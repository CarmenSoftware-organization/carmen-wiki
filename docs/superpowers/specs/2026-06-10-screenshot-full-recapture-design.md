# Full Screenshot Re-capture — Design

**Date:** 2026-06-10
**Goal:** Refresh every Inventory-book wiki screenshot to the most-current frontend state, picking up new/changed routes since the last run, then publish to the dev Wiki.js instance — without regressing pages that already embed working images.

## Decisions

| Decision | Choice |
|----------|--------|
| Route scope | **Re-discover routes** from `carmen-inventory-frontend-react/routes` first, then capture (Inventory book only; Platform book out of scope) |
| Seed IDs | **Re-discover all seedIds** fresh against the tenant the frontend (`:3000`) is logged into — do not reuse possibly-stale `seed-ids.json` |
| Locale | **EN only** — capture EN, images shared by both EN + TH pages (current convention) |
| Publish | **Commit + push** — commit PNGs to `carmen-wiki`, commit manifest/seed changes to e2e repo, upload assets to Wiki.js, sync any changed pages |
| Orchestration | **Approach B — Overwrite-in-place + coverage diff** (see below) |

## Orchestration: Approach B (Overwrite-in-place)

Keep existing PNGs. Re-discover to update the manifest and seed IDs, re-capture overwriting on success, and **leave the previous image in place for any route that fails this run** so no page that already embeds an image regresses to a 404. Produce a coverage diff so we can see exactly what changed.

Rejected alternatives:
- **A. Clean-slate** (delete all, capture fresh) — any route that fails this run (incl. the ~14 historically uncapturable: PR admin-perm gap, inventory-adjustment 500, counting-grid timeouts) would turn its embed into a 404. Unacceptable regression.
- **C. Two-phase temp + diff promote** — more control than needed for a refresh; too slow.

## Pipeline (existing tooling — no new code expected)

**Capture side — `carmen-inventory-frontend-e2e/tests/wiki-screenshots/`:**
1. `bun run wiki:seed` — `seed-manifest.ts` regenerates `manifest.ts` from frontend `app/**/page.tsx`. `inferModuleSlug` is deterministic → existing routes keep their module/slug (embeds stay valid); only the route delta changes.
2. `bun run tests/wiki-screenshots/discover-seeds.ts` — regenerate `seed-ids.json` (validated detail-record IDs for the current tenant).
3. `bun run wiki:capture` — Playwright `wiki-screenshots` project, EN locale, bounded waits (domcontentloaded + capped network/skeleton settle, no fullPage), writes to `../carmen-wiki/assets/screenshots/inventory/<module>/<slug>.png`.
4. `bun run wiki:coverage` — coverage report.

**Publish side — `carmen-wiki/scripts/`:**
5. `upload_assets.sh <changed.png>...` — upload to Wiki.js Asset Manager.
6. `push_pages.py <changed.md>...` — only if any page content/embeds changed (GraphQL `pages.update`).

## Known risks & handling

1. **Asset-folder mapping** — `upload_assets.sh` only knows folder ids for ~17 wiki modules. Re-discovery may surface routes whose `inferModuleSlug` module is not in that map (sub-entities like `category`, `currency`). Handling: after capture, diff the set of module folders that have changed PNGs against the folder-id map; for any unmapped module, decide upload target (extend the map or fold into the correct wiki folder) before uploading. Do not silently skip.
2. **Slug stability** — verified low risk: the current manifest was generated by the same `inferModuleSlug`, so re-seeding yields identical module/slug for unchanged routes. Confirm via `git diff manifest.ts` before capturing; if any existing route's module/slug changed, reconcile embeds.
3. **Tenant validity** — seed discovery and capture must run against the same logged-in tenant. Verify the frontend session tenant before discovery; if detail pages 403/empty, the tenant lacks data.
4. **Historically uncapturable routes** — best-effort. Log them in the coverage diff; their existing images stay (Approach B). Do not block the run on them.

## Execution phases

- **Phase 0 — Prep:** verify `:3000` healthy + tenant has data; confirm `scripts/.env` has `WIKI_API_URL`/`WIKI_API_TOKEN`; clean git state in both repos.
- **Phase 1 — Re-discover routes:** run `wiki:seed`; review `git diff manifest.ts` for the route delta + slug stability; reconcile module mapping for new routes.
- **Phase 2 — Re-discover seeds:** run `discover-seeds.ts`; sanity-check `seed-ids.json` count vs route count.
- **Phase 3 — Capture (EN):** run `wiki:capture`; then `wiki:coverage`.
- **Phase 4 — Coverage diff:** report newly-captured / still-failing / new-route / removed-route / stale-kept.
- **Phase 5 — Commit + push:** commit e2e (manifest, seed-ids) and carmen-wiki (PNGs); `upload_assets.sh` changed PNGs (after resolving risk #1); `push_pages.py` only for changed pages; verify a sample renders on the dev Wiki.js.

## Success criteria

- Coverage ≥ previous 120/134 (no net regression); ideally higher if new routes captured.
- No page that previously embedded a working image now 404s.
- Changed assets visible on `http://dev.blueledgers.com:3987/`.
- Coverage diff documented in the run summary.
