# ใบรับสินค้า (Goods Receive Note) — Test Scenarios — Audit & Config

> **At a Glance**
> **Persona:** Audit / Config (Auditor + System Administrator) &nbsp;·&nbsp; **โมดูล:** [[good-receive-note]] &nbsp;·&nbsp; **Scenario:** ~29
> **หมวด:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **Coverage E2E:** map กับ `501-grn.spec.ts` ใน `../carmen-inventory-frontend-e2e/` (พื้นผิว audit-log และ config โดยมาก planned coverage manual / API-test)

หน้านี้จับ scenario การทดสอบที่ persona Audit / Config — แยกเป็นสอง sub-persona ที่ไม่ใช่ transaction: **Auditor** (read-only เคร่งครัดบนชุดข้อมูล GRN ทั้งหมด; รันคิวรี activity-log, review ผลลัพธ์ three-way-match, trace lot-recall) และ **System Administrator** (ตั้งค่ารูปแบบ generate เลข lot, RBAC role และเกณฑ์การอนุมัติ, รหัสภาษี / สกุลเงิน / reason code และ endpoint integration ไปยังโมดูล [[purchase-order]] / [[inventory]] / Finance / Vendor) — ขับเคลื่อนตรง ๆ ในโมดูล `good-receive-note` ทั้งสอง sub-persona ไม่เข้าร่วมใน state machine `draft / saved / committed / voided` บน `enum_good_received_note_status`: คิวรีของ Auditor return ข้อมูลโดยไม่เขียน state ใด ๆ และการแก้ของ Sysadmin เปลี่ยนกติกาที่ apply กับ GRN **ในอนาคต** (GRN in-flight `draft` / `saved` เก็บ snapshot กติกาที่บังคับใช้ตอน save; GRN `committed` immutable) Rule ID ที่ share ที่ test ของ persona นี้ exercise คือ `GRN_AUTH_009` (สิทธิ์ export read-only), `GRN_AUTH_010` (segregation Receiver ≠ Purchaser บังคับใช้ที่ commit, ตั้งค่าใน panel RBAC), `GRN_AUTH_011` (workflow-derived authorization ที่ `workflow_current_stage` ปัจจุบัน), `GRN_POST_010` (ข้อห้าม void หลัง commit ที่ gate co-authorisation Inventory Manager + Finance reversal workflow), `GRN_XMOD_001`–`GRN_XMOD_010` (การ wire integration ที่ Sysadmin ดูแลข้าม PO / Inventory / Finance / Vendor) และกติกา snapshot ที่ปกป้อง GRN in-flight จากการเปลี่ยน config ของ Sysadmin (บรรยายใน Section 3 ของ [03-user-flow-audit-config.md](./03-user-flow-audit-config.md) และไม่ได้กำหนด rule ID แยก — อยู่ในชั้น persistence ของ timestamp effective-from) Prefix เดียวบน test ID คือ `AUD-` Scenario จัดกลุ่มเป็น **happy path** (คิวรี activity-log ของ Auditor และ lot-recall trace, การเปลี่ยนรูปแบบเลข lot และการปรับ RBAC ของ Sysadmin, การดูแลรหัสภาษี / reason-code, cutover integration) **permission** (Auditor read-only allow, Auditor write deny, Sysadmin configure allow, Sysadmin no-transact deny, การ export ฟิลด์ sensitive approval, co-authorisation elevated หลัง commit) **validation** (การรักษา snapshot บน GRN in-flight, การบังคับ export-approval, การป้องกัน deadlock บนการเปลี่ยน RBAC, การ check backwards-compatibility ของรูปแบบเลข lot, ขอบเขต scan date-range, การ reject shape รหัสภาษี, save ที่ no-op) และ **edge case** รอบขอบเขต snapshot in-flight, rollback config เสีย, gap audit-log ระหว่าง lot-recall, query timeout, mass-RBAC activation และการชนรูปแบบเลข lot Handoff ข้าม persona ที่ pivot จาก persona นี้ (Auditor → Quality / Recall lead, Sysadmin → cross-module change board) อยู่ใน [04-test-scenarios.md](./04-test-scenarios.md)

## 2. Permission / Authorization

| # | Scenario | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| AUD-PERM-01 | Auditor อ่าน activity log และผลลัพธ์ three-way-match | **Allow read-only** `GRN_AUTH_009` ให้ Auditor (รวมใต้สิทธิ์ View, export reports) access read เต็มข้าม `doc_status` ทั้งหมด (`draft / saved / committed / voided`) และ JSON `workflow_history` ประวัติการแก้บรรทัด comment thread การเชื่อม credit-note และผลลัพธ์ three-way-match หน้าจอโมดูล audit render render-only; ไม่มี affordance Save / Commit / Void / Edit ปรากฏ; endpoint API ที่อยู่เบื้องหลัง reject การเรียก write ใด ๆ จาก token role Auditor ด้วย `"Auditor role is read-only — write actions are not permitted on tb_good_received_note."` Map กับ AUD-HP-01..03 |
| AUD-PERM-02 | Auditor พยายามแก้บรรทัด GRN หรือ post journal | **Deny — role scope** การเรียก API ตรงไปยัง `PATCH /grn/{id}` หรือ `POST /grn/{id}/commit` จาก token Auditor ถูก reject ที่ชั้น RBAC ด้วย `403 Forbidden` และเหตุผล `"Auditor role is not authorised for write actions on the GRN module."`; ไม่มี `doc_status` transition; ไม่มี append `workflow_history`; การ reject เองถูก log บน audit-of-audit trail สำหรับ compliance เทียบเท่า SOX Mirror ของขอบเขต read-only ของ AUD-PERM-01 ฝั่ง write |
| AUD-PERM-03 | Sysadmin ตั้งค่ารูปแบบเลข lot, RBAC, รหัสภาษี / reason code, integration | **Allow — config scope only** Role Sysadmin ถือ panel configuration (รูปแบบเลข lot, RBAC role และเกณฑ์การอนุมัติรวมการบังคับ segregation `GRN_AUTH_010`, รหัสภาษี / สกุลเงิน / reason code, endpoint integration ไปยัง [[purchase-order]] / [[inventory]] / Finance / Vendor ตาม `GRN_XMOD_001`–`GRN_XMOD_010`) GRN configuration console render panel สี่ตัวเฉพาะสำหรับ Sysadmin; role Receiver / Inventory Manager / Finance / Purchaser ไม่เห็น console Map กับ AUD-HP-05..08 |

## 4. Edge Case

| # | Scenario | เงื่อนไข | Expected |

| AUD-EDGE-01 | ขอบเขต snapshot in-flight — Sysadmin save ระหว่าง GRN `saved` ที่ Receiver เปิด | Receiver มี GRN `G-AE1` เปิดที่ `doc_status = saved` ที่เวลา `T`; Sysadmin save อัตรารหัสภาษีใหม่ที่ `T + 100ms`; Receiver คลิก **Commit** ที่ `T + 1s` | **In-flight เก็บ snapshot ตามกติกา snapshot** Snapshot รหัสภาษีของ `G-AE1` ที่จับตอน save (เวลา `T - X`) preserve ผ่าน commit; commit อ่าน snapshot ที่ freeze ของ `G-AE1` ไม่ใช่อัตราใหม่ระดับ tenant; `doc_status` เลื่อน `saved → committed` ที่อัตรา snapshot; อัตราใหม่ apply กับ GRN ที่สร้างที่ `T + 100ms` หรือทีหลัง Mirror decision branch กติกา snapshot ของ Section 3 ของ [03-user-flow-audit-config.md](./03-user-flow-audit-config.md) |
| AUD-EDGE-02 | Rollback การเปลี่ยน config เสียหลัง partial in-flight effect | Sysadmin save config endpoint integration เสียที่ `T` ที่ทำให้ AP-accrual fan-out `GRN_POST_006` เสีย 2 นาที (ไม่มี GRN in-flight `saved → committed` ระหว่าง window เสีย — จับโดย monitoring ที่ `T + 90s`); Sysadmin roll back ที่ `T + 2min` | **คืน config ก่อนหน้าจาก configuration history; ไม่มี effect in-flight ให้ undo** Configuration history เก็บเวอร์ชันก่อนหน้าตาม Section 2.2 ขั้น 5 ของ [03-user-flow-audit-config.md](./03-user-flow-audit-config.md); Sysadmin คลิก **Roll back to previous version**; config ก่อนหน้า active พร้อม `effective_from = T + 2min`; GRN ใหม่ commit สะอาดภายใต้ endpoint ที่คืน; configuration-history แสดงสอง entry (save เสียที่ `T`, rollback ที่ `T + 2min`) สำหรับ Auditor review ถ้ามี GRN commit ระหว่าง window เสียพร้อม fan-out failure เส้นทาง recovery จะ route ไปยังคิว AP-investigation และ `workflow_history` ของ GRN จะพก event failure — แต่ไม่มี GRN commit ในเคสนี้ |
| AUD-EDGE-03 | Lot-recall trace พร้อม gap audit-log | Auditor รัน lot-recall trace บน `LOT-GAP-7`; แถว `tb_inventory_transaction_detail` ที่อยู่เบื้องหลังรวมแถวหนึ่งที่ `workflow_history` ของ GRN ต้นทางขาด entry `committed` (data-integrity gap จาก migration ประวัติศาสตร์); inventory transaction เองมีอยู่ | **Surface gap ใน trace report** Lot trace return chain of custody ผ่าน `tb_inventory_transaction_detail.lot_no` → `tb_inventory_transaction` → `tb_good_received_note_detail_item.inventory_transaction_id` → `tb_good_received_note.id` ถูกต้อง; panel audit-log บน GRN ที่กระทบ flag `"workflow_history missing the committed entry — inventory side is consistent (transaction present), audit-log side has a gap; recommend forensics."`; การ export chain-of-custody รวม gap flag สำหรับ external auditor; ไม่มี auto-repair, ไม่มี auto-fill ของ entry ที่ขาด (รักษา integrity forensic) |
| AUD-EDGE-04 | Query timeout ของ Auditor บนชุดข้อมูลใหญ่ | Auditor กรอก filter ที่แม้ภายใน cap date-range (AUD-VAL-05) return จำนวน candidate-row เกิน tenant query-cost budget (เช่น 500K GRN event สำหรับ scan 12 เดือน all-vendor) | **Server timeout พร้อม graceful degradation** Backend cancel คิวรีที่ timeout ที่ config (เช่น 30s) และ return `"Audit-log query exceeded compute budget; partial result up to T+30s returned. | AUD-EDGE-05 | Mass-RBAC activation — 200 ผู้ใช้ใหม่ได้รับ role Receiver ครั้งเดียว | Tenant onboard business unit ใหม่และ Sysadmin bulk-grant role `Receiving Clerk` ให้ 200 ผู้ใช้ใหม่ผ่าน CSV import บน panel RBAC | **Allow พร้อม effective-from แบบ batch; entry activity-log ต่อผู้ใช้** Import process 200 ผู้ใช้ใน transaction เดียว; impact preview ที่ขั้น 4 ยืนยันไม่มี conflict segregation-of-duties (ไม่มีใน 200 เป็น Purchaser บน PO active ใด ๆ); ที่ save 200 ผู้ใช้ทั้งหมดเห็น scope Receiver ที่ความพยายาม action ถัดไปตาม decision branch Section 3 — ไม่มีการ cancel transaction ย้อนหลัง; entry configuration-history เขียนต่อผู้ใช้ (200 entry) สำหรับ Auditor visibility; เวลา transaction รวมต่ำกว่า bulk-RBAC budget ของ tenant; ถ้าผู้ใช้คนเดียวล้มเหลว conflict check ทั้ง batch roll back atomically เพื่อรักษา invariant all-or-nothing |
