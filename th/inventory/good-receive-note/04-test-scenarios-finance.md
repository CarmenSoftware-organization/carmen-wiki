# ใบรับสินค้า (Goods Receive Note) — Test Scenarios — Finance

> **At a Glance**
> **Persona:** Finance (Officer / AP Clerk + Finance Manager / Controller) &nbsp;·&nbsp; **โมดูล:** [[good-receive-note]] &nbsp;·&nbsp; **Scenario:** ~30
> **หมวด:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **Coverage E2E:** map กับ `501-grn.spec.ts` ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ scenario การทดสอบที่ persona Finance (**Finance Officer / AP Clerk** ที่โต๊ะ three-way match รายวัน บวก **Finance Manager / Controller** ที่ขอบเขต sign-off ปิดงวด) ขับเคลื่อนตรง ๆ ในโมดูล `good-receive-note` Finance เป็นผู้เข้าร่วม **หลัง commit** — การเข้าสู่ flow นี้สันนิษฐานว่า Inventory Manager ได้ fire transition `saved → committed` บนเส้นทาง Receiver แล้ว ซึ่งได้ขึ้น inventory accrual (`Dr Inventory / Cr GRN Clearing`, `GRN_POST_006`) และเพิ่มสต๊อกคงคลังไปแล้ว การเข้าร่วมของ Finance ครอบคลุมสองกิจกรรมที่ชัดเจน: **การปรับการเงินก่อน AP** บน GRN `committed` (แก้วิธีจัดสรร extra-cost — `manual`, `by_value`, `by_qty` ตาม `enum_allocate_extra_cost_type` — และรหัสภาษีขณะ invoice vendor ยังไม่มา; อนุญาตโดย `GRN_AUTH_007`); และ **three-way match** เมื่อ invoice มาถึง (match `PO ↔ GRN ↔ Invoice` บน qty และราคาภายใน tolerance ของ tenant, post journal ฝั่ง AP `Dr GRN Clearing / Cr AP-Trade` เมื่อสำเร็จตาม `GRN_POST_008`, flag ความคลาดเคลื่อนกลับให้ Purchaser เมื่อล้มเหลวตาม `GRN_POST_009`) Finance Manager รัน **sign-off ปิดงวด** เพิ่ม — reconcile inventory sub-ledger กับบัญชี GL inventory control, age ยอด GRN Clearing เปิด และ lock งวด สิ่งสำคัญ: **enum `doc_status` ของ GRN ไม่ถูก transition โดย Finance** — `saved → committed` เป็น posting event เดียวที่เอกสารผ่าน; ผลลัพธ์ match อยู่บน match flag แยก ไม่อยู่บน enum สี่สถานะ Scenario จัดกลุ่มเป็น **happy path** (ปรับการจัดสรรก่อน AP, three-way match สะอาด, match GRN บางส่วน, ราคาภายใน tolerance auto-pass, ปรับ FX, sign-off ปิดงวด) **permission** (allow ปรับก่อน AP, allow three-way match, allow post AP, deny override match ล้มเหลว, SoD Finance ≠ Purchaser/Receiver, lock งวดโดย Finance Manager) **validation** (negative test กับการ flag ความคลาดเคลื่อน, GRN ขาด, GRN voided, split หลาย invoice, block งวดที่ปิด, match invoice ซ้ำ) และ **edge case** รอบขอบเขต tolerance, อัตรา FX เท่ากัน, รวมหลาย GRN กับ invoice เดียว (1:N), การ post invoice พร้อมกัน และความแม่นยำ decimal บนยอด AP Handoff ข้าม persona ที่ pivot จาก Finance (Finance ↔ Purchaser bounce-back บนความคลาดเคลื่อนที่ flag, Finance ↔ Controller ที่ปิดงวด) อยู่ใน [04-test-scenarios.md](./04-test-scenarios.md)
