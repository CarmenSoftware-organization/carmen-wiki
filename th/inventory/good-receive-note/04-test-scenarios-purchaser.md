# ใบรับสินค้า (Goods Receive Note) — Test Scenarios — Purchaser

> **At a Glance**
> **Persona:** Purchaser (Procurement Officer + Department Manager) &nbsp;·&nbsp; **โมดูล:** [[good-receive-note]] &nbsp;·&nbsp; **Scenario:** ~20
> **หมวด:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **Coverage E2E:** map กับ `501-grn.spec.ts` ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ scenario การทดสอบที่ persona Purchaser (**Purchaser / Procurement Officer** ที่ขึ้น PO ต้นทาง บวก subset **Department Manager** ที่เป็นเจ้าของ cost-centre ที่ GRN post เข้า) ขับเคลื่อนตรง ๆ ในโมดูล `good-receive-note` Purchaser เป็นผู้เข้าร่วม **review-only** บน GRN — พวกเขา **ไม่** สร้าง GRN ที่ dock (เส้นทาง Receiver) **ไม่** save การกรอกบรรทัด **ไม่** post commit ที่ไม่อาจย้อนคืน (เส้นทาง Inventory Manager) และ **ไม่** แก้เอกสาร GRN ใน state ใด ๆ งานของพวกเขาถูกจำกัดโดยสามกิจกรรม: (1) react กับการแจ้งเตือนที่ fire โดย transition `draft → saved` ของ Receiver (variance-flagged) หรือ `saved → committed` ของ Inventory Manager บน GRN ที่ PO ต้นทางพวกเขาเป็นเจ้าของ (2) review การรับเทียบกับ PO ที่พวกเขาขึ้น — `received_qty` vs `pending_qty`, `accepted_qty` vs `received_qty`, lot / expiry บน `tb_inventory_transaction_detail` ที่ link, ใบส่งของแนบ และ price drift เทียบ `[[vendor-pricelist]]` — และ (3) เป็นเจ้าของการแก้ฝั่ง vendor สำหรับ variance ทุกอย่าง (ตาม short-ship, เจรจา credit / replacement สำหรับเสียหาย, ขึ้น amendment PO, เจรจาราคา drift ใหม่) Subset Department Manager ทำการ review เดียวกันที่ scope กับบรรทัดที่ post กับ cost-centre ของแผนกตน และ sign off ผลกระทบงบประมาณ Segregation of duties (`PO_AUTH_010` carry มาจากโมดูล PO ต้นน้ำ) ห้ามผู้ใช้ที่ขึ้นหรือ transmit PO post GRN กับมันชัดเจน; ชุดทดสอบด้านล่าง verify ขอบเขตนั้นทุกแถว permission Scenario จัดกลุ่มเป็น **happy path** (รับการแจ้งเตือน, ปิดการรับสะอาด, ตาม short-ship, escalate price-variance, sign-off cost-centre ของ Department Manager, ขึ้น amendment PO เพื่อแก้ over-receipt) **permission** (scope view ตามการเป็นเจ้าของ PO, SoD บน modify / create / commit, scope read cost-centre ของ Department Manager) **validation** (การ deny action เปลี่ยน state ที่เป็นของ Receiver / Inventory Manager, การ deny override cost-centre บนเอกสาร GRN, SoD block self-receipt) และ **edge case** (เวลาการแจ้งเตือนตอน commit, ขอบเขต tolerance variance ที่ขับเคลื่อนการตัดสินใจ chase, review รวมหลาย PO, review ข้าม cost-centre ของ Department Manager) Handoff ข้าม persona ที่ pivot จาก Purchaser (Scenario 4, 6 ของภาพรวมแม่ — handoff ปฏิเสธคุณภาพและ handoff price-variance) อยู่ใน [04-test-scenarios.md](./04-test-scenarios.md) ไม่ใช่ที่นี่

| PUR-VAL-03 | Department Manager override การจัดสรร cost-centre บนบรรทัด GRN ไม่ได้ | GRN `committed` post กับ cost-centre `CC-A` (resolve จาก mapping default ของ location inventory); Department Manager ไม่เห็นด้วยกับการจัดสรรที่ resolve และต้องการ `CC-B`; พยายาม PATCH ฟิลด์ `cost_centre_id` ของ inventory transaction | **Reject** — terminal state + role check Server return `"GRN at status committed is locked; cost-centre re-allocation must be performed by Finance via journal adjustment."` cost-centre คง `CC-A`; GRN ไม่เปลี่ยน; เส้นทางที่ถูกของ Department Manager คือเขียน comment sign-off บน activity log ของ GRN flag ความไม่เห็นด้วยและ handoff ไป **Finance** สำหรับการจัดสรร journal ใหม่ (ย้าย cost-centre ผ่าน entry GL ชดเชย ไม่ใช่ mutate เอกสาร GRN) ยืนยันขอบเขต PUR-HP-05 |

## 4. Edge Case

| # | Scenario | เงื่อนไข | Expected |
| - | -------- | -------- | -------- |
| PUR-EDGE-01 | เวลาการแจ้งเตือน — Purchaser รับการแจ้งเตือนตอน commit ไม่ใช่ก่อน | Receiver ขับ GRN สะอาด `draft → saved → committed` ในกะเดียว เส้นทาง Receiver fire การแจ้งเตือนบน transition `draft → saved` เฉพาะถ้ามี comment variance บนอย่างน้อยหนึ่งบรรทัด; transition `saved → committed` ของ Inventory Manager fire ไม่มีเงื่อนไข | สำหรับการรับสะอาด (ไม่มี comment variance) Purchaser รับการแจ้งเตือน **หนึ่ง** ครั้งที่ `saved → committed` carry flag `clean`; transition `draft → saved` ไม่ fire การแจ้งเตือน Purchaser สำหรับการรับ variance-flagged Purchaser รับการแจ้งเตือน **สอง** ครั้ง — หนึ่งบน `saved` (flag เช่น `short`) เพื่อให้การ chase vendor สามารถเริ่มขณะ GRN ยัง uncommitted และหนึ่งบน `committed` ยืนยันการ posting; ทั้งสองการแจ้งเตือน deep-link ไปยัง GRN เดียวกันโดย state พัฒนาระหว่างสองคลิก ไม่มีการแจ้งเตือนซ้ำ ไม่มีการแจ้งเตือนพลาด (idempotency guard บน `tb_user_notification` key ตาม `(grn_id, doc_status_to, recipient_id)`) |
| PUR-EDGE-02 | Threshold variance ที่ขอบเป๊ะของกฎตัดสินใจ chase | GRN `committed` กับ PO ของตนที่ `received_qty = pending_qty × (1 − tenant_short_tolerance)` เป๊ะ — ที่ขอบของ "ยอมรับการรับ short" vs "chase vendor"; tolerance short ของ tenant คือ `5%`; `order_qty = 10` ดังนั้นขอบคือ `received_qty = 9.5` | **ที่ขอบ (`received_qty = 9.5`):** flag สรุป variance ของ GRN อาจ render เป็น `clean` (ภายใต้ config tenant: short-within-tolerance) หรือ `short` (ภายใต้ config เข้มงวด: shortfall ใด trigger chase) Decision branch ของ Purchaser ใน [03-user-flow-purchaser.md](./03-user-flow-purchaser.md) Section 3 ("การรับสะอาด" vs "การรับ short") ตาม config ของ tenant — ที่ขอบเป๊ะ การ chase อยู่ที่ดุลพินิจของ Purchaser และ comment activity-log สะท้อน **หนึ่งหน่วยใต้ขอบ (`received_qty = 9.49`):** ชัดเจน `short` chase บังคับ **หนึ่งหน่วยเหนือ (`received_qty = 9.51`):** ชัดเจน `clean` ไม่ chase การเลื่อน `po_status` ของ PO ขับโดยอิสระโดย `PO_POST_007` และไม่กระทบโดยการตัดสินใจนี้ |
| PUR-EDGE-03 | Review รวมหลาย PO — GRN เดียวคลุมสอง PO ของ Purchaser | vendor เดียวกันส่งในรถบรรทุกเดียวคลุม `PO-A` (บรรทัดเดียว pending 4) และ `PO-B` (สองบรรทัด pending 6 ทั้งคู่); Receiver รวมเป็น GRN เดียวสามบรรทัดตาม RCV-EDGE-06; ทั้งสอง PO เป็นของ Purchaser คนเดียวกัน; บรรทัดหนึ่ง short 1 หน่วย อีกสองสะอาด | Purchaser รับการแจ้งเตือน **หนึ่ง** ครั้งต่อ state transition ของ GRN (ไม่ใช่หนึ่งต่อ PO) carry หมายเลข PO ทั้งสองและ breakdown variance ต่อบรรทัด การเปิด GRN render ทั้งสามบรรทัด (scope ความเป็นเจ้าของ cross-PO รวมสำหรับ view read เพราะ Purchaser เป็นเจ้าของทั้งสอง PO); flag สรุป variance คือ `short` (บรรทัดใด short flip flag ทั้ง GRN); chase scope กับบรรทัด short หนึ่งและระบุ PO ที่มัน sit กับ (`PO-A` บรรทัด `L1` หรือ `PO-B` บรรทัด `L1` / `L2`); บรรทัดสะอาดไม่ต้องการ action Panel pricelist-deviation render ต่อบรรทัด ต่อ window (vendor / item / วันที่รับ) การแก้ (chase บนบรรทัด short) log หนึ่งครั้งบน activity log ของ GRN พร้อม reference PO / บรรทัด; ไม่ต้องการ amendment PO ถ้าการส่งครั้งถัดไปจะปิดยอด Map กับ TC-GRN-040002, TC-GRN-040004 จาก review ฝั่ง Purchaser |
| PUR-EDGE-04 | Review ของ Department Manager span GRN หลาย cost-centre | GRN `committed` รวมการส่ง vendor เดียวข้ามสอง location inventory ที่ cost-centre default เป็นของ **สอง** Department Manager คนละแผนก (`CC-A` เป็นของ Dept Mgr A, `CC-B` เป็นของ Dept Mgr B); สามบรรทัดรวม | Department Manager แต่ละคนรับการแจ้งเตือน scope กับบรรทัด cost-centre ของตนเท่านั้น; เปิด GRN render บรรทัดบน cost-centre ของตนเต็ม (พร้อม context header) และ mask บรรทัดบน cost-centre อื่น (ตาม PUR-PERM-06) Panel pricelist-deviation render เฉพาะสำหรับบรรทัดในขอบเขต Department Manager แต่ละคน sign off อย่างอิสระบนบรรทัด cost-centre ของตน (PUR-HP-05) — sign-off บันทึกเป็น entry activity-log แยกพร้อม `cost_centre_id` และ `actor_id` แยกแยะพวกเขา GRN เอง **ไม่** ถูก mutate โดย signoff ใด (PUR-VAL-03); ความไม่เห็นด้วย cross-cost-centre บนบรรทัด share เป็นไปไม่ได้ (แต่ละบรรทัดมี `cost_centre_id` ที่ resolve หนึ่งเป๊ะ) |
