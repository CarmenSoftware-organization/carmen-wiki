# ใบรับสินค้า (Goods Receive Note) — Test Scenarios — Receiver

> **At a Glance**
> **Persona:** Receiver (Store Keeper / Receiving Clerk + Inventory Manager) &nbsp;·&nbsp; **โมดูล:** [[good-receive-note]] &nbsp;·&nbsp; **Scenario:** ~31
> **หมวด:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **Coverage E2E:** map กับ `501-grn.spec.ts` ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ scenario การทดสอบที่ persona Receiver (**Store Keeper / Receiving Clerk** ที่ dock บวก **Store Manager / Inventory Manager** ที่เป็นเจ้าของการ post `saved → committed` ที่ไม่อาจย้อนคืน) ขับเคลื่อนตรง ๆ ในโมดูล `good-receive-note` การเข้าร่วมของ Receiver เริ่มเมื่อ PO มาถึงที่ `po_status ∈ {sent, partial}` (หรือสำหรับ `doc_type = manual` เมื่อเกิดการรับ ad-hoc ที่ไม่มี PO ต้นทาง) และจบเมื่อ GRN ออกจากโซน receiver — ผ่าน `committed` (handoff ไป Finance สำหรับ three-way match), `voided` จาก `draft` / `saved` (ไม่มีผลกระทบ inventory / GL) หรือ variance ที่ flag handoff ไปยัง Purchaser ([03-user-flow-receiver.md](./03-user-flow-receiver.md) Section 4) Scenario จัดกลุ่มเป็น **happy path** (เดินสี่สถานะจาก `draft → saved → committed`, GRN manual, GRN บางส่วน, batch commit, จับ lot/expiry และยอมรับปัญหาคุณภาพพร้อม variance) **RBAC** (scope Store Keeper เทียบ Inventory Manager, segregation of duties บังคับใช้โดย `PO_AUTH_010` ที่สืบทอดจาก PO ต้นน้ำ และ co-authorisation elevated ที่ต้องการสำหรับ reversal หลัง commit) **validation** (negative test กับ `GRN_VAL_001`–`GRN_VAL_014` ที่ Receiver trigger ได้ตอน save หรือ commit) และชุด **edge case** เล็ก ๆ รอบขอบเขต tolerance, ความแม่นยำ decimal บน qty lot, post พร้อมกันบน PO เดียวกัน, sweep auto-commit สิ้นงวดจับ GRN `saved` เก่า และผลคุณภาพหลายบรรทัดผสม Handoff ข้าม persona ที่ pivot จาก Receiver (Scenario 1, 2, 3, 4, 7, 9, 10 ในภาพรวมแม่) อยู่ใน [04-test-scenarios.md](./04-test-scenarios.md) ไม่ใช่ที่นี่

| RCV-PERM-06 | Reversal หลัง commit — ต้อง co-authorisation elevated | **Deny เมื่อขึ้นโดย Inventory Manager คนเดียว** Reversal หลัง commit ของ GRN `committed` (`committed → voided`) ต้อง co-authorisation โดย **Inventory Manager AND Finance** (หรือ System Administrator บนเหตุ audit) Action **Void** ผู้ใช้คนเดียวบน GRN `committed` block ด้วย `"Post-commit reversal requires co-authorisation from Finance."`; เฉพาะ flow สองฝ่ายเท่านั้นที่ fire inventory transaction ชดเชย, reversal cost-layer, decrement `received_qty` PO และ entry AP reverse Receiver อาจขึ้น GRN replacement หลังจากนั้นถ้าจะบันทึกการรับจริงใหม่ Map กับ TC-GRN-120003 (Void Committed GRN) |
| RCV-PERM-07 | Receiver / Inventory Manager void GRN `draft` หรือ `saved` พร้อมเหตุผล | **Allow** Void ก่อน commit อยู่ในขอบเขตของเส้นทาง Receiver (sub-persona ใด ๆ): `draft → voided` และ `saved → voided` อนุญาตทั้งคู่พร้อมข้อความเหตุผลที่ต้องระบุและไม่สร้างผลกระทบ inventory / GL (RCV-HP-09) ปุ่ม void enable บน `draft` และ `saved` สำหรับ role Receiver / Inventory Manager และ disable บน `committed` |

## 4. Edge Case

| # | Scenario | เงื่อนไข | Expected |
| - | -------- | -------- | -------- |
| RCV-EDGE-01 | ขอบเขต tolerance เป๊ะบน over-receipt | บรรทัด PO `L1`: `order_qty = 10`, `received_qty = 0`, `cancelled_qty = 0`; over-receipt tolerance ของ tenant = `5%`; Receiver กรอก `received_qty = 10.5` (เป๊ะที่ขอบบน) `accepted_qty = 10.5` | **Accepted** — `GRN_VAL_009` clause สอง (tolerance ของ tenant ผ่อน pending-qty cap) `received_qty = 10.5` ภายใน tolerance (`10 × 1.05 = 10.5`); save และ commit สำเร็จทั้งคู่; `tb_purchase_order_detail.received_qty = 0 → 10.5`; การประเมินบรรทัด `received_qty (10.5) ≥ order_qty − cancelled_qty (10)` → รับเต็ม; `po_status = sent → completed` ตาม `PO_POST_007` ต้นน้ำ Setting `received_qty = 10.51` เกินขอบและถูก reject ตาม RCV-VAL-08 |
| RCV-EDGE-02 | ความแม่นยำ decimal บน qty lot | บรรทัดอนุญาตความแม่นยำ 3-dp; Receiver กรอก `received_qty = 5.123`, `accepted_qty = 4.998`; การจับ multi-lot sum เป็น `received_qty` บนแถว `tb_inventory_transaction_detail` ที่ link (เช่น `lot A = 3.000`, `lot B = 2.123`); conversion factor `1.234567` | **Accepted ที่ storage** `received_qty`, `accepted_qty` และ qty ต่อ lot ทั้งหมดเก็บที่ความแม่นยำ UoM ของบรรทัด (3 dp); `received_base_qty = Round(5.123 × 1.234567, 3)` คำนวณตาม `GRN_VAL_008`; qty ต่อ lot บน `tb_inventory_transaction_detail` ต้อง sum เป๊ะเป็น `received_qty` ของบรรทัด (ไม่มี rounding drift carry เข้า total); variance คุณภาพ `received_qty − accepted_qty = 0.125`; Decimal ที่เก็บทั้งหมด fit ใน `Decimal(20, 5)` สำหรับเงินและ `Decimal(15, 5)` สำหรับอัตรา |
| RCV-EDGE-03 | Post GRN พร้อมกันบน PO เดียวกันจากสอง session Receiver | Receiver A (Store Keeper) และ Receiver B (session แยก) ทั้งคู่เปิด `PO-X` (`po_status = sent`, `received_qty = 0`, `order_qty = 10`); A save และ commit `received_qty = 6` ที่ `T`; B save และ commit `received_qty = 4` ที่ `T + 500ms` กับบรรทัดเดียวกัน ทั้งคู่ snapshot `doc_version = N` | **Commit แรกชนะ atomically** GRN ของ A commit, `tb_purchase_order_detail.received_qty = 0 → 6`, `po_status = sent → partial`, `doc_version → N+1` Submit ของ B reconcile กับ state ใหม่ — ทั้งโมดูล GRN re-read `pending_qty = 4` และ `received_qty = 4` ของ B commit สะอาด (เลื่อน `received_qty → 10`, `po_status → completed`) หรือ optimistic-concurrency guard reject B ด้วย `"This PO was modified by another user. Please refresh and re-apply your changes."` และ B retry กับ snapshot ใหม่ ไม่มี double-write ไม่มี lost update ไม่มี `received_qty > order_qty` overflow นอก over-receipt tolerance ไม่มี on-hand เพิ่มซ้ำ |
| RCV-EDGE-04 | Sweep auto-commit สิ้นงวดจับ GRN `saved` เก่า | System Administrator ตั้งค่า sweep schedule ที่ commit GRN `saved` ที่เก่ากว่า grace window ของ tenant (เช่น 24h); GRN สองใบนั่งใน `saved` เกิน threshold — ใบหนึ่งใช้ได้ ใบหนึ่ง PO ต้นทาง `voided` แล้ว | **Sweep รันชุดกฎ commit-time เดียวกัน** GRN ที่ใช้ได้: `GRN_VAL_011`–`GRN_VAL_014` ทั้งหมดผ่าน → `saved → committed` โดย sweep; การเขียน inventory, cost layer, การเลื่อน PO, AP accrual fire ทั้งหมดภายใต้ context system / scheduled user; activity log บันทึก `{ doc_status: 'committed', action: 'auto_commit_sweep' }` GRN ที่ PO voided: `GRN_VAL_013` block → sweep ข้าม log ความล้มเหลว route เอกสารกลับไปคิว Inventory Manager สำหรับแก้ manual (void ตาม RCV-PERM-07 หรือ re-anchor กับ PO ที่ใช้ได้); ไม่มีการเขียน inventory Finance หยิบ GRN ที่ commit ใหม่สำหรับ matching; sweep เอง audit ได้ผ่าน activity log ตาม `audit trail` |
| RCV-EDGE-05 | GRN หลายบรรทัดพร้อมผลคุณภาพผสม | PO `PO-M` ที่ `po_status = sent`, สามบรรทัด: `L1` (`order_qty = 10`), `L2` (`order_qty = 5`), `L3` (`order_qty = 8`) Vendor ส่ง `L1 = 10` (รับทั้งหมด), `L2 = 5` (1 หน่วยเสียหาย → `accepted_qty = 4`), `L3 = 8` (3 หน่วยหมดอายุขณะมาถึง → `accepted_qty = 5`) | **ทั้งสามบรรทัด commit บน GRN เดียวกัน** กรอกต่อบรรทัด: `L1` `received_qty = 10, accepted_qty = 10`; `L2` `received_qty = 5, accepted_qty = 4`; `L3` `received_qty = 8, accepted_qty = 5` Inventory เพิ่ม: `+10` บน `L1`, `+4` บน `L2`, `+5` บน `L3` (รวม `+19` ไม่ใช่ `+23`) `received_qty` บรรทัด PO เลื่อนทั้งสาม: `L1 0 → 10`, `L2 0 → 5`, `L3 0 → 8`; ทุกบรรทัดผ่าน `received_qty + cancelled_qty ≥ order_qty` → `po_status = sent → completed` ตาม `PO_POST_007` ต้นน้ำ สรุป variance บน GRN แสดง `L2` reject = 1, `L3` reject = 3; Purchaser ได้รับ variance handoff หลายบรรทัดสำหรับการแก้ฝั่ง vendor บน `L2` และ `L3` (return / credit note / replacement); PO ไม่ auto-correct สำหรับ variance คุณภาพ |
| RCV-EDGE-06 | รวมหลาย PO เป็น GRN เดียวกับ vendor / สกุลเงิน / location เดียว | vendor เดียวกัน สกุลเงินเดียวกัน location ส่งเดียวกันข้าม `PO-A` (บรรทัดเดียว pending 4) และ `PO-B` (สองบรรทัด pending 6 ทั้งคู่); vendor ส่งรถบรรทุกเดียวคลุมทั้งสอง PO | **GRN เดียวสามบรรทัด span สอง PO** GRN สร้างผ่านเส้นทาง multi-PO; header สืบทอด `vendor_id`, `currency_id`, `exchange_rate` ที่ share; แถว GRN detail สามแถวแต่ละแถวพก `purchase_order_detail_id` ของตน Save รัน `GRN_VAL_009` ต่อบรรทัดกับยอด pending ของ PO แต่ละใบอย่างอิสระ Commit เลื่อน `received_qty` บนแถว `tb_purchase_order_detail` สามแถวที่ต่างกันใน transaction เดียว; `PO-A` flip `sent → completed`, `PO-B` flip `sent → completed` (หรือ `partial` ถ้าไม่ทุกบรรทัดรับเต็ม); inventory transaction เขียนหนึ่งครั้งต่อบรรทัด GRN; AP accrual หนึ่งคลุมการส่งที่รวม Map กับ TC-GRN-040002, TC-GRN-040004 |
