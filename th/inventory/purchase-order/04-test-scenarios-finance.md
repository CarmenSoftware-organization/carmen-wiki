# ใบสั่งซื้อ (Purchase Order) — Test Scenarios — Finance

> **At a Glance**
> **Persona:** Finance (Officer / AP Clerk + Finance Manager) &nbsp;·&nbsp; **Module:** [purchase-order](/th/inventory/purchase-order) &nbsp;·&nbsp; **Scenarios:** ~25
> **Categories:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** map ไปยัง `401-po.spec.ts`, `403-po-finance-ap-match.spec.ts` ใน `../carmen-inventory-frontend-e2e/`

หน้านี้ capture test scenarios ที่ Finance persona (**Finance Officer / Accounts Payable** clerk ที่รัน three-way match ประจำวันและ post AP liability บวก **Finance Manager** ที่ exercise pre-transmission financial sign-off บน POs ที่ high-value หรือ FX-sensitive) ขับเคลื่อนโดยตรงในโมดูล `purchase-order` Finance มีสองจุดสัมผัสที่ต่างกันบนวงจรชีวิต PO: **pre-transmission review** ขณะที่ PO ยังอยู่ที่ `po_status = in_progress` (Finance Manager sign off currency, exchange rate, tax codes, และ totals ก่อน `PO_POST_004` ส่ง PO ให้ vendor) และ **post-receipt three-way match** หลัง GRN posting (Finance Officer capture vendor invoice, look up matching PO และ GRN(s) และรัน match ภายใต้ `PO_POST_008` / `PO_POST_009`) Three-way match คือ activity สำคัญของ Finance — บน success, AP clear GRN accrual และ post vendor liability; บน failure, invoice hold ใน dispute และความคลาดเคลื่อน flag กลับให้ Purchaser สำหรับ resolution PO เอง **ไม่** transition โดย three-way match (`PO_POST_008` ระบุชัดเจน) — match outcome อยู่บนเร็คคอร์ด invoice และ PO เก็บสถานะ fulfilment ที่ถึง (`partial`, `completed`, หรือ `closed`) Scenarios จัดกลุ่มเป็น happy paths ที่อธิบายใน [03-user-flow-finance.md](./03-user-flow-finance.md), RBAC boundary ที่บังคับโดย `PO_AUTH_009` (Finance Officer scope — read-only ข้าม PO + match + post AP ฝั่ง AP) และสิทธิ์ pre-transmission approver ของ Finance Manager ภายใต้ `PO_AUTH_011` (stage-gated approval), กฎ validation รอบ qty / price tolerance, missing GRN, FX, closed-period, และ duplicate-invoice guards, และชุดเล็กของ edge cases รอบ tolerance boundaries, FX no-op cases, multi-invoice matching, concurrency, และ decimal precision บน AP amounts Cross-persona handoffs ที่ pivot บน Finance (`X-PO-05` pre-transmission send-back, `X-PO-07` three-way-match dispute bounce-back, `X-PO-09` AP posting close-out) อยู่ใน parent overview ไม่ใช่ที่นี่

## 2. Permission / Authorization

| # | Scenario | Expected behaviour (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| FIN-PERM-01 | Finance Officer รัน three-way match บน PO ที่ `po_status ∈ {partial, completed, closed}` | **Allow** ตาม `PO_AUTH_009` (Finance Officer read-only บน PO) และ AP-side permission เพื่อ capture + match invoices Match algorithm ภายใต้ `PO_POST_008` execute; บน success, `PO_POST_008` post AP; บน failure, `PO_POST_009` park invoice ใน dispute เร็คคอร์ด PO เองไม่ mutate โดย Finance Officer (read-only) |
| FIN-PERM-02 | Finance Officer post AP liability บน three-way-match success | **Allow** AP posting คือ AP-module action driven โดย `PO_POST_008` (cross-module rule `PO_XMOD_007`) Finance Officer ถือ AP-post right ฝั่ง AP; โมดูล PO consulted (read-only) เท่านั้นสำหรับ matching reference data — `vendor_id`, `currency_id`, line `unit_price`, GRN linkage GL entries เขียนไปยัง AP / inventory-clearing / VAT accounts ตาม tenant mapping |
| FIN-PERM-03 | Finance Officer พยายาม override failed three-way match (force-post AP แม้มี qty / price discrepancy) | **Deny — Manager-only** Override ของ failed match ต้องการ Finance Manager role (หรือ AP Manager ตาม tenant configuration) UI ของ Finance Officer ซ่อน / disable action **Override and Post** บน failed match; invoice ยังที่ state dispute `PO_POST_009` direct API call return `"Override of failed three-way match requires the Finance Manager role."` |
| FIN-PERM-04 | Finance Manager sign off high-value PO ที่ pre-transmission review stage | **Allow** ตาม `PO_AUTH_011` (stage-gated approval) เมื่อ user ปรากฏใน `tb_purchase_order.user_action.execute` ที่ `workflow_current_stage` ปัจจุบัน Signoff post `PO_POST_003` (ภายใน `in_progress`) หรือบน final stage, fire `PO_POST_004` (`in_progress → sent`) ต่ำกว่า high-value threshold, workflow อาจ bypass Finance stage ทั้งหมด |
| FIN-PERM-05 | Finance Officer พยายาม post AP โดยไม่มี three-way match สำเร็จ | **Deny** AP posting gate บน `PO_POST_008` success — AP module reject direct post เทียบกับ unmatched / failed-match invoice Server return `"Cannot post AP liability — three-way match has not succeeded on invoice <invoice_no>. Resolve qty / price discrepancy or escalate to the Finance Manager."` GL entries ไม่เขียน; invoice ยังที่ state `pending_match` หรือ `dispute` |
| FIN-PERM-06 | Finance Officer พยายาม modify PO header (vendor, currency, price, qty) | **Deny — Purchaser เท่านั้น** ตาม `PO_AUTH_002` (Purchaser scope) และ `PO_AUTH_010` (segregation of duties), Finance Officer ไม่มี edit right บนเร็คคอร์ด PO เอง — เฉพาะ read access ภายใต้ `PO_AUTH_009` หน้า detail PO ใน Finance Officer's view เป็น read-only; AP-module screens คือที่เดียวที่ Finance สามารถ act และ screens เหล่านั้นไม่สามารถ mutate PO direct API edit call return `"You are not authorised to edit this PO at the current stage."` |
| FIN-PERM-07 | Finance Manager send back PO ที่ pre-transmission review stage | **Allow** ตาม `PO_POST_005`, Finance Manager (หรือ stage approver ใด ๆ) อาจ reject PO ที่ review stage พร้อม reason text; `po_status = in_progress → draft`; `workflow_current_stage` reset; `last_action = rejected`; rejection comment persist ใน `tb_purchase_order_comment` (type `system`); ownership return ไปยัง Purchaser เพื่อแก้ไขและ resubmit |

## 3. Validation / Error

| # | Scenario | Trigger | Expected error |
| - | -------- | ------- | -------------- |
| FIN-VAL-01 | Qty discrepancy นอก tolerance — flag กลับให้ Purchaser | PO `PO-Q1` ที่ `po_status = completed` (qty 10 รับผ่าน GRN); vendor invoice `INV-Q1` สำหรับ qty 12 @ matching unit price; tenant qty tolerance = `0%` | Reject match ตาม `PO_POST_009` Server return `"Three-way match failed — invoice qty (12) exceeds GRN accepted qty (10) outside tenant tolerance (0%)."` Invoice hold ใน state `dispute`; AP posting block; `system` comment append บน PO บันทึก failure (line, dimension = qty, magnitude); deviation record เปิดบน vendor / vendor-pricelist side; Purchaser notify ผ่าน activity log เพื่อตาม credit note (over-invoicing) หรือ supplementary GRN (under-receipt) |
| FIN-VAL-02 | Price discrepancy นอก tolerance — flag กลับให้ Purchaser | PO `PO-P1` ที่ `po_status = completed` (qty 100 @ PO `unit_price = ฿50.00`); vendor invoice `INV-P1` สำหรับ qty 100 @ `฿55.00`; tenant price tolerance = `1.5%` (actual variance = `10%`) | Reject match ตาม `PO_POST_009` Server return `"Three-way match failed — invoice unit price (฿55.00) is outside the price-tolerance band (±1.5% of PO unit price ฿50.00)."` Invoice hold ใน dispute; AP posting block; `system` comment append บน PO + deviation บน vendor pricelist; Purchaser ตาม credit note (over-billing) หรือ PO amendment ภายใต้ `PO_VAL_016` (การเปลี่ยนราคา post-`sent` จำกัด) |
| FIN-VAL-03 | Missing GRN — ไม่สามารถ match โดยไม่มี | PO `PO-N1` ที่ `po_status = sent` (ยังไม่มี GRN posted); vendor invoice `INV-N1` รับ | Reject match Server return `"Three-way match cannot run — no GRN posted against PO <po_no>. Confirm receipt with the Receiver / Purchaser before re-presenting the invoice."` Invoice park ใน `pending_match` (ไม่อยู่ใน dispute); AP posting block; ถ้า vendor ส่งของแต่ GRN ยังไม่ posted, Receiver / Purchaser notify เพื่อตาม GRN — เมื่อ posted, match re-run อัตโนมัติ ถ้า vendor invoice ก่อนการส่งของ, invoice bounce กลับให้ vendor พร้อม non-receipt notice |
| FIN-VAL-04 | Invoice สำหรับ voided PO — rejected | PO `PO-V1` ที่ `po_status = voided` (voided โดย Procurement Manager ตาม `PO_AUTH_007`); vendor invoice `INV-V1` มาถึงอ้างอิง voided `po_no` | Reject match Server return `"PO <po_no> is at status voided — no invoice can be matched against this PO."` Invoice flag สำหรับ return ให้ vendor; AP posting block; activity log บันทึก attempt; เส้นทาง resolution คือ vendor correction (vendor ต้องถอน invoice) — ไม่มี action ฝั่ง PO ว่าง เพราะ `voided` เป็น terminal ตาม `PO_POST_010` |
| FIN-VAL-05 | Currency mismatch เกิน FX tolerance — rejected | PO `PO-C1` ที่ `po_status = completed` ด้วย `currency_id = USD`; vendor invoice `INV-C1` รับใน `EUR` แต่ tenant policy ไม่อนุญาต `EUR` invoicing สำหรับ vendor นี้ (single-currency contract ตาม `PO_VAL_013`) | Reject match Server return `"Invoice currency (EUR) does not match PO currency (USD); dual-currency invoicing is not permitted for this vendor per tenant policy."` Invoice hold ใน dispute; AP posting block; `system` comment append บน PO; Purchaser flag เพื่อติดต่อ vendor สำหรับ re-invoicing ใน `USD` แตกต่างจาก FIN-HP-05 happy path ที่ dual-currency อนุญาต |
| FIN-VAL-06 | Duplicate invoice match attempt — rejected | PO `PO-D1` matched + AP-posted แล้วเทียบกับ invoice `INV-D1` (state = `approved for payment`); invoice ที่สอง `INV-D2` ด้วย vendor invoice number เดียวกัน capture เทียบกับ PO เดียวกัน | Reject ที่ AP-capture stage (ก่อน match รัน) Server return `"Duplicate invoice — vendor invoice number <invoice_no> from <vendor> has already been captured and matched against PO <po_no>."` Invoice ที่สองไม่เขียน; AP posting block; activity log บันทึก attempt; Purchaser / vendor notify สำหรับ re-issue ถ้า invoice ที่สองเป็น legitimate amendment (ซึ่งในกรณีนั้น vendor ต้องใช้ invoice number ที่ distinct โดยทั่วไปด้วย suffix `-CN` หรือ `-AM`) |
| FIN-VAL-07 | AP posting พยายามเทียบกับ closed accounting period — rejected | PO `PO-CP` ที่ `po_status = completed`; vendor invoice `INV-CP` matched สะอาด; current GL period สำหรับ invoice date (`2026-04-15`) เป็น **closed** (period-close cutoff applied) | Reject ที่ AP-post stage Server return `"Cannot post AP — accounting period <yyyy-mm> is closed. Either re-date the invoice to the open period or request period re-open from the Finance Manager."` GL entries ไม่เขียน; invoice hold ที่ state `pending_post`; เส้นทาง resolution คือ Finance Manager period re-open (audit-logged) หรือ invoice re-dating กับ vendor concurrence PO ไม่เปลี่ยน |

## 4. Edge Cases

| # | Scenario | เงื่อนไข | Expected |
| - | -------- | --------- | -------- |
| FIN-EDGE-01 | Exact tolerance boundary บนทั้ง price และ qty | PO `PO-B1` ที่ `po_status = completed` (qty 100 @ `฿50.00`); vendor invoice `INV-B1` สำหรับ qty 102 @ `฿50.75`; tenant qty tolerance = `2%` (upper bound = 102); tenant price tolerance = `1.5%` (upper bound = `฿50.75`) | ยอมรับ — ทั้งสอง dimensions land บน boundary พอดี `PO_POST_008` fire: qty variance `(102 − 100) / 100 = 2%` และ price variance `(50.75 − 50.00) / 50.00 = 1.5%` ทั้งคู่เท่ากับ tolerance ไม่เกิน (inclusive boundary ตาม tenant configuration) AP post ที่ค่า invoiced; qty over-receipt reconcile เทียบกับ GRN over-receipt tolerance ที่ post โดย Receiver; price variance flow ไปยัง PPV ข้าม bound ใด (qty = 103 หรือ price = `฿50.76`) จะ trigger `PO_POST_009` ตาม FIN-VAL-01 / FIN-VAL-02 |
| FIN-EDGE-02 | FX rate ตรงกับ PO snapshot พอดี — ไม่มี FX adjustment posted | PO `PO-NX` ที่ `po_status = completed` ด้วย `currency_id = USD`, `exchange_rate = 36.50`; vendor invoice `INV-NX` รับใน `USD` (single-currency match); invoice-date `USD → THB` rate happen เป็น `36.50` พอดี (ตรงกับ PO snapshot ที่ 5 dp) | ยอมรับ ไม่มี FX entry `PO_POST_008` fire: match ผ่านบน qty + price; AP post invoice ใน `USD`; base-currency `THB` value ใช้ invoice-date rate `36.50` — ซึ่งเท่ากับ PO `exchange_rate` snapshot — ดังนั้น realised FX gain/loss entry คือ `฿0.00` และไม่เขียนไปยัง GL (no-op suppression ตาม tenant policy) แตกต่างจาก FIN-HP-05 ที่ cross-rate ต่างจาก snapshot |
