# ใบสั่งซื้อ (Purchase Order) — Test Scenarios — Receiver

> **At a Glance**
> **Persona:** Receiver (Store Keeper + Inventory Manager) &nbsp;·&nbsp; **Module:** [purchase-order](/th/inventory/purchase-order) &nbsp;·&nbsp; **Scenarios:** ~23
> **Categories:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** ยังไม่มี Receiver spec เฉพาะ; shared coverage ใน `tests/401-po.spec.ts` (TC-PO-340001..340003 GRN sync, TC-PO-060501..060504 Close) ใน `../carmen-inventory-frontend-e2e/`

หน้านี้ capture test scenarios ที่ Receiver persona (**Store Keeper** ที่ dock บวก **Inventory Manager** ที่กำกับการปิดสำหรับ location) ขับเคลื่อนโดยตรงในโมดูล `purchase-order` การ involve ของ Receiver เริ่มเมื่อ PO ถึง `po_status = sent` และจบเมื่อ PO ออกจาก receipt zone — ทั้งผ่าน `completed` (รับครบ), `partial` (open balance carry ไปข้างหน้า) หรือ `closed` (Inventory Manager early-terminate `partial` PO) GRN posting เองอยู่ในโมดูล `[good-receive-note](/th/inventory/good-receive-note)` ปลายน้ำ; หน้านี้ capture เฉพาะ **PO-side effects** — วิธีที่ GRN flip `tb_purchase_order.po_status` ผ่าน `PO_POST_006` (`sent → partial`) และ `PO_POST_007` (`* → completed`), วิธีที่ Inventory Manager ปิด `partial` PO ด้วย remainder เขียนเป็น `cancelled_qty` ผ่าน `PO_POST_011`, และวิธีที่ PO line counters (`received_qty`, `cancelled_qty`) advance เทียบกับ `order_qty` ([03-user-flow-receiver.md](./03-user-flow-receiver.md) Section 2) Scenarios จัดกลุ่มเป็น happy paths ที่อธิบายใน user flow, RBAC boundary ที่บังคับโดย `PO_AUTH_008` (Receiver / Inventory Manager scope) และ `PO_AUTH_010` (segregation of duties — ผู้ post GRN ≠ PO buyer / transmitter) เทียบกับ `PO_AUTH_001`–`PO_AUTH_007` (Purchaser / Manager scope), กฎ validation รอบ `received_qty`, `accepted_qty`, over-receipt tolerance และ voided / closed-PO state guards และชุดเล็กของ edge cases รอบ tolerance boundaries, decimal precision, concurrency และ multi-currency context Cross-persona handoffs ที่ pivot บน Receiver (`X-PO-03`, `X-PO-04`, `X-PO-06`, `X-PO-08`) อยู่ใน parent overview ไม่ใช่ที่นี่

## 1. Happy Path

Review variance summary (ไม่มี short, ไม่มี over, ไม่มี quality reject) 5. คลิก **Post** | GRN detail row เขียน; `tb_purchase_order_detail.received_qty` ขึ้น `0 → 10`; bridge `tb_purchase_order_detail_tb_purchase_request_detail.received_qty` update สัดส่วน; inventory on-hand เพิ่มโดย `accepted_qty = 10` (จัดการโดย `[good-receive-note](/th/inventory/good-receive-note)` / `[inventory](/th/inventory/inventory)`) ทุกบรรทัด active ตอนนี้พอใจ `received_qty + cancelled_qty ≥ order_qty` → `po_status = sent → completed` ตาม `PO_POST_007`; `history` append `{ po_status: 'completed', action: 'received' }`; PO กลายเป็น read-only สำหรับ receipt; ตำแหน่ง matched-but-unbilled ส่งต่อให้ Finance สำหรับ three-way match ตาม `PO_POST_008` |

## 2. Permission / Authorization

| # | Scenario | Expected behaviour (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| RCV-PERM-01 | Receiver post GRN เทียบกับ PO ที่ `po_status = sent` | **Allow** ตาม `PO_AUTH_008` (`po_status ∈ {sent, partial}`) `[good-receive-note](/th/inventory/good-receive-note)` Create flow ยอมรับ PO selection; counters ฝั่ง PO update; `po_status` transition ตาม `PO_POST_006` / `PO_POST_007` บนพื้นฐาน line-wise evaluation |
| RCV-PERM-02 | Receiver พยายาม GRN เทียบกับ PO ที่ `po_status = draft` | **Deny** ตาม `PO_AUTH_008` (status ไม่อยู่ใน `{sent, partial}`) PO ไม่ปรากฏใน vendor / PO picker ของโมดูล GRN; direct API call return `"GRN cannot be posted against PO at status draft — PO must be sent or partial."`; counters ไม่เปลี่ยน |
| RCV-PERM-03 | Receiver พยายาม modify PO header (vendor, currency, delivery date) | **Deny — Purchaser-only** ตาม `PO_AUTH_010` (segregation of duties) และ `PO_AUTH_002` (Purchaser scope) header / line edits สงวนสำหรับ buyer ที่ assigned; Receiver ไม่สามารถ edit header หรือ line price / qty / discount หน้า detail PO ใน Receiver's view เป็น read-only สำหรับ header / lines — เฉพาะปุ่ม **Receive** และ (สำหรับ Inventory Manager) **Close** ที่ interactive direct API edit call return `"You are not authorised to edit this PO at the current stage"` |
| RCV-PERM-04 | ผู้ post GRN = PO buyer / transmitter (segregation of duties) | **Deny — segregation of duties** ตาม `PO_AUTH_010` user ที่ระบุโดย `tb_purchase_order.buyer_id` หรือเป็น `last_action_by_id` บน transition `sent` ต้อง **ไม่** เป็น user คนเดียวกันที่ post GRN GRN module reject post ที่ create-time ด้วย `"You created or transmitted this PO; another user must post the GRN against it."`; ไม่มี GRN row เขียน; PO counters ไม่เปลี่ยน |
| RCV-PERM-05 | Receiver (Store Keeper) พยายามปิด `partial` PO (early termination) | **Deny เมื่อ role = Store Keeper เท่านั้น; allow เมื่อ role = Inventory Manager** ตาม `PO_AUTH_008` และ `PO_POST_011` Close (`partial → closed`) action ต้องการ Inventory Manager role (หรือ Procurement Manager override ผ่าน `PO_AUTH_007`) UI ของ Store Keeper ซ่อน / disable ปุ่ม **Close** บน PO header; direct API call return `"Close from status partial requires the Inventory Manager role."` |
| RCV-PERM-06 | Receiver พยายาม GRN เทียบกับ PO ที่ `po_status = completed` หรือ `closed` | **Deny — terminal status** ตาม `PO_POST_007` / `PO_POST_011`, `completed` และ `closed` เป็น terminal สำหรับวัตถุประสงค์ receipt; ไม่รับ GRNs เพิ่ม PO ไม่ปรากฏใน open-PO list ของโมดูล GRN; direct API call return `"PO is at terminal receipt status <status> — no further receipts can be posted."` |
| RCV-PERM-07 | Inventory Manager ปิด `partial` PO ที่พวกเขาไม่ได้ raise | **Allow** ตาม `PO_AUTH_008` Inventory Manager authority ผูกกับ delivery location ไม่ใช่กับ PO authorship; segregation of duties (`PO_AUTH_010`) ใช้กับ GRN posting ไม่ใช่กับ PO closure Close สำเร็จด้วย reason text append ไปยัง `tb_purchase_order_comment` |

## 3. Validation / Error

| # | Scenario | Trigger | Expected error |
| - | -------- | ------- | -------------- |
| RCV-VAL-01 | Over-receipt โดยไม่มี tolerance — `received_qty > pending balance` | PO line `L1`: `order_qty = 10`, `received_qty = 0`, `cancelled_qty = 0`; tenant over-receipt tolerance disable หรือตั้งเป็น `0%`; Receiver ใส่ `received_qty = 12` | Reject — `PO_XMOD_004` Server return `"Received quantity exceeds the pending balance and tenant over-receipt tolerance is not configured."`; GRN line cap ที่ pending qty (10) หรือ post block ทั้งหมด; PO counters ไม่เปลี่ยน |
| RCV-VAL-02 | Zero `received_qty` บน GRN line | Receiver ใส่ `received_qty = 0` บนบรรทัดและคลิก **Post** | Reject — `PO_XMOD_003` / GRN-side validation Server return `"Received quantity must be greater than zero on every GRN line; remove the line if nothing was received against it."`; GRN ไม่สามารถ post ด้วย zero-qty line; PO counters ไม่เปลี่ยน |
| RCV-VAL-03 | GRN เทียบกับ `voided` PO | PO `PO-V` ที่ `po_status = voided` (voided โดย Procurement Manager ตาม `PO_AUTH_007`); Receiver พยายามเปิด GRN flow จากโมดูล GRN โดยตรง | Reject — `PO_AUTH_007` (voided เป็น terminal) Server return `"PO is at status voided — no GRN can be posted."`; PO ไม่ปรากฏใน open-PO list; direct API call block ที่ GRN create step |
| RCV-VAL-04 | `accepted_qty > received_qty` (quality เกิน physical receipt) | Receiver ใส่ `received_qty = 8`, `accepted_qty = 10` บนบรรทัดและคลิก **Post** | Reject — GRN-side validation Server return `"Accepted quantity cannot exceed received quantity on any GRN line."`; inequality `accepted_qty ≤ received_qty` บังคับใช้ที่ save และ post; GRN ไม่สามารถ post; PO counters ไม่เปลี่ยน |
| RCV-VAL-05 | Partial GRN เมื่อ remaining qty เป็น unreceivable (negative pending) | PO line `L1`: `order_qty = 10`, `received_qty = 8`, `cancelled_qty = 2` (ดังนั้น pending balance = 0); Receiver พยายาม post GRN ด้วย `received_qty = 1` บน `L1` | Reject — `PO_XMOD_003` Server return `"PO line has no remaining open balance — received_qty (8) + cancelled_qty (2) = order_qty (10)."`; GRN line ไม่สามารถ post บน `L1`; ถ้าบรรทัดอื่นบน PO ยังเปิด, ตัวเหล่านั้นอาจ post อิสระ |
| RCV-VAL-06 | GRN เทียบกับ PO ที่ vendor flip เป็น `is_active = false` หลัง `sent` | PO `PO-W` ที่ `po_status = sent`; `tb_vendor.is_active = false` บน vendor ของ PO (vendor blacklisted post-transmission) | Reject — vendor-state guard Server return `"Vendor is no longer active and cannot deliver further goods. Escalate to the Procurement Manager."`; GRN create step block; เส้นทาง resolution คือ Procurement Manager void (`PO_AUTH_007`) หรือ vendor master fix-up |

## 4. Edge Cases

| # | Scenario | เงื่อนไข | Expected |
| - | -------- | --------- | -------- |
| RCV-EDGE-01 | Exact tolerance boundary บน over-receipt | PO line `L1`: `order_qty = 10`, `received_qty = 0`, `cancelled_qty = 0`; tenant over-receipt tolerance = `5%`; Receiver ใส่ `received_qty = 10.5` (พอดี upper bound) | ยอมรับ — `PO_XMOD_004` `received_qty = 10.5` อยู่ภายใน tolerance (`10 × 1.05 = 10.5`); GRN post; `tb_purchase_order_detail.received_qty = 10.5`; line evaluation: `received_qty (10.5) ≥ order_qty − cancelled_qty (10)` → รับครบ; `po_status = sent → completed` ตาม `PO_POST_007` การตั้ง `received_qty = 10.51` จะ breach bound และ reject ตาม RCV-VAL-01 |
| RCV-EDGE-02 | Decimal precision บน `accepted_qty` | Receiver ใส่ `received_qty = 5.123`, `accepted_qty = 4.998` บนบรรทัดที่ order UoM อนุญาต 3-dp precision; conversion factor `1.234567`, `base_qty` คำนวณใน storage UoM | ยอมรับที่ storage; `received_qty` และ `accepted_qty` เก็บที่ precision UoM ของบรรทัด (3 dp); `base_qty` คำนวณใหม่บนบรรทัดเป็น `Round(received_qty × conv_factor, 3)` สำหรับ inventory roll-up; quality variance `received_qty − accepted_qty = 0.125` track บน GRN; PO `received_qty` aggregate โดยไม่มี precision loss; Decimals ทั้งหมดเก็บพอดีใน `Decimal(20, 5)` สำหรับเงินและ `Decimal(15, 5)` สำหรับอัตรา |
| RCV-EDGE-03 | Concurrent GRN posts บน PO เดียวกันจากสอง Receiver sessions | Receiver A และ B ทั้งคู่เปิด `PO-X` (`po_status = sent`, `received_qty = 0`, `order_qty = 10`); A post `received_qty = 6` ที่ `T`; B post `received_qty = 4` ที่ `T + 500ms` เทียบกับบรรทัดเดียวกัน ทั้งคู่ด้วย `doc_version = 3` snapshots | Post แรกชนะ atomically: GRN ของ A post, `tb_purchase_order_detail.received_qty = 0 → 6`, `po_status = sent → partial`, `doc_version → 4` Post ของ B reconcile เทียบกับ state ใหม่ — ทั้ง GRN module re-read `pending_qty = 4` และ post `received_qty = 4` ของ B สะอาด (advance `received_qty → 10`, `po_status → completed`) หรือ optimistic-concurrency guard reject B ด้วย `"This PO was modified by another user. Please refresh and re-apply your changes."` และ B retry ไม่มี double-write, ไม่มี lost-update, ไม่มี `received_qty > order_qty` overflow นอก over-receipt tolerance |
| RCV-EDGE-04 | Multi-currency context — PO base currency สำคัญสำหรับ valuation | PO `PO-F` ด้วย `currency_id = USD`, `exchange_rate = 36.50` (เทียบกับ base `THB`); line `L1`: `order_qty = 10`, `price = $50.00`; Receiver post GRN `received_qty = 10`, `accepted_qty = 10` | GRN post ที่สกุลเงิน transaction ของ PO (`USD`); inventory valuation dual-posted — transaction value `$500.00` และ base value `Round($500.00 × 36.50, 2) = ฿18,250.00` ตาม `PO_CALC_008`–`PO_CALC_011`; `exchange_rate` snapshot ของ PO ณ time ของ PO submission คืออัตราที่ใช้สำหรับ inventory accrual ที่ driven โดย GRN ไม่ใช่อัตรา spot ปัจจุบัน; `po_status = sent → completed` ตาม `PO_POST_007`; three-way match ปลายน้ำ (Finance) reconcile invoice สกุลเงิน transaction เทียบกับ PO / GRN สกุลเงิน transaction |
