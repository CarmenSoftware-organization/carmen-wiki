# ใบสั่งซื้อ (Purchase Order) — Test Scenarios — Purchaser

> **At a Glance**
> **Persona:** Purchaser (Procurement Officer) &nbsp;·&nbsp; **Module:** [[purchase-order]] &nbsp;·&nbsp; **Scenarios:** ~34
> **Categories:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** map ไปยัง `tests/402-po-purchaser-journey.spec.ts` และ `tests/401-po.spec.ts` ใน `../carmen-inventory-frontend-e2e/`

หน้านี้ capture test scenarios ที่ Purchaser persona (เรียกอีกชื่อว่า **Procurement Officer**) ขับเคลื่อนโดยตรงในโมดูล `purchase-order` Purchaser เป็นเจ้าของ PO ตั้งแต่การสร้างไปจนถึงการส่งให้ vendor — ช่วงจาก `po_status = draft` ถึง `sent` ([03-user-flow-purchaser.md](./03-user-flow-purchaser.md) Section 2) สองเส้นทางการสร้างมาบรรจบที่ flow เดียวกัน: **manual PO** (`po_type = manual`) ที่ยกขึ้นโดย procurement โดยตรง และ **PR-sourced PO** (`po_type = purchase_request`) ที่ materialise ผ่าน Convert-to-PO จากโมดูล [[purchase-request]] ต้นน้ำผ่านตาราง bridge `tb_purchase_order_detail_tb_purchase_request_detail` ([01-data-model.md](./01-data-model.md) Section 2.5) Scenarios จัดกลุ่มเป็น happy paths ที่อธิบายใน user flow, RBAC boundary ที่บังคับโดย `PO_AUTH_001`–`PO_AUTH_003` และ `PO_AUTH_006` (ขอบเขต edit / submit / transmit ของ Purchaser) เทียบกับ `PO_AUTH_004`, `PO_AUTH_005`, `PO_AUTH_007` (escalation ไปยัง Procurement Manager), กฎ validation ใน [02-business-rules.md](./02-business-rules.md) Section 2 (`PO_VAL_001`–`PO_VAL_016`) ที่ Purchaser สามารถ trigger จาก create / edit / submit time และชุดเล็กของ edge cases รอบ ๆ decimal precision, currency, concurrency, และการเปลี่ยนแปลง state ของ vendor-master Cross-persona handoffs ที่ pivot บน Purchaser (`X-PO-01`, `X-PO-02`, `X-PO-05`, `X-PO-10`) อยู่ใน parent overview ไม่ใช่ที่นี่

## 2. Permission / Authorization

| # | Scenario | Expected behaviour (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| PUR-PERM-01 | Purchaser เปิด PO ของตัวเอง (current user ตรงกับ `tb_purchase_order.buyer_id`, หรืออยู่ใน `user_action.execute[]` ที่ `workflow_current_stage` ปัจจุบัน) | **Allow** read ข้ามทุก statuses และ actions create / edit / submit / transmit ตาม `PO_AUTH_001`–`PO_AUTH_003` และ `PO_AUTH_006` Detail page load พร้อม header + items + Item Details panel (TC-PO-060301, TC-PO-060302) |
| PUR-PERM-02 | Purchaser ดู PO ของ Purchaser อื่น (RBAC-scoped read) | **Allow** read เมื่อ tenant's RBAC grant `purchase-order:read` ให้ทุก users `enum_stage_role = purchase` (default); **deny** write actions (Edit / Submit / Send-to-Vendor / Close) เพราะ `PO_AUTH_002` ต้องการ user เป็น buyer ที่ assigned หรือถือ `workflow_current_stage` ปัจจุบัน Direct API call เพื่อ edit return `"You are not authorised to edit this PO at the current stage"` |
| PUR-PERM-03 | Purchaser edit `draft` PO ที่พวกเขาเป็นเจ้าของ | **Allow** ตาม `PO_AUTH_002` (`po_status ∈ {draft, in_progress}` และ user เป็น buyer ที่ assigned) ฟิลด์ header และบรรทัดทั้งหมด editable; ปุ่ม **Edit / Delete / Submit** ปรากฏบน `draft` (TC-PO-060303); การ modify qty / เพิ่มบรรทัด / cancel edit / submit ทั้งหมดสำเร็จ (TC-PO-060401..TC-PO-060405) |
| PUR-PERM-04 | Purchaser edit `sent` PO ที่พวกเขาเป็นเจ้าของ | **Allow** พร้อม field restriction ตาม `PO_VAL_016` — เฉพาะ `cancelled_qty` และ note ต่อบรรทัดที่ mutable post-`sent` Vendor / currency / line price / line qty edits ถูก reject ด้วย `"PO can no longer be amended at status sent. | PUR-PERM-05 | Purchaser พยายาม approve PO ของตัวเอง (`in_progress → sent` self-approval ที่ total ≤ tenant high-value threshold) | **Allow เฉพาะหาก workflow definition assign Purchaser ไปยัง approval stage** (ตาม `PO_AUTH_004` — "below the threshold, Procurement Officer can self-approve to `sent` if the workflow allows") **Deny** เมื่อ workflow route `in_progress` ไปยัง Procurement Manager โดยไม่คำนึงถึง amount หรือเมื่อ segregation of duties (`PO_AUTH_010`) ตั้งค่าให้ force approver ที่ต่าง Direct call เพื่อ advance stage จาก user ที่ไม่อยู่ใน list ถูก reject โดย `PO_AUTH_011` ด้วย `"You are not in the authorised approver set for this stage"` |
| PUR-PERM-06 | Purchaser void `draft` PO ของตัวเองผ่าน soft-delete (abandon path) | **Deny direct hard-delete; allow escalation** `PO_AUTH_005` สงวน soft-delete-in-draft action ให้ Procurement Manager เส้นทางของ Purchaser คือ abandon-as-draft (ทิ้ง PO ไว้ใน `draft` ไม่กำหนดเวลา) หรือ escalate ไปยัง Procurement Manager สำหรับ soft-delete Direct API call เพื่อ delete จาก Purchaser role ถูก reject ด้วย `"Soft-delete from draft is restricted to the Procurement Manager"` UI เปิดเผยปุ่ม `Delete` บน draft (TC-PO-060303) แต่ API บังคับ role check |

Void or close instead."`; เฉพาะ `cancelled_qty` และ note ต่อบรรทัดที่ mutable post-`sent` |

## 4. Edge Cases

| # | Scenario | เงื่อนไข | Expected |
| - | -------- | --------- | -------- |
| PUR-EDGE-01 | Zero-quantity submit-time guard รวมกับ line save-time guard | Purchaser พยายาม save บรรทัดด้วย `order_qty = 0` (caught ที่ save โดย `PO_VAL_008`); แล้วพยายาม bypass UI โดย submit PO ด้วยบรรทัด edited เป็น `0` post-save ผ่าน direct API call | ทั้ง save-line และ submit-time runs ของ `PO_VAL_008` reject บรรทัดด้วย `"Order quantity must be greater than zero and a unit of measure is required."`; submit abort; บรรทัดยังที่ค่า valid ก่อน; ไม่มี `tb_purchase_order_detail` row mutate |
| PUR-EDGE-02 | ความแม่นยำ decimal สูงสุดบน line `price` | Line `price = 99999999999.99999` (15 integer + 5 fraction digits, upper bound ของ `Decimal(20, 5)`) ด้วย `order_qty = 1.99999`; tax_rate `7%`, ไม่มี discount | ยอมรับที่ storage; `sub_total_price = Round(99999999999.99999 × 1.99999, 2)` คำนวณตาม `PO_CALC_001`; การปัดเศษใช้ half-up ผ่าน `PO_CALC_012`; `base_qty = Round(1.99999 × conv_factor, 3)` ตาม `PO_CALC_006`; header `total_qty` aggregate ใน base UoM ที่ 3 dp ตาม `PO_CALC_011`; Decimals ทั้งหมดเก็บพอดีใน `Decimal(20, 5)` สำหรับเงินและ `Decimal(15, 5)` สำหรับอัตรา |
| PUR-EDGE-03 | Back-dated `delivery_date` เท่ากับ `order_date` (lower bound ของ `PO_VAL_006`) | ตั้ง `order_date = 2026-05-15`, `delivery_date = 2026-05-15`; คลิก Save / Submit | ยอมรับ — `PO_VAL_006` บังคับ `delivery_date >= order_date` (inclusive) Same-day case ผ่าน; PO save และ submit ปกติ การตั้ง `delivery_date = order_date − 1d` จะ fail (covered โดย PUR-VAL-04) |
| PUR-EDGE-04 | การ reject multi-currency consolidation ใน From-PR wizard | ติ๊ก PRs ที่อนุมัติแล้วสองใบใน wizard: `PR-A` ด้วย `currency_id = THB` และ `PR-B` ด้วย `currency_id = USD`, ทั้งคู่สำหรับ `vendor_id = V1` | Wizard auto-group ด้วย `(vendor_id, currency_id)` → **สอง** กลุ่ม draft-PO (หนึ่งต่อ currency) ไม่ใช่หนึ่ง แต่ละ PO ที่ได้พอใจ `PO_VAL_013` direct API call เพื่อ force ทั้งสองบรรทัดเข้า PO เดียวถูก reject ด้วย `"All lines on a PO must share the header vendor and currency. Split into separate POs by vendor+currency."` (`PO_VAL_013`) |
| PUR-EDGE-05 | Concurrent edits บน draft PO เดียวกันจากสอง Purchaser sessions | Purchaser A และ Purchaser B ทั้งคู่ load draft PO `PO-X` ที่ `doc_version = 3`; A save qty change ที่ `T` (`doc_version → 4`); B save price change ที่ `T + 1s` ใช้ stale `doc_version = 3` | Save แรกชนะ: qty change ของ A persist, `doc_version = 4` save ของ B ถูก reject โดย optimistic-concurrency guard ด้วย `"This PO was modified by another user. Please refresh and re-apply your changes."`; client ของ B prompt refresh ไม่มี double-write หรือ lost-update เกิด |
| PUR-EDGE-06 | Transmit ขณะที่ vendor email หาย / vendor flip inactive | PO ที่ `po_status = sent`; `tb_vendor.email IS NULL` สำหรับ vendor บน PO หรือ `tb_vendor.is_active = false` หลังจาก PO ถูกอนุมัติ | **Send to Vendor** (TC-PO-060502) บน vendor ที่ email หายถูก reject ด้วย `"Vendor email is not configured — set up the transmission channel before sending."`; บน inactive / blacklisted vendor ถูก reject ด้วย `"Vendor is no longer active and cannot receive new POs."` PO ยังคงเป็น `sent` (status transition fire แล้วที่ final approval ตาม `PO_POST_004`); `email` / `approval_date` columns ไม่ update; audit comment append Purchaser ต้อง escalate ไปยัง Procurement Manager สำหรับ void (`PO_AUTH_007`) หรือ update vendor master และ retry |
| PUR-EDGE-07 | Submit PO ทันทีหลัง soft-delete บน `po_no` เดียวกัน (uniqueness re-use) | PO ก่อนที่ `po_no = PO-2026-000123` ถูก soft-deleted (`deleted_at IS NOT NULL`) ตาม `PO_POST_012`; Purchaser draft PO ใหม่และ numbering service issue `po_no` เดียวกัน | ยอมรับ — unique index `@@unique([po_no, deleted_at])` อนุญาต reuse เพราะ `deleted_at` ต่างกัน PO ใหม่ save และ submit ปกติ; row ที่ soft-deleted ก่อนยังอยู่ในฐานข้อมูลสำหรับ audit Hard-delete-in-draft สงวนสำหรับ Procurement Manager (`PO_AUTH_005`) ดังนั้น scenario นี้มีต้นทางจาก Manager-driven cleanup |
