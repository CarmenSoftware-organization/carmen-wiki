# ใบขอซื้อ (Purchase Request) — Test Scenarios — Purchaser

> **At a Glance**
> **Persona:** Purchaser (Procurement Officer — สะพาน PR-to-PO) &nbsp;·&nbsp; **โมดูล:** [purchase-request](/th/inventory/purchase-request) &nbsp;·&nbsp; **Scenario:** ~33
> **หมวด:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** map ไปยัง `tests/304-pr-purchaser-journey.spec.ts`, `tests/310-pr-template.spec.ts` และ `tests/301-pr.spec.ts` (fixture purchaseTest) ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ test scenario ที่ persona Purchaser ขับโดยตรงในโมดูล `purchase-request` Purchaser (ตำแหน่งอื่นคือ **Procurement Officer**) เป็น persona สะพานระหว่างฝั่ง PR ต้นน้ำและฝั่ง PO ปลายน้ำของห่วงโซ่ procure-to-pay; พวกเขา **ไม่** อนุมัติเนื้อหา PR ตอนที่ PR ถึงคิวของพวกเขา มันได้ผ่าน chain อนุมัติทั้งหมดแล้วและ `pr_status = approved` (`PR_POST_005`) Scenario ถูก group เป็น happy path ที่อธิบายใน [03-user-flow-purchaser.md](./03-user-flow-purchaser.md) Section 2 (validate vendor allocation, รัน Allocate Vendor, lookup pricelist deviation, consolidate ตาม vendor + currency, แปลงเป็น PO — เต็มและบางส่วน), ขอบเขต RBAC ที่บังคับโดย `PR_AUTH_008` (เฉพาะ `enum_stage_role = purchase` ที่แปลงได้), กฎ cross-module / PO conversion ใน [02-business-rules.md](./02-business-rules.md) Section 6 (ตาราง bridge `tb_purchase_order_detail_tb_purchase_request_detail`, consolidation ตาม `(vendor_id, currency_id)`, partial conversion, snapshot FX rate ตอนแปลง) และชุดเล็กของ edge ความสมบูรณ์ของ bridge และ concurrency Handoff ข้าม persona ที่ pivot off Purchaser (`X-PR-01`, `X-PR-04`, `X-PR-07`) อยู่ในภาพรวมหลัก ไม่ใช่ที่นี่

## 3. Validation / Error

| # | Scenario | Trigger | Error ที่คาด |
| - | -------- | ------- | -------------- |
| PUR-VAL-01 | Consolidate สอง PR ที่ share vendor แต่ต่าง currency | เลือกบรรทัด A (`vendor_id = V1`, `currency_id = THB`) และบรรทัด B (`vendor_id = V1`, `currency_id = USD`); พยายาม merge เข้า group draft PO เดียว | Reject — workbench grouping key คือ `(vendor_id, currency_id)` ตาม Section 6 ของ [02-business-rules.md](./02-business-rules.md) สองบรรทัด materialize เป็น **สอง** draft PO แยก (หนึ่งต่อ currency) บังคับเข้า group เดียวผ่าน API call ถูก reject ด้วย `"Cannot consolidate lines with mismatched currency on a single PO"` |
| PUR-VAL-02 | Consolidate สอง PR ที่ share currency แต่ต่าง vendor | เลือกบรรทัด A (`vendor_id = V1`, `currency_id = THB`) และบรรทัด B (`vendor_id = V2`, `currency_id = THB`); พยายาม merge เข้า group draft PO เดียว | Reject — grouping `(vendor_id, currency_id)` ต้องการ `vendor_id` ตรง บรรทัด materialize เป็นสอง draft PO (หนึ่งต่อ vendor) API call เพื่อ merge reject ด้วย `"Cannot consolidate lines with mismatched vendor on a single PO"` |
| PUR-VAL-03 | จำนวนแปลงบางส่วนเกิน `approved_base_qty` | บรรทัด `L1` มี `approved_base_qty = 100`, ไม่มีแถว bridge ก่อน; Purchaser ตั้ง convert-qty เป็น `120` และรัน Convert to PO | Reject — จำนวนแปลงบางส่วนต่อบรรทัดต้อง satisfy `0 < convert_qty ≤ approved_base_qty − Σ(จำนวน bridge-linked ก่อน)` Server reject ด้วย `"Convert quantity exceeds the open quantity on this PR line"`; guard ความสมบูรณ์ bridge ปฏิเสธการเขียนแถวที่ผลรวมจะ over-bridge บรรทัดต้นทาง |
| PUR-VAL-04 | จำนวนแปลงบางส่วนเกินจำนวนเปิดที่เหลือหลังการแปลงบางส่วนก่อน | บรรทัด `L1` `approved_base_qty = 100`; แถว bridge ก่อนบันทึก `converted_qty = 60` แล้ว; Purchaser ตั้ง convert-qty เป็น `50` (60 + 50 = 110 > 100) | Reject — กฎเดียวกับ PUR-VAL-03; server reject ด้วย `"Convert quantity exceeds the open quantity on this PR line"` และแสดงค่าจำนวนเปิด (40) ใน toast error |
| PUR-VAL-05 | Pricelist deviation เกิน tolerance trigger warning / บังคับการตัดสินใจ | บรรทัด snapshot `pricelist_price = 100.00000 ฿`; แถว pricelist active ปัจจุบันคือ `108.00000 ฿` (+8 %, นอก tolerance `±5 %` ที่ตั้ง) | Panel deviation แสดง warning สีแดง Purchaser ต้องเลือกหนึ่งในสามทาง explicit (ตาม [03-user-flow-purchaser.md](./03-user-flow-purchaser.md) Section 3): (a) **Accept snapshot** — ยืนยันและดำเนินต่อที่ `100.00000 ฿`; (b) **Refresh to current** — pull `108.00000 ฿` ลงบนบรรทัด PO; (c) **Raise concern / send back** — bounce PR ไปยัง Requestor Convert-to-PO ถูก **block** จนกว่าจะเลือกทาง API call ตรงโดยไม่มีการตัดสินใจ deviation explicit ถูก reject ด้วย `"Pricelist deviation exceeds tolerance — explicit accept/refresh/send-back required before conversion"` |
| PUR-VAL-06 | ขาดการจัดสรร vendor บล็อกการแปลง | บรรทัด `L1` มี `vendor_id IS NULL` (Requestor ไม่ allocate, Purchaser ยังไม่รัน Allocate Vendor); Purchaser tick `L1` และรัน Convert to PO | Reject — target bridge ต้องการ vendor Server reject ด้วย `"Cannot convert: line has no vendor allocation. Run Allocate Vendor first."`; workbench flag บรรทัดสีแดงและตัดจาก group จนกว่าจะตั้งการจัดสรร vendor |
| PUR-VAL-07 | PR ที่แปลงแล้ว (`pr_status = completed`) ไม่สามารถ reopen สำหรับการแปลงเพิ่ม | PR ต้นทางถูก bridge เต็มในรอบก่อน (`PR_POST_007` พลิกเป็น `completed`); Purchaser เปิดจาก filter **completed** และคลิก Convert to PO ผ่าน UI stale / API call ตรง | Reject — `completed` คือ terminal สำหรับการแปลง Server reject ด้วย `"This PR has already been fully converted and cannot be reopened"`; workbench filter PR `completed` ออกจาก pool การเลือก |
| PUR-VAL-08 | PR ถูก void โดย Finance / sys-admin ระหว่าง session ของ Purchaser | Purchaser load PR `approved`; ใน background `PR_AUTH_007` พลิก `pr_status` เป็น `voided`; Purchaser คลิก Convert to PO | Reject — guard state-machine: `voided` คือ terminal Endpoint conversion reject ด้วย `"This PR is voided and cannot be converted"` Soft budget commitment ของ PR ถูกปล่อยโดย `PR_POST_006` |
| PUR-VAL-09 | Master vendor flip inactive ระหว่าง PR submit และการแปลง | vendor `V1` ที่ snapshot บนบรรทัด active ตอน PR submit; วันนี้ `tb_vendor.is_active = false` (หรือ flag blacklist set); Purchaser คลิก Convert to PO | Reject — lookup vendor-master live invalidate การจัดสรร Server reject ด้วย `"Vendor V1 is no longer active and cannot be used for new POs. Run Allocate Vendor to pick an alternative."`; Purchaser ต้อง reassign vendor ก่อนลองใหม่ |
| PUR-VAL-10 | Bounce-back ไป Requestor โดยไม่ให้เหตุผล | เปิด dialog Send-Back-to-Requestor, ทิ้ง textarea เหตุผลว่าง, คลิก Confirm | Reject — เหตุผล mandatory ตาม `PR_POST_008` (system comment immutable ต้องการ payload) และ contract dialog UI disable ปุ่ม Confirm เมื่อเหตุผลว่าง; API call ตรงถูก reject ด้วย `"A reason is required to send back this PR"` |

## 4. Edge Case

| # | Scenario | เงื่อนไข | คาดหวัง |
| - | -------- | --------- | -------- |
| PUR-EDGE-01 | พยายามแปลงด้วยศูนย์บรรทัด (ไม่มีบรรทัดที่เลือก) | Purchaser เปิด workbench Convert-to-PO, tick ไม่มีบรรทัด, และคลิก Convert to PO | Reject — endpoint conversion ต้องการอย่างน้อยหนึ่งแถว bridge Server reject ด้วย `"Select at least one line to convert"`; UI ก็ disable ปุ่ม Convert-to-PO เมื่อไม่มีบรรทัด tick ไม่มีการสร้าง `tb_purchase_order` |
| PUR-EDGE-02 | ทุกบรรทัดถูกเลือกแต่แต่ละบรรทัดมี convert-qty = 0 | Purchaser tick ทุกบรรทัดแต่ตั้ง `convert_qty = 0` บนทุกแถว, แล้วคลิก Convert to PO | Reject — `convert_qty > 0` ต่อบรรทัดเป็นที่ต้องการ (เลียนแบบขอบเขตล่างของ PUR-VAL-03) Server reject ด้วย `"Convert quantity must be positive"`; UI flag ทุกแถว qty ศูนย์สีแดงและ disable Convert to PO จนกว่าอย่างน้อยหนึ่งแถวมี convert-qty บวก |
| PUR-EDGE-03 | Snapshot FX rate เท่ากับอัตรา PR พอดี (ไม่มี divergence) | `exchange_rate` PR = `35.50000`; อัตรา FX ขององค์กรวันนี้ก็ `35.50000`; บรรทัด unbridged หนึ่ง | ยอมรับ — PO snapshot `exchange_rate = 35.50000` (เดียวกับ PR) แถว bridge เขียน; ยอดสกุลฐานฝั่ง PR และฝั่ง PO match กันพอดี กฎ snapshot ไม่เปลี่ยนไม่ว่าอัตราเคลื่อน (PUR-HP-08) หรือคงที่ (case นี้) |
| PUR-EDGE-04 | ความสมบูรณ์ของตาราง bridge เมื่อการแปลงบางส่วนจบบรรทัดเปิดสุดท้ายเต็ม | PR ต้นทางมีสามบรรทัด (`L1`, `L2`, `L3`); `L1` และ `L2` ถูก bridge เต็มในรอบก่อน (`pr_status` ยังคง `approved` เพราะ `L3` ยังมี qty เปิด `40`); Purchaser ตอนนี้ bridge `L3` สำหรับ `40` ที่เหลือ | แถว bridge เขียน; Σ qty bridge-linked สำหรับ `L3` ตอนนี้เท่ากับ `L3.approved_base_qty`; ทั้งสามบรรทัดตอนนี้ถูก bridge เต็ม `PR_POST_007` fire และพลิก `pr_status` จาก `approved` เป็น `completed`; soft commitment สำหรับ `L3` แข็งตัวเข้า PO ใหม่; PR ตกจากคิว Approved PRs |
| PUR-EDGE-05 | พยายามแปลงพร้อมกันบน PR line เดียวกัน | Purchaser A และ Purchaser B ทั้งคู่ load PR `approved` `L1` (`approved_base_qty = 100`, ไม่มี bridge ก่อน); A tick `L1` ที่ `60` และคลิก Convert to PO ที่ `T`; B tick `L1` ที่ `60` และคลิก Convert to PO ที่ `T+1s` (หลังแถว bridge ของ A commit แต่ก่อน client ของ B refresh) | การแปลงแรกชนะ: แถว bridge เขียนสำหรับ A ด้วย `converted_qty = 60`, qty เปิดของบรรทัดต้นทางตอนนี้ `40` การแปลงที่สอง (B) ถูก reject ที่ server — guard ความสมบูรณ์ bridge re-check `Σ(prior bridge-linked qty) + new converted_qty ≤ approved_base_qty` กับสถานะปัจจุบันและ reject ด้วย `"Convert quantity exceeds the open quantity on this PR line"` (เส้นทางเดียวกับ PUR-VAL-04) UI ของ B prompt refresh ไม่มีการสร้าง double-bridge |
| PUR-EDGE-06 | ความแม่นยำสูงสุดของ convert qty | บรรทัด `L1` มี `approved_base_qty = 1.99999 PCS` (Decimal(15, 5)); Purchaser ตั้ง `convert_qty = 1.99998` | ยอมรับ — validation `convert_qty` ผ่าน (`> 0` และ `≤ open_qty` ที่ความแม่นยำ 5-dp เต็ม); แถว bridge เขียนด้วย `converted_qty = 1.99998`; บรรทัดเหลือ `0.00001 PCS` เปิด; PR ยังคงเป็น `approved` จนกว่า `0.00001` ที่เหลือจะถูก bridge หรือ PR ถูกยกเลิก explicit |
