# ใบขอซื้อ (Purchase Request) — Test Scenarios — Procurement Manager

> **At a Glance**
> **Persona:** Procurement Manager (อนุมัติมูลค่าสูง + ตั้งค่า Allocate Vendor) &nbsp;·&nbsp; **โมดูล:** [purchase-request](/th/inventory/purchase-request) &nbsp;·&nbsp; **Scenario:** ~29
> **หมวด:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** ยังไม่มี spec Procurement Manager เฉพาะ; เส้นทาง escalation exercise ผ่าน block `gmTest` ใน `tests/301-pr.spec.ts` ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ test scenario ที่ persona Procurement Manager ขับโดยตรงในโมดูล `purchase-request` Procurement Manager เป็นเจ้าของ **สอง surface** ที่อธิบายใน [03-user-flow-procurement-manager.md](./03-user-flow-procurement-manager.md): surface **transactional** — stage อนุมัติ escalated สุดท้ายที่ fire เมื่อ `base_total_amount` ข้าม threshold มูลค่าสูงที่ตั้งตาม `PR_AUTH_005` และใช้ UI review-and-decide เดียวกันกับ chain Approver พื้นฐานด้วย scope กว้างกว่า (รวม override send-back ของ Approver ก่อนหน้า) — และ surface **configurational** — ชุดกฎ Allocate Vendor (น้ำหนัก vendor ranking, override priority ต่อ vendor และ Stuck PR Oversight bulk action) ที่ขับ flow การเลือก vendor ปลายน้ำของทุก Purchaser Scenario ถูก group เป็น happy path จาก [03-user-flow-procurement-manager.md](./03-user-flow-procurement-manager.md) Section 2 (escalated มูลค่าสูง approve / reject / send-back, ปรับน้ำหนัก ranking, ปรับ priority Allocate Vendor, bulk-action บน PR ที่ค้าง, override กฎ delegation), ขอบเขต RBAC จาก `PR_AUTH_002`–`PR_AUTH_008` ใน [02-business-rules.md](./02-business-rules.md) Section 4, กฎ validation ที่ fire บนแต่ละ surface (`PR_AUTH_005` semantic threshold, `PR_VAL_013` บน `approved_qty`, snapshot-preservation ตาม `PR_CALC_006`) และชุดเล็กของ boundary / concurrency case ที่ unique กับการมีทั้งสิทธิ์ transactional และ configurational Handoff ข้าม persona ที่ pivot off Procurement Manager (`X-PR-05` threshold escalation, `X-PR-10` returned-PR round trip เมื่อ PR escalated ถูกส่งกลับเป็น draft) อยู่ในภาพรวมหลัก ไม่ใช่ที่นี่

## 2. Permission / Authorization

| # | Scenario | พฤติกรรมที่คาด (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| PM-PERM-01 | Procurement Manager เปิด PR escalated (ผู้ใช้ปัจจุบันอยู่ใน `user_action.execute[]` สำหรับ stage escalated) | **Allow** read และ action bar เต็ม (Approve / Send Back / Reject / Split-Reject) `PR_AUTH_002` ผ่านสำหรับ stage escalated; stage escalated เองเป็น stage role `approve` โดย routing `PR_AUTH_005` ดังนั้น `PR_AUTH_003` และ `PR_AUTH_004` ให้สิทธิ์ระดับบรรทัดและระดับ header เดียวกันกับ chain Approver พื้นฐาน |
| PM-PERM-02 | Procurement Manager เปิด PR ที่ **ไม่** escalate ไปหา (PR ปัจจุบันอยู่ที่ stage Budget Controller ซึ่งเป็นก่อนหน้าใน chain) | **Deny action, read-only.** `PR_AUTH_002` คำนวณ `user_action.execute[]` ใหม่สำหรับ stage ปัจจุบันบนทุกการ transition; Procurement Manager ไม่อยู่ใน list ผู้ทำ Stage 2 Visibility list อาจยังให้แถวปรากฏใต้ **All Documents** ขึ้นกับ grant แต่ปุ่ม Approve / Send Back / Reject ถูก disable พร้อมคำอธิบาย inline; API call ตรงถูก reject ที่ server ด้วย `"You are not authorised to act on this PR at the current stage"` |
| PM-PERM-03 | Procurement Manager แก้กฎ Allocate Vendor (surface configurational) | **Allow.** การดูแลกฎจำกัดเฉพาะ `enum_stage_role = purchase` ตาม `PR_AUTH_008`; Procurement Manager ถือ role นั้นบน surface configurational Configuration audit trail บันทึกทุกการแก้พร้อม `user_id` ของ Procurement Manager |
| PM-PERM-04 | Procurement Manager พยายามแก้กฎ Allocate Vendor ขณะที่ **conversion lock** ระดับระบบ active (`PR_AUTH_007` analog — Finance / sys-admin freeze ชุดกฎชั่วคราวระหว่าง window การแปลง year-end) | **Deny.** Surface configurational ถูก lock ขณะ flag conversion-lock เปิด; rule editor เป็น read-only พร้อม banner อธิบาย window lock; API call ตรงเพื่ออัปเดตชุดกฎถูก reject ด้วย `"Allocate Vendor rules are locked during the active conversion window — contact System Administrator."` Lock เองดูแลโดย scope admin แบบ `PR_AUTH_007` (Finance / sys-admin เท่านั้น) |

| PM-PERM-06 | ผู้ใช้ non-purchase (Requestor / HOD / Finance) เปิด **Vendor Allocation Rules** และพยายาม save การเปลี่ยนน้ำหนัก | **Deny — role ผิด.** `PR_AUTH_008` จำกัดการดูแลกฎเฉพาะ `enum_stage_role = purchase` UI ซ่อนหรือ disable ปุ่ม **Save** สำหรับ role non-purchase; API call ตรงเพื่ออัปเดตชุดกฎถูก reject ด้วย `"You are not authorised to edit Allocate Vendor rules"` |
| PM-PERM-07 | Configurational delegation — Procurement Manager พยายาม delegate สิทธิ์ **rule-maintenance** ให้ผู้ใช้คนอื่น | **Deny by default.** `PR_AUTH_006` ครอบคลุม delegation ของสิทธิ์อนุมัติ transactional เท่านั้น; สิทธิ์ configurational (การดูแลกฎ) **ไม่** delegate ได้ตาม [03-user-flow-procurement-manager.md](./03-user-flow-procurement-manager.md) Section 3 UI delegation scope สิทธิ์ที่ delegate ได้เฉพาะ surface transactional และ disable / ซ่อน toggle configurational; API call ตรงเพื่อขยาย delegation ไปยังการดูแลกฎถูก reject ด้วย `"Allocate Vendor rule maintenance cannot be delegated"` |

## 3. Validation / Error

| # | Scenario | Trigger | Error ที่คาด |
| - | -------- | ------- | -------------- |
| PM-VAL-01 | Threshold อนุมัติไม่ match — PR จริง ๆ ไม่เกิน threshold ของ Procurement Manager | PR ที่ misroute ด้วย `base_total_amount = 50,000 ฿` ลงในคิว Escalated PRs แต่ threshold escalation ที่ตั้งคือ `> 100,000 ฿` ตาม `PR_AUTH_005`; Procurement Manager คลิก Approve | Reject — `PR_AUTH_005` re-validate threshold ตอน action Server reject ด้วย `"This PR does not meet the escalated-stage threshold and should not be at this stage"`; action ถูก block และ workflow engine ถูก flag สำหรับการซ่อมแซม stage-routing PR ยังคงเป็น `in_progress` ที่ stage ก่อนหน้าที่ควรหยุด |
| PM-VAL-02 | การเปลี่ยนกฎโดยไม่มี diff (no-op save) | เปิด Vendor Allocation Rules → คลิก **Save** โดยไม่เปลี่ยนน้ำหนักหรือ override | Reject — ผู้เขียน configuration ต้องการ diff ไม่ว่างกับชุดกฎ active Server reject ด้วย `"No changes detected — nothing to save"` UI ก็ disable ปุ่ม **Save** จนกว่าจะมีการเปลี่ยนที่ staged อย่างน้อยหนึ่ง ไม่มีแถว effective-from ใหม่เขียนและไม่มี notification fire |
| PM-VAL-03 | Commit การเปลี่ยนกฎขณะที่ PR กลาง flow ยังพึ่งกฎเก่า — snapshot preservation ตาม `PR_CALC_006` | PR `in_progress` มี `vendor_id`, `vendor_name`, `pricelist_detail_id`, `pricelist_price` ที่ snapshot จาก PR submit ภายใต้ชุดกฎ **เก่า**; Procurement Manager save ชุดกฎใหม่ห้านาทีต่อมา; Approver แล้ว approve PR | **อนุญาตให้ save แต่ snapshot ถูกรักษา.** การ save configurational ไม่ถูก block โดย PR กลาง flow (ตาม [03-user-flow-procurement-manager.md](./03-user-flow-procurement-manager.md) Section 3); อย่างไรก็ตาม PR กลาง flow คง allocation ที่ snapshot **เก่า** ผ่านไปจนถึงการแปลง PO กฎใหม่ใช้กับ: (a) PR ที่สร้างหลัง timestamp effective-from, (b) PR ที่ส่งกลับเป็น `draft` ผ่าน send-back, และ (c) PR ที่รันผ่าน bulk re-allocation ครั้งเดียว ความพยายามใด ๆ เพื่อ re-rank PR `in_progress` ย้อนหลังผ่าน API call ตรงถูก reject ด้วย `"Cannot re-allocate vendor on an in-flight PR — use Stuck PR Oversight bulk re-allocation"` |
| PM-VAL-04 | น้ำหนัก vendor priority ไม่รวมเป็น 100 % | เปิด Scoring Weights และใส่ `{ vendor_rank: 50, lowest_price: 30, last_receiving: 30 }` (ผลรวม = 110) | Reject — น้ำหนัก scoring ต้องรวมเป็น 100 พอดี ข้ามสามเกณฑ์ Server reject ด้วย `"Scoring weights must sum to 100"` UI ก็ surface indicator live `Σ = 110 ≠ 100` และ disable ปุ่ม **Save** จนกว่าผลรวมจะถูก Panel simulation ไม่ re-rank จนกว่า validation จะผ่าน |
| PM-VAL-05 | น้ำหนัก scoring ไม่ valid — ค่าลบ | ใส่ `{ vendor_rank: -10, lowest_price: 60, last_receiving: 50 }` | Reject — น้ำหนัก scoring แต่ละตัวต้อง satisfy `0 ≤ weight ≤ 100` Server reject ด้วย `"Scoring weight must be a non-negative percentage (0–100)"`; UI flag input ลบด้วย error inline และ disable **Save** เส้นทางเดียวกัน reject ค่าเหนือ 100 ด้วยข้อความเดียวกัน |
| PM-VAL-06 | Approve / Reject / Send Back โดยไม่มีเหตุผลที่จำเป็น | เปิด dialog **Reject** หรือ **Send Back**, ทิ้ง textarea เหตุผลว่าง, คลิก Confirm | Reject — เหตุผล mandatory ตาม contract dialog สำหรับ Reject (`PR_AUTH_004` + `PR_POST_006`) และ Send Back (`PR_POST_003`); ทั้งสองต้องการ payload system comment immutable (`PR_POST_008`) UI disable ปุ่ม Confirm เมื่อเหตุผลว่าง; API call ตรงถูก reject ด้วย `"A reason is required to reject this PR"` (Reject) หรือ `"A reason is required to send back this PR"` (Send Back) Approve อนุญาต comment optional |
| PM-VAL-07 | แก้ `approved_qty` ที่ stage escalated แต่ละเมิด `PR_VAL_013` | บนบรรทัดที่ `requested_qty = 100`, Procurement Manager พิมพ์ `approved_qty = 120` แล้วคลิก Approve | `PR_VAL_013` — reject ด้วย `"Approved quantity must be positive and may not exceed requested quantity"` Stage escalated สืบทอดกฎ validation ระดับบรรทัด Approver พื้นฐานทั้งหมด; กฎเข้มเป็น `> 0` และ `≤ requested_qty` evaluate หลังแปลง UoM โดยใช้ `approved_unit_conversion_factor` |
| PM-VAL-08 | Override send-back ของ Approver ที่พยายามบน PR ที่ยุติแล้ว | Approver ก่อนหน้าได้ issue Send Back และ PR กลับเป็น `draft`; Requestor ยังไม่ resubmit; Procurement Manager พยายาม override send-back จาก UI stale โดยคลิก Approve | Reject — guard state-machine: PR ไม่อยู่ใน `in_progress` ที่ stage escalated อีกแล้ว `PR_AUTH_002` re-evaluate กับ `workflow_current_stage` ปัจจุบัน (stage `create` ของ Requestor) และพบ Procurement Manager ไม่อยู่ใน `user_action.execute[]` ของ stage นั้น Server reject ด้วย `"This PR has been sent back to the Requestor and cannot be approved without resubmission"` UI ของ Procurement Manager prompt refresh |

## 4. Edge Case

| # | Scenario | เงื่อนไข | คาดหวัง |
| - | -------- | --------- | -------- |
| PM-EDGE-01 | PR กลาง flow vs ขอบเขต snapshot กฎใหม่ | PR-A submit ที่ `T-0` ภายใต้ชุดกฎ `R1`; vendor snapshot ถ่ายตอน submit Procurement Manager save ชุดกฎ `R2` ที่ `T+5m` PR-B สร้างที่ `T+6m` และ submit ที่ `T+7m` | PR-A ยังคง snapshot `R1` ผ่านการอนุมัติและเข้าการแปลง PO — คอลัมน์ `vendor_id`, `vendor_name`, `pricelist_*` บน `tb_purchase_request_detail` ไม่เปลี่ยน Vendor snapshot ของ PR-B ตอน submit ถ่ายภายใต้ `R2` — น้ำหนักใหม่, override ต่อ vendor ใหม่ใช้ ขอบเขตคือ `timestamp` effective-from ของ `R2` ไม่ใช่เวลา approve ของ PR เลียนแบบ pattern snapshot exchange-rate `PR_CALC_006` |
| PM-EDGE-02 | Boundary threshold พอดี — `base_total_amount` ลงที่ขอบเขตล่างของ Procurement Manager พอดี | Threshold escalation ตั้งที่ `base_total_amount > 100,000.00000 ฿`; Approver ปรับ `approved_qty` ให้ `base_total_amount` ใหม่ = `100,000.00000 ฿` (พอดี) | เปรียบเทียบ threshold เข้มเป็น `>` ตาม `PR_AUTH_005` ดังนั้นค่าขอบเขตพอดี **ไม่** trigger escalation; PR **ไม่** route ไป Procurement Manager และ finalise ที่ stage Finance ปกติแทน ค่า `100,000.00001 ฿` จะ escalate (ตัวเลข threshold เฉพาะตั้งค่าได้ต่อองค์กร; operator ไม่ใช่ตัวเลขคือ invariant — pattern เดียวกับ APP-EDGE-05) |
| PM-EDGE-03 | Bulk-action failure บางส่วน — บาง PR สำเร็จ, บาง fail | Procurement Manager เลือกสิบ PR จาก Stuck PR Oversight สำหรับ Escalation Ping เจ็ดผ่าน `PR_AUTH_002` สำหรับ Procurement Manager และมี stage ปัจจุบัน valid; สอง scope ไปยัง business unit ที่ Procurement Manager ไม่มีสิทธิ์; หนึ่งเพิ่ง void โดย Finance (`PR_AUTH_007`) เมื่อนาทีที่แล้ว | First-pass aggregation: `PR_AUTH_002` และ guard state-machine รันต่อ PR เจ็ด PR ที่มีสิทธิ์, valid ได้ ping และ comment `type = system` บันทึก bulk source (`PR_POST_008`) สอง PR ที่ไม่มีสิทธิ์ถูกตัดออกแบบเงียบพร้อม log line อธิบาย หนึ่ง PR ที่ void ถูก reject ที่ guard state-machine ด้วย `"This PR is voided and cannot be acted on"` และอยู่ใน breakdown failure Dialog ผลแสดง `7 succeeded / 3 failed (2 unauthorised, 1 voided)` ไม่มี rollback transactional บางส่วน; PR สำเร็จยังคง ping; PR fail ถูกทิ้งไม่แตะ |
| PM-EDGE-04 | Chain delegation ไป Procurement Manager หมดกลางการตัดสินใจ | Procurement Manager A ลาและ delegate authority stage escalated ให้ Procurement Manager B ตาม `PR_AUTH_006` สำหรับ window `[T-2d, T]`; B เปิด PR escalated ที่ `T-1m`, เดิน review, และคลิก Approve ที่ `T+30s` (หลัง delegation หมด) | Reject ที่ server `PR_AUTH_002` re-evaluate `user_action.execute[]` กับ clock ปัจจุบัน; delegate B ที่หมดอายุไม่อยู่ใน list อีก Server คืน `"Your delegation has expired — please refresh"` UI surface ข้อความและ prompt refresh Procurement Manager A ดั้งเดิม (หรือ delegate ใหม่) ต้องลงมือ pattern เดียวกับ APP-EDGE-04 ใน scenario Approver |
| PM-EDGE-05 | Save configurational vs concurrent one-off re-allocation กลาง flow | Procurement Manager เปิด **Stuck PR Oversight** และเริ่ม one-off re-allocation สำหรับสาม PR ที่ `T`; ก่อน bulk job commit, Procurement Manager B save ชุดกฎใหม่ที่ `T+1s` | One-off re-allocation อ่านชุดกฎในขณะที่ re-allocation per-PR แต่ละตัวรัน Per-PR re-allocation ที่รันก่อน `T+1s` ใช้ `R_old`; ที่รันหลัง `T+1s` ใช้ `R_new` ไม่มีการรับประกัน transactional ว่าทั้งสาม PR ใช้ชุดกฎเดียวกันใน window write concurrent Audit log ของ bulk-action บันทึก timestamp ของชุดกฎที่ใช้สำหรับแต่ละ PR เพื่อให้ divergence สังเกตได้ ในเชิงปฏิบัติพบไม่บ่อย — `Stuck PR Oversight` และ rule editor บันทึกว่าใช้โดย Procurement Manager หนึ่งคนทีละครั้ง |
| PM-EDGE-06 | Save ชุดกฎลงที่ขณะที่ PR ใหม่กำลัง submit พอดี | ที่ `T`, Requestor คลิก Submit บน PR-X (ที่ trigger routine vendor-snapshot); ที่ instant `T` เดียวกัน, Procurement Manager save ชุดกฎ `R2` (effective-from `T`) | Tie-break โดย system clock ตอน write ถ้า routine snapshot ของ PR-X resolve ชุดกฎที่ `T-1ms`, มันอ่าน `R1`; ถ้าที่ `T+1ms`, อ่าน `R2` ทั้งสองผลลัพธ์ valid — contract คือ "ใช้ชุดกฎใดก็ตามที่ effective ในขณะ snapshot" ไม่ใช่ "ให้ PR-X ชุดกฎที่ต้องการ" Audit trail บนทั้งแถว configuration และแถว PR detail บันทึก timestamp เพื่อให้การเลือกสังเกตได้ |
