# ใบขอซื้อ (Purchase Request) — Test Scenarios — Approver

> **At a Glance**
> **Persona:** Approver (Department Head, Budget Controller, Finance Officer / Manager) &nbsp;·&nbsp; **โมดูล:** [purchase-request](/th/inventory/purchase-request) &nbsp;·&nbsp; **Scenario:** ~33
> **หมวด:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** map ไปยัง `tests/303-pr-approver-journey.spec.ts`, `tests/311-pr-returned-flow.spec.ts` และ `tests/301-pr.spec.ts` (fixture hodTest / fcTest / gmTest) ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ test scenario ที่ persona Approver ขับโดยตรงในโมดูล `purchase-request` Approver ครอบคลุมสาม decision-maker กลางใน chain อนุมัติ — **Department Head** (Stage 1), **Budget Controller** (Stage 2) และ **Finance Officer / Manager** (Stage 3) — ทั้งหมด share UI review-and-decide เดียวกัน Scenario ถูก group เป็นสี่ happy-path action ที่อธิบายใน [03-user-flow-approver.md](./03-user-flow-approver.md) Section 2 (approve, send-back, reject, split-reject), ขอบเขต RBAC ที่บังคับโดย `PR_AUTH_002`–`PR_AUTH_006` ใน [02-business-rules.md](./02-business-rules.md) Section 4, กฎ validation ที่ fire บน approval action (โดยเฉพาะ `PR_VAL_013` บน `approved_qty` และ `PR_VAL_016` บน optimistic concurrency) และชุดเล็กของ boundary / concurrency case ที่มาจาก state machine หลาย stage Handoff ข้าม persona ที่ pivot off Approver (`X-PR-02`, `X-PR-03`, `X-PR-05`, `X-PR-06`, `X-PR-07`) อยู่ในภาพรวมหลัก ไม่ใช่ที่นี่

## 2. Permission / Authorization

| # | Scenario | พฤติกรรมที่คาด (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| APP-PERM-01 | Approver เปิด PR ในคิวของตน (ผู้ใช้ปัจจุบันอยู่ใน `user_action.execute[]` สำหรับ `workflow_current_stage`) | **Allow** read และปุ่ม action (Approve / Send Back / Reject) `PR_AUTH_002` ผ่านสำหรับ stage ปัจจุบัน |
| APP-PERM-02 | Approver เปิด PR ที่ stage ต่างกัน (เช่น HOD เปิด PR ที่อยู่ stage Budget Controller ปัจจุบัน) | **Deny action, read-only.** ปุ่ม action (Approve / Send Back / Reject / toolbar Split) ถูก disable พร้อมคำอธิบาย inline; `PR_AUTH_002` คำนวณ `user_action.execute[]` ใหม่ทุกการ transition stage และ HOD ไม่อยู่ใน list นั้นแล้ว Visibility list อาจยังให้แถวปรากฏใต้ **All Documents** ขึ้นกับ grant |
| APP-PERM-03 | Approver คลิก **Approve** ที่ stage ปัจจุบัน | **Allow.** `PR_AUTH_002` ผ่านสำหรับ stage ปัจจุบัน; `PR_AUTH_003` ให้สิทธิ์ approve / send-back / reject เป็นสิทธิ์ระดับบรรทัด; `PR_POST_004` (หรือ `PR_POST_005` ถ้าสุดท้าย) commit transition |
| APP-PERM-04 | Approver คลิก **Approve** บน PR ที่ already advanced past stage ของพวกเขา (race หลัง approver concurrent อื่นลงมือ) | **Deny.** `PR_AUTH_002` ฝั่ง server reject เพราะ `user_action.execute[]` ถูกคำนวณใหม่สำหรับ stage ใหม่และผู้ใช้ที่เรียกไม่อยู่ใน list นั้นแล้ว; UI แสดงข้อความ "PR has moved to a later stage — please refresh" และ disable action bar ตอน refresh |
| APP-PERM-05 | Department Head approve PR จากแผนกที่ตนไม่รับผิดชอบ | **Deny.** Routing default chain Stage 1 scope `user_action.execute[]` ไปยังแผนกของ requestor (หรือ HOD ที่ตั้งสำหรับแผนกใน `tb_workflow`) HOD แผนกอื่นไม่อยู่ใน list; `PR_AUTH_002` บล็อก action |
| APP-PERM-06 | Approver พยายาม approve PR ที่ตน submit เอง (segregation of duties) | **Deny.** Role stage `create` และ role stage `approve` ปลายน้ำ disjoint ตามนิยาม workflow แม้ผู้ใช้เดียวกันปรากฏในทั้งสอง stage โดย misconfig `user_action.execute[]` ถูกคำนวณใหม่ต่อ stage และเจ้าของ create-stage ถูก filter ออกก่อน action รัน (ดู [02-business-rules.md](./02-business-rules.md) Section 4, chain default) |
| APP-PERM-07 | ผู้ใช้ Delegate (ตั้งตาม `PR_AUTH_006`) ลงมือกับ PR ที่ delegate ใน window delegation | **Allow.** Delegation ถูก materialize เข้า `user_action.execute[]` สำหรับ window delegation; delegate สืบทอดสิทธิ์ approve / send-back / reject / split-reject เฉพาะ window นั้น `last_action_by_id` บันทึก delegate; comment audit บันทึกแหล่งที่มา delegation |
| APP-PERM-08 | Approver เปิด URL PR ใด ๆ ด้วย fixture `noAuth` / session หมดอายุ | **Deny — redirect ไป login.** ไม่มี auth context, `user_action.execute[]` ไม่สามารถ evaluate กับผู้ใช้ปัจจุบัน; `PR_AUTH_002` ผ่านไม่ได้ Covered โดย fixture `noAuthTest` ใน `301-pr.spec.ts` |

## 3. Validation / Error

| # | Scenario | Trigger | Error ที่คาด |
| - | -------- | ------- | -------------- |
| APP-VAL-01 | Approver ลงมือกับ PR ที่ตนไม่มีสิทธิ์ | Approver ที่ **ไม่อยู่** ใน `user_action.execute[]` สำหรับ `workflow_current_stage` เรียก endpoint Approve โดยตรง (bypass ปุ่ม UI ที่ disabled) | `PR_AUTH_002` — reject ด้วย `"You are not authorised to act on this PR at the current stage"` List ถูกคำนวณใหม่ทุกการ transition stage; client ที่ stale ตกในเส้นทางนี้ |
| APP-VAL-02 | Approver คลิก **Send Back** โดยไม่ให้เหตุผล | เปิด dialog send-back, ทิ้ง textarea เหตุผลว่าง, คลิก Confirm | Reject — เหตุผล mandatory ตาม contract dialog; ปุ่ม **Confirm** ก็ disabled ใน UI เมื่อเหตุผลว่าง แต่การเรียก API ตรงถูก reject ที่ server ด้วย `"A reason is required to send back this PR"` Comment `type = system` ที่ `PR_POST_003` เขียนต้องมีข้อความเหตุผล; ถ้าไม่มี write fail |
| APP-VAL-03 | Approver คลิก **Reject** โดยไม่ให้เหตุผล | เปิด dialog reject, ทิ้งเหตุผลว่าง, คลิก Confirm | Reject — `PR_AUTH_004` + `PR_POST_006` ต้องการ system comment immutable พร้อมเหตุผล reject (`PR_POST_008`); server reject การเรียกเหตุผลว่างด้วย `"A reason is required to reject this PR"` |
| APP-VAL-04 | Approver ตั้ง `approved_qty = 0` และคลิก Approve | บนบรรทัด, ใส่ `0` ใน input approved-qty | `PR_VAL_013` — reject ด้วย `"Approved quantity must be positive and may not exceed requested quantity"` กฎเข้มเป็น `> 0` ไม่ใช่ `≥ 0` (ดู APP-EDGE-01 — UI ยังถือ `approved_qty = 0` เป็น reject ต่อบรรทัดเมื่อ surface ผ่าน bulk split toolbar; เส้นทาง validation ใต้ toolbar ยัง reject save raw ของ `0`) |
| APP-VAL-05 | Approver ตั้ง `approved_qty > requested_qty` | บนบรรทัดที่ `requested_qty = 100`, ใส่ `120` ใน input approved-qty | `PR_VAL_013` — reject ด้วย `"Approved quantity must be positive and may not exceed requested quantity"` เปรียบเทียบทำหลังแปลง UoM โดยใช้ `approved_unit_conversion_factor` |
| APP-VAL-06 | Approver แก้ `approved_qty` แต่ omit `approved_unit_id` / `approved_unit_conversion_factor` | Force-save ด้วย `approved_qty` ที่ไม่ null และ `approved_unit_id` null (โดยทั่วไปเข้าถึงผ่าน API call ตรงเท่านั้น) | `PR_VAL_013` — reject ด้วย `"Approved quantity must be positive and may not exceed requested quantity"` (กฎยังต้องการ `approved_unit_id` และ `approved_unit_conversion_factor` ต้องส่งมาด้วย `approved_qty`) |
| APP-VAL-07 | Approver พยายาม approve PR ที่ `workflow_id` required ไม่ active อีกต่อไป (scope เปลี่ยนหรือแถวถูก soft-delete กลางบิน) | แถว workflow ใน `tb_workflow` ถูก deactivate หรือ scope flip จาก `purchase-request` หลัง PR submit; Approver คลิก Approve | Reject — `PR_VAL_004` ถูก re-evaluate บน action; `"A valid PR workflow must be selected"` PR ค้างอย่างมีประสิทธิภาพจนกว่า admin re-activate workflow valid หรือ void PR (`PR_AUTH_007`) |
| APP-VAL-08 | Approver แก้ `approved_qty` ตาม `doc_version` ที่ stale (มี concurrent edit อื่นเกิด) | สอง Approver (หรือ Approver และ delegate) load PR เดียวกัน; A save edit `approved_qty`; B (ยังถือ `doc_version` เก่า) คลิก Approve | `PR_VAL_016` — reject ด้วย `"Document was modified by another user; reload and retry"` Write สำเร็จ bump `doc_version` ขึ้น 1 |
| APP-VAL-09 | Split-reject โดยไม่มีเหตุผลต่อบรรทัดบนบรรทัดที่ reject | Mark `L2` เป็น **reject** แต่ทิ้งเหตุผลต่อบรรทัดว่าง; คลิก Approve ที่ header | Reject — การ reject ระดับบรรทัดต้องการเหตุผลตาม `PR_AUTH_003` และ contract write comment ต่อบรรทัด (`PR_POST_008` ต้องการ payload ของ comment) UI ก็ disable ปุ่ม Approve จนกว่าทุกบรรทัด reject จะมีเหตุผลไม่ว่าง |
| APP-VAL-10 | Approver พยายามลงมือหลัง PR ถูก void โดย Finance / sys-admin | `pr_status` ของ PR เพิ่งพลิกเป็น `voided` (`PR_AUTH_007`); Approver load หน้าก่อน void และคลิก Approve | Reject — guard state-machine: `voided` คือ terminal และ freeze เอกสาร Endpoint Approve / Send Back / Reject reject ด้วย `"This PR is voided and cannot be acted on"` |

## 4. Edge Case

| # | Scenario | เงื่อนไข | คาดหวัง |
| - | -------- | --------- | -------- |
| APP-EDGE-01 | `approved_qty = 0` ถือเป็น rejection ต่อบรรทัด | Approver ใช้ toolbar **Split** แบบ bulk เพื่อตั้ง `approved_qty = 0` บน `L2` ขณะที่ทิ้ง `L1` ที่ requested qty | ตาม contract toolbar `approved_qty = 0` ถูก materialize เป็น **reject** ต่อบรรทัดพร้อมเหตุผลที่ system สร้าง; บรรทัด save ด้วย `current_stage_status = rejected` (ไม่ถูก reject โดย `PR_VAL_013` เพราะเส้นทาง toolbar ไม่เขียน `0` ไปคอลัมน์ — เขียน `null` + flag reject) Edit ตรงของ `approved_qty = 0` นอก toolbar split ยังถูก reject โดย `PR_VAL_013` |
| APP-EDGE-02 | Boundary `approved_qty > requested_qty` | `requested_qty = 12.00000`, Approver ใส่ `approved_qty = 12.00001` | Reject ตาม `PR_VAL_013` — กฎเข้มเป็น `≤` evaluate ที่ความแม่นยำ `Decimal(15, 5)` เต็ม; เท่ากัน (`12.00000 == 12.00000`) ถูกต้อง อะไรเหนือทศนิยมที่ 5 ไม่ |
| APP-EDGE-03 | Approve พร้อมกันโดยสอง delegate ของ stage เดียวกัน | HOD delegate ให้ผู้ใช้สองคน; ทั้งคู่เปิด PR ภายในไม่กี่วินาทีและทั้งคู่คลิก Approve | Approve แรกชนะและ bump `doc_version` + เลื่อน `workflow_current_stage`; Approve ที่สองถูก reject — โดย `PR_VAL_016` ถ้า client ที่สองยังถือ `doc_version` เก่า หรือโดย `PR_AUTH_002` ถ้า client refresh แล้ว (PR advanced past Stage 1 แล้วและ delegate ที่สองไม่อยู่ใน `user_action.execute[]`) Last-write-wins **ไม่ใช่** นโยบาย |
| APP-EDGE-04 | Window delegation หมดกลางการตัดสินใจ | Delegate เปิด PR ที่ `T-1m`; delegation หมดที่ `T`; delegate คลิก Approve ที่ `T+30s` | Reject ที่ server `PR_AUTH_002` re-evaluate `user_action.execute[]` กับ clock ปัจจุบัน; delegate ที่หมดอายุไม่อยู่ใน list อีก UI surface `"Your delegation has expired — please refresh"` Approver ดั้งเดิม (หรือ delegate ใหม่) ต้องลงมือ |
| APP-EDGE-05 | Boundary threshold — ยอด header ลงที่ threshold escalation พอดี | `tb_workflow` ตั้ง routing Stage 4 ที่ `base_total_amount > 100,000.00000 ฿`; Approver ปรับ `approved_qty` ให้ `base_total_amount` ใหม่ = `100,000.00000 ฿` (พอดี) | เปรียบเทียบ threshold เข้มเป็น `>` ตาม `PR_AUTH_005` ดังนั้นค่าขอบเขตพอดี **ไม่** trigger escalation Stage 4; PR route ไป stage ปกติถัดไป ค่า `100,000.00001 ฿` จะ trigger escalation (ตัวเลข threshold เฉพาะตั้งค่าได้ต่อองค์กร — กฎคือ operator เปรียบเทียบ ไม่ใช่ตัวเลข) |
| APP-EDGE-06 | Send-back ได้รับโดย Requestor ขณะ Approver ที่สอง load PR | Approver A คลิก Send Back ที่ `T`; Approver B (parallel reviewer / observer) อยู่บนหน้า detail อยู่แล้วและคลิก Approve ที่ `T+2s` | Reject สำหรับ B `PR_AUTH_002` re-evaluate บน action และพบ `workflow_current_stage` rolled back ไป stage `create`; B ไม่อยู่ใน `user_action.execute[]` ของ stage นั้น UI prompt B ให้ refresh |
| APP-EDGE-07 | ความแม่นยำทศนิยมสูงสุดในการปรับ `approved_qty` | บรรทัดมี `requested_qty = 1.99999`; Approver ใส่ `approved_qty = 1.99998` | ยอมรับ — `PR_VAL_013` ผ่าน (`> 0` และ `≤ requested_qty`); ค่า persist ตรงที่ 5 ทศนิยมบน `Decimal(15, 5)`; roll-up header ถัดไป round ตามนโยบายใน [02-business-rules.md](./02-business-rules.md) Section 3 |
