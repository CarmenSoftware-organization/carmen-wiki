# ใบขอซื้อ (Purchase Request) — Test Scenarios — Audit & Config

> **At a Glance**
> **Persona:** Audit / Config (Auditor read-only + System Administrator config) &nbsp;·&nbsp; **โมดูล:** [[purchase-request]] &nbsp;·&nbsp; **Scenario:** ~30
> **หมวด:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** ยังไม่มี `30X-pr-audit-config-journey.spec.ts` เฉพาะ; เส้นทางอ่าน audit-log บางส่วนผ่าน assertion activity-log ใน `tests/301-pr.spec.ts` ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ test scenario ที่แกน persona **Audit / Config** ขับในโมดูล `purchase-request` สอง role share แกนนี้แม้จะนั่งบนสองฝั่งของ read / write: **Auditor** เป็น persona อ่านอย่างเดียวที่เดิน activity trail immutable (`workflow_history`, `tb_purchase_request_comment` ตาม `PR_POST_008`, ประวัติ `current_stage_status` ระดับบรรทัด, snapshot vendor / pricelist / exchange-rate จาก `PR_CALC_006`) เพื่อ verify การปฏิบัติตามนโยบาย, segregation of duties และความสมบูรณ์ของ trail; **System Administrator** เป็น persona configuration ที่เป็นเจ้าของ stage workflow, amount threshold (`PR_AUTH_005`), กฎ delegation และ window (`PR_AUTH_006`), default ต่อประเภท PR, master tax / currency ที่ feed `PR_CALC_002`–`PR_CALC_006` และ assignment ผู้ใช้ / role บน `user_action.execute[]` ไม่มี role ใดอยู่บนเส้นทาง happy ของ request-to-PO Scenario ถูก group เป็น happy path จาก [03-user-flow-audit-config.md](./03-user-flow-audit-config.md) Section 2 (Auditor: query PR ตามงวด, drill ลง activity log, export พร้อมการอนุมัติทุติยภูมิ, flag การละเมิดนโยบาย; Sysadmin: เปลี่ยน stage workflow, ปรับ threshold, ตั้ง window delegation, เปลี่ยน default ประเภท PR, เปลี่ยน master tax / currency), ขอบเขต RBAC จาก `PR_AUTH_002`, `PR_AUTH_005`–`PR_AUTH_008` ใน [02-business-rules.md](./02-business-rules.md) Section 4, กฎ validation ที่ fire บนแต่ละ surface (snapshot preservation ตาม `PR_CALC_006`, นโยบายการอนุมัติ export, การป้องกัน deadlock configuration, การเช็คช่วง threshold, limit scan ช่วงวันที่) และชุดของ boundary case ที่ unique กับการมีทั้ง audit surface passive และ configuration surface active Handoff ข้าม persona ที่ pivot off แกน Audit / Config (Auditor case-file escalation ที่ trigger void `PR_AUTH_007` ของ Sysadmin, Sysadmin rollback กระทบ submission ของ Requestor ในเวลาต่อมา) อยู่ในภาพรวมหลัก ไม่ใช่ที่นี่

## 2. Permission / Authorization

Section 2 แบ่ง Auditor (allow read-only / deny write) จาก Sysadmin (allow configure / deny transact)

| # | Scenario | พฤติกรรมที่คาด (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| AUD-PERM-01 | Auditor เปิด activity log ของ PR ใด ๆ ใน scope | **Allow read-only.** Role `auditor` ให้ read บน `tb_purchase_request`, `tb_purchase_request_detail`, `workflow_history`, `tb_purchase_request_comment`, และ `user_action` ไม่มีสิทธิ์ mutation; action bar บนหน้า PR detail ซ่อนสำหรับ Auditor Drill-down เองถูก log ใน query log ฝั่ง audit |
| AUD-PERM-02 | Auditor พยายามแก้ PR, เปลี่ยน `approved_qty` หรือคลิก Approve / Reject / Send Back / Void | **Deny — role read-only.** `PR_AUTH_002` ต้องการผู้ใช้อยู่ใน `user_action.execute[]` สำหรับ stage ปัจจุบัน; Auditor **ไม่เคย** อยู่ใน list นั้นโดยการตั้งค่า UI ซ่อน control write; API call ตรงไปยัง `PATCH /pr/:id` หรือ endpoint action workflow ใด ๆ ถูก reject ด้วย `"You are not authorised to act on this PR — auditor role is read-only"` |
| AUD-PERM-03 | Sysadmin เปิด **PR Workflow Settings** และ save การเปลี่ยน stage / threshold / delegation | **Allow.** Role `sysadmin` ให้ write บนตาราง configuration workflow, ตารางกฎ threshold, ตาราง delegation, ตาราง default ประเภท PR และตาราง master tax / currency ทุก save เขียนไปยัง configuration audit log ด้วย `user_id` ของ Sysadmin Analog `PR_AUTH_007` และ `PR_AUTH_008`: scope configuration ถือโดย `sysadmin` exclusive |
| AUD-PERM-04 | Sysadmin พยายาม approve / reject / send-back PR (action transactional) | **Deny — role ผิด.** Sysadmin **ไม่อยู่** ใน `user_action.execute[]` สำหรับ stage transactional ใด ๆ โดย default (เว้นแต่ assign ชัดเจนเป็น user grant แยก ซึ่งไม่แนะนำสำหรับ segregation of duties) UI ซ่อนปุ่ม Approve / Reject / Send Back; API call ตรงถูก reject โดย `PR_AUTH_002` ด้วย `"You are not authorised to act on this PR at the current stage — sysadmin role does not include transactional approval"` Action เดียวที่เปลี่ยนสถานะ PR ที่ Sysadmin ทำได้คือ **void** ระดับสูงภายใต้ `PR_AUTH_007` (ดู `AUD-PERM-06`) |
| AUD-PERM-05 | Auditor export report CSV / PDF ที่รวมฟิลด์ sensitive | **Allow, gated บนการอนุมัติทุติยภูมิ.** Auditor สามารถ submit คำขอ export; export นั่งใน `pending` จนกว่า data-export approver ที่ตั้งจะปล่อย (ตาม `AUD-HP-03`) UI surface สถานะ pending; API call ตรงไปยัง `GET /audit/export/:id/download` บน export `pending` ถูก reject ด้วย `"Export is pending approval"` Export non-sensitive scope (เช่น report count-only) bypass การอนุมัติทุติยภูมิแต่ยัง log |
| AUD-PERM-06 | Sysadmin ทำ void ระดับสูงบน PR กลาง flow หรือ approved (`PR_AUTH_007`) | **Allow พร้อม audit trail.** `PR_AUTH_007` สงวน action `void` ให้ scope Sysadmin / Finance Sysadmin เปิด PR, คลิก **Void**, พิมพ์เหตุผล mandatory, ยืนยัน Server รัน `PR_AUTH_007` → `PR_POST_006`: `pr_status` พลิกเป็น `voided` (terminal); commitment soft / hard ปล่อย; `workflow_history` append; comment `type = system` จับเหตุผล (`PR_POST_008`); Auditor เห็น void event ครั้งถัดไปที่ query นี่เป็น action เดียวของ Sysadmin ที่เปลี่ยนสถานะ PR — แตกต่างจาก save configuration |
| AUD-PERM-07 | ผู้ใช้ non-auditor (Requestor / Approver / Purchaser) พยายามเปิด **PR Activity Queries** | **Deny — role ผิด.** Workspace Audit จำกัดเฉพาะ role `auditor` และ `sysadmin` UI ซ่อน sidebar entry **Audit** สำหรับ role อื่น; navigation route ตรงถูก reject ด้วย `"You are not authorised to view the audit workspace"` Persona transactional เห็นเฉพาะ PR ของตัวเองและ activity log ของ PR ที่อยู่ใน scope อยู่แล้ว (read-only บน PR เหล่านั้นเท่านั้น ผ่านแท็บ activity-log เดียวกัน ไม่ใช่ query builder) |

## 3. Validation / Error

Section 3 ครอบคลุม validation failure บนทั้งสอง surface: Sysadmin configuration write ที่ละเมิด snapshot, deadlock, range หรือกฎ window; ความพยายาม query / export ของ Auditor ที่ละเมิด limit scan หรือ gate อนุมัติ

| # | Scenario | Trigger | Error ที่คาด |
| - | -------- | ------- | -------------- |
| AUD-VAL-01 | Sysadmin พยายามเปลี่ยน workflow ที่จะ re-route PR กลาง flow (snapshot preservation ตาม `PR_CALC_006`) | Sysadmin ลบ Stage 2 จาก workflow ที่มี 18 PR นั่งที่ Stage 2 ตอนนี้; preview panel แสดง `18 PRs would be left without a stage if re-routed`; Sysadmin คลิก Save | **อนุญาตให้ save แต่ PR กลาง flow คง snapshot เก่า.** ตาม pattern snapshot-preservation ใน [02-business-rules.md](./02-business-rules.md) Section 6 การ save **ไม่** ถูก block; 18 PR กลาง flow ดำเนินต่อภายใต้ configuration 3-stage เก่าจนกว่าจะออกจาก `in_progress`; เฉพาะ PR ที่สร้างหลัง `effective_from` เห็น chain 2-stage ใหม่ Preview panel surface count กลาง flow เพื่อความโปร่งใส API call ตรงใด ๆ ที่พยายาม re-route PR กลาง flow ย้อนหลัง (`POST /pr/:id/re-route`) ถูก reject ด้วย `"Cannot re-route an in-flight PR — snapshot preserved per PR_CALC_006"` |
| AUD-VAL-02 | Auditor พยายาม download export ฟิลด์ sensitive โดยไม่มีการอนุมัติทุติยภูมิ | Auditor submit export ที่รวม `requester_name` และ free-text justification เต็ม; export-approver ยังไม่อนุมัติ; Auditor เรียก `GET /audit/export/:id/download` ตรง | Reject ด้วย `"Export is pending approval — sensitive fields require secondary release"` UI ก็ disable ปุ่ม **Download** จนกว่า approver จะปล่อย export นโยบายการอนุมัติ export บังคับใช้ฝั่ง server; ไม่มี UI-only bypass |
| AUD-VAL-03 | Sysadmin ลบผู้ใช้สุดท้ายจาก `user_action.execute[]` บน stage ที่มี PR กลาง flow รออยู่ (deadlock) | Workflow Stage 2 ปัจจุบันมีผู้ใช้ที่ assigned หนึ่ง `approver-1@blueledgers.com`; PR หนึ่ง `in_progress` ที่ Stage 2; Sysadmin เปิด editor และลบ `approver-1@blueledgers.com` โดยไม่ assign แทน | Reject pre-save ด้วย `"Stage 2 has 1 in-flight PR but would have 0 eligible approvers after this change — assign a replacement or activate a delegation before saving"` Validator จับ deadlock ที่ validate-before-save ตาม [03-user-flow-audit-config.md](./03-user-flow-audit-config.md) Section 3 แขนงการตัดสินใจ; Sysadmin ต้องเพิ่มผู้ใช้แทน, activate delegation `PR_AUTH_006` ที่ครอบคลุม stage ที่ได้รับผลกระทบ หรือยอมรับ deadlock ด้วย toggle override ที่ explicit ที่ log แยกใน configuration audit log |
| AUD-VAL-04 | Sysadmin activate delegation ด้วย `end_at < start_at` หรือด้วย `start_at` ในอดีต | Sysadmin พิมพ์ `start_at = 2026-06-14 00:00`, `end_at = 2026-06-01 23:59` และคลิก Save (window กลับ); หรือ `start_at = 2026-04-01 00:00` สำหรับ window ที่ผ่านไปแล้ว | Reject pre-save ด้วย `"Delegation end_at must be after start_at"` สำหรับ window กลับ หรือ `"Delegation start_at must be in the future or now"` สำหรับ case window ผ่านไปแล้ว UI ก็ disable ปุ่ม Save จนกว่าวันที่จะ valid API call ตรงถูก reject ด้วยข้อความเดียวกัน Validator รันตาม semantic window `PR_AUTH_006` |
| AUD-VAL-05 | Auditor รัน query โดยไม่มี filter หรือด้วยช่วงวันที่เกิน limit scan ที่ตั้ง | Auditor คลิก **Run query** ด้วย chip filter ว่าง หรือด้วย `pr_date` ระหว่าง `2020-01-01` และ `2026-04-30` (range > limit 12 เดือนที่ตั้ง) | Reject ด้วย `"Query must include at least one filter to prevent unbounded scans"` สำหรับ case ว่าง หรือ `"Date range exceeds 12-month limit — split the query or request an offline export"` สำหรับ case over-range UI ก็ live-validate ช่วงวันที่และ disable **Run query** จนกว่า range จะอยู่ใน limit Limit ตั้งค่าได้ต่อองค์กร; validator รันตอน submit |
| AUD-VAL-06 | Sysadmin ใส่ค่า threshold ไม่ valid (ลบ, ศูนย์, ไม่ใช่ตัวเลข หรือเกิน cap ต่อ currency) | Sysadmin พิมพ์ `escalation_threshold = -50000` สำหรับกฎ routing มูลค่าสูง หรือ `escalation_threshold = 0` หรือวาง `"abc"` หรือพิมพ์ค่าเหนือ cap ต่อ currency `9,999,999,999.99999 ฿` | Reject pre-save ด้วยข้อความที่เหมาะสม: `"Threshold must be a positive amount"` สำหรับลบ / ศูนย์, `"Threshold must be numeric"` สำหรับไม่ใช่ตัวเลข, `"Threshold exceeds the configured per-currency cap"` สำหรับ case over-cap UI flag input ไม่ถูก inline และ disable **Save** Validator รันเช็ค range / type เดียวกันที่ปกป้อง `PR_VAL_*` บนฟิลด์ transactional |
| AUD-VAL-07 | Sysadmin save configuration ที่ไม่มี diff (no-op save) | Sysadmin เปิด **PR Workflow Settings**, ไม่แก้, คลิก **Save** | Reject ด้วย `"No changes detected — nothing to save"` ผู้เขียน configuration ต้องการ diff ไม่ว่างกับ version active; UI disable **Save** จนกว่าจะมีการเปลี่ยนที่ staged อย่างน้อยหนึ่ง ไม่มีแถว `effective_from` ใหม่เขียนและไม่มี notification fire (เลียนแบบ `PM-VAL-02`) |
| AUD-VAL-08 | Sysadmin พยายามลบแถว tax-code หรือ currency-rate ที่ snapshot ของ PR กลาง flow อ้างถึง | Sysadmin พยายามลบแถว `VAT_STANDARD` `7%` `effective_from = 2025-01-01` ขณะที่ 42 PR กลาง flow ถือแถวนั้นใน snapshot `vat_rate` ตาม `PR_CALC_002`–`PR_CALC_004` | Reject ด้วย `"Cannot delete tax code in use by 42 in-flight PR snapshots — archive (set effective_to) instead of delete"` Validator บังคับว่าแถว master-data ที่อ้างถึงโดย snapshot active ใด ๆ สามารถถูก **archive** (เพิ่ม `effective_to` เพื่อให้แถวไม่ใช้กับ PR ใหม่) แต่ไม่ **ลบ** ทันที pattern เดียวกันปกป้องแถว currency-rate, แถว master vendor และแถว pricelist |

## 4. Edge Case

Section 4 ครอบคลุม boundary และ concurrency case ที่ unique กับแกน persona audit / config `AUD-EDGE-01` ถึง `AUD-EDGE-03` ฝั่ง configuration; `AUD-EDGE-04` ถึง `AUD-EDGE-06` ฝั่ง audit

| # | Scenario | เงื่อนไข | คาดหวัง |
| - | -------- | --------- | -------- |
| AUD-EDGE-01 | PR กลาง flow vs ขอบเขต snapshot configuration ใหม่ | PR-A submit ที่ `T-0` ภายใต้ workflow `WF1` และ threshold `T1` Sysadmin save `WF2` และ `T2` ที่ `T+10m` (change ticket เดียว) PR-B สร้างที่ `T+11m` และ submit ที่ `T+12m` PR-A อนุมัติผ่านที่ `T+20m` | PR-A เดินผ่าน chain stage ของ `WF1` end-to-end และ evaluate กับ `T1` สำหรับ escalation — snapshot configuration ที่ถ่ายตอน submit ถือผ่านทุกการอนุมัติและเข้าการแปลง PO Workflow snapshot ของ PR-B ตอน submit คือ `WF2` ด้วย threshold `T2` ขอบเขตคือ timestamp `effective_from` ของ configuration ใหม่ ไม่ใช่เวลา approve เลียนแบบ pattern snapshot exchange-rate `PR_CALC_006` และ pattern snapshot vendor-allocation `PM-EDGE-01` |
| AUD-EDGE-02 | Sysadmin rollback การเปลี่ยน configuration ไม่ดี | Sysadmin save `WF2` ที่ `T` ที่ `T+30m`, step verify-activation (`03-user-flow-audit-config.md` Section 2 step 6) พบว่า `WF2` แนะนำ bug routing สำหรับประเภท PR `capital_expense` Sysadmin revert ไป `WF1` ที่ `T+35m` ระหว่าง `T` และ `T+35m` 4 PR ถูกสร้างภายใต้ `WF2` | การ rollback เองเป็น configuration save ของตัวเองด้วย `effective_from = T+35m` ของตัวเอง; configuration audit log บันทึกทั้งการเปลี่ยน forward (`T`) และ rollback (`T+35m`); 4 PR ที่สร้างระหว่าง `T` และ `T+35m` คง snapshot `WF2` ตาม pattern snapshot-preservation (semantic snapshot ปกป้องพวกเขาจากการ re-route ย้อนหลัง); PR ใหม่หลัง `T+35m` ใช้ `WF1` Auditor สามารถ correlate rollback log กับ 4 PR และตัดสินว่าแต่ละต้องการการแก้ไขนอก band (โดยทั่วไป: void `PR_AUTH_007` ของ Sysadmin ต่อ PR ถ้า misrouting unrecoverable) |
| AUD-EDGE-03 | Activation delegation จำนวนมากที่ขอบเขตองค์กร | Delegation bulk activate ที่ `start_at = 2026-06-01 00:00` UTC สำหรับ 23 approver ข้ามหลาย business unit Clock ที่ `start_at` trigger activation event; session ผู้ใช้บางคนกลาง action ตอน activation fire | Activation process delegate-by-delegate ในลำดับ deterministic; สำหรับผู้ delegate แต่ละคน ระบบ augment `user_action.execute[]` บนทุก stage ที่พวกเขาเป็นเจ้าของ (ตาม `PR_AUTH_006`); notification re-fan ใน batch แต่แต่ละ PR ถูกแจ้งครั้งเดียวต่อ delegate เพื่อป้องกัน storm Session กลาง action ที่ activation เห็น delegate ใหม่สะท้อนบน refresh ถัดไป; request server กลาง flow ที่เริ่มก่อน activation complete ภายใต้ `user_action.execute[]` ก่อน activation (first-write-wins) Configuration audit log บันทึก bulk activation เป็น event เดียวพร้อม breakdown ระดับ delegate ในเชิงปฏิบัติพบไม่บ่อย — delegation bulk schedule นอกเวลาทำการตาม convention |
| AUD-EDGE-04 | Auditor พบช่องว่างใน audit log (system event ขาด) | Drill-down บน `PR-202604-0077` แสดง state transition `in_progress → approved` ที่ `T` แต่ไม่มีแถว comment `type = system` ที่ `T` เพื่อบันทึก action และไม่มีแถว `workflow_history` ที่สอดคล้อง | Auditor flag PR ใน case file พร้อมหลักฐานช่องว่าง (timestamp `T`, แถวที่ขาด) Flag ไม่เปลี่ยนสถานะ PR Target routing = `Compliance` บวก tag แยกไปยังทีม platform สำหรับ review ความสมบูรณ์ log ตามกฎ immutability (`PR_POST_008`) audit trail ตั้งใจให้ตรวจจับการแทรกแซง — ช่องว่างเองเป็น anomaly ที่รายงานได้และ case file กลายเป็นหลักฐาน Sysadmin จะ *ไม่* พยายาม backfill แถวที่ขาด; แทน เปิดการสอบสวนระดับ platform |
| AUD-EDGE-05 | Query Auditor เกิน limit ขนาด dataset และ timeout | Auditor รัน `"All status transitions for a PR"` บน `pr_date 2025-01-01 ถึง 2025-12-31` สำหรับทุก business unit โดยไม่มี filter requestor; ประมาณ count result > 500,000 event; query เกิน timeout ที่ตั้ง | Server คืน `"Query exceeds maximum result size — narrow the filter set or split into smaller windows"` (หรือถ้า timeout fire กลาง stream `"Query timed out after Ns — split or schedule as an offline export"`) UI surface panel suggest refinement: "Try narrowing by business unit, quarter, or PR type." Query ยังคง log ใน query log ฝั่ง audit เป็นความพยายามที่ fail พร้อม filter set และเหตุผล failure — มีประโยชน์สำหรับ review audit-of-audit ของ Auditor เอง |
| AUD-EDGE-06 | Save configuration ลงที่ขณะ PR ใหม่กำลัง submit พอดี | ที่ `T`, Requestor คลิก Submit บน PR-X (trigger routine snapshot สำหรับ workflow, threshold, tax, currency, vendor); ใน instant `T` เดียวกัน Sysadmin save configuration ใหม่ `C2` ด้วย `effective_from = T` | Tie-break โดย system clock ตอน write ถ้า routine snapshot ของ PR-X resolve configuration ที่ `T-1ms`, อ่าน `C1`; ที่ `T+1ms` อ่าน `C2` ทั้งสองผลลัพธ์ valid — contract คือ "ใช้ configuration ใดก็ตามที่ effective ในขณะ snapshot" ไม่ใช่ "ให้ PR-X version ที่ต้องการ" Configuration audit log และ record snapshot ของ PR บันทึกทั้งสอง timestamp เพื่อให้การเลือกสังเกตได้โดย Auditor เลียนแบบ `PM-EDGE-06` |
