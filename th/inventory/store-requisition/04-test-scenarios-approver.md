# ใบเบิกของสโตร์ (Store Requisition) — Test Scenarios — Approver

> **At a Glance**
> **Persona:** Approver (Department Head + approver ระดับสูงขึ้น) &nbsp;·&nbsp; **โมดูล:** [store-requisition](/th/inventory/store-requisition) &nbsp;·&nbsp; **Scenarios:** ~27
> **หมวดหมู่:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** map ไปยัง `tests/701-sr.spec.ts` (TC-SR-060001..060005, 070001..070003, 080001..080002, 090001+, 100001+, 110001+) ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ scenario test ที่ persona Approver (**Department Head** ที่ขั้นอนุมัติแรก และ approver ระดับสูงขึ้น — Operations Manager / Cost Controller — ที่ขั้นหลัง) ขับเคลื่อนโดยตรงในโมดูล `store-requisition` การมีส่วนร่วมของ Approver เริ่มเมื่อ SR transition เป็น `in_progress` และ workflow ลงที่ขั้นอนุมัติที่ Approver อยู่ใน `user_action.execute`; จบเมื่อบรรทัดทั้งหมดถูก action (เดิน workflow ไป fulfillment) หรือเมื่อบรรทัดทั้งหมดถูก reject (auto-cancellation ตาม `SR_POST_004` tail) Scenarios จัดกลุ่มเป็น **happy path** (approve เต็มต่อบรรทัด, ตัดลงพร้อม `approved_message`, reject พร้อม `reject_message`, send-back พร้อม `review_message`, ผลผสมบน SR เดียว, multi-tier escalation, delegation), **RBAC** (segregation-of-duties `Requester ≠ Approver` ตาม `SR_AUTH_011`, authorization ที่ gate ตามขั้นตาม `SR_AUTH_014`, สิทธิ์ที่ delegate), **validation** (test เชิงลบกับ `SR_VAL_010` — `approved_qty > requested_qty`, `reject_message` ว่าง) และ **edge case** เรื่อง approver พร้อมกันบนบรรทัดต่าง, time-out / SLA escalation, threshold มูลค่าของ approver ขั้นสอง และขอบเขต all-lines-rejected auto-cancel Handoff ข้าม persona ที่ pivot จาก Approver (Scenarios 1, 3, 4, 5 ในภาพรวมแม่) อยู่ใน [04-test-scenarios.md](./04-test-scenarios.md) ไม่ใช่ที่นี่

## 2. Permission / Authorization

| # | Scenario | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| APR-PERM-01 | Approver ใน `user_action.execute` ที่ขั้นปัจจุบันกระทำบน SR | **Allow** Action approve / trim / reject / send-back ล้วน enable บนบรรทัดขณะ `doc_status = in_progress` และ `workflow_current_stage` ตรงกัน ตาม `SR_AUTH_005` |
| APR-PERM-02 | ผู้ใช้ที่ **ไม่** อยู่ใน `user_action.execute` พยายามอนุมัติ | **Deny — workflow-stage gating** ตาม `SR_AUTH_014` ผู้ใช้เห็น SR ใน activity feed แต่ปุ่ม **Approve** ถูกซ่อน / disable; การเรียก API โดยตรง reject ด้วย `"You are not assigned to act on this requisition at the current stage."` ไม่เขียน `approved_qty` Map ไปยัง TC-SR-070002 (Unauthorized User Attempts to Approve) |
| APR-PERM-03 | Requester พยายามอนุมัติ SR ของตน | **Deny — SoD `Requester ≠ Approver`** ตาม `SR_AUTH_011` ผู้ใช้คนเดียวถือทั้ง Requester และ Approver rights แต่ถูก block บน SR ของตนด้วย `"You raised this requisition; another user must approve it."` ไม่เขียน Workflow ค้างจนกว่า approver อื่นจะกระทำ (หรือ Inventory Controller void เชิงบริหาร) |
| APR-PERM-04 | Approver พยายามตั้ง `approved_qty > requested_qty` | **Deny — value cap** ตาม `SR_VAL_010` หน้าจอ block การป้อนฝั่ง client; server reject ผ่าน `"Approved quantity cannot exceed requested quantity; to grant more, the requester must amend and resubmit."` Approver ต้องตัดเป็น `requested_qty` หรือส่งบรรทัดกลับขอ requester แก้ |
| APR-PERM-05 | Approver reject บรรทัดโดยไม่มี `reject_message` | **Deny — เหตุผลบังคับ** ตาม `SR_VAL_010` ส่วนสอง Dialog Reject ต้องการ `reject_message` ไม่ว่าง; server reject submission ว่างด้วย `"Rejection reason is required."` |
| APR-PERM-06 | Approver พยายามแก้ `requested_qty` บนบรรทัด | **Deny — ขอบเขต Approver** Approver ตั้ง `approved_qty` (และคอลัมน์ลายเซ็น) ได้ แต่ไม่ใช่ `requested_qty` (ซึ่งเป็นขอบเขต requester และล็อกตอน submit) ฟิลด์ `requested_qty` ของบรรทัด read-only บนหน้าจออนุมัติ การเปลี่ยน requested quantity ต้องส่งบรรทัดกลับเพื่อแก้ (APR-HP-04) |
| APR-PERM-07 | Approver พยายามกระทำบน SR `completed` | **Deny — สถานะจุดสิ้นสุด** ตามกฎกลาง `completed` ล็อกข้าม persona Approver อาจตรวจสอบลายเซ็นเชิงประวัติศาสตร์สำหรับ audit แต่ re-approve หรือ revoke ไม่ได้ |

## 3. Validation / Error

| # | Scenario | Trigger | Error ที่คาดหวัง |
| - | -------- | ------- | --------------- |
| APR-VAL-01 | `approved_qty > requested_qty` ต่อบรรทัด | Approver พยายาม `approved_qty = 30` บนบรรทัดที่ `requested_qty = 25` | **Reject** — `SR_VAL_010` ส่วนแรก Server คืน `"Approved quantity cannot exceed requested quantity; to grant more, the requester must amend and resubmit."` |
| APR-VAL-02 | `approved_qty < 0` | Approver พยายามปริมาณที่อนุมัติติดลบ (ข้อผิดพลาดการป้อนข้อมูลหรือ API mis-use) | **Reject** — `SR_VAL_008` (quantity invariant reject ติดลบทุกขั้น) Server คืน `"Quantities must satisfy 0 ≤ issued_qty ≤ approved_qty ≤ requested_qty."` |
| APR-VAL-03 | Reject บรรทัดด้วย `reject_message` ว่าง | Approver คลิก Reject และ submit โดยไม่ป้อนเหตุผล | **Reject** — `SR_VAL_010` ส่วนสอง Server คืน `"Rejection reason is required."` |
| APR-VAL-04 | Send-back ด้วย `review_message` ว่าง | Approver คลิก Send Back และ submit โดยไม่ป้อนเหตุผล | **Reject** — กลุ่มเดียวกันกับ VAL-03 Server คืน `"Review message is required when sending an SR back for correction."` |
| APR-VAL-05 | Approver กระทำบนบรรทัดนอกขั้นของตน | Approver อยู่ใน `user_action.execute` สำหรับขั้น 1 แต่พยายามกระทำบนบรรทัดที่ `current_stage_status` อยู่ที่ขั้น 2 (ผ่านขั้น 1 แล้ว — ลำดับผิดปกติ) | **Reject** — `SR_AUTH_014` Server คืน `"You are not assigned to act on this requisition at the current stage."` |
| APR-VAL-06 | Warning เกินงบประมาณตอนอนุมัติ | Cost-centre ของเอาท์เลตอยู่ใน ฿1,000 ของงบประมาณรายเดือน; SR `Σ approved_qty × cost_per_unit` จะดันเกิน | **Warn (ไม่ block)** — หน้าจอแสดง warning เกินงบประมาณถ้า Finance ได้เชื่อมโมดูล budget เข้ากับ view ของ approver; Approver อาจดำเนินต่อ (บันทึกการรับทราบ) หรือตัด Server อนุญาตการอนุมัติแต่ log `system` comment สำหรับ Finance ติดตาม Map ไปยัง TC-SR-070003 (Budget Exceeded During Approval) |

## 4. Edge Cases

| # | Scenario | เงื่อนไข | คาดหวัง |
| - | -------- | -------- | ------- |
| APR-EDGE-01 | Approver พร้อมกันบนบรรทัดต่างกัน | สอง approver (ทั้งคู่ใน `user_action.execute` สำหรับขั้นเดียวกัน) เปิด SR เดียวกันและแต่ละคลิก approve บนบรรทัดต่างกันพร้อมกัน | **Locking ระดับบรรทัด** ระบบล็อกที่ระดับบรรทัดผ่าน optimistic concurrency (`doc_version` บน detail) แต่ละบรรทัดยอมรับ approver แรกที่กระทำ; approver ที่สองกระทำบนบรรทัดเดียวกันได้รับ message "line already actioned" และถูกขอให้ refresh ไม่มี double-write ไม่มีลายเซ็นสูญ Workflow เดินเมื่อบรรทัดทั้งหมดถูก action โดย approver บางคน |
| APR-EDGE-02 | บรรทัดทั้งหมด reject → SR cancel อัตโนมัติ | Approver reject บรรทัด active ทุกบรรทัดบน SR (แต่ละบรรทัดมี `reject_message` ของตน); submit decision | **SR cancel อัตโนมัติ** หลังการเขียนต่อบรรทัด `Σ approved_qty = 0` ข้ามบรรทัด active trigger `SR_POST_004` tail; `doc_status = in_progress → cancelled` อัตโนมัติ; SR จุดสิ้นสุด; requester ถูกแจ้งต่อบรรทัด Approver ไม่ต้อง cancel ด้วยมือ — ระบบตรวจจับ state all-rejected |
| APR-EDGE-03 | Time-out / SLA escalation | SR ค้างใน queue ของ Approver เกิน SLA window ของ tenant (เช่น 48 ชม.) โดยไม่มี action | **Escalation ของ workflow** Tenant config ตัดสินใจ: (a) workflow auto-เดินไปยัง approver ระดับสูงขึ้นโดย Approver เดิมถูกแจ้ง; (b) workflow แจ้ง Inventory Controller; (c) workflow ไม่ทำอะไร ทิ้ง SR ค้าง (ต้องการการแทรกแซงด้วยมือ) อำนาจของ Approver ไม่สูญ — แต่ SR ถูก flag ใน variance dashboard สำหรับ review stale-approval |
| APR-EDGE-04 | Approver ตัดเป็น on-hand ต้นทางพอดีตอนอนุมัติ | On-hand ต้นทางที่ Approver เห็น = 15; Approver ตัด `approved_qty = 15` (ตรงกับ on-hand) | **อนุมัติยอมรับ** ตอน commit on-hand ต้นทางอาจขยับ (SR อื่น / การบริโภค) `SR_VAL_013` run ที่ commit เทียบกับ on-hand live — ถ้า on-hand ตกต่ำกว่า 15 Fulfiller short-fulfill; ถ้าเพิ่มขึ้น ไม่กระทบ (Fulfiller cap ที่ `approved_qty = 15` อยู่แล้ว) Approver **ไม่** เห็นความเสี่ยง stock-out ตอน issue ที่อนุมัติ — นั่นเป็นความรับผิดชอบของ Fulfiller |
| APR-EDGE-05 | Multi-tier ที่ขั้นสองตัดเพิ่ม | Approver ขั้นแรกอนุมัติบรรทัดที่ 20; Approver ขั้นสองตัดเพิ่มเป็น 15 ด้วย `approved_message` ของตน | **ลายเซ็นที่สองทับลายเซ็นแรก** คอลัมน์ลายเซ็นต่อบรรทัด (`approved_by_id`, `approved_by_name`, `approved_date_at`, `approved_message`) เป็น last-writer-wins แต่ timeline `history` JSON ต่อบรรทัดจับลายเซ็นทั้งสองตามลำดับ (`{seq:1, name:'stage1', by: bann, qty: 20}`, `{seq:2, name:'stage2', by: ops-mgr, qty: 15}`) Fulfiller กระทำกับ `approved_qty = 15` สุดท้าย Auditor อ่าน chain เต็มจาก `history` |
| APR-EDGE-06 | Approver กระทำแล้วแก้ทันทีใน session เดียวกัน | Approver อนุมัติบรรทัด 1 ที่ 25 จากนั้นรู้ตัวว่าตั้งใจตัด; พยายาม re-set `approved_qty = 20` | **อนุญาตการตัดสินใจใหม่ใน session ก่อน workflow เดิน** ตราบเท่าที่ขั้น workflow ยังไม่เดิน (โดยทั่วไปเพราะบรรทัดอื่นบนขั้นยังไม่ถูก action) Approver อาจอัปเดตลายเซ็นของตนบนบรรทัด; `approved_date_at` อัปเดต, `approved_qty` ล่าสุดคือค่า canonical, JSON `history` บันทึกการเปลี่ยน เมื่อ workflow เดิน (บรรทัดทั้งหมดถูก action submit decision) บรรทัดถูกล็อกที่ขั้นนี้ |
| APR-EDGE-07 | Bulk approve จาก list อนุมัติ | Approver เลือก SR หลายตัวใน list และ bulk-approve (บรรทัดทั้งหมดบนแต่ละ SR ที่ `requested_qty`) | **Bulk approve fan-out ไปสู่การเขียนต่อ SR ต่อบรรทัด** แต่ละ SR ถูกประมวลผลตามลำดับ; กฎต่อบรรทัด (`SR_VAL_010`) apply กับแต่ละ; ความล้มเหลวบน SR หนึ่ง (เช่น `approved_qty > requested_qty` บนบรรทัดที่ mark สำหรับ manual) ไม่หยุดอื่น; Approver เห็นสรุปผลต่อ SR Map ไปยัง TC-SR-060003 (Bulk Action — Export Selected Requisitions; bulk approve เป็นรูปแบบพี่น้อง) |
