# ใบเบิกของสโตร์ (Store Requisition) — Test Scenarios — Requester

> **At a Glance**
> **Persona:** Requester (Outlet Manager ที่ปลายทางบริโภค) &nbsp;·&nbsp; **โมดูล:** [store-requisition](/th/inventory/store-requisition) &nbsp;·&nbsp; **Scenarios:** ~29
> **หมวดหมู่:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** map ไปยัง `tests/701-sr.spec.ts` (TC-SR-010001..010005, 020001..020003, 030001..030004, 040001..040005, 050001..050005) ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ scenario test ที่ persona Requester (**Outlet Manager** ที่ปลายทางบริโภค) ขับเคลื่อนโดยตรงในโมดูล `store-requisition` การมีส่วนร่วมของ Requester เริ่มเมื่อความต้องการสต๊อกถูกระบุที่เอาท์เลต (event ผลิตที่วางแผน, การละเมิด par-level, ใบ event banquet หรือ auto-create ที่ขับโดย recipe จาก `[recipe](/th/inventory/recipe)`) และจบเมื่อ SR ออกจาก requester zone — ผ่าน submit (handoff ไปยัง Approver), withdraw (`SR_AUTH_004` → `cancelled`) หรือบรรทัดทั้งหมดถูก reject โดย approver (`SR_POST_004` tail) Scenarios จัดกลุ่มเป็น **happy path** (เดิน `draft`: create / ป้อนบรรทัด / submit; auto-create จาก recipe; resubmit หลัง send-back), **RBAC** (Requester ไม่ได้กำหนดให้ department, carry-over ของ segregation-of-duties ที่ขั้น approve), **validation** (test เชิงลบกับ `SR_VAL_001`–`SR_VAL_009` ที่ Requester trigger ได้ที่ save หรือ submit) และชุดเล็ก ๆ ของ **edge case** เรื่องโหมด soft vs hard ของความพร้อมต้นทาง บรรทัด cost-dimension split ความแม่นยำทศนิยมบน `requested_qty` และการ submit พร้อมกันบนเอาท์เลตเดียวกัน Handoff ข้าม persona ที่ pivot จาก Requester (Scenarios 1, 2, 3, 4, 11, 12 ในภาพรวมแม่) อยู่ใน [04-test-scenarios.md](./04-test-scenarios.md) ไม่ใช่ที่นี่

## 2. Permission / Authorization

| # | Scenario | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| REQ-PERM-01 | Outlet Manager สร้าง SR ที่ `doc_status = draft` และ submit | **Allow** Requester เป็นเจ้าของ transition `draft → in_progress` ที่แก้ไขได้สำหรับ SR ของ department ของตน `New SR`, ป้อนบรรทัด, upload attachment และ `Submit` ล้วน enable บน SR ที่ยังไม่ submit Approver และ Fulfiller เห็นเอกสารใน `in_progress` หลัง submit แต่ Requester ไม่อาจแก้ได้อีก Map ไปยัง TC-SR-010001 + TC-SR-050001 |
| REQ-PERM-02 | ผู้ใช้ที่ **ไม่ได้** กำหนดให้ department พยายามสร้าง SR ให้ department นั้น | **Deny — RBAC** ผู้ใช้ `requestor@blueledgers.com` ไม่เป็นสมาชิก `Department-Main-Kitchen` การพยายามสร้าง SR ด้วย `department_id = Department-Main-Kitchen` ถูก reject ตอนสร้างด้วย `"You are not authorized to create requisitions for this department."`; ไม่มีแถว `tb_store_requisition` เขียน ตาม `SR_AUTH_001` Map ไปยัง TC-SR-010002 |
| REQ-PERM-03 | Requester แก้ SR `draft` ของตน | **Allow** การแก้ส่วนหัวและบรรทัด upload attachment withdraw — ล้วน enable ขณะ `doc_status = draft` ตาม `SR_AUTH_002` |
| REQ-PERM-04 | Requester พยายามแก้ SR `in_progress` ของตน (หลัง submit) | **Deny — เอกสารออกจากมือ requester แล้ว** Endpoint แก้ reject ด้วย `"SR at status in_progress is not editable by the requester; the document is under approval / fulfilment."` ข้อยกเว้น: ถ้า workflow ได้ route SR กลับมาขั้น requester (send-back จาก approver) การแก้ enable อีกครั้งที่ขั้น requester เท่านั้น ตาม `SR_AUTH_002` ส่วนสอง |
| REQ-PERM-05 | Requester พยายามอนุมัติ SR ของตน | **Deny — SoD** แม้ Requester ถือ Approver role ด้วย `SR_AUTH_011` block action approve บน SR ของตนด้วย `"You raised this requisition; another user must approve it."`; ไม่เขียน `approved_qty`; workflow ไม่เดินต่อ Requester ต้องรอ approver อื่น |
| REQ-PERM-06 | Requester พยายามถอน SR ของตนเกินขั้นอนุมัติแรก | **Deny — workflow เดินต่อ** เมื่อ approver ใดกระทำบนบรรทัดใด (`approved_by_id IS NOT NULL` บนบรรทัดอย่างน้อยหนึ่ง) ปุ่ม **Withdraw** ถูกซ่อน / disable การเรียก API โดยตรงคืน `"Cannot withdraw an SR that has been actioned by an approver; ask the approver to reject if needed."` ตาม `SR_AUTH_004` ส่วนสอง |
| REQ-PERM-07 | Requester พยายามแก้ SR `completed` | **Deny — สถานะจุดสิ้นสุด** การแก้ส่วนหัวและบรรทัดถูก block ข้าม persona ทั้งหมด (ไม่ใช่แค่ Requester) บน SR `completed`; เส้นทางแก้หลัง commit เดียวคือ `[inventory-adjustment](/th/inventory/inventory-adjustment)` Map ไปยัง permission-denial ข้าม persona |

## 3. Validation / Error

| # | Scenario | Trigger | Error ที่คาดหวัง |
| - | -------- | ------- | --------------- |
| REQ-VAL-01 | ขาด `sr_no` ตอนสร้าง | นโยบายเลขของ tenant ล้มเหลวในการกำหนด `sr_no` (ปัญหาระบบ); requester คลิก **Save** | **Reject ตอนสร้าง** — `SR_VAL_001` Server คืน `"SR reference number is required and must be unique."` (ระบบ / Sysadmin ต้องสอบสวน numbering service) |
| REQ-VAL-02 | ขาดสถานที่ต้นทาง / ปลายทางตอน submit | `from_location_id` หรือ `to_location_id` ทิ้งเป็น null หลัง requester รีบกรอก form; คลิก **Submit** | **Reject ตอน submit** — `SR_VAL_002` Server คืน `"Source and destination locations are required and must differ."` error เดียวกันถ้าต้นทาง = ปลายทาง Map ไปยัง TC-SR-050002 |
| REQ-VAL-03 | ประเภทการเคลื่อนย้าย / ประเภทปลายทางไม่ตรง | Requester เลือก `sr_type = 'issue'` แต่ปลายทางเป็น `tb_location.location_type = 'inventory'` (คลัง ไม่ใช่สถานที่บริโภค) | **Reject ตอน save / submit** — `SR_VAL_003` Server คืน `"Movement type 'issue' requires a 'direct' destination; selected destination is 'inventory'."` Requester ต้องเปลี่ยน `sr_type` เป็น `'transfer'` หรือเลือกปลายทางอื่น |
| REQ-VAL-04 | ขาด `sr_date` หรือ `expected_date < sr_date` | Requester ทิ้ง `sr_date` เป็น null หรือป้อน `expected_date = sr_date − 1 day`; คลิก **Save** หรือ **Submit** | **Reject** — `SR_VAL_004` Server คืน `"Requisition date is required; expected date must be on or after requisition date."` Map ไปยัง TC-SR-010004 (ขาด expected delivery date) |
| REQ-VAL-05 | ขาด `requestor_id` / `department_id` ตอน submit | Submit fire โดยไม่บรรจุ requester / department (bug ระบบ — แต่เป็นไปได้ผ่าน API) | **Reject ตอน submit** — `SR_VAL_005` Server คืน `"Requester and requesting department are required and the requester must belong to that department."` |
| REQ-VAL-06 | SR ว่าง (ไม่มีบรรทัด) ตอน submit | Requester ลืมเพิ่มบรรทัด; คลิก **Submit** | **Reject ตอน submit** — `SR_VAL_006` cascade Server คืน `"SR must contain at least one line item before it can be submitted."` Map ไปยัง TC-SR-050003 (line items ว่าง) |
| REQ-VAL-07 | สินค้า inactive / discontinued บนบรรทัด | Requester เลือกสินค้าที่ถูก mark inactive ตั้งแต่นั้น (active ใน catalog cache แต่ inactive ใน master) | **Reject ตอน save line / submit** — `SR_VAL_006` Server คืน `"Product is required and must be active for issue from the source location."` |
| REQ-VAL-08 | product+dimension ซ้ำบน SR เดียวกัน | Requester เพิ่ม `Rice-1kg` ด้วย `dimension = [{cost_centre: 'Banquet'}]` สองครั้ง (แทนที่จะแก้บรรทัดที่มี) | **Reject ตอน save line** — `SR_VAL_007` Server คืน `"This product is already on the SR with the same cost-dimension allocation; edit the existing line or use a different dimension to split."` ตาม unique index `SRT1_*` |
| REQ-VAL-09 | `requested_qty = 0` หรือติดลบบนบรรทัด | Requester ทิ้งปริมาณที่ default 0 หรือพิมพ์ `-25` โดยบังเอิญ | **Reject ตอน submit** — `SR_VAL_008` Server คืน `"Quantities must satisfy 0 ≤ issued_qty ≤ approved_qty ≤ requested_qty; requested quantity must be greater than zero at submit."` Save อาจ warn-only ที่ 0; submit block Map ไปยัง TC-SR-020002 (ปริมาณไม่ถูกต้อง) |
| REQ-VAL-10 | On-hand ต้นทางไม่พอ (โหมด hard) | Tenant ในโหมด hard `SR_VAL_009`; `requested_qty = 100`; on-hand ต้นทาง = 60; available (on-hand หักจอง) = 50 | **Reject ตอน submit** — `SR_VAL_009` hard Server คืน `"Requested quantity 100 exceeds available stock 50 at source location WH-Central."` Requester ลด `requested_qty` เป็น 50 หรือต่ำกว่า หรือตั้ง SR แยกกับต้นทางอื่น Map ไปยัง TC-SR-020003 (สต๊อกไม่พอ) และ TC-SR-030002 (negative inventory) |

## 4. Edge Cases

| # | Scenario | เงื่อนไข | คาดหวัง |
| - | -------- | -------- | ------- |
| REQ-EDGE-01 | Warning ความพร้อมต้นทางโหมด soft | Tenant ในโหมด soft `SR_VAL_009`; `requested_qty = 100`; on-hand ต้นทาง = 60; available = 50 | **Submit อนุญาตพร้อม warning** หน้าจอแสดง soft warning ("Requested quantity exceeds available stock; the Approver may trim and the Fulfiller may need to short-fulfil"); requester เลือก submit ทั้งที่มี Approver เห็น warning ตอนอนุมัติและตัดสินใจตัดหรืออนุมัติตามเดิม Variance คือผลลัพธ์ที่คาดหวังตอน fulfillment |
| REQ-EDGE-02 | Cost-dimension split บนสินค้าเดียวกัน | การป้อนสินค้าเดียวกันสองแถวตาม REQ-HP-03; สองแถวมีค่า JSON `dimension` ต่างกัน | สองแถว persist (unique index `SRT1_*` รวม `dimension`); ไหลอย่างอิสระผ่านการอนุมัติ (แต่ละสามารถ trim / reject แยก) และ fulfillment (แต่ละเขียน inventory transaction แยก); journal entries แยกต้นทุนข้ามสอง cost-centre ตอน commit Edge: product+dimension เหมือนกันเป็น duplicate ตาม REQ-VAL-08 |
| REQ-EDGE-03 | ความแม่นยำทศนิยมบน `requested_qty` | Outlet manager ป้อน `requested_qty = 12.345` (3 ทศนิยม อยู่ใน `Decimal(20, 5)` พอดี) | **ยอมรับที่ storage** ปริมาณถูกเก็บที่ความแม่นยำ 5dp; แสดงผลปัด half-up 3dp ("12.345"); ไหลผ่านการอนุมัติและ issue ไม่เปลี่ยน; `received_base_qty` / `cost_per_unit` ของ inventory transaction ที่ลิงก์ใช้สัญญาความแม่นยำเดียวกัน ตาม `SR_CALC_007` |
| REQ-EDGE-04 | การ submit พร้อมกันบนเอาท์เลตเดียวกัน (สอง requester, สอง SR กับต้นทางเดียวกัน) | เอาท์เลตมีสอง requester (Outlet Manager และรอง); ทั้งคู่ submit SR กับ `WH-Central` พร้อมกัน ทั้งคู่ขอ `Rice-1kg` ปริมาณที่รวมกันเกิน on-hand ต้นทาง | **การ submit ทั้งสองสำเร็จแยกกัน** แต่ละ SR ผ่าน `SR_VAL_009` บน snapshot ของตน (check ต่อ SR ไม่ aggregate ข้าม submit พร้อมกัน) race ถูก resolve ที่อนุมัติ (Approver เห็นทั้งคู่และตัดได้) และที่ fulfillment (`SR_VAL_013` check live ตอน commit จับการ over-commit) SR หนึ่ง commit เต็ม; อีกตัว commit บางส่วนหรือเจอ stock-out Variance บันทึกบนส่วน |
| REQ-EDGE-05 | `description` ยาวและ attachment จำนวนมาก | Requester เขียน `description` 4,000 ตัวอักษรและแนบไฟล์ PDF / รูป 10 ไฟล์ผ่าน `tb_store_requisition_comment.attachments` | **ยอมรับใน Prisma limits** `description` เป็น `VarChar` (ไม่มี cap ความยาว); `attachments` JSONB array มี file ref 10 ตัว (`{originalName, fileToken, contentType}`) Storage backend (S3) ถือไฟล์; SR ถือเฉพาะ token Approver / Fulfiller เห็น attachment ใน activity log |
| REQ-EDGE-06 | Auto-save ระหว่างป้อนบรรทัด | Requester อยู่ระหว่างป้อนบรรทัด; ระบบ auto-save ทุก 60 วินาที (TC-SR-040003); requester ปิด browser โดยไม่ save ระบุ | **Draft persist** Auto-save เขียน line state ที่กำลังทำลง `tb_store_requisition` / `tb_store_requisition_detail` ทุก 60 วินาทีขณะ `doc_status = draft`; เมื่อเปิดใหม่ requester เห็น state auto-save ล่าสุด ไม่มีข้อมูลสูญใน window 60 วินาที; ข้อมูลใน window สูญหายเมื่อ crash hard Map ไปยัง TC-SR-040003 |
