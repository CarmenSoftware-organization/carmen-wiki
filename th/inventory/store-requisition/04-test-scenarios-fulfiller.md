# ใบเบิกของสโตร์ (Store Requisition) — Test Scenarios — Fulfiller

> **At a Glance**
> **Persona:** Fulfiller (Store Keeper / Warehouse Supervisor ที่ต้นทาง) &nbsp;·&nbsp; **โมดูล:** [store-requisition](/th/inventory/store-requisition) &nbsp;·&nbsp; **Scenarios:** ~27
> **หมวดหมู่:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** map ไปยัง `tests/701-sr.spec.ts` (TC-SR-120001..120004+ Issuance) ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ scenario test ที่ persona Fulfiller (**Store Keeper / Warehouse Supervisor** ที่สถานที่ต้นทาง) ขับเคลื่อนโดยตรงในโมดูล `store-requisition` การมีส่วนร่วมของ Fulfiller เริ่มเมื่อ SR transition ไปขั้น fulfillment ของ `in_progress` (หลังการอนุมัติทั้งหมดเสร็จและมีบรรทัดอย่างน้อยหนึ่งที่ `approved_qty > 0`) และจบที่ commit — ผ่าน `in_progress → completed` (posting event ครั้งเดียว) หรือผ่านการขัดจังหวะก่อน commit (ระบบ / SoD ล้มเหลว) ที่ส่ง SR กลับ queue ของ fulfiller Scenarios จัดกลุ่มเป็น **happy path** (fulfillment เต็มตรงกับ `approved_qty`, pick lot เดียว, pick multi-lot บนบรรทัดเดียว, fulfillment บางส่วนตอน stock-out ที่ issue, `sr_type = transfer` จับคู่กับการเพิ่ม on-hand ปลายทาง, `sr_type = issue` พร้อม debit cost-centre ปลายทาง), **RBAC** (SoD Approver ≠ Fulfiller ตาม `SR_AUTH_012`, authorization gate ตามขั้น, สิทธิ์ fulfiller scope ตาม location), **validation** (test เชิงลบกับ `SR_VAL_008` — `issued_qty > approved_qty`, `SR_VAL_011`–`SR_VAL_014` — gate ตอน commit) และ **edge case** เรื่อง commit ในงวดปิด, optimistic concurrency ตอน commit, ความสมบูรณ์ของข้อมูล lot สำหรับสินค้าเน่าเสีย และผลผสมหลายบรรทัด (บางบรรทัด issue เต็ม อื่นขาด อื่นเป็นศูนย์) Handoff ข้าม persona ที่ pivot จาก Fulfiller (Scenarios 1, 2, 6, 8, 9, 10 ในภาพรวมแม่) อยู่ใน [04-test-scenarios.md](./04-test-scenarios.md) ไม่ใช่ที่นี่

## 2. Permission / Authorization

| # | Scenario | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| FUL-PERM-01 | Fulfiller ใน `user_action.execute` ที่ขั้น fulfillment กระทำบน SR ที่ต้นทางของตน | **Allow** การหยิบ ป้อน `issued_qty` เลือก lot เขียน comment ต่อบรรทัด และ **Commit** ล้วน enable ขณะ `doc_status = in_progress` และ `workflow_current_stage = ขั้น fulfillment` ตาม `SR_AUTH_007` |
| FUL-PERM-02 | Fulfiller จากต้นทางอื่นพยายามกระทำ | **Deny — location scope** อำนาจ Fulfiller เป็นต่อ-location: fulfiller ที่ `WH-North` ไม่อาจกระทำบน SR ที่ source จาก `WH-Central` การเรียก API โดยตรง reject ด้วย `"You are not authorized to fulfil requisitions from source location WH-Central."` ตามข้อจำกัด location ของ `SR_AUTH_007` + การ gate ขั้นของ `SR_AUTH_014` (fulfiller ไม่อยู่ใน `user_action.execute`) |
| FUL-PERM-03 | Fulfiller พยายามตั้ง `issued_qty > approved_qty` | **Deny — value cap** ตาม `SR_VAL_008` หน้าจอ block การป้อนฝั่ง client; server reject ด้วย `"Quantities must satisfy 0 ≤ issued_qty ≤ approved_qty ≤ requested_qty."` Fulfiller ต้อง cap ที่ `approved_qty` หรือ escalate ไปยัง Inventory Controller เพื่อแก้การอนุมัติ (ซึ่งต้องการการเปิด SR ใหม่ — โดยทั่วไปไม่อนุญาต; แทนนั้นตั้ง SR ใหม่สำหรับส่วนเกิน) |
| FUL-PERM-04 | Approver พยายาม fulfill SR ที่ตนอนุมัติเหนือ threshold SoD | **Deny — SoD `Approver ≠ Fulfiller`** ตาม `SR_AUTH_012` SR `SR-G` ยอดรวม = ฿15,000 (เหนือ threshold ผ่อนคลาย ฿5,000) ผู้ใช้คนเดียวถือทั้ง Approver และ Fulfiller rights แต่ถูก block ตอน commit ด้วย `"You approved a line on this requisition; another user must issue the goods."` SR อยู่ที่ `in_progress`; รอง fulfiller หรือ shift handover จำเป็น |
| FUL-PERM-05 | Fulfiller พยายามกระทำบน SR `completed` | **Deny — สถานะจุดสิ้นสุด** ตามกฎกลาง `completed` ล็อกข้าม persona Fulfiller อาจตรวจสอบ lot และปริมาณเชิงประวัติศาสตร์สำหรับ audit แต่ re-commit หรือแก้ไม่ได้ การแก้หลัง commit ผ่าน `[inventory-adjustment](/th/inventory/inventory-adjustment)` |
| FUL-PERM-06 | Fulfiller พยายามข้ามการเลือก lot สำหรับสินค้าควบคุม lot | **Deny ตอน commit** ตาม `SR_VAL_012` ถ้าสินค้าควบคุม lot หรือเน่าเสียและ `tb_inventory_transaction_detail` ที่ลิงก์ไม่มี `lot_no` (หรือไม่มี `expiry_date` สำหรับเน่าเสีย) commit reject ด้วย `"Issue posting requires lot information for lot-controlled items and a valid unit cost on every issued line; line <seq> is missing data on the linked inventory transaction."` Fulfiller ต้องกรอก lot sub-form ให้ครบก่อน re-attempt commit |

## 3. Validation / Error

| # | Scenario | Trigger | Error ที่คาดหวัง |
| - | -------- | ------- | --------------- |
| FUL-VAL-01 | `issued_qty > approved_qty` บนบรรทัด | Fulfiller ป้อน `issued_qty = 12` บนบรรทัดที่ `approved_qty = 10` | **Reject** — `SR_VAL_008` Server คืน `"Quantities must satisfy 0 ≤ issued_qty ≤ approved_qty ≤ requested_qty."` สาเหตุพบบ่อย: typo / ข้อผิดพลาดการป้อนข้อมูล |
| FUL-VAL-02 | `issued_qty < 0` บนบรรทัด | Fulfiller ป้อนปริมาณติดลบโดยบังเอิญ | **Reject** — `SR_VAL_008` Server คืน invariant error เดียวกัน |
| FUL-VAL-03 | Commit โดยไม่มีบรรทัดที่อนุมัติ | บรรทัดทั้งหมดมี `approved_qty = 0` (เช่น approver ทุกคน reject แต่ auto-cancel ไม่ fire — bug ระบบหรือ race) | **Reject ตอน commit** — `SR_VAL_011` Server คืน `"SR must contain at least one approved line (approved quantity > 0) before it can be committed."` Inventory Controller สอบสวน SoR; โดยทั่วไป SR ควรเป็น `cancelled` แล้ว |
| FUL-VAL-04 | ขาดข้อมูล lot บน inventory transaction ที่ลิงก์ (สินค้าควบคุม lot) | Fulfiller commit โดยไม่บรรจุ lot sub-form บนสินค้าควบคุม lot (ข้าม เลื่อน หรือลบโดยบังเอิญ) | **Reject ตอน commit** — `SR_VAL_012` Server คืน `"Issue posting requires lot information for lot-controlled items and a valid unit cost on every issued line; line <seq> is missing data on the linked inventory transaction."` Fulfiller เปิด lot sub-form, เลือก lot, re-commit |
| FUL-VAL-05 | On-hand ต้นทางต่ำกว่า `issued_qty` ตอน commit (check live) | check `SR_VAL_013` live: SR อื่น consume on-hand ต้นทางระหว่างการเลือก lot ของ Fulfiller กับ commit; on-hand ตอนนี้ < `issued_qty` | **Reject ตอน commit** — `SR_VAL_013` Server คืน `"Source stock-out at issue: line <seq> requires <issued_qty> but only <on_hand> is available at <from_location_name>. Reduce issued_qty to the available quantity or cancel the line."` Fulfiller ลด `issued_qty` เป็น on-hand live เขียน system comment re-commit (รูปแบบ partial-fulfilment FUL-HP-04) |
| FUL-VAL-06 | การพยายาม commit ในงวดปิด | วันที่ post สำหรับ commit (โดยทั่วไป `now()`) อยู่ในงวดบัญชีปิด | **Reject ตอน commit** — `SR_VAL_014` Server คืน `"Cannot commit SR <sr_no>: posting date falls in a closed accounting period."` Fulfiller escalate ไป Finance เพื่อเปิดงวดหรือไปยัง Inventory Controller เพื่อ void SR; SR อยู่ที่ `in_progress` |
| FUL-VAL-07 | Optimistic-concurrency ไม่ตรงตอน commit | ผู้ใช้อื่น (fulfiller พร้อมกันในแท็บอื่น หรือระบบ background) ได้อัปเดต `doc_version` บน SR ระหว่างอ่านกับ commit | **Reject ตอน commit** — optimistic-concurrency ทั่วไป Server คืน `"This SR was modified by another user. Please refresh and re-apply your changes."` Fulfiller refresh, ป้อน state ต่อบรรทัดที่ไม่ได้ save อีกครั้ง, re-commit |

## 4. Edge Cases

| # | Scenario | เงื่อนไข | คาดหวัง |
| - | -------- | -------- | ------- |
| FUL-EDGE-01 | ความแม่นยำทศนิยมบน `issued_qty` และ multi-lot split | สินค้าควบคุม lot, `approved_qty = 5.500` (3-dp); Fulfiller split ข้ามสอง lot: `Lot-A = 2.250`, `Lot-B = 3.250`; conversion factor `1.000000` (base UoM) | **ยอมรับที่ storage** ปริมาณต่อ lot เก็บที่ความแม่นยำ 5dp ตาม `Decimal(20, 5)`; ผลรวมต้องเท่ากับ `issued_qty = 5.500` พอดี (lot-sub-form validate ทั้งฝั่ง client และ server); ไม่มี rounding drift เข้าสู่ยอดรวม Display ปัด half-up 3dp ตาม `SR_CALC_007` |
| FUL-EDGE-02 | ข้อมูล lot มีบางส่วน (เลือก lot แต่ไม่มี expiry สำหรับเน่าเสีย) | Fulfiller เลือก lot แต่ `expiry_date` ของ lot เป็น null ในข้อมูลต้นทาง (data-quality issue ตอนรับ) | **Reject ตอน commit** — `SR_VAL_012` Check มอง `expiry_date` ไม่เป็น null สำหรับสินค้าเน่าเสีย Fulfiller escalate ไป Inventory Controller; ปัญหาอยู่ที่ source-receipt ไม่ใช่ที่ SR Fix ก่อน commit: Inventory Controller post inventory-adjustment เพื่อบรรจุ expiry ของ lot; จากนั้น re-attempt commit SR |
| FUL-EDGE-03 | Event Stock-OUT ใน sub-ledger inventory-transaction reconciliation | หลัง commit query sub-ledger inventory `Σ outQty for inventory_doc_type = store_requisition, document_id = SR-A.id` ถูก reconcile กับ GL credit ของบัญชี inventory ต้นทาง | **Match ภายใน tolerance** การ reconcile เป็นความรับผิดชอบของ Finance / Inventory Controller (persona Audit / Config); บทบาท Fulfiller คือการป้อน `issued_qty` ถูกต้องและการเลือก lot ตาม `SR_XMOD_003` (`enum_inventory_doc_type = store_requisition`) |
| FUL-EDGE-04 | Commit trigger การ consume cost-layer FIFO จากหลาย lot | ต้นทางใช้ costing FIFO; บรรทัดมี `issued_qty = 10`, layer เก่าสุดเหลือ 4 หน่วย, layer ถัดไปเหลือ 30 | **สองแถวการ consume cost-layer** โมดูล costing (`[costing](/th/inventory/costing)`) consume 4 หน่วยจาก layer เก่าสุดที่ `cost_per_unit` ของมัน จากนั้น 6 หน่วยจาก layer ถัดไปที่ `cost_per_unit` ของมัน; `unit_cost` ที่แสดงบนบรรทัดคือ blend ถ่วงน้ำหนัก; GL entry ใช้ยอดรวม blend; ตาม `SR_XMOD_004` |
| FUL-EDGE-05 | การเลือก lot ก่อน commit save เป็น draft state | Fulfiller ได้ป้อน `issued_qty` และเลือก lot สำหรับ 2 ใน 3 บรรทัด จากนั้นกะของตนจบ | **การเลือก lot persist** State ต่อบรรทัดก่อน commit (issued qty + การเลือก lot) ถูก save ตำแหน่งบน `tb_store_requisition_detail` และแถว `tb_inventory_transaction_detail` ที่ staged (inventory transaction อาจถูกสร้างใน state `staged` / non-committed จากนั้น finalise ที่ commit) Fulfiller ถัดไป (รอง / กะถัดไป) เปิด SR เห็น state ที่กำลังทำ ทำบรรทัดที่เหลือเสร็จ และ commit |
| FUL-EDGE-06 | SR แบบ transfer ที่สถานที่ปลายทาง inactive ปัจจุบัน | `sr_type = transfer`; `tb_location` ปลายทางถูก mark `is_active = false` (การปิดเชิงบริหาร) ระหว่างอนุมัติกับ commit | **Reject ตอน commit** — `SR_VAL_002` (สถานที่ต้องอ้างอิงแถว active) Server คืน `"Destination location is no longer active; cannot transfer."` Fulfiller escalate ไป Inventory Controller เพื่อ reactivate location หรือ void SR |
| FUL-EDGE-07 | การแปลง UoM ตอน issue (issue ใน UoM ต่างจากอนุมัติ) | สินค้ามีสอง UoM (`Bottle` และ `Case = 12 Bottle`); อนุมัติใน `Case = 5`; Fulfiller หยิบใน `Bottle` (60 ขวดรวม = 5 case) | **การแปลงจัดการที่ชั้น inventory-transaction** ตาม `SR_VAL_008` value cap อยู่ใน UoM ของบรรทัด (โดยทั่วไป UoM ที่ขอ); การแปลงเป็น base UoM เกิดบน inventory transaction (`received_unit_conversion_factor` บน `tb_inventory_transaction_detail`) Fulfiller ป้อนค่าเทียบเท่าใน base UoM; `issued_qty` ของบรรทัด SR สะท้อนค่า requested-UoM การ consume cost-layer อยู่ใน base UoM |
| FUL-EDGE-08 | Batch commit (หลาย SR ใน transaction window เดียว) | บทบาทเทียบเท่า Inventory Manager (Fulfiller ที่มีสิทธิ์สูง) commit หลาย SR `in_progress` ที่ปลายกะ | **การประเมินต่อ SR ใน batch** SR แต่ละตัวถูกประเมินอย่างอิสระต่อ `SR_VAL_011`–`SR_VAL_014`; ความล้มเหลวบน SR หนึ่ง (เช่น block งวดปิด) ทิ้ง SR นั้นที่ `in_progress`; SR ที่สำเร็จทั้งหมด commit ใน transaction window เดียวกัน; activity log บันทึกผลต่อ SR หมายเหตุ: รูปแบบ batch ของโมดูล SR พบน้อยกว่าของ GRN; tenant config ตัดสินใจ |
