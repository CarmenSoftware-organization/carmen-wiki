# ใบเบิกของสโตร์ (Store Requisition) — Test Scenarios — Receiver

> **At a Glance**
> **Persona:** Receiver (ผู้แทนเอาท์เลตปลายทาง) &nbsp;·&nbsp; **โมดูล:** [store-requisition](/th/inventory/store-requisition) &nbsp;·&nbsp; **Scenarios:** ~20
> **หมวดหมู่:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** manual / planned ณ เวลานี้; automated coverage ที่ใกล้ที่สุดอยู่ implicit ใน `tests/701-sr.spec.ts` TC-SR-120001..120004 (Issuance) ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ scenario test ที่ persona Receiver (**ผู้แทนเอาท์เลตปลายทาง** ที่เอาท์เลตบริโภคหรือคลังปลายทาง) ขับเคลื่อนโดยตรงในโมดูล `store-requisition` การมีส่วนร่วมของ Receiver เริ่มเมื่อ SR transition เป็น `doc_status = completed` (Fulfiller ได้ commit issue ที่ต้นทางและ inventory transactions ที่ลิงก์ถูกเขียน) และจบเมื่อการรับถูกยืนยันเต็มหรือความคลาดเคลื่อนได้รับการ resolve โดย Inventory Controller ผ่าน `[inventory-adjustment](/th/inventory/inventory-adjustment)` Receiver **ไม่มี** อำนาจ commit บน SR — SR เป็นจุดสิ้นสุดที่ `completed` แล้วเมื่อรับเอกสาร; บทบาทคือ **การยืนยัน การยอมรับ และการส่งสัญญาณความคลาดเคลื่อน** Scenarios จัดกลุ่มเป็น **happy path** (การยืนยันตรงเต็ม, ตรงเต็มหลายบรรทัด, การยืนยัน lot กับ inventory transaction ที่ลิงก์, การยืนยันปลายทาง paired-GRN สำหรับการโอนระหว่างคลัง), **RBAC** (Receiver ที่ปลายทางเท่านั้น, ไม่อาจแก้ SR, ไม่อาจ post adjustment โดยตรง), **validation** (test เชิงลบเรื่องความสมบูรณ์ของการรายงานความคลาดเคลื่อน) และ **edge case** เรื่อง short receipt, over receipt, การส่ง lot ผิด, สินค้าเสีย, มาช้าเทียบกับ `expected_date` และเส้นทาง missing-on-arrival Handoff ข้าม persona ที่ pivot จาก Receiver (Scenario 7 ในภาพรวมแม่ — flag ความคลาดเคลื่อนไปยัง Inventory Controller) อยู่ใน [04-test-scenarios.md](./04-test-scenarios.md) ไม่ใช่ที่นี่

## 2. Permission / Authorization

| # | Scenario | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| RCV-PERM-01 | Receiver ที่เอาท์เลตปลายทาง append comment ยืนยัน | **Allow** การเขียน `user` comment บนส่วนหัว SR หรือบรรทัด (ผ่าน `tb_store_requisition_comment` / `tb_store_requisition_detail_comment`) enable สำหรับผู้ใช้ที่มีสิทธิ์อ่าน `to_location_id` ของ SR บทบาท Receiver ไม่ต้องการสิทธิ์เขียนคอลัมน์ `tb_store_requisition_detail` ของ SR — เฉพาะตาราง comment |
| RCV-PERM-02 | Receiver ที่เอาท์เลตอื่นพยายามยืนยัน | **Deny — destination scope** Receiver ที่ `WH-North` ไม่อาจยืนยัน SR ที่ `to_location_id = Main-Kitchen` SR detail read-only สำหรับผู้ใช้ที่ไม่ใช่ปลายทางตาม default (RBAC); action **Add Comment** ถูกซ่อน / disable การเรียก API โดยตรง reject ด้วย `"You are not authorized to acknowledge requisitions delivered to Main-Kitchen."` |
| RCV-PERM-03 | Receiver พยายามเปลี่ยน `doc_status` หรือแก้ `issued_qty` | **Deny — Receiver scope** Receiver ไม่มีอำนาจเหนือ `doc_status` หรือคอลัมน์ใดบน `tb_store_requisition_detail` Detail view read-only บนฟิลด์เหล่านั้น; พื้นผิวการเขียนเดียวคือตาราง comment ตาม `SR_AUTH_008` การแก้ข้อผิดพลาด `issued_qty` Receiver flag ความคลาดเคลื่อนและ Inventory Controller post adjustment ผ่าน `[inventory-adjustment](/th/inventory/inventory-adjustment)` |
| RCV-PERM-04 | Receiver พยายาม void หรือ cancel SR `completed` | **Deny — สถานะจุดสิ้นสุด + Receiver scope** `completed` ล็อกข้าม persona (ไม่มี transition `completed → voided`); Receiver โดยเฉพาะไม่มีอำนาจ void ที่สถานะใด การแก้หลัง commit ใช้ inventory-adjustment |
| RCV-PERM-05 | Receiver พยายาม post inventory adjustment โดยตรง | **Deny — สิทธิ์ adjustment** การ post inventory-adjustment เป็นสิทธิ์ของ Inventory Controller (persona Audit / Config) ไม่ใช่ Receiver Receiver flag ความคลาดเคลื่อนผ่าน comment; Inventory Controller ตัดสินใจ adjustment ตาม `SR_XMOD_009` (การแก้หลัง commit ไหลผ่าน inventory-adjustment ไม่ใช่ผ่าน action ของ Receiver) |

## 3. Validation / Error

| # | Scenario | Trigger | Error ที่คาดหวัง |
| - | -------- | ------- | --------------- |
| RCV-VAL-01 | Receiver flag ความคลาดเคลื่อนด้วย comment ว่าง | Receiver คลิก **Flag Discrepancy** และ submit โดยไม่ป้อนข้อความ | **Reject** — body comment บังคับสำหรับ flag ความคลาดเคลื่อน Server คืน `"Discrepancy comment requires a description of the issue (e.g. received quantity vs expected, lot mismatch, damage)."` |
| RCV-VAL-02 | Receiver flag ความคลาดเคลื่อนโดย received เท่ากับ `issued_qty` | Receiver เปิด form ความคลาดเคลื่อนแต่ป้อน `received = issued_qty` (ไม่มี gap จริง) | **Warn — ไม่ตรวจพบความคลาดเคลื่อน** UI แสดง soft warning ("Received quantity matches issued quantity — is there still a discrepancy to flag? E.g. wrong lot, damaged goods?") ผู้ใช้ยืนยันหรือแก้ Comment ยังอาจ save ได้ (เช่น flag lot mismatch โดยไม่มี gap ปริมาณ) แต่ระบบ flag การป้อนเป็นความคลาดเคลื่อนไม่เชิงปริมาณ |
| RCV-VAL-03 | Receiver พยายาม flag ความคลาดเคลื่อนบน SR ที่ไม่ใช่ `completed` | SR อยู่ที่ `doc_status = in_progress` (ยังที่ขั้น fulfillment); Receiver พยายาม flag ความคลาดเคลื่อน | **Reject — เร็วเกินไป** Server คืน `"Discrepancy flags are available only after the SR has been committed (doc_status = completed)."` Receiver รอ Fulfiller commit; ถ้า SR ค้างที่ `in_progress` เกิน SLA Inventory Controller แทรกแซง |

## 4. Edge Cases

| # | Scenario | เงื่อนไข | คาดหวัง |
| - | -------- | -------- | ------- |
| RCV-EDGE-01 | รับขาดที่ปลายทาง (`received < issued_qty`) | SR `SR-E` `completed` ด้วย `issued_qty = 10` บนบรรทัด 1; การส่งจริง = 7 (ขาด 3 — อาจแตกในขนส่ง นับผิดที่ต้นทาง หรือสูญหายในขนส่ง) | Receiver flag ความคลาดเคลื่อนบนบรรทัด 1 ผ่าน `tb_store_requisition_detail_comment` ด้วย: `received_qty = 7`, `gap = 3`, สาเหตุที่สงสัย, รูปกล่องแตกที่แนบ SR ยังคงเป็น `completed` Inventory Controller ถูกแจ้ง (event ระบบขึ้นกับ config); สอบสวนการนับฝั่งต้นทาง vs ปลายทาง; post inventory-adjustment เพื่อ reconcile on-hand ปลายทาง (ปลายทางถูกเพิ่ม 10 แล้วตอน commit ฝั่งต้นทางสำหรับ `transfer` หรือคิด 10 เป็น expense สำหรับ `issue` — adjustment แก้โดย -3) |
| RCV-EDGE-02 | รับเกินที่ปลายทาง (`received > issued_qty`) | การส่งจริง = 12 เมื่อ SR แสดง `issued_qty = 10` | Receiver flag ความคลาดเคลื่อนรับเกิน Inventory Controller สอบสวน: (a) records ต้นทางผิด (ต้นทางส่ง 12 แต่บันทึก 10 — adjustment แก้เพิ่ม on-hand ปลายทาง 2 และลด on-hand ต้นทาง 2 ให้ตรงจริง) หรือ (b) การส่งจริงรวมส่วนเกิน pilferage-bound (พบน้อย) SR ยังคงเป็น `completed`; reconciliation ผ่าน adjustment |
| RCV-EDGE-03 | ได้รับ lot ผิด | `tb_inventory_transaction_detail.lot_no = "Lot-A"` ที่ลิงก์ของ SR; สินค้าจริงถือ label `Lot-B` | Receiver flag ความคลาดเคลื่อน lot ผิด สำหรับสินค้าไม่เน่าเสียมัก cosmetic (สินค้าถูก lot บันทึกผิด); สำหรับสินค้าเน่าเสียสำคัญเพราะการติดตาม expiry แตก Inventory Controller ตัดสินใจ: (a) comment แก้ lot ถ้า lot จริงอยู่ใน records ต้นทางแต่ถูกบันทึกผิดบน SR นี้ (ไม่มี adjustment on-hand แค่แก้ lot-trace) หรือ (b) inventory-adjustment เต็มถ้า lot จริงมาจาก cost layer ต่าง (การแก้ cost-of-goods) |
| RCV-EDGE-04 | สินค้าเสียตอนมาถึง | การส่งจริงมาถึงเสียในการขนส่ง (กล่องบุบ container รั่ว) ปริมาณถูกแต่สภาพใช้ไม่ได้ | Receiver บันทึกความเสียหายพร้อมรูปและปริมาณที่กระทบ สองเส้นทาง: (a) **ยอมรับและ write off** — Receiver ยอมรับสินค้าที่ปลายทาง Inventory Controller post inventory-adjustment พร้อมเหตุผล `damaged-on-arrival` ลด on-hand ปลายทางตามปริมาณที่เสีย; (b) **ปฏิเสธส่วนที่เสีย** — Receiver ปฏิเสธจริงที่ dock Inventory Controller ประสานกับต้นทาง / ผู้ขาย / ประกันสำหรับ logistics ส่งคืน SR ยังคงเป็น `completed` ทั้งสองทาง; การจัดการความเสียหายอยู่ในกระบวนการ adjustment / return |
| RCV-EDGE-05 | มาถึงช้าเทียบกับ `expected_date` | SR `SR-F` `expected_date = 2026-05-12`; การส่งจริงมาถึง 2026-05-14 (ช้า 2 วัน) การผลิตที่เอาท์เลตเลื่อน | Receiver append comment บันทึกความช้า; Inventory Controller อาจติดตามรูปแบบช้าเรื้อรังสำหรับ review supply-chain (คลังใด สินค้าใด route ใดที่มาถึงช้าสม่ำเสมอ) ไม่มีการเปลี่ยนสถานะบน SR; ไม่ต้องการ adjustment เว้นแต่มีปัญหาปริมาณ / คุณภาพด้วย |
| RCV-EDGE-06 | สินค้าไม่มาถึงเลย (missing-on-arrival) | SR `SR-G` `completed` 3 วันที่แล้ว; สินค้าไม่เคยมาถึงปลายทาง; records ต้นทางแสดงส่งแล้ว, records transit แสดงกำลังดำเนินการแต่ไม่มีการส่ง | Receiver flag `received_qty = 0` เทียบกับ `issued_qty` ไม่เป็นศูนย์ — ความคลาดเคลื่อนวิกฤต Inventory Controller เริ่มการค้นหา (records transit, การ check ปลายทางสำรอง, security review); ถ้ากู้คืนไม่ได้ post compensating loss adjustment ที่ปลายทางผ่าน `[inventory-adjustment](/th/inventory/inventory-adjustment)` (ต้นทางถูก debit แล้วตอน commit; สินค้าที่หายเป็น event loss write off ไปบัญชี loss / shrinkage ที่เหมาะสมโดย Finance) |
| RCV-EDGE-07 | Receiver = Requester (เอาท์เลตเล็ก คนเดียวสวมหมวกสอง) | Outlet Manager `purchase@blueledgers.com` เป็นทั้ง Requester และ Receiver สำหรับ `SR-H` ตั้ง SR, Approver และ Fulfiller กระทำ, SR commit และผู้ใช้คนเดียวกันยืนยันการรับ | **ไม่มี SoD บังคับใช้ระหว่าง Requester กับ Receiver** ระบบ **ไม่** block การยืนยันตัวเอง (SoD อยู่ระหว่าง Requester / Approver และ Approver / Fulfiller — ไม่ใช่ Requester / Receiver) Audit trail แสดง `created_by_id` เดียวกันบน draft เริ่มและบน comment ยืนยัน Tenant ที่กังวลเรื่อง overlap อาจตั้งค่าขั้น workflow ที่ต้องการผู้ใช้อื่นยืนยันได้ แต่ไม่ใช่ default |
| RCV-EDGE-08 | การรับปริมาณสูงปลายกะ | Receiver มี 20 SR ที่ต้องยืนยันปลายกะ | **การยืนยัน bulk (ถ้ารองรับ)** Tenant config อาจให้ action bulk-acknowledge ที่ append comment template เดียวกันข้ามหลาย SR; มิเช่นนั้น Receiver ทำทีละตัว ทั้งสองทาง audit จับแต่ละการยืนยันแยก Flag ความคลาดเคลื่อนต้องเป็นต่อ SR เสมอ (ไม่มี bulk discrepancy) |
