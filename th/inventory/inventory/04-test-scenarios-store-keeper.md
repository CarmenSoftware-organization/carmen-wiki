# คลังสินค้า (Inventory) — Test Scenarios — Store Keeper

> **At a Glance**
> **Persona:** Store Keeper &nbsp;·&nbsp; **โมดูล:** [[inventory]] &nbsp;·&nbsp; **Scenarios:** ~29
> **หมวด:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** map ไปยัง `501-grn.spec.ts`, `720-stock-issue.spec.ts` ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ test scenarios ที่ Store Keeper persona ขับโดยตรงในโมดูล `inventory` Store Keeper เริ่ม adjustments non-GRN / non-SR (manual stock-in สำหรับ found stock / count overage / vendor-replacement, manual stock-out สำหรับ breakage / expiry / count shortage), ดำเนินการ physical และ spot counts ที่ระดับ location และทำงานภายใน auto-approve threshold สำหรับเอกสารต่ำกว่า cost-impact ความเป็นเจ้าของโมดูล inventory ของพวกเขาเริ่มเมื่อระบุความคลาดเคลื่อนที่ floor และจบเมื่อเอกสาร source (`tb_stock_in` / `tb_stock_out` / `tb_count_stock`) อย่างใดอย่างหนึ่ง auto-post (ต่ำกว่า threshold) หรือส่งต่อให้ Inventory Controller เพื่ออนุมัติ Scenarios จัดกลุ่มเป็น **happy paths** (stock-in สำหรับ found stock, stock-out สำหรับ breakage, multi-line count-derived rollup, FOC / no-cost stock-in สำหรับ vendor-supplied free replacement, perishable / lot-tracked entry), **RBAC** (Store Keeper scope vs Inventory Controller scope, segregation of duties on receive-then-write-off), **validation** (negative tests against `INV_VAL_001`–`INV_VAL_013` ที่ Store Keeper สามารถ trigger ที่ submit time) และ **edge cases** รอบขอบเขต tolerance, decimal precision บน lot qty, posts concurrent, gates location-type และพฤติกรรม count-rollup การส่งต่อข้าม persona ที่ pivot off Store Keeper (Scenarios 1, 2, 4, 7, 9, 10 ในภาพรวม parent) อาศัยอยู่ใน [04-test-scenarios.md](./04-test-scenarios.md) ไม่ใช่ที่นี่

## 2. Permission / Authorization

| # | Scenario | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| SK-PERM-01 | Store Keeper raise stock-in / stock-out ต่ำกว่า threshold | **Allow** ตาม `INV_AUTH_001` / `INV_AUTH_002` Store Keeper ถือสิทธิ์ create บนเอกสาร `tb_stock_in` / `tb_stock_out` ภายใน scope `tb_user_location` ต่ำกว่า auto-approve threshold, submit auto-advance `draft → completed` และ post ไปยัง inventory ตรง ๆ |
| SK-PERM-02 | Store Keeper พยายามอนุมัติ stock-out pending ของ Store Keeper คนอื่น | **Deny — ต้องการ Inventory Controller** ตาม `INV_AUTH_003` เฉพาะ Inventory Controller (หรือสูงกว่า) สามารถ fire transition `in_progress → completed` บนเอกสารเหนือ auto-approve threshold; ต่ำกว่า threshold ระบบ auto-fire Store Keeper พยายามอนุมัติเอกสาร `in_progress` ของ Store Keeper คนอื่นผ่าน API return `"Approval of above-threshold inventory adjustments requires the Inventory Controller role."` เอกสารอยู่ `in_progress`; ไม่มี inventory transaction |
| SK-PERM-03 | Store Keeper พยายามแก้ไข inventory transaction ที่ completed (`posted`) | **Deny — terminal state** ตาม `INV_POST_012` ทุก `tb_inventory_transaction` คือ immutable บน write; การกลับรายการคือ compensating transaction ใหม่ ไม่ใช่ edit `PATCH /inventory-transaction/{id}` โดยตรงจาก role ใด return `"Inventory transactions are immutable; reverse via a compensating stock-in / stock-out."` Store Keeper raise compensating stock-in / stock-out แทน |
| SK-PERM-04 | Store Keeper พยายาม post ไปยัง location นอก scope `tb_user_location` | **Deny — scope** Location picker filter ไปยัง scope ของ user; การเรียก API ตรงด้วย `location_id` นอก scope return `"You do not have permission to post to location <LOC-X>."` `INV_VAL_003` run ที่ submit แต่ scope check run ที่ API gateway ก่อนนั้น — ทั้งสอง layer block |
| SK-PERM-05 | Store Keeper พยายาม post ไปยัง location `direct`-cost หรือ `inactive` | **Deny ตาม `INV_VAL_009` / `INV_VAL_003`** Direct-cost locations ไม่ถือ balance; stock-in / stock-out ไม่มีความหมายที่นั่นและ reject Locations inactive (soft-deleted หรือ `is_active = false`) filter จาก picker; การ submit API ตรง return error validation ที่สอดคล้อง |
| SK-PERM-06 | Segregation of duties — write-off ของสต๊อกที่ Store Keeper รับเองแต่แรก | **Deny เมื่อเหนือ SoD threshold** ตาม `INV_AUTH_010` เมื่อ Store Keeper พยายาม write off lot ที่ตนเองรับ (GRN `created_by_id` ดั้งเดิมตรงกับ user ของ stock-out) และผลกระทบต้นทุนเกิน tenant SoD threshold submit reject ด้วย `"You created the receipt for this lot; an independent adjuster must initiate the write-off (SoD INV_AUTH_010)."` Write-offs เล็กปกติยกเว้น Controller (หรือ Store Keeper คนอื่น) เริ่มแทน |
| SK-PERM-07 | Store Keeper พยายาม post เข้างวด `closed` หรือ `locked` | **Deny ตาม `INV_VAL_008`** Server return `"Cannot post into period <YYMM>: period is <status>. Re-open the period (closed only) or post a current-period restatement (locked)."` เอกสารอยู่ `draft`; Store Keeper เปลี่ยนวันที่ transaction ไปยังงวด open ปัจจุบันหรือ escalate ไปยัง Finance Manager เพื่อ re-open งวด closed งวด locked re-open ไม่ได้ |

## 3. Validation / Error

| # | Scenario | Trigger | Error ที่คาดหวัง |
| - | -------- | ------- | -------------- |
| SK-VAL-01 | Missing สินค้าหรือ location ที่ submit (`INV_VAL_002`) | Store Keeper เพิ่ม line stock-in ด้วย `product_id = null` (เคลียร์โดย mis-click) หรือ `location_id = null` (scope picker ไม่เลือก); คลิก Submit | **Reject ที่ submit** — `INV_VAL_002` Server return `"Product is required on every inventory transaction detail line; location is required for non-rollup movements."` Atomic — เอกสารทั้งหมด reject ไม่ใช่แค่ line Store Keeper pick field ที่หายไปและ re-submit |

| SK-VAL-04 | ความพยายาม balance ติดลบ (`INV_VAL_005`) | Outbound stock-out สำหรับ `P-1` ที่ `LOC-A`; qty available ที่ `LOT-1` คือ 5; Store Keeper ป้อน `qty = 7` | **Reject ที่ submit** — `INV_VAL_005` Server return `"Outbound movement would drive on-hand at (LOC-A, P-1, LOT-1) below zero. Available: 5, requested: 7."` เอกสารอยู่ที่ `draft`; Store Keeper ลด qty เป็น 5 (หรือเป็น multi-lot split — 5 จาก `LOT-1`, ส่วนที่เหลือจาก `LOT-2`) หรือ escalate ไปยัง Controller เพื่อ investigate ถ้าความคลาดเคลื่อนแนะนำ inbound ที่หายไป Map ไปยัง parent Scenario 10 |
| SK-VAL-05 | Existing-lot collision (`INV_VAL_006`) | Store Keeper ป้อน `lot_no` claim "new lot" แต่ lot มีอยู่แล้วสำหรับ `(product_id, location_id)` เดียวกัน; หรือป้อน outbound จาก `lot_no` ที่ไม่มีอยู่ที่ location | **Reject ที่ submit** — `INV_VAL_006` สำหรับ inbound new-lot collision: `"Lot <lot_no> already exists for product <product> at location <location>; lot identity must be unique."` สำหรับ outbound missing-lot: `"Lot <lot_no> at location <location> is not available for consumption."` |
| SK-VAL-06 | Cost ลบ (`INV_VAL_007`) | การเรียก API ตรงด้วย `cost_per_unit = -฿5.00` บน stock-in (ปกติเป็น attempt bypass validation ฝั่ง client) | **Reject ที่ submit** — `INV_VAL_007` Server return `"Cost per unit and qty must be non-negative; in_qty and out_qty are mutually exclusive on a single cost-layer row."` เอกสารอยู่ `draft` |
| SK-VAL-07 | Post ย้อนหลังเข้างวดที่ปิด (`INV_VAL_008`) | Store Keeper ตั้งวันที่ transaction ของ stock-in ไปยังวันที่ภายในงวดที่ปิดแล้ว (เช่นลืม post เมื่อวานนี้; งวดปิดข้ามคืน) | **Reject ที่ submit** — `INV_VAL_008` Server return `"Cannot post into period <YYMM>: period is closed. Re-open the period or post a current-period restatement (locked)."` Store Keeper reset วันที่เป็นวันนี้หรือ escalate ไปยัง Finance Manager Map ไปยัง parent Scenario 11 |
| SK-VAL-08 | Stock-in ไปยัง direct-cost location (`INV_VAL_009`) | Location picker filter เป็น inventory-type แต่ direct API call ด้วย `location_id` อ้างอิง direct-cost location พยายาม post stock-in | **Reject ที่ submit** — `INV_VAL_009` Server return `"Cannot post outbound movement to direct-cost location <location>. Direct locations bypass inventory; consumption is implicit at receipt."` (Symmetrically reject สำหรับ stock-out; direct locations ไม่ถือ balance ที่จะปรับ) Store Keeper pick valid inventory-type location |
| SK-VAL-09 | Consignment-to-inventory implicit transfer reject (`INV_VAL_010`) | สต๊อกของ consignment location ถูกนำเข้าความเป็นเจ้าของโดยไม่มีเอกสาร re-classification อย่างชัดเจน; direct API call พยายามเขียน transfer จาก consignment ไปยัง inventory | **Reject ตาม `INV_VAL_010`** — server return `"Consignment-to-inventory ownership transfer requires an explicit re-classification stock-in document."` Store Keeper raise stock-in อย่างชัดเจนสำหรับฝั่ง inventory และ (ภายใต้สิทธิ์ elevated) stock-out จากฝั่ง consignment |

## 4. Edge Cases

| # | Scenario | Condition | Expected |
| - | -------- | --------- | -------- |
| SK-EDGE-01 | Exact tolerance boundary บน threshold | Auto-approve threshold = `฿500.00`; Store Keeper submit stock-in ด้วย `qty × cost_per_unit = ฿500.00` พอดี | **Boundary inclusive — auto-approve** ระบบถือว่า `≤` เป็นเงื่อนไข auto-approve เอกสาร advance `draft → completed` โดยไม่มีการส่งต่อ Controller `฿500.01` จะเกิน threshold และ route ไปยัง Controller |
