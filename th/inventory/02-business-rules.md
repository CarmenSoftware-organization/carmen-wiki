# คลังสินค้า — กฎทางธุรกิจ (Inventory — Business Rules)

> **At a Glance**
> **ตระกูลกฎ:** `INV_VAL_*` การตรวจสอบ &nbsp;·&nbsp; `INV_AUTH_*` สิทธิ์ &nbsp;·&nbsp; `INV_CALC_*` การคำนวณ &nbsp;·&nbsp; `INV_POST_*` การ posting &nbsp;·&nbsp; `INV_XMOD_*` ข้ามโมดูล
> **จำนวนกฎ:** ประมาณ 60 กฎ
> **ผู้ใช้งาน:** ผู้เขียนเทสและนักพัฒนา — ทุก rule ID ถูก anchor จากหน้า `04-test-scenarios*`
> **วงจรชีวิตสถานะ:** Section 5.1 (ที่มี) ถือ callout ความไม่ตรงระหว่าง Live UI vs BRD

## 1. ภาพรวม

หน้านี้รวบรวมกฎทางธุรกิจปฏิบัติการที่ปกครอง **โมดูล inventory** — ledger ที่โมดูลผลิตการเคลื่อนไหวอื่นทุกตัว (GRN receipts, store-requisition issues / transfers, physical / spot counts, credit notes, manual stock-in / stock-out) post เข้า เนื่องจาก inventory ขับเคลื่อนด้วย movement แทนที่จะด้วยเอกสาร ชุดกฎตรงนี้จึงดูแตกต่างจาก catalog ของ GRN / PR / PO: ไม่มีวงจรชีวิต `doc_status` ที่ต้องจัดการ ไม่มีความก้าวหน้า save / approve / commit และไม่มี "posting event" เดียวที่ fire ต่อ row แต่ละ transaction คือ posting event ของตัวเอง; การตรวจสอบรันก่อน **transaction ถูกเขียน** การคำนวณ populate cost-layer และ (ปลายน้ำ) period snapshot การกำหนดสิทธิ์เป็น RBAC-plus-source-document ผลกระทบของ posting คือ ledger ทันทีและ (ที่ใช้ได้) GL fan-out และกระบวนการ period-end คือชุดกฎแยกที่ lock ledger จากการเปลี่ยนแปลงย้อนหลัง

จุดเชิงโครงสร้างสองจุดให้สีกับทุกกฎด้านล่างและคุ้มค่าที่จะกล่าวซ้ำตั้งแต่ต้น **ประการแรก** **ไม่มี row `tb_stock_balance`** — on-hand ปัจจุบันที่ `(location_id, product_id, lot_no)` derive เป็น `last_snapshot.closing_qty + Σ (cost_layer.in_qty − cost_layer.out_qty)` ตั้งแต่ snapshot scope ที่ key นั้น ดังนั้นกฎที่ดูเหมือน "เพิ่ม balance" จริง ๆ หมายถึง "เขียน row `tb_inventory_transaction_cost_layer` ใหม่ที่มี `in_qty` / `out_qty` / `cost_per_unit` ที่ถูกต้องเพื่อให้ผลรวม derive ก้าวหน้าอย่างถูกต้อง" (ดู [[inventory/01-data-model]] § 5 item 1) **ประการที่สอง** `location_type` (`inventory`, `direct`, `consignment` บน `tb_location.location_type`) คือ **การตั้งค่าที่ส่งผลมากที่สุด** สำหรับพฤติกรรม posting — กำหนดว่า movement จะ debit สินทรัพย์ Inventory, Department Expense หรือ memo register และว่า movement สร้าง row `tb_inventory_transaction_cost_layer` หรือไม่ ในตอนแรก กฎหลายข้อด้านล่างถูก gate ด้วย `location_type` และพฤติกรรมของกฎเปลี่ยนตามนั้น

## 2. กฎการตรวจสอบ (Validation Rules)

Rule ID ตาม `INV_VAL_NNN` การตรวจสอบรัน **ณ เวลา post** — กล่าวคือ เมื่อโมดูล source (GRN, SR, count, credit note, manual stock-in / stock-out) พยายามเขียน row `tb_inventory_transaction` และลูก ไม่มีการแยก save-time / commit-time เพราะไม่มีสถานะ draft บนตัว transaction; เอกสาร source ถือความก้าวหน้านั้น

| `INV_VAL_007` | ทุก field ที่เป็นจำนวนเงินไม่ติดลบ: `cost_per_unit ≥ 0` บน `tb_inventory_transaction_detail`; `in_qty ≥ 0` และ `out_qty ≥ 0` (ไม่ใช่ทั้งสอง non-zero บน row cost-layer เดียวกัน) บน `tb_inventory_transaction_cost_layer`; `average_cost_per_unit ≥ 0` | Post | Reject ด้วย `"Cost per unit และ qty ต้องไม่ติดลบ; in_qty และ out_qty เป็น mutually exclusive บน cost-layer row เดียว"` |
| `INV_VAL_008` | งวดบัญชีที่บรรจุวันที่ transaction ไม่อยู่ใน `closed` หรือ `locked` บน `tb_period.status` Postings ย้อนหลัง (วันที่ transaction ในงวด `closed` / `locked`) ถูก reject ทันที การ re-open งวด `closed` เป็น action elevated แยก (`INV_AUTH_*`); งวด `locked` ไม่สามารถ re-open โดย role ใด | Post | Reject ด้วย `"ไม่สามารถ post เข้างวด <YYMM>: งวดเป็น <status> Re-open งวด (closed เท่านั้น) หรือ post restatement งวดปัจจุบัน (locked)"` |

| `INV_VAL_010` | เมื่อ `location_type = consignment` ความเป็นเจ้าของเป็นของ vendor จนถึง consumption Inbound (`good_received_note`) post memo cost-layer row แต่ **ไม่** ยก AP liability หรือ debit Inventory; outbound (`issue`) post COGS และ AP พร้อมกันตาม `INV_POST_006` Cross-type transfers จาก `consignment → inventory` (เข้ารับความเป็นเจ้าของ) ต้องการเอกสาร re-classification `tb_stock_in` ที่ชัดเจน และไม่เป็นโดยปริยาย | Post | Reject implicit consignment-to-inventory transfers ด้วย `"การถ่ายโอนความเป็นเจ้าของ consignment-to-inventory ต้องการเอกสาร re-classification stock-in ที่ชัดเจน"` |

| `INV_VAL_013` | Soft-delete บน `tb_inventory_transaction`: `deleted_at` **เท่านั้น** ถูก set เป็นส่วนของ sequence compensating-transaction (เขียน reversal แล้ว mark `deleted_at` ของ original ถ้า tenant policy คือ "ซ่อน transactions ที่ reverse แล้ว") Direct soft-delete โดยไม่มี compensating reversal ถูก reject | Soft-delete | Reject ด้วย `"ไม่สามารถ soft-delete inventory transaction โดยไม่มี compensating reversal"` |

## 3. กฎการคำนวณ (Calculation Rules)

ค่าจำนวนเงินทั้งหมดเก็บเป็น `Decimal(20, 5)` บน cost-layer และ detail rows การปัดเศษสำหรับการแสดงเป็น half-up ทศนิยม 2 ตำแหน่งสำหรับจำนวนเงินและ 3 ตำแหน่งสำหรับปริมาณ การคำนวณกลางใช้ precision 5dp เต็ม; เฉพาะค่าสุดท้ายที่เก็บ / แสดงถูกปัดเศษ

Rule ID ตาม `INV_CALC_NNN`

| Rule ID | สูตร |
| ------- | ------- |
| `INV_CALC_001` (detail total_cost) | `total_cost = qty × cost_per_unit` (signed ตาม sign ของ `qty`) เก็บที่ 5dp บน `tb_inventory_transaction_detail` |
| `INV_CALC_002` (cost-layer total_cost — inbound) | สำหรับ `in_qty > 0`: `total_cost = in_qty × cost_per_unit` บน `tb_inventory_transaction_cost_layer` |
| `INV_CALC_003` (cost-layer total_cost — outbound) | สำหรับ `out_qty > 0`: `total_cost = out_qty × cost_per_unit` ที่ `cost_per_unit` ถูกเลือกโดย costing method (ดู `INV_CALC_005` / `INV_CALC_006`) |
| `INV_CALC_004` (on-hand ปัจจุบัน — derive) | `on_hand_qty(location, product, lot) = last_snapshot.closing_qty + Σ_layer (cost_layer.in_qty − cost_layer.out_qty)` สำหรับ layers ที่ `created_at > last_snapshot.snapshot_at` scope ที่ key เมื่อไม่มี snapshot ก่อนหน้า ผลรวมเริ่มที่ 0 |
| `INV_CALC_005` (FIFO outbound cost pick) | สำหรับสินค้าที่ `costing_method = FIFO` (กำหนดบน `tb_product`): เมื่อ post outbound layer iterate rows `tb_inventory_transaction_cost_layer` ที่ `(location_id, product_id)` เรียงโดย `lot_seq_no` ascending บริโภค `out_qty` จากยอดคงเหลือของแต่ละ layer จนกว่าจะพอใจ แต่ละ layer ที่บริโภคผลิต outbound cost-layer row ของตัวเองที่มี `cost_per_unit = layer.cost_per_unit`; transaction outbound จึงอาจมี cost-layer rows **หลาย** rows สำหรับ detail row เดียวเมื่อ consumption ครอบคลุม lot |
| `INV_CALC_006` (Weighted-average outbound cost pick) | สำหรับสินค้าที่ `costing_method = WEIGHTED_AVERAGE`: outbound `cost_per_unit = current_average_cost(location, product) ณ เวลา post` average ปัจจุบันคือ `tb_inventory_transaction_cost_layer.average_cost_per_unit` ล่าสุดที่ `(location_id, product_id)` สำหรับสินค้านั้น (refresh โดย inbound ทุกครั้ง — ดู `INV_CALC_007`) |
| `INV_CALC_007` (Weighted-average refresh on inbound) | เมื่อ inbound post ภายใต้ WEIGHTED_AVERAGE: `new_average = (prior_on_hand × prior_average + in_qty × in_cost_per_unit) / (prior_on_hand + in_qty)` ปัดเป็น 5dp เขียนไปยัง `average_cost_per_unit` บน inbound cost-layer row ใหม่ การอ่านต่อมาที่ `(location_id, product_id)` รับค่านี้ |
| `INV_CALC_008` (Period snapshot — opening) | `opening_qty = previous_period.closing_qty`; `opening_cost_per_unit = previous_period.closing_cost_per_unit`; `opening_total_cost = opening_qty × opening_cost_per_unit` เขียนโดย period-open job (`enum_inventory_doc_type = open`, `enum_transaction_type = open_period`) |
| `INV_CALC_009` (Period snapshot — buckets receipt / issue / adjustment) | สำหรับงวดที่ปิด: `receipt_qty = Σ in_qty สำหรับ cost layers inbound`; `receipt_total_cost = Σ in_qty × cost_per_unit`; ทำนองเดียวกัน `issue_qty / issue_total_cost = Σ out_qty × cost_per_unit`; `adjustment_qty / adjustment_total_cost = Σ จาก layers adjustment_in / adjustment_out / credit_note_*` ผลรวมทั้งหมด scope ที่ key งวด × location × product × lot |
| `INV_CALC_010` (Period snapshot — closing) | `closing_qty = opening_qty + receipt_qty − issue_qty + adjustment_qty`; `closing_cost_per_unit = closing_total_cost / closing_qty` (set เป็น weighted average ปลายงวด; สำหรับ FIFO คือ `cost_per_unit` ของ lot ที่เหลือ); `closing_total_cost = opening_total_cost + receipt_total_cost − issue_total_cost + adjustment_total_cost + diff_amount` bucket `diff_amount` จับ effects ของ price-revaluation (credit-note-amount adjustments) |
| `INV_CALC_011` (Credit-note amount adjustment) | สำหรับ `enum_transaction_type = credit_note_amount`: เขียน cost-layer row ด้วย `in_qty = out_qty = 0`, `diff_amount = vendor_credit_amount` (signed; ลบสำหรับ vendor concession ลดต้นทุน), `transaction_type = credit_note_amount` Cost บน lot ดั้งเดิมถูกคำนวณใหม่: `new_lot_cost_per_unit = (original_lot_total_cost + diff_amount) / original_lot_qty` Consumption ต่อมาจาก lot เดียวกันรับ cost ที่ revalued; ส่วนที่บริโภคไปแล้วไม่ปรับย้อนหลัง (price variance ไหลผ่าน `diff_amount`) |
| `INV_CALC_012` (การปัดเศษ) | การปัดเศษทั้งหมดเป็น half-up ค่าที่เก็บใช้ precision ของ column; การคำนวณกลางถือ 5dp; การ aggregation on-the-fly อ่านค่าที่ปัดเศษซ้ำจากแต่ละขั้น |

### 3.1 ตัวอย่างใช้งาน (สินค้า Weighted-average, ฿ THB)

สินค้าที่มี `costing_method = WEIGHTED_AVERAGE` ไม่มี on-hand ก่อนหน้า สาม movements ตามลำดับเวลาที่ location ประเภท inventory เดียวกัน:

- **GRN receipt #1** (`enum_transaction_type = good_received_note`): `in_qty = 100`, `cost_per_unit = ฿10.00`
  - Cost layer: `in_qty = 100`, `cost_per_unit = ฿10.00`, `total_cost = ฿1,000.00`
  - `new_average = (0 × 0 + 100 × 10.00) / (0 + 100) = ฿10.00000` ตาม `INV_CALC_007`
- **GRN receipt #2** (`good_received_note`): `in_qty = 50`, `cost_per_unit = ฿14.00`
  - Cost layer: `in_qty = 50`, `cost_per_unit = ฿14.00`, `total_cost = ฿700.00`
  - `new_average = (100 × 10.00 + 50 × 14.00) / (100 + 50) = (1,000 + 700) / 150 = ฿11.33333` ตาม `INV_CALC_007`
- **Store-requisition issue** (`issue`): `out_qty = 80`
  - Outbound cost-layer row: `out_qty = 80`, `cost_per_unit = ฿11.33333` (เลือกจาก `current_average_cost` ตาม `INV_CALC_006`), `total_cost = ฿906.67` (`80 × 11.33333` ปัด 2dp)
  - `new_average` คงที่ `฿11.33333` (issues ไม่เปลี่ยน average ภายใต้ weighted-average; เฉพาะ inbound ที่เปลี่ยน)
- On-hand ปัจจุบันตาม `INV_CALC_004`: `0 (ไม่มี snapshot ก่อน) + (100 + 50 − 80) = 70 units`; การตีมูลค่า `70 × 11.33333 = ฿793.33`

สำหรับ **FIFO** variant ของสินค้าเดียวกัน (`INV_CALC_005`): issue จะบริโภคจาก receipt #1 ก่อน (`lot_seq_no` ascending) — `out_qty = 80` ที่ `cost_per_unit = ฿10.00` (หนึ่ง cost-layer row), `total_cost = ฿800.00` ยอดคงเหลือของ receipt #1 กลายเป็น `100 − 80 = 20` units ที่ `฿10.00`; ยอดของ receipt #2 ไม่ถูกแตะที่ `50` units ที่ `฿14.00` การตีมูลค่า inventory closing: `20 × 10.00 + 50 × 14.00 = ฿900.00`

### 3.2 ตัวอย่างใช้งาน (location_type = direct)

Receipt ของสินค้าไปยัง location `direct` `INV_VAL_009` อนุญาต inbound การ commit GRN เขียน row `tb_inventory_transaction` (สำหรับ audit) แต่ **ไม่** สร้าง row `tb_inventory_transaction_cost_layer` — receipt expense ถูกรับรู้ทันที:

- **GRN receipt** (`good_received_note`, location_type = direct): `qty = 30`, `cost_per_unit = ฿8.00`
  - Inventory transaction header + detail เขียน (audit trail)
  - **ไม่มี cost-layer row เขียน** (ไม่มี asset ledger entry)
  - GL: `Dr Department Expense ฿240.00 / Cr AP ฿240.00` ตาม [[good-receive-note/02-business-rules]] `GRN_POST_006` direct-variant
  - On-hand ที่ `(direct_location, product, *)` คงที่ 0 (direct locations ไม่ถือ balance)

## 5. กฎ Posting

โมดูล inventory ไม่มีวงจรชีวิต `doc_status` ของเอกสาร — ทุก transaction ที่ post คือ immutable ตอนเขียน state diagram จึง degenerate: แต่ละ transaction transition `(none) → posted` และอาจผ่าน reversal compensating ถูก mark `deleted_at` (reverse เชิงตรรกะแต่ไม่เคยลบ physical) กฎที่น่าสนใจคือ **fan-out effects** ที่ fire เมื่อเขียน

Rule ID ตาม `INV_POST_NNN`

| Rule ID | Transition / Event | Effects |
| ------- | ------------------ | ------- |
| `INV_POST_001` | Post inbound (`good_received_note`, `transfer_in`, `adjustment_in`) ไปยัง location ประเภท `inventory` | Insert `tb_inventory_transaction` (header) และ `tb_inventory_transaction_detail` (ต่อสินค้า `qty > 0`) สำหรับสินค้าที่มี costing method ที่ต้องการ layer tracking ให้ insert หนึ่ง row `tb_inventory_transaction_cost_layer` ที่มี `in_qty > 0`, `cost_per_unit`, `lot_no`, `lot_index`, `lot_seq_no` และ `at_period` set; คำนวณ `average_cost_per_unit` ใหม่ตาม `INV_CALC_007` และเขียนบน layer row ใหม่ GL: **Dr Inventory** ที่ `total_cost` / **Cr** บัญชี credit ของเอกสาร source (AP สำหรับ GRN, Inventory ที่ location อื่นสำหรับ transfers, Adjustment Suspense สำหรับ stock-in) |
| `INV_POST_002` | Post outbound (`issue`, `transfer_out`, `adjustment_out`) จาก location ประเภท `inventory` | Insert `tb_inventory_transaction` + `tb_inventory_transaction_detail` (`qty < 0`) Cost-layer rows ขึ้นกับ costing method: FIFO เขียน outbound layer rows หนึ่งหรือมากกว่าบริโภคจาก `lot_seq_no` ascending จนกว่า `out_qty` พอใจ (แต่ละ row ถือ `cost_per_unit` ของ lot); weighted-average เขียน outbound layer row เดียวที่ `average_cost_per_unit` ปัจจุบัน `INV_VAL_005` (no-negative-balance) ตรวจก่อนเขียน GL: **Dr** บัญชีปลายทาง (COGS / Department Expense สำหรับ issues, Inventory ที่ destination สำหรับ transfers, Adjustment Suspense สำหรับ stock-out) / **Cr Inventory** ที่ cost ที่เลือก |
| `INV_POST_003` | Post receipt ไปยัง location ประเภท `direct` | Insert header + detail สำหรับ audit **ไม่มี cost-layer row** ไม่มีผลกระทบต่อ balance GL: **Dr Department Expense** / **Cr** บัญชี credit ของเอกสาร source (AP สำหรับ GRN เป็นต้น) นี่คือเส้นทาง direct-cost `INV_VAL_009` |
| `INV_POST_004` | Post receipt ไปยัง location ประเภท `consignment` | Insert header + detail สำหรับ audit Insert cost-layer row flag เป็น consignment (โดยปกติผ่าน `dimension` หรือ `info` JSON; ไม่ผ่าน enum value แยก) **ไม่ยก AP liability ไม่ debit Inventory asset ณ receipt** Memo-only — ของถูกติดตามเพื่อการมองเห็นแต่ความเป็นเจ้าของยังคงอยู่กับ vendor |
| `INV_POST_005` | Post issue จาก location ประเภท `consignment` | GL สองขาแบบ atomic: **Dr COGS / Cr AP** ที่ cost consignment Cost-layer row บันทึก consumption (`out_qty > 0`); รายการ AP ถูกยก ณ เวลา consumption ไม่ใช่ ณ receipt นี่คือรูปแบบ payment-on-consumption consignment ที่บันทึกใน `../carmen/docs/Inventory/location-type-and-financial-treatment.md` |
| `INV_POST_006` | Transfer ระหว่าง locations (`store_requisition` transfer ที่ source ≠ destination) | Cost-layer rows สองอันภายใต้ `tb_inventory_transaction` เดียว (หรือสอง transactions ที่ coupled ขึ้นกับ posting model ของโมดูล SR): `transfer_out` ที่ source (`out_qty > 0`, FIFO หรือ WA cost pick), `transfer_in` ที่ destination (`in_qty > 0` ด้วย `cost_per_unit = transfer_out.cost_per_unit`, `parent_lot_no = source_lot`) `INV_VAL_005` รันเทียบกับ balance ของ source location GL: โดยปกติ **Dr Inventory@Dest / Cr Inventory@Source** — net zero บนบัญชี control Inventory ถ้าทั้ง locations เป็น inventory-type ถ้า destination เป็น direct transfer hit expense แทน (หายาก; โดยปกติใช้คู่ re-classification stock-in / stock-out) |
| `INV_POST_007` | Credit-note amount adjustment (`credit_note_amount`) | Insert cost-layer row ด้วย `in_qty = out_qty = 0`, `diff_amount = vendor_credit_amount` (signed), `transaction_type = credit_note_amount` อัปเดต `cost_per_unit` ของ lot ดั้งเดิมตาม `INV_CALC_011` Consumption ปลายน้ำจาก lot เดียวกันรับ cost ที่ revalued; ส่วนที่บริโภคไปแล้วไม่ปรับย้อนหลัง GL: **Dr AP / Cr Inventory** ที่ amount credit (vendor concession ลด AP และ inventory carrying value) |
| `INV_POST_008` | Credit-note quantity adjustment (`credit_note_quantity`) | Insert cost-layer row ด้วย `out_qty = credit_qty` บริโภคจาก lot ของ receipt ดั้งเดิมที่ `cost_per_unit` ของ lot GL: **Dr AP / Cr Inventory** ที่ `out_qty × cost_per_unit` คู่ฝั่ง receipt บนปริมาณของ GRN source ไม่ถูกแก้ไข (`committed` คือปลายทาง); credit-note คือการกลับรายการแบบสมมาตร |
| `INV_POST_009` | Period close (`enum_inventory_doc_type = close`) | System job กวาด transactions เปิดทั้งหมดในงวดและเขียน rows `tb_period_snapshot` ต่อ `(location_id, product_id, lot_no, lot_index)` — `closing_qty / closing_cost_per_unit / closing_total_cost` และ bucket aggregates ตาม `INV_CALC_009` เขียน cost-layer row หนึ่ง row ต่อ lot ที่ได้รับผลกระทบด้วย `transaction_type = close_period` Set `tb_period.status = closed` Postings ย้อนหลังเข้างวดนี้ตอนนี้ถูก reject ตาม `INV_VAL_008` |
| `INV_POST_010` | Period open (`enum_inventory_doc_type = open`) | System job (โดยปกติ chain กับ `INV_POST_009`) เปิดงวดถัดไป — set `tb_period.status = open` บนงวดใหม่และเขียน rows `tb_inventory_transaction_cost_layer` ด้วย `transaction_type = open_period`, `in_qty = previous.closing_qty`, `cost_per_unit = previous.closing_cost_per_unit`, `lot_seq_no` รักษาจาก lot ที่ปิด ลำดับ `lot_seq_no` ของงวดก่อนหน้าส่งต่อ รักษาลำดับ FIFO ข้ามขอบงวด |
| `INV_POST_011` | Period lock (`closed → locked` บน `tb_period.status`) | การอนุญาตตาม `INV_AUTH_006` ไม่อนุญาต re-open อีก ไม่มี fan-out ฝั่ง transaction |
| `INV_POST_012` | Reversal (compensating transaction) | Post `tb_inventory_transaction` ใหม่ด้วย `qty` negate เทียบกับ `qty` ของ detail ดั้งเดิม; FIFO บริโภคจาก / reverse lot ดั้งเดิมเพื่อรักษา cost-layer integrity Transaction ดั้งเดิม **ไม่** ถูกแก้ไข ถ้า tenant policy คือ "ซ่อน transactions ที่ reverse แล้ว" set `deleted_at` บน row ดั้งเดิม **หลังจาก** compensating transaction post (ตาม `INV_VAL_013`) Use case: แก้ไข stock-out ที่ post ผิด reverse receipt สำหรับสินค้าที่ภายหลังพบว่าเป็นของ PO อื่น เป็นต้น |

State diagram (degenerate, single-state):

```
[*] → posted → (compensating reversal optional) → posted (with deleted_at set on original)
```

Transaction row เป็น immutable หลังเขียน; reversal คือ row ใหม่ ไม่ใช่การเปลี่ยนสถานะ

| `INV_XMOD_008` | Finance / GL | กระทบยอด Inventory-to-GL รันเป็นขั้นตอน period-close ผลรวมของ cost-layer `total_cost` และ `diff_amount` ทั่ว movements ประเภท `inventory` ทั้งหมดในงวดต้องเท่ากับการเปลี่ยน net ของบัญชี control Inventory GL Variance flag exception การกระทบยอดที่ Finance สืบสวนก่อน period lock (`INV_AUTH_006`) |
| `INV_XMOD_009` | [[product]] | `costing_method` ของสินค้า (FIFO vs WEIGHTED_AVERAGE) ถูกอ่านโดยโมดูล inventory ณ ทุกการ post outbound การเปลี่ยน costing method บนสินค้าที่มี on-hand ไม่เป็นศูนย์ถูก **บล็อก** (ประวัติ cost-layer จะ inconsistent ภายใต้ method ใหม่); การเปลี่ยนแปลงต้องการ drain stock ก่อนหรือรัน migration entry one-time ภายใต้การอนุญาต elevated |
| `INV_XMOD_010` | โมดูลทั้งหมดที่ผลิตการเคลื่อนไหว | ทุกโมดูลที่ mutate inventory (GRN, SR, count, spot-check, credit note, manual stock-in / stock-out) เขียนผ่าน API `tb_inventory_transaction` เดียวกัน ไม่มี bypass เฉพาะโมดูล; `inventory_doc_type` แบบ polymorphic คือตัวแยกเดียวที่ระดับ ledger นี่คือ chokepoint เดียวที่ทำให้ audit trail เชื่อถือได้ |
