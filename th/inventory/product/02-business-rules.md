# สินค้า (Product) — Business Rules

> **At a Glance**
> **family ของกฎ:** `PRD_VAL_*` validation &nbsp;·&nbsp; `PRD_AUTH_*` permission &nbsp;·&nbsp; `PRD_CALC_*` calc &nbsp;·&nbsp; `PRD_LIFE_*` lifecycle &nbsp;·&nbsp; `PRD_XMOD_*` cross-module
> **จำนวนกฎ:** ประมาณ 65 กฎ
> **กลุ่มผู้ใช้:** ผู้เขียน test + developer — ทุก rule ID ถูก anchor จากหน้า `04-test-scenarios*`
> **วงจรชีวิตของสถานะ:** Section 5.1 (เมื่อมี) มี callout ความแตกต่างของ Live UI vs BRD

## 1. ภาพรวม

หน้านี้บันทึกกฎทางธุรกิจเชิงปฏิบัติการที่ควบคุม **โมดูลข้อมูลหลักของสินค้า** — แคตตาล็อกที่โมดูลอื่นทุกตัวอ้างอิง ต่างจากโมดูลธุรกรรม (PR, PO, GRN, SR, inventory) สินค้าเป็น **master-data** ดังนั้นชุดกฎดูต่างไป ไม่มีเอกสาร workflow `doc_status` พร้อมการเปลี่ยน `draft → in_progress → completed` ไม่มี event การ post ที่ยิง journal entry ไม่มีหน้าต่าง period-end ที่ล็อก record วงจรชีวิตง่ายกว่า — `create → active → deprecated (inactive) → soft-deleted` — และกฎที่น่าสนใจจะรวมอยู่ที่ **การ validate ตอน create/update** (ความไม่ซ้ำ ความสอดคล้องของการจำแนก ความถูกต้องของ unit / conversion) **การคำนวณ default ที่สืบทอด** (ค่าความคลาดเคลื่อน tax profile cascade จากหมวดหมู่ผ่าน item-group ไปยังสินค้า) **การกำหนดสิทธิ์** (ใครสร้าง ใครปิดใช้ ใคร import / export ใครลบ) **guard ของวงจรชีวิต** (ไม่สามารถปิดใช้สินค้าบน PR / PO / SR เปิด ไม่สามารถลบสินค้าที่มีคลังไม่เป็นศูนย์หรือถูกอ้างอิงโดยสูตรที่ publish) และ **กฎด้านการอ่านข้ามโมดูล** (ผู้บริโภคเห็นแคตตาล็อก active snapshot รอดจากการ re-code สินค้า)

จุดโครงสร้างสองจุดให้สีกับทุกกฎด้านล่างและควรย้ำตั้งแต่ต้น **ประการแรก** **แคตตาล็อกคือรากของ dependency** — ทุกตารางธุรกรรมปลายน้ำของคลังสินค้าอ้างอิงสินค้าผ่าน `product_id` ดังนั้นกฎวงจรชีวิตส่วนใหญ่ด้านล่าง guard ต่อ **สถานะ in-use ในโมดูลปลายน้ำ** มากกว่าต่อสถานะ local ของสินค้า **ประการที่สอง** **วิธีการคิดต้นทุนไม่ได้อยู่บนสินค้า** — อยู่บน `tb_business_unit.calculation_method` (platform schema) ดังนั้นกฎต่อสินค้าเรื่อง FIFO vs Weighted-Average ที่ปรากฏใน carmen/docs จึงไม่บังคับใช้ได้ที่ชั้นของสินค้า `standard_cost` ของสินค้าคือต้นทุนอ้างอิงที่ใช้โดยการประเมินมูลค่า count-variance (`enum_physical_count_costing_method = standard`) และโดย recipe baselining ไม่ใช่โดย FIFO / WA cost-pick engine ดู [[product/01-data-model]] § 5 สำหรับแคตตาล็อกความแตกต่างเต็ม

## 2. กฎการตรวจสอบความถูกต้อง

Rule ID ตามรูปแบบ `PRD_VAL_NNN` การ validate รันที่ **create** และ **update** บน product master และตารางสนับสนุน (`tb_unit`, `tb_unit_conversion`, `tb_product_category`, `tb_product_sub_category`, `tb_product_item_group`, `tb_product_location`, `tb_product_tb_vendor`)

| ------- | --------- | ------------- | ----------------- |

| `PRD_VAL_002` | `tb_product.name` ไม่ว่างและไม่ใช่ whitespace ร่วมกับ `code` และ `deleted_at` ทูเปิล `(code, name, deleted_at)` ไม่ซ้ำ | Create / Update | ปฏิเสธด้วย `"Product name is required."` `400 Bad Request` |

| `PRD_VAL_004` | `tb_product.product_item_group_id` (เมื่อตั้ง) อ้างอิง `tb_product_item_group` ที่ active ไม่ถูก soft-delete แคตตาล็อก production ต้องการ item-group; null ได้รับอนุญาตใน schema สำหรับ staging ของ import แต่ปฏิเสธที่ user-edit submit | Create / Update | ปฏิเสธด้วย `"Item group is required (or selected item group is inactive/deleted)."` `400 Bad Request` |
| `PRD_VAL_005` | `tb_product.barcode` (เมื่อตั้ง) ไม่ซ้ำภายใน `(barcode, deleted_at)` ไม่มี schema constraint — บังคับใช้ที่ application ตาม [[product/01-data-model]] § 5 รายการ 5 การเลือกรูปแบบ (UPC, EAN, CODE128) ไม่ validate ที่ schema; application อาจตรวจสอบรูปแบบตามความยาว | Create / Update | ปฏิเสธด้วย `"Barcode <barcode> is already assigned to product <other_code>."` `409 Conflict` |
| `PRD_VAL_006` | `tb_product.price_deviation_limit` และ `tb_product.qty_deviation_limit` อยู่ใน `[0, 100]` (เปอร์เซ็นต์) default `0` (ไม่มี tolerance) ค่าลบถูกปฏิเสธที่ submit | Create / Update | ปฏิเสธด้วย `"Deviation limits must be between 0 and 100 percent."` `400 Bad Request` |
| `PRD_VAL_007` | `tb_product.standard_cost` เป็น `≥ 0` ต้นทุนอ้างอิงไม่ติดลบ ศูนย์ได้รับอนุญาต (เช่น สำหรับสินค้า FOC / promotion) | Create / Update | ปฏิเสธด้วย `"Standard cost cannot be negative."` `400 Bad Request` |
| `PRD_VAL_008` | `tb_product_category.code` และ `tb_product_item_group.code` ไม่ว่างและไม่ซ้ำต่อ `productcategory_code_u` / `productitemgroup_code_name_product_subcategory_u` `tb_product_sub_category.code` default เป็น `""` ตาม schema (ดู [[product/01-data-model]] § 5 รายการ 10) — UI ของ application ต้องการ แต่ schema อนุญาตให้ว่าง | Create / Update | ปฏิเสธด้วย `"Category / sub-category / item-group code is required and must be unique within its scope."` `400 / 409` ตามที่เหมาะสม |
| `PRD_VAL_009` | `tb_product_sub_category.product_category_id` อ้างอิงหมวดหมู่ parent ที่ active; `tb_product_item_group.product_subcategory_id` อ้างอิง sub-category parent ที่ active วงกลมเป็นไปไม่ได้โดยการสร้าง schema (ไม่มี self-reference) | Create / Update | ปฏิเสธด้วย `"Parent category is inactive or deleted."` `400 Bad Request` |

| `PRD_VAL_013` | แถว `tb_product_tb_vendor` มี `(vendor_id, product_id, deleted_at)` ไม่ซ้ำต่อ `product_vendor_vendor_product_u` ทั้ง FK ต้องอ้างอิงแถวที่ active ไม่ถูกลบ | Create / Update | ปฏิเสธซ้ำด้วย `"Vendor <vendor> is already mapped to product <product>."` `409 Conflict` |
| `PRD_VAL_014` | `tb_product.tax_profile_id` (เมื่อตั้ง) อ้างอิง `tb_tax_profile` ที่ active ไม่ตั้ง (`null`) ได้รับอนุญาต — กฎการสืบทอด (Section 3, `PRD_CALC_002`) เดินขึ้นห่วงโซ่การจำแนกเพื่อหาค่าไม่เป็น null | Create / Update | ปฏิเสธด้วย `"Selected tax profile is inactive or deleted."` `400 Bad Request` |
| `PRD_VAL_015` | การเปลี่ยน `tb_product.product_status_type` ถูกจำกัด: `active → inactive` อนุญาต (subject ต่อ in-use guard `PRD_LIFE_*`); `inactive → active` อนุญาต (re-activation); ไม่มีค่าอื่นอนุญาต (`discontinued` **ไม่ได้** อยู่ใน enum ตาม [[product/01-data-model]] § 5 รายการ 4) | Update | ปฏิเสธค่าผิด: `"Product status must be 'active' or 'inactive'."` `400 Bad Request` |
| `PRD_VAL_016` | `tb_product.info`, `tb_product.dimension` และ `tb_product.certification` เป็น JSON ที่ถูกต้อง ไม่มีการ validate รูปทรงระดับ schema; application อาจบังคับใช้กฎรูปทรงเฉพาะ tenant (เช่น `info.allergens` เป็น array ของ code สารก่อภูมิแพ้จาก list ที่รู้จัก) แต่ schema validate แค่ syntax JSONB | Create / Update | ปฏิเสธ JSON ผิดรูปแบบด้วย `400 Bad Request` การละเมิดกฎรูปทรงเป็น application-layer |

## 3. กฎการคำนวณ

Rule ID ตามรูปแบบ `PRD_CALC_NNN` Product master ไม่มี ledger ที่ post และไม่มีการคำนวณต่อเอกสาร แต่เป็นเจ้าของกฎ **default ที่สืบทอด / cascade** และกฎ **ฟิลด์ที่ derive** ที่โมดูลปลายน้ำบริโภค

| Rule ID | สูตร |
| ------- | ------- |
| `PRD_CALC_001` (เส้นทางการจำแนก) | เส้นทางการจำแนกของสินค้าคือ `tb_product_category.name / tb_product_sub_category.name / tb_product_item_group.name` resolve โดยเดินขึ้น `tb_product.product_item_group_id → tb_product_item_group.product_subcategory_id → tb_product_sub_category.product_category_id` แสดงเป็น string เดียวที่อ่านได้สำหรับแสดงผล ("Beverages / Hot / Coffee Beans") คำนวณใหม่ทุกครั้งที่อ่าน ไม่มีคอลัมน์เก็บ |
| `PRD_CALC_002` (การสืบทอด tax profile) | tax profile ที่มีผลสำหรับสินค้าคือค่าไม่เป็น null แรกเดินขึ้นห่วงโซ่การจำแนก: `tb_product.tax_profile_id` ⟶ `tb_product_item_group.tax_profile_id` ⟶ `tb_product_sub_category.tax_profile_id` ⟶ `tb_product_category.tax_profile_id` ถ้าทั้งสี่เป็น null ไม่มีภาษีใช้ (อัตราที่มีผล 0%, "ยกเว้น") Application อ่านห่วงโซ่ที่ทุก PR/PO/GRN line create เพื่อกำหนด tax profile ที่ใช้ |
| `PRD_CALC_003` (การสืบทอดค่าความคลาดเคลื่อน) | cascade เดียวกับ `PRD_CALC_002` สำหรับ `price_deviation_limit` และ `qty_deviation_limit` — ค่าแรกที่ไม่เป็นศูนย์เดินขึ้น `product → item-group → sub-category → category` ศูนย์ถือเป็น "ไม่ได้ตั้ง" (ผ่านไป) Floor คือ 0% เพดาน 100% กฎ procurement / receiving (`PR_VAL_*`, `GRN_VAL_*`) อ่านค่าที่มีผลนี้ |
| `PRD_CALC_004` (การสืบทอด is_used_in_recipe) | cascade เดียวกันสำหรับ `is_used_in_recipe` และ `is_sold_directly` ค่าระดับสินค้าถ้าตั้งชนะ; มิฉะนั้นสืบทอดจาก item-group → sub-category → category (หมายเหตุ: `null` คือผ่านไป; default `true` / `false` บนคอลัมน์ไม่ใช่เหมือนกับการสืบทอด — application แยกแยะ "user set false" จาก "unset") |
| `PRD_CALC_005` (Conversion factor) | conversion factor base-to-target สำหรับแถว `tb_unit_conversion` คือ `to_unit_qty / from_unit_qty` ใช้แปล qty บนบรรทัดเอกสารปลายน้ำกลับเป็นหน่วยคลังฐานของสินค้า (`tb_product.inventory_unit_id`) ลำดับ resolution ที่ document-line save: (a) ถ้าหน่วยของเอกสารเท่ากับหน่วยฐาน factor = 1; (b) ค้นหา `tb_unit_conversion` ที่ match `(product_id, unit_type, from_unit_id, to_unit_id)`; (c) ถ้าไม่มีแถวอยู่สำหรับคู่หน่วยที่ขอ การ save ของ document-line ถูกปฏิเสธ |
| `PRD_CALC_006` (การแปลงหลายขั้น) | เมื่อคู่หน่วยที่ขอไม่มีแถว `tb_unit_conversion` ตรง แต่เส้นทาง transitive มีอยู่ (เช่น `KG → G` และ `G → MG` นิยาม ไม่มี `KG → MG` ตรง) engine จัด factor ตามเส้นทาง subject ต่อความสอดคล้องสองทิศทาง `PRD_VAL_011` วงกลมหรือเส้นทางคลุมเครือ flag สำหรับ review |
| `PRD_CALC_007` (precision การแสดงผลของหน่วยฐานที่มีผล) | precision การแสดงผลสำหรับ qty ในหน่วยคลังฐานอ่านจาก `tb_unit.decimal_place` (default `2`) การเก็บข้อมูลพื้นฐานคือ `Decimal(20, 5)` การปัดเศษการแสดงผลคือ half-up |
| `PRD_CALC_008` (Last receiving cost — derived) | "last receiving cost" แสดงบนมุมมองรายละเอียดของสินค้าคือ **derive** ไม่ได้เก็บ (ตาม [[product/01-data-model]] § 5 รายการ 12) resolve เป็น `cost_per_unit` ของแถว inbound `tb_inventory_transaction_cost_layer` ล่าสุดที่ `(product_id, *)` พร้อม `transaction_type ∈ {good_received_note, adjustment_in, transfer_in}` GRN ต้นทาง, vendor, date อ่านโดย join กลับไปยัง source-document ผ่าน polymorphic `inventory_doc_type` / `inventory_doc_no` แสดงข้างเคียง `standard_cost` เพื่อเปรียบเทียบ |
| `PRD_CALC_009` (On-hand ปัจจุบัน — derived, ไม่ได้อยู่บนสินค้า) | "on-hand ปัจจุบันต่อ location" บนมุมมองรายละเอียดของสินค้าคือ **derive** จาก inventory cost-layer ledger (ดู [[inventory/01-data-model]] § 5 รายการ 1) ต่อ location: `last_period_snapshot.closing_qty + Σ (cost_layer.in_qty − cost_layer.out_qty)` ตั้งแต่ snapshot กำหนดขอบเขตด้วย `(product_id, location_id, lot_no)` ไม่มีโมเดล `tb_stock_balance` หน้าสินค้า render read-model นี้ |
| `PRD_CALC_010` (Cost-tier inheritance) | เมื่อสินค้าไม่มี `standard_cost` ตั้ง (`= 0`) recipe-baseline cost falls back ไปที่ `cost_per_unit` ของ cost-layer inbound ล่าสุด (เหมือน `PRD_CALC_008`) เมื่อแม้แต่นั้นก็ไม่มี (สินค้าไม่เคยรับ) recipe-baseline cost คือ `0` — สูตรที่รวมสินค้าแสดง cost-incomplete warning |

## 5. กฎวงจรชีวิต

Product เป็น master-data — วงจรชีวิตคือ `(none) → active → inactive → soft-deleted` พร้อม option ที่จะ **re-activate** จาก `inactive` ไม่มี document workflow; กฎวงจรชีวิตด้านล่างคือ **guard ของการเปลี่ยน** ที่ gate การย้ายระหว่างสถานะ

Rule ID ตามรูปแบบ `PRD_LIFE_NNN`

| Rule ID | การเปลี่ยน / Event | ผลกระทบ / Guard |
| ------- | ------------------ | ---------------- |
| `PRD_LIFE_001` | `(none) → active` (create) | Insert `tb_product` พร้อม `product_status_type = active`, `is_active = true`, กฎ validation `PRD_VAL_001`–`PRD_VAL_016` ผ่านทั้งหมด default ที่สืบทอด (`PRD_CALC_002`–`PRD_CALC_004`) อ่านที่การแสดงครั้งแรก; ไม่มี snapshot ที่เก็บ Activity log บันทึก `{ event: 'create', by: <user>, at: <ts> }` |
| `PRD_LIFE_002` | `active → inactive` (ปิดใช้) | Validation: **ไม่มีบรรทัด PR / PO / SR เปิด** ที่ `doc_status ∉ {completed, cancelled, voided}` อาจอ้างอิง `product_id` นี้ Soft-block (Product Administrator สามารถ override ด้วยข้อความเหตุผลใน activity log) ถ้าสินค้าถูกอ้างอิงโดยสูตรที่ publish — recipe re-costing และ explosion ของ theoretical-consumption จะพัง เมื่อ override application อาจ flag สูตรที่ได้รับผลกระทบสำหรับ review อัตโนมัติ ในการปิดใช้บริสุทธิ์ (ไม่มีเอกสารเปิด ไม่มี reference ของสูตรที่ publish) การ update ตั้ง `product_status_type = inactive`; `is_active` ไม่เปลี่ยน Activity log บันทึกการเปลี่ยน |
| `PRD_LIFE_003` | `inactive → active` (re-activate) | Validation: ห่วงโซ่การจำแนกยังถูกต้อง (item-group / sub-category / category parent ไม่ถูกลบ) ค่าการสืบทอดคำนวณใหม่ที่การอ่านครั้งต่อไป Activity log บันทึก re-activation; recipe-review flag จาก `PRD_LIFE_002` ถูกล้าง |
| `PRD_LIFE_004` | `active | inactive → soft-deleted` (ลบ) | Hard guard: **on-hand ปัจจุบันไม่เป็นศูนย์** ที่ location ใด (derive ตาม `PRD_CALC_009`) บล็อกการลบด้วย `"Cannot delete product with non-zero on-hand. Drain inventory first via stock-out / write-off, then re-attempt."` `409 Conflict` **บรรทัดเอกสารที่ไม่ถูก soft-delete** ที่อ้างอิงสินค้า (PR / PO / GRN / SR / count / spot-check / credit-note / pricelist / recipe-ingredient) บล็อกด้วย `"Product is referenced by <N> documents and cannot be deleted."` `409 Conflict` **วัตถุดิบสูตรที่ไม่ถูก soft-delete** ที่อ้างอิงสินค้าบล็อกคล้ายกัน เมื่อ guard ผ่านทั้งหมด: ตั้ง `deleted_at`, `deleted_by_id`; `(code, name)` ใช้งานได้ใหม่หลัง soft-delete (constraint unique กำหนดขอบเขตด้วย `deleted_at`) |
| `PRD_LIFE_005` | `is_active = true → false` (hard-disable) | แตกต่างจาก `product_status_type = inactive` — `is_active = false` ลบสินค้าจาก picker ทุกตัวรวม admin view (ตาม [[product/01-data-model]] § 5 รายการ 14) subject ต่อ guard เดียวกับ `PRD_LIFE_002` บวกกฎเพิ่มเติม: `is_active = false` หมายถึง `product_status_type = inactive` (application บังคับใช้การรวมนี้; schema อนุญาตแต่ไม่บังคับ) |
| `PRD_LIFE_006` | Bulk import — create/update | bulk import ผ่าน Excel / CSV รัน row-level validation ตาม `PRD_VAL_*` และ dry-run preview แถวที่ failure validation แสดงใน downloadable error report; แถวที่ผ่านถูก insert / update ใน transaction เดียว Partial-success mode คือ default (แถวที่ผ่าน commit, แถวที่ fail ถูกข้ามและรายงาน); strict mode (all-or-nothing) เป็น option ต่อ job `created_by_id` / `updated_by_id` บันทึก Product Administrator ที่รัน import |
| `PRD_LIFE_007` | Bulk import — soft-delete | mass soft-delete ผ่าน import (คอลัมน์ `action = 'delete'`) ได้รับอนุญาตแต่แต่ละแถว subject ต่อ guard `PRD_LIFE_004`; แถวที่ fail รายงานรายตัวใน error report ไม่มี bypass "force delete" — in-use guard เป็น absolute ที่ชั้นของสินค้า |
| `PRD_LIFE_008` | การเปลี่ยนสถานะตามตารางเวลา | application อาจรองรับการตั้งเวลาการเปลี่ยนสถานะให้มีผลในวันที่อนาคต (เช่น `effective_at = 2026-06-01 deactivate`) Implementation คือ scheduled job ที่รันการเปลี่ยนภายใต้อำนาจของผู้ขอเดิม; ไม่มีคอลัมน์ `effective_at` ระดับ schema บน `tb_product` subject ต่อ lifecycle guard เดียวกันในเวลาที่รัน (ดังนั้น scheduled deactivation อาจล้มเหลวถ้ามีเอกสารเปิดในเวลารัน — scheduled job log การล้มเหลวและแจ้งผู้ขอ) |
| `PRD_LIFE_009` | Restore จาก soft-delete | Product Administrator อาจ restore สินค้าที่ soft-deleted (ล้าง `deleted_at` และ `deleted_by_id`) Validation รันใหม่ที่ restore — ถ้า `(code, name)` ถูก re-use โดยสินค้า live อื่นในระหว่างนั้น restore ถูกปฏิเสธด้วย `"A live product with this code/name already exists. Restore is blocked."` `409 Conflict` |
| `PRD_LIFE_010` | การจัดระเบียบการจำแนกใหม่ | การย้ายสินค้าระหว่าง item-group (กล่าวคือ update `tb_product.product_item_group_id`) ได้รับอนุญาตแต่ update default ที่สืบทอดสำหรับ tax-profile และ ค่าความคลาดเคลื่อนตาม `PRD_CALC_002` / `PRD_CALC_003` การเปลี่ยนเป็น **prospective** — เอกสารเปิดที่ snapshot tax-profile เก่าเก็บ snapshot; เอกสารใหม่อ่านค่าสืบทอดใหม่ Activity log บันทึก reclassification |

State diagram (เรียบง่าย สามสถานะ):

```
[*] ──create──► active ◄──reactivate── inactive
                │                          │
                ├──deactivate──────────────┘
                │
                └──delete──► soft-deleted ──restore──► active

(soft-deleted คือ terminal ในการใช้งานปกติ; restore เป็น action พิเศษของ Product Administrator)
```

| `PRD_XMOD_003` | [[costing]] | Costing อ่าน `tb_product.standard_cost` สำหรับวิธี count-costing `standard` (`enum_physical_count_costing_method = standard`); ดู [[costing/01-data-model]] § 2.6 Costing **ไม่ได้** อ่านวิธีการคิดต้นทุนจากสินค้า — วิธีอยู่ที่ `tb_business_unit.calculation_method` (ตาม [[product/01-data-model]] § 5 รายการ 7) บทบาทของสินค้าใน costing คือแหล่งของต้นทุนอ้างอิงและแหล่งของ recipe-baseline ไม่ใช่ตัวเลือก cost-flow |

| `PRD_XMOD_005` | [[recipe]] | บรรทัดวัตถุดิบของสูตรอ้างอิงสินค้าผ่าน `tb_recipe_ingredient.product_id`; picker filter ไปที่ `is_used_in_recipe = true` (ค่าที่มีผลตาม `PRD_CALC_004`) การปิดใช้สินค้าที่ถูกอ้างอิงโดยสูตรที่ publish trigger soft-block ตาม `PRD_LIFE_002` พร้อม override + recipe-review flag; การลบ hard-block |
| `PRD_XMOD_006` | [[purchase-request]] / [[purchase-order]] | บรรทัด procurement อาจอ้างอิง order-unit ผ่าน `tb_unit_conversion` (พร้อม `unit_type = order_unit`) qty บนบรรทัด PR / PO ถูกเก็บในหน่วยของเอกสาร; ระบบแปลงเป็นหน่วยคลังฐานที่การรับ (`good-receive-note`) ตาม `PRD_CALC_005` Conversion factor ที่ขาดบล็อกการ save บรรทัดด้วย `"No conversion factor defined for unit <X> → <inventory_unit> on product <code>."` |
| `PRD_XMOD_007` | [[good-receive-note]] | `accepted_qty` และ `cost_per_unit` ของ GRN populate `tb_inventory_transaction_cost_layer.in_qty` และ `.cost_per_unit` ตาม [[inventory/02-business-rules]] `INV_XMOD_001` `price_deviation_limit` และ `qty_deviation_limit` ของสินค้า (ค่าที่มีผลตาม `PRD_CALC_003`) gate ว่าบรรทัด GRN ได้รับการอนุมัติอัตโนมัติหรือต้องการการอนุมัติเกินเกณฑ์ |
| `PRD_XMOD_008` | [[store-requisition]] | บรรทัด SR อ้างอิงสินค้า; `tb_product_location.par_qty` ของ store ที่ออกของอาจขับเคลื่อนการกำหนดขนาด requisition อัตโนมัติ การ post SR บริโภคจากคลังที่ `(location, lot)` ของสินค้าตามกฎ FIFO / WA cost-pick (ซึ่งอ่านจาก `tb_business_unit.calculation_method` ไม่ใช่จากสินค้า) |
| `PRD_XMOD_009` | [[physical-count]] / [[spot-check]] | เอกสาร count อ้างอิงสินค้าผ่าน `product_id` การประเมินมูลค่า count-variance อ่านอย่างใดอย่างหนึ่งของ `tb_product.standard_cost` หรือ cost-layer ตาม `enum_physical_count_costing_method` ตาม [[costing/01-data-model]] § 2.6 การจำแนกของสินค้า (category → item-group) ขับเคลื่อนการ group count-sweep สำหรับการนับสต๊อกเต็ม |
| `PRD_XMOD_010` | [[inventory-adjustment]] | เอกสาร stock-in / stock-out ด้วยมืออ้างอิงสินค้า; in-use guard ของการลบสินค้า (`PRD_LIFE_004`) เห็นเอกสารเหล่านี้เป็น reference เมื่อ `doc_status` ของพวกเขาไม่ใช่ terminal |
| `PRD_XMOD_011` | Activity log / audit | ทุกการเปลี่ยน `tb_product` (create, update, การเปลี่ยนสถานะ, soft-delete, restore) เขียน entry activity-log entry ตาราง comment (`tb_product_comment`) มีการอภิปรายที่ user driven; activity log เป็น system-driven surface ทั้งสองสามารถ query ได้โดย Auditor (`PRD_AUTH_011`) |
| `PRD_XMOD_012` | Integration ภายนอก (POS, e-commerce, vendor catalog feed, sustainability DB) | Outbound: product master ถูกอ่านโดย POS / e-commerce สำหรับการ publish menu / catalog (filter ไปที่ `is_sold_directly = true` สำหรับ POS) Inbound: vendor catalog feed อาจสร้าง / update แถว `tb_product_tb_vendor` อัตโนมัติภายใต้ system user; sustainability DB อาจ update JSON `tb_product.certification` Integration endpoint ตั้งค่าโดย Sysadmin ตาม `PRD_AUTH_010`; การเขียนของ integration subject ต่อกฎ validation `PRD_VAL_*` เดียวกัน |
