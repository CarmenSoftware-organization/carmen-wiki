# สินค้า (Product) — Test Scenarios — Product Admin

> **At a Glance**
> **Persona:** Product Administrator &nbsp;·&nbsp; **โมดูล:** [product](/th/inventory/product) &nbsp;·&nbsp; **scenario:** ~59
> **หมวด:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Lifecycle &nbsp;·&nbsp; Bulk &nbsp;·&nbsp; Edge Case
> **การครอบคลุม E2E:** map ไปยัง `100-product.spec.ts`, `101-product-category.spec.ts` ใน `../carmen-inventory-frontend-e2e/` (surface CRUD ส่วนใหญ่ manual / planned)

หน้านี้บันทึก test scenario ที่ persona Product Administrator ขับเคลื่อนตรงในโมดูล `product` พวกเขาเป็นเจ้าของ CRUD เต็มบน `tb_product`, การจำแนก (`tb_product_category` / `tb_product_sub_category` / `tb_product_item_group`), หน่วย (`tb_unit`), conversion (`tb_unit_conversion`), location mapping (`tb_product_location`), vendor mapping (`tb_product_tb_vendor`) และรัน bulk import / export การเป็นเจ้าของด้านแคตตาล็อกของพวกเขาเริ่มที่ create และจบที่ soft-delete (หรือ restore โดยพิเศษ) scenario จัดกลุ่มเป็น **happy path** (สร้างเดี่ยว CRUD การจำแนก การนิยามหน่วยและ conversion location mapping vendor mapping bulk import สำเร็จ) **RBAC** (scope Product Administrator vs scope Cost Controller / Inventory Controller การ segregate of duties บนการเปลี่ยน standard-cost) **validation** (negative test เทียบกับ `PRD_VAL_001`–`PRD_VAL_018`) **lifecycle** (ปิดใช้ / re-activate / soft-delete / restore พร้อม in-use guard `PRD_LIFE_*`) **bulk operation** (import dry-run, error report, partial-success, strict-commit) และ **edge case** (การแก้พร้อมกัน การ override การสืบทอด การจัดระเบียบใหม่ของการจำแนก restore-vs-recreate hard-disable) handoff ข้าม persona ที่ pivot จาก Product Administrator (Scenario 1, 2, 3, 4, 5, 6, 7, 11, 12, 13, 14, 16 ใน parent overview) อยู่ใน [04-test-scenarios.md](./04-test-scenarios.md) ไม่ใช่ที่นี่

## 2. Permission / Authorization

| # | Scenario | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| PA-PERM-01 | Product Administrator CRUD บน product master | **Allow** ตาม `PRD_AUTH_001` CRUD เต็มบนแถว `tb_product` subject ต่อกฎ validation `PRD_VAL_*` และ lifecycle guard `PRD_LIFE_*` |
| PA-PERM-02 | Product Administrator ตั้งค่า `tb_unit` และ `tb_unit_conversion` | **Allow** ตาม `PRD_AUTH_002` subject ต่อ in-use guard `PRD_VAL_017` ของการลบหน่วย |
| PA-PERM-03 | Product Administrator รัน import / export | **Allow** ตาม `PRD_AUTH_003` import job รันภายใต้ `user_id` ของ Administrator; การ export ฟิลด์ที่ละเอียดอ่อน (`standard_cost`, vendor mapping) ได้รับอนุญาตให้ role นี้ |
| PA-PERM-04 | Product Administrator อนุมัติการเปลี่ยน standard-cost ของตนเองเกินเกณฑ์ SoD | **Deny — ต้องการ signature ที่สอง** ตาม `PRD_AUTH_012` ผู้อนุมัติการเปลี่ยน `standard_cost` เกินเกณฑ์ SoD ของ tenant ต้องไม่ใช่ user คนเดียวกับที่ submit การ submit stage การเปลี่ยน; การอนุมัติต้องการ Cost Controller หรือ Finance การส่ง API ตรงที่พยายาม bypass return `"Self-approval of standard-cost change above SoD threshold is not permitted."` |
| PA-PERM-05 | Product Administrator แก้นโยบายตัวเลข `tb_product_location` (min / max / par / reorder) | **อนุญาตที่ระดับ schema — แต่ตามธรรมเนียมเป็นอำนาจ Inventory-Controller** ตาม `PRD_AUTH_004` Product Administrator สามารถสร้างแถว mapping เพื่อเปิดสินค้าที่ location; ค่าตัวเลข (`min_qty` / `max_qty` / `par_qty` / `re_order_qty`) ตามธรรมเนียมดูแลโดย Inventory Controller ตาม [inventory/02-business-rules](/th/inventory/inventory/02-business-rules) `INV_AUTH_004` Tenant workflow อาจบังคับใช้สิ่งนี้ด้วย UI gate; schema อนุญาต PA เขียนค่า |
| PA-PERM-06 | Product Administrator พยายามอนุมัติบรรทัด PR / PO / GRN | **Deny — นอกขอบเขต** อำนาจของ Product Administrator อยู่บน product master ไม่ใช่บนเอกสารธุรกรรม การอนุมัติ PR / PO / GRN / SR ต้องการ role ของโมดูลตามลำดับ (Purchaser / Approver, Receiver ฯลฯ) |
| PA-PERM-07 | Product Administrator พยายาม lock accounting period | **Deny — ต้องการ Finance Manager** ตาม [inventory/02-business-rules](/th/inventory/inventory/02-business-rules) `INV_AUTH_006` period lock เป็นอำนาจ Finance Manager Product Administrator ไม่มีอำนาจการจัดการ period |
| PA-PERM-08 | สิทธิ์อ่านอย่างเดียวของ Auditor บนสินค้าที่ soft-deleted | **Allow read-only** ตาม `PRD_AUTH_011` Auditor เห็นแถว soft-deleted สำหรับ traceability ไม่มี write authority |
| PA-PERM-09 | System Administrator ตั้งค่า RBAC สำหรับโมดูล product | **Allow** ตาม `PRD_AUTH_010` Sysadmin ตั้งค่า role และ integration endpoint ไม่สามารถแก้ product master โดยตรง (separation of duties) |
| PA-PERM-10 | Product Administrator ดู activity log | **Allow** ตาม `PRD_XMOD_011` activity log query ได้โดย Product Administrator และโดย Auditor |

## 3. Validation / Error

| # | Scenario | Trigger | Error ที่คาดหวัง |
| - | -------- | ------- | -------------- |
| PA-VAL-01 | code สินค้าซ้ำ (`PRD_VAL_001`) | สร้างสินค้าใหม่ด้วย `code = COF-001` ขณะที่สินค้า live อื่นมี code นั้น | **ปฏิเสธที่ submit** — server return `"Product code COF-001 already exists. Choose a different code or restore the existing soft-deleted product."` `409 Conflict` Product Administrator เลือก code อื่นหรือสอบสวนสินค้าที่ soft-deleted |
| PA-VAL-02 | ชื่อสินค้าว่าง (`PRD_VAL_002`) | สร้างสินค้าใหม่ด้วย `name` ว่าง / whitespace | **ปฏิเสธที่ submit** — `"Product name is required."` `400 Bad Request` |
| PA-VAL-03 | เปลี่ยน inventory unit บนสินค้าที่มีประวัติ (`PRD_VAL_003`) | สินค้าที่มีอยู่กับ 10 GRN รับบน `KG`; พยายามเปลี่ยน `inventory_unit_id` เป็น `LB` | **ปฏิเสธที่ submit** — `"Inventory unit cannot be changed for a product with existing inventory history."` `409 Conflict` Administrator ต้อง (หายาก) drain คลัง soft-delete และสร้างใหม่ภายใต้ code ต่างพร้อมหน่วยใหม่ |
| PA-VAL-04 | การอ้างอิงการจำแนกที่ inactive / deleted (`PRD_VAL_004`, `PRD_VAL_009`) | สร้างสินค้าใหม่เลือก `tb_product_item_group_id` ที่ soft-deleted | **ปฏิเสธที่ submit** — `"Item group is required (or selected item group is inactive/deleted)."` `400 Bad Request` Picker ปกติ filter โหนด inactive / deleted ออก; การส่ง API ตรงตรวจสอบใหม่ |
| PA-VAL-05 | Barcode ชน (`PRD_VAL_005`) | สินค้าที่มีอยู่มี `barcode = '8851234567890'`; สร้างสินค้าใหม่ด้วย barcode เดียวกัน | **ปฏิเสธที่ submit** — `"Barcode 8851234567890 is already assigned to product <other_code>."` `409 Conflict` บังคับใช้ที่ application (ไม่มี schema `@unique`) |
| PA-VAL-06 | ค่าความคลาดเคลื่อนนอกช่วง (`PRD_VAL_006`) | ตั้ง `price_deviation_limit = 150` (เกิน 100%) | **ปฏิเสธที่ submit** — `"Deviation limits must be between 0 and 100 percent."` `400 Bad Request` |
| PA-VAL-07 | standard cost ลบ (`PRD_VAL_007`) | ตั้ง `standard_cost = -50` | **ปฏิเสธที่ submit** — `"Standard cost cannot be negative."` `400 Bad Request` |
| PA-VAL-08 | Self-conversion ในแถว unit-conversion (`PRD_VAL_010`) | นิยาม `tb_unit_conversion` ด้วย `from_unit_id = KG`, `to_unit_id = KG` | **ปฏิเสธที่ submit** — `"Conversion factor must reference two distinct units and have positive qty on both sides."` `400 Bad Request` |
| PA-VAL-09 | ความไม่สอดคล้องสองทิศทางใน conversion (`PRD_VAL_011`) | conversion ที่มีอยู่ `1 KG = 1000 G`; พยายามเพิ่ม `1 KG = 1100 G` สำหรับสินค้า / unit_type เดียวกัน | **ปฏิเสธที่ submit** — `"Conversion factor is inconsistent with existing conversions for this product. Expected 1000, got 1100."` `409 Conflict` การตรวจสอบระดับ application ข้ามแถว |
| PA-VAL-10 | Min > max บนนโยบาย location (`PRD_VAL_012`) | ตั้ง `min_qty = 50`, `max_qty = 20` บนแถว `tb_product_location` | **ปฏิเสธที่ submit** — `"Min must not exceed max; reorder qty should be at or above min."` `400 Bad Request` |
| PA-VAL-11 | vendor mapping ซ้ำ (`PRD_VAL_013`) | แถว `tb_product_tb_vendor` ที่มีอยู่สำหรับ `(vendor_id = V1, product_id = P1)`; พยายามเพิ่มอีก | **ปฏิเสธที่ submit** — `"Vendor V1 is already mapped to product P1."` `409 Conflict` |
| PA-VAL-12 | tax profile inactive (`PRD_VAL_014`) | ตั้ง `tb_product.tax_profile_id` เป็น `tb_tax_profile` ที่ soft-deleted | **ปฏิเสธที่ submit** — `"Selected tax profile is inactive or deleted."` `400 Bad Request` Picker ปกติ filter |
| PA-VAL-13 | การเปลี่ยนสถานะผิด (`PRD_VAL_015`) | API call ตรงพยายาม `product_status_type = 'discontinued'` | **ปฏิเสธที่ submit** — `"Product status must be 'active' or 'inactive'."` `400 Bad Request` ค่า `discontinued` ไม่อยู่ใน enum (ตาม [product/01-data-model](/th/inventory/product/01-data-model) § 5 รายการ 4) |
| PA-VAL-14 | JSON ผิดรูปแบบใน info / dimension / certification (`PRD_VAL_016`) | การส่ง API ด้วย JSON ผิดรูปแบบใน `info` | **ปฏิเสธที่ submit** — `400 Bad Request` (JSONB syntax error) การละเมิดกฎรูปทรงเป็น application-layer |
| PA-VAL-15 | ลบหน่วยที่ในใช้ (`PRD_VAL_017`) | พยายาม soft-delete `tb_unit` "POUND" ที่ถูกอ้างอิงโดย 12 สินค้า | **ปฏิเสธที่ submit** — `"Unit POUND is in use by 12 products / 30 conversions / N document lines and cannot be deleted."` `409 Conflict` Map ไปยัง parent Scenario 14 |
| PA-VAL-16 | ลบหมวดหมู่พร้อมลูก (`PRD_VAL_018`) | พยายาม soft-delete `tb_product_category` "Beverages" พร้อม 5 sub-category ใต้ | **ปฏิเสธที่ submit** — `"Category cannot be deleted: 5 sub-categories / N item-groups / M products still classified beneath it."` `409 Conflict` |

## 4. Lifecycle

## 5. Edge Cases

| # | Scenario | เงื่อนไข | คาดหวัง |
| - | -------- | --------- | -------- |
| PA-EDGE-01 | การแก้พร้อมกันบน header สินค้าเดียวกัน | Administrator สองคนเปิดสินค้า P1 พร้อมกัน; ทั้งสองแก้ `description`; A save ที่ T; B save ที่ T+5s ด้วย stale data | คอลัมน์ audit ของสินค้ารวม `updated_at` / `updated_by_id` แต่ไม่มีฟิลด์ `version` (ตาม [product/01-data-model](/th/inventory/product/01-data-model) — ไม่มีคอลัมน์ optimistic-concurrency บน `tb_product`) Last-write-wins; การแก้ของ B overwrite ฟิลด์ `description` ของ A Best practice: ธรรมเนียม tenant บังคับ review ร่วมผ่าน comment ก่อนแก้; audit log รักษาทั้งสองการเปลี่ยน |
| PA-EDGE-02 | การ override การสืบทอดที่ระดับสินค้า | หมวดหมู่มี `tax_profile_id = TAX-7`; สินค้า override ด้วย `tax_profile_id = TAX-0` (ยกเว้น) | tax profile ที่มีผลตาม `PRD_CALC_002` อ่านระดับสินค้าก่อน; ผลลัพธ์คือ `TAX-0` Activity log บันทึก override |
| PA-EDGE-03 | ค่าสืบทอดทุกระดับการจำแนกเป็น null | สินค้าไม่มี `tax_profile_id`; item-group ไม่มี; sub-category ไม่มี; category ไม่มี | tax profile ที่มีผลคือ `null` ตาม `PRD_CALC_002`; อัตราที่มีผล 0% ("ยกเว้น") บรรทัด procurement / receiving คำนวณภาษีที่ 0% |
| PA-EDGE-04 | Multi-hop conversion ผ่านเส้นทาง transitive | Conversion ที่นิยาม: `1 BAG = 5 KG` และ `1 KG = 1000 G`; ไม่มีแถว `BAG → G` ตรง บรรทัด PR กรอกใน `BAG` | ตาม `PRD_CALC_006`: engine จัด `1 BAG = 5 KG × 1000 G/KG = 5000 G` Conversion สำเร็จ ถ้าแถว `BAG → G` ตรงมีอยู่ด้วย factor ต่าง ความไม่สอดคล้อง flag ตาม `PRD_VAL_011` |
| PA-EDGE-05 | Precision ทศนิยมบน unit conversion | `from_unit_qty = 1`, `to_unit_qty = 0.12345` (5dp) | เก็บที่ 5dp บน `Decimal(20, 5)` Conversion factor ที่มีผล `0.12345` แสดงปัดเศษตาม `tb_unit.decimal_place` (โดยทั่วไป 2dp → `0.12`); การคำนวณกลางถือ 5dp |
| PA-EDGE-06 | Hard-disable พร้อม `product_status_type` ยัง active | API ตรงตั้ง `is_active = false` แต่เหลือ `product_status_type = active` | ธรรมเนียม application บังคับใช้ `is_active = false → product_status_type = inactive` ตาม `PRD_LIFE_005` การรวมได้รับอนุญาตที่ระดับ schema แต่ application normalise ที่การอ่าน (สินค้า inactive อย่างมีประสิทธิภาพ) Best practice: backend service hook คู่คอลัมน์ให้สอดคล้องกัน |
| PA-EDGE-07 | Sub-category พร้อม code string ว่าง (quirk ของ schema) | sub-category `code = ""` ตาม [product/01-data-model](/th/inventory/product/01-data-model) § 5 รายการ 10 | อนุญาตที่ระดับ schema (`@default("")`) UI ของ application ต้องการ code ไม่ว่างที่ create; แถว legacy / migrate อาจมี code ว่าง constraint unique `productsubcategory_code_name_u` กำหนดขอบเขตความไม่ซ้ำบน `(code, name, deleted_at)` ดังนั้นสอง sub-category พร้อม code ว่างได้รับอนุญาตถ้าชื่อต่างกัน |
| PA-EDGE-08 | Restore สินค้าที่ soft-deleted พร้อมห่วงโซ่การจำแนกถูกลบบางส่วน | สินค้าที่ soft-deleted P1; item-group parent ของมันถูก soft-deleted ในระหว่างนั้น (ไม่มีสินค้าอื่นอ้างอิง ดังนั้นการลบสำเร็จ) | Restore ตาม `PRD_LIFE_009` validate ห่วงโซ่การจำแนก ถ้า item-group parent ยังคง soft-deleted restore ถูกปฏิเสธด้วย `"Parent item group is inactive or deleted."` Administrator ต้อง restore item-group ก่อน แล้วสินค้า |
| PA-EDGE-09 | Bulk export ด้วยฟิลด์ที่ละเอียดอ่อน | Administrator export แคตตาล็อกรวม `standard_cost` และ vendor mapping สำหรับการส่ง audit | ตาม `PRD_AUTH_003` role Administrator อนุญาตการ export ฟิลด์ที่ละเอียดอ่อน Export job log คำขอ; audit log บันทึก `{ event: 'export', fields: [...], by: pa@blueledgers.com, at: <ts> }` Export คู่ขนานของ Auditor ตาม `PRD_AUTH_011` ต้องการการอนุมัติรองตามธรรมเนียม tenant |
| PA-EDGE-10 | ปริมาณ activity log บนสินค้าที่ถูกแก้บ่อย | สินค้าเดียวรับ standard-cost update 50 ครั้ง/เดือนจาก supplier feed integration | entry activity-log ทั้ง 50 รักษาไว้; ไม่มีการ auto-prune ของ activity log ภายในหน้าต่างการเก็บข้อมูลมาตรฐาน การ query Audit ตาม `PRD_AUTH_011` return ประวัติเต็ม Tenant อาจตั้งค่าการเก็บถาวร activity-log ตามนโยบายการเก็บข้อมูล |
