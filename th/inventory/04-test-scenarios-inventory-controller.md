# คลังสินค้า (Inventory) — Test Scenarios — Inventory Controller

> **At a Glance**
> **Persona:** Inventory Controller &nbsp;·&nbsp; **โมดูล:** [[inventory]] &nbsp;·&nbsp; **Scenarios:** ~26
> **หมวด:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** map ไปยัง `720-stock-issue.spec.ts`, `900-period-end.spec.ts` ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ test scenarios ที่ Inventory Controller persona ขับโดยตรงในโมดูล `inventory` Controller เป็นเจ้าของความถูกต้องของยอด: อนุมัติเอกสาร `tb_stock_in` / `tb_stock_out` เหนือ Store Keeper auto-approve threshold ตาม `INV_AUTH_003`, commit count-variance rollups ที่เกิดจาก physical / spot counts ตาม `INV_XMOD_003` / `INV_XMOD_004`, ดูแล policy สต๊อกต่อสินค้า / ต่อ location บน `tb_product_location` ตาม `INV_AUTH_004` และเซ็นรับรอง variance ก่อนปิดงวด ความเป็นเจ้าของโมดูล inventory ของพวกเขาเริ่มเมื่อ Store Keeper escalate เอกสารหรือ count complete และจบเมื่อ (a) เอกสาร post บนการอนุมัติ Controller (ต่ำกว่า Finance threshold), (b) เอกสาร escalate ต่อไปยัง Finance สำหรับผลกระทบต้นทุนใหญ่มาก, (c) count commit และ rollup post หรือ (d) บันทึกการเซ็นรับรอง variance ปิดงวด Scenarios จัดกลุ่มเป็น **happy paths** (อนุมัติ, count-variance commit, แก้ไข stock-policy, เซ็นรับรองปิดงวด), **RBAC** (Controller scope vs Store Keeper / Finance / Sysadmin, ป้องกัน self-approval), **validation** (negative tests บน approval gates) และ **edge cases** (concurrent approvals, count rollup ด้วยผลผสม, routing threshold-boundary)

## 2. Permission / Authorization

| - | -------- | --------------------------------------- |
| IC-PERM-01 | Controller อนุมัติ adjustment เหนือ Store-Keeper-threshold | **Allow ตาม `INV_AUTH_003`** ปุ่ม **Approve** enable บนเอกสารที่ `in_progress` ภายใน threshold ของ Controller; คลิก fire การ post ต่ำกว่า Controller threshold แต่เหนือ Store Keeper auto-approve — Controller คือผู้อนุมัติ เหนือ Controller threshold — route ไปยัง Finance ตาม Scenario IC-PERM-04 |

## 3. Validation / Error

| # | Scenario | Trigger | Error ที่คาดหวัง |
| - | -------- | ------- | -------------- |
| IC-VAL-01 | Controller อนุมัติ stock-out ที่จะขับ on-hand ติดลบ | เอกสาร `tb_stock_out` submit ขณะที่ on-hand คือ `10`; ระหว่าง submit และ approve, stock-out อื่น post ลด on-hand เป็น `3`; `qty = 7` ของเอกสารจะขับ on-hand ติดลบตอนนี้ | **Reject ที่ approve** — `INV_VAL_005` Server return `"Outbound movement would drive on-hand below zero at (location, product, lot). Available: 3, requested: 7."` เอกสารอยู่ `in_progress`; Controller route เอกสารกลับไปยัง Store Keeper พร้อมสถานะใหม่ หรือ split / ลด qty ในนามของ Store Keeper (ด้วย edit-and-approve action) ตาม auth model ของระบบ |
| IC-VAL-02 | Controller อนุมัติ adjustment ในงวด `closed` ตอนนี้ | เอกสาร submit ขณะงวด `open`; ระหว่าง submit และ approve, งวด auto-close (rare race กับ job ปิดงวด scheduled); วันที่ transaction อยู่ในงวด `closed` ตอนนี้ | **Reject ที่ approve** — `INV_VAL_008` Server return `"Cannot post into period <YYMM>: period is closed."` Controller อัปเดตวันที่ transaction เป็นงวด open ปัจจุบัน (ปกติด้วยการเห็นด้วยของ Store Keeper) และ re-approve หรือ route ไปยัง Finance Manager เพื่อ re-open งวด closed สำหรับกรณีพิเศษ |
| IC-VAL-03 | Controller อนุมัติ count rollup ด้วย aggregate ไม่ตรง | Count rollup ที่ผลกระทบต้นทุน aggregate เปลี่ยนระหว่างเวลา count-commit และเวลา Controller-confirm (เพราะการเปลี่ยน physical inventory interim เปลี่ยน system qty) | **Reject ที่ commit** — defensive guard: server return `"Count rollup aggregate has changed since review. Refresh the variance dashboard and re-commit."` Controller refresh count, re-review variance lines และ re-commit ด้วย aggregate ปัจจุบัน |
| IC-VAL-04 | Controller พยายาม commit count ด้วย unresolved `re-count` lines | Dashboard count แสดง count ด้วยหนึ่งหรือมากกว่า lines flag `re-count`; Controller คลิก Commit อย่างไรก็ตาม (mis-click) | **Reject ที่ commit** — server return `"Count cannot be committed while <N> lines are flagged for re-count. Resolve the re-count lines (Store Keeper re-counts) before committing."` Controller รอ re-count เสร็จหรือลบ flag `re-count` บน line (escalate เป็น `post` หรือ `investigate`) และ re-commit |
| IC-VAL-05 | Controller พยายามแก้ policy ด้วย bounds ไม่ถูกต้อง | Controller ตั้ง `min_qty = 20`, `max_qty = 15` บน `tb_product_location` (max < min — ไม่ coherent) | **Reject ที่ save** — shape validation: server return `"max_qty must be ≥ min_qty on stock-policy row."` Controller แก้ค่าหนึ่งและ re-save (Shape rules ที่คล้าย: `re_order_qty ≥ 0`, `par_qty ≥ 0`) |
| IC-VAL-06 | Controller เซ็นรับรองงวดด้วยเอกสารที่อยู่ระหว่างทาง | Controller คลิก **Sign off variance review for 2026-04** ขณะที่เอกสาร `tb_stock_out` สำหรับงวดยังอยู่ที่ `doc_status = in_progress` (ยังไม่ approve) | **Reject ที่ sign-off** — server return `"Cannot sign off period <2026-04> while <N> in-flight inventory documents remain at draft or in_progress."` Controller resolve เอกสารที่อยู่ระหว่างทาง (approve หรือ reject แต่ละ) ก่อน re-sign-off Finance period-close run respect gate เดียวกัน |

## 4. Edge Cases

| # | Scenario | Condition | Expected |
| - | -------- | --------- | -------- |
| IC-EDGE-01 | Threshold boundary บน routing อนุมัติ Controller | เอกสาร aggregate cost impact = `฿5,000.00` (Finance threshold พอดี) | **Boundary inclusive — ที่ threshold, การอนุมัติของ Controller คือ terminal** ระบบถือว่า `≤` เป็น Controller-final `฿5,000.01` จะ route ไปยัง Finance Boundary-inclusive rule เดียวกันกับ SK-EDGE-01 |
| IC-EDGE-02 | Count-rollup ด้วย overage และ shortage บนสินค้าเดียวกันข้าม lots | Count ระบุ `LOT-1` short 3, `LOT-2` over 2 สำหรับ `P-1` ที่ `LOC-A` | **สองเอกสาร rollup ตาม parent Scenario 4 / SK-EDGE-05** Single `tb_stock_in` (overage บน `LOT-2`, qty 2) และ single `tb_stock_out` (shortage บน `LOT-1`, qty 3); ไม่ netted Controller อนุมัติทั้งสองใน commit เดียว รักษา audit ต่อ lot |
| IC-EDGE-03 | Approvals concurrent บนเอกสารต่างใน queue | Controller A approve เอกสาร `D-1` ที่ `T`; Controller B (peer) approve เอกสาร `D-2` (`(location, product, lot)` ต่างกัน) ที่ `T+200ms` | **ทั้งสองสำเร็จ — ไม่มี row contention ที่แชร์** แต่ละการอนุมัติคือ row-level บนเอกสาร; transactions ทั้งสอง post; ไม่มี conflict Peer-controller model รองรับผ่าน RBAC — Controllers หลายคนอาจมีสิทธิ์อนุมัติเดียวกันบน scope ที่ทับซ้อนหรือไม่ทับซ้อน |
| IC-EDGE-04 | Approval concurrent บนเอกสารเดียวกันโดยสอง Controllers | Controllers ทั้งสองคลิก Approve บน `D-1` พร้อมกัน | **คลิกแรกชนะ; ที่สอง reject** ผ่าน optimistic concurrency คลิกที่แพ้ return `"This document was approved by another user at <timestamp>. Please refresh."` เอกสาร `completed`; หนึ่ง inventory transaction; ชุด cost-layer rows เดียว; ไม่มี double-post |
| IC-EDGE-05 | แก้ stock-policy ระหว่าง replenishment ที่อยู่ระหว่างทาง | Purchase request ถูก raise สำหรับ `P-1` ที่ `LOC-A` ตาม `min_qty = 10` เก่า; Controller raise `min_qty` เป็น 15 ขณะที่ PR ยัง `draft` | **PR ไม่ได้รับผลกระทบโดย policy snapshot rule** PR เก็บ snapshot create-time ของ policy (`min_qty = 10` ที่ create); policy ใหม่ใช้ prospective กับ raises PR ต่อมา การแก้ไขของ Controller บน `tb_product_location` สำเร็จ; การประเมิน PR ที่อยู่ระหว่างทางกับ `min_qty` ไม่ re-run |
| IC-EDGE-06 | Sign-off ปิดงวดด้วย count line ที่ flag `investigate` | Count line flag `investigate` โดย Controller ก่อนหน้าในงวด; resolution ยัง pending (Finance review); Controller พยายาม sign-off | **Reject ที่ sign-off** ตาม IC-VAL-06 variant: server return `"Cannot sign off period <2026-04> while <1> count variance line is flagged investigate (route via Finance)."` Resolution เกิดก่อน sign-off: หรือ Finance clear flag investigate (มีหรือไม่มี post) หรือ Controller un-flag หลังจาก resolution |
