# การปรับสต๊อก (Inventory Adjustment) — Test Scenarios — Store Keeper

> **At a Glance**
> **Persona:** Store Keeper &nbsp;·&nbsp; **โมดูล:** [inventory-adjustment](/th/inventory/inventory-adjustment) &nbsp;·&nbsp; **Scenarios:** ~38
> **ประเภท:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **การครอบคลุม E2E:** map ไปยัง `501-grn.spec.ts`, `720-stock-issue.spec.ts` ใน `../carmen-inventory-frontend-e2e/`

หน้านี้ capture test scenarios ที่ persona Store Keeper ขับเคลื่อนโดยตรงในโมดูล `inventory-adjustment` Store Keeper ริเริ่มเอกสาร `tb_stock_in` (ขาเข้า write-on) และ `tb_stock_out` (ขาออก write-off), แนบหลักฐาน, เลือก reason code, กรอกข้อมูล product / qty / lot และ submit — ลงที่ `completed` (ต่ำกว่า threshold, ไม่ใช่ new-lot) หรือ `in_progress` (เหนือ threshold, new-lot หรือ reasons requires-quality-check) Scenarios group เป็น **happy paths** (auto-approve stock-in / stock-out, การ route new-lot, การ submit multi-line, lot override สำหรับ expiry write-off), **RBAC / permission** (scope Store Keeper vs scope Inventory Controller, SoD บน receive-then-write-off), **validation** (เทสลบต่อ `ADJ_VAL_001`–`ADJ_VAL_014` ที่ Store Keeper สามารถ trigger ที่เวลา submit) และ **edge cases** รอบขอบ threshold, ความแม่นยำทศนิยม, posts พร้อมกัน, gate location-type และผลของ reason-flag Handoffs ข้าม persona ที่ pivot จาก Store Keeper (Scenarios 1, 2, 4, 9, 10, 11, 14, 15, 16 ในภาพรวม parent) อยู่ใน [04-test-scenarios.md](./04-test-scenarios.md) ไม่ใช่ที่นี่

## 2. Permission / Authorization

| # | Scenario | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| SK-PERM-01 | Store Keeper สร้าง stock-in / stock-out ภายใน scope, ต่ำกว่า threshold | **Allow** ตาม `ADJ_AUTH_001` / `ADJ_AUTH_002`, อำนาจสร้างภายใน scope `tb_user_location`; ต่ำกว่า threshold auto-approve `draft → completed` ตาม `ADJ_POST_001` / `ADJ_POST_002` |
| SK-PERM-02 | Store Keeper พยายามอนุมัติเอกสาร `in_progress` ของ Store Keeper อื่น | **Deny — Inventory Controller required** ตาม `ADJ_AUTH_004`, การเปลี่ยน `in_progress → completed` เหนือ auto-approve ต้องการ Inventory Controller (หรือ Finance สำหรับ above-Controller-threshold) API call return `"Approval of above-threshold adjustments requires the Inventory Controller role."` เอกสารคงอยู่ `in_progress` |
| SK-PERM-03 | Store Keeper พยายามแก้ไขเอกสาร `completed` | **Deny — terminal state** ตาม `ADJ_VAL_013` และ [inventory](/th/inventory/inventory) `INV_POST_012`, เอกสาร `completed` เป็น immutable `PATCH /stock-in/{id}` (หรือ `/stock-out/{id}`) return `"Cannot edit a completed adjustment. Void and create a new compensating adjustment."` |
| SK-PERM-04 | Store Keeper พยายาม post ไปยัง location นอก scope `tb_user_location` | **Deny — scope** Picker filter ไปยัง scope ของผู้ใช้; การ submit API โดยตรงด้วย `location_id` นอก scope return `"You do not have permission to post to location <LOC-X>."` `ADJ_VAL_003` รันที่ submit; scope check ที่ gateway ทั้งสองเลเยอร์ block |
| SK-PERM-05 | Store Keeper พยายาม post ไปยัง location direct-cost หรือ inactive | **Deny ตาม `ADJ_VAL_003` / `ADJ_POST_007`** locations direct-cost reject; locations inactive filter จาก picker; API โดยตรง return validation error |
| SK-PERM-06 | การละเมิด SoD — Store Keeper write off receipt ของตัวเองเหนือ threshold | **Deny ตาม `ADJ_AUTH_010`** เมื่อ `tb_stock_out.created_by_id` ตรงกับ `tb_good_received_note.created_by_id` ของ GRN ต้นทางของ lot และต้นทุนเหนือ SoD threshold submit return `"You created the receipt for this lot; an independent adjuster must initiate the write-off (SoD)."` การ write-off ขนาดเล็กตามปกติได้รับการยกเว้น Map ไปยัง parent Scenario 14 |
| SK-PERM-07 | Store Keeper พยายาม post เข้างวด `closed` หรือ `locked` | **Deny ตาม `ADJ_VAL_011` / [inventory](/th/inventory/inventory) `INV_VAL_008`** Return `"Cannot post into period <YYMM>: period is <closed/locked>."` เอกสารคงอยู่ `draft` Store Keeper เปลี่ยนวันที่หรือ escalate ไปยัง Finance Manager re-open (เฉพาะ closed) Map ไปยัง parent Scenario 10 |
| SK-PERM-08 | Store Keeper พยายามใช้ reason ของทิศทางผิด | **Deny ตาม `ADJ_VAL_002`** เอกสาร `tb_stock_in` ด้วย reason ทิศทาง `STOCK_OUT` (หรือในทางกลับกัน) ถูก reject: `"Adjustment reason is required and must match the document direction."` Picker ปกติ filter; การ call API โดยตรง re-check Map ไปยัง parent Scenario 15 |
| SK-PERM-09 | Store Keeper พยายาม void เอกสาร `completed` โดยตรง | **Deny — ต้องใช้ compensating reversal** ตาม `ADJ_POST_004`, void เป็น action ของ Controller / Finance และต้องการ compensating reversal ก่อน SK API call return `"Voiding a completed adjustment requires the Inventory Controller or Finance role and proceeds via a compensating reversal."` |

## 3. Validation / Error

| # | Scenario | Trigger | Expected error |
| - | -------- | ------- | -------------- |
| SK-VAL-01 | เลขที่เอกสาร collision บน auto-generation race (`ADJ_VAL_001`) | สอง simultaneous submits ชน `si_no` เดียวกันหลังจาก 3x retry | Submit fail ด้วย `"Stock-in number <si_no> already exists for this tenant."` UI reload ด้วย `si_no` ใหม่ผ่าน numbering service |
| SK-VAL-02 | Reason code ขาดหรือ direction mismatch (`ADJ_VAL_002`) | `adjustment_type_id = null` หรือ `tb_adjustment_type.type` mismatch ทิศทางเอกสาร | Reject ที่ save / submit ด้วย `"Adjustment reason is required and must match the document direction."` |
| SK-VAL-03 | Location ขาดหรือ inactive (`ADJ_VAL_003`) | `location_id = null` หรือ location ที่อ้างอิงมี `is_active = false` / `deleted_at` set / `location_type = direct` | Reject ด้วย `"Location is required and must be an inventory- or consignment-type location."` (null/inactive); `"Direct-cost locations cannot be the target of an adjustment."` (direct-type) |
| SK-VAL-04 | Description ว่างที่ submit (`ADJ_VAL_004`) | ฟิลด์ description เว้นว่างเมื่อ submit | Reject ที่ submit ด้วย `"Description is required for audit purposes."` (Soft-fail ที่ save พร้อม warning) |
| SK-VAL-05 | แผนก / cost-centre ขาดใน `dimension` (`ADJ_VAL_005`) | JSON `dimension` ว่างหรือไม่มี entry `department`; tenant policy require | Reject ที่ submit ด้วย `"Department / cost-centre is required (set via dimension)."` |
| SK-VAL-06 | Product inactive หรือ product ไม่เปิดที่ location (`ADJ_VAL_006`) | `product_id` อ้างอิง `tb_product` inactive หรือไม่มีแถว `tb_product_location` | Reject ด้วย `"Product <product_code> is not active or not enabled at location <location_code>."` |
| SK-VAL-07 | ปริมาณ zero หรือ negative (`ADJ_VAL_007`) | Line `qty = 0` หรือ `qty < 0` | Reject ด้วย `"Quantity must be greater than zero on every line."` |
| SK-VAL-08 | ต้นทุน negative (`ADJ_VAL_008`) | Line `cost_per_unit < 0` (โดยทั่วไป client-side validation bypass attempt) | Reject ด้วย `"Cost per unit must be non-negative."` |
| SK-VAL-09 | Lot identity collision หรือ expiry ขาด (`ADJ_VAL_009`) | Stock-in new-lot collision กับ `(product, location, lot_no)` ที่มีอยู่; หรือ stock-out จาก lot ที่ไม่มี; หรือเน่าเสียง่าย new-lot โดยไม่มี expiry | Reject ด้วยข้อความ error ที่ตรงกัน (`"Lot <lot_no> already exists..."` / `"Lot <lot_no>... is not available for consumption."` / `"Expiry date is required for perishable product <product> on new lot <lot_no>."`) |
| SK-VAL-10 | Required-document ขาด (`ADJ_VAL_010`) | Reason flag `info.requiresDocument = true`; ไม่มี comment ด้วย JSON `attachments` ที่ไม่ว่าง | Reject ที่ submit ด้วย `"Supporting document attachment is required for this adjustment reason."` Map ไปยัง parent Scenario 16 |
| SK-VAL-11 | วันที่ย้อนหลังเข้างวด closed (`ADJ_VAL_011`) | `si_date` / `so_date` ในงวด `closed` | Reject ด้วย `"Cannot post into period <YYMM>: period is closed."` Map ไปยัง parent Scenario 10 |
| SK-VAL-12 | Negative-balance บน stock-out (`ADJ_VAL_012`) | Stock-out `qty` > ที่มีที่ lot ที่เลือก | Reject ด้วย `"Outbound movement would drive on-hand at (LOC-A, P-1, LOT-1) below zero. Available: 5, requested: 7."` เอกสารคงอยู่ `draft` SK ลด qty / เลือก lot อื่น / escalate Map ไปยัง parent Scenario 9 |
| SK-VAL-13 | ความพยายามแก้ไขบนเอกสาร `completed` (`ADJ_VAL_013`) | `PATCH` ไปยัง `tb_stock_in.id` ที่ `doc_status = completed` | Reject ด้วย `"Cannot edit a completed adjustment. Void and create a new compensating adjustment."` |
