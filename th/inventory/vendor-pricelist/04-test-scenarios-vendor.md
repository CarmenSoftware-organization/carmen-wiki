# รายการราคาผู้ขาย (Vendor Pricelist) — Test Scenarios — Vendor

> **At a Glance**
> **Persona:** Vendor (external party — portal ที่ authenticate ด้วย token เท่านั้น) &nbsp;·&nbsp; **โมดูล:** [[vendor-pricelist]] &nbsp;·&nbsp; **Scenario:** ~25
> **Categories:** Happy Path &nbsp;·&nbsp; Permission (N/A) &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** ไม่มี — vendor portal เป็น surface สาธารณะแยก; coverage ที่ระดับ API / integration (token middleware, output validator, auto-save) รอ roadmap item ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ test scenario ที่ exercise การ interact ของ persona **Vendor** กับโมดูล `vendor-pricelist` ไม่เหมือนกับ persona external ส่วนใหญ่ใน Carmen suite vendor ที่นี่ขับ system state โดยตรงผ่าน **portal session ที่ authenticate ด้วย token** — ที่เดียวที่ external party ปฏิบัติการภายในระบบ Carmen ด้วยผลคงทนใน-ระบบ ([03-user-flow-vendor.md](./03-user-flow-vendor.md) Section 1) Surface ของ vendor มีขอบเขต: เปิด portal ผ่าน URL ต่อ vendor, หยิบสกุลเงิน submission, ป้อนหรือ upload ราคาสำหรับรายการสินค้าของ template, save draft (auto-save ทุก ~30 วินาที) และคลิก Submit การ approve, reject, inactivation, expiry และการควบคุมนโยบาย token ทั้งหมดขับโดย persona ภายใน (Purchaser / Manager / Finance Manager / Sysadmin / cron) เพราะ vendor ไม่มี Carmen login และไม่มี matrix RBAC ฝั่ง Carmen section Permission / Authorization ด้านล่างลดลงเป็นแถว N/A เดียวโดยตั้งใจ — นโยบาย portal-token เองถูก test เป็นข้อกังวลการตั้งค่าฝั่ง Sysadmin ใน [04-test-scenarios-audit-config.md](./04-test-scenarios-audit-config.md) และการแยกหน้าที่ (`VPL_AUTH_014`) ตรวจสอบจากฝั่ง Purchaser / Auditor การ handoff ข้าม persona ที่ pivot จาก submission ของ vendor (`X-VPL-01`, `X-VPL-04`, `X-VPL-05`, `X-VPL-07`, `X-VPL-08`) อยู่ใน parent overview ไม่ใช่ที่นี่

## 2. Permission / Authorization

| # | Scenario | พฤติกรรมที่คาด |
| - | -------- | ------------------ |
| VPL-VND-PERM-01 | Vendor เป็น external party ที่ไม่มี Carmen system login; RBAC ใน-ระบบเป็น N/A นโยบาย portal-token (การหมดอายุ token, IP allowlist, ขีดจำกัด concurrent-session, การตรวจจับ suspicious-activity) เป็น surface การควบคุมการเข้าถึงเดียวสำหรับ vendor และถูก test เป็นข้อกังวลการตั้งค่าฝั่ง Sysadmin | N/A — ไม่มี Carmen login สำหรับ vendor และไม่มี matrix RBAC ใน-ระบบที่จะแจกแจง "permission" ของ vendor ลดลงเป็น token validity ตาม `VPL_AUTH_007` และการแยกหน้าที่ตาม `VPL_AUTH_014` (user ที่ถือ token ไม่สามารถเป็น Carmen user ที่ approve pricelist ด้วย) — ตรวจสอบทั้งคู่จากฝั่ง Sysadmin / Purchaser / Auditor การ test unit นโยบาย portal-token (การหมดอายุ, IP allowlist, session limit, การ propagate การ revoke) ทำแคตตาล็อกใน [04-test-scenarios-audit-config.md](./04-test-scenarios-audit-config.md) |

## 3. Validation / Error

| # | Scenario | Trigger | Error ที่คาด |
| - | -------- | ------- | -------------- |
| VPL-VND-VAL-01 | การเข้า portal หลัง token expiration | Campaign `end_date < now()`; cron `VPL_POST_014` revoke token; vendor คลิก URL วันถัดไป | Reject — `VPL_AUTH_007` Portal return `401 — token expired or revoked` พร้อมข้อความ help ชี้นำ vendor ให้ติดต่อทีมจัดซื้อผ่านข้อมูลผู้ติดต่อบน email ต้นฉบับ ไม่สร้างแถว pricelist; ไม่เปิด portal session |
| VPL-VND-VAL-02 | การเข้า portal จาก IP ที่ไม่อยู่ใน allowlist | Sysadmin ตั้งค่า IP allowlist สำหรับ tenant (เช่น เครือข่ายสำนักงานของ vendor เฉพาะ); vendor พยายามเข้าจากเครือข่ายมือถือส่วนตัวนอก allowlist | Reject — `VPL_AUTH_007` Portal return `401 — access from this IP is not authorised for this token` พร้อมแนวทางใช้เครือข่ายที่ approve `system` comment ต่อความพยายามเขียนใน `tb_request_for_pricing_detail_comment` จับ IP นอก allowlist สำหรับบันทึก Auditor (ช่วยให้ Sysadmin tune allowlist ถ้า traffic ที่ถูกต้องถูกบล็อก) |
| VPL-VND-VAL-03 | ขีดจำกัด concurrent session เกิน | ขีดจำกัด tenant default = 5 session พร้อมกันต่อ token; ทีมจัดซื้อของ vendor เปิด portal ใน 6 browser tab พร้อมกัน | Reject ที่ 6 — `VPL_AUTH_007` Portal return `429 — concurrent session limit exceeded` ไปยัง tab ที่ 6; 5 แรกยัง active Vendor สามารถปิด tab active ตัวหนึ่งเพื่อปลด slot จากนั้น retry ที่ 6 ขีดจำกัด tenant-configurable; ขีดจำกัดต่ำกว่า route ไปการ escalation |
| VPL-VND-VAL-04 | Submit pricelist ที่มีแถว detail zero | Vendor เปิด portal, save draft header (`tb_pricelist` แถวมีอยู่ที่ `status = draft`) แต่ไม่ได้เพิ่มแถว detail; คลิก Submit | Reject — `VPL_VAL_023` Server return `"Pricelist must contain at least one valid line item to submit."` ปุ่ม Submit ถูก disable ใน UI จนกว่าจะมีแถวอย่างน้อยหนึ่งที่ผ่าน validation ระดับบรรทัด; การเรียก API ตรง (เส้นทางไม่ปกติสำหรับ vendor) ถูก reject |
| VPL-VND-VAL-05 | Submit การตั้งราคา MOQ-tier ในลำดับเพิ่มขึ้น (validator จับ inversion ที่ submit) | Vendor ป้อน `MOQ 1 @ ฿10.00`, แล้ว `MOQ 50 @ ฿11.00`, แล้ว `MOQ 100 @ ฿12.00` (เพิ่มขึ้น ไม่ใช่ทิศทาง bulk-discount ที่ตั้งใจ) | Save ส่ง warning ตาม `VPL_VAL_020`; submit reject Server return `"MOQ-tier pricing must be non-increasing as MOQ quantity increases."` คำแนะนำการแก้ inline highlight tier ที่ผิด; vendor ต้องปรับก่อน Submit fire ได้ |
| VPL-VND-VAL-06 | Submit แถวที่มี `price_without_tax < 0` | Vendor ป้อน `price_without_tax = -10.00` (บางทีผ่าน Excel upload จาก workbook ที่ malform) | Reject — `VPL_VAL_021` Server return `"Price-without-tax must be non-negative; tax_amt and price must reconcile to the configured rounding rule (5 dp internal, 2 dp display)."` Error inline บนแถว; Submit บล็อกจนกว่าจะแก้ |
| VPL-VND-VAL-07 | เลือกสกุลเงินที่ไม่อนุญาต | Vendor หยิบ `BTC` (หรือสกุลเงินอื่นที่ไม่อนุญาต) จาก picker ที่ขยายสมมติ | Reject — `VPL_AUTH_008` UI ไม่ list สกุลเงินที่ไม่อนุญาตใน picker; การเรียก API ตรงด้วย `currency_id` ที่ไม่อนุญาตถูก reject Server return `"Currency is not permitted for this tenant."` |
| VPL-VND-VAL-08 | เพิ่มสินค้าเดียวกันสองครั้งในโหมด online-entry (บังคับซ้ำผ่านการ manipulate DOM ตรง) | Browser ของ vendor ถูก compromise หรือ vendor พยายามการเรียก API ตรงด้วยสองแถว `tb_pricelist_detail` สำหรับ `(pricelist_id, product_id, unit_id, moq_qty)` เดียวกัน | Reject — `VPL_VAL_019` + DB unique index `pricelist_detail_pricelist_id_product_id_unit_id_moqqty_u` Server return `"MOQ quantity is required and must be non-negative; duplicate (product, unit, MOQ) rows are not permitted."` UI ป้องกันการซ้ำ; DB คือ guard fallback |

## 4. Edge Cases

| # | Scenario | เงื่อนไข | คาด |
| - | -------- | --------- | -------- |
| VPL-VND-EDGE-01 | Auto-save ระหว่าง portal-session network drop | Vendor บน surface online-entry; การเชื่อมต่อ network drop ขณะพิมพ์; vendor พิมพ์ 20 วินาทีโดยไม่มี network | Auto-save buffer locally ใน browser และ retry ที่การเชื่อมต่อ network ครั้งต่อไป ตอนสำเร็จ auto-save ตามทันค่าที่พิมพ์ล่าสุด; indicator "saved" ปรากฏใน UI ถ้า network down ที่จุด auto-save 30-second ค่า save ก่อนหน้ารักษาบน server; buffer local รักษาจนกว่า browser ปิดหรือ navigate ออกจากหน้า ไม่มี data loss ในกรณีปกติ; พฤติกรรมที่บันทึกสำหรับกรณีเลวร้ายสุด (browser crash ระหว่างการตัดการเชื่อมต่อ: อาจสูญเสียการพิมพ์สูงสุด 30 วินาทีล่าสุด) |
| VPL-VND-EDGE-02 | Switch วิธีการ submission mid-flight | Vendor ป้อน 5 แถวออนไลน์ แล้วตัดสินใจ switch ไป Excel upload | Portal เสนอ **Switch to Excel upload** พร้อม prompt ยืนยัน ตอนยืนยัน 5 แถว online ที่ป้อนถูกทิ้ง (`system` comment จับการทิ้ง); surface upload render ที่การ Excel upload แถวใหม่ parse เข้า `submission_method` finalise ที่ moment ของการคลิก Submit — ดังนั้น vendor สามารถ switch ไปกลับก่อน Submit โดยไม่ล็อกวิธีใด หลัง Submit `submission_method` เป็น immutable |
| VPL-VND-EDGE-03 | Vendor user สองคนจากบริษัท vendor เดียวกันเปิด portal บนเครื่องที่ต่างกัน | Vendor `V1` มี staff จัดซื้อ `A` และ `B`; ทั้งคู่ได้รับ invitation forward ภายใน; ทั้งคู่คลิก URL พร้อมกันจากเครื่องที่ต่างกัน | ทั้งสอง session ถูกต้อง (ภายในขีดจำกัด concurrent-session) ใช้แถว `tb_pricelist` เดียวกัน (หนึ่ง pricelist ต่อ invitation) การแก้จาก `A` และ `B` อยู่ภายใต้ guard optimistic-concurrency บน `doc_version` — save แรกชนะ; save ที่สองเห็นเวอร์ชัน stale และถูก prompt refresh พฤติกรรมจริง: นี่เกิดขึ้นน้อยเพราะ user สองคน coordinate ออฟไลน์ แต่กลไกรักษา |
