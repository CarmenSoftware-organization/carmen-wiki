# คลังสินค้า — Test Scenarios — Finance (Inventory — Test Scenarios — Finance)

> **At a Glance**
> **Persona:** Finance (Officer / Inventory Accountant + Finance Manager) &nbsp;·&nbsp; **โมดูล:** [[inventory]] &nbsp;·&nbsp; **Scenarios:** ~28
> **หมวด:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** map ไปยัง `900-period-end.spec.ts` ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ test scenarios ที่ Finance persona (Finance Officer / Inventory Accountant สำหรับ reconciliation ประจำวัน, บวก Finance Manager / Controller สำหรับสิทธิ์ period-lock) ขับโดยตรงในโมดูล `inventory` Finance คือสิทธิ์การตีมูลค่า — role ที่อนุมัติ adjustments เหนือ Controller-threshold ตาม `INV_AUTH_005`, run inventory-to-GL reconciliation ตาม `INV_XMOD_008`, orchestrate การปิดงวดตาม `INV_POST_009` / `INV_POST_010` และ (ที่ระดับ Finance Manager) advance `tb_period.status = closed → locked` ตาม `INV_AUTH_006` / `INV_POST_011` Finance **ไม่** post inventory transactions ตรง ๆ — การแก้ไขไหลผ่านเอกสาร stock-in / stock-out / credit-note ที่ Finance approve; period-end fan-out fire ภายใต้ system context ตาม `INV_AUTH_007` ความเป็นเจ้าของโมดูล inventory ของพวกเขาเริ่มเมื่อ (a) Controller escalate adjustment ผลกระทบต้นทุน, (b) reconciliation pass เป็นระยะถึงเวลา หรือ (c) Controller sign off variance สำหรับงวดที่ปิด และจบเมื่อ (a) เอกสาร post, (b) reconciliation สะอาด, (c) งวด transition `open → closed` หรือ (d) Finance Manager advance `closed → locked` Scenarios จัดกลุ่มเป็น **happy paths** (อนุมัติผลกระทบต้นทุน, pass reconciliation, ปิดงวด + open-next, period lock, period re-open ภายใน audit window), **RBAC** (Finance scope vs Controller / Sysadmin, Finance Manager exclusivity บน lock), **validation** (reconciliation variance, close-with-open-items, lock-after-window) และ **edge cases** (boundary variance, FX revaluation ที่ปิดงวด, multi-period reconciliation, เส้นทางแก้ไข locked-period)

## 2. Permission / Authorization

| # | Scenario | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| FIN-PERM-01 | Finance Officer อนุมัติ adjustments เหนือ Controller-threshold | **Allow ตาม `INV_AUTH_005`** Finance คือ signature ที่สองบนเอกสารเหนือ Controller cap ปุ่ม Approve enable ใน queue อนุมัติผลกระทบต้นทุน |
| FIN-PERM-02 | Finance Officer run inventory-to-GL reconciliation | **Allow** Reconciliation run ภายใต้ Finance role ตาม `INV_XMOD_008`; dashboard คือ read+post (อ่าน sub-ledger sums, post compensating journals เพื่อ align) |
| FIN-PERM-03 | Finance Officer พยายาม lock งวด | **Deny — Finance Manager เท่านั้น ตาม `INV_AUTH_006`** ปุ่ม **Lock** ซ่อนสำหรับ Finance Officer role; การเรียก API ตรง return `"Period lock requires the Finance Manager role."` Officer run close (`closed → open → closed`); Manager advance เป็น `locked` |
| FIN-PERM-04 | Finance Manager re-open งวดที่ closed | **Allow ด้วย audit log ตาม `INV_AUTH_006`** Action Re-open ต้องการ justification string; action audit-logged Re-opens หลายครั้งของงวดเดียวกันภายใน audit window อนุญาตแต่ scrutinise มากขึ้น; งวด locked re-open ไม่ได้โดย role ใด |
| FIN-PERM-05 | ใครก็ตามพยายาม re-open งวด locked | **Deny ตาม `INV_AUTH_006`** Locked คือ terminal Server return `"Period <YYMM> is locked; locked periods cannot be re-opened by any role."` เส้นทางการแก้ไขเดียวคือ restatement งวด-ปัจจุบัน |
| FIN-PERM-06 | Finance Officer พยายามแก้ inventory transaction ที่ `posted` | **Deny — terminal state ตาม `INV_POST_012`** กฎเดียวกันกับ Controller / Store Keeper: ทุก transaction immutable Finance สามารถ post compensating credit-note หรืออนุมัติ corrective stock-in / stock-out |
| FIN-PERM-07 | Finance Officer พยายาม config period definition / costing method | **Deny — Sysadmin เท่านั้น ตาม `INV_AUTH_008`** Finance consume configuration; ไม่แก้ Period definitions สำหรับ cadences ปกติ auto-generate โดย scheduled job; งวดยกเว้นต้องการ Sysadmin action |

## 3. Validation / Error

| # | Scenario | Trigger | Error ที่คาดหวัง |
| - | -------- | ------- | -------------- |
| FIN-VAL-01 | อนุมัติ stock-out ที่จะขับ on-hand ติดลบที่ approve time | Race เดียวกันกับ IC-VAL-01 แต่ที่ Finance threshold: ระหว่างการอนุมัติ Controller และการอนุมัติ Finance, on-hand ที่ `(location, product, lot)` ลดลงเพิ่มเติมเนื่องจาก posts อื่น | **Reject ที่ Finance approve** — `INV_VAL_005` re-check ตาม `INV_POST_002` Server return error ติดลบเดียวกัน; Finance return เอกสารไปยัง Controller สำหรับ re-evaluation (split / ลด qty / re-route) |
| FIN-VAL-02 | Pass reconciliation mark สะอาดด้วย variance เหนือ tolerance | Finance Officer พยายามคลิก **Mark reconciliation clean** บน variance `LOC-A` ของ `฿250.00` (เหนือ tolerance `฿1.00`) โดยไม่ resolve | **Reject ที่ mark-clean** — server return `"Variance ฿250.00 exceeds tolerance ฿1.00; resolve via compensating journal or corrective adjustment before marking clean."` Finance ต้อง drill, resolve, re-run reconciliation จากนั้น mark |
| FIN-VAL-03 | Close งวดพยายามด้วยเอกสาร source ที่ `in_progress` | Finance คลิก **Close period 2026-04** ขณะที่เอกสาร `tb_stock_in` สำหรับงวดยังอยู่ `in_progress` | **Reject ที่ close** — server return `"Cannot close period 2026-04: <N> source documents are still at draft or in_progress. Resolve all in-flight documents before closing."` Finance route กลับไปยัง Controller / Store Keeper สำหรับ resolution |
| FIN-VAL-04 | Close งวดพยายามโดยไม่มี Controller variance sign-off | ตาม FIN-HP-05 — gate เดียวกันจากฝั่ง close-orchestration | **Reject ที่ close** ดู FIN-HP-05 Pre-close checklist แสดง sign-off ที่ขาด; ปุ่ม **Close period** disable จนกว่า Controller sign off |
| FIN-VAL-05 | Close งวดพยายามด้วย reconciliation variance ที่ไม่ได้แก้ไข | Reconciliation สำหรับงวดปิดไม่ได้ run หรือ run ล่าสุดมี variance ไม่ได้แก้ไขเหนือ tolerance | **Reject ที่ close** — server return `"Cannot close period 2026-04: inventory-to-GL reconciliation has unresolved variance above tolerance (variance: ฿X, tolerance: ฿Y)."` Finance ต้อง resolve ก่อน close |
| FIN-VAL-06 | Period re-open โดย Finance Officer (ไม่ใช่ Manager) | Finance Officer พยายาม re-open งวด `closed` ผ่าน API | **Reject** ตาม FIN-PERM-03 Server return `"Period re-open requires the Finance Manager role."` |
| FIN-VAL-07 | Period lock พยายามก่อน audit window ผ่าน | Finance Manager พยายาม lock `2026-04` (closed) ที่ current date `2026-04-15` (audit window ยังไม่ผ่าน; tenant policy = 60 วัน) | **Reject ที่ lock ด้วย override path** — server return `"Period 2026-04 audit window has not yet passed (lock-eligible date: 2026-06-01). Confirm early lock with audit-grade justification to proceed."` ถ้า Finance Manager confirm ด้วย justification, lock fire; มิฉะนั้นอยู่ `closed` Tenant policy อาจห้าม early lock อย่างชัดเจน |
| FIN-VAL-08 | Lock พยายามด้วยคำขอ audit-correction ที่เปิด | Finance Manager พยายาม lock `2026-04` (closed, audit window ผ่าน) แต่ audit-correction-request tracker แสดงหนึ่ง item เปิด | **Reject ที่ lock** — server return `"Cannot lock period 2026-04: 1 open audit-correction request remains."` Finance Manager resolve request (ปกติ credit-note post หรือ restatement งวด-ปัจจุบัน) ก่อนลอง lock ใหม่ |

## 4. Edge Cases

| # | Scenario | Condition | Expected |
| - | -------- | --------- | -------- |
| FIN-EDGE-01 | Reconciliation variance ตรง tolerance พอดี | Variance `฿1.00` พอดีเท่ากับ tolerance `฿1.00` | **Boundary inclusive — สะอาด** ถือว่า `≤ tolerance` เป็นสะอาด `฿1.01` จะ fail Pattern inclusive เดียวกันกับ SK-EDGE-01 / IC-EDGE-01 |
| FIN-EDGE-02 | Close attempts concurrent โดย Finance Officers สองคน | Officer A คลิก **Close period 2026-04** ที่ `T`; Officer B คลิกที่ `T+200ms` | **คลิกแรกชนะ; ที่สอง reject** Close ของ A fire; งวด transition เป็น `closed` คลิกของ B return `"Period 2026-04 was closed at <timestamp> by <user A>. Refresh."` ไม่มี double-close; ไม่มี double-snapshot-write |
| FIN-EDGE-03 | FX revaluation ที่ปิดงวด (multi-currency inventory) | สต๊อกที่มาจากสกุลเงินต่างประเทศ (เช่น `P-USD` import จาก USD vendor) นั่งที่ `2026-04` close; FX rate ที่ `2026-04-30` ต่างจาก rate ที่ receipt | **Snapshot ใช้ต้นทุน receipt-time; FX revaluation คือ GL journal แยก** `tb_period_snapshot.closing_cost_per_unit` ใช้ `cost_per_unit` ของ cost-layer ซึ่งอยู่ใน base currency ที่ receipt-time FX FX revaluation เป็น rate ปัจจุบัน (ถ้า tenant policy ต้องการ) คือ manual journal ที่ Finance post ไปยังบัญชี FX Gain/Loss; inventory snapshot เองไม่ **ถูก** rewrite ด้วย FX rate ใหม่ cost-layer ledger รักษาต้นทุนดั้งเดิม |
| FIN-EDGE-04 | Multi-period reconciliation (งวดก่อนยัง open) | `2026-03` และ `2026-04` ทั้งคู่ `open` (Finance ยังไม่ run `2026-03` close เพราะ Q1 audit hold); Finance พยายาม reconcile `2026-04` | **Allow reconciliation บนงวด open ปัจจุบัน** Sub-ledger sums ต่องวด scope บน `at_period` (`YYMM`); reconciliation run กับ GL entries timestamped ภายในงวด การ run close Finance ต้อง run งวดตามลำดับ: close `2026-03` ต้องเกิดก่อน close `2026-04` (มิฉะนั้น opening ของ `2026-04` ไม่มี closing ก่อนที่ valid ให้อ่าน) |
| FIN-EDGE-05 | การแก้ไข locked-period ผ่าน restatement งวด-ปัจจุบัน | งวด `2026-03` คือ `locked`; auditor ระบุ write-off ที่พลาดที่ควร post ใน `2026-03`; Finance post corrective stock-out ในงวด open ปัจจุบัน `2026-05` ด้วย note link กลับไปยัง `2026-03` | **Posts สะอาดใน `2026-05`; audit trail ผ่าน `note` / `info` JSON** วันที่ transaction ของ stock-out แก้คือ `2026-05` (งวด open ปัจจุบัน); field `note` carry reference ข้ามงวด การเปิดเผย restatement ในรายงานการเงินสำหรับ `2026-05` คือความกังวลรายงานของ Finance ไม่ใช่ความกังวลโมดูล inventory; โมดูลแค่ post movement |
| FIN-EDGE-06 | Decimal precision บน AP-reconciliation amount | Sum cost-layer เกี่ยวข้องกับจำนวนทศนิยมเล็กมาก; rounding สะสมเป็น `฿2.34` variance ข้าม rows หลายพัน | **5dp ต่อ row; aggregate rounded เป็น 2dp; absorb ใน reconciliation tolerance** แต่ละ cost-layer row stored ที่ 5dp; aggregation re-read step ก่อนหน้า rounded ตาม `INV_CALC_012`; rounding drift สะสมอยู่ภายใน tolerance ปกติ (เช่น `฿1.00` ต่อ cost-centre ต่องวดสำหรับบัญชีใช้งาน) Drift เหนือ tolerance อย่างต่อเนื่องบ่งชี้ปัญหาเชิงระบบและ route ไปยัง Sysadmin / Auditor สำหรับ investigation |
