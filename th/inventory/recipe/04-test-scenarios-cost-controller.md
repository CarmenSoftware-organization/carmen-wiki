# สูตรอาหาร (Recipe) — Test Scenarios — Cost Controller

> **At a Glance**
> **Persona:** Cost Controller (+ Cost Control Department) &nbsp;·&nbsp; **โมดูล:** [[recipe]] &nbsp;·&nbsp; **scenario:** ~26
> **หมวด:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **การครอบคลุม E2E:** ไม่มีในเวลานี้ — E2E ภายใน Cost-Controller เป็นช่องว่าง; การครอบคลุมใกล้เคียงคือทางอ้อมผ่าน `tests/701-sr.spec.ts` ใน `../carmen-inventory-frontend-e2e/`

หน้านี้บันทึก test scenario ที่ persona Cost Controller (Cost Controller + Cost Control Department) ขับเคลื่อนตรงในโมดูล `recipe` Cost Controller อ่านข้าม recipe library และเขียนคอลัมน์ cost / pricing เท่านั้น (ตาม `REC_AUTH_006`); งานหลักของพวกเขาคือการติดตาม cost drift การ co-approve การ publish off-target (ตาม `REC_AUTH_007`) และการสอบสวน variance ทฤษฎี-vs-จริง scenario จัดกลุ่มเป็น **happy path** (การแก้ cost-only; co-approval ที่การ publish off-target; การ verify cascade ของ sub-recipe; dashboard variance; การปรับเป้าหมายระดับหมวดหมู่) **RBAC** (cost controller ที่ไม่มี `recipe:edit-cost`; พยายามแก้วัตถุดิบ; ด้าน auditor read-only) **validation** (negative test เทียบกับขอบ % cost ของ `REC_VAL_008` ความสมบูรณ์ pricing-history) และ **edge case** รอบ cost drift ของวัตถุดิบ high-fanout การ cascade sub-recipe ลึก precision สกุลเงิน การตั้งค่าเป้าหมาย multi-tenant handoff ข้าม persona ที่ pivot จาก Cost Controller (Scenario 2, 3, 13 ใน parent overview) อยู่ใน [04-test-scenarios.md](./04-test-scenarios.md) ไม่ใช่ที่นี่

## 2. Permission / Authorization

| # | Scenario | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| CC-PERM-01 | Cost Controller แก้ `target_food_cost_percentage`, `selling_price`, `labor_cost_percentage`, `overhead_percentage` | **Allow** ตาม `REC_AUTH_006` การแก้สำเร็จบนสูตร `DRAFT` และ `PUBLISHED`; แถว `tb_recipe_pricing_history` เขียน |
| CC-PERM-02 | Cost Controller พยายามแก้ `qty` ของวัตถุดิบ `wastage_percentage` link sub-recipe | **Deny — Chef-only** Cost Controller มี `recipe:edit-cost` แต่ไม่มี `recipe:edit` (edit เต็ม) UI ซ่อนการควบคุมการแก้วัตถุดิบ; API call ตรง return `"You are not authorized to edit ingredient lines; that authority is the Chef's."` |
| CC-PERM-03 | Cost Controller พยายาม publish สูตร | **Deny — Chef-only** Publish ต้องการ `recipe:publish` ที่อยู่ด้าน Chef บทบาทของ Cost Controller คือ co-approval ไม่ใช่ publish; action publish ยังคงอยู่กับ Chef ตาม `REC_AUTH_003` |
| CC-PERM-04 | Cost Controller co-approve การ publish off-target | **Allow** ตาม `REC_AUTH_007` Co-approval ปรากฏใน audit log และปลดบล็อกการ publish ของ Chef |
| CC-PERM-05 | User ของ Cost Control Department พยายามเปลี่ยน cost-setting ระดับหมวดหมู่ | **ประสานกับ Sysadmin** การเขียนจริงอยู่บน `tb_recipe_category.default_cost_settings` ที่ Sysadmin เป็นเจ้าของ Cost Control Department ยกคำขอการเปลี่ยน; Sysadmin execute Co-authorship คือรูปแบบปฏิบัติการ |
| CC-PERM-06 | Cost Controller อ่าน pricing history | **Allow** `recipe:read-history` เป็นส่วนของ role cost controller; การเข้าถึง `tb_recipe_pricing_history` เต็มสำหรับการวิเคราะห์ |
| CC-PERM-07 | Cost Controller พยายามอ่านสูตร `DRAFT` นอก scope ของพวกเขา | **Allow (โดยทั่วไป)** Cost Controller มักมีการอ่านข้าม library (หมวดหมู่ใดก็ได้) เพื่อสนับสนุนการวิเคราะห์ข้ามหมวด นโยบาย tenant อาจ scope; default คือเปิดอ่าน |

## 3. Validation / Error

| # | Scenario | Trigger | Error ที่คาดหวัง |
| - | -------- | ------- | -------------- |
| CC-VAL-01 | ค่า percentage ผิด | Cost Controller ตั้ง `target_food_cost_percentage = -5` หรือ `150` | **Reject** — `REC_VAL_008` Server return `"Cost percentage values must be between 0 and 100."` |
| CC-VAL-02 | `selling_price ≤ cost_per_portion` บนสูตร `PUBLISHED` | Cost Controller ตั้ง `selling_price = 0` หรือต่ำกว่า `cost_per_portion` ปัจจุบัน | **Reject** — analog `REC_VAL_018` ที่เวลา edit Server return `"Selling price must be greater than cost per portion; review pricing."` (หมายเหตุ: schema อนุญาต `selling_price` nullable ดังนั้นการตั้งเป็น null bypass สิ่งนี้; การตั้งเป็นค่าบวกใต้ cost ถูกปฏิเสธ) |
| CC-VAL-03 | `change_reason` ของ Pricing-history ว่างบนการแก้ cost-only | Cost Controller save โดยไม่มี `change_reason` | **Reject** — การ validate ระดับ application Server return `"Change reason is required for pricing edits."` (ข้อความเหตุผลจำเป็นสำหรับความชัดเจน audit ตาม `REC_POST_010`) |
| CC-VAL-04 | การตรวจสอบความสมบูรณ์ rollup cost ล้มเหลว | `tb_recipe.cost_per_portion` ไม่ match `(total_ingredient_cost + labor_cost + overhead_cost) / base_yield` ใน tolerance การปัดเศษ | **Reject ที่ save (การตรวจสอบความสมบูรณ์)** — analog `REC_VAL_013` ที่ระดับ header Server return `"Recipe cost rollup is inconsistent; please refresh and re-save."` (บ่งบอกว่า cascade ต้นน้ำไม่ใช้ — การ recover: refresh และ save ใหม่) |
| CC-VAL-05 | การ suppress rapid-fire ของ Pricing-history | Cost Controller save ค่าเดียวกันสองครั้งใน 5 วินาที (ไม่มีการเปลี่ยนจริง) | **Warn — no-op** Server ตรวจจับค่าซ้ำ ทางเลือก suppress การ insert `tb_recipe_pricing_history` ที่สอง (หรือ insert ด้วย `change_reason = "no-change save (idempotent)"` ขึ้นอยู่กับนโยบาย tenant) |

## 4. Edge Cases

| # | Scenario | เงื่อนไข | คาดหวัง |
| - | -------- | --------- | -------- |
| CC-EDGE-01 | Cost drift ของวัตถุดิบ high-fanout | สินค้า `Premium Olive Oil` ใช้ใน 25 สูตร; cost เพิ่ม 18% | โมดูล Costing emit event drift ตาม `REC_XMOD_006`; โมดูล recipe refresh `cost_per_unit` บนทุกบรรทัดที่อ้างอิงสินค้า (25 สูตร); เขียนแถว `tb_recipe_pricing_history` ต่อสูตรที่ได้รับผลกระทบด้วย `change_reason = "cost-drift update from costing module"`; สูตรที่ `actual_food_cost_percentage` ใหม่เกิน tolerance flag ใน dashboard ประสิทธิภาพ: bulk update เสร็จในเวลาที่ tenant ยอมรับได้ |
| CC-EDGE-02 | Sub-recipe cascade ลึก (3 ระดับ) | การเปลี่ยน cost บนสินค้าใบไม้ `Beef Bones` cascade ผ่าน `Brown Stock` → `Sauce Au Poivre` → `Composite Plate` (สูตรที่ publish) | Cascade ซ้อน 3 ระดับตาม `REC_POST_006`; แต่ละระดับคำนวณ `cost_per_portion` ใหม่และเขียนแถว `tb_recipe_pricing_history` ด้วย `change_reason` ที่อ้างอิงแหล่งต้นกำเนิดของ cascade; cascade สอดคล้องภายใน (การเปลี่ยน cost สุดท้ายบน parent กระทบยอดกับการเปลี่ยน cost ของใบไม้ × multiplier การใช้) |
| CC-EDGE-03 | กรณีมุม precision สกุลเงิน | การเปลี่ยน cost ของ ฿0.001 ต่อหน่วยบนวัตถุดิบ high-volume (เช่น ข้าว 10g ต่อ portion) | เก็บที่ precision 5dp เต็ม; roll up เป็น `net_cost` ที่ 5dp; display ปัดเศษเป็น 2dp สำหรับสกุลเงินตาม `REC_CALC_015` การเปลี่ยน 0.001 อาจไม่เปลี่ยน cost-per-portion ที่แสดงให้เห็นแต่ roll up ตลอด volume สูงใน pricing history |
| CC-EDGE-04 | การแก้ cost-only พร้อมกัน | สอง Cost Controller แก้ pricing ของสูตรเดียวกันพร้อมกัน | Optimistic-concurrency: save แรกชนะ; save ที่สอง return conflict; user ที่สอง refresh แต่ละ save ที่ยอมรับเขียนแถว `tb_recipe_pricing_history` ของตัวเอง |
| CC-EDGE-05 | การ re-apply retroactive ระดับหมวดหมู่ | Sysadmin เปลี่ยนเป้าหมายของหมวด `Mains` จาก 32% เป็น 33%; Cost Control Department ขอ re-apply retroactive ให้สูตร `Mains` ทั้งหมด | operation ด้วยมือ: batch job iterate สูตรที่ไม่ถูก soft-delete ทั้งหมดใน `Mains` และเขียนเป้าหมายใหม่; แถว `tb_recipe_pricing_history` per-recipe อาจถูกเขียนตาม `REC_POST_010` (ขึ้นอยู่กับนโยบาย tenant บน bulk operation) |
| CC-EDGE-06 | การแก้ pricing variant | Cost Controller แก้ `selling_price` บน `tb_recipe_yield_variant` เฉพาะ | `selling_price` per-variant update; `food_cost_percentage` / `gross_margin` / `gross_margin_percentage` ของ variant คำนวณใหม่; แถว pricing history อาจเป็น variant-scope (พร้อม `variant_id` populate บนแถว); pricing ของสูตรฐานไม่ได้รับผลกระทบ |
| CC-EDGE-07 | การอ่านสถานะเวอร์ชันประวัติ | Cost Controller drill เข้า `tb_recipe_version` สำหรับ snapshot 6 เดือนเก่าเพื่อสอบสวนคำถาม margin | `tb_recipe_version.recipe_data` / `ingredients_data` JSON ให้ snapshot เต็ม; Cost Controller สามารถคำนวณ margin ประวัติที่จุดเวลานั้น; ไม่มีอำนาจแก้บน snapshot ประวัติ (อ่านอย่างเดียวผ่าน `recipe:read-history`) |
| CC-EDGE-08 | นโยบายการ truncate Pricing-history | สูตรอายุยาวสะสมแถว `tb_recipe_pricing_history` 500+ ตลอดอายุ | ทุกแถวรักษาสำหรับ audit; UI จัดหน้า; query indexed อย่างเหมาะสมบน `recipe_id` + `effective_date` Tenant อาจมีนโยบาย archival สำหรับแถวเก่ามาก (ย้ายไป cold storage) แต่โมดูล recipe ไม่ลบแถวตาม default |
