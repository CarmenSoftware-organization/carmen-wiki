# สูตรอาหาร (Recipe) — Test Scenarios — Chef

> **At a Glance**
> **Persona:** Chef (Chef / Kitchen Manager + Kitchen Staff read-only) &nbsp;·&nbsp; **โมดูล:** [recipe](/th/inventory/recipe) &nbsp;·&nbsp; **scenario:** ~38
> **หมวด:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **การครอบคลุม E2E:** ไม่มีในเวลานี้ — E2E ภายในสูตรเป็นช่องว่าง; การครอบคลุมใกล้เคียงที่สุดคือ `tests/701-sr.spec.ts` สำหรับ recipe-driven SR auto-create ใน `../carmen-inventory-frontend-e2e/`

หน้านี้บันทึก test scenario ที่ persona Chef (Chef / Kitchen Manager + subset Kitchen Staff read-only) ขับเคลื่อนตรงในโมดูล `recipe` การมีส่วนร่วมของ Chef เริ่มที่การสร้างสูตร (`DRAFT` จาก create-time) และดำเนินต่อผ่านทุกการ revise และผ่าน archive; Kitchen Staff เป็น read-only ที่เวลาบริการ scenario จัดกลุ่มเป็น **happy path** (create / save / publish; in-place edit บน `PUBLISHED`; un-publish round-trip; archive; clone; การใช้ sub-recipe; yield variant) **RBAC** (chef ที่ไม่มี `recipe:publish`; pastry chef scope ไปยังหมวด Desserts; Kitchen Staff พยายามแก้) **validation** (negative test เทียบกับ `REC_VAL_001`–`REC_VAL_018` ที่ Chef สามารถ trigger ที่ save หรือ publish) และ **edge case** รอบการตรวจจับ cycle ของ sub-recipe ความแม่นยำทศนิยม การแก้พร้อมกัน สูตรใหญ่ และบรรทัดวัตถุดิบ variant-scope handoff ข้าม persona ที่ pivot จาก Chef (Scenario 1, 2, 4, 5, 7, 8, 9, 12, 14 ใน parent overview) อยู่ใน [04-test-scenarios.md](./04-test-scenarios.md) ไม่ใช่ที่นี่

## 2. Permission / Authorization

| # | Scenario | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| CHEF-PERM-01 | Chef สร้างและ publish สูตรในหมวดของตัวเอง | **Allow** Chef ถือ `recipe:create / edit / publish` บน `Mains` CHEF-HP-01 ผ่าน CHEF-HP-03 สำเร็จ ตาม `REC_AUTH_001` / `REC_AUTH_003` |
| CHEF-PERM-02 | Pastry Chef (scope ไปยัง Desserts) พยายามสร้างใน Mains | **Deny — scope RBAC** Pastry Chef มี `recipe:create` เฉพาะบน `Desserts` ความพยายาม create ด้วย `category = Mains` ถูกปฏิเสธที่ API ด้วย `"You are not authorized to create recipes in category Mains."`; ไม่มีแถว `tb_recipe` เขียน |
| CHEF-PERM-03 | Kitchen Staff พยายามแก้สูตร `PUBLISHED` | **Deny — read-only** Kitchen Staff มี `recipe:read` เท่านั้น ความพยายามเขียน API ตรง return `"Kitchen Staff role is read-only on the recipe library."` |
| CHEF-PERM-04 | Chef ที่ไม่มี `recipe:publish` พยายาม publish | **Deny — RBAC** Chef มี `recipe:create / edit` แต่ไม่มี `recipe:publish` ปุ่ม publish return `"You are not authorized to publish recipes."`; สูตรยังคง `DRAFT` ตาม `REC_AUTH_003` |
| CHEF-PERM-05 | Chef พยายามแก้สูตร `PUBLISHED` โดยไม่มี permission `recipe:edit-published` | **Deny — RBAC** Tenant ต้องการ `recipe:edit-published` สำหรับการแก้ `PUBLISHED`; chef ที่ไม่มี permission ไม่สามารถ save in-place ขอ permission หรือใช้ un-publish round-trip ถ้านโยบาย tenant อนุญาต ตาม `REC_AUTH_002` clause ที่สอง |
| CHEF-PERM-06 | Chef พยายามลบสูตร `PUBLISHED` | **Deny — ต้อง archive** สูตร `PUBLISHED` ไม่สามารถ soft-delete; ต้อง archive ก่อนตาม `REC_AUTH_014` API call soft-delete ตรง return `"PUBLISHED recipes must be archived, not deleted."` |
| CHEF-PERM-07 | Chef พยายาม archive สูตรที่มี linkage menu-item active | **Warn — ประสานกับ F&B Ops** UI แสดงคำเตือน "สูตรนี้ link กับ N menu item active; archive จะทำลาย linkage นั้น ประสานกับ F&B Ops ก่อนดำเนินการ" รูปแบบ confirm-to-proceed Schema พื้นฐานอนุญาต archive; การลบ menu-link เป็นนโยบาย application |

## 3. Validation / Error

| # | Scenario | Trigger | Error ที่คาดหวัง |
| - | -------- | ------- | -------------- |
| CHEF-VAL-01 | `code` ขาดที่ save | นโยบายการเลขของ tenant ล้มเหลวในการกำหนด `code`; chef คลิก **Save** | **Reject** — `REC_VAL_001` Server return `"Recipe code is required and the (code, name) pair must be unique."` |
| CHEF-VAL-02 | `name` ขาดที่ save | Chef เว้น `name` ว่าง; คลิก **Save** | **Reject** — `REC_VAL_002` Server return `"Recipe name is required."` |
| CHEF-VAL-03 | `category_id` / `cuisine_id` ขาดหรือ inactive | Chef เลือกหมวดหมู่ที่ถูก soft-delete ตั้งแต่นั้น (race UI) หรือ cuisine `is_active = false` | **Reject** — `REC_VAL_003` / `REC_VAL_004` Server return `"Recipe category is required and must reference an active category."` / `"Cuisine type is required and must reference an active cuisine."` |
| CHEF-VAL-04 | `base_yield = 0` หรือลบที่ publish | Chef กรอก `base_yield = 0`; คลิก **Publish** | **Reject ที่ publish** — `REC_VAL_005` Server return `"Recipe yield must be greater than zero and a yield unit is required."` (Save อาจเป็น warn-only บน 0) |
| CHEF-VAL-05 | `prep_time` หรือ `cook_time` ลบ | Chef กรอก `-5` สำหรับ prep time โดยบังเอิญ | **Reject** — `REC_VAL_006` Server return `"Prep time and cook time must be non-negative."` |
| CHEF-VAL-06 | ค่า percentage ผิด (> 100) บน % cost | Chef ตั้ง `target_food_cost_percentage = 150` | **Reject** — `REC_VAL_008` Server return `"Cost percentage values must be between 0 and 100."` |
| CHEF-VAL-07 | วัตถุดิบที่ `qty ≤ 0` หรือ `wastage_percentage ≥ 100` | Chef ตั้ง `qty = 0` หรือ `wastage_percentage = 100` บนบรรทัด | **Reject ที่ publish** — `REC_VAL_009` Server return `"Ingredient quantity must be greater than zero; cost per unit must be non-negative; wastage percentage must be in [0, 100)."` (Save อาจเป็น warn-only บน 0 qty) |
| CHEF-VAL-08 | Discriminator วัตถุดิบไม่ตรง | Chef ตั้ง `ingredient_type = product` แต่ populate `sub_recipe_id` (หรือกลับกัน) | **Reject** — `REC_VAL_010` Server return `"Ingredient discriminator mismatch: type 'product' requires 'product_id' populated and the referenced row to be active."` |
| CHEF-VAL-09 | Sub-recipe เป็น `DRAFT` หรือ `ARCHIVED` ที่ save / publish ของ parent | Chef อ้างอิง sub-recipe `Burger Sauce` แต่มันถูกย้ายเป็น `DRAFT` โดย chef อื่นตั้งแต่นั้น | **Reject** — `REC_VAL_014` Server return `"Sub-recipe 'Burger Sauce' is not currently published; the parent recipe cannot reference an unpublished or archived sub-recipe."` |
| CHEF-VAL-10 | การตรวจจับ cycle ของ Sub-recipe | Chef พยายามเพิ่ม Recipe A เป็น sub-recipe ของตัวเอง หรือ loop A → B → A | **Reject** — `REC_VAL_011` Server return `"Sub-recipe cycle detected: recipe 'A' cannot reference itself directly or through another sub-recipe."` |
| CHEF-VAL-11 | การ conversion UoM ไม่ครบ | Chef ตั้ง `ingredient_unit = g`, `inventory_unit = kg` แต่เว้น `conversion_factor` null | **Reject** — `REC_VAL_012` Server return `"Unit conversion is incomplete: when the recipe unit and inventory unit differ, a positive conversion factor is required."` |
| CHEF-VAL-12 | Publish โดยไม่มีวัตถุดิบ | Chef พยายาม publish สูตรที่ไม่มีแถว `tb_recipe_ingredient` | **Reject ที่ publish** — `REC_VAL_015` Server return `"Recipe must contain at least one ingredient line before it can be published."` |
| CHEF-VAL-13 | Publish โดยไม่มีขั้นตอนการเตรียม | Chef พยายาม publish สูตรที่ไม่มีแถว `tb_recipe_preparation_step` | **Reject ที่ publish** — `REC_VAL_016` Server return `"Recipe must contain at least one preparation step before it can be published."` |
| CHEF-VAL-14 | Publish ด้วย `cost_per_portion = 0` | Rollup cost ผลิตศูนย์ (cost วัตถุดิบทั้งหมดศูนย์ ไม่มี labor / overhead) | **Reject ที่ publish** — `REC_VAL_017` Server return `"Recipe cost rollup is invalid or zero — verify ingredient costs, labor, overhead, and yield before publishing."` |
| CHEF-VAL-15 | Publish ด้วย `selling_price ≤ cost_per_portion` | Chef ตั้ง `selling_price = ฿80`, `cost_per_portion = ฿99.23` | **Reject ที่ publish** — `REC_VAL_018` Server return `"Selling price must be greater than cost per portion at publish; review pricing."` |

## 4. Edge Cases

| # | Scenario | เงื่อนไข | คาดหวัง |
| - | -------- | --------- | -------- |
| CHEF-EDGE-01 | สูตรใหญ่ — 40 วัตถุดิบ 25 prep step 3 yield variant | จานประกอบซับซ้อน (จานหลายส่วนประกอบ) | ทุกแถว persist ภายในขีดจำกัด `Decimal(20, 5)` ของ Prisma และ JSONB; rollup cost เสร็จในเวลาที่ tenant ยอมรับได้; snapshot JSON `tb_recipe_version` ใหญ่แต่ persist; การ test regression ประสิทธิภาพยืนยันว่า publish ใช้เวลา < 2s |
| CHEF-EDGE-02 | Sub-recipe ซ้อนลึก (3 ระดับ) | จานประกอบ → mother sauce → สต๊อก → สินค้าใบไม้ | Cost cascade ซ้อน 3 ระดับ; theoretical-consumption fan-out บนการขายเมนูซ้อน 3 ระดับ; OUT movement post เทียบกับสินค้าใบไม้ตาม `REC_XMOD_004` ไม่มี cycle (ตาม `REC_VAL_011`) |
| CHEF-EDGE-03 | ความแม่นยำทศนิยมบน `qty` และ `cost_per_unit` | Chef กรอก `qty = 12.34567` (5dp) และ `cost_per_unit = ฿0.12345` | ทั้งสองเก็บที่ความแม่นยำ 5dp เต็มตาม `Decimal(20, 5)`; `net_cost` คำนวณที่ความแม่นยำเต็มด้วยการปัดเศษ half-up เป็น 5dp; display ปัดเศษเป็น 3dp สำหรับ qty และ 2dp สำหรับสกุลเงินตาม `REC_CALC_015` |
| CHEF-EDGE-04 | บรรทัดวัตถุดิบ variant-scope | สูตรที่มีสอง variant `Small` และ `Large`; วัตถุดิบหนึ่ง (เช่น ไข่ทั้งฟอง) ปรากฏเฉพาะใน `Large` (ปริมาณขั้น ไม่ใช่ linear scale) | ตั้ง `tb_recipe_ingredient.tb_recipe_yield_variantId` เป็น id ของ variant `Large`; วัตถุดิบใช้เฉพาะเมื่อ variant `Large` อยู่ใน scope; การ explode theoretical consumption เคารพ variant scope |
| CHEF-EDGE-05 | การแก้พร้อมกันโดยสอง chef บน `DRAFT` เดียวกัน | Executive Chef และ Sous Chef ทั้งคู่เปิดและ save `DRAFT` เดียวกัน | Optimistic-concurrency: save แรกชนะ; save ที่สอง return 409 conflict; chef ที่สอง refresh และ resolve ด้วยมือ Schema สูตรไม่มีคอลัมน์ `doc_version` บน `tb_recipe` โดยตรง — concurrency บังคับใช้ผ่านการตรวจสอบ timestamp `updated_at` ที่ application layer |
| CHEF-EDGE-06 | Clone ของสูตร `ARCHIVED` | Chef ต้องการ revive จานที่เลิกใช้; clone สูตร archive | แถว `DRAFT` ใหม่ที่ header / วัตถุดิบ / ขั้นตอน / variant copy; `code` และ `name` ล้างให้ chef ตั้ง; `archived_at = null`, `published_at = null` บน clone; ต้นฉบับ archive ยังไม่ถูกแตะใน `ARCHIVED` |
| CHEF-EDGE-07 | Yield variant ด้วยปริมาณขั้น (non-linear scale) | Variant `Large Burger` ใช้ 2 patty (ไม่ใช่ 1.8 patty จาก linear scale) | ตั้ง scope วัตถุดิบของ variant บนบรรทัด patty (`tb_recipe_ingredient.tb_recipe_yield_variantId = Large.id`) ด้วย `qty = 2`; variant ฐานใช้บรรทัด patty ของตัวเองด้วย `qty = 1`; การ roll-up cost แยกแยะ cost per-variant อย่างถูกต้อง |
| CHEF-EDGE-08 | Re-publish หลัง un-publish รักษา chain เวอร์ชัน | สูตรเป็น `PUBLISHED` (v1), un-publish เป็น `DRAFT`, แก้, re-publish | chain `tb_recipe_version`: v1 (published = true, "initial publication"), v2 (published = false, บน save ระหว่าง `DRAFT`), v3 (published = true, "seasonal refresh") หรือขึ้นอยู่กับนโยบาย tenant การ save `DRAFT` ไม่เขียนแถวเวอร์ชันและเฉพาะการ publish เท่านั้น (v1, v2 เท่านั้น — ทั้งสอง published) chain audit สอดคล้องภายใน |
