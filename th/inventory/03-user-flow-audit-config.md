# คลังสินค้า (Inventory) — User Flow — Audit & Config

> **At a Glance**
> **Persona:** Audit / Config (Auditor + System Administrator) &nbsp;·&nbsp; **โมดูล:** [[inventory]] &nbsp;·&nbsp; **ขั้นตอน workflow:** ผู้สังเกตการณ์นอกเส้นทาง — Sysadmin ตั้งค่า `tb_location` types, `tb_product.costing_method`, `tb_adjustment_type`, `tb_period`, thresholds, RBAC, integration endpoints; Auditor อ่าน transactions / cost-layer / snapshots, run lot-recall traces และ period-snapshot reconciliation queries &nbsp;·&nbsp; **สิทธิ์สำคัญ:** Sysadmin เป็นเจ้าของ configuration (`INV_AUTH_008`); Auditor อ่านอย่างเดียวข้าม dataset เต็มรวม rows ที่ soft-deleted
> **สิ่งที่ persona นี้ทำ:** ตั้งค่า primitives เชิงโครงสร้างของโมดูล inventory (Sysadmin); audit chains ของ movement / cost-layer / period-snapshot และ lot-recall traces (Auditor)

## 1. บทบาทในโมดูลนี้

Persona **Audit / Config** รวม role ที่ไม่ใช่เชิงธุรกรรมสองตัวเข้าใน flow เดียวเพราะทั้งคู่ไม่ได้มีส่วนร่วมในวงจรชีวิต movement-and-period ของ inventory **System Administrator** sub-persona ตั้งค่า primitives เชิงโครงสร้างที่ทุกการกระทำ inventory อื่นขึ้นอยู่กับ: นิยาม `tb_location` (โดยเฉพาะ `location_type` ตาม `enum_location_type` — `inventory` / `direct` / `consignment` — และ `physical_count_type`), costing method ต่อสินค้าบน `tb_product.costing_method` (FIFO / WEIGHTED_AVERAGE — ซึ่ง engine post cost-layer อ่านที่ทุก outbound), รายการ reason-code `tb_adjustment_type` ที่ Store Keeper เลือกและ Inventory Controller / Finance อนุมัติ, นิยาม `tb_period` สำหรับงวดบัญชี, ค่า threshold ที่ขับ routing ของ Store Keeper auto-approve / Controller-route / Finance-route, นิยาม role RBAC และ scope mapping `tb_user_location` ตาม `INV_AUTH_008` และ endpoints integration กับโมดูล source ต้นน้ำ (GRN, SR, count, credit note, manual stock-in / stock-out) **Auditor** sub-persona คือ **อ่านอย่างเดียวอย่างเคร่งครัด** ข้าม dataset เต็มของ inventory — rows `tb_inventory_transaction` ทั้งหมดรวมถึง soft-deleted (compensated) reversals, ledger `tb_inventory_transaction_cost_layer` เต็ม, ทุก `tb_period_snapshot`, ประวัติ configuration และ activity logs ข้ามโมดูล (GRN commits, SR posts, count completions) — และ run **รูปแบบ query เฉพาะสองอย่าง**: (1) **lot-recall traces** ที่เดิน cost-layer ledger ไปข้างหน้าจาก GRN รับเข้าไปยังทุก movement consumption ปลายน้ำ (ตามรูปแบบ recall ใน [[good-receive-note/03-user-flow-audit-config]]) และ (2) **period-snapshot reconciliation queries** ที่ตรวจสอบว่า rows snapshot ตรงกับกิจกรรม cost-layer สำหรับงวดที่ปิด (การป้องกันของ audit ต่อการ drift ของ ledger แบบเงียบ) ทั้งสอง sub-personas ไม่เคยปรากฏใน transactional flow: การแก้ไขของ Sysadmin เปลี่ยนกฎที่ใช้กับ movement **อนาคต** (เอกสาร source ที่อยู่ระหว่างทางที่ `draft` / `in_progress` เก็บ snapshot ของกฎที่ใช้บังคับเมื่อสร้าง; `posted` inventory transactions คือ immutable และเก็บ snapshot configuration), และ queries ของ Auditor return data โดยไม่เขียน state ใด สำคัญ การบังคับใช้ segregation-of-duties `INV_AUTH_010` (write-off ≠ original receiver เหนือ threshold) และการห้ามล็อกงวด `INV_AUTH_006` (งวด locked คือ immutable สำหรับทุก roles) อาศัยอยู่ใน RBAC / posting-rule layer ที่ persona นี้เป็นเจ้าของ configuration ของ

## 2. จุดเข้าและ Flow หลัก

**จุดเข้า:** เฉพาะ sub-persona ไม่ทับซ้อนกับมุมมองรายการ inventory เชิงธุรกรรม

- **Auditor — workspace audit-log inventory** — หน้าจออ่านอย่างเดียวที่แสดง rows `tb_inventory_transaction` พร้อม `inventory_doc_type` source linkage, กิจกรรม `tb_inventory_transaction_cost_layer`, rollups `tb_period_snapshot` และ feed ประวัติ configuration Filter facets: ช่วงวันที่, location, สินค้า, lot number, ประเภทเอกสาร source (GRN / SR / count / credit note / stock-in / stock-out), งวด
- **Sysadmin — console configuration inventory** — หน้าจอ administrative scope โดย tenant พร้อม panels สำหรับ **Locations** (นิยาม, `location_type`, `physical_count_type`, delivery point linkage), **Products** (costing method ต่อสินค้า — โดยทั่วไปอุทิศให้ console config ของ [[product]] module พร้อม pass-through อ่านอย่างเดียวที่นี่), **Adjustment Types** (reason codes `tb_adjustment_type` สำหรับ stock-in / stock-out), **Periods** (นิยาม rows `tb_period`, ตาราง period-cadence), **Thresholds** (Store-Keeper auto-approve, Inventory-Controller approval cap, Finance approval ceiling), **RBAC** (`tb_user_location` mapping, role-to-action scope), **Integrations** (endpoints โมดูล source)

### 2.1 Flow Auditor (อ่านอย่างเดียว, lot-trace + period-snapshot verify) — 6 ขั้นตอน

1. **เปิด workspace audit inventory** scope ไปยังงวดที่ review รวม rows `tb_inventory_transaction` join ไปยังเอกสาร source (ผ่าน polymorphic `inventory_doc_no`), cost-layer ledger และ rollups period-snapshot เข้ามุมมองเดียวที่ navigate ได้
2. **Filter scope** Scope การ investigation ทั่วไป: lot เฉพาะ (สำหรับ recall), location เฉพาะ (สำหรับการวิเคราะห์ shrinkage), สินค้าเฉพาะ (สำหรับการ review ผลการดำเนินงาน vendor / cost-variance), งวดเฉพาะ (สำหรับ audit ปิดงวด) ผล filter เป็น render-only; ไม่มี edit affordances
3. **Drill เข้า movement** เปิด row `tb_inventory_transaction` เฉพาะเพื่อดู header (source-document linkage), detail rows (บรรทัด ledger ต่อสินค้า / ต่อ lot) และ cost-layer rows (in_qty / out_qty / cost_per_unit / lot_no พร้อม FIFO ordering ตาม `lot_seq_no`) เดิน source linkage กลับไปยังเอกสารต้นทาง (เช่น GRN, SR, count) สำหรับ chain audit เต็ม
4. **Optional: lot-recall trace** (trace เต็มสำหรับ lot เฉพาะ) ป้อน `lot_no` และ run trace **Backward** trace: ระบุทุก inventory transaction ที่แนะนำ lot (ผ่าน `current_lot_no` บน inbound detail row หรือผ่าน `lot_no` บน cost-layer's `transaction_type = good_received_note` หรือ `adjustment_in` rows) **Forward** trace: ระบุทุก movement ปลายน้ำที่บริโภคจาก lot (ผ่าน `from_lot_no` บน outbound detail rows และ `transaction_type = issue` / `transfer_out` / `adjustment_out` / `credit_note_quantity` cost-layer rows) ผลิตรายงาน chain-of-custody เต็ม
5. **Optional: period-snapshot reconciliation query** สำหรับงวดที่ปิด ตรวจสอบว่า `Σ tb_inventory_transaction_cost_layer.in_qty − Σ out_qty` (ในงวด, scope ไปยัง location / สินค้า / lot key) เท่ากับ `tb_period_snapshot.closing_qty − opening_qty` สำหรับ key เดียวกัน Gap บ่งชี้ ledger drift — defensive audit query
6. **Export รายงาน** Audit trail หรือ lot-trace exports เป็น CSV / PDF สำหรับ recall file, external auditor binder, รายงาน incident-response หรือ working papers ปิดงวด Sensitive-field exports (unit costs, vendor terms ผ่าน joined tables) ต้องการการอนุมัติรองตามรูปแบบ audit ใน [[good-receive-note/03-user-flow-audit-config]] (Auditor / Controller co-approval)

### 2.2 Flow Sysadmin (การเปลี่ยน configuration) — 6 ขั้นตอน

1. **ระบุความจำเป็น configuration** Triggers รวมการเปิด location ใหม่ (ต้องการ row `tb_location` ที่มี `location_type`, delivery point), การ retire location เก่า (ต้องการการยืนยันว่า on-hand เป็นศูนย์ — ดูกิ่งการตัดสินใจ), การ switch costing method ของสินค้า (ต้องการแผน migration — ดูกิ่งการตัดสินใจ), การเพิ่ม / retire reason code adjustment, การกำหนดงวดบัญชีใหม่ (โดยทั่วไป automate แต่งวด exception เช่น short period สำหรับการเปลี่ยน fiscal-year ลงที่นี่), การปรับ threshold การอนุมัติ, การให้ scope `tb_user_location` ให้ผู้ใช้ใหม่ หรือการ rewire integration endpoint (หายาก)
2. **เปิด console configuration inventory** และ navigate ไปยัง panel ที่เกี่ยวข้อง
3. **ปรับ configuration** แก้ fields ที่เกี่ยวข้องตามกฎ shape validation ของ panel (location code ไม่ซ้ำใน tenant, costing method หนึ่งใน {FIFO, WEIGHTED_AVERAGE}, adjustment type code ไม่ซ้ำ, period start/end ไม่ทับซ้อน, threshold ไม่เป็นลบ, RBAC role / action / scope triple well-formed, integration endpoint URL well-formed)
4. **Preview และ impact analysis** ก่อนบันทึก console run impact preview: เอกสาร source ที่อยู่ระหว่างทางกี่ฉบับ (`tb_stock_in.doc_status = draft` / `in_progress` ฯลฯ) อ้างอิง configuration ที่ได้รับผลกระทบ, rows `tb_inventory_transaction` กี่อันในงวด open ปัจจุบันที่เสี่ยงต่อการที่ change กระทบการรายงาน, sessions ของผู้ใช้ใดที่ถือ scope role / location RBAC ที่ได้รับผลกระทบ, โมดูลปลายน้ำใดบริโภค configuration ที่ได้รับผลกระทบ (เช่น การเปลี่ยน `location_type` ของ location จาก `inventory` เป็น `direct` จะกระทบ [[good-receive-note]] receipt posting) Preview flag blockers (active references ที่จะ deadlock) และ warnings (changes ที่ต้องการ user-side adoption)
5. **Save change** Config persist ด้วย effective-from timestamp; เวอร์ชันก่อนหน้าเก็บไว้ในประวัติ configuration สำหรับ audit Change reflect ทันทีสำหรับ movements **ใหม่** ที่สร้างหลังการบันทึก; เอกสาร source ที่อยู่ระหว่างทางที่ `draft` / `in_progress` เก็บ snapshot ของ configuration ที่ใช้บังคับเมื่อสร้าง (นิยาม location, costing method, threshold, RBAC ที่ create time) เว้นแต่ field จะถูก recompute อย่างชัดเจนโดยการ re-save **Posted inventory transactions** เก็บ configuration ประวัติศาสตร์เสมอผ่าน snapshot ของเอกสาร source — `cost_per_unit` ของ cost-layer คำนวณที่ post time ด้วยกฎที่ปัจจุบันในขณะนั้นและ immutable
6. **Effective สำหรับ movements ใหม่** Movements ใหม่ pick up configuration ใหม่ Activity log ประวัติ configuration แสดง change สำหรับ Auditor review ใต้ audit-trail ของ Sysadmin

## 3. กิ่งการตัดสินใจ

- **Auditor lot-recall investigation** (อ่านอย่างเดียว, escalates): Auditor ส่งมอบ trace chain-of-custody และส่งต่อให้ **Quality / Recall lead** (การ book write-off, customer notification) — ไม่แก้เอกสารหรือ transaction ใด ไม่เริ่ม write-off; การดำเนินการ recall อาศัยอยู่ใน [[inventory-adjustment]] (write-off ผ่าน `tb_stock_out`) และในเส้นทาง credit-note ของ [[good-receive-note]] (vendor concession) Deliverable ของ Auditor คือรายงาน trace
- **Sysadmin location-type change ถูก block โดย on-hand ไม่เป็นศูนย์**: ความพยายามเปลี่ยน `tb_location.location_type` จาก `inventory` เป็น `direct` (หรือการเปลี่ยน inventory ↔ non-inventory ใด) บน location ที่มี on-hand ไม่เป็นศูนย์ถูก **reject** ที่ impact preview ตามการต้อง drain ของ `INV_AUTH_008` การแก้ไข: drain on-hand (transfer ไปยัง location inventory อื่นหรือ write off ผ่าน stock-out) จากนั้นลองเปลี่ยน location-type ใหม่ ระบบแสดง summary on-hand ในข้อความ rejection
- **Sysadmin costing-method change ถูก block โดย on-hand ไม่เป็นศูนย์**: การ switch `costing_method` ของสินค้า (FIFO ↔ WEIGHTED_AVERAGE) บนสินค้าที่มี on-hand ไม่เป็นศูนย์ถูก **block** ตาม `INV_XMOD_009` เพราะประวัติ cost-layer ผูกเชิงโครงสร้างกับ method (FIFO ต้องการ `lot_seq_no` ordering; WA อ่าน `average_cost_per_unit`) การแก้ไข: drain stock (หายาก) หรือ run one-time elevated migration script ที่ re-base cost layers ภายใต้ method ใหม่ — นี่คือ event ที่ประสานงานโดย Finance ไม่ใช่การกระทำ Sysadmin ปกติ
- **Sysadmin retire adjustment-type ถูก block โดย references ประวัติศาสตร์**: ความพยายาม hard-delete row `tb_adjustment_type` ที่อ้างอิงโดยเอกสาร `tb_stock_in` / `tb_stock_out` ประวัติศาสตร์ถูก **reject** Soft-deprecate (mark inactive) เสนอเป็นเส้นทางการแก้ไข — รักษา audit-trail integrity ในขณะที่ป้องกันไม่ให้ code ถูก pick บนเอกสารใหม่ รูปแบบเดียวกันกับกฎ retirement reason-code ของ [[good-receive-note]]
- **Sysadmin period definition change**: งวดมาตรฐาน auto-generate โดย scheduled job; งวด exception (short period สำหรับการเปลี่ยน fiscal-year, การปรับขอบเขต mid-period สำหรับ event corporate) ลงที่นี่เป็นการแก้ Sysadmin ครั้งเดียว Change ไม่สามารถทับซ้อนงวดที่มีอยู่; ขอบเขตต้อง align กับขอบเขตทั้งวัน; งวดไม่สามารถ redefine เมื่อมี `tb_inventory_transaction` ใดที่มีวันที่ในงวด Gates แน่น
- **Sysadmin RBAC change ที่กระทบ active session**: รูปแบบเดียวกันกับ flow Audit / Config ของ [[good-receive-note]] — change มีผลบนความพยายามกระทำต่อไป ไม่ใช่ย้อนหลัง Force-session-revocation เสนอเป็น option การเปลี่ยนกฎ SoD (`INV_AUTH_010`) re-evaluate ที่ adjustment post ถัดไป ไม่ใช่ที่ moment ของการเปลี่ยนกฎ
- **Sysadmin threshold change retroactive scope**: การลด Store-Keeper auto-approve threshold หมายถึงเอกสารที่อยู่ระหว่างทางต่ำกว่า threshold เก่าที่ยัง `draft` ตอนนี้ route ผ่านการอนุมัติ Controller บน submit ถัดไป; เอกสารที่ submit แล้ว (`in_progress` หรือหลัง) อยู่ในเส้นทาง routing ดั้งเดิม การเพิ่ม threshold หมายถึงเอกสาร draft ใหม่ route ผ่าน auto-approve บ่อยขึ้น; เอกสาร `in_progress` ที่อยู่ระหว่างทาง complete บนเส้นทางปัจจุบัน
- **Auditor sensitive-field export ต้องการการอนุมัติรอง**: exports รวม unit costs, vendor terms (joined ผ่าน `inventory_doc_no → tb_good_received_note.vendor_id`) หรือ PII (user identities เกินกว่า role) ต้องการการอนุมัติรองตามรูปแบบ audit ใน [[good-receive-note/03-user-flow-audit-config]] Plain activity-log exports (movement counts, lot lists, anonymised) ไม่ต้องการการอนุมัติรอง
- **Sysadmin period-lock RBAC enforcement** (ห้าม — locked คือ immutable): ความพยายามของ Sysadmin หรือ Finance Manager re-open งวด `locked` ถูก reject อย่างชัดเจนตาม `INV_AUTH_006` — locked คือ terminal บทบาทของ Sysadmin ในกิ่งนี้คือเพื่อให้แน่ใจว่า RBAC layer gate lock → re-open path อย่างถูกต้อง (ไม่มี role มี permission advance `locked → open`); เส้นทางการแก้ไขเดียวสำหรับงวดที่ locked คือ restatement งวด-ปัจจุบัน (compensating adjustment ในงวด open ปัจจุบัน)
- **Sysadmin integration endpoint cutover**: การตัด endpoint integration โมดูล source (เช่น GRN module API endpoint, SR module API endpoint, period-end job runner) run dual-write window ที่ทั้ง endpoints เก่าและใหม่รับ transaction-side fan-out ตามด้วยการกระทบยอดและ hard cutover ระหว่าง dual-write window Auditor monitor รายงานการกระทบยอดสำหรับ fan-out mismatches; mismatches แก้ไขก่อน hard cutover

## 4. จุดออก / การส่งต่อ

การมีส่วนร่วมของ persona Audit / Config บนสาย inventory ที่กำหนดจบที่หนึ่งในสามขอบเขต:

- **Auditor — trace / report สร้าง** lot-trace หรือ audit-log export ส่งมอบให้ผู้ขอ (external auditor, Quality / Recall lead, Controller, Data Protection Officer) Auditor ไม่แก้ inventory transaction ใด ไม่เริ่มเอกสารปลายน้ำใด (stock-in, stock-out, credit note); deliverable คือ artefact อ่านอย่างเดียว การส่งต่อ follow-up จาก report (write-off recall, vendor blacklisting, period-lock re-review) อาศัยอยู่ใน flow ของ persona เจ้าของ — Store Keeper / Inventory Controller / Finance / Finance Manager
- **Sysadmin — configuration saved, effective สำหรับ movements ใหม่** Configuration ใหม่ persist ด้วย effective-from timestamp; ประวัติ configuration บันทึก change สำหรับ audit Movements ใหม่ pick up กฎใหม่; เอกสาร source `draft` / `in_progress` ที่อยู่ระหว่างทางเก็บ snapshot ของกฎที่ใช้บังคับที่ create time เว้นแต่ recompute อย่างชัดเจน; transactions `posted` คือ immutable Sysadmin แจ้งผู้ใช้ชุมชนที่ได้รับผลกระทบ (Store Keepers, Inventory Controllers, Finance) เกี่ยวกับ rule change ผ่านช่อง release-note มาตรฐาน
- **Sysadmin — change reject โดย impact preview, escalated** ที่ impact preview block change (on-hand ไม่เป็นศูนย์ที่ location ที่ type จะเปลี่ยน, references ที่อยู่ระหว่างทางไปยัง code ที่ retire, RBAC deadlock, integration-endpoint dependency) Sysadmin escalate ไปยัง controller หรือ change board ข้ามโมดูลเพื่อกำหนดเวลา migration window, ขอเส้นทาง soft-deprecation หรือ rollback change ที่ขอ โมดูล inventory เองไม่ transition; configuration ยังคงที่ pre-change state จนกว่าแผน migration จะอนุมัติ

- Sibling: [03-user-flow-inventory-controller.md](./03-user-flow-inventory-controller.md) — persona ที่กิจกรรมอนุมัติ, count-variance reviews และ stock-policy edits ป้อนการ review variance และ policy-compliance ของ Auditor; approval threshold ของ Controller ตั้งค่าโดย Sysadmin
