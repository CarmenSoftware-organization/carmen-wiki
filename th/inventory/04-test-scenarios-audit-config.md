# คลังสินค้า (Inventory) — Test Scenarios — Audit & Config

> **At a Glance**
> **Persona:** Audit / Config (Auditor + System Administrator) &nbsp;·&nbsp; **โมดูล:** [[inventory]] &nbsp;·&nbsp; **Scenarios:** ~32
> **หมวด:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** map ไปยัง `080-location.spec.ts`, `031-adjustment-type.spec.ts`, `900-period-end.spec.ts` ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ test scenarios ที่ Audit / Config persona ขับในโมดูล `inventory` Persona แบ่งเป็นสอง non-transactional sub-personas: **Auditor** (อ่านอย่างเดียวอย่างเคร่งครัดข้าม dataset เต็มของ inventory — `tb_inventory_transaction`, `tb_inventory_transaction_cost_layer`, `tb_period_snapshot`, ประวัติ configuration; run lot-recall traces, audit-log queries และ period-snapshot reconciliation queries) และ **System Administrator** (ตั้งค่า `tb_location` รวม `location_type` ตาม `enum_location_type` และ `physical_count_type`, costing method ต่อสินค้าบน `tb_product.costing_method` — เป็นเจ้าของโดย config โมดูล [[product]], surface ที่นี่, reason codes `tb_adjustment_type`, นิยาม `tb_period`, ค่า threshold tier ที่ route Store Keeper / Controller / Finance approval, นิยาม role RBAC และ scope mapping `tb_user_location` ตาม `INV_AUTH_008` และ integration endpoints ไปยังโมดูล source) ทั้งสอง sub-personas ไม่มีส่วนร่วมในวงจรชีวิต movement inventory — queries ของ Auditor return data โดยไม่เขียน state ใด; การแก้ไขของ Sysadmin ใช้กับ movements อนาคต (เอกสาร source ที่อยู่ระหว่างทางเก็บ snapshot ของกฎที่ใช้บังคับเมื่อสร้าง; `posted` inventory transactions immutable ตาม `INV_POST_012`) Rule IDs ที่ shared ที่ tests ของ persona นี้ exercise คือ `INV_AUTH_006` (สิทธิ์ lock / re-open งวด — Sysadmin ตั้งค่า RBAC), `INV_AUTH_008` (config scope ของ Sysadmin และ drain requirement บนการเปลี่ยน location-type / costing-method), `INV_AUTH_009` (Auditor read-only access), `INV_AUTH_010` (config SoD), `INV_POST_011` / `INV_POST_012` (rules lock และ reversal-via-compensation ที่ Sysadmin RBAC layer gate) และ `INV_XMOD_009` (drain requirement การเปลี่ยน costing-method) Scenarios จัดกลุ่มเป็น **happy paths** (Auditor audit-log query, lot-recall trace, period-snapshot reconciliation query; Sysadmin location-type config, การเปลี่ยน costing-method ด้วย drain, การดูแล adjustment-type, แก้ไข threshold, RBAC scope grant, integration cutover), **permission** (Auditor read-only allow, Sysadmin config-only deny on transact), **validation** (drain requirement บน type change, period definition non-overlap, snapshot preservation บนเอกสารที่อยู่ระหว่างทาง, RBAC deadlock prevention) และ **edge cases** (in-flight snapshot boundary, rollback ของ config ที่ไม่ดี, lot-trace gap, query timeout, mass-RBAC, period-history hard-delete protection)

## 2. Permission / Authorization

| # | Scenario | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| AUD-PERM-01 | Auditor อ่าน inventory transactions, cost-layer ledger, period snapshots, ประวัติ configuration | **Allow อ่านอย่างเดียวตาม `INV_AUTH_009`** Scope อ่านเต็มข้าม inventory entities ทั้งหมดรวม rows ที่ soft-deleted (compensated) Render-only screens; ไม่มี Save / Edit / Post / Approve affordances; backing API endpoints reject Auditor write attempts |
| AUD-PERM-02 | Auditor พยายามแก้ inventory entity ใด | **Deny — role scope** Direct API write (`POST /inventory-transaction`, `PATCH /inventory-transaction/{id}`, `POST /stock-in` ฯลฯ) จาก Auditor token return `403 Forbidden` ด้วย `"Auditor role is read-only on the inventory module."` Rejection เองถูก log บน audit-of-audit trail |
| AUD-PERM-03 | Sysadmin config location / costing method / adjustment types / periods / thresholds / RBAC / integrations | **Allow — config scope ตาม `INV_AUTH_008`** Console configuration render panels สำหรับ Sysadmin role; users เชิงธุรกรรม (Store Keeper, Controller, Finance) ไม่เห็น console |
| AUD-PERM-04 | Sysadmin พยายาม post inventory transaction | **Deny — separation โดยการออกแบบ** Sysadmin role คือ configuration-only; ไม่อยู่ใน `tb_user_location` mapping ใดโดย default; ไม่อยู่ใน approval workflow ใด Direct API `POST /stock-in` จาก Sysadmin token return `"Sysadmin role is not authorised to transact on the inventory module — configuration only."` นี่คือ separation ที่จงใจที่ให้ Sysadmin ดูแล `INV_AUTH_010` โดยไม่ bypass |
| AUD-PERM-05 | Auditor export sensitive-field report — ต้องการ secondary approval | **Allow ด้วย secondary approval** ตาม AUD-HP-04, sensitive-field exports (unit cost, vendor terms, PII ผ่าน joined tables) ต้องการ Controller หรือ DPO co-approval Plain audit-log exports (movement counts, lot lists, anonymised user roles) bypass สิ่งนี้ |
| AUD-PERM-06 | Sysadmin พยายาม period-lock หรือ re-open elevated | **Deny — Finance Manager เท่านั้น ตาม `INV_AUTH_006`** บทบาทของ Sysadmin คือ config RBAC ที่ gate lock / re-open; action เองคือของ Finance Manager Sysadmin พยายาม advance `tb_period.status = closed → locked` (หรือ `locked → open`) return `"Period state advancement requires the Finance Manager role; locked → open is forbidden for all roles."` |
| AUD-PERM-07 | User non-audit / non-sysadmin access workspace audit / config | **Deny** Store Keeper / Controller / Finance navigate ไปยัง `/admin/inventory-config` หรือ `/audit/inventory-log` ได้รับ `403 Forbidden` audit-trail visibility ของแต่ละ role scope ไปยัง actions ของตน (Store Keeper เห็น activity log ของเอกสารตนเองภายในหน้าจอ operational ฯลฯ) |

| AUD-VAL-09 | Snapshot rule — Sysadmin เปลี่ยน adjustment-type reason ที่ in-flight draft อ้างอิง | Sysadmin พยายาม retire reason code `OLD_REASON` ที่อ้างอิงโดย 1 เอกสาร `tb_stock_in.doc_status = draft`; impact preview surface reference | **Block hard-retire; เสนอเส้นทางตาม Section 3 ของ [03-user-flow-audit-config.md](./03-user-flow-audit-config.md)** Server return `"Cannot retire OLD_REASON: 1 in-flight draft document references it. Resolution: (a) wait for draft to commit or void; (b) bulk re-save drafts against new config; (c) soft-deprecate."` Sysadmin pick resolution; draft ที่อยู่ระหว่างทางเก็บ snapshot create-time ตาม snapshot rule |

## 4. Edge Cases

| # | Scenario | Condition | Expected |
| - | -------- | --------- | -------- |
| AUD-EDGE-01 | In-flight snapshot boundary ที่ threshold change | Store Keeper มีเอกสาร `tb_stock_in` เปิดที่ `draft` ที่ `T`; Sysadmin save Store Keeper auto-approve threshold สูงขึ้นที่ `T + 100ms`; Store Keeper คลิก Submit ที่ `T + 1s` | **Snapshot ใช้ที่ submit time ไม่ใช่ที่ draft create time** Routing เอกสารที่ submit อ่าน threshold **ปัจจุบัน** (สูงใหม่); ถ้าเอกสารตอนนี้ต่ำกว่า threshold ใหม่, auto-approve แทน route ไปยัง Controller Mirror snapshot rule แต่ทิศทางกลับ — threshold ประเมิน lazily เอกสาร stock-in / stock-out อ่าน threshold ที่จุดตัดสินใจ routing |
| AUD-EDGE-02 | Rollback ของ config change ไม่ดีหลังไม่มีผล in-flight | Sysadmin save threshold ค่าไม่ดีที่ `T` ที่ route เอกสารมูลค่าสูงต่ำกว่า threshold ไม่ถูกต้อง (เช่น typo เปลี่ยน `฿500` เป็น `฿50,000`); จับได้โดย monitoring ที่ `T + 5min`; rolls back | **Restore config ก่อนจากประวัติ configuration** ตามการเก็บ config-history `INV_AUTH_008`, เวอร์ชันก่อนเก็บไว้; Sysadmin คลิก **Roll back to previous version**; threshold ก่อนกลายเป็น active ทันที; submits ใหม่ใช้ค่า restore ไม่มี movements ใน `T..T+5min` ได้รับผลกระทบเพราะไม่มี Store Keeper submit ใน window; ถ้ามี submit เอกสารเหล่านั้นจะตาม bad routing path และต้องการ Finance review เพื่อตัดสินว่า reposting จำเป็นหรือไม่ |
| AUD-EDGE-03 | Lot-recall trace ด้วย gap (orphan inventory transaction) | Row `tb_inventory_transaction_cost_layer.lot_no = LOT-GAP-9` มีอยู่แต่เอกสาร source ของ inventory transaction (เช่น GRN ด้วย `inventory_doc_no = X`) soft-delete โดย migration error ประวัติศาสตร์; lot มีอยู่แต่ provenance พัง | **Surface gap ในรายงาน trace** Forward trace จาก `LOT-GAP-9` return consumption ปลายน้ำสำเร็จ; backward trace flag `"Source document <X> for inventory transaction <Y> is soft-deleted or missing; provenance gap. Recommend forensics."` Export chain-of-custody รวม gap flag สำหรับ external auditor; ไม่มี auto-repair (รักษา forensic integrity) |
| AUD-EDGE-04 | Auditor query timeout บน dataset ขนาดใหญ่ | Auditor ป้อน filter spanning 12 เดือนข้ามทุก locations; row count เกิน compute budget tenant | **Graceful timeout ด้วยผลบางส่วน** Server cancel ที่ timeout ที่ตั้ง (เช่น 30s); return ผลบางส่วนถึง T+30s ด้วย flag `truncated = true`; แนะนำ offline export job สำหรับ dataset เต็ม Pattern เดียวกันกับโมดูล GRN audit |
| AUD-EDGE-05 | Mass-RBAC activation — N users granted ข้าม N locations | Onboarding ของ locations ใหม่ 10 ด้วย Store Keepers ใหม่ 10 คนต่อ location (100 users × 10 locations = 1000 `tb_user_location` mappings); Sysadmin bulk-imports | **Allow ด้วย atomic transaction; entry ประวัติ configuration ต่อ mapping** All-or-nothing — ถ้า mapping เดียวใด fail (เช่น SoD conflict กับ role ที่มีอยู่), batch ทั้งหมด rollback; บนความสำเร็จ 1000 entries ประวัติ configuration เขียน; users เห็น scope บน login ถัดไป |
| AUD-EDGE-06 | Period history hard-delete protection | Row `tb_period_snapshot` มีอยู่สำหรับ `2026-03` (งวดที่ปิด); Sysadmin หรือ Finance Manager พยายาม soft-delete หรือ hard-delete period definition | **Reject delete บนงวดด้วย snapshots** Server return `"Period 2026-03 has <N> snapshot rows and <M> inventory transactions; periods with movements cannot be deleted."` Period definition immutable post-movements; รักษา audit chain Protection เดียวกันขยายไปยัง row `tb_period_snapshot` deletion (ห้าม) และ row `tb_inventory_transaction_cost_layer` deletion (เฉพาะผ่าน compensating-transaction soft-delete pattern ของ `INV_VAL_013`) |
