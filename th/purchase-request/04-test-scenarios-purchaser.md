# ใบขอซื้อ — กรณีทดสอบ — เจ้าหน้าที่จัดซื้อ (Purchaser)

หน้านี้รวบรวมกรณีทดสอบที่ persona ของ Purchaser เป็นผู้ขับเคลื่อนโดยตรงในโมดูล `purchase-request` Purchaser (อีกชื่อหนึ่งคือ **Procurement Officer**) เป็น persona ที่เป็นสะพานเชื่อมระหว่างฝั่ง PR ต้นน้ำและฝั่ง PO ปลายน้ำของห่วงโซ่ procure-to-pay; **ไม่** ทำหน้าที่อนุมัติเนื้อหา PR เมื่อ PR เข้าถึงคิวของ Purchaser แล้วแปลว่าผ่านห่วงโซ่ผู้อนุมัติทั้งหมดและ `pr_status = approved` (`PR_POST_005`) กรณีทดสอบถูกจัดเป็น happy path ตามที่อธิบายไว้ใน [03-user-flow-purchaser.th.md](./03-user-flow-purchaser.th.md) หัวข้อ 2 (ตรวจการจัดสรร vendor, รัน Allocate Vendor, เปิดดู pricelist deviation, consolidate ตาม vendor + currency, convert to PO — เต็มและบางส่วน), ขอบเขต RBAC ที่บังคับโดย `PR_AUTH_008` (เฉพาะ `enum_stage_role = purchase` เท่านั้นที่ convert ได้), กฎ cross-module / PO conversion ใน [02-business-rules.th.md](./02-business-rules.th.md) หัวข้อ 6 (bridge table `tb_purchase_order_detail_tb_purchase_request_detail`, consolidation ตาม `(vendor_id, currency_id)`, partial conversion, FX rate snapshot ตอน conversion) และชุด edge case เกี่ยวกับ bridge integrity และ concurrency จำนวนเล็ก ๆ เส้นทาง cross-persona ที่หมุนรอบ Purchaser (`X-PR-01`, `X-PR-04`, `X-PR-07`) อยู่ในหน้าภาพรวมพ่อ ไม่ได้อยู่ในหน้านี้

## 1. Happy Path

| PUR-HP-09 | Bounce-back ไปที่ Requestor เพื่อให้เคลียร์ประเด็น vendor / spec | PR `approved` มี 1 บรรทัดที่ vendor ทำตาม `delivery_date` ที่ขอไม่ทันและไม่มี vendor ทางเลือกที่รับได้; มีข้อความเหตุผลพร้อม | 1. เปิด PR → กด **Send Back to Requestor** (หรือ action Bounce-back) 2. Dialog บังคับใส่เหตุผล — พิมพ์ `"Vendor V1 lead time 21 days, requested delivery in 7 days. Please revise delivery_date or scope."` 3. confirm | `PR_POST_003` ทำงาน: `workflow_current_stage` re-open ไปยัง stage `create` ของ Requestor; `pr_status` กลับเป็น `draft`; `last_action = reviewed`; soft budget commitment ถูกปลด; Requestor ได้รับแจ้งพร้อมข้อความเหตุผล; เขียน comment `type = system` ลง PR (`PR_POST_008`); PR หลุดจากคิว Approved PRs ของ Purchaser และจะกลับเข้ามาเมื่อ Requestor resubmit และ chain ผู้อนุมัติเคลียร์อีกครั้ง |

## 2. Permission / Authorization

| # | กรณี | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | ---- | --------------------------------------- |
| PUR-PERM-01 | Purchaser เปิด PR `approved` (ผู้ใช้ปัจจุบันมี `enum_stage_role = purchase` และอยู่ใน `user_action.execute[]` ของ stage `purchase`) | **Allow** อ่านได้และเห็น action ทั้ง conversion และ vendor-allocation (Allocate Vendor / Convert to PO / Send Back to Requestor) `PR_AUTH_002` ผ่านสำหรับ stage `purchase`; `PR_AUTH_008` ให้สิทธิ์ conversion |
| PUR-PERM-02 | Purchaser รัน **Convert to PO** บน PR `approved` | **Allow** `PR_AUTH_008` (`enum_stage_role = purchase` เป็นเจ้าของการจัดสรร vendor และการแปลงเป็น PO); `PR_POST_007` รันการเขียน bridge และเปลี่ยน `pr_status` โมดูล PO ([[purchase-order]]) สร้างแถว `tb_purchase_order` ใหม่ |
| PUR-PERM-03 | Purchaser พยายาม convert PR ที่ยัง `pr_status = in_progress` (อยู่ระหว่าง chain อนุมัติ) | **Deny — wrong status** `PR_AUTH_008` และ `PR_POST_007` ต่างต้องการ PR ต้นทางเป็น `approved` (หรือ `approved` ที่ partial converted แล้วยังมี open line) endpoint Convert-to-PO reject PR ต้นทาง `in_progress` ด้วย `"This PR has not completed approval and cannot be converted to a PO"` workbench กรอง PR แบบนี้ออกจาก pool การเลือก |
| PUR-PERM-04 | Purchaser พยายามแก้ header ของ PR (เช่น `delivery_date`, `description`, `currency_id`) | **Deny — edit reserved** การแก้ header สงวนไว้ให้ Requestor ตอน `draft` (`PR_AUTH_001`) และผู้อนุมัติตอน `in_progress` เฉพาะฟิลด์ `approved_qty` แคบ ๆ (`PR_AUTH_003`) หน้ารายละเอียดของ Purchaser เป็น read-mostly: input ของ header ถูก disable; การเรียก API ตรง ๆ เพื่อ update column ของ `tb_purchase_request` header ถูก reject ด้วย `"This PR is in approved status and only conversion-related fields may be modified"` |
| PUR-PERM-05 | Purchaser พยายามแก้เนื้อหาบรรทัด PR (`requested_qty`, `approved_qty`, `description`, `product_id`) | **Deny — edit reserved** บรรทัด `approved_qty` ถูกตั้งโดย chain ของผู้อนุมัติ (`PR_AUTH_003` + `PR_VAL_013`) และ freeze ตอน `approved` Purchaser แก้ได้แค่ snapshot ของ vendor / pricelist บนบรรทัดและเขียน converted-qty ลง bridge table — ไม่ใช่แก้บรรทัด source การเรียก API ตรง ๆ ถูก reject ด้วย `"This field is read-only for the purchase stage role"` |
| PUR-PERM-06 | Purchaser พยายาม **approve** PR (ใช้ action ของผู้อนุมัติ เช่น `approve` / `send-back-to-prior-approver`) | **Deny — different role** ตาม `PR_AUTH_008` บทบาท `purchase` แยกจาก `approve` endpoint ของ action ผู้อนุมัติ reject การเรียกด้วย `"You are not authorised to approve at this stage"`; `PR_AUTH_002` ไม่ผ่านเพราะ Purchaser ไม่ได้อยู่ใน `user_action.execute[]` ของ stage `approve` ใด ๆ การ bounce-back ไปที่ Requestor (PUR-HP-09) เป็นเส้นทางแยกที่อนุญาต |
| PUR-PERM-07 | Purchaser trigger **Send Back to Requestor** (bounce-back) บน PR `approved` | **Allow** การ bounce-back เป็น handoff ที่ระบุไว้ฝั่ง Purchaser (`X-PR-07`) route PR ไปที่ Requestor ผ่าน path send-back มาตรฐาน (`PR_POST_003`) Purchaser **ไม่** แก้เนื้อหา — แค่ route กลับพร้อมเหตุผล |
| PUR-PERM-08 | ผู้ใช้ที่ไม่ใช่ purchase (Requestor / HOD / Finance) เปิด PR `approved` และกด **Convert to PO** | **Deny — wrong role** `PR_AUTH_008` จำกัด action ให้เฉพาะ role ที่มี `enum_stage_role = purchase` endpoint Convert-to-PO reject การเรียกโดยไม่สนว่าผู้ใช้เข้ามาจากไหน UI ซ่อนหรือ disable ปุ่มสำหรับ role ที่ไม่ใช่ purchase; การเรียก API ตรง ๆ ตอบกลับ `"You are not authorised to convert this PR to a PO"` |

## 3. Validation / Error

| # | กรณี | Trigger | ข้อผิดพลาดที่คาดหวัง |
| - | ---- | ------- | -------------------- |
| PUR-VAL-01 | Consolidate PR สองฉบับที่ vendor เดียวกันแต่ currency ต่างกัน | เลือกบรรทัด A (`vendor_id = V1`, `currency_id = THB`) และบรรทัด B (`vendor_id = V1`, `currency_id = USD`) แล้วพยายามรวมเป็น draft PO กลุ่มเดียว | Reject — key ของการจัดกลุ่มใน workbench คือ `(vendor_id, currency_id)` ตามหัวข้อ 6 ของ [02-business-rules.th.md](./02-business-rules.th.md) บรรทัดทั้งสองจะแยกเป็น draft PO **สอง** ใบ (หนึ่งใบต่อ currency) การบังคับให้รวมเป็นกลุ่มเดียวผ่าน API ถูก reject ด้วย `"Cannot consolidate lines with mismatched currency on a single PO"` |
| PUR-VAL-02 | Consolidate PR สองฉบับที่ currency เดียวกันแต่ vendor ต่างกัน | เลือกบรรทัด A (`vendor_id = V1`, `currency_id = THB`) และบรรทัด B (`vendor_id = V2`, `currency_id = THB`) แล้วพยายามรวมเป็น draft PO กลุ่มเดียว | Reject — การจัดกลุ่ม `(vendor_id, currency_id)` ต้องให้ `vendor_id` ตรงกัน บรรทัดทั้งคู่จะแยกเป็น draft PO 2 ใบ (หนึ่งใบต่อ vendor) การเรียก API เพื่อรวมถูก reject ด้วย `"Cannot consolidate lines with mismatched vendor on a single PO"` |
| PUR-VAL-03 | qty ของ partial-conversion เกิน `approved_base_qty` | บรรทัด `L1` มี `approved_base_qty = 100`, ไม่มีแถว bridge ก่อนหน้า; Purchaser ตั้ง convert-qty เป็น `120` แล้วรัน Convert to PO | Reject — qty ของ partial-conversion ต่อบรรทัดต้องอยู่ในเงื่อนไข `0 < convert_qty ≤ approved_base_qty − Σ(qty ของ bridge ก่อนหน้า)` server reject ด้วย `"Convert quantity exceeds the open quantity on this PR line"`; bridge integrity guard ปฏิเสธการเขียนแถวที่ผลรวมจะ over-bridge บรรทัด source |
| PUR-VAL-04 | qty ของ partial-conversion เกิน open qty ที่เหลือหลังจาก partial conversion ก่อนหน้า | บรรทัด `L1` `approved_base_qty = 100`; แถว bridge ก่อนหน้าบันทึก `converted_qty = 60` ไว้แล้ว; Purchaser ตั้ง convert-qty เป็น `50` (60 + 50 = 110 > 100) | Reject — กฎเดียวกับ PUR-VAL-03; server reject ด้วย `"Convert quantity exceeds the open quantity on this PR line"` และแสดงค่า open-qty (40) ใน error toast |
| PUR-VAL-05 | Pricelist deviation เกิน tolerance trigger warning / บังคับให้ตัดสินใจ | บรรทัด snapshot `pricelist_price = 100.00000 ฿`; แถว pricelist active ปัจจุบันคือ `108.00000 ฿` (+8 % เกิน tolerance `±5 %` ที่ตั้งไว้) | Panel deviation แสดง warning สีแดง Purchaser ต้องเลือกอย่างชัดเจน 1 ใน 3 path (ตาม [03-user-flow-purchaser.th.md](./03-user-flow-purchaser.th.md) หัวข้อ 3): (a) **Accept snapshot** — ยืนยันและทำต่อที่ `100.00000 ฿`; (b) **Refresh to current** — ดึง `108.00000 ฿` ไปยังบรรทัด PO; (c) **Raise concern / send back** — bounce PR กลับไปยัง Requestor Convert-to-PO ถูก **block** จนกว่าจะเลือก path การเรียก API ตรง ๆ โดยไม่มีการตัดสินใจเรื่อง deviation ที่ชัดเจน ถูก reject ด้วย `"Pricelist deviation exceeds tolerance — explicit accept/refresh/send-back required before conversion"` |
| PUR-VAL-06 | ไม่มีการจัดสรร vendor บล็อกการ convert | บรรทัด `L1` มี `vendor_id IS NULL` (Requestor ไม่ได้จัดสรร, Purchaser ยังไม่รัน Allocate Vendor); Purchaser tick `L1` แล้วรัน Convert to PO | Reject — เป้าหมายของ bridge ต้องมี vendor server reject ด้วย `"Cannot convert: line has no vendor allocation. Run Allocate Vendor first."`; workbench flag บรรทัดเป็นสีแดงและกันออกจากกลุ่มจนกว่าจะตั้งการจัดสรร vendor |
| PUR-VAL-07 | PR ที่ convert ครบแล้ว (`pr_status = completed`) เปิดใหม่เพื่อ convert เพิ่มไม่ได้ | PR ต้นทางถูก bridge ครบในรอบก่อนหน้า (`PR_POST_007` flip เป็น `completed`); Purchaser เปิดจาก filter **completed** แล้วกด Convert to PO ผ่าน UI ที่ stale / API ตรง ๆ | Reject — `completed` เป็น terminal สำหรับ conversion server reject ด้วย `"This PR has already been fully converted and cannot be reopened"`; workbench กรอง PR ที่ `completed` ออกจาก pool การเลือก |
| PUR-VAL-08 | PR ถูก void โดย Finance / sys-admin ระหว่าง session ของ Purchaser | Purchaser load PR `approved` ไว้; ระหว่างนั้น `PR_AUTH_007` flip `pr_status` เป็น `voided`; Purchaser กด Convert to PO | Reject — state-machine guard: `voided` เป็น terminal endpoint conversion reject ด้วย `"This PR is voided and cannot be converted"` soft budget commitment ของ PR ถูกปลดโดย `PR_POST_006` ไปแล้ว |
| PUR-VAL-09 | Vendor master flip เป็น inactive ระหว่าง PR submit กับการ convert | vendor `V1` ที่ snapshot ไว้บนบรรทัด active ตอน PR submit; วันนี้ `tb_vendor.is_active = false` (หรือ flag blacklist ถูกตั้ง); Purchaser กด Convert to PO | Reject — การ lookup vendor master สดทำให้การจัดสรรเดิมเป็นโมฆะ server reject ด้วย `"Vendor V1 is no longer active and cannot be used for new POs. Run Allocate Vendor to pick an alternative."`; Purchaser ต้องกำหนด vendor ใหม่ก่อนลองอีกครั้ง |
| PUR-VAL-10 | Bounce-back ไปยัง Requestor โดยไม่กรอกเหตุผล | เปิด dialog Send-Back-to-Requestor, ปล่อยช่อง reason ว่าง, กด Confirm | Reject — เหตุผลเป็น mandatory ตาม `PR_POST_008` (system comment ที่แก้ไม่ได้ต้องมี payload) และ contract ของ dialog UI disable ปุ่ม Confirm เมื่อ reason ว่าง; การเรียก API ตรง ๆ ถูก reject ด้วย `"A reason is required to send back this PR"` |

## 4. Edge Cases

| # | กรณี | เงื่อนไข | ผลที่คาดหวัง |
| - | ---- | -------- | ------------ |
| PUR-EDGE-01 | พยายาม convert โดยไม่มีบรรทัดเลือก | Purchaser เปิด workbench Convert-to-PO, ไม่ tick บรรทัดใด, แล้วกด Convert to PO | Reject — endpoint conversion ต้องการแถว bridge อย่างน้อยหนึ่งแถว server reject ด้วย `"Select at least one line to convert"`; UI ก็ disable ปุ่ม Convert-to-PO เมื่อไม่มีบรรทัด tick ไม่มีการสร้างแถว `tb_purchase_order` |
| PUR-EDGE-02 | tick ทุกบรรทัดแต่ convert-qty = 0 ทุกแถว | Purchaser tick ทุกบรรทัดแต่ตั้ง `convert_qty = 0` ทุกแถว แล้วกด Convert to PO | Reject — ต้อง `convert_qty > 0` ต่อบรรทัด (สะท้อนขอบล่างเดียวกับ PUR-VAL-03) server reject ด้วย `"Convert quantity must be positive"`; UI flag ทุกแถวที่ qty = 0 เป็นสีแดงและ disable Convert to PO จนกว่ามีอย่างน้อยหนึ่งแถวที่ convert-qty เป็นบวก |
| PUR-EDGE-03 | FX rate snapshot เท่ากับ FX rate ของ PR (ไม่มี divergence) | PR `exchange_rate = 35.50000`; FX rate ขององค์กรวันนี้ก็เท่ากับ `35.50000`; มี 1 บรรทัดที่ยังไม่ bridge | ผ่าน — PO snapshot `exchange_rate = 35.50000` (เท่ากับ PR) เขียนแถว bridge; ยอดในสกุลฐานฝั่ง PR และ PO เท่ากันเป๊ะ กฎ snapshot ไม่เปลี่ยนไม่ว่า rate จะเปลี่ยน (PUR-HP-08) หรือคงเดิม (เคสนี้) |
| PUR-EDGE-04 | Bridge-table integrity เมื่อ partial conversion ปิด open line สุดท้ายของ PR | PR ต้นทางมี 3 บรรทัด (`L1`, `L2`, `L3`); `L1` และ `L2` ถูก bridge ครบในรอบก่อนหน้า (`pr_status` ยังเป็น `approved` เพราะ `L3` ยังมี open qty `40`); ตอนนี้ Purchaser bridge `L3` อีก `40` ที่เหลือ | เขียนแถว bridge; Σ qty ที่ link ผ่าน bridge ของ `L3` = `L3.approved_base_qty`; ทั้ง 3 บรรทัด bridge ครบแล้ว `PR_POST_007` ทำงานและ flip `pr_status` จาก `approved` เป็น `completed`; soft commitment ของ `L3` แข็งเป็น hard commitment บน PO ใหม่; PR หลุดจากคิว Approved PRs |
| PUR-EDGE-05 | พยายาม convert พร้อมกันบนบรรทัด PR เดียวกัน | Purchaser A และ Purchaser B ต่าง load `approved` PR `L1` (`approved_base_qty = 100`, ไม่มี bridge ก่อนหน้า); A tick `L1` ที่ `60` แล้วกด Convert to PO ที่ `T`; B tick `L1` ที่ `60` แล้วกด Convert to PO ที่ `T+1s` (หลัง bridge ของ A commit แล้วแต่ก่อนที่ client ของ B refresh) | conversion ครั้งแรกชนะ: เขียนแถว bridge ให้ A ที่ `converted_qty = 60`, open qty ของบรรทัดต้นทางเหลือ `40` conversion ครั้งที่สอง (B) ถูก reject ที่ server — bridge-integrity guard ตรวจ `Σ(qty bridge ก่อนหน้า) + new converted_qty ≤ approved_base_qty` ใหม่ตามสถานะปัจจุบันแล้ว reject ด้วย `"Convert quantity exceeds the open quantity on this PR line"` (path เดียวกับ PUR-VAL-04) UI ของ B แจ้งให้ refresh ไม่มีการสร้าง double-bridge |
| PUR-EDGE-06 | ความละเอียดสูงสุดของ convert qty | บรรทัด `L1` มี `approved_base_qty = 1.99999 PCS` (Decimal(15, 5)); Purchaser ตั้ง `convert_qty = 1.99998` | ผ่าน — การ validate `convert_qty` ผ่าน (`> 0` และ `≤ open_qty` ที่ความละเอียดเต็ม 5 dp); เขียนแถว bridge ด้วย `converted_qty = 1.99998`; บรรทัดยังเหลือ open `0.00001 PCS`; PR ยังเป็น `approved` จนกว่าจะ bridge ส่วนที่เหลือ `0.00001` หรือ cancel PR อย่างชัดเจน |
