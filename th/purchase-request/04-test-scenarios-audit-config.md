# ใบขอซื้อ — กรณีทดสอบ — ผู้ตรวจสอบและผู้ดูแลระบบ (Audit / Config)

หน้านี้รวบรวมกรณีทดสอบที่แกน persona **Audit / Config** เป็นผู้ขับเคลื่อนในโมดูล `purchase-request` สองบทบาทใช้แกนนี้ร่วมกันแม้จะอยู่คนละฝั่งของการอ่าน / เขียน: **Auditor (ผู้ตรวจสอบ)** เป็น persona แบบอ่านอย่างเดียวที่เดิน activity trail ที่แก้ไขไม่ได้ (`workflow_history`, `tb_purchase_request_comment` ตาม `PR_POST_008`, ประวัติ `current_stage_status` ระดับบรรทัด, snapshot ของ vendor / pricelist / exchange-rate จาก `PR_CALC_006`) เพื่อยืนยันการปฏิบัติตามนโยบาย, การแบ่งหน้าที่ (segregation of duties) และความสมบูรณ์ของ trail ส่วน **System Administrator (ผู้ดูแลระบบ)** เป็น persona เชิงการตั้งค่าที่ดูแล workflow stage, amount threshold (`PR_AUTH_005`), กฎและช่วงเวลา delegation (`PR_AUTH_006`), default รายชนิด PR, master ของ tax / currency ที่ป้อน `PR_CALC_002`–`PR_CALC_006` และการกำหนด user / role บน `user_action.execute[]` ทั้งสองบทบาทไม่ได้อยู่บน happy path ของ request-to-PO กรณีทดสอบจัดเป็น happy path ตาม [03-user-flow-audit-config.th.md](./03-user-flow-audit-config.th.md) หัวข้อ 2 (Auditor: query PR ตามช่วงเวลา, drill เข้า activity log, export พร้อมการอนุมัติชั้นที่สอง, flag การละเมิดนโยบาย; Sysadmin: เปลี่ยน workflow stage, ปรับ threshold, ตั้ง delegation window, เปลี่ยน PR-type default, เปลี่ยน master tax / currency), ขอบเขต RBAC จาก `PR_AUTH_002`, `PR_AUTH_005`–`PR_AUTH_008` ใน [02-business-rules.th.md](./02-business-rules.th.md) หัวข้อ 4, validation ที่ทำงานบนแต่ละพื้นผิว (การคง snapshot ตาม `PR_CALC_006`, นโยบายการอนุมัติ export, การกัน deadlock ของ configuration, การตรวจช่วงค่า threshold, การจำกัด scan ตามช่วงวัน) และชุด edge case เฉพาะกรณีที่ persona มีทั้งพื้นผิวการตรวจสอบและการตั้งค่า เส้นทาง cross-persona ที่หมุนรอบ Audit / Config (Auditor escalate แฟ้มคดีจน Sysadmin ต้อง void ด้วย `PR_AUTH_007`, Sysadmin rollback ที่กระทบ Requestor ถัดไป) อยู่ในหน้าภาพรวมพ่อ ไม่ได้อยู่ในหน้านี้

## 2. Permission / Authorization

หัวข้อ 2 แยกกรณี Auditor (อ่านได้ / เขียนไม่ได้) ออกจาก Sysadmin (ตั้งค่าได้ / ทำรายการไม่ได้)

| # | กรณี | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | ---- | --------------------------------------- |
| AUD-PERM-01 | Auditor เปิด activity log ของ PR ใด ๆ ที่อยู่ใน scope | **Allow แบบอ่านอย่างเดียว** role `auditor` ให้สิทธิ์อ่าน `tb_purchase_request`, `tb_purchase_request_detail`, `workflow_history`, `tb_purchase_request_comment` และ `user_action` ไม่มีสิทธิ์ mutation; แถบ action บนหน้ารายละเอียด PR ถูกซ่อนสำหรับ Auditor การ drill-down เองถูกบันทึกใน audit-side query log |
| AUD-PERM-02 | Auditor พยายามแก้ PR, เปลี่ยน `approved_qty` หรือกด Approve / Reject / Send Back / Void | **Deny — role อ่านอย่างเดียว** `PR_AUTH_002` ต้องการให้ผู้ใช้อยู่ใน `user_action.execute[]` ของ stage ปัจจุบัน; Auditor **ไม่** อยู่ใน list นั้นตาม configuration UI ซ่อนปุ่ม write; การเรียก API ตรง ๆ ไปยัง `PATCH /pr/:id` หรือ workflow action endpoint ใด ๆ ถูก reject ด้วย `"You are not authorised to act on this PR — auditor role is read-only"` |
| AUD-PERM-03 | Sysadmin เปิด **PR Workflow Settings** และ save การเปลี่ยน stage / threshold / delegation | **Allow** role `sysadmin` ให้สิทธิ์เขียนบนตาราง configuration ของ workflow, threshold rule, delegation, PR-type default และ master tax / currency การ save ทุกครั้งเขียนลง audit log ของ configuration พร้อม `user_id` ของ Sysadmin analog ของ `PR_AUTH_007` และ `PR_AUTH_008`: scope ของ configuration ถือไว้เฉพาะ `sysadmin` |
| AUD-PERM-04 | Sysadmin พยายาม approve / reject / send-back PR (action เชิงรายการ) | **Deny — wrong role** Sysadmin **ไม่** อยู่ใน `user_action.execute[]` ของ stage เชิงรายการใด ๆ โดย default (เว้นแต่ถูก grant แบบ user แยกต่างหากซึ่งไม่แนะนำเพื่อรักษา segregation of duties) UI ซ่อนปุ่ม Approve / Reject / Send Back; การเรียก API ตรง ๆ ถูก reject โดย `PR_AUTH_002` ด้วย `"You are not authorised to act on this PR at the current stage — sysadmin role does not include transactional approval"` action ที่เปลี่ยน PR state เพียงอย่างเดียวที่ Sysadmin ทำได้คือ **void** ขั้นยกระดับตาม `PR_AUTH_007` (ดู `AUD-PERM-06`) |
| AUD-PERM-05 | Auditor export report CSV / PDF ที่มี sensitive field | **Allow แต่ติด gate ของการอนุมัติชั้นที่สอง** Auditor ส่งคำขอ export ได้; export อยู่ในสถานะ `pending` จนกว่า data-export approver ที่ตั้งค่าไว้จะปล่อย (ตาม `AUD-HP-03`) UI โชว์สถานะ pending; การเรียก API ตรง ๆ ไปยัง `GET /audit/export/:id/download` บน export ที่ยัง `pending` ถูก reject ด้วย `"Export is pending approval"` Export แบบ scope ไม่ sensitive (เช่น report นับจำนวนอย่างเดียว) ข้าม secondary approval ได้แต่ยังถูก log |
| AUD-PERM-06 | Sysadmin void PR ระหว่างทางหรือ PR ที่อนุมัติแล้วขั้นยกระดับ (`PR_AUTH_007`) | **Allow พร้อม audit trail** `PR_AUTH_007` สงวน action `void` ไว้ให้ scope Sysadmin / Finance Sysadmin เปิด PR กด **Void** พิมพ์เหตุผลที่บังคับ ยืนยัน Server รัน `PR_AUTH_007` → `PR_POST_006`: `pr_status` เปลี่ยนเป็น `voided` (terminal); soft / hard commitment ถูกปลด; `workflow_history` ถูก append; comment `type = system` บันทึกเหตุผล (`PR_POST_008`); Auditor เห็น void event ตอน query รอบถัดไป นี่เป็น action **เดียว** ของ Sysadmin ที่เปลี่ยน PR state — ต่างจากการ save configuration |
| AUD-PERM-07 | ผู้ใช้ที่ไม่ใช่ auditor (Requestor / Approver / Purchaser) พยายามเปิด **PR Activity Queries** | **Deny — wrong role** workspace Audit สงวนไว้ให้ role `auditor` และ `sysadmin` UI ซ่อนเมนู Sidebar **Audit** สำหรับ role อื่น; การ navigate route ตรง ๆ ถูก reject ด้วย `"You are not authorised to view the audit workspace"` persona เชิงรายการเห็นเฉพาะ PR ของตนเองและ activity log ของ PR ที่อยู่ใน scope ของตน (อ่านได้เฉพาะบน PR เหล่านั้น ผ่าน tab activity log เดียวกัน ไม่ใช่ผ่าน query builder) |

## 3. Validation / Error

หัวข้อ 3 ครอบคลุม validation ที่ล้มเหลวบนทั้งสองพื้นผิว: การเขียน configuration ของ Sysadmin ที่ผิดกฎ snapshot, deadlock, range หรือ window; การ query / export ของ Auditor ที่ผิดข้อจำกัด scan หรือประตูอนุมัติ

| # | กรณี | Trigger | ข้อผิดพลาดที่คาดหวัง |
| - | ---- | ------- | -------------------- |
| AUD-VAL-01 | Sysadmin พยายามเปลี่ยน workflow ที่จะ re-route PR ระหว่างทาง (การคง snapshot ตาม `PR_CALC_006`) | Sysadmin ลบ Stage 2 ออกจาก workflow ที่ตอนนี้มี 18 PR ค้างอยู่ที่ Stage 2; panel preview โชว์ว่า `18 PRs would be left without a stage if re-routed`; Sysadmin กด Save | **อนุญาตให้ save แต่ PR ระหว่างทางคง snapshot เดิม** ตามแพทเทิร์น snapshot preservation ใน [02-business-rules.th.md](./02-business-rules.th.md) หัวข้อ 6 การ save **ไม่** ถูก block; 18 PR ระหว่างทางทำงานต่อภายใต้ configuration 3-stage เดิมจนกว่าจะออกจาก `in_progress`; เฉพาะ PR ที่สร้างหลัง `effective_from` เท่านั้นที่เห็น chain 2-stage panel preview โชว์จำนวน in-flight เพื่อความโปร่งใส การเรียก API ตรง ๆ ที่พยายาม re-route PR ระหว่างทางย้อนหลัง (`POST /pr/:id/re-route`) ถูก reject ด้วย `"Cannot re-route an in-flight PR — snapshot preserved per PR_CALC_006"` |
| AUD-VAL-02 | Auditor พยายามดาวน์โหลด export ที่มี sensitive field โดยไม่มี secondary approval | Auditor ส่งคำขอ export ที่รวม `requester_name` และ justification เต็ม; export-approver ยังไม่ได้อนุมัติ; Auditor เรียก `GET /audit/export/:id/download` ตรง ๆ | reject ด้วย `"Export is pending approval — sensitive fields require secondary release"` UI ก็ disable ปุ่ม **Download** จนกว่าผู้อนุมัติจะปล่อย export นโยบายการอนุมัติ export ถูกบังคับฝั่ง server ไม่มี bypass แบบ UI-only |
| AUD-VAL-03 | Sysadmin ลบ user คนสุดท้ายจาก `user_action.execute[]` ของ stage ที่มี PR ค้างอยู่ (deadlock) | workflow Stage 2 ปัจจุบันมี user `approver-1@blueledgers.com` 1 คน; 1 PR `in_progress` ที่ Stage 2; Sysadmin เปิด editor และลบ `approver-1@blueledgers.com` โดยไม่กำหนดผู้แทน | reject ก่อน save ด้วย `"Stage 2 has 1 in-flight PR but would have 0 eligible approvers after this change — assign a replacement or activate a delegation before saving"` validator จับ deadlock ในขั้น validate-before-save ตาม [03-user-flow-audit-config.th.md](./03-user-flow-audit-config.th.md) หัวข้อ 3 สาขาการตัดสินใจ; Sysadmin ต้องเพิ่ม user ผู้แทน, เปิด delegation ตาม `PR_AUTH_006` ที่ครอบ stage ที่ได้รับผลกระทบ หรือยอมรับ deadlock ด้วย toggle override อย่างชัดเจนซึ่งจะถูก log แยกใน audit log ของ configuration |
| AUD-VAL-04 | Sysadmin เปิด delegation ที่ `end_at < start_at` หรือ `start_at` อยู่ในอดีต | Sysadmin พิมพ์ `start_at = 2026-06-14 00:00`, `end_at = 2026-06-01 23:59` และกด Save (window กลับด้าน); หรือ `start_at = 2026-04-01 00:00` สำหรับ window ที่ผ่านไปแล้ว | reject ก่อน save ด้วย `"Delegation end_at must be after start_at"` สำหรับ window กลับด้าน หรือ `"Delegation start_at must be in the future or now"` สำหรับ window ที่ผ่านไปแล้ว UI ก็ disable ปุ่ม Save จนกว่าวันจะถูกต้อง การเรียก API ตรง ๆ ถูก reject ด้วยข้อความเดียวกัน validator รันตาม semantics ของ window ใน `PR_AUTH_006` |
| AUD-VAL-05 | Auditor รัน query โดยไม่มี filter หรือใช้ช่วงวันที่เกินขีดจำกัดการ scan | Auditor กด **Run query** โดย filter chip ว่าง หรือ `pr_date` ระหว่าง `2020-01-01` ถึง `2026-04-30` (ช่วง > ขีดจำกัด 12 เดือนที่ตั้งไว้) | reject ด้วย `"Query must include at least one filter to prevent unbounded scans"` กรณีว่าง หรือ `"Date range exceeds 12-month limit — split the query or request an offline export"` กรณีเกินช่วง UI ก็ validate ช่วงวันแบบ live และ disable **Run query** จนกว่าช่วงจะอยู่ในขีดจำกัด ค่าจำกัดตั้งได้ต่อองค์กร; validator รันที่เวลา submit |
| AUD-VAL-06 | Sysadmin ใส่ค่า threshold ไม่ถูกต้อง (ติดลบ, ศูนย์, ไม่ใช่ตัวเลข หรือเกินเพดานต่อสกุลเงิน) | Sysadmin พิมพ์ `escalation_threshold = -50000` สำหรับกฎ route มูลค่าสูง หรือ `escalation_threshold = 0` หรือ paste `"abc"` หรือพิมพ์ค่าเกินเพดานต่อสกุลเงินที่ตั้งไว้ `9,999,999,999.99999 ฿` | reject ก่อน save ด้วยข้อความเหมาะสม: `"Threshold must be a positive amount"` กรณีติดลบ / ศูนย์, `"Threshold must be numeric"` กรณีไม่ใช่ตัวเลข, `"Threshold exceeds the configured per-currency cap"` กรณีเกินเพดาน UI inline-flag input ที่ไม่ดีและ disable **Save** validator รันการตรวจ range / type ชุดเดียวกับที่ปกป้อง `PR_VAL_*` บน field เชิงรายการ |
| AUD-VAL-07 | Sysadmin save configuration ที่ไม่มี diff (no-op save) | Sysadmin เปิด **PR Workflow Settings** ไม่ได้แก้ใด ๆ กด **Save** | reject ด้วย `"No changes detected — nothing to save"` configuration writer ต้องการ diff ที่ไม่ว่างเทียบกับ version ที่ active; UI disable **Save** จนกว่าจะมี staged change อย่างน้อยหนึ่งรายการ ไม่มีการเขียนแถว `effective_from` ใหม่และไม่ส่ง notification (สะท้อน `PM-VAL-02`) |
| AUD-VAL-08 | Sysadmin พยายามลบแถว tax-code หรือ currency-rate ที่ถูกอ้างอิงโดย snapshot ของ PR ระหว่างทาง | Sysadmin พยายามลบแถว `VAT_STANDARD` `7%` `effective_from = 2025-01-01` ขณะที่ 42 PR ระหว่างทางถือแถวนั้นใน snapshot `vat_rate` ตาม `PR_CALC_002`–`PR_CALC_004` | reject ด้วย `"Cannot delete tax code in use by 42 in-flight PR snapshots — archive (set effective_to) instead of delete"` validator บังคับว่าแถว master-data ที่ถูกอ้างอิงโดย snapshot ใด ๆ ที่ active สามารถ **archive** ได้ (เพิ่ม `effective_to` เพื่อให้แถวไม่ apply กับ PR ใหม่) แต่ห้าม **delete** ออกจริง แพทเทิร์นเดียวกันปกป้องแถว currency-rate, แถว vendor master และแถว pricelist |

## 4. Edge Cases

หัวข้อ 4 ครอบคลุม edge case ของ boundary และ concurrency เฉพาะแกน persona Audit / Config `AUD-EDGE-01` ถึง `AUD-EDGE-03` เป็นฝั่ง configuration; `AUD-EDGE-04` ถึง `AUD-EDGE-06` เป็นฝั่ง audit

| # | กรณี | เงื่อนไข | ผลที่คาดหวัง |
| - | ---- | -------- | ------------ |
| AUD-EDGE-01 | ขอบเขต snapshot ระหว่าง PR กำลังเดินอยู่กับ configuration ใหม่ | PR-A submit ที่ `T-0` ภายใต้ workflow `WF1` และ threshold `T1` Sysadmin save `WF2` และ `T2` ที่ `T+10m` (change ticket เดียว) PR-B ถูกสร้างที่ `T+11m` และ submit ที่ `T+12m` การ approve ของ PR-A เคลียร์ที่ `T+20m` | PR-A เคลียร์ผ่าน chain stage ของ `WF1` แบบจบโดยถูกประเมินกับ `T1` สำหรับการ escalate — snapshot ของ configuration ที่ take ตอน submit อยู่ตลอด chain การ approve และเข้า PO conversion snapshot workflow ของ PR-B ตอน submit คือ `WF2` พร้อม threshold `T2` ขอบเขตคือ `effective_from` ของ configuration ใหม่ ไม่ใช่เวลาที่ approve สะท้อนแพทเทิร์น snapshot ของ `PR_CALC_006` exchange rate และแพทเทิร์น snapshot vendor allocation ของ `PM-EDGE-01` |
| AUD-EDGE-02 | Sysadmin rollback การเปลี่ยน configuration ที่ผิดพลาด | Sysadmin save `WF2` ที่ `T` ที่ `T+30m` ขั้น verify activation (`03-user-flow-audit-config.th.md` หัวข้อ 2 ขั้นที่ 6) พบว่า `WF2` ทำให้เกิดบั๊กในการ route สำหรับชนิด PR `capital_expense` Sysadmin revert กลับเป็น `WF1` ที่ `T+35m` ระหว่าง `T` กับ `T+35m` มีการสร้าง 4 PR ภายใต้ `WF2` | การ rollback เองเป็นการ save configuration ที่มี `effective_from = T+35m` ของตัวเอง; audit log ของ configuration บันทึกทั้งการเปลี่ยน forward (`T`) และการ rollback (`T+35m`); 4 PR ที่สร้างระหว่าง `T` กับ `T+35m` คง snapshot `WF2` ไว้ตามแพทเทิร์น snapshot preservation (semantics snapshot ปกป้องไม่ให้ถูก re-route ย้อนหลัง); PR ใหม่ที่สร้างหลัง `T+35m` ใช้ `WF1` Auditor สามารถ correlate log การ rollback กับ 4 PR และตัดสินว่าแต่ละฉบับต้องการการแก้ไขนอกระบบหรือไม่ (โดยทั่วไป: Sysadmin void ด้วย `PR_AUTH_007` ต่อ PR ถ้า misroute แก้ไม่ได้) |
| AUD-EDGE-03 | Mass-delegation activation ที่ขอบเขตขององค์กร | bulk delegation activate ที่ `start_at = 2026-06-01 00:00` UTC สำหรับ 23 ผู้อนุมัติข้าม business unit หลายแห่ง clock ที่ `start_at` trigger activation event; บาง session ของ user อยู่ระหว่าง action ตอนที่ activation ฟัง | activation ประมวลผล delegate ทีละราย ในลำดับ deterministic; สำหรับผู้มอบหมายแต่ละราย ระบบ augment `user_action.execute[]` บน stage ที่ผู้มอบหมายเป็นเจ้าของทั้งหมด (ตาม `PR_AUTH_006`); notification re-fan เป็น batch แต่แต่ละ PR ถูกแจ้งอย่างมาก 1 ครั้งต่อ delegate เพื่อกัน storm session ที่อยู่ระหว่าง action ตอน activation จะเห็น delegate ใหม่ใน refresh ครั้งถัดไป; request server ระหว่างทางที่เริ่มก่อน activation จบภายใต้ `user_action.execute[]` ก่อน activation (first-write-wins) audit log ของ configuration บันทึก bulk activation เป็น event เดียวพร้อม breakdown ระดับ delegate ในเชิง operational เกิดยาก — bulk delegation ตั้งเวลานอกเวลาทำการตามธรรมเนียม |
| AUD-EDGE-04 | Auditor พบช่องว่างใน audit log (event ของระบบหายไป) | drill-down บน `PR-202604-0077` โชว์การเปลี่ยนสถานะ `in_progress → approved` ที่ `T` แต่ไม่มีแถว comment `type = system` ที่ `T` บันทึก action นั้น และไม่มีแถว `workflow_history` ที่สอดคล้อง | Auditor flag PR ในแฟ้มคดีพร้อมหลักฐานช่องว่าง (timestamp `T`, แถวที่หายไป) flag ไม่เปลี่ยน PR state target การ route = `Compliance` บวก tag แยกถึงทีม platform เพื่อตรวจ integrity ของ log ตามกฎ immutability (`PR_POST_008`) audit trail มีไว้เพื่อตรวจหารอยการแก้ไข — ช่องว่างเองเป็นความผิดปกติที่ต้องรายงาน และแฟ้มคดีกลายเป็นหลักฐาน Sysadmin จะ *ไม่* พยายาม backfill แถวที่หายไป; แต่จะเปิดการสอบสวนระดับ platform |
| AUD-EDGE-05 | query ของ Auditor เกินขนาด dataset และ timeout | Auditor รัน `"All status transitions for a PR"` ครอบช่วง `pr_date 2025-01-01 to 2025-12-31` ทุก business unit โดยไม่มี filter ผู้ร้องขอ; จำนวนผลลัพธ์โดยประมาณ > 500,000 event; query เกิน timeout ที่ตั้งไว้ | server ส่ง `"Query exceeds maximum result size — narrow the filter set or split into smaller windows"` (หรือถ้า timeout ฟังระหว่าง stream: `"Query timed out after Ns — split or schedule as an offline export"`) UI โชว์ panel แนะนำการ refine: "ลอง narrow ตาม business unit, ไตรมาส หรือชนิด PR" query ยังถูก log ใน audit-side query log เป็นความพยายามที่ล้มเหลวพร้อม filter set และเหตุผลที่ล้ม — มีประโยชน์ต่อการ audit-of-audit ของ Auditor เอง |
| AUD-EDGE-06 | save configuration ลงพอดีกับช่วงที่ PR ใหม่กำลัง submit | ที่ `T`, Requestor กด Submit บน PR-X (trigger routine ของ snapshot สำหรับ workflow, threshold, tax, currency, vendor); ในวินาทีเดียวกัน `T`, Sysadmin save configuration ใหม่ `C2` ด้วย `effective_from = T` | Tie-break ตาม clock ระบบที่เวลา write ถ้า routine snapshot ของ PR-X resolve configuration ที่ `T-1ms` จะอ่าน `C1`; ถ้าที่ `T+1ms` จะอ่าน `C2` ทั้งสองผลถูกต้อง — contract คือ "ใช้ configuration ที่ effective ณ ช่วงเวลาของ snapshot" ไม่ใช่ "ให้ PR-X ใช้ version ที่ preferred" audit log ของ configuration และแถว snapshot ของ PR บันทึก timestamp ทั้งคู่ ทำให้ Auditor สังเกตการเลือกได้ สะท้อน `PM-EDGE-06` |
