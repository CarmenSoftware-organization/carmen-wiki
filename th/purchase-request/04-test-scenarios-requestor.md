# ใบขอซื้อ — กรณีทดสอบ — ผู้ร้องขอ (Requestor)

หน้านี้รวบรวมกรณีทดสอบที่ persona ของ Requestor เป็นผู้ขับเคลื่อนโดยตรงในโมดูล `purchase-request` ครอบคลุม happy path ตามที่อธิบายไว้ใน [03-user-flow-requestor.th.md](./03-user-flow-requestor.th.md) หัวข้อ 2, ขอบเขต RBAC ที่มาจาก `PR_AUTH_001` และข้อกำหนดที่อนุญาตให้แก้ไขเฉพาะตอนเป็น draft ตามที่ระบุใน [02-business-rules.th.md](./02-business-rules.th.md), validation rule `PR_VAL_001`..`PR_VAL_012` ที่ Requestor สามารถ trigger ได้ด้วยตนเอง และชุด edge case จำนวนเล็ก ๆ ที่เกิดจากความละเอียดของ `Decimal(15, 5)` / `Decimal(20, 5)` และจาก send-back loop เส้นทาง cross-persona ที่เริ่มที่ Requestor แล้วเด้งกลับมาจาก Approver (`X-PR-02`, `X-PR-10`) จะอยู่ในหน้าภาพรวมพ่อ ไม่ได้อยู่ในหน้านี้

## 1. Happy Path

| # | กรณี | เงื่อนไขก่อน | ขั้นตอน | ผลที่คาดหวัง |
| - | ---- | ------------ | ------- | ------------ |
| REQ-HP-01 | สร้างและ submit PR แบบ minimal (บรรทัดเดียว, สกุลเงินฐาน) | Requestor `requestor@blueledgers.com` login อยู่; มี `tb_workflow` active สำหรับ scope `purchase-request`; มี `tb_product` active อย่างน้อย 1 รายการ; สกุลเงินฐานเป็น THB | 1. จากหน้ารายการ PR กด **Create New PR** 2. กรอกส่วนหัว: เลือก `department_id`, ตั้ง `delivery_date` ≥ `pr_date`, ปล่อย `currency_id` ตามค่าเริ่มต้นของฐาน, เลือก `workflow_id`, ใส่ description 3. เพิ่ม 1 บรรทัดที่มี `requested_qty > 0`, `requested_unit_id` ถูกต้อง, `pricelist_price` และ `delivery_date` ในอนาคต 4. บันทึกส่วนหัว (ระบบมี auto-save เมื่อขาดการเชื่อมต่อ) 5. เปิดแท็บ **Review** ตรวจ subtotal / discount / tax / grand total 6. กด **Submit** | มีแถวใหม่ใน `tb_purchase_request` ที่ `pr_status = draft → in_progress`; `pr_no` ถูก auto-generate และไม่ซ้ำ; `last_action = submitted`; `workflow_current_stage` = stage แรกของการอนุมัติ; soft budget commitment ถูกบันทึก; ผู้อนุมัติ stage แรกได้รับการแจ้งเตือน; Requestor เห็น PR บน dashboard **My PRs** |

| REQ-HP-03 | บันทึก draft วันนี้, กลับมาแก้ไขวันถัดไป แล้ว submit | Requestor login อยู่; ยังไม่ได้ submit | 1. **Create New PR** กรอกส่วนหัว เพิ่ม 1 บรรทัด 2. กด **Save** (ไม่ submit) → PR ถูกบันทึกที่ `pr_status = draft` 3. logout / ปิดแท็บ 4. วันถัดไป login กลับมา เปิด PR จาก **My PRs** 5. แก้ description และ qty ของ line 1 (`PR_AUTH_001` อนุญาตเพราะยังเป็น `draft` และ Requestor เป็นเจ้าของ) 6. Submit | Draft แสดงผลตอน login ครั้งถัดไปด้วย state ล่าสุด; แก้ไขได้เพราะ `pr_status = draft` และ `requestor_id == auth.user.id`; ตอน submit `doc_version` เพิ่มขึ้นตาม `PR_VAL_016`; สถานะเปลี่ยน `draft → in_progress` |

| REQ-HP-05 | ยกเลิก draft ของตัวเองจากหน้ารายละเอียด | PR อยู่ที่ `pr_status = draft`; Requestor เป็นเจ้าของ; ยังไม่เคย submit | 1. เปิดหน้ารายละเอียด PR 2. กด **Delete** / **Cancel** 3. ยืนยันใน modal | `pr_status` เปลี่ยนเป็น `cancelled`; ไม่มี workflow stage ใดถูก advance; ไม่เคยมี soft budget commitment เกิดขึ้น (เพราะยังไม่ถึง submit); audit log บันทึก cancel event; PR หายไปจาก active list และปุ่ม edit ถูกซ่อน |
| REQ-HP-06 | สร้าง PR จาก template, แก้ไข, submit | มี `tb_purchase_request_template` active อย่างน้อย 1 รายการ พร้อมแถว detail ที่ Requestor มองเห็นได้; template มี `workflow_id` สำหรับ scope `purchase-request` | 1. **Create from Template** เลือก template แรกใน picker 2. Submit | PR ใหม่เป็นอิสระจาก template; แถว `tb_purchase_request_template_detail` ไม่ถูกแก้; PR เปลี่ยนสถานะ `draft → in_progress`; workflow ตาม `workflow_id` ของ template |
| REQ-HP-07 | Submit PR ที่มี discount และ tax ระดับบรรทัด | ตามเงื่อนไข REQ-HP-01; เปิดใช้งานอินพุต tax profile และ discount | 1. เพิ่มบรรทัดที่ `pricelist_price = 185.00000`, `discount_rate = 5`, `tax_rate = 7`, `is_discount_adjustment = false`, `is_tax_adjustment = false` 2. ตรวจ live preview ให้แสดง `sub_total_price = 2,220.00000`, `discount_amount = 111.00000`, `net_amount = 2,109.00000`, `tax_amount = 147.63000`, `total_price = 2,256.63000` สำหรับ `requested_qty = 12` 3. | REQ-HP-08 | Submit PR สกุลเงินต่างประเทศ (บรรทัด USD, base = THB) | ตามเงื่อนไข REQ-HP-01; มี `tb_currency` ของ USD active; มี `exchange_rate` ที่ใช้ได้ ณ วันที่ ≤ `pr_date` | 1. เพิ่มบรรทัด USD ที่ `pricelist_price = 5.20000`, `requested_qty = 12`, `discount_rate = 5`, `tax_rate = 7`, `exchange_rate = 35.50000` 2. ตรวจ preview ของ base ให้ได้ `base_total_price ≈ 2,251.74180 ฿` 3. Submit | `exchange_rate` ถูก snapshot บนบรรทัดตอน submit และคงที่ตลอดอายุเอกสาร (อนุมัติใหม่ไม่ดึงเรตใหม่); คอลัมน์ `base_*` คำนวณตาม `PR_CALC_006`; soft budget commitment บันทึกในสกุลเงินฐาน |

## 2. Permission / Authorization

| # | กรณี | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | ---- | --------------------------------------- |
| REQ-PERM-01 | Requestor เปิด PR ที่ตัวเองเป็นเจ้าของ (`requestor_id == auth.user.id`) ใน `pr_status = draft` | **Allow** ทั้งอ่านและแก้ไข `PR_AUTH_001` ให้สิทธิ์แก้ไขเฉพาะกับเจ้าของขณะที่ PR ยังเป็น `draft` |
| REQ-PERM-02 | Requestor เปิด PR ที่คนอื่นเป็นเจ้าของใน `pr_status = draft` | **Deny edit, read-only** (ไม่มี toolbar Edit / Delete / Submit) `PR_AUTH_001` สงวนสิทธิ์แก้ไขให้เจ้าของหรือ delegate ที่ระบุไว้เท่านั้น; การมองเห็นใน list เองขึ้นกับสิทธิ์ **All Documents** / ระดับแผนก ถ้า PR อยู่นอกแผนกของ Requestor จะไม่ปรากฏใน list เลย |
| REQ-PERM-03 | Requestor แก้ไข draft ของตัวเอง (description ส่วนหัวและ qty ของ 1 บรรทัด) | **Allow** `PR_AUTH_001` ผ่าน; `doc_version` เพิ่มขึ้นตอนบันทึกตาม `PR_VAL_016` |
| REQ-PERM-04 | Requestor พยายามแก้ไข PR ของตัวเองหลัง submit (`pr_status = in_progress`) | **Deny** state-machine lock — ปุ่ม edit ถูก disable ทางเดียวที่จะกลับมาแก้ได้คือให้ผู้อนุมัติ **Send Back** PR กลับมา `draft` |
| REQ-PERM-05 | Requestor ยกเลิก / ลบ PR `draft` ของตัวเอง | **Allow** เป็น action ของเจ้าของบนเอกสารที่ยังไม่ submit; ยังไม่มี workflow stage ใดถูก advance, ไม่มี soft commitment ที่ต้องปล่อย |
| REQ-PERM-06 | Requestor พยายามยกเลิก PR ของตัวเองหลัง submit (`in_progress` หรือ `approved`) | **Deny** PR ที่ submit แล้วสามารถกลับเข้า chain ได้ผ่าน **Send Back** ของผู้อนุมัติ, ถูก reject โดยผู้อนุมัติ (`PR_AUTH_004` → `cancelled`) หรือถูก **void** โดย Finance / sys-admin (`PR_AUTH_007` → `voided`) เท่านั้น; Requestor จะไม่เห็นปุ่ม Cancel บน PR ที่ submit แล้ว |
| REQ-PERM-07 | Requestor พยายามอนุมัติ PR ของตัวเอง | **Deny** segregation of duties — บทบาท stage `create` และ stage `approve` ปลายน้ำเป็นชุดที่ไม่ทับซ้อนกันตาม workflow definition (ดู [02-business-rules.th.md](./02-business-rules.th.md) หัวข้อ 4 default chain) แม้จะตั้ง misconfig ให้ Requestor อยู่ใน approver stage `user_action.execute[]` ถูก recompute ทุก stage transition และเจ้าของ stage `create` จะถูก filter ออก |
| REQ-PERM-08 | Requestor ที่ใช้ fixture `noAuth` / session หมดอายุเปิด URL ของ PR ใด ๆ | **Deny — redirect ไปหน้า login** เมื่อไม่มี auth context ก็ไม่มี `auth.user.id` ให้เทียบกับ `requestor_id`; `PR_AUTH_001` ใช้ไม่ได้; ครอบคลุมโดย fixture `noAuthTest` ใน `301-pr.spec.ts` |

## 3. Validation / Error

| # | กรณี | Trigger | ข้อผิดพลาดที่คาดหวัง |
| - | ---- | ------- | -------------------- |
| REQ-VAL-01 | Submit โดยไม่มี `pr_no` (หรือ `pr_no` ซ้ำ) | header insert ของสอง client แข่งกัน หรือ seed test สร้างแถวที่ `pr_no` ชนกับแถวที่ยังไม่ถูกลบ | `PR_VAL_001` — reject ด้วย `"PR reference number is required and must be unique"` มี unique index `PR0_pr_no_u` บน `(pr_no, deleted_at)` บังคับ |
| REQ-VAL-02 | Submit โดยไม่มี `requestor_id` / snapshot `requestor_name` ที่ valid | บัญชี Requestor ถูก deactivate ระหว่างแก้ไข หรือ snapshot หาย | `PR_VAL_002` — reject ด้วย `"Requestor is required"` |
| REQ-VAL-03 | Submit โดย `department_id` ว่าง หรือชี้ไปแผนกที่ Requestor ไม่ได้สังกัด | header `department_id` ถูกเคลียร์ หรือกำหนดเป็นค่าที่อยู่นอก `user.memberships` | `PR_VAL_003` — reject ด้วย `"Department is required and must match requestor membership"` |
| REQ-VAL-04 | Submit โดยไม่มี `workflow_id` (หรือชี้ไป workflow ที่ scope ≠ `purchase-request`) | header `workflow_id` ถูกเคลียร์ | `PR_VAL_004` — reject ด้วย `"A valid PR workflow must be selected"` |
| REQ-VAL-05 | Submit โดย `pr_date` อยู่ในอนาคต | ตั้ง `pr_date` เป็นวันพรุ่งนี้ | `PR_VAL_005` — reject ด้วย `"PR date cannot be in the future"` |
| REQ-VAL-06 | Submit PR ที่ไม่มีบรรทัดที่ยังไม่ถูกลบเลย | บันทึกแค่ส่วนหัว แล้วกด Submit | `PR_VAL_006` — reject ด้วย `"A PR must contain at least one line item"` ปุ่ม Submit ก็ถูก disable ใน UI ตาม `TC-PR-050604` |
| REQ-VAL-07 | บันทึกบรรทัดที่ไม่มี `product_id` (หรือ product ที่ inactive) | ใส่แค่คำอธิบายอิสระ ไม่เลือก product หรือ product ถูก soft-delete ในแคตตาล็อก | `PR_VAL_007` — reject ด้วย `"Product is required on every line"` ระดับ DB ก็บังคับ NOT NULL |
| REQ-VAL-08 | บันทึกบรรทัดที่ `requested_qty = 0` (หรือเป็นลบ หรือไม่มี `requested_unit_id`) | พิมพ์ `0` ในช่อง qty ของบรรทัดใหม่ | `PR_VAL_008` — reject ด้วย `"Requested quantity must be greater than zero and have a unit"` |
| REQ-VAL-09 | บันทึกบรรทัดที่ `delivery_date` ก่อน `pr_date` | ตั้ง `delivery_date` ของบรรทัดเป็นวันที่ก่อน `pr_date` (ครอบคลุมโดย `TC-PR-050211`) | `PR_VAL_009` — reject ด้วย `"Delivery date cannot be earlier than the PR date"` |
| REQ-VAL-10 | เพิ่มบรรทัดที่สองที่มี `(product_id, location_id, dimension)` ซ้ำกับที่อยู่บน PR แล้ว | ใส่บรรทัดซ้ำภายใน PR เดียวกัน | `PR_VAL_010` — reject ด้วย `"Same product cannot be requested twice for the same location and dimension"` unique index `PR1_purchase_request_product_location_dimension_u` บังคับ |
| REQ-VAL-11 | บันทึกบรรทัดที่ `currency_id` หาย หรือ `exchange_rate ≤ 0` หรือ `exchange_rate_date` อยู่หลัง `pr_date` | บังคับเคลียร์ `exchange_rate` เป็น `0` หรือตั้ง `exchange_rate_date` เป็นวันพรุ่งนี้ | `PR_VAL_011` — reject ด้วย `"Currency and exchange rate are required and must be effective on or before the PR date"` |
| REQ-VAL-12 | บันทึกบรรทัดที่ `tax_rate = 110` (หรือ `discount_rate = -5`) | พิมพ์อัตราที่อยู่นอกช่วง 0–100 | `PR_VAL_012` — reject ด้วย `"Tax and discount rates must be between 0 and 100"` |
| REQ-VAL-13 | สองแท็บ browser แก้ไข draft เดียวกัน แท็บที่สองพยายามบันทึก | แท็บ A บันทึก description; แท็บ B (ที่ยังถือ `doc_version` เดิม) กด Save | `PR_VAL_016` — reject ด้วย `"Document was modified by another user; reload and retry"` การบันทึกที่สำเร็จจะ bump `doc_version` ขึ้น 1 |

## 4. Edge Cases

| # | กรณี | เงื่อนไข | ผลที่คาดหวัง |
| - | ---- | -------- | ------------ |
| REQ-EDGE-01 | ขอบเขตจำนวนเป็นศูนย์ | บันทึกบรรทัดที่ `requested_qty = 0.00000` | ถูก reject ตาม `PR_VAL_008` — กฎคือ `> 0` strict ไม่ใช่ `≥ 0`; ค่าติดลบก็ควรถูก reject ที่ฝั่ง form เช่นกัน |
| REQ-EDGE-02 | ความละเอียดทศนิยมสูงสุด | กรอกบรรทัดด้วย `requested_qty = 1.99999`, `pricelist_price = 0.00001`, `discount_rate = 99.99999`, `tax_rate = 99.99999`, `exchange_rate = 35.12345` | ถูกบันทึกที่ 5 ตำแหน่งทศนิยมพอดีบนคอลัมน์ `Decimal(15, 5)` / `Decimal(20, 5)`; ทศนิยมตำแหน่งที่ 6 ถูกปัด half-up ก่อนใช้ในขั้นถัดไปตาม rounding policy ใน [02-business-rules.th.md](./02-business-rules.th.md) หัวข้อ 3; display อาจตัดเหลือ 2 ตำแหน่งแต่ค่าที่บันทึกยังคงไว้ที่ 5 ตำแหน่ง |
| REQ-EDGE-03 | `requested_date` เท่ากับ `pr_date` | `delivery_date == pr_date` (วันเดียวกัน) | ผ่าน — `PR_VAL_009` ระบุว่า "on or after `pr_date`" ดังนั้นเท่ากันก็ valid |
| REQ-EDGE-04 | `requested_date` ในอนาคตไกล | ตั้ง `delivery_date` ของบรรทัดเป็น `2099-12-31` | ผ่านโดย `PR_VAL_009`; stage ปลายน้ำ (Approver, Purchaser) อาจ flag ใน validation ของตัวเอง แต่ submit ของ Requestor ไม่ถูกบล็อก |
| REQ-EDGE-05 | สกุลเงินบนบรรทัดต่างจากสกุลเงิน default ของแผนก | ส่วนหัว `currency_id = USD` แต่ default ของแผนก Requestor คือ THB | อนุญาตที่ stage Requestor — การแปลงเป็นสกุลเงินฐานทำผ่าน `PR_CALC_006` โดยใช้ `exchange_rate` ที่ snapshot ไว้; soft budget commitment บันทึกในสกุลฐาน (THB); chain Approver ยังสามารถ send back พร้อมเหตุผล "ให้ใช้สกุลเงินของแผนก" ได้แต่นั่นเป็นเรื่องปลายน้ำ |
| REQ-EDGE-06 | แก้ไข draft เดียวกันแบบ concurrent | เปิด draft เดียวกันใน 2 แท็บ ทั้งสองแก้ฟิลด์ต่างกันแล้วกด Save แทบจะพร้อมกัน | ครั้งแรกชนะและ bump `doc_version`; ครั้งที่สองถูก reject โดย `PR_VAL_016` และ UI แจ้งให้ reload; ไม่ใช่ last-write-wins |
| REQ-EDGE-07 | ผู้อนุมัติ send-back ขณะที่ Requestor กำลังแก้ไขอยู่ | ผู้อนุมัติกด **Send Back** บน PR ที่อยู่ใน `in_progress`; Requestor มี PR เปิดอยู่ในมุมมอง `draft` (หรือกำลังจ้องหน้ารายละเอียด) | เมื่อ refresh / save ครั้งถัดไป client เห็น `pr_status` กลับมาที่ `in_progress` พร้อม `workflow_current_stage` ที่เลื่อนกลับ; edit เปิดได้อีกครั้ง (lock `in_progress` เดิมถูกปล่อย); การแก้ไข local ที่ยังไม่บันทึกถูก merge เข้าใน draft ที่แก้ไขได้ (`PR_VAL_016` ยังคุมแต่ละ save อยู่) |
