# ใบรับสินค้า (Goods Receive Note)

> **At a Glance**
> **วัตถุประสงค์ของโมดูล:** บันทึกการรับสินค้าจริงตามใบ PO โพสต์การเคลื่อนไหวสต๊อกเข้า อัปเดต costing และสร้างรายการ AP (`Received` → `Committed` → `Voided`) &nbsp;·&nbsp; **ผู้ใช้:** Store Keeper / Receiver, Inventory Manager, Purchaser, Finance / AP &nbsp;·&nbsp; **เอนทิตี/ตารางหลัก:** `tb_good_receive_note`, `tb_good_receive_note_detail`, `InventoryStatus`, `JournalEntry`, FIFO lot layers &nbsp;·&nbsp; **หน้าย่อย:** 12

## 1. ภาพรวม

**ใบรับสินค้า (Goods Receive Note — GRN)** คือเอกสารที่บันทึกการรับสินค้าจริงจากผู้ขายอย่างเป็นทางการและเขียนเข้าคลังสินค้า แต่ละ GRN ประกอบด้วยส่วนหัว — หมายเลขอ้างอิง วันที่รับ ผู้ขาย จุดส่งของ หมายเลขใบกำกับและวันที่ สกุลเงินและอัตราแลกเปลี่ยน flag consignment/cash ที่เลือกใช้ได้ และตัวบ่งชี้ค่าใช้จ่ายเพิ่มเติม — และรายการสินค้าหนึ่งรายการขึ้นไป แต่ละรายการบรรจุข้อมูลสินค้า คลังจัดเก็บ ปริมาณที่สั่ง (เมื่อมาจาก PO) ปริมาณที่รับ ปริมาณ FOC ราคาต่อหน่วย ส่วนลด ภาษี ข้อมูล lot สำหรับสินค้าที่ต้องสืบย้อนได้ และยอดรวมต่อบรรทัดที่ระบบคำนวณให้ ส่วนหัวจะรวบยอดจากรายการเหล่านี้เป็น subtotal, discount, tax, extra costs และ grand total ทั้งในสกุลเงินที่ใช้บันทึกธุรกรรมและสกุลเงินฐาน

GRN เดินตามวงจรชีวิตที่จำกัดอย่างเคร่งครัด: `Received` (เรียกว่า Draft ก็ได้ — แก้ไขได้ ไม่กระทบสต๊อกหรือ GL) → `Committed` (immutable มีการโพสต์ inventory และ journal entry) → `Voided` (เข้าได้จากสถานะ Received เท่านั้น — ย้อนกลับ draft) การ commit GRN เป็นเหตุการณ์เดียวที่เปลี่ยนแปลงระบบ: สร้างการเคลื่อนไหวสต๊อกเข้าต่อบรรทัด อัปเดตปริมาณคงคลังและต้นทุนล่าสุดบน `InventoryStatus` สร้าง FIFO cost layer ใหม่หรือคำนวณ weighted-average ใหม่ตามวิธี costing ของสินค้า เขียน journal entry กับบัญชี AP และ inventory และอัปเดตปริมาณคงเหลือบน PO ต้นทาง เมื่อ commit แล้วเอกสารจะถูกล็อก — การแก้ไขต้องผ่านการ void เอกสารต้นทางหรือการปรับชดเชย

GRN เป็นปลายทางการรับของในการ **จับคู่สามทาง** (three-way match: PO ↔ GRN ↔ ใบกำกับจากผู้ขาย) สามารถสร้างได้จาก PO เปิดหนึ่งใบหรือมากกว่า (รองรับการรวมหลาย PO เมื่อ vendor, currency และเงื่อนไขตรงกัน) หรือสร้างด้วยมือสำหรับการรับของที่ไม่มี PO ต้นทาง การรับของบางส่วนถือเป็นเรื่องปกติ — GRN ติดตามปริมาณที่รับเทียบกับที่สั่งต่อบรรทัด และส่งปริมาณคงเหลือกลับไปยัง PO ต้นทาง ซึ่งจะย้ายไปสถานะ `Partial` จนกว่าจะรับครบหรือยกเลิก จากนั้นย้ายไป `Closed` การยกเลิกของผู้ขายจะถูกบันทึกบน GRN ส่งปริมาณที่ยกเลิกกลับไปยัง PO และป้องกันการรับเพิ่มเติมเมื่อ PO ถูกปิด

## 2. บริบททางธุรกิจ

GRN คือจุดควบคุมที่โลกของจริงพบกับบัญชี จนกว่าฝ่ายรับของจะรับรองสินค้า PO ยังเป็นเพียง commitment เมื่อ GRN ถูก commit มูลค่าสต๊อกจะอยู่บนงบดุล หนี้ทางบัญชีกับผู้ขาย (AP) ถูกรับรู้ และใบกำกับจากผู้ขายสามารถจับคู่และจ่ายได้ ธุรกิจโรงแรมมีกำไรขั้นต้นจากต้นทุนอาหารที่บางเฉียบและต้องพึ่งพาจุดตรวจนี้เพื่อจับการส่งของขาด ส่งของเกิน บรรทัดที่ส่งมาน้อย และการคลาดเคลื่อนของราคาก่อนที่จะกลายเป็นความผิดพลาดของคลังสินค้าหรือการจ่ายเงินเกิน การบังคับให้หมายเลขใบกำกับ + ผู้ขายเป็น unique การกำหนด cost allocation ก่อน commit การ review ที่แยกตามบทบาท การกระทบสต๊อกเฉพาะตอนโพสต์ และสถานะหลังโพสต์ที่ immutable มีอยู่เพื่อให้ทุกการรับของอธิบายได้และรองรับ three-way match กับใบกำกับที่จะตามมา

ฝั่งการเงินก็สะท้อนสิ่งนี้ GRN ที่ commit แต่ละใบจะเขียน journal entry ที่เดบิต inventory (หรือ expense สำหรับสินค้าที่ไม่ใช่ inventory) ตามมูลค่าที่รับบวกค่าใช้จ่ายเพิ่มเติมที่จัดสรร (freight, duties, handling) เครดิต accounts payable ตามภาระต่อผู้ขาย และบันทึกภาษีแยกตามวิธี tax-inclusive หรือ tax-exclusive ของบรรทัด การรับของข้ามสกุลเงินจะ dual-post ทั้งสกุลธุรกรรมและสกุลฐานโดยใช้อัตราแลกเปลี่ยนที่ตั้งค่าได้ การรับ consignment ถูกติดตามเป็น `Consignment In` ที่ไม่กระทบมูลค่าสต๊อกของกิจการ และสินค้าที่ไม่ใช่ inventory จะข้ามผลกระทบต่อสต๊อกไปเลยขณะยังคงสร้างรายการ AP

ความปลอดภัยของอาหารและการควบคุมคุณภาพอยู่ทับชั้นบนทั้งหมดนี้ สินค้าที่เน่าเสียง่ายจะมีหมายเลข lot (สร้างอัตโนมัติหรือเขียนทับด้วยมือ) และวันหมดอายุที่เดินทางไปกับ stock layer เพื่อให้ FIFO consumption การตัดจำหน่ายตามวันหมดอายุ และเหตุการณ์เรียกคืน (recall) สามารถสืบย้อนกลับไปยัง GRN ต้นทาง workflow การรับของสามารถ flag การส่งของให้ตรวจสอบคุณภาพ — รับ ปฏิเสธ หรือรับบางส่วน — โดยปริมาณที่ปฏิเสธจะป้อนเข้าสู่ตัวชี้วัดผลงานผู้ขาย ความคลาดเคลื่อนของราคาระหว่าง PO กับการรับของจะถูกแสดงเพื่อตรวจสอบเทียบกับ vendor pricelist ก่อน commit

## 3. แนวคิดสำคัญ

- **Receiving Lot**: หมายเลข lot ที่กำหนดให้แต่ละบรรทัดของสินค้าที่ติดตาม lot ตอนรับของ เชื่อมสต๊อกที่รับเข้ามากับ batch หนึ่ง ๆ พร้อมวันหมดอายุและ flag FOC ที่ใส่ได้ ระบบสร้างหมายเลข lot อัตโนมัติในรูปแบบที่ตั้งค่าได้ (ส่วนประกอบของวันที่ ตัวระบุสินค้า ลำดับท้าย) และอนุญาตให้เขียนทับด้วยมือ ประวัติ lot ถูกเก็บไว้ตั้งแต่ต้นจนจบ เพื่อให้ FIFO consumption การตัดจำหน่ายตามวันหมดอายุ และเหตุการณ์ recall สืบย้อนกลับไปยัง GRN ต้นทางได้
- **Three-Way Match**: การกระทบยอดเอกสารสามฉบับ — ใบสั่งซื้อ (สั่งอะไร) GRN (รับอะไร) และใบกำกับจากผู้ขาย (เรียกเก็บอะไร) — เพื่ออนุญาตการจ่ายเงินให้ผู้ขาย GRN คือ leg ฝั่งรับของ ปริมาณและราคาที่ตรงกันจะปลดล็อกการอนุมัติใบกำกับ ส่วนความคลาดเคลื่อน (ส่งขาด เรียกเก็บเกิน ราคาไม่ตรง) จะ flag ใบกำกับให้ AP review
- **Partial Receipt**: GRN ที่ทำให้ PO ที่ค้างอยู่สำเร็จเพียงบางส่วน ปริมาณที่รับสามารถน้อยกว่าที่สั่งได้ และ PO ต้นทางจะย้ายไปสถานะ `Partial` พร้อมปริมาณคงเหลือที่เปิดสำหรับ GRN ในอนาคต สามารถออก GRN หลายใบกับ PO เดียวกันได้จนกว่า PO จะรับครบหรือ `Closed`
- **Over/Under Receipt**: ความคลาดเคลื่อนของปริมาณระหว่าง PO กับ GRN รับขาด (รับน้อยกว่าที่สั่ง) จะปล่อย PO ไว้ที่ `Partial` รับเกิน (รับมากกว่าที่สั่ง) ถูกจำกัดด้วย validation (`Received quantity cannot exceed ordered minus already-received minus cancelled`) และต้องมีการแก้ไข PO ต้นทางหรือปฏิเสธส่วนที่เกิน การยกเลิกของผู้ขายจะถูกบันทึกเป็นปริมาณที่ยกเลิกซึ่งปิดยอดคงเหลือที่เปิดอยู่
- **Receipt Posting (Commit)**: เหตุการณ์เดียวที่อัปเดตสต๊อกและ GL ตอน commit ระบบจะ: (1) เขียนการเคลื่อนไหวสต๊อกเข้าต่อบรรทัด (2) อัปเดต `InventoryStatus.QuantityOnHand`, `LastUnitCost`, `TotalCost` (3) สร้าง FIFO cost layer ใหม่หรือคำนวณ `AverageCostTracking` ใหม่ตามวิธี costing ของสินค้า (4) สร้าง journal entry (เดบิต inventory เครดิต AP พร้อมการจัดการภาษีและค่าใช้จ่ายเพิ่มเติมแยก) (5) อัปเดตปริมาณคงเหลือบน PO ต้นทาง และ (6) ล็อก GRN ไม่ให้แก้ไขต่อ รองรับการ commit แบบเดี่ยว แบบ batch และ auto-commit ที่สิ้นงวด
- **Quality Hold / Inspection**: ประตูที่เลือกใช้ได้ก่อน commit ที่สินค้าที่รับมาจะถูกตรวจสอบและอาจถูกรับ ปฏิเสธ หรือรับบางส่วน ปริมาณที่ปฏิเสธจะไม่ถูกนับเข้าสต๊อกและป้อนเข้าตัวชี้วัดผลงานผู้ขาย การรับแบบมีเงื่อนไขจะถูกบันทึกพร้อมหมายเหตุบนบรรทัด นี่คือสิ่งที่ป้องกันไม่ให้ความผิดพลาดด้านความปลอดภัยอาหาร (ซีลแตก อุณหภูมิผิด กล่องเสียหาย) เข้าสู่คลัง
- **Extra Cost Allocation**: ส่วนประกอบ landed cost (freight, handling, duties, บริการบุคคลที่สาม) ที่เพิ่มลงใน GRN และจัดสรรข้ามรายการสินค้าตามปริมาณหรือตามมูลค่า ส่วนเพิ่มเติมที่จัดสรรจะไหลเข้าสู่การคำนวณ last-cost ของสินค้า — `Last Cost = (Net Amount + Extra Costs) / (Received Quantity + FOC Quantity)` — เพื่อให้ valuation ของ inventory สะท้อนต้นทุนส่งมอบจริง ไม่ใช่แค่ราคาต่อหน่วย
- **FOC (Free of Charge)**: โหมดการรับของที่สินค้าจากผู้ขายมาในราคา 0 (ตัวอย่าง โบนัสส่งเสริม การแทนสินค้าที่เสียหาย) ปริมาณ FOC ไม่ถูกรวมใน `Subtotal` และ `Last Price` แต่รวมในตัวหารของ `Last Cost` และในปริมาณ lot เพื่อให้ต้นทุนต่อหน่วยเฉลี่ยถูกเจือจางอย่างถูกต้องโดยไม่ตีค่าการรับสูงเกินจริง
- **Transaction Types**: แต่ละบรรทัดของ GRN จะ resolve เป็นหนึ่งใน movement type สามแบบตอน commit — `Stock In` (inventory ที่กิจการเป็นเจ้าของแบบปกติ ผลกระทบเต็มต่อคงคลังและ valuation), `Consignment In` (สต๊อกที่ผู้ขายเป็นเจ้าของและฝากที่สถานที่ของกิจการ ติดตามแยก ไม่กระทบสต๊อกของกิจการหรือ AP จนกว่าจะถูกใช้) หรือ `Non-Inventory` (ลงเป็น expense ทันที ไม่กระทบสต๊อกแต่มี AP และภาษีเต็ม)

## 4. บทบาทและ Persona

| Role | ความรับผิดชอบ |
|------|----------------|
| Store Keeper / Receiving Clerk | รับการส่งของจริงที่ dock นับและตรวจสอบสินค้าเทียบกับ PO และใบส่งของของผู้ขาย สร้าง GRN ในสถานะ `Received` (Draft) แนบใบส่งสินค้าและหลักฐานคุณภาพ บันทึกหมายเลข lot และวันหมดอายุ และเอกสารความคลาดเคลื่อน การส่งของขาด หรือบรรทัดที่ปฏิเสธ |
| Store Manager / Inventory Manager | กระทบยอด GRN กับสต๊อกจริงที่รับ ดูแลการติดตาม lot/batch และการกำหนด storage location ติดตามรูปแบบการรับและความคลาดเคลื่อน และ commit GRN รายใบหรือดำเนินการ batch commit ที่สิ้นกะหรือสิ้นงวด |
| Purchaser / Procurement Officer | เป็นเจ้าของ PO ต้นทางที่ GRN ถูกจับคู่ด้วย ตรวจสอบข้อมูลการรับของสำหรับ PO ที่ตนออก สอบสวนปัญหาผลงานผู้ขาย (สาย ขาด เสียหาย ผิดสินค้า) และประสานการแก้ปัญหากับผู้ขายสำหรับความคลาดเคลื่อน |
| Finance Team / AP Clerk | ตรวจสอบ GRN เทียบกับใบกำกับจากผู้ขาย (three-way match) ตรวจสอบการจัดสรรค่าใช้จ่ายเพิ่มเติมและการจัดการภาษี ปรับและสรุปรายการ GRN ก่อนโพสต์ AP กระทบยอด inventory sub-ledger กับ GL และเซ็นรับรองกิจกรรมการรับของที่สิ้นงวด |
| Department Manager | ตรวจสอบ GRN ที่กระทบ cost-centre ของแผนก ยืนยันว่าสินค้าที่รับตรงกับที่สั่งสำหรับแผนก และติดตามความคลาดเคลื่อนของราคาเทียบกับ vendor pricelist |
| System Administrator | ดูแลรูปแบบการสร้างหมายเลข lot ตั้งค่าสิทธิ์ผู้ใช้และเกณฑ์อนุมัติ บริหารรหัสภาษี อัตราแลกเปลี่ยน และรหัสเหตุผลสำหรับการยกเลิกและปฏิเสธ และดูแลการเชื่อมต่อกับโมดูล PO, Inventory, Finance และ Vendor |

- [[master-data/vendor]] — vendor master ที่อ้างอิงโดยส่วนหัว GRN
- [[master-data/currency]] — สกุลธุรกรรมและอัตราแลกเปลี่ยนสำหรับการรับของที่ dual-post
- [[master-data/tax-profile]] — รหัสภาษีที่ใช้กับบรรทัด GRN
- [[master-data/credit-term]] — เงื่อนไขการชำระเงินที่ copy จาก vendor master ลง GRN
- [[master-data/extra-cost-type]] — ส่วนประกอบ landed cost (freight, duties, handling) ที่จัดสรรข้ามบรรทัด
- [[master-data/delivery-point]] — สถานที่รับของที่สินค้าได้รับการรับจริง
- [[master-data/unit]] — หน่วยนับสำหรับปริมาณที่รับ
- [[master-data/location]] — สถานที่จัดเก็บที่แต่ละบรรทัดเขียนการเคลื่อนไหวสต๊อกเข้า
- [[system-config/workflow]] — workflow อนุมัติ / commit สำหรับการอนุญาต GRN
- [[system-config/period]] — ประตูงวดบัญชีสำหรับการโพสต์ GRN
- [[system-config/running-code]] — การกำหนดลำดับหมายเลขเอกสาร GRN
- [[reporting-audit/activity]] — บันทึกการเปลี่ยนสถานะและการแก้ไข GRN สำหรับ audit
- [[reporting-audit/attachment]] — ใบส่งสินค้า ใบส่งของ และหลักฐานคุณภาพที่แนบกับแต่ละ GRN
