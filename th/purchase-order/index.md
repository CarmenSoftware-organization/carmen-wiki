# ใบสั่งซื้อ (Purchase Order)

## 1. ภาพรวม

**ใบสั่งซื้อ (Purchase Order — PO)** คือเอกสารผูกพันภายนอกอย่างเป็นทางการที่ผู้ซื้อออกให้กับผู้ขาย ซึ่งผูกพันองค์กรให้จัดซื้อสินค้าหรือบริการที่ระบุไว้ตามราคาต่อหน่วย ปริมาณ วันที่ส่งมอบ และเงื่อนไขการชำระเงินที่ตกลงกัน PO แต่ละใบประกอบด้วยส่วนหัว — หมายเลขอ้างอิงที่ไม่ซ้ำกัน ผู้ขาย วันที่สั่งซื้อ วันที่ต้องการรับของ จุดส่งของ สกุลเงินและอัตราแลกเปลี่ยน เงื่อนไขการชำระเงินและการส่งมอบ สถานะ ผู้สร้าง และยอดรวมที่รวบยอด — และรายการสินค้าหนึ่งรายการขึ้นไปที่บรรจุข้อมูลสินค้าจากแคตตาล็อกหรือคำอธิบายแบบอิสระ ปริมาณที่สั่ง หน่วยนับ ราคาต่อหน่วย ส่วนลด การจัดการภาษี ปริมาณ FOC และลิงก์ traceability กลับไปยังบรรทัด Purchase Request ต้นทาง ยอดรวมระดับส่วนหัว (subtotal, total discount, total tax, grand total) ถูกคำนวณจากค่าระดับบรรทัดที่ปัดเศษแล้ว และ dual-post ทั้งในสกุลเงินที่ใช้บันทึกธุรกรรมและสกุลเงินฐาน

วงจรชีวิตของ PO ขับเคลื่อนด้วยสถานะ: `Draft` (แก้ไขได้ ยังไม่มีการผูกพัน) → `Sent` (ส่งให้ผู้ขายแล้วและสร้างข้อผูกพันงบประมาณที่แน่นอน) → ตามด้วย `Partially Received` เมื่อมีการ post GRN ใส่ → `Fully Received` เมื่อทุกบรรทัดถูก match → `Closed` เมื่อ PO ถูกปิดทางธุรการ (หรือ `Voided` หากถูกยกเลิกก่อนการรับของ) การลบ PO ทำได้เฉพาะในสถานะ `Draft` เท่านั้น PO ที่ active แล้วทำได้แค่ void หรือ close เพื่อรักษา audit trail ไว้ การแก้ไข (ราคา ปริมาณ วันส่งของ เงื่อนไขผู้ขาย) บน PO ที่เปิดอยู่จะถูก version และบันทึกใน activity log การปิด PO แบบสั้น — ยอมรับการรับของบางส่วนเป็นการส่งมอบขั้นสุดท้าย — เป็นการกระทำที่ตั้งใจซึ่งจะปลดข้อผูกพันส่วนที่เหลือ

PO เกิดขึ้นได้ทั้งแบบ manual (สร้าง PO เปล่าจากศูนย์) หรือโดยการแปลงจาก Purchase Request หนึ่งใบหรือมากกว่าที่ผ่านการอนุมัติ เมื่อเลือก PR หลายใบเพื่อแปลง ระบบจะจัดกลุ่มโดยอัตโนมัติตาม **vendor + currency** ผลิตเป็น PO หนึ่งใบต่อแต่ละการรวมที่ไม่ซ้ำกัน รวมบรรทัด PR เข้าไว้ในนั้นในขณะที่ยังคง traceability จาก PR สู่ PO ในทุกบรรทัด PO จึงเป็นเอกสารที่ผู้ขายจะส่งของตามนั้น ผู้รับสร้าง Good Receive Note และ three-way match (PO ↔ GRN ↔ ใบแจ้งหนี้ผู้ขาย) เคลียร์สินค้าก่อน post เข้าบัญชีเจ้าหนี้

## 2. บริบททางธุรกิจ

PO คือจุดที่คำขอภายในกลายเป็นข้อผูกพันภายนอก ก่อนหน้านี้รายจ่ายมีอยู่เพียงในรูปของ soft commitment ต่องบประมาณเท่านั้น เมื่อออก PO ก็จะแปลงเป็นข้อผูกพันที่แน่นอนพร้อมภาระผูกพันทางกฎหมายต่อผู้ขายตามเงื่อนไขที่ตกลงกัน การเปลี่ยนผ่านเพียงครั้งเดียวนี้คือสิ่งที่ให้อำนาจฝ่ายการเงินและฝ่ายจัดซื้อในการควบคุมการใช้จ่ายที่ไม่ผ่านระบบ โดยการบังคับให้ทุกข้อผูกพันภายนอกผ่าน PO ที่มีเอกสารกำกับ พร้อมหมายเลขอ้างอิงที่ไม่ซ้ำกัน ผู้ขายที่ได้รับการอนุมัติ ราคา pricelist ที่ผ่านการตรวจสอบ และการตรวจสอบงบประมาณ องค์กรจะป้องกันการสั่งซื้อที่ไม่ผ่านระบบและทำให้ใบแจ้งหนี้ทุกใบในอนาคตต้องมีการอนุมัติที่ match กัน

โมดูลนี้คือกระดูกสันหลังของการเชื่อมต่อในห่วงโซ่ procure-to-pay ฝั่งต้นน้ำ PR ป้อนเข้ามาพร้อมการจัดสรรผู้ขายและปริมาณที่อนุมัติ PO ผูกพันปริมาณและราคาเหล่านั้นกับผู้ขาย โมดูล GRN รับของเข้ามาตาม PO และตรวจสอบปริมาณที่สั่ง vs ที่รับ vs ที่ยอมรับ โมดูลคลังสินค้าเพิ่ม on-order ตอน PO ส่ง และ on-hand ตอน GRN post โมดูล vendor-pricelist ให้และตรวจสอบราคาต่อหน่วย ฝ่ายการเงินรับข้อผูกพันที่แน่นอน และเมื่อรับใบแจ้งหนี้ก็จะรัน three-way match ก่อน post เข้าบัญชีเจ้าหนี้ Document management (attachments, comments, activity log) ทำให้ PO ทุกใบมี audit trail ครบถ้วน — ใครสร้าง อะไรถูกแก้ ส่งเมื่อไหร่ ใครรับ และเมื่อไหร่ที่ถูกปิด

ความถูกต้องทางการเงินถูกบังคับใช้ที่ชั้นการคำนวณ Item subtotal, discount, net amount, tax และ total ถูกปัดเศษแยกที่ระดับบรรทัดด้วย half-up (banker's) rounding โดยใช้ทศนิยม 3 ตำแหน่งสำหรับปริมาณ 2 ตำแหน่งสำหรับเงิน และ 5 ตำแหน่งสำหรับอัตราแลกเปลี่ยน ยอดรวมระดับส่วนหัวของ PO roll up จากค่าบรรทัดที่ปัดเศษแล้ว PO ข้ามสกุลเงิน dual-post พร้อมการจัดการอัตราแลกเปลี่ยนที่ชัดเจน PO ต้องกระทบยอดได้สะอาดกับ PR ต้นทางและกับ GRN และใบแจ้งหนี้ที่ตามมา ดังนั้นวินัยการปัดเศษแบบเดียวกันจึงต้องเดินตลอดจนจบของการคำนวณ procure-to-pay

## 3. แนวคิดสำคัญ

- **PO Header**: เรคคอร์ดระดับธุรกรรมที่บรรจุผู้ขาย หมายเลขอ้างอิง วันที่สั่งและวันที่ต้องการรับ สกุลเงินและอัตราแลกเปลี่ยน จุดส่งของ เงื่อนไขการชำระเงินและการส่งมอบ สถานะ ยอดรวม และข้อมูล audit ส่วนหัวเป็นตัวผูกรายการสินค้าทั้งหมดเป็นข้อผูกพันใบเดียวต่อผู้ขายรายเดียวในสกุลเงินเดียว
- **PO Line / PO Item**: บรรทัดบน PO ที่แทนสินค้าหนึ่งรายการหรือรายการแบบอิสระพร้อมปริมาณที่สั่ง หน่วยนับ ราคาต่อหน่วย ส่วนลด อัตราภาษี ปริมาณ FOC ยอดรวมต่อบรรทัดที่ระบบคำนวณ และฟิลด์ traceability (`prItemId`, `prNumber`) เมื่อมาจาก Purchase Request บรรทัดคือหน่วยของการรับของ three-way match และการ post AP
- **Delivery Terms**: ข้อกำหนด Incoterm หรือเทียบเท่าที่ระบุว่ากรรมสิทธิ์เปลี่ยนมือที่ใด ใครจ่ายค่าขนส่งและประกัน และภาระการส่งมอบของผู้ขายสิ้นสุดที่ใด (เช่น จุดส่งของ การขนถ่าย on-premise) อยู่บนส่วนหัวและใช้โดยฝ่ายรับของและฝ่ายการเงิน
- **Payment Terms**: เงื่อนไขเครดิตที่ตกลงกับผู้ขาย (เช่น net 30, 2/10 net 30, COD) มาจาก vendor master คัดลอกลง PO header ตอนสร้าง และใช้โดย AP เพื่อคำนวณวันครบกำหนดและช่วงส่วนลดบนใบแจ้งหนี้ที่ตามมา
- **Amendment**: การแก้ไขที่ควบคุมไว้บน PO ที่ active — ราคา ปริมาณ วันส่งของ เงื่อนไข หรือการเพิ่ม/ลบบรรทัด — บันทึกเป็นเหตุการณ์แบบ version ใน activity log การแก้ไขจะปรับข้อผูกพันที่เปิดอยู่และ propagate ไปยังงบประมาณและ on-order ของคลังสินค้า การยืนยันใหม่จากผู้ขายมักจำเป็นสำหรับการเปลี่ยนแปลงที่สำคัญ
- **Open vs Closed PO**: PO ที่ **open** มีปริมาณคงค้างที่ต้องรับหรือยังไม่ถูกปิดทางธุรการ PO ที่ **closed** ถูก finalise — รับครบและปิดแล้ว หรือ short-close พร้อมปล่อยข้อผูกพันที่เหลือ PO ที่ปิดแล้วไม่สามารถรับ GRN เพิ่มได้และจะ read-only ยกเว้นเพื่อการรายงานและ audit
- **Three-Way Match**: การควบคุม AP ที่เปรียบเทียบบรรทัด PO (สิ่งที่สั่ง) บรรทัด GRN (สิ่งที่รับและยอมรับ) และบรรทัดใบแจ้งหนี้ผู้ขาย (สิ่งที่เรียกเก็บ) ในด้านปริมาณและราคาก่อนอนุมัติใบแจ้งหนี้เพื่อชำระเงิน ความคลาดเคลื่อนจะถูกส่งกลับไปยังฝ่ายจัดซื้อหรือฝ่ายการเงินเพื่อแก้ไข
- **Voided PO**: PO ที่ active ที่ถูกยกเลิกก่อนมี GRN ใด ๆ post การ void จะปล่อยข้อผูกพันงบประมาณ ทำให้ PO แก้ไขไม่ได้ และเก็บเรคคอร์ดไว้สำหรับ audit PO ที่มีอย่างน้อยหนึ่ง GRN ไม่สามารถ void ได้ — ต้องปิดแทน
- **Vendor + Currency Grouping**: กฎการแปลงที่แบ่งชุด PR ที่เลือกเป็น PO หนึ่งใบต่อแต่ละการรวมของ vendor และ currency ที่ไม่ซ้ำกัน เพื่อให้แน่ใจว่า PO แต่ละใบเป็น single-vendor และ single-currency รวมบรรทัด PR ที่มีสิทธิ์เข้าด้วยกัน และรักษาแนวปฏิบัติการจัดซื้อให้สะอาด
- **PR-to-PO Traceability**: ลิงก์ถาวรจากบรรทัด PO กลับไปยังบรรทัด PR ต้นทาง (`prItemId`, `prNumber`) คงอยู่ผ่านการแก้ไขและการรับของบางส่วน เพื่อให้ผู้ตรวจสอบและผู้ปฏิบัติงานสามารถ trace สินค้าที่รับใด ๆ กลับไปยังความต้องการที่ขอได้
- **FOC (Free of Charge)**: ฟิลด์ระดับบรรทัดสำหรับสินค้าที่ผู้ขายให้ฟรี (ตัวอย่าง โบนัสโปรโมชั่น) ปริมาณ FOC ไม่นับรวมใน subtotal ของ PO แต่ flow ผ่านไปยัง GRN เพื่อให้ฝั่งรับของบันทึกในคลังสินค้า
- **Exchange Rate**: อัตราการแปลงที่บันทึกบน PO header ตอนสร้าง ใช้ในการ dual-post ยอด PO ในสกุลเงินฐาน ล็อกอยู่บน PO เพื่อให้ข้อผูกพันและการรับของและใบแจ้งหนี้ที่ตามมากระทบยอดบน basis ที่มั่นคง

## 4. บทบาทและ Personas

| บทบาท | ความรับผิดชอบ |
|------|----------------|
| Procurement Officer / Purchaser (เจ้าหน้าที่จัดซื้อ) | สร้าง PO แบบ manual หรือโดยการแปลง PR ที่อนุมัติแล้ว ตรวจสอบการจัดสรรผู้ขายและราคา pricelist ตั้งเงื่อนไขการส่งมอบและการชำระเงิน ส่ง PO ให้ผู้ขาย จัดการการแก้ไขและติดตามผล และ void หรือปิด PO ตามวงจรชีวิตที่กำหนด |
| Procurement Manager (ผู้จัดการฝ่ายจัดซื้อ) | กำกับดูแลฟังก์ชันจัดซื้อ อนุมัติ PO และการแก้ไขที่มูลค่าสูงหรือมีความสำคัญเชิงกลยุทธ์ จัดการความสัมพันธ์และลำดับผู้ขาย และปรับแต่งกฎการแปลงและการจัดกลุ่ม ถืออำนาจ delete-in-draft และ override |
| Vendor (ผู้ขาย) | คู่ค้าภายนอกที่รับ PO ยืนยันการตอบรับ ส่งมอบสินค้าตามเงื่อนไขที่ตกลง และออกใบแจ้งหนี้ที่ถูก match กับ PO และ GRN ก่อนการชำระเงิน |
| Receiver / Store Keeper (ผู้รับสินค้า / พนักงานคลัง) | บทบาทปลายน้ำที่รับสินค้าจริง สร้าง GRN ตาม PO และตรวจสอบปริมาณที่รับและที่ยอมรับทีละบรรทัด trigger การเพิ่ม on-hand ของคลังสินค้าและการเปลี่ยนสถานะ partial-/fully-received บน PO |
| Inventory Manager (ผู้จัดการคลังสินค้า) | จัดการการรับของสำหรับสถานที่ ดูแลการสร้าง GRN และปิด PO เมื่อการรับของเสร็จสมบูรณ์หรือถูกยอมรับเป็นการส่งมอบขั้นสุดท้าย |
| Finance Officer / Accounts Payable (เจ้าหน้าที่การเงิน / บัญชีเจ้าหนี้) | ตรวจสอบเงื่อนไข PO และความถูกต้องทางการเงิน รัน three-way match บนใบแจ้งหนี้ผู้ขายเทียบกับ PO และ GRN post หนี้ AP ตอน match ใบแจ้งหนี้ และ flag ความคลาดเคลื่อนกลับไปยังฝ่ายจัดซื้อ |
| Finance Manager (ผู้จัดการฝ่ายการเงิน) | ตรวจสอบ PO และการแก้ไขที่มูลค่าสูงก่อนการส่ง ตรวจสอบการจัดการสกุลเงินและอัตราแลกเปลี่ยน และกำกับดูแลการควบคุมและการรายงานฝั่ง AP |
| System Administrator (ผู้ดูแลระบบ) | กำหนดค่าการกำหนดหมายเลข PO การเปลี่ยนสถานะ RBAC การเชื่อมต่อ vendor และ pricelist การเชื่อมต่องบประมาณและคลังสินค้า document templates และกฎการแปลง/จัดกลุ่ม |
| Auditor (ผู้ตรวจสอบ) | เข้าถึง PO การแก้ไข และ activity log ในโหมด read-only เพื่อตรวจสอบความสอดคล้องกับนโยบาย การแยกหน้าที่ ความสมบูรณ์ของ three-way match และ traceability ปลายต่อปลายตั้งแต่ PR ผ่าน PO และ GRN ไปจนถึงใบแจ้งหนี้ |
