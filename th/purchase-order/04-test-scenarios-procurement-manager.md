# ใบสั่งซื้อ (Purchase Order) — Test Scenarios — Procurement Manager

> **At a Glance**
> **Persona:** Procurement Manager (อนุมัติมูลค่าสูง + การปรับ rule configuration) &nbsp;·&nbsp; **Module:** [[purchase-order]] &nbsp;·&nbsp; **Scenarios:** ~29
> **Categories:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** map ไปยัง `401-po.spec.ts`, `403-po-approver-journey.spec.ts` ใน `../carmen-inventory-frontend-e2e/`

หน้านี้ capture test scenarios ที่ Procurement Manager persona ขับเคลื่อนในโมดูล `purchase-order` ต่างจาก Purchaser — ที่เป็นเจ้าของเอกสาร end-to-end — Procurement Manager engage กับ PO ข้าม **สอง surfaces ที่แตกต่าง** ([03-user-flow-procurement-manager.md](./03-user-flow-procurement-manager.md) Section 1) **Transactional surface** คือ gate การอนุมัติมูลค่าสูงที่ Purchaser self-clear ไม่ได้ภายใต้ `PO_AUTH_004`: POs ที่ `tb_purchase_order.total_amount` เกิน tenant threshold (หรือมี pricelist deviation เกิน tolerance ตาม `PO_XMOD_006`) escalate ไปยังคิว **My Approvals** ของ Manager ที่ Manager รัน **Approve & Transmit** (`PO_POST_004`), **Return to Buyer** (`PO_POST_005`), หรือ **Reject** (`PO_POST_010`) **Configurational surface** คือ workbench ปรับ rule — vendor ranking และ allocation, Convert-to-PO `(vendor_id, currency_id)` grouping, unit conversion factors, pricelist tolerance band, และ high-value threshold เอง — ที่ edits ส่งผลกระทบต่อ POs ใหม่เท่านั้นเพราะ in-flight POs เก็บ snapshot ของ rule version ก่อน Manager เพิ่มเติมถือ **override authorities** ที่ Purchaser ใช้ไม่ได้: soft-delete-in-draft (`PO_AUTH_005`), void จาก non-terminal state ใด ๆ (`PO_AUTH_007`, `PO_POST_010`), และ early-close จาก `partial → closed` (`PO_AUTH_008`, `PO_POST_011`) Scenarios ด้านล่าง mirror รูปแบบ five-section เดียวกับไฟล์ Purchaser: happy paths trace ทั้ง transactional escalation cycle และ configurational rule edit; permission section pin `PO_AUTH_004`–`PO_AUTH_007` ลงเทียบกับ stages ข้างเคียง; validation section ครอบคลุม threshold mismatches, snapshot preservation, และ rule-edit integrity; และ edge cases ครอบคลุม boundary-amount POs, in-flight rule changes, bulk failures, และ delegation timing

## 2. Permission / Authorization

| # | Scenario | Expected behaviour (allow/deny + เหตุผล) |

## 3. Validation / Error

| # | Scenario | Trigger | Expected error |
| - | -------- | ------- | -------------- |
| PM-VAL-01 | PM พยายาม approve PO ที่ stage มูลค่าสูงที่ `total_amount` ไม่เกิน threshold อีกต่อไป (Purchaser-side edit ลด amount หลัง submit) | เปิด PO; คลิก **Approve PO** ใน footer | Reject — workflow layer ตรวจจับ threshold mismatch เมื่อ re-evaluate stage ที่ confirm time Dialog แสดง `"This PO no longer requires high-value approval — the buyer will self-approve on resubmit."` และ PO ถูก route กลับเป็น `draft` อัตโนมัติสำหรับ re-submission (หรือ hold ที่ stage ปัจจุบันขึ้นอยู่กับ tenant's workflow policy) ไม่มี status change เป็น `sent`; `PO_POST_004` ไม่ fire |
| PM-VAL-02 | PM save configuration rule edit ที่ค่าใหม่เหมือนกับค่าปัจจุบัน (ไม่มี diff) | เปิด **Vendor Ranking & Allocation** → tweak sort field แล้ว revert เป็นค่าดั้งเดิม → **Save** | Reject — save handler คำนวณ diff และปฏิเสธเขียน no-op version Server return `"Rule change rejected — new value is identical to the current version."` ไม่มี rule version ใหม่เขียน; ไม่มี audit-log entry; ไม่มี configuration-change notification fire |
| PM-VAL-03 | PM update high-value threshold ขณะที่ borderline PO นั่งที่ `in_progress` บน stage มูลค่าสูง — confirm ว่า in-flight PO **ไม่** re-evaluate เทียบกับ threshold ใหม่ | เปิด **Workflow Threshold**; raise threshold จาก ฿100k เป็น ฿150k; borderline PO ที่ `total_amount = ฿120k` นั่งที่ `in_progress` บน stage มูลค่าสูง Save | ยอมรับ — threshold ใหม่ใช้ได้สำหรับ POs ใหม่เท่านั้น In-flight PO ดำเนินต่อผ่าน stage มูลค่าสูงบน threshold เก่าตามหลักการ snapshot (ดู [03-user-flow-procurement-manager.md](./03-user-flow-procurement-manager.md) Section 3 — "Save and apply going forward") ความพยายาม re-route in-flight PO ใด ๆ gate หลัง tenant's "Save and re-route in-flight" option ที่ tenants ส่วนใหญ่ disable เพื่อ audit safety Server บันทึก rule change ด้วย `effective_from = now()` และ audit log capture in-flight-PO warning ที่แสดง |
| PM-VAL-04 | PM update **Vendor Ranking** scoring weights ที่ไม่ sum เป็น 1.0 (หรือ 100%) | เปิด **Vendor Ranking & Allocation** → ปรับ on-time-delivery weight เป็น 0.5, three-way-match-success เป็น 0.3, deviation-rate เป็น 0.3 (sum = 1.1) Save | Reject — server return `"Scoring weights must sum to 1.0 (or 100 %)."` ไม่มี rule version ใหม่เขียน UI อาจบังคับ constraint inline ผ่าน live-sum readout |
| PM-VAL-05 | PM update unit-conversion factor เป็น `0` หรือค่า negative | เปิด **Unit Conversion** สำหรับ product `P1`; ตั้ง `order_unit_conversion_factor = 0` หรือ `-0.5` Save | Reject — validation re-use `PO_VAL_009` (`order_unit_conversion_factor > 0`) Server return `"Order UoM must have a positive conversion factor to base UoM."` ไม่มี rule version ใหม่เขียน PO ใหม่ใด ๆ ที่สร้างเทียบกับ `P1` ยังคงใช้ factor ก่อน |
| PM-VAL-06 | PM พยายาม **Send Back** หรือ **Reject** PO โดยไม่ให้ reason | เปิด `in_progress` PO → mark items เป็น Review (หรือ Reject) → คลิก footer **Send Back** (หรือ **Reject**) → ปล่อย reason field ว่าง → confirm | Reject — ปุ่ม confirm ของ dialog ยัง disabled ขณะที่ reason field ว่าง (สำหรับ Send-Back path ที่ reason field บังคับตาม [03-user-flow-procurement-manager.md](./03-user-flow-procurement-manager.md) Section 2) สำหรับ Reject path reason documented เป็น optional ใน TC-PO-070310 แต่ server-side guard ยอมรับ reason ว่างและบันทึก `tb_purchase_order_comment` entry ว่าง Tenants ที่ต้องการ reason บน Reject ตั้งค่า field เป็น mandatory ผ่าน workflow definition |
| PM-VAL-07 | PM พยายาม action ใด ๆ (Approve / Send Back / Reject / Void / Close) บน PO ที่ terminal state แล้ว (`voided`, `completed`, `closed`) | เปิด `voided` PO; toolbar buttons ซ่อน / disabled; force direct API call เพื่อ advance status | Reject — ตาม `PO_VAL_015` ("Status transitions follow the state machine in Section 5; out-of-order transitions are blocked") Server return `"Invalid status transition from voided to sent"` (หรือ from-state เปรียบเทียบ) Terminal-state guard บังคับใช้ที่ API layer โดยไม่คำนึงถึง UI button-disable state |
| PM-VAL-08 | PM พยายาม early-close `partial` PO ที่ทุกบรรทัดรับครบแล้ว (`received_qty = order_qty − cancelled_qty` บนทุกบรรทัด) | เปิด `partial` PO ที่ทุกบรรทัดพอใจเงื่อนไข full-receipt; คลิก **Close** | Reject — close handler ตรวจจับว่า PO ควรอยู่ที่ `completed` ผ่าน `PO_POST_007` (transition ที่ driven โดย GRN ยังไม่รัน หรือรันแล้วแต่ race ทิ้ง status ไว้ที่ `partial`) Server return `"PO is fully received — please refresh; the document will move to completed."` หรืออีกทางหนึ่งระบบ auto-correct เป็น `completed` ใน read ถัดไป `PO_POST_011` ไม่ fire เพราะไม่มี remainder ให้เขียนกลับเป็น `cancelled_qty` |

## 4. Edge Cases

| # | Scenario | เงื่อนไข | Expected |
| - | -------- | --------- | -------- |
| PM-EDGE-01 | Boundary ของ high-value threshold — `total_amount` เท่ากับ tenant threshold พอดี | Tenant threshold = ฿100,000.00; PO ที่ `total_amount = ฿100,000.00` พอดี; PO submit เป็น `in_progress` | Tenant policy ตัดสิน Default interpretation ตาม `PO_AUTH_004` คือ `total_amount > threshold` (strict greater-than) route ไปยัง Manager; `total_amount = threshold` อาจ self-approve โดย Purchaser Workflow definition's stage rule encode comparator (`>` vs `>=`) เส้นทางใดก็ตามที่ถูกใช้ post-conditions ถูก: route ไปยัง PM ⇒ `user_action.execute` ของ Manager populated; route ไปยัง Purchaser ⇒ `user_action.execute` ของ Purchaser populated ต้องไม่มีความคลุมเครือที่ comparator — threshold เป็น single-sided |
| PM-EDGE-02 | Boundary ระหว่าง in-flight PO snapshot กับ rule version ใหม่ | PO `A` submit ที่ `T0` และ snapshot vendor-ranking version `v1` แล้วนั่งในคิวมูลค่าสูง ที่ `T1` PM save vendor-ranking `v2` PO `B` submit ที่ `T2 > T1` | PO `A` เก็บ snapshot `v1` ผ่าน approve / send-back / reject — แม้เมื่อ Manager re-open หน้า detail, `v1` คือ version of record สำหรับ vendor-allocation context ที่แสดงบนหน้า PO `B` ใช้ `v2` หลักการเดียวกันใช้กับ conversion factor บนบรรทัด — `tb_purchase_order_detail.order_unit_conversion_factor` denormalise บนบรรทัดที่ line-save time และ subsequent rule edit ไม่ retroactively update existing PO lines (ดู `PUR-EDGE-02` ใน [04-test-scenarios-purchaser.md](./04-test-scenarios-purchaser.md)) |
| PM-EDGE-03 | Bulk action ด้วย failure บางส่วน — POs บางตัวใน multi-selection อยู่ที่ terminal states หรือเพิ่ง modify โดย user อื่น | PM เลือก 12 POs สำหรับ **Bulk Void**: 9 ที่ `sent`, 2 ที่ `partial`, 1 ที่ `completed` (terminal) PO `sent` หนึ่งใบใน 9 เพิ่ง transition เป็น `partial` โดย GRN posting ใน session อื่น | Bulk-action handler ประมวลผลแต่ละ PO อิสระ **Result:** PO 1 `completed` skip (terminal — `PO_POST_010` reject); PO 1 `sent`-now-`partial` void จาก `partial` state ปัจจุบัน (ยัง non-terminal); อีก 10 void สำเร็จ Summary toast รายงาน `11 of 12 voided, 1 skipped` และ row-level breakdown มีให้บนคลิก ไม่มี failure เดียว abort bulk operation — แต่ละ PO เป็น transaction ของตัวเอง |
| PM-EDGE-04 | Workflow approval-stage delegation expire mid-decision | Backup approver grant temporary delegation บน stage มูลค่าสูงด้วย `valid_until = T_end` Manager เปิดหน้า detail PO ที่ `T_end − 30s`, mark items Approved, คลิก **Approve PO** ที่ `T_end + 10s` | Stage check ที่ confirm time re-evaluate `user_action.execute` เทียบกับเวลาปัจจุบัน หาก delegation expire แล้ว backup approver ไม่อยู่ในชุดอีกต่อไปและ API call return `"You are not in the authorised approver set for this stage"` ตาม `PO_AUTH_011` Page-level state บน client (items marked Approved) preserve แต่ document-level transition ไม่ fire; user ต้อง refresh และ retry (ถ้า delegation renew) หรือ escalate PO ไปยัง primary approver |
| PM-EDGE-05 | Save vs. submit race บน configuration rule — สอง Managers edit rule เดียวกันขนานกัน | Manager A และ B ทั้งคู่เปิด **Vendor Ranking & Allocation** ที่ version `v3` A save re-rank ที่ `T` (rule advance เป็น `v4`) B save tolerance-band change ที่ `T + 1s` ใช้ stale `v3` baseline | Save แรกชนะ: re-rank ของ A persist เป็น `v4` save ของ B ถูก reject โดย optimistic-concurrency guard ด้วย `"Configuration rule was modified by another user. Please refresh and re-apply your changes."` Client ของ B refresh เป็น `v4`, B re-apply tolerance-band change เทียบกับ `v4`, และ save ที่สามผลิต `v5` Audit log แสดงทั้ง `v4` และ `v5` (A และ B ตามลำดับ) ด้วย rejected race บันทึกเป็น metric ไม่ใช่เป็น rule version ไม่มี double-write หรือ lost-update เกิด |
