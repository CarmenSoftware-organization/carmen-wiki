# ใบสั่งซื้อ — กรณีทดสอบ — ผู้จัดการฝ่ายจัดซื้อ (Procurement Manager)

หน้านี้รวบรวมกรณีทดสอบที่ persona ของ Procurement Manager เป็นผู้ขับเคลื่อนในโมดูล `purchase-order` ต่างจาก Purchaser ที่เป็นเจ้าของเอกสารตั้งแต่ต้นจนจบ Procurement Manager เข้าเกี่ยวข้องกับ PO ผ่าน **สอง surface ที่แตกต่างกัน** ([03-user-flow-procurement-manager.md](./03-user-flow-procurement-manager.md) หัวข้อ 1) **Transactional surface** คือ gate การอนุมัติมูลค่าสูงที่ Purchaser ไม่สามารถอนุมัติเองได้ภายใต้ `PO_AUTH_004`: PO ที่มี `tb_purchase_order.total_amount` เกิน threshold ของ tenant (หรือ pricelist deviation เกิน tolerance ตาม `PO_XMOD_006`) จะถูก escalate เข้าคิว **My Approvals** ของ Manager ซึ่ง Manager จะดำเนินการ **Approve & Transmit** (`PO_POST_004`), **Return to Buyer** (`PO_POST_005`) หรือ **Reject** (`PO_POST_010`) **Configurational surface** คือ workbench สำหรับปรับ rule — vendor ranking and allocation, Convert-to-PO `(vendor_id, currency_id)` grouping, unit conversion factors, pricelist tolerance band และ high-value threshold ซึ่งการ edit จะมีผลกับ PO ใหม่เท่านั้น เพราะ PO ที่กำลังอยู่ระหว่างทาง (in-flight) จะเก็บ snapshot ของ rule version ก่อนหน้าไว้ Manager ยังมี **override authority** ที่ Purchaser ไม่มีคือ soft-delete-in-draft (`PO_AUTH_005`), void จาก state ที่ไม่ใช่ terminal (`PO_AUTH_007`, `PO_POST_010`) และ early-close จาก `partial → closed` (`PO_AUTH_008`, `PO_POST_011`) กรณีทดสอบด้านล่างมีโครงสร้าง 5 หัวข้อเหมือนหน้า Purchaser: happy path ครอบคลุมทั้ง transactional escalation cycle และ rule edit ใน configurational; หัวข้อ permission ระบุขอบเขต `PO_AUTH_004`–`PO_AUTH_007` เทียบกับ stage ข้างเคียง; หัวข้อ validation ครอบคลุม threshold mismatch, การคง snapshot และความสมบูรณ์ของ rule edit; ส่วน edge case ครอบคลุม PO ที่อยู่ที่ขอบ amount, การแก้ rule ขณะ in-flight, partial failure ของ bulk และ timing ของ delegation

## 3. Validation / Error

| # | กรณี | Trigger | ข้อความ error ที่คาดหวัง |
| - | ---- | ------- | ------------------------ |
| PM-VAL-01 | PM พยายาม approve PO ที่ stage high-value ซึ่ง `total_amount` ไม่เกิน threshold แล้ว (การ edit ฝั่ง Purchaser ลด amount หลัง submit) | เปิด PO; กด **Approve PO** ใน footer | Reject — workflow layer ตรวจพบ threshold mismatch เมื่อ re-evaluate stage ณ เวลา confirm Dialog แสดง `"This PO no longer requires high-value approval — the buyer will self-approve on resubmit."` และ PO ถูก route กลับไป `draft` อัตโนมัติเพื่อ re-submit (หรือถือไว้ที่ stage ปัจจุบันขึ้นกับ workflow policy ของ tenant) ไม่มี status change เป็น `sent`; `PO_POST_004` ไม่ fire |
| PM-VAL-02 | PM save configuration rule edit ที่ค่าใหม่เหมือนค่าปัจจุบัน (ไม่มี diff) | เปิด **Vendor Ranking & Allocation** → ปรับ sort field แล้ว revert กลับเป็นค่าเดิม → **Save** | Reject — save handler คำนวณ diff และปฏิเสธการเขียน version ที่เป็น no-op Server คืน `"Rule change rejected — new value is identical to the current version."` ไม่มี rule version ใหม่ถูกเขียน; ไม่มี audit-log entry; ไม่มี configuration-change notification fire |
| PM-VAL-03 | PM update high-value threshold ขณะที่ PO borderline กำลังนั่งที่ `in_progress` บน stage high-value — ยืนยันว่า PO in-flight **ไม่** ถูก re-evaluate กับ threshold ใหม่ | เปิด **Workflow Threshold**; เพิ่ม threshold จาก ฿100k เป็น ฿150k; PO borderline ที่ `total_amount = ฿120k` กำลังนั่งที่ `in_progress` บน stage high-value Save | Accepted — threshold ใหม่มีผลกับ PO ใหม่เท่านั้น PO in-flight ดำเนินต่อผ่าน stage high-value บน threshold เก่าตาม snapshot principle (ดู [03-user-flow-procurement-manager.md](./03-user-flow-procurement-manager.md) หัวข้อ 3 — "Save and apply going forward") การพยายาม re-route PO in-flight ถูก gate ภายใต้ตัวเลือก "Save and re-route in-flight" ของ tenant ซึ่ง tenant ส่วนใหญ่ disable เพื่อความปลอดภัยของ audit Server บันทึก rule change ด้วย `effective_from = now()` และ audit log จับ warning เรื่อง PO in-flight ที่แสดงไว้ |
| PM-VAL-04 | PM update **Vendor Ranking** scoring weights ที่รวมไม่เป็น 1.0 (หรือ 100 %) | เปิด **Vendor Ranking & Allocation** → ปรับ on-time-delivery weight เป็น 0.5, three-way-match-success เป็น 0.3, deviation-rate เป็น 0.3 (รวม = 1.1) Save | Reject — Server คืน `"Scoring weights must sum to 1.0 (or 100 %)."` ไม่มี rule version ใหม่ถูกเขียน UI อาจ enforce constraint inline ผ่าน live-sum readout ก็ได้ |
| PM-VAL-05 | PM update unit-conversion factor เป็น `0` หรือค่าติดลบ | เปิด **Unit Conversion** สำหรับ product `P1`; set `order_unit_conversion_factor = 0` หรือ `-0.5` Save | Reject — validation ใช้ `PO_VAL_009` ซ้ำ (`order_unit_conversion_factor > 0`) Server คืน `"Order UoM must have a positive conversion factor to base UoM."` ไม่มี rule version ใหม่ถูกเขียน PO ใหม่ที่สร้างกับ `P1` ยังคงใช้ factor ก่อนหน้า |
| PM-VAL-06 | PM พยายาม **Send Back** หรือ **Reject** PO โดยไม่ระบุเหตุผล | เปิด PO `in_progress` → mark items เป็น Review (หรือ Reject) → กด footer **Send Back** (หรือ **Reject**) → ปล่อย reason field ว่าง → ยืนยัน | Reject — ปุ่ม confirm ของ dialog ยังคง disable ขณะที่ reason field ว่าง (สำหรับเส้นทาง Send Back ที่ reason field เป็น mandatory ตาม [03-user-flow-procurement-manager.md](./03-user-flow-procurement-manager.md) หัวข้อ 2) สำหรับเส้นทาง Reject reason ถูกระบุไว้ว่า optional ใน TC-PO-070310 แต่ guard ฝั่ง server รับ empty reason และบันทึก `tb_purchase_order_comment` entry ที่ว่าง Tenant ที่ต้องการ reason บน Reject กำหนด field เป็น mandatory ผ่าน workflow definition |
| PM-VAL-07 | PM พยายาม action ใด ๆ (Approve / Send Back / Reject / Void / Close) บน PO ที่อยู่ที่ terminal state แล้ว (`voided`, `completed`, `closed`) | เปิด PO ที่ `voided`; ปุ่ม toolbar ถูก hide / disable; force การเรียก API ตรงเพื่อเลื่อน status | Reject — ตาม `PO_VAL_015` ("Status transitions follow the state machine in Section 5; out-of-order transitions are blocked") Server คืน `"Invalid status transition from voided to sent"` (หรือ from-state ที่เทียบเคียง) Terminal-state guard ถูก enforce ที่ API layer ไม่ว่าสถานะปุ่ม UI disable จะเป็นอย่างไร |
| PM-VAL-08 | PM พยายาม early-close PO ที่ `partial` ซึ่งทุกบรรทัดรับครบแล้ว (`received_qty = order_qty − cancelled_qty` ทุกบรรทัด) | เปิด PO ที่ `partial` ซึ่งทุกบรรทัดผ่าน full-receipt condition; กด **Close** | Reject — close handler ตรวจพบว่า PO ควรอยู่ที่ `completed` ผ่าน `PO_POST_007` (transition ที่ขับเคลื่อนโดย GRN ยังไม่ได้ run หรือ run แล้วแต่ race ทำให้ status ค้างที่ `partial`) Server คืน `"PO is fully received — please refresh; the document will move to completed."` หรือระบบ auto-correct เป็น `completed` ในการ read ครั้งถัดไป `PO_POST_011` ไม่ fire เพราะไม่มี remainder ให้เขียนกลับเป็น `cancelled_qty` |

## 4. Edge Cases

| # | กรณี | เงื่อนไข | ผลที่คาดหวัง |
| - | ---- | -------- | ------------ |
| PM-EDGE-01 | ขอบเขต threshold high-value — `total_amount` เท่ากับ threshold ของ tenant พอดี | Threshold ของ tenant = ฿100,000.00; PO ที่ `total_amount = ฿100,000.00` พอดี; PO submit ไปที่ `in_progress` | ขึ้นกับ policy ของ tenant การตีความ default ตาม `PO_AUTH_004` คือ `total_amount > threshold` (strict greater-than) จะ route ไปที่ Manager; `total_amount = threshold` Purchaser อาจ self-approve ได้ Stage rule ใน workflow definition encode comparator (`>` vs `>=`) ไม่ว่าจะเลือกเส้นทางใด post-condition ถูกต้อง: route ไปที่ PM ⇒ `user_action.execute` ของ Manager ถูกเติม; route ไปที่ Purchaser ⇒ `user_action.execute` ของ Purchaser ถูกเติม ต้องไม่มีความกำกวมที่ comparator — threshold เป็น single-sided |
| PM-EDGE-02 | ขอบเขต snapshot ของ PO in-flight vs. rule version ใหม่ | PO `A` ถูก submit ที่ `T0` และ snapshot vendor-ranking version `v1` แล้วนั่งในคิว high-value ที่ `T1` PM save vendor-ranking `v2` PO `B` ถูก submit ที่ `T2 > T1` | PO `A` คง snapshot `v1` ไว้ตลอดการ approve / send-back / reject — แม้เมื่อ Manager เปิดหน้ารายละเอียดใหม่ `v1` คือ version of record สำหรับ vendor-allocation context ที่แสดงบนหน้า PO `B` ใช้ `v2` หลักการเดียวกันใช้กับ conversion factor บนบรรทัด — `tb_purchase_order_detail.order_unit_conversion_factor` ถูก denormalise เข้าบรรทัดที่เวลา save บรรทัด และการแก้ rule ในภายหลังไม่ retroactive กับบรรทัด PO ที่มีอยู่แล้ว (ดู `PUR-EDGE-02` ใน [04-test-scenarios-purchaser.md](./04-test-scenarios-purchaser.md) ด้วย) |
| PM-EDGE-03 | Bulk action พร้อม partial failure — PO บางส่วนใน multi-selection อยู่ที่ terminal state หรือเพิ่งถูก modify โดย user อื่น | PM เลือก 12 PO สำหรับ **Bulk Void**: 9 ที่ `sent`, 2 ที่ `partial`, 1 ที่ `completed` (terminal) หนึ่งใน 9 PO ที่ `sent` เพิ่ง transition เป็น `partial` จากการ post GRN ใน session อื่น | Bulk-action handler ประมวลผลแต่ละ PO อย่างอิสระ **ผลลัพธ์:** 1 PO ที่ `completed` ถูกข้าม (terminal — `PO_POST_010` ปฏิเสธ); 1 PO `sent`-ที่กลายเป็น-`partial` ถูก void จาก state `partial` ปัจจุบัน (ยังไม่ใช่ terminal); อีก 10 PO ถูก void สำเร็จ Toast สรุปรายงาน `11 of 12 voided, 1 skipped` และมี row-level breakdown ให้กดดู ไม่มี failure เดียวที่ abort bulk operation — แต่ละ PO เป็น transaction ของตัวเอง |
| PM-EDGE-04 | Delegation ของ approval-stage ใน workflow หมดอายุระหว่างการตัดสินใจ | Backup approver ได้รับ delegation ชั่วคราวบน stage high-value พร้อม `valid_until = T_end` Manager เปิดหน้ารายละเอียด PO ที่ `T_end − 30s`, mark items เป็น Approved, กด **Approve PO** ที่ `T_end + 10s` | การตรวจสอบ stage ณ เวลา confirm re-evaluate `user_action.execute` เทียบกับเวลาปัจจุบัน หากเลย delegation หมดอายุแล้ว backup approver จะไม่อยู่ใน set อีกต่อไป และการเรียก API คืน `"You are not in the authorised approver set for this stage"` ตาม `PO_AUTH_011` Page-level state บน client (items ที่ mark Approved) ถูก preserve แต่ document-level transition ไม่ fire; user ต้อง refresh แล้ว retry (กรณี delegation ถูก renew) หรือ escalate PO ไปยัง primary approver |
| PM-EDGE-05 | Save vs. submit race บน configuration rule — Manager สองคนแก้ rule เดียวกันพร้อมกัน | Manager A และ Manager B ทั้งคู่เปิด **Vendor Ranking & Allocation** ที่ version `v3` A save การ re-rank ที่ `T` (rule เลื่อนเป็น `v4`) B save การเปลี่ยน tolerance-band ที่ `T + 1s` โดยใช้ baseline `v3` ที่ stale | First save wins: การ re-rank ของ A ถูก persist เป็น `v4` Save ของ B ถูกปฏิเสธโดย optimistic-concurrency guard ด้วย `"Configuration rule was modified by another user. Please refresh and re-apply your changes."` Client ของ B refresh เป็น `v4`, B re-apply การเปลี่ยน tolerance-band เทียบกับ `v4`, และ save ครั้งที่สามผลิต `v5` Audit log แสดงทั้ง `v4` และ `v5` (A และ B ตามลำดับ) พร้อม race ที่ถูกปฏิเสธบันทึกเป็น metric ไม่ใช่ rule version ไม่มี double-write หรือ lost-update เกิดขึ้น |
