# ใบสั่งซื้อ — เส้นทางผู้ใช้งาน — เจ้าหน้าที่จัดซื้อ (Purchaser)

## 1. บทบาทในโมดูลนี้

**เจ้าหน้าที่จัดซื้อ (Purchaser)** (มีอีกชื่อหนึ่งว่า **Procurement Officer**) เป็นเจ้าของ PO ตั้งแต่การสร้างจนถึงการส่งให้ผู้ขาย — ช่วงตั้งแต่ `draft` ถึง `sent` มีสองเส้นทางการสร้างที่มาบรรจบในเส้นทางเดียวกัน: **PO แบบ manual** (`po_type = manual`) ที่ฝ่ายจัดซื้อสร้างเองโดยตรง และ **PO ที่มาจาก PR** (`po_type = purchase_request`) ที่เกิดจากการ Convert-to-PO ในโมดูลต้นน้ำ [[purchase-request]] ซึ่งเขียน 1 แถวต่อคู่ (บรรทัด PO, บรรทัด PR) ลงใน bridge table `tb_purchase_order_detail_tb_purchase_request_detail` ([01-data-model.md](./01-data-model.md) หัวข้อ 2.5) เมื่อ draft ถูกสร้างแล้ว Purchaser กรอก (หรือ inherit แล้วตรวจ) ส่วนหัว — `vendor_id`, `currency_id`, `exchange_rate`, `credit_term_id`, `order_date`, `delivery_date`, `workflow_id` — แล้วไล่แต่ละบรรทัดเพื่อตรวจราคากับ [[vendor-pricelist]], ปรับจำนวน / ส่วนลด / ภาษี / FOC ในขอบเขตที่ได้รับอนุญาต, ตั้งเงื่อนไขการส่งของและการชำระเงิน และ submit (`draft → in_progress`, `PO_AUTH_003` และ `PO_POST_002`) Purchaser ยังถือ action การส่ง PO ในจังหวะอนุมัติขั้นสุดท้าย (`PO_AUTH_006`, `PO_POST_004`), จัดการการแก้ไข PO ภายใต้ข้อจำกัด post-`sent` ของ `PO_VAL_016`, รัน bounce-back เพื่อส่ง PR กลับให้ผู้ร้องขอเมื่อประเด็น vendor หรือ spec แก้ไม่ได้ และ void PO ใน `draft` เมื่อจำเป็น (`PO_AUTH_007` สงวนการ void จาก state ที่ไม่ใช่ `draft` ให้ Procurement Manager) Purchaser ทำงานภายใต้ `enum_stage_role = purchase` (สะท้อนจาก `PR_AUTH_008` ทางฝั่ง PR)

## 2. จุดเริ่มต้นและเส้นทางหลัก

**จุดเริ่มต้น:** Sidebar → โมดูล **Purchase Order** → **Create Purchase Order** สำหรับ PO แบบ manual หรือ Sidebar → โมดูล **Purchase Request** → คิว **Approved PRs** → workbench **Convert to PO** สำหรับ PO ที่มาจาก PR Notification ทาง in-app และอีเมล "Purchase Request [PR-ID] Ready for PO Conversion" deep-link เข้า workbench โดยตรง ทั้งสองเส้นทางพา Purchaser ไปที่หน้ารายละเอียด PO เดียวกันที่ `po_status = draft`

**เส้นทางหลัก (happy path):**

1. เลือกเส้นทางการสร้าง สำหรับ **PO แบบ manual**: จากหน้า landing ของโมดูล PO กด **Create Purchase Order** แล้วกรอกส่วนหัว (vendor, currency, exchange rate, credit term, order date, delivery date, workflow) บนฟอร์มเปล่า สำหรับ **PO ที่มาจาก PR**: จาก workbench Convert-to-PO เลือกบรรทัดที่อนุมัติแล้วจาก PR หนึ่งบรรทัดหรือมากกว่า ระบบจัดกลุ่มตาม `(vendor_id, currency_id)` และสร้าง draft `tb_purchase_order` หนึ่งใบต่อกลุ่ม insert บรรทัด `tb_purchase_order_detail` ที่ตรงกัน และเขียน bridge entry ด้วยค่า `pr_detail_qty > 0` ต่อคู่ (บรรทัด PO, บรรทัด PR) (`PO_VAL_014`) snapshot ฝั่ง PR — product, UoM, location, delivery point — ถูก denormalise ลงบนแถว bridge ในจังหวะแปลง
2. เปิด **draft PO** ส่วนหัวถูก pre-fill จาก vendor master และ workflow ที่เลือก (เส้นทาง manual) หรือ inherit จาก PR ต้นทาง (เส้นทาง PR-sourced) ตรวจว่า `vendor_id` อ้างถึง vendor ที่ active และไม่อยู่ใน blacklist (`PO_VAL_002`), `currency_id` และ `exchange_rate > 0` (`PO_VAL_003`), `credit_term_id` ถูกตั้งเมื่อ vendor ต้องการ (`PO_VAL_005`) และ `delivery_date >= order_date` (`PO_VAL_006`) `po_no` ที่ unique ถูกสร้างโดย numbering service (`PO_VAL_001`)
3. ไล่ดูแต่ละ **บรรทัด PO** ในแท็บ **Items** บรรทัดที่มาจาก PR ใช้ bridge row เป็นตัวขับ snapshot; บรรทัด manual Purchaser เลือกสินค้าจาก catalog เอง สำหรับทุกบรรทัด ตรวจว่า `product_id` active (`PO_VAL_007`), `order_qty > 0` พร้อม `order_unit_id` (`PO_VAL_008`) และ conversion factor ไป base UoM เป็นบวก (`PO_VAL_009`) บรรทัด FOC (`is_foc = true`) อนุญาตให้ `price = 0` (`PO_VAL_010`)
4. ตรวจ **ราคากับ vendor pricelist ปัจจุบัน** ระบบเทียบ `price` ของแต่ละบรรทัดกับ `tb_pricelist_detail` ที่ active resolve จาก `product_id`, vendor, location และ effective date ตัวบ่งชี้ deviation flag บรรทัดที่อยู่นอกแถบ tolerance ที่ตั้งไว้ (เช่น `±5%`) เมื่อมี deviation Purchaser เลือกได้ว่า (a) คงราคาบรรทัดและเดินหน้าต่อ (บรรทัดที่มาจาก PR อาจคง snapshot ของ PR ไว้), (b) refresh เป็นราคา pricelist ปัจจุบัน หรือ (c) สำหรับบรรทัดที่มาจาก PR ยกประเด็นและ trigger bounce-back ไปยังผู้ร้องขอก่อน submit
5. ตั้งหรือยืนยัน **เงื่อนไขการส่งของและการชำระเงิน** บนส่วนหัว เงื่อนไขการชำระเงิน (`credit_term_id` snapshot เป็น `credit_term`) มาจาก vendor master เป็นค่าเริ่มต้น; Incoterm / clause การส่งของถูกเก็บบนส่วนหัวและสะท้อนใน `delivery_point_id` ของแต่ละแถว bridge `tb_purchase_order_detail` ปรับ `delivery_date` รายบรรทัดได้หากต้องการ schedule แบบเหลื่อม tax profile และ discount rate ถูกตรวจตาม `PO_VAL_010` และ `PO_VAL_011`
6. ดูยอดส่วนหัวคำนวณใหม่ subtotal, discount, net, tax และ total ของบรรทัดคำนวณตาม `PO_CALC_001`–`PO_CALC_005`; การ dual-post ในสกุลเงินฐานใช้ `exchange_rate` ที่ถูก lock ผ่าน `PO_CALC_006`; บรรทัด FOC ส่งผ่านจำนวนแต่ไม่ส่งผ่านเงินตาม `PO_CALC_007`; ส่วนหัว roll up `total_price`, `total_tax`, `total_amount` และ `total_qty` ตาม `PO_CALC_008`–`PO_CALC_011`; การ rounding ใช้ half-up ตาม `PO_CALC_012`
7. เปิดแท็บ **Attachments** และ **Comments** แล้วแนบเอกสารสนับสนุน (vendor quote, memo ภายใน) หรือบันทึกสำหรับห่วงโซ่ผู้อนุมัติ activity log เก็บทุกเหตุการณ์การ save รวมถึงการเปลี่ยนแปลงส่วนหัวและบรรทัดผ่าน `tb_purchase_order_comment` และ `tb_purchase_order_detail_comment`
8. รันการตรวจ submit-time PO ต้องมีบรรทัดที่ยังไม่ถูก soft-delete อย่างน้อย 1 บรรทัด (`PO_VAL_012`), ทุกบรรทัดต้องใช้ `vendor_id` และ `currency_id` เดียวกับส่วนหัว (single-vendor / single-currency invariant, `PO_VAL_013`) และบรรทัดที่มาจาก PR ต้องมีแถว bridge (`PO_VAL_014`)
9. กด **Submit for approval** (`PO_AUTH_003`, `PO_POST_002`) `po_status` เปลี่ยน `draft → in_progress`, `last_action = submitted`, `workflow_current_stage` เลื่อนไปยัง stage อนุมัติแรก และ `user_action.execute` ถูก populate จาก workflow definition ส่งต่อไปยังผู้อนุมัติ stage แรก (โดยทั่วไปคือ Procurement Manager สำหรับ PO มูลค่าสูงที่เกิน threshold ที่ตั้งใน tenant; ต่ำกว่า threshold Purchaser อาจมีสิทธิ์ self-approve ตาม `PO_AUTH_004`)
10. เมื่ออนุมัติขั้นสุดท้าย (`in_progress → sent`, `PO_POST_004`), workflow engine trigger action ส่ง PO Purchaser (หรือระบบ ถ้าตั้ง auto-transmit ภายใต้ `PO_AUTH_006`) ส่ง PO ให้ผู้ขายผ่าน channel ที่ตั้งไว้ — email, EDI หรือ vendor portal — และระบบตั้ง `tb_purchase_order.email` และ `approval_date` `po_status` กลายเป็น `sent` และ PO กลายเป็น commitment ที่ผูกพันกับผู้ขาย; soft budget commitment กลายเป็น vendor liability
11. ติดตาม PO บน dashboard **Open POs** Purchaser ติดตามการตอบรับของผู้ขาย ตามเรื่องความล่าช้า และดูการ post GRN (ขับโดยโมดูล [[good-receive-note]]) ที่ flip `po_status` จาก `sent` เป็น `partial` และในที่สุดเป็น `completed` ผ่าน `PO_POST_006` และ `PO_POST_007` คอลัมน์ `received_qty` รายบรรทัดและ `received_qty` ใน bridge ถูก update ทุกครั้งที่ post GRN
12. จัดการคำขอแก้ไข post-`sent` ตาม `PO_VAL_016` หลังจาก `sent` แล้วแก้ได้เฉพาะ `cancelled_qty` และ note รายบรรทัด — การเปลี่ยน vendor / currency / บรรทัดที่เป็นเนื้อหาต้อง void ยอดเปิดและออก PO ใหม่ Purchaser เขียน comment ทุกครั้งที่แก้ไข เพื่อให้ activity log เก็บประวัติการเปลี่ยนแปลง

## 3. สาขาการตัดสินใจ

- **ถ้า pricelist deviation ของบรรทัดที่มาจาก PR เกิน tolerance** (ราคาปัจจุบันเทียบ `pricelist_price` ที่ PR snapshot ไว้อยู่นอกแถบ `±X%`): Purchaser เห็น flag deviation ในขั้นตอน 4 และเลือกหนึ่งในสามเส้นทาง (a) **Accept snapshot** — บรรทัด PO รับราคาที่ PR freeze ไว้; PO submit ด้วยราคานั้น (b) **Refresh to current** — บรรทัด PO ถูก update เป็นราคาปัจจุบันจาก `tb_pricelist_detail` ก่อน submit; snapshot ของ PR ไม่เปลี่ยน แต่ PO บันทึกราคาใหม่ (c) **Bounce-back ไปยังผู้ร้องขอ** — Purchaser ยกเลิกการแปลงบรรทัดที่ได้รับผลกระทบและ trigger เส้นทาง send-back มาตรฐานของ PR; `pr_status` กลับเป็น `draft`, soft commitment ถูกปลด และบรรทัดเข้าห่วงโซ่ผู้อนุมัติของ PR ใหม่หลังผู้ร้องขอแก้ไข เหตุผลของการ bounce-back ถูกบันทึกใน `tb_purchase_request_comment` เพื่อการตรวจสอบ ดู [[purchase-request]] หัวข้อ "สาขาการตัดสินใจ" สำหรับเส้นทางฝั่งรับ
- **ถ้าบรรทัดของ PO แบบ manual ไม่มี vendor allocation หรือไม่ match กับ pricelist**: บรรทัดจะผ่าน `PO_VAL_002` / `PO_VAL_007` ไม่ได้จนกว่าจะตั้งค่า vendor + product + price ที่ถูกต้อง Purchaser เปิด vendor picker และ product picker เลือก combination ที่ถูกต้อง แล้วระบบเขียน snapshot ของ vendor และ pricelist context ลงบนบรรทัด PO แบบ manual ไม่ต้องมีแถว bridge (`PO_VAL_014` ใช้เฉพาะเมื่อ `po_type = purchase_request`)
- **ถ้าจะส่งเพียงบางบรรทัดของ PR ที่เลือกไว้ในรอบนี้ (partial PR conversion)**: ใน workbench Convert-to-PO (ขั้นตอน 1) Purchaser ติ๊กเฉพาะบรรทัด (และจำนวนที่จะแปลง) ที่จะส่งในรอบนี้; บรรทัดหรือจำนวนที่ไม่ติ๊กยังคงเป็นยอดเปิดบน PR ต้นทาง bridge table บันทึกเฉพาะ link PR-line → PO-line ที่ถูกสร้างพร้อมจำนวน PR ต้นทางยังคงอยู่ที่ `pr_status = approved` จนกว่าทุกบรรทัดจะ bridge ครบหรือถูก cancel แล้ว flip เป็น `completed` (ตาม `PR_POST_007`) Purchaser (หรือเพื่อนร่วมทีม) สามารถรันการแปลงรอบที่สองได้ภายหลังจากคิว Approved PRs เดิม
- **ถ้าต้องแก้ไขหลังจาก `po_status = sent`** (แก้ราคา, ลดจำนวน, เลื่อนวันส่งที่ตกลงกับ vendor): ตาม `PO_VAL_016` หลังจาก `sent` แล้วแก้ได้เฉพาะ `cancelled_qty` และ note รายบรรทัด สำหรับการลดจำนวน Purchaser เขียนยอดคงเหลือที่ตกลงเป็น `cancelled_qty` ให้ `received_qty + cancelled_qty = order_qty` ของบรรทัดที่ได้รับผลกระทบ แล้วบันทึกการแก้ไขใน `tb_purchase_order_comment` สำหรับการเปลี่ยนเนื้อหา (ราคาใหม่, สินค้าใหม่, vendor ใหม่) ต้อง void ยอดเปิดผ่าน Procurement Manager (`PO_AUTH_007`) แล้วออก PO ใหม่ การแก้ไขต้องสื่อสารกับ vendor ผ่าน channel เดียวกับที่ใช้ส่ง PO ครั้งแรก (email / EDI / portal)
- **ถ้า Purchaser ต้อง void ใน draft** (PO ถูกสร้างผิด, ความต้องการเปลี่ยนก่อน submit): จากหน้ารายละเอียด PO ที่ `po_status = draft` Purchaser สามารถ soft-delete ได้ (`PO_AUTH_005` สงวนการ hard-delete-in-draft ให้ Procurement Manager; เส้นทางของ Purchaser คือทิ้งเป็น draft โดยไม่ submit หรือเสนอ Procurement Manager เพื่อ soft-delete) การ void จาก `in_progress`, `sent` หรือ `partial` **ไม่อยู่ในสิทธิ์ของ Purchaser** — ต้องเป็น Procurement Manager (`PO_AUTH_007`, `PO_POST_010`)
- **ถ้าต้องเคลียร์กับ vendor ระหว่างเตรียม draft** (spec กำกวม, MOQ ขัดแย้ง, lead-time ไม่ทันวันที่ขอ): สำหรับ PO ที่มาจาก PR Purchaser **ไม่** แก้เนื้อหา PR เอง — แต่ trigger เส้นทาง send-back ฝั่ง PR ซึ่งส่ง PR กลับให้ผู้ร้องขอที่ `draft` พร้อมเหตุผลของการเคลียร์ ที่บันทึกใน `tb_purchase_request_comment` ผู้ร้องขอแก้ไขแล้ว submit ใหม่ผ่านห่วงโซ่ผู้อนุมัติ PR ทั้งหมด; PR เข้าคิว Approved PRs อีกครั้ง และ Purchaser มารับใหม่เพื่อ Convert-to-PO รอบใหม่ สำหรับ PO แบบ manual Purchaser เคลียร์กับ vendor โดยตรงและแก้ draft PO ก่อน submit — ไม่มี PR ต้นน้ำให้ bounce กลับ

## 4. จุดสิ้นสุด / การส่งต่อ

บทบาทของ Purchaser ต่อ PO แต่ละฉบับสิ้นสุดที่หนึ่งในสี่จุดต่อไปนี้; สถานะเอกสารแต่ละจุดส่งต่อยึดตามค่า `enum_purchase_order_doc_status` ณ จุดส่งต่อ

- **Submit มาตรฐาน → การอนุมัติ** — Purchaser กด submit และ `po_status` เปลี่ยน `draft → in_progress` (`PO_AUTH_003`, `PO_POST_002`) ส่งต่อไปยังผู้อนุมัติ stage แรก (โดยทั่วไปคือ **Procurement Manager** สำหรับ PO มูลค่าสูงที่เกิน threshold ของ tenant; ต่ำกว่า threshold Purchaser อาจ self-approve ได้ถ้า workflow อนุญาต) บทบาทของ Purchaser กลับมาเล่นต่อถ้าผู้อนุมัติ send-back PO (ขั้นตอน 9 ด้านบนจะ route PO กลับเป็น `draft` ตาม `PO_POST_005`)
- **อนุมัติขั้นสุดท้าย → ส่งให้ผู้ขาย** — เมื่ออนุมัติ stage สุดท้าย (`in_progress → sent`, `PO_POST_004`), Purchaser (หรือตัวจัดการ auto-transmit) ส่ง PO ให้ **ผู้ขาย (Vendor)** ผ่าน email / EDI / portal ภายใต้ `PO_AUTH_006` ส่งต่อไปยัง Vendor; สถานะเอกสาร ณ จุดส่งต่อคือ `sent` PO กลายเป็น commitment ผูกพัน และ soft budget commitment กลายเป็น hard commitment ดูไฟล์ persona Vendor สำหรับเส้นทางฝั่งภายนอก
- **วงรอบการแก้ไข (Amendment loop)** — เมื่อมีคำขอแก้ไข PO ที่อยู่ใน `sent` (จาก Receiver, Finance หรือ Vendor), Purchaser กลับเข้ามาในเส้นทางที่ขั้นตอนการแก้ไขภายใต้ข้อจำกัด `PO_VAL_016` (หลัง `sent` แก้ได้เฉพาะ `cancelled_qty` และ note รายบรรทัด) การเปลี่ยนเนื้อหาส่งต่อให้ **Procurement Manager** เพื่อ void + ออกใหม่; การเปลี่ยนเล็กน้อย (จำนวนที่ vendor ส่งไม่ครบ, note) Purchaser จัดการเองในเส้นทางเดิมและบันทึกการแก้ไขใน `tb_purchase_order_comment` PO ไม่เปลี่ยนสถานะ — ยังคงอยู่ที่ `sent` หรือ `partial` จนกว่าจะมีการ post GRN ครั้งต่อไป
- **Void ใน draft / Bounce-back** — เมื่อ PO ไปต่อไม่ได้ Purchaser ออกจากเส้นทางด้วยการ (a) เสนอ **Procurement Manager** เพื่อ soft-delete ใน `draft` (`PO_AUTH_005`) หรือ void จาก state ที่สูงกว่า (`PO_AUTH_007`, `PO_POST_010`) — terminal state `voided` — หรือ (b) สำหรับ PO ที่มาจาก PR trigger เส้นทาง send-back ฝั่ง PR ซึ่งส่ง PR ต้นทางกลับให้ **ผู้ร้องขอ (Requestor)** ที่ `pr_status = draft` (`PR_POST_003` ฝั่ง PR) และทิ้ง draft PO ผู้ร้องขอแก้ไขแล้ว submit ใหม่; PR เข้าห่วงโซ่ผู้อนุมัติอีกครั้งและในที่สุดกลับมาที่คิวของ Purchaser เพื่อ Convert-to-PO รอบใหม่ ดู [[purchase-request]] § เส้นทาง Requestor สำหรับฝั่งรับของ bounce-back

Transition ที่ขับโดยการรับสินค้า (`sent → partial → completed` ผ่าน `PO_POST_006`/`PO_POST_007` และ `partial → closed` ผ่าน `PO_POST_011`) ไม่ใช่ action ของ Purchaser — ขับโดย **Receiver** ผ่านการ post GRN และสำหรับการปิดก่อนกำหนด โดย **Procurement Manager** หรือ **Inventory Manager** (`PO_AUTH_008`) Purchaser ติดตาม transition เหล่านี้บน dashboard Open POs แต่ไม่ trigger โดยตรง
