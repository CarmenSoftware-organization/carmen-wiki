# ใบสั่งซื้อ (Purchase Order) — Test Scenarios — Vendor

> **At a Glance**
> **Persona:** Vendor (ฝ่ายภายนอก — ไม่มี Carmen login) &nbsp;·&nbsp; **Module:** [[purchase-order]] &nbsp;·&nbsp; **Scenarios:** ~15
> **Categories:** Happy Path &nbsp;·&nbsp; Permission (N/A) &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** ไม่มี vendor spec เฉพาะ (ไม่มี Carmen login); vendor-driven events exercised ทางอ้อมผ่าน `tests/401-po.spec.ts` (TC-PO-030001..030004 send-to-vendor, TC-PO-340001..340003 GRN-sync) ใน `../carmen-inventory-frontend-e2e/`

หน้านี้ capture test scenarios ที่ exercise interaction ของ **Vendor** persona กับโมดูล `purchase-order` ต่างจากไฟล์ persona อื่น ๆ Vendor เป็น **ฝ่ายภายนอกที่ไม่มี Carmen system login** ([03-user-flow-vendor.md](./03-user-flow-vendor.md) Section 1) — ทุก effect ฝั่งระบบของ action ของ vendor บันทึกโดย internal persona (**Purchaser** สำหรับ acknowledgement และ amendment, **Receiver** สำหรับ GRN postings เทียบกับ `PO_POST_006` / `PO_POST_007`, **Finance** persona สำหรับ invoice capture และ three-way match) Scenarios ด้านล่างจึงอธิบาย **vendor-driven events** (transmitted PO received, acknowledgement returned, shipment dispatched, invoice issued) ร่วมกับ system effects ที่ internal personas บันทึกแทน vendor เพราะ vendor ไม่มี role ใน-ระบบ ไฟล์จงใจสั้นกว่าไฟล์ scenario persona อื่น ๆ — Permission / Authorization section ลดเหลือเพียง row N/A เดียว และ cross-persona handoffs (`X-PO-01`, `X-PO-05`, three-way-match) อยู่ใน parent overview ไม่ใช่ที่นี่

## 2. Permission / Authorization

| # | Scenario | Expected behaviour |
| - | -------- | ------------------ |
| VND-PERM-01 | Vendor เป็นฝ่ายภายนอกไม่มี system login; in-system permissions เป็น N/A ทุก vendor responses บันทึกโดย internal personas (Purchaser / Receiver / Finance) | N/A — ไม่มี Carmen login สำหรับ vendor และไม่มี UI surface ให้ test ทุก effect ของ vendor action เข้าระบบผ่าน Purchaser (acknowledgement / amendment comments บน `tb_purchase_order_comment`), Receiver (GRN postings ขับเคลื่อน `PO_POST_006` / `PO_POST_007`) หรือ Finance persona (invoice capture และ three-way match ตาม `PO_POST_008` / `PO_XMOD_007`) ที่ vendor portal ตั้งค่า portal ทำหน้าที่เป็น authenticated channel ของ vendor เข้า Carmen แต่เป็น concern แยกจาก RBAC matrix ของโมดูล PO (`PO_AUTH_001`–`PO_AUTH_011`) |

## 3. Validation / Error

| # | Scenario | Trigger | Expected error |
| - | -------- | ------- | -------------- |
| VND-VAL-01 | Invoice รับสำหรับ PO ที่ไม่มีอยู่ | Finance พยายาม capture vendor invoice ที่อ้างอิง `po_no` ไม่อยู่ใน `tb_purchase_order` (พิมพ์ผิดหรือกุขึ้น) | Reject ที่ invoice capture — server return `"Referenced PO not found. Please verify the PO number with the vendor."` ไม่มี invoice row เขียน; ไม่มี three-way match พยายาม; audit entry บันทึกสำหรับ capture ที่ failed Finance ติดต่อ vendor ผ่าน Purchaser เพื่อ clarify `po_no` ที่ถูก |
| VND-VAL-02 | Invoice รับก่อน PO ถูกส่ง (order-of-operations violation) | Vendor ออก invoice ขณะที่ PO ยังอยู่ที่ `po_status ∈ {draft, in_progress}` (การส่ง **Send to Vendor** ยังไม่ fire) | Reject ที่ invoice capture ตาม cross-module precondition ว่า PO ต้องอยู่ที่ `po_status ∈ {sent, partial, completed}` สำหรับ vendor-side document ใด ๆ ที่จะแนบ Server return `"Invoice cannot be matched — PO has not been transmitted to the vendor."` นี่ catches premature invoicing ที่ vendor pre-empt PO หรือใช้ `po_no` ผิดสำหรับเอกสารอื่น |
| VND-VAL-03 | ปริมาณ invoice เกิน GRN quantity ที่บันทึก (flag สำหรับ three-way match) | Vendor invoice 12 units เทียบกับ `L1` ที่ `received_qty = 10` (vendor over-billed); Finance capture invoice และรัน three-way match ตาม `PO_POST_008` และ `PO_XMOD_007` | Three-way match fail — `Σ invoice.qty (12) > Σ GRN.received_qty (10)` บน `L1` Match result บันทึกเป็น `failed` ด้วย reason `"Invoice qty exceeds matched GRN qty on line L1 (12 > 10)"`; AP-accrual liability **ไม่** cleared; invoice park ในคิว match-exception; Finance escalate ไปยัง Purchaser เพื่อ confirm ว่า excess 2 units เป็น (a) vendor error ที่ต้อง credit-noted, (b) missed GRN ที่ต้อง posted, หรือ (c) over-receipt ที่ตกลงภายใน tenant tolerance ตาม `PO_XMOD_004` PO status ไม่เปลี่ยนโดย match failure |
| VND-VAL-04 | Invoice รับสำหรับ `completed` PO ที่ fully invoiced แล้ว | Vendor re-send invoice (duplicate หรือ error) เทียบกับ PO ที่ `po_status = completed` ที่ matched invoice posted แล้ว | Reject ที่ invoice capture — server return `"PO is already fully invoiced and matched. Reject the duplicate or raise as a credit / debit memo."` duplicate-invoice guard check AP module's prior-match registry สำหรับ `po_no` และ line-level invoiced totals Activity log บันทึก duplicate-capture attempt; Finance notify Purchaser ที่ confirm rejection กับ vendor ผ่าน transmission channel ดั้งเดิม |
| VND-VAL-05 | Invoice สกุลเงินผิดเทียบกับ PO base currency (FX adjustment) | PO ที่ `currency_id = USD`, `exchange_rate = 35.50` เทียบกับ base currency `THB`; vendor ออก invoice ใน `EUR` (ต่างจากทั้ง transaction และ base currency บน PO) | Reject by default — server return `"Invoice currency (EUR) does not match the PO transaction currency (USD)."` Three-way match operate ในสกุลเงิน transaction ของ PO; cross-currency invoices ต้องการ FX adjustment capture ที่ invoice time ที่ govern โดย `PO_CALC_005` (FX rounding เป็น base currency) และ `PO_CALC_009` (header rolls maintained ในทั้งสกุลเงิน transaction และ base) ที่ tenant policy อนุญาต single-step FX conversion, invoice capture พร้อมค่า `exchange_rate_invoice` ที่ชัดเจนและ match re-run เทียบกับ converted amount; มิฉะนั้น Finance return invoice ให้ vendor สำหรับ re-issue ในสกุลเงิน PO |

## 4. Edge Cases

| # | Scenario | เงื่อนไข | Expected |
| - | -------- | --------- | -------- |
| VND-EDGE-01 | Vendor ปฏิเสธ PO หลัง acknowledgement (bounce-back ผ่าน Purchaser) | PO ที่ `po_status = sent`; vendor ตอบรับครั้งแรก (VND-HP-02) แล้ว stock-out / price-disagreement force ปฏิเสธ | Purchaser log การปฏิเสธใน `tb_purchase_order_comment` (type `system`) และ re-route สำหรับ amendment ภายในข้อจำกัด post-`sent` ของ `PO_VAL_016` (เฉพาะ `cancelled_qty` และ note ต่อบรรทัดที่ mutated) หรือ escalate ไปยัง **Procurement Manager** สำหรับ void ภายใต้ `PO_AUTH_007` / `PO_POST_010` หาก voided, `po_status` transition `sent → voided` และ soft commitment reverse; vendor-side notification ส่งกลับผ่าน channel ดั้งเดิมที่ supported acknowledgement comment ก่อนของ vendor เก็บไว้สำหรับ audit |
| VND-EDGE-02 | Vendor partial-ships ของผิด (Receiver บันทึก discrepancy) | PO line `L1 : order_qty = 10` (product `P1`); vendor ส่ง 8 units ของ `P1` และ 2 units ของ product ที่ไม่เกี่ยวข้อง `P_wrong` (substitution / packing error) | Receiver เปิด GRN เทียบกับ `L1` และบันทึก 8 รับเทียบกับ `L1` (flip `po_status → partial` ตาม `PO_POST_006`) บวก variance comment ใน `tb_purchase_order_comment` อ้างอิง `P_wrong` และ 2 units ที่ไม่เกี่ยวข้อง การส่งของผิด **ไม่** post ไปยัง inventory ภายใต้ `L1`; Purchaser notify เพื่อ initiate return / replacement / credit note loop กับ vendor ผ่าน amendment surface write-off ที่ตกลงของ 2 units ที่เหลือของ `P1` เขียนไปยัง `cancelled_qty` บน `L1` (ปิดบรรทัดที่ `partial → closed` ภายใต้ `PO_POST_011` เมื่อไม่มี fulfilment เพิ่ม) และ items ผิดส่งกลับให้ vendor บนเอกสารแยก |
| VND-EDGE-03 | Vendor ออกจากธุรกิจ mid-PO (Procurement Manager void ตาม `PO_AUTH_007`) | PO ที่ `po_status = partial` ด้วย `received_qty = 6` บน `L1 : order_qty = 10`; vendor หยุดค้าก่อนส่ง balance 4 units | Procurement Manager เปิด PO detail และ execute **Void** ภายใต้ `PO_AUTH_007` (action ที่ Purchaser ถูก deny ตาม `PUR-PERM-07`) `po_status` transition `partial → voided` ผ่าน `PO_POST_010`; `is_active = false`; 6 units ที่รับแล้ว preserve บน `tb_purchase_order_detail.received_qty` และ AP-accrual liability ของพวกเขายังคงอยู่ภายใต้ความรับผิดชอบของ AP ตาม `PO_XMOD_007`; 4 units ที่ unfulfilled reverse จาก soft commitment; audit comment capture reason `"Vendor out of business"` ที่ POs มากได้รับผลกระทบ Manager อาจใช้เส้นทาง bulk-void (covered เป็น `PM-HP-09` ใน [04-test-scenarios-procurement-manager.md](./04-test-scenarios-procurement-manager.md)) |
| VND-EDGE-04 | Vendor confirm quantity ผิดใน acknowledgement เริ่มต้น (correction ผ่าน Purchaser amendment) | Vendor acknowledge ใน VND-HP-02 แต่ระบุว่าสามารถ supply ได้เฉพาะ 8 units เทียบกับ `L1 : order_qty = 10`; PO ที่ `po_status = sent`, `received_qty = 0` บน `L1` | ตาม `PO_VAL_016` เฉพาะ `cancelled_qty` และ note ต่อบรรทัดที่ mutable post-`sent` Purchaser execute amendment loop (covered เป็น `PUR-HP-05` ใน [04-test-scenarios-purchaser.md](./04-test-scenarios-purchaser.md)): ตั้ง `cancelled_qty = 2` บน `L1` ดังนั้น `received_qty + cancelled_qty = 0 + 2 = 2` และ effective open qty กลายเป็น `order_qty − received_qty − cancelled_qty = 10 − 0 − 2 = 8`; บันทึก comment ใน `tb_purchase_order_comment` ด้วย reference ของ vendor; `po_status` ยังคงเป็น `sent` GRN ปลายน้ำเทียบกับ `L1` อาจรับได้ถึง 8 Purchaser ไม่ต้อง void PO — amendment surface พอสำหรับ quantity-down correction ที่ acknowledged โดย vendor |
