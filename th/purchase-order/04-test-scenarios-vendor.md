# ใบสั่งซื้อ — กรณีทดสอบ — ผู้ขาย (Vendor)

หน้านี้รวบรวมกรณีทดสอบที่ใช้ตรวจสอบการมีปฏิสัมพันธ์ของ persona **ผู้ขาย (Vendor)** กับโมดูล `purchase-order` ต่างจากไฟล์ persona อื่น ๆ ผู้ขายเป็น**บุคคลภายนอกที่ไม่มี login ในระบบ Carmen** ([03-user-flow-vendor.md](./03-user-flow-vendor.md) หัวข้อ 1) — ผลของการกระทำของผู้ขายทุกอย่างที่ปรากฏในระบบนั้นถูกบันทึกโดย persona ภายในแทน (**Purchaser** สำหรับการตอบรับและการ amend, **Receiver** สำหรับการ post GRN ตาม `PO_POST_006` / `PO_POST_007`, persona **Finance** สำหรับการรับใบแจ้งหนี้และการทำ three-way match) กรณีทดสอบด้านล่างจึงเป็นการอธิบาย**เหตุการณ์ที่ขับเคลื่อนโดยผู้ขาย** (รับ PO ที่ส่ง, ตอบรับ, จัดส่ง, ออกใบแจ้งหนี้) พร้อมกับผลในระบบที่ persona ภายในบันทึกแทนผู้ขาย เพราะผู้ขายไม่มี role ในระบบ ไฟล์นี้จึงสั้นกว่าไฟล์กรณีทดสอบของ persona อื่น ๆ โดยตั้งใจ — หัวข้อ Permission / Authorization ลดเหลือเพียงแถวเดียวที่เป็น N/A และ handoff ระหว่าง persona (`X-PO-01`, `X-PO-05`, three-way match) อยู่ในหน้าภาพรวมพ่อ ไม่ได้อยู่ในหน้านี้

## 1. Happy Path

| # | กรณี | เงื่อนไขก่อน | ขั้นตอน | ผลที่คาดหวัง |
| - | ---- | ------------ | ------- | ------------ |
| VND-HP-01 | รับ PO ที่ถูกส่ง (email PDF หรือ vendor-portal callback) | PO ที่ `po_status = sent` หลังจาก Purchaser กด **Send to Vendor**; `tb_purchase_order.email` ถูกตั้งและ `approval_date = now()`; ตั้งค่า email หรือ portal callback endpoint ของผู้ขายบน `tb_vendor` | 1. Carmen ส่ง PO ผ่าน channel ที่ตั้งค่าไว้ (email PDF, EDI feed หรือ vendor-portal link) 2. กล่องเมล / portal ของผู้ขายได้รับเอกสารที่อ้างถึง `po_no`, บรรทัด, qty, price, delivery date และ payment terms | PO ถูกส่งถึงผู้ขาย; `po_status` ยังเป็น `sent` (การส่งเอกสารไม่เปลี่ยน status — `PO_POST_004` ทำงานแล้วตอน final approval); ฝั่ง Carmen `tb_purchase_order.email` ถูกบันทึก และ activity log บันทึกเหตุการณ์การส่ง จากจุดนี้ soft budget commitment กลายเป็น liability ที่ผูกกับผู้ขาย |
| VND-HP-02 | ตอบรับ PO (บันทึกโดย Purchaser เป็น comment) | PO ที่ `po_status = sent`; ผู้ขายยืนยันการยอมรับ price, quantity, delivery date และ payment terms ผ่าน channel การส่งเดิม | 1. ผู้ขายตอบกลับ (email / portal / โทรศัพท์) ยืนยันการยอมรับ 2. Purchaser เปิดหน้ารายละเอียด PO และเพิ่ม comment ลงใน `tb_purchase_order_comment` เพื่อบันทึกการตอบรับ, วันที่ยืนยัน, และ reference ของผู้ขาย (กรณีที่ตั้งค่า vendor portal ไว้ portal callback จะเขียน comment เดียวกันให้อัตโนมัติ) | แถว comment ถูกผนวกเข้า `tb_purchase_order_comment` ด้วยประเภท `system` (หรือ `user` ถ้าป้อนด้วยมือโดย Purchaser), วันที่ยืนยัน และ reference ของผู้ขาย; `po_status` ยังเป็น `sent`; ไม่มีการเขียน `cancelled_qty` หรือการปรับ quantity; activity log แสดงเหตุการณ์การตอบรับ ต่อจากนี้ PO ถือว่าผู้ขาย "ยืนยัน" แล้วสำหรับการ monitor การส่งของ |
| VND-HP-03 | จัดส่งเต็มจำนวนตาม delivery date ที่ตกลง (Receiver บันทึก GRN ตรงจำนวน) | PO ที่ `po_status = sent` มีบรรทัด `L1 : order_qty = 10`, `received_qty = 0`, `cancelled_qty = 0`; ผู้ขายจัดส่ง 10 หน่วยตาม `delivery_date` ที่ตกลง | 1. ผู้ให้บริการขนส่งของผู้ขายส่ง 10 หน่วยครบไปยังสถานที่รับของพร้อม delivery note ที่อ้างถึง `po_no` 2. Receiver เปิด PO ในโมดูล [[good-receive-note]] และ post GRN ที่ qty 10 ตรงกับ `L1` | การ post GRN เปลี่ยน `po_status` จาก `sent → completed` ผ่าน `PO_POST_007` เพราะ `received_qty + cancelled_qty = order_qty` ในทุกบรรทัด (`L1`: `10 + 0 = 10`); `tb_purchase_order_detail.received_qty = 10` บน `L1`; inventory on-hand เพิ่มขึ้นในโมดูล [[inventory]] ตาม `PO_XMOD_008`; โมดูล AP ตั้ง inventory-accrual liability ตาม `PO_XMOD_007` กับ GRN; PO ไปถึง terminal receipt state `completed` |
| VND-HP-04 | จัดส่งบางส่วนแล้วส่งครั้งที่สอง (Receiver บันทึก partial แล้วเต็ม) | PO ที่ `po_status = sent` มี `L1 : order_qty = 10`; ผู้ขายส่งวันนี้ได้ 6 และอีก 4 สัปดาห์หน้า | 1. ผู้ขายส่ง 6 หน่วยพร้อม delivery note ที่ระบุว่า partial 2. Receiver post GRN #1 qty 6 กับ `L1` 3. ผู้ขายส่ง 4 หน่วยที่เหลือสัปดาห์ต่อมา 4. Receiver post GRN #2 qty 4 กับ `L1` | หลัง GRN #1: `received_qty = 6`, `po_status` เปลี่ยน `sent → partial` ตาม `PO_POST_006` เพราะ `received_qty < order_qty − cancelled_qty` บน `L1`; หลัง GRN #2: `received_qty = 10`, `po_status` เปลี่ยน `partial → completed` ตาม `PO_POST_007` เพราะทุกบรรทัดเข้าเงื่อนไข `received_qty + cancelled_qty = order_qty`; ทั้งสองเหตุการณ์ GRN ปรากฏบน activity log; AP-side accrual ตั้งต่อ GRN ตาม `PO_XMOD_007`; inventory เพิ่มขึ้น 2 ครั้ง |
| VND-HP-05 | ออกใบแจ้งหนี้ตรงกับ PO และ GRN (three-way match ของ Finance ผ่าน) | PO ที่ `po_status = completed` (หรือ `partial` สำหรับใบแจ้งหนี้บางส่วน); มี GRN บันทึกตาม VND-HP-03 / VND-HP-04; ใบแจ้งหนี้ AP ของผู้ขายอ้างถึง `po_no` และ quantity ที่ส่ง | 1. ผู้ขายส่งใบแจ้งหนี้ AP (กระดาษ / PDF / EDI) 2. persona Finance รับใบแจ้งหนี้ที่อ้างถึง PO และ GRN ที่ผูกไว้ 3. Finance รัน three-way match (PO ↔ GRN ↔ invoice) ตาม canonical match rule (`PO_POST_008` + `PO_XMOD_007`) 4. เมื่อ match สำเร็จ Finance post AP liability | three-way match สำเร็จ — `Σ invoice.qty` เท่ากับ `Σ GRN.received_qty` ในทุกบรรทัด และ `Σ invoice.amount` เท่ากับ `Σ matched (qty × price)` ภายใน tolerance ของ tenant; AP-side inventory-accrual liability ที่ตั้งไว้ตอน post GRN ถูก clear และแทนที่ด้วย vendor invoice liability ตาม `PO_XMOD_007`; `po_status` ไม่เปลี่ยนเพราะ match (status PO ถูกตั้งโดยการ post GRN ไม่ใช่โดย match); ใบแจ้งหนี้ถูก post และผู้ขายเข้าสู่รอบการจ่ายเงิน |

## 2. Permission / Authorization

| # | กรณี | พฤติกรรมที่คาดหวัง |
| - | ---- | ------------------ |
| VND-PERM-01 | ผู้ขายเป็นบุคคลภายนอก ไม่มี login ในระบบ สิทธิ์ในระบบเป็น N/A การตอบของผู้ขายทุกอย่างถูกบันทึกโดย persona ภายใน (Purchaser / Receiver / Finance) | N/A — ไม่มี Carmen login ของผู้ขาย และไม่มี UI surface ให้ทดสอบ ผลของการกระทำของผู้ขายทุกอย่างเข้าสู่ระบบผ่าน Purchaser (comment การตอบรับ / amendment บน `tb_purchase_order_comment`), Receiver (การ post GRN ที่ขับ `PO_POST_006` / `PO_POST_007`) หรือ persona Finance (การรับใบแจ้งหนี้และ three-way match ตาม `PO_POST_008` / `PO_XMOD_007`) กรณีที่ตั้งค่า vendor portal ไว้ portal ทำหน้าที่เป็น channel ที่ผู้ขาย authenticate เข้ามาใน Carmen แต่เป็นเรื่องแยกจากเมทริกซ์ RBAC ของโมดูล PO (`PO_AUTH_001`–`PO_AUTH_011`) |

## 3. Validation / Error

| # | กรณี | Trigger | Error ที่คาดหวัง |
| - | ---- | ------- | ---------------- |
| VND-VAL-01 | ใบแจ้งหนี้สำหรับ PO ที่ไม่มีอยู่ | Finance พยายามรับใบแจ้งหนี้ที่อ้างถึง `po_no` ที่ไม่มีใน `tb_purchase_order` (พิมพ์ผิดหรือสร้างเอง) | Reject ที่ขั้นรับใบแจ้งหนี้ — server ตอบกลับ `"Referenced PO not found. Please verify the PO number with the vendor."` ไม่มีการเขียนแถว invoice; ไม่ทำ three-way match; บันทึก audit entry สำหรับการ capture ที่ล้มเหลว Finance ติดต่อผู้ขายผ่าน Purchaser เพื่อยืนยัน `po_no` ที่ถูกต้อง |
| VND-VAL-02 | ใบแจ้งหนี้มาก่อนที่ PO จะส่งออก (ผิดลำดับการดำเนินการ) | ผู้ขายออกใบแจ้งหนี้ขณะ PO ยัง `po_status ∈ {draft, in_progress}` (action **Send to Vendor** ยังไม่ทำงาน) | Reject ที่ขั้นรับใบแจ้งหนี้ตามเงื่อนไข cross-module ที่ PO ต้องอยู่ที่ `po_status ∈ {sent, partial, completed}` จึงจะแนบเอกสารฝั่งผู้ขายได้ server ตอบกลับ `"Invoice cannot be matched — PO has not been transmitted to the vendor."` กรณีนี้จับการออกใบแจ้งหนี้ก่อนเวลาที่ผู้ขายชิงออกก่อน PO หรือใช้ `po_no` ผิดสำหรับเอกสารอื่น |
| VND-VAL-03 | จำนวนในใบแจ้งหนี้เกินจำนวนที่บันทึก GRN (ถูกตั้ง flag ให้ three-way match) | ผู้ขายแจ้งหนี้ 12 หน่วยกับ `L1` ที่มี `received_qty = 10` (ผู้ขายแจ้งเกิน); Finance รับใบแจ้งหนี้และรัน three-way match ตาม `PO_POST_008` และ `PO_XMOD_007` | three-way match ล้มเหลว — `Σ invoice.qty (12) > Σ GRN.received_qty (10)` บน `L1` ผล match บันทึกเป็น `failed` ด้วยเหตุผล `"Invoice qty exceeds matched GRN qty on line L1 (12 > 10)"`; AP-accrual liability **ไม่** ถูก clear; ใบแจ้งหนี้เข้าคิว match-exception; Finance escalate ไปที่ Purchaser เพื่อยืนยันว่า 2 หน่วยที่เกินคือ (ก) ความผิดพลาดของผู้ขายที่ต้องออก credit note, (ข) GRN ที่ตกหล่นและต้องไป post หรือ (ค) over-receipt ที่ตกลงกันภายใน tenant tolerance ตาม `PO_XMOD_004` status PO ไม่เปลี่ยนเพราะ match ล้มเหลว |
| VND-VAL-04 | ใบแจ้งหนี้สำหรับ PO ที่ `completed` แล้วและออกใบแจ้งหนี้เต็มแล้ว | ผู้ขายส่งใบแจ้งหนี้ซ้ำ (ซ้ำหรือผิดพลาด) กับ PO ที่ `po_status = completed` ซึ่งใบแจ้งหนี้ที่ matched แล้วได้ post ไปแล้ว | Reject ที่ขั้นรับใบแจ้งหนี้ — server ตอบกลับ `"PO is already fully invoiced and matched. Reject the duplicate or raise as a credit / debit memo."` ตัวกัน duplicate-invoice ตรวจสอบ prior-match registry ของโมดูล AP สำหรับ `po_no` และยอด invoiced รายบรรทัด activity log บันทึกความพยายาม capture ซ้ำ; Finance แจ้ง Purchaser ผู้ซึ่งจะยืนยันการ reject กับผู้ขายผ่าน channel การส่งเดิม |
| VND-VAL-05 | ใบแจ้งหนี้ใช้สกุลเงินผิดเทียบกับ base currency ของ PO (ปรับ FX) | PO ที่ `currency_id = USD`, `exchange_rate = 35.50` เทียบกับ base currency `THB`; ผู้ขายออกใบแจ้งหนี้เป็น `EUR` (ต่างจากทั้ง transaction และ base currency ของ PO) | Reject โดยค่าเริ่มต้น — server ตอบกลับ `"Invoice currency (EUR) does not match the PO transaction currency (USD)."` three-way match ทำงานใน transaction currency ของ PO; ใบแจ้งหนี้ข้ามสกุลเงินต้องการ FX adjustment ที่ capture ตอนรับ invoice ซึ่งถูกกำกับโดย `PO_CALC_005` (FX rounding ไปยัง base currency) และ `PO_CALC_009` (ยอดรวมหัวเก็บทั้งใน transaction และ base currency) เมื่อ policy ของ tenant อนุญาตการแปลง FX ขั้นเดียว ใบแจ้งหนี้จะ capture พร้อมค่า `exchange_rate_invoice` ที่ระบุชัดเจน และรัน match ใหม่กับยอดที่แปลงแล้ว; มิฉะนั้น Finance ส่งใบแจ้งหนี้กลับให้ผู้ขายออกใหม่ในสกุลเงินของ PO |

## 4. Edge Cases

| # | กรณี | เงื่อนไข | ผลที่คาดหวัง |
| - | ---- | ------- | ------------ |
| VND-EDGE-01 | ผู้ขายปฏิเสธ PO หลังตอบรับแล้ว (bounce-back ผ่าน Purchaser) | PO ที่ `po_status = sent`; ผู้ขายเคยตอบรับไปแล้ว (VND-HP-02) จากนั้นเกิด stock-out / ราคาไม่ลงตัว ทำให้ต้องปฏิเสธ | Purchaser บันทึกการปฏิเสธลง `tb_purchase_order_comment` (ประเภท `system`) และเลือกระหว่าง re-route ไป amendment ภายใต้ข้อจำกัดหลัง `sent` ของ `PO_VAL_016` (แก้ได้เพียง `cancelled_qty` และ note รายบรรทัด) หรือ escalate ไปที่ **Procurement Manager** เพื่อ void ตาม `PO_AUTH_007` / `PO_POST_010` ถ้า void แล้ว `po_status` เปลี่ยน `sent → voided` และ soft commitment ถูก reverse; ส่ง notification ฝั่งผู้ขายกลับผ่าน channel เดิมที่รองรับ comment การตอบรับก่อนหน้าของผู้ขายถูกเก็บไว้สำหรับ audit |
| VND-EDGE-02 | ผู้ขายส่งของบางส่วนผิดรายการ (Receiver บันทึก discrepancy) | บรรทัด PO `L1 : order_qty = 10` (สินค้า `P1`); ผู้ขายส่ง 8 หน่วยของ `P1` และ 2 หน่วยของสินค้าที่ไม่เกี่ยว `P_wrong` (สลับของ / pack ผิด) | Receiver เปิด GRN กับ `L1` และบันทึกรับ 8 ที่ `L1` (ทำให้ `po_status → partial` ตาม `PO_POST_006`) พร้อม comment variance ใน `tb_purchase_order_comment` อ้างถึง `P_wrong` และ 2 หน่วยที่ไม่เกี่ยว การส่งของผิดรายการ**ไม่** post เข้า inventory ภายใต้ `L1`; Purchaser ถูกแจ้งให้เริ่มลูป return / replacement / credit note กับผู้ขายผ่าน amendment surface การเขียน write-off สำหรับ 2 หน่วยที่เหลือของ `P1` ที่ตกลงกันถูกบันทึกเข้า `cancelled_qty` บน `L1` (ปิดบรรทัดที่ `partial → closed` ตาม `PO_POST_011` เมื่อไม่มีการส่งของเพิ่มเติม) และส่งคืนของผิดให้ผู้ขายในเอกสารแยก |
