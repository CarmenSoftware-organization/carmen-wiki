# ใบสั่งซื้อ — Test Scenarios — Audit / Config

> **At a Glance**
> **Persona:** Audit / Config (Auditor + System Administrator) &nbsp;·&nbsp; **Module:** [[purchase-order]] &nbsp;·&nbsp; **Scenarios:** ~30
> **Categories:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** map ไปยัง `401-po.spec.ts` (transactional paths เท่านั้น) ใน `../carmen-inventory-frontend-e2e/`; audit-log query และ config-management surfaces covered โดย API / integration tests

หน้านี้ capture test scenarios ที่แกน persona **Audit / Config** ขับเคลื่อนโดยตรงในโมดูล `purchase-order` สอง roles ที่แตกต่างนั่งบนแกนนี้: **Auditor** (read-only ข้าม procurement chain เต็ม — Purchase Request → Purchase Order → Goods Receive Note → vendor invoice / AP posting) ที่ใช้ `ActivityLogTab`, `workflow_history`, `tb_purchase_order_comment` (`PO_POST_005`, `PO_POST_009`) ของโมดูล PO และ cross-document bridges (`tb_purchase_order_detail_tb_purchase_request_detail`, GRN back-references ตาม `PO_XMOD_003`, three-way-match records ภายใต้ `PO_POST_008`) เพื่อ verify policy compliance, segregation of duties (`PO_AUTH_010`), three-way-match integrity (`PO_POST_008` / `PO_POST_009`), และ traceability end-to-end; และ **System Administrator** (configuration เท่านั้น) ที่เป็นเจ้าของ PO numbering scheme, workflow definition (`tb_purchase_order.workflow_id`, stages, `user_action.execute[]` membership ตาม `PO_AUTH_011`), RBAC role-to-permission map สำหรับ `PO_AUTH_001`–`PO_AUTH_011`, integration wiring (vendor, vendor-pricelist, budget, inventory), กฎ PR-to-PO conversion และ vendor+currency grouping ตาม carmen/docs § 2.3.2, document templates ที่อ้างอิงที่ transmission ตาม `PO_POST_004`, และ tax / currency / FX rate sources ที่ `PO_CALC_001`–`PO_CALC_011` consume ทั้งสอง roles ไม่ transition PO ข้าม `enum_purchase_order_doc_status`: Auditor exit ผ่าน generated report (ไม่มี PO state change), Sysadmin exit ผ่าน saved configuration ด้วย `effective_from` timestamp (POs ที่ `in_progress` / `sent` / `partial` แล้วเก็บ snapshot ดั้งเดิม; POs ใหม่ใช้ configuration ใหม่) Scenarios จัดกลุ่มเป็น happy paths ที่อธิบายใน [03-user-flow-audit-config.md](./03-user-flow-audit-config.md), RBAC boundary ที่แยก read-only-audit จาก configure-only-sysadmin จาก no-direct-transact, กฎ validation รอบ snapshot preservation, export approval, deadlock prevention, delegation windows และ scan limits และชุดเล็กของ edge cases รอบ in-flight boundaries, rollback, mass delegation, audit-log gaps และ config vs in-flight race conditions Shared `401-po.spec.ts` E2E spec exercise transactional paths เท่านั้น; audit-log query และ configuration-management surfaces covered โดย API / integration tests วันนี้ (ไม่มี audit-config E2E spec เฉพาะที่นี้)

## 2. Permission / Authorization

| # | Scenario | Expected behaviour (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| AUD-PERM-01 | Auditor เปิด PO ใด ๆ ใน `po_status` ใด ๆ (read-only ข้าม chain) | **Allow** ตาม Auditor role's read-only mandate, Auditor สามารถเปิด PR, PO, GRN, invoice, และ AP records ข้ามทุก statuses รวมถึง terminal (`completed`, `closed`, `voided`) หน้า detail PO ใน Auditor's view เป็น read-only: edit / submit / approve / void / close / transmit actions ไม่ expose ใน UI; direct API edit calls return `"You are not authorised to edit this PO at the current stage."` ไม่มี `tb_purchase_order` mutation ทำได้ |
| AUD-PERM-02 | Auditor พยายาม write action (edit a line, void, comment เป็น actor) บน PO | **Deny** Auditor ไม่มี write surface ในโมดูล `purchase-order` — ไม่บน `tb_purchase_order`, ไม่บน `tb_purchase_order_detail`, ไม่บน `tb_purchase_order_comment` (นอกเหนือจาก audit-side case file ที่เป็น store แยก) UI ซ่อน write actions; direct API attempts return `"Read-only role — write actions are not permitted from the Audit workspace."` Audit-side case-file notes เขียนโดย Auditor แต่ไม่ใช่ `tb_purchase_order_comment` entries (ไม่มี PO record change) |
| AUD-PERM-03 | Sysadmin เปิด configuration surface ใด ๆ (numbering, workflow, RBAC, integration, PR-to-PO rules, templates) | **Allow** Sysadmin role ถือ configure rights บนทุก surface ที่ list ใน [03-user-flow-audit-config.md](./03-user-flow-audit-config.md) § 2.2 แต่ละ save บันทึกใน configuration audit log พร้อม identity ของ Sysadmin, version diff, `effective_from` timestamp, และ affected-population count ใด ๆ จาก preview panel |
| AUD-PERM-04 | Sysadmin พยายามสร้าง / submit / approve / transmit PO จาก configuration workspace | **Deny — segregation** Sysadmin role ตั้งค่ารูลแต่ **ไม่** execute transactions บนพวกเขา ตาม `PO_AUTH_001` (Procurement Officer สร้าง), `PO_AUTH_003` (Procurement Officer submit), `PO_AUTH_004` (Procurement Manager approve ที่ high-value), `PO_AUTH_006` (Procurement Officer / Manager transmit), Sysadmin role ไม่อยู่ใน subjects เหล่านี้ UI ไม่ expose ปุ่ม "New PO" ใน configuration workspace; direct API attempts return `"Sysadmin role cannot transact in the purchase-order module — use the Procurement Officer or Procurement Manager role."` |
| AUD-PERM-05 | Auditor request export ที่รวม sensitive fields (full identities, pricelist snapshots, deviations, attachments) | **Allow — พร้อม secondary approval gate** ตาม data-export policy, exports ที่รวม sensitive fields เข้า state `pending`; Auditor ไม่สามารถ bypass Export approver release export หลังจากนั้น CSV / PDF materialise และ download link บันทึกใน audit case file พร้อม identity ของ approver AUD-HP-05 เดิน happy path; AUD-VAL-02 ครอบคลุม unapproved case |
| AUD-PERM-06 | Sysadmin execute elevated void บน PO ที่ deadlocked / orphaned ผ่าน `PO_AUTH_007` escalation | **Allow — เฉพาะ** เมื่อ audit finding หรือ workflow deadlock ต้องการ terminate non-compliant PO และไม่มี Procurement Manager พร้อม Sysadmin ถือ elevated path เพื่อ void ภายใต้ `PO_AUTH_007` (tenant-configured override) Action นี้ต้องการ (a) audit finding reference หรือ deadlock-warning reference จาก configuration audit log, (b) Sysadmin ต้อง confirm ด้วย second-factor / step-up auth, และ (c) reason บังคับใน `tb_purchase_order_comment` Void เอง fire `PO_POST_010` (`* → voided`); override บันทึกใน **ทั้ง** `tb_purchase_order_comment` (`system`) **และ** configuration audit log สำหรับ cross-reference นี่คือเส้นทางเดียวที่ Sysadmin สัมผัสเร็คคอร์ด PO โดยตรง |
| AUD-PERM-07 | User ที่ไม่ใช่ Auditor หรือ Sysadmin (เช่น `purchase@blueledgers.com`, `requestor@blueledgers.com`) เปิด Audit workspace หรือ Configuration workspace | **Deny** ตาม RBAC, เฉพาะ users ที่มี Auditor role สามารถเปิด Audit workspace และเฉพาะ users ที่มี Sysadmin role สามารถเปิด Configuration workspace UI ซ่อนทั้งสอง sidebar entries สำหรับ users ที่ไม่มี role; direct URL navigation return `"Audit workspace requires the Auditor role."` หรือ `"Configuration workspace requires the System Administrator role."` Transactional personas (Purchaser, Receiver, Finance Officer) มี activity views ของตัวเองบน PO record (`ActivityLogTab` มองเห็นที่นั่น) แต่ไม่สามารถเปิด cross-document audit query builder |

| AUD-VAL-04 | Delegation window ที่ `end_at < start_at` ถูก reject ตอน save | Sysadmin เปิด delegation ใหม่; `start_at = 2026-06-14T00:00:00Z`; `end_at = 2026-06-01T00:00:00Z` (swap ผิดพลาด) | Reject save Server return `"Delegation window is invalid — end_at (2026-06-01T00:00:00Z) must be strictly greater than start_at (2026-06-14T00:00:00Z)."` ไม่มี delegation row เขียน; configuration audit log บันทึก rejected attempt AUD-HP-09 คือ happy-path counterpart พร้อม ordering ที่ถูก |
| AUD-VAL-05 | Auditor submit empty filter set บน date-range query | Auditor เปิด **Audit → Procurement Activity Queries**; คลิก Apply โดยไม่ตั้ง filter ใด ๆ | Reject query Server return `"Empty filter set — at least one of date range, business unit, vendor, or buyer must be specified to prevent unbounded scans across the procurement chain."` ไม่มี result set render; ไม่มี audit-side activity-store entry เขียน Auditor ต้องเพิ่มอย่างน้อยหนึ่ง filter chip; ที่พบบ่อยที่สุดคือ date range ที่ใช้ใน AUD-HP-01 |
| AUD-VAL-06 | Sysadmin save threshold-range ด้วย band ที่ invalid | Sysadmin เปิด threshold editor; ตั้ง high-value threshold เป็น `-50000` (negative) หรือ `lower > upper` บน multi-tier threshold | Reject save Server return `"Threshold value is invalid — must be a non-negative number; multi-tier thresholds must satisfy lower < upper."` Configuration ไม่ persist; preview panel ไม่ advance Sysadmin revise input และ re-save Configuration audit log บันทึก rejected attempt เพื่อป้องกัน silent retries |
| AUD-VAL-07 | Sysadmin save no-op (ไม่มี change จริง) | Sysadmin เปิด config surface, เปิด editor, และคลิก Save โดยไม่เปลี่ยน field ใด ๆ | Reject save Server return `"No changes detected — save was not persisted."` Configuration version ไม่เพิ่ม; ไม่มี audit-log entry เขียน นี่ป้องกัน audit-log inflation จาก no-op saves ที่ผิดพลาด Configuration page return เป็น view mode |
| AUD-VAL-08 | Sysadmin พยายาม delete referenced master-data record (เช่น `tax_id` ที่ active PO line อ้างอิง) | Sysadmin เปิด **Configuration → Tax Codes**; เลือก tax code ที่ปัจจุบันอ้างอิงโดยอย่างน้อยหนึ่ง `tb_purchase_order_detail.tax_id` บน non-terminal PO; คลิก **Delete** | Reject delete Server return `"Cannot delete tax code <id> — referenced by N active PO line(s). Archive the tax code instead to preserve historical references."` Archive flow คือเส้นทางที่ยอมรับได้เท่านั้น: archived codes ยังคงใช้ได้บน historical PO snapshots แต่ไม่ selectable สำหรับ PO lines ใหม่ นี่รักษา calculation-rule integrity ใน `PO_CALC_001`–`PO_CALC_011` สำหรับ in-flight และ historical POs |

## 4. Edge Cases

| # | Scenario | เงื่อนไข | Expected |
| - | -------- | --------- | -------- |
| AUD-EDGE-01 | In-flight snapshot boundary — config saved พอดีที่ submission | PO `PO-SB-01` ที่ `po_status = draft`; Purchaser คลิก **Submit** ที่ `T`; Sysadmin save workflow change ด้วย `effective_from = T` (millisecond เดียวกัน) | Race resolve โดย atomic `effective_from` comparison ถ้า timestamp submit ของ `PO-SB-01` ตาม `PO_POST_002` strictly น้อยกว่า `effective_from`, PO capture OLD workflow snapshot (`user_action.execute[]`, stage chain, threshold band) และ configuration ใหม่ apply เฉพาะกับ submissions ถัดไป ถ้า timestamp submit ของ `PO-SB-01` เป็น `≥ effective_from`, PO capture NEW workflow snapshot `workflow_history` entry แรกของ PO บันทึก version configuration ที่มัน bind ไปดังนั้น audit chain ชัดเจน Configuration audit log บันทึก in-flight count และ bordering PO ID สำหรับ forensic clarity |
| AUD-EDGE-02 | Rollback ของ config change ที่ไม่ดี | Sysadmin save RBAC config ใหม่ที่ `T1`; verification ที่ step 6 ของ happy-path flow พบว่า approver ที่ต้องการถูก remove โดยผิดพลาด; Sysadmin revert เป็น version ก่อนที่ `T2 > T1` | Rollback เองเป็น configuration save ด้วย `effective_from = T2` POs ที่สร้างระหว่าง `T1` และ `T2` **ไม่** retroactively re-evaluated (snapshot semantics); จบภายใต้ broken-RBAC snapshot — โดยทั่วไป rescued ผ่าน one-off delegation (AUD-HP-09) หรือ `PO_AUTH_007` void ถ้า deadlocked สมบูรณ์ POs ที่สร้างหลัง `T2` ใช้ rolled-back configuration Configuration audit log เก็บ **ทั้ง** versions (forward change ที่ `T1` และ rollback ที่ `T2`); trail สะอาด preserve กลไก snapshot boundary ของ AUD-EDGE-01 ก็ apply ที่ `T2` |
| AUD-EDGE-03 | Mass-delegation activation ที่ organisation boundary | Sysadmin ตั้ง delegation window `start_at = 2026-12-25T00:00:00+07:00 .. 2026-12-31T23:59:59+07:00` ครอบคลุม 12 approvers (year-end holiday); PO load ระหว่าง window สูงผิดปกติ (300+ POs in flight) | Window activation atomic ที่ `start_at` — ทั้ง 12 delegates inherit `user_action.execute[]` membership สำหรับ scopes ของพวกเขาที่ instant เดียวกัน Notifications สำหรับ 300+ in-flight POs batch (ไม่ใช่ per-PO 300 emails แต่ per-delegate digests) เพื่อหลีกเลี่ยง notification storms Stage routing ของแต่ละ PO ตอนนี้เสนอ **ทั้ง** approver ดั้งเดิม **และ** delegate; ทั้งสองอาจ approve บน `end_at`, ทั้ง 12 สิทธิ์ที่ inherit drop atomically; in-flight POs ดำเนินต่อด้วย approvers ดั้งเดิมของพวกเขา Configuration audit log บันทึก mass activation, delegate count, และ PO-affected count สำหรับ forensic reconstruction |
| AUD-EDGE-04 | Audit-log gap — system event ที่หาย | Auditor เดิน timeline ของ `PO-GAP-01`; พบว่า `workflow_history` มี entry `submitted` ที่ `T0` และ entry `approved (final)` ที่ `T2` แต่ **หาย** intermediate `approved` entry ใด ๆ (`PO_POST_003`) ที่ควรนั่งระหว่างพวกเขาสำหรับ multi-stage workflow | Gap เองเป็น audit finding Auditor flag `PO-GAP-01` ใน case file ด้วย note "missing intermediate workflow_history entry" แนบ timeline excerpt + configured workflow stage count (ที่บอกว่าควรมี N entries) และ route ไปยัง Sysadmin สำหรับการ investigate สาเหตุที่เป็นไปได้: (a) replay-log corruption (escalate ไปยัง ops), (b) transition ที่ bypass writer (escalate ไปยัง engineering), (c) soft-deleted comment (`PO_POST_012` ไม่ apply กับ `workflow_history` — workflow_history เป็น append-only) Auditor **ไม่** ปิด case เป็น "no findings"; case ยังเปิดจนกว่า gap ถูกอธิบาย |
| AUD-EDGE-05 | Query timeout บน huge dataset | Auditor ตั้ง date range = `2024-01-01 .. 2026-05-15` (29 เดือน) + ไม่มี filter อื่น เทียบกับ tenant ที่มี 80,000+ POs ในขอบเขต | Query กระทบ date-range scan limit (AUD-VAL-05 ป้องกัน empty case แต่ 29-month single-filter ยังเสี่ยง timeout) ระบบ chunk query — return rows 1,000 แรก + cursor "Continue to next page" persist query plan และรัน remainder asynchronously Auditor สามารถ page ผ่าน synchronously หรือ schedule full-extract export (ซึ่งจากนั้นเข้า AUD-HP-05 / AUD-VAL-02 export-approval flow) ไม่มี timeout error surface ให้ user; experience degrade gracefully เป็น paged + async |
| AUD-EDGE-06 | Config save vs in-flight PO submission race | Sysadmin คลิก Save ที่ `T`; พร้อมกัน 5 Purchasers คลิก Submit บน 5 draft POs ที่ต่างกันใน browser tabs ข้างเคียง; ทั้ง 6 actions มาถึง backend ภายใน window 500 ms เดียวกัน | Backend serialise โดย timestamp `effective_from` บน configuration row และ `PO_POST_002` `last_action_at_date` บนแต่ละ PO แต่ละ PO submission tag atomically ด้วย version configuration active ที่ instant ของการเขียน `PO_POST_002` Result: ของ 5 submissions, บางตัวอาจ capture OLD snapshot และบางตัว NEW ขึ้นอยู่กับ order — แต่ `workflow_history` entry แรกของแต่ละ PO บันทึก ชัดเจนว่า version ใดที่มัน bind ไป ไม่มี PO จบใน hybrid state; ไม่มี submission ถูก reject เป็น "configuration in transition" Configuration audit log บันทึก count ของ in-flight submissions ที่ resolve เทียบกับแต่ละด้านของ boundary สำหรับ forensic clarity |
