# ใบสั่งซื้อ — กรณีทดสอบ — ผู้ตรวจสอบและผู้ดูแลระบบ (Audit / Config)

หน้านี้รวบรวมกรณีทดสอบที่ persona axis **ผู้ตรวจสอบและผู้ดูแลระบบ (Audit / Config)** เป็นผู้ขับเคลื่อนโดยตรงในโมดูล `purchase-order` มีสอง role ที่อยู่บน axis นี้: **ผู้ตรวจสอบ (Auditor)** (read-only ตลอด chain การจัดซื้อทั้งสาย — Purchase Request → Purchase Order → Goods Receive Note → ใบแจ้งหนี้ผู้ขาย / การ post AP) ซึ่งใช้ `ActivityLogTab` ของโมดูล PO, `workflow_history`, `tb_purchase_order_comment` (`PO_POST_005`, `PO_POST_009`) และ bridge ข้ามเอกสาร (`tb_purchase_order_detail_tb_purchase_request_detail`, back-reference จาก GRN ตาม `PO_XMOD_003`, record three-way-match ภายใต้ `PO_POST_008`) เพื่อ verify การปฏิบัติตามนโยบาย, segregation of duties (`PO_AUTH_010`), ความถูกต้องของ three-way-match (`PO_POST_008` / `PO_POST_009`) และความสามารถในการสืบกลับตลอด chain; และ **ผู้ดูแลระบบ (System Administrator)** (ตั้งค่าเท่านั้น) ซึ่งดูแลรูปแบบเลขที่ PO, นิยาม workflow (`tb_purchase_order.workflow_id`, stage, สมาชิก `user_action.execute[]` ตาม `PO_AUTH_011`), แผนที่ RBAC role-to-permission สำหรับ `PO_AUTH_001`–`PO_AUTH_011`, การเชื่อม integration (vendor, vendor-pricelist, งบประมาณ, inventory), กฎ PR-to-PO conversion และ vendor+currency grouping ตาม carmen/docs § 2.3.2, document template ที่อ้างอิงเมื่อส่งตาม `PO_POST_004` และแหล่ง tax / สกุลเงิน / FX rate ที่ใช้ใน `PO_CALC_001`–`PO_CALC_011` ทั้งสอง role ไม่ transition PO ข้าม `enum_purchase_order_doc_status`: ผู้ตรวจสอบออกผ่านการสร้างรายงาน (PO state ไม่เปลี่ยน), ผู้ดูแลระบบออกผ่านการบันทึก configuration พร้อม timestamp `effective_from` (PO ที่อยู่ที่ `in_progress` / `sent` / `partial` แล้วยังคง snapshot เดิม; PO ใหม่ใช้ configuration ใหม่) กรณีทดสอบถูกจัดเป็น happy path ตามที่อธิบายใน [03-user-flow-audit-config.md](./03-user-flow-audit-config.md), ขอบเขต RBAC ที่แยก read-only-audit ออกจาก configure-only-sysadmin ออกจาก no-direct-transact, validation rule รอบ ๆ การคง snapshot, การอนุมัติ export, การป้องกัน deadlock, หน้าต่างเวลา delegation และข้อจำกัด scan และชุด edge case จำนวนเล็ก ๆ เกี่ยวกับขอบ in-flight, การ rollback, mass delegation, audit-log gap และ race ระหว่าง config กับ in-flight สเป็ค E2E `401-po.spec.ts` ที่ใช้ร่วมกัน exercise เส้นทาง transactional เท่านั้น; surface การ query audit-log และการจัดการ configuration ครอบคลุมโดย API / integration test ในปัจจุบัน (ยังไม่มี audit-config E2E spec เฉพาะ)

## 2. Permission / Authorization

| # | กรณี | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | ---- | ---------------------------------------- |
| AUD-PERM-01 | ผู้ตรวจสอบเปิด PO ใด ๆ ใน `po_status` ใด ๆ (read-only ตลอด chain) | **Allow** ตาม mandate read-only ของ role ผู้ตรวจสอบ ผู้ตรวจสอบสามารถเปิด record PR, PO, GRN, ใบแจ้งหนี้ และ AP ในทุกสถานะรวมถึง terminal (`completed`, `closed`, `voided`) หน้ารายละเอียด PO ในมุมมองของผู้ตรวจสอบเป็น read-only: action edit / submit / approve / void / close / transmit ไม่ถูก expose ใน UI; การเรียก API edit ตรง ๆ ตอบกลับ `"You are not authorised to edit this PO at the current stage."` ไม่มีทาง mutate `tb_purchase_order` |
| AUD-PERM-02 | ผู้ตรวจสอบพยายาม write action (แก้บรรทัด, void, comment เป็นผู้กระทำ) บน PO | **Deny** ผู้ตรวจสอบไม่มี write surface ในโมดูล `purchase-order` — ไม่ได้บน `tb_purchase_order`, ไม่ได้บน `tb_purchase_order_detail`, ไม่ได้บน `tb_purchase_order_comment` (นอกจาก case file ฝั่ง audit ซึ่งเป็น store แยก) UI ซ่อน write action; การเรียก API ตรง ๆ ตอบกลับ `"Read-only role — write actions are not permitted from the Audit workspace."` Note ใน case file ฝั่ง audit ถูก write โดยผู้ตรวจสอบแต่ **ไม่ใช่** entry `tb_purchase_order_comment` (PO record ไม่เปลี่ยน) |
| AUD-PERM-03 | ผู้ดูแลระบบเปิด configuration surface ใด ๆ (numbering, workflow, RBAC, integration, PR-to-PO rule, template) | **Allow** role ผู้ดูแลระบบถือสิทธิ์ configure บนทุก surface ที่ระบุใน [03-user-flow-audit-config.md](./03-user-flow-audit-config.md) § 2.2 การ save แต่ละครั้งถูกบันทึกใน configuration audit log พร้อม identity ของผู้ดูแลระบบ, diff ของ version, timestamp `effective_from` และจำนวน affected-population จาก preview panel |
| AUD-PERM-04 | ผู้ดูแลระบบพยายาม create / submit / approve / transmit PO จาก configuration workspace | **Deny — segregation** role ผู้ดูแลระบบตั้งค่ากฎแต่ **ไม่** execute transaction บนกฎเหล่านั้น ตาม `PO_AUTH_001` (Procurement Officer create), `PO_AUTH_003` (Procurement Officer submit), `PO_AUTH_004` (Procurement Manager approve ที่ high-value), `PO_AUTH_006` (Procurement Officer / Manager transmit) role ผู้ดูแลระบบไม่ได้เป็น subject ในใด ๆ เหล่านี้ UI ไม่ expose ปุ่ม "New PO" ใน configuration workspace; การเรียก API ตรง ๆ ตอบกลับ `"Sysadmin role cannot transact in the purchase-order module — use the Procurement Officer or Procurement Manager role."` |

## 3. Validation / Error

| # | กรณี | Trigger | Error ที่คาดหวัง |
| - | ---- | ------- | ---------------- |
| AUD-VAL-01 | ผู้ดูแลระบบเปลี่ยน config ขณะมี PO ในกระบวนการ — snapshot ถูกคง | PO `PO-SP-01` ที่ `po_status = in_progress` ภายใต้เกณฑ์ high-value เดิม `฿1,000,000`; ผู้ดูแลระบบเปลี่ยนเกณฑ์เป็น `฿2,000,000` ขณะที่ `PO-SP-01` ยังอยู่กลาง workflow | การ save อนุญาต (กฎไม่ถูก lock โดย PO ในกระบวนการ) `PO-SP-01` ดำเนินต่อภายใต้ band เกณฑ์ที่ snapshot ไว้และ chain stage; การ routing **ไม่** retroactively เปลี่ยน Preview panel surface จำนวน PO ที่ได้รับผลกระทบก่อน save Configuration audit log บันทึกการเปลี่ยนแปลงและจำนวน in-flight เมื่อ save, PO ใหม่ที่สร้างหลัง `effective_from` ใช้เกณฑ์ใหม่ `฿2,000,000`; `PO-SP-01` finish workflow ปัจจุบันภายใต้ routing เดิม `฿1,000,000` ไม่มี mutation `tb_purchase_order` เกิดขึ้นจากการ save config |
| AUD-VAL-02 | พยายาม bypass การอนุมัติ export | ผู้ตรวจสอบ login; ขอ export CSV ที่มี field sensitive; data-export approver **ยัง**ไม่อนุมัติ | Reject การ download server ตอบกลับ `"Export requires approval — sensitive fields detected (buyer identity, pricelist snapshot, deviation detail). Export is in pending state until release by an export approver."` Link download ไม่ถูกสร้าง; export ที่ pending ยังอยู่ใน case file ผู้ตรวจสอบคลิกผ่านไม่ได้; การเรียก API ตรง ๆ เพื่อ materialise ตอบกลับ `403 Forbidden — approval required` AUD-HP-05 เป็นคู่ happy-path |
| AUD-VAL-03 | ผู้ดูแลระบบพยายาม save config ที่จะ deadlock PO ที่อยู่ในกระบวนการ | PO `PO-DL-01` รออยู่ที่ stage `S2` ด้วย user เดียว `last.approver@blueledgers.com` ใน `user_action.execute[]`; ผู้ดูแลระบบเปิด workflow editor และพยายามลบ `last.approver@blueledgers.com` ออกจาก `S2` | Preview panel flag deadlock: `"Saving this change will leave PO-DL-01 with no eligible approver at stage S2 — the PO will time out at this stage and require manual intervention (one-off delegation or void under PO_AUTH_007)."` ผู้ดูแลระบบถูกถามให้ (a) cancel, (b) เพิ่ม approver ทดแทนใน `S2` ก่อน save หรือ (c) save ทั้งที่มี warning ที่รับทราบแล้ว ถ้า save ทั้งที่มี warning, configuration audit log บันทึกทั้งการเปลี่ยนแปลงและ deadlock warning เพื่อ audit attribution |
| AUD-VAL-04 | หน้าต่างเวลา delegation ด้วย `end_at < start_at` ถูก reject ที่ save | ผู้ดูแลระบบเปิด delegation ใหม่; `start_at = 2026-06-14T00:00:00Z`; `end_at = 2026-06-01T00:00:00Z` (สลับกันโดยพลาด) | Reject การ save server ตอบกลับ `"Delegation window is invalid — end_at (2026-06-01T00:00:00Z) must be strictly greater than start_at (2026-06-14T00:00:00Z)."` ไม่มีแถว delegation ถูก write; configuration audit log บันทึกความพยายามที่ถูก reject AUD-HP-09 เป็นคู่ happy-path ที่มีลำดับถูกต้อง |
| AUD-VAL-05 | ผู้ตรวจสอบ submit ชุด filter ว่างบน query ช่วงวันที่ | ผู้ตรวจสอบเปิด **Audit → Procurement Activity Queries**; คลิก Apply โดยไม่ตั้ง filter ใด ๆ | Reject query server ตอบกลับ `"Empty filter set — at least one of date range, business unit, vendor, or buyer must be specified to prevent unbounded scans across the procurement chain."` ไม่มีชุด result ถูก render; ไม่มี entry activity-store ฝั่ง audit ถูก write ผู้ตรวจสอบต้องเพิ่ม filter chip อย่างน้อยหนึ่งอัน; ที่พบบ่อยที่สุดคือช่วงวันที่ที่ใช้ใน AUD-HP-01 |
| AUD-VAL-06 | ผู้ดูแลระบบ save threshold-range ด้วย band ที่ไม่ valid | ผู้ดูแลระบบเปิด threshold editor; ตั้งเกณฑ์ high-value เป็น `-50000` (ลบ) หรือเป็น `lower > upper` บนเกณฑ์หลายระดับ | Reject การ save server ตอบกลับ `"Threshold value is invalid — must be a non-negative number; multi-tier thresholds must satisfy lower < upper."` Configuration ไม่ถูก persist; preview panel ไม่เดินหน้า ผู้ดูแลระบบแก้ input และ save ใหม่ Configuration audit log บันทึกความพยายามที่ถูก reject เพื่อป้องกันการ retry แบบเงียบ |
| AUD-VAL-07 | ผู้ดูแลระบบ save no-op (ไม่มีการเปลี่ยนแปลงจริง) | ผู้ดูแลระบบเปิด config surface, เปิด editor และคลิก Save โดยไม่เปลี่ยน field ใด ๆ | Reject การ save server ตอบกลับ `"No changes detected — save was not persisted."` Version ของ configuration ไม่ถูก increment; ไม่มี entry audit-log ถูก write นี่ป้องกัน audit-log inflation จากการ save no-op โดยพลาด หน้า configuration กลับสู่ view mode |
| AUD-VAL-08 | ผู้ดูแลระบบพยายาม delete record master-data ที่ถูกอ้างอิง (เช่น `tax_id` ที่อ้างอิงโดยบรรทัด PO ที่ active) | ผู้ดูแลระบบเปิด **Configuration → Tax Codes**; เลือก tax code ที่ปัจจุบันถูกอ้างอิงโดยอย่างน้อยหนึ่ง `tb_purchase_order_detail.tax_id` บน PO ที่ไม่ terminal; คลิก **Delete** | Reject การ delete server ตอบกลับ `"Cannot delete tax code <id> — referenced by N active PO line(s). Archive the tax code instead to preserve historical references."` Flow archive เป็นเส้นทางเดียวที่ยอมรับได้: code ที่ archive ยังคง valid บน snapshot PO ประวัติแต่ไม่สามารถเลือกได้สำหรับบรรทัด PO ใหม่ นี่คงความถูกต้องของกฎ calculation ใน `PO_CALC_001`–`PO_CALC_011` สำหรับ PO ในกระบวนการและประวัติ |

## 4. Edge Cases

| # | กรณี | เงื่อนไข | ผลที่คาดหวัง |
| - | ---- | -------- | ------------ |
| AUD-EDGE-01 | ขอบ in-flight snapshot — config save พอดีตอน submission | PO `PO-SB-01` อยู่ที่ `po_status = draft`; Purchaser คลิก **Submit** ที่ `T`; ผู้ดูแลระบบ save การเปลี่ยน workflow ด้วย `effective_from = T` (millisecond เดียวกัน) | Race ถูกแก้โดยการเปรียบเทียบ `effective_from` แบบ atomic ถ้า timestamp submit `PO_POST_002` ของ `PO-SB-01` น้อยกว่า `effective_from` strictly, PO capture workflow snapshot **เดิม** (`user_action.execute[]`, chain stage, band threshold) และ configuration ใหม่ใช้กับ submission ที่ตามมาเท่านั้น ถ้า timestamp submit ของ `PO-SB-01` คือ `≥ effective_from`, PO capture workflow snapshot **ใหม่** entry `workflow_history` แรกของ PO บันทึก version ของ configuration ที่ผูกไว้ ดังนั้น audit chain ชัดเจน Configuration audit log บันทึกจำนวน in-flight และ ID ของ PO ที่อยู่บนขอบเพื่อความชัดเจนทางนิติวิทยา |
| AUD-EDGE-02 | Rollback ของการเปลี่ยน config ที่ผิด | ผู้ดูแลระบบ save RBAC config ใหม่ที่ `T1`; verification ที่ขั้นตอน 6 ของ happy-path flow พบว่า approver ที่จำเป็นถูกลบออกโดยพลาด; ผู้ดูแลระบบ revert ไป version ก่อนหน้าที่ `T2 > T1` | Rollback เองคือการ save configuration ด้วย `effective_from = T2` PO ที่สร้างระหว่าง `T1` ถึง `T2` **ไม่** ถูก re-evaluate ย้อนหลัง (snapshot semantics); finish ภายใต้ snapshot RBAC ที่เสีย — โดยทั่วไป rescue ผ่าน one-off delegation (AUD-HP-09) หรือ void `PO_AUTH_007` ถ้า deadlock เต็มที่ PO ที่สร้างหลัง `T2` ใช้ configuration ที่ rollback Configuration audit log เก็บทั้งสอง version (การเปลี่ยน forward ที่ `T1` และ rollback ที่ `T2`); คง trail สะอาด กลไก snapshot boundary ของ AUD-EDGE-01 ใช้ได้ที่ `T2` ด้วย |
| AUD-EDGE-03 | การ activate mass-delegation ที่ขอบ organization | ผู้ดูแลระบบตั้งหน้าต่างเวลา delegation `start_at = 2026-12-25T00:00:00+07:00 .. 2026-12-31T23:59:59+07:00` ครอบคลุม approver 12 คน (วันหยุดสิ้นปี); ภาระ PO ระหว่างหน้าต่างเวลาสูงผิดปกติ (PO 300+ ฉบับในกระบวนการ) | การ activate หน้าต่างเวลาเป็น atomic ที่ `start_at` — ผู้รับมอบอำนาจทั้ง 12 คนสืบทอดสมาชิก `user_action.execute[]` สำหรับ scope ของตน ณ instant เดียวกัน Notification สำหรับ PO 300+ ฉบับในกระบวนการถูก batch (ไม่ใช่ email ต่อ PO 300 ฉบับ แต่เป็น digest ต่อ delegate) เพื่อหลีกเลี่ยง notification storm Stage routing ของแต่ละ PO ตอนนี้เสนอทั้ง approver เดิมและผู้รับมอบอำนาจ; ใครก็ได้สามารถอนุมัติ ที่ `end_at` สิทธิ์ที่สืบทอดทั้ง 12 ตัวถูกตัดแบบ atomic; PO ในกระบวนการดำเนินต่อกับ approver เดิม Configuration audit log บันทึกการ activate mass, จำนวน delegate และจำนวน PO ที่ได้รับผลกระทบเพื่อการ reconstruct ทางนิติวิทยา |
| AUD-EDGE-04 | Audit-log gap — system event ขาด | ผู้ตรวจสอบเดิน timeline ของ `PO-GAP-01`; พบว่า `workflow_history` มี entry `submitted` ที่ `T0` และ entry `approved (final)` ที่ `T2` แต่ **ขาด** entry `approved` กลาง (`PO_POST_003`) ที่ควรอยู่ระหว่างนั้นสำหรับ workflow หลาย stage | Gap เองเป็น audit finding ผู้ตรวจสอบ flag `PO-GAP-01` ใน case file ด้วย note "ขาด entry workflow_history กลาง", แนบ excerpt ของ timeline + จำนวน stage ของ workflow ที่ตั้งค่าไว้ (ซึ่งบอกว่าควรมี N entry) และ route ไปยังผู้ดูแลระบบเพื่อสืบสวน สาเหตุที่เป็นไปได้: (a) replay-log corruption (escalate ไป ops), (b) transition ที่ bypass writer (escalate ไป engineering), (c) comment ที่ soft-delete (`PO_POST_012` ไม่ใช้กับ `workflow_history` — workflow_history เป็น append-only) ผู้ตรวจสอบ **ไม่** ปิด case ว่า "no findings"; case เปิดอยู่จนกว่า gap จะได้รับการอธิบาย |
| AUD-EDGE-05 | Query timeout บนชุดข้อมูลขนาดใหญ่ | ผู้ตรวจสอบตั้งช่วงวันที่ = `2024-01-01 .. 2026-05-15` (29 เดือน) + ไม่มี filter อื่น เทียบกับ tenant ที่มี PO 80,000+ ฉบับใน scope | Query ชน scan limit ของช่วงวันที่ (AUD-VAL-05 ป้องกันกรณีว่าง แต่ filter เดียว 29 เดือนยังเสี่ยง timeout) ระบบ chunk query — ส่งคืน 1,000 แถวแรก + cursor "Continue to next page" ก่อน, persist query plan และรันส่วนที่เหลือแบบ asynchronous ผู้ตรวจสอบสามารถเลือก page ผ่านแบบ synchronous หรือ schedule full-extract export (ซึ่งจากนั้นเข้า flow การอนุมัติ export ของ AUD-HP-05 / AUD-VAL-02) ไม่มี timeout error ปรากฏให้ user; ประสบการณ์ degrade graceful เป็น paged + async |
| AUD-EDGE-06 | Race ระหว่าง config save กับ submission ของ PO ในกระบวนการ | ผู้ดูแลระบบคลิก Save ที่ `T`; ในเวลาเดียวกัน Purchaser 5 คนคลิก Submit บน PO draft ที่แตกต่างกัน 5 ฉบับใน tab browser ใกล้กัน; ทั้ง 6 action มาถึง backend ภายในหน้าต่างเวลา 500 ms เดียวกัน | Backend serialise โดย timestamp `effective_from` บนแถว configuration และ `last_action_at_date` ของ `PO_POST_002` บนแต่ละ PO การ submit แต่ละ PO ถูก tag แบบ atomic ด้วย version configuration ที่ active ณ instant ของการ write `PO_POST_002` ผลลัพธ์: จาก 5 submission, บางอันอาจ capture snapshot **เดิม** และบางอัน **ใหม่** ขึ้นกับลำดับ — แต่ entry `workflow_history` แรกของแต่ละ PO บันทึกชัดเจนว่าผูกกับ version ใด ไม่มี PO ลงเอยใน state hybrid; ไม่มี submission ถูก reject ว่า "configuration in transition" Configuration audit log บันทึกจำนวน in-flight submission ที่ resolve เทียบกับแต่ละด้านของขอบเพื่อความชัดเจนทางนิติวิทยา |
