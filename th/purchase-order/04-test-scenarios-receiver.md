# ใบสั่งซื้อ — กรณีทดสอบ — ผู้รับสินค้า (Receiver)

หน้านี้รวบรวมกรณีทดสอบที่ persona ของผู้รับสินค้า (**Store Keeper** ที่จุดรับ บวก **Inventory Manager** ที่กำกับการปิดรับของสถานที่) เป็นผู้ขับเคลื่อนโดยตรงในโมดูล `purchase-order` Receiver เข้ามาเกี่ยวข้องเมื่อ PO ขึ้น `po_status = sent` และจบเมื่อ PO ออกจากโซนการรับสินค้า — ทาง `completed` (รับครบ), `partial` (ยังคงค้าง) หรือ `closed` (Inventory Manager ยุติ partial PO ก่อนกำหนด) การ post GRN เองอยู่ในโมดูลปลายน้ำ `[[good-receive-note]]`; หน้านี้รวบรวมเฉพาะ **ผลกระทบฝั่ง PO** — GRN flip `tb_purchase_order.po_status` ผ่าน `PO_POST_006` (`sent → partial`) และ `PO_POST_007` (`* → completed`) อย่างไร Inventory Manager ปิด PO ที่ `partial` พร้อมเขียนยอดที่เหลือไปที่ `cancelled_qty` ผ่าน `PO_POST_011` อย่างไร และตัวนับบนบรรทัด PO (`received_qty`, `cancelled_qty`) เคลื่อนเทียบกับ `order_qty` อย่างไร ([03-user-flow-receiver.md](./03-user-flow-receiver.md) หัวข้อ 2) กรณีทดสอบถูกจัดเป็น happy path ตามที่อธิบายใน user flow, ขอบเขต RBAC ที่บังคับโดย `PO_AUTH_008` (ขอบเขต Receiver / Inventory Manager) และ `PO_AUTH_010` (segregation of duties — ผู้ post GRN ≠ buyer / transmitter ของ PO) เทียบกับ `PO_AUTH_001`–`PO_AUTH_007` (ขอบเขต Purchaser / Manager), validation rule รอบ ๆ `received_qty`, `accepted_qty`, tolerance ของการรับเกิน และด่านสถานะ voided / closed PO และชุด edge case จำนวนเล็ก ๆ เกี่ยวกับขอบ tolerance, ความละเอียดทศนิยม, concurrency และบริบทหลายสกุลเงิน เส้นทาง cross-persona ที่หมุนรอบ Receiver (`X-PO-03`, `X-PO-04`, `X-PO-06`, `X-PO-08`) อยู่ในหน้าภาพรวมพ่อ ไม่ได้อยู่ในหน้านี้

## 2. Permission / Authorization

| # | กรณี | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | ---- | ---------------------------------------- |
| RCV-PERM-01 | Receiver post GRN เทียบกับ PO ที่ `po_status = sent` | **Allow** ตาม `PO_AUTH_008` (`po_status ∈ {sent, partial}`) flow Create ของ `[[good-receive-note]]` รับการเลือก PO; ตัวนับฝั่ง PO update; `po_status` เปลี่ยนตาม `PO_POST_006` / `PO_POST_007` ตามผลการประเมินรายบรรทัด |
| RCV-PERM-02 | Receiver พยายาม post GRN เทียบกับ PO ที่ `po_status = draft` | **Deny** ตาม `PO_AUTH_008` (status ไม่อยู่ใน `{sent, partial}`) PO ไม่ปรากฏใน vendor / PO picker ของโมดูล GRN; เรียก API ตรง ๆ ตอบกลับ `"GRN cannot be posted against PO at status draft — PO must be sent or partial."`; ตัวนับไม่เปลี่ยน |
| RCV-PERM-03 | Receiver พยายามแก้ header ของ PO (vendor, currency, delivery date) | **Deny — Purchaser only** ตาม `PO_AUTH_010` (segregation of duties) และ `PO_AUTH_002` (ขอบเขต Purchaser) การแก้ header / บรรทัด สงวนไว้ให้ buyer ที่ถูกจัดให้; Receiver ไม่สามารถแก้ทั้ง header และราคา / qty / discount ของบรรทัด หน้ารายละเอียด PO ในมุมมองของ Receiver เป็น read-only สำหรับ header / บรรทัด — มีเพียงปุ่ม **Receive** และ (สำหรับ Inventory Manager) **Close** เท่านั้นที่ใช้งานได้ เรียก API แก้ตรง ๆ ตอบกลับ `"You are not authorised to edit this PO at the current stage"` |
| RCV-PERM-04 | ผู้ post GRN = buyer / transmitter ของ PO (segregation of duties) | **Deny — segregation of duties** ตาม `PO_AUTH_010` ผู้ใช้ที่ระบุโดย `tb_purchase_order.buyer_id` หรือ `last_action_by_id` บน transition `sent` ห้ามเป็นคนเดียวกับผู้ post GRN โมดูล GRN reject การ post ตอน create-time ด้วย `"You created or transmitted this PO; another user must post the GRN against it."`; ไม่บันทึกแถว GRN; ตัวนับ PO ไม่เปลี่ยน |
| RCV-PERM-05 | Receiver (Store Keeper) พยายามปิด PO ที่ `partial` (early termination) | **Deny เมื่อ role = Store Keeper เท่านั้น; allow เมื่อ role = Inventory Manager** ตาม `PO_AUTH_008` และ `PO_POST_011` action Close (`partial → closed`) ต้องใช้ role Inventory Manager (หรือ Procurement Manager override ผ่าน `PO_AUTH_007`) UI ของ Store Keeper ซ่อน / ปิดปุ่ม **Close** บน header ของ PO; เรียก API ตรง ๆ ตอบกลับ `"Close from status partial requires the Inventory Manager role."` |
| RCV-PERM-06 | Receiver พยายาม post GRN เทียบกับ PO ที่ `po_status = completed` หรือ `closed` | **Deny — status เป็น terminal** ตาม `PO_POST_007` / `PO_POST_011` `completed` และ `closed` เป็น terminal สำหรับการรับ; ไม่รับ GRN เพิ่ม PO ไม่ปรากฏในรายการ open-PO ของโมดูล GRN; เรียก API ตรง ๆ ตอบกลับ `"PO is at terminal receipt status <status> — no further receipts can be posted."` |
| RCV-PERM-07 | Inventory Manager ปิด PO ที่ `partial` ที่ตนไม่ได้สร้าง | **Allow** ตาม `PO_AUTH_008` อำนาจของ Inventory Manager ผูกกับสถานที่จัดส่ง ไม่ใช่ผูกกับผู้สร้าง PO; segregation of duties (`PO_AUTH_010`) ใช้กับการ post GRN ไม่ใช่กับการปิด PO การปิดสำเร็จพร้อมเหตุผลที่ผนวกใน `tb_purchase_order_comment` |

## 3. Validation / Error

| # | กรณี | Trigger | Error ที่คาดหวัง |
| - | ---- | ------- | ---------------- |
| RCV-VAL-01 | รับเกินโดยไม่มี tolerance — `received_qty > ยอดค้าง` | บรรทัด PO `L1`: `order_qty = 10`, `received_qty = 0`, `cancelled_qty = 0`; tenant ปิด tolerance รับเกินหรือตั้งเป็น `0%`; Receiver กรอก `received_qty = 12` | Reject — `PO_XMOD_004` server ตอบกลับ `"Received quantity exceeds the pending balance and tenant over-receipt tolerance is not configured."`; บรรทัด GRN ถูก cap ที่ยอดค้าง (10) หรือบล็อกการ post ทั้งหมด; ตัวนับ PO ไม่เปลี่ยน |
| RCV-VAL-02 | `received_qty = 0` บนบรรทัด GRN | Receiver กรอก `received_qty = 0` บนบรรทัดแล้วกด **Post** | Reject — validation ฝั่ง GRN / `PO_XMOD_003` server ตอบกลับ `"Received quantity must be greater than zero on every GRN line; remove the line if nothing was received against it."`; GRN post ไม่ได้กับบรรทัด qty 0; ตัวนับ PO ไม่เปลี่ยน |
| RCV-VAL-03 | GRN เทียบกับ PO ที่ `voided` | PO `PO-V` ที่ `po_status = voided` (Procurement Manager void ตาม `PO_AUTH_007`); Receiver พยายามเปิด flow GRN จากโมดูล GRN โดยตรง | Reject — `PO_AUTH_007` (voided เป็น terminal) server ตอบกลับ `"PO is at status voided — no GRN can be posted."`; PO ไม่ปรากฏในรายการ open-PO; เรียก API ตรง ๆ ถูกบล็อกตั้งแต่ขั้น create ของ GRN |
| RCV-VAL-04 | `accepted_qty > received_qty` (คุณภาพเกินรับจริง) | Receiver กรอก `received_qty = 8`, `accepted_qty = 10` บนบรรทัดแล้วกด **Post** | Reject — validation ฝั่ง GRN server ตอบกลับ `"Accepted quantity cannot exceed received quantity on any GRN line."`; inequality `accepted_qty ≤ received_qty` ถูกบังคับตอน save และ post; GRN post ไม่ได้; ตัวนับ PO ไม่เปลี่ยน |
| RCV-VAL-05 | GRN บางส่วนเมื่อยอดที่เหลือรับไม่ได้ (pending ติดลบ) | บรรทัด PO `L1`: `order_qty = 10`, `received_qty = 8`, `cancelled_qty = 2` (ยอดค้าง = 0); Receiver พยายาม post GRN ด้วย `received_qty = 1` บน `L1` | Reject — `PO_XMOD_003` server ตอบกลับ `"PO line has no remaining open balance — received_qty (8) + cancelled_qty (2) = order_qty (10)."`; บรรทัด GRN post ไม่ได้บน `L1`; ถ้ามีบรรทัดอื่นบน PO ที่ยังเปิด บรรทัดเหล่านั้น post ต่างหากได้ |
| RCV-VAL-06 | GRN เทียบกับ PO ที่ vendor ถูก flip เป็น `is_active = false` หลัง `sent` | PO `PO-W` ที่ `po_status = sent`; `tb_vendor.is_active = false` บน vendor ของ PO (vendor ถูก blacklist หลังการส่ง) | Reject — vendor-state guard server ตอบกลับ `"Vendor is no longer active and cannot deliver further goods. Escalate to the Procurement Manager."`; ขั้น create ของ GRN ถูกบล็อก; เส้นทางแก้คือ Procurement Manager void (`PO_AUTH_007`) หรือ fix vendor master |

## 4. Edge Cases

| # | กรณี | เงื่อนไข | ผลที่คาดหวัง |

| RCV-EDGE-01 | ขอบ tolerance พอดีบน over-receipt | บรรทัด PO `L1`: `order_qty = 10`, `received_qty = 0`, `cancelled_qty = 0`; tenant ตั้ง tolerance รับเกิน = `5%`; Receiver กรอก `received_qty = 10.5` (พอดีขอบบน) | ยอมรับ — `PO_XMOD_004` `received_qty = 10.5` อยู่ใน tolerance (`10 × 1.05 = 10.5`); GRN post; `tb_purchase_order_detail.received_qty = 10.5`; การประเมินบรรทัด: `received_qty (10.5) ≥ order_qty − cancelled_qty (10)` → รับเต็ม; `po_status = sent → completed` ตาม `PO_POST_007` ตั้ง `received_qty = 10.51` จะข้ามขอบและถูก reject ตาม RCV-VAL-01 |
