# ใบสั่งซื้อ — เส้นทางผู้ใช้งาน — ผู้ตรวจสอบและผู้ดูแลระบบ (Audit / Config)

## 1. บทบาทในโมดูลนี้

แกน persona **Audit / Config** รวมสองบทบาทที่แตกต่างกันแต่ทั้งคู่อยู่ **นอกเส้นทางหลักของรายการ (transactional happy path)** ของโมดูล `purchase-order` ทั้งสองบทบาทเป็นกลไกสำคัญสำหรับการกำกับดูแลและการทำงานของระบบ **Auditor (ผู้ตรวจสอบ)** เป็น persona แบบอ่านอย่างเดียวที่สนใจ **ห่วงโซ่การจัดซื้อแบบ end-to-end** — Purchase Request → Purchase Order → Goods Receive Note → vendor invoice / AP posting — และใช้ activity log ของโมดูล PO (`ActivityLogTab`), `workflow_history`, `tb_purchase_order_comment` (`PO_POST_005`, `PO_POST_009`) และ bridge ระหว่างเอกสาร (PR→PO ผ่าน `tb_purchase_order_detail_tb_purchase_request_detail`, PO→GRN ผ่าน back-reference ของ GRN detail ตาม `PO_XMOD_003`, PO/GRN→invoice ผ่าน record ของ three-way match ภายใต้ `PO_POST_008`) เพื่อยืนยันการปฏิบัติตามนโยบาย, segregation of duties (`PO_AUTH_010`: Purchaser ≠ Receiver), ความครบถ้วนของ three-way match (`PO_POST_008` / `PO_POST_009`) และ traceability เต็มห่วงโซ่ตั้งแต่ requisition ผ่าน commitment, การรับสินค้า และ payable Auditor **ไม่มีพื้นผิวการเขียน (write surface)** ในโมดูล: ไม่สามารถ approve, transmit, void, close, แก้บรรทัด หรือเปลี่ยนสถานะ PO ได้ **System Administrator (ผู้ดูแลระบบ)** เป็น persona เชิงการตั้งค่าที่ดูแล **พื้นผิวนโยบายและ integration** ของโมดูล — รูปแบบ PO numbering ที่ขับ `tb_purchase_order.po_no`, นิยาม workflow ที่อ้างโดย `tb_purchase_order.workflow_id` (stage, `stage_role` และ membership ของ `user_action.execute[]` ที่ควบคุมทุก transition ตาม `PO_AUTH_011`), map RBAC ของ role-to-permission สำหรับ `PO_AUTH_001`–`PO_AUTH_011`, การเชื่อม integration ไปยัง vendor, vendor-pricelist (snapshot เมื่อแปลง PR-to-PO), budget (soft-commit เมื่อ submit ตาม `PO_POST_002`) และ inventory (เพิ่ม on-hand ผ่าน GRN ตาม `PO_XMOD_003`), กฎการแปลง PR-to-PO และการ group ตาม vendor+currency, document template ที่ใช้ตอน transmit (`PO_POST_004`) และแหล่งอัตรา tax / currency / FX ที่ป้อนให้ `PO_CALC_001`–`PO_CALC_011` ทั้งสองบทบาทไม่ได้อยู่ใน happy path ของ create-to-close; แต่ละบทบาทมีจุดเริ่มต้น, พื้นผิว และ exit เป็นของตัวเอง: Auditor exit ด้วย report ที่ถูกสร้างขึ้นโดยไม่เปลี่ยนสถานะ PO ส่วน Sysadmin exit ด้วยการตั้งค่าที่ถูกบันทึกซึ่งมีผลกับ PO ใหม่ในอนาคต โดยรักษา snapshot semantics ของ PO ที่อยู่ใน `in_progress` หรือเลยไปแล้ว ทั้งคู่ถูกบันทึกในไฟล์เดียวกันบนแกน persona นี้เพราะทั้งคู่อยู่นอกเส้นทางหลักของรายการและใช้ pattern เดียวกันคือ "นอกเส้นทาง เน้นการกำกับดูแล"

## 2. จุดเริ่มต้นและเส้นทางหลัก

ทั้งสองบทบาทมีจุดเริ่มต้นและเส้นทางแยกกัน อธิบายเป็นหัวข้อย่อยด้านล่าง

### 2.1. เส้นทาง Auditor

**จุดเริ่มต้น:** Sidebar → workspace **Audit** → **Procurement Activity Queries** (หรือเมื่อเริ่มจากเอกสารที่รู้แล้ว: Sidebar → โมดูล **Purchase Order** → เปิด PO → tab **Activity Log** / tab **Related Documents**) Auditor เริ่มที่หน้า query builder ที่ scope อยู่กับ document family ของการจัดซื้อ (PR / PO / GRN / invoice) ไม่ใช่ที่คิว My POs หรือ review queue ที่ persona เชิงรายการใช้

**เส้นทางหลัก (happy path — Auditor):**

1. จาก **Audit → Procurement Activity Queries** เลือก template ของ audit query (เช่น "All POs voided in period", "All POs transmitted by buyer", "Three-way-match exceptions", "Segregation-of-duties violations", "All status transitions for a PO", "PR-to-PO conversions in period") หรือสร้าง query แบบเฉพาะกิจกับ `workflow_history`, `tb_purchase_order_comment`, snapshot ของ header / detail และ bridge ระหว่างเอกสาร
2. ใส่ **filter**: ช่วงวัน (`order_date`, `approval_date`, `last_action_at_date` หรือ `created_at`), business unit, buyer, vendor, สกุลเงิน, ค่า `po_status`, `po_type` (`manual` เทียบกับ `purchase_request`) และ band ของ `total_amount` รวมถึง flag การฟัง high-value threshold filter chip ปรากฏเหนือตารางผลลัพธ์; filter ว่างเปล่าจะถูกปฏิเสธเพื่อกัน scan ที่ไม่มีขอบเขตตลอดทั้งห่วงโซ่
3. ตรวจ **ผลลัพธ์ที่ได้**: แต่ละแถวคือหนึ่ง PO (หรือหนึ่ง event ขึ้นกับลักษณะ query) พร้อม `po_no`, vendor, buyer, สถานะปัจจุบัน, action ล่าสุด, ผู้ดำเนินการล่าสุด และข้อเท็จจริงเชิง audit ที่เกี่ยวข้อง (เช่น เหตุผล void, จำนวน GRN, ผลของ invoice match) เรียงตามคอลัมน์ใด ๆ; คลิกที่แถวเพื่อ drill เข้า **ห่วงโซ่ทั้งหมด** ของ PO นั้น
4. ในหน้าผลลัพธ์ drill-down เดิน **forward chain** — จาก timeline ของ `po_status` บน PO ผ่านการ post GRN ที่เชื่อมโยง (`PO_POST_006` / `PO_POST_007` ขับโดย [[good-receive-note]]) ไปยัง record ของ three-way match บน vendor invoice (`PO_POST_008`) และการ post AP liability — และ **back chain** — จาก header ของ PO ผ่าน PR→PO bridge `tb_purchase_order_detail_tb_purchase_request_detail` ย้อนไปยัง Purchase Request ต้นทางและ trail การอนุมัติของมัน ยืนยันว่าห่วงโซ่ต่อเนื่อง (ไม่มีช่องว่าง, ไม่มี timestamp ที่ผิดลำดับ), segregation of duties ถูกเคารพ (`PO_AUTH_010`: `buyer_id` / `last_action_by_id` ณ transition `sent` ≠ ผู้ใช้ที่ post GRN ใด ๆ), ทุก transition มีทั้งผู้ดำเนินการและเหตุผลในกรณีที่บังคับ และการ group แบบ vendor+currency ของ PR→PO (carmen/docs § 2.3.2) ถูก apply อย่างถูกต้อง
5. หากพบความผิดปกติ (เช่น approval ที่บันทึกจากผู้ใช้นอก `user_action.execute[]` ของ stage ตาม `PO_AUTH_011`, void โดยไม่มี comment เหตุผลใน `tb_purchase_order_comment`, การ override three-way-match ที่ข้าม `PO_POST_009` หรือ pairing PO/GRN ของผู้ใช้เดียวกันที่ละเมิด `PO_AUTH_010`) ให้ **flag** PO ในแฟ้มคดี audit พร้อม note การ flag **ไม่** เปลี่ยนแปลง PO — มันเขียนเฉพาะใน store ฝั่ง audit เท่านั้น
6. **Export report** เป็น CSV / PDF สำหรับช่วงเวลาที่กำหนดหรือสำหรับแฟ้มคดี การ export ฟิลด์ที่อ่อนไหว (เช่น snapshot ของ pricing vendor, ตัวตนของ buyer / approver, ข้อความ justification เต็ม, payload ของไฟล์แนบ) ต้องการการอนุมัติชั้นที่สองตามนโยบาย data-export — Auditor ส่งคำขอ export และผู้อนุมัติการ export ปล่อยให้ดำเนินการ report ที่ export ออกมาและบันทึกการอนุมัติเองก็เป็นวัตถุ audit

### 2.2. เส้นทาง System Administrator

**จุดเริ่มต้น:** Sidebar → workspace **Configuration** → **PO Numbering & Templates** (สำหรับรูปแบบ `po_no` และ document template), **PO Workflow Settings** (สำหรับ stage, high-value threshold, การกำหนด `user_action.execute[]` ตาม `PO_AUTH_011`), **RBAC & Roles** (สำหรับ map role-to-permission ของ `PO_AUTH_001`–`PO_AUTH_011`), **Integration Settings** (vendor, vendor-pricelist, budget, inventory) หรือ **PR-to-PO Rules** (สำหรับการ group ตาม vendor+currency และ default ของการแปลง) แต่ละพื้นผิวเป็นหน้าแยกภายใต้ workspace เดียวกัน

**เส้นทางหลัก (happy path — Sysadmin, การเปลี่ยนการตั้งค่า):**

1. **ระบุความต้องการการตั้งค่า** ทริกเกอร์มาจากภายนอก — เช่น Finance ขอ high-value threshold ใหม่, การปรับโครงสร้างแผนกเปลี่ยนผู้รับผิดชอบ stage อนุมัติขั้นสุดท้าย, integration vendor-portal ใหม่ go live, รูปแบบ `po_no` ต้องเปลี่ยน prefix ปี, กฎ group PR-to-PO ต้องเพิ่มมิติ delivery location หรือ template การ transmit ต้องเพิ่ม block ทะเบียนภาษี เปิด ticket การเปลี่ยนแปลงและแนบเอกสารอ้างอิงนโยบาย (memo / approval) ก่อนเปิดพื้นผิวการตั้งค่า
2. **เปิดหน้าการตั้งค่าที่เกี่ยวข้อง** สำหรับ numbering: **Configuration → PO Numbering & Templates** → เลือกแถว scheme → แก้ prefix / sequence / กฎ reset สำหรับ workflow / threshold / RBAC: **Configuration → PO Workflow Settings** → เลือกแถว workflow (รายต่อ business unit / รายต่อ `po_type`) → เปิด stage editor สำหรับ integration: **Configuration → Integration Settings** → เลือก integration (vendor / pricelist / budget / inventory) → แก้ endpoint, credential, sync window สำหรับ conversion / grouping: **Configuration → PR-to-PO Rules**
3. **ปรับการตั้งค่า** ใน editor แบบ staged: เปลี่ยน scheme numbering, เพิ่ม / ลบ / จัดลำดับ workflow stage, แก้ high-value threshold ที่ทริกเกอร์ gate อนุมัติของ Procurement Manager ตาม `PO_AUTH_004`, assign หรือถอนผู้ใช้ออกจาก `user_action.execute[]` ตาม `PO_AUTH_011`, ปรับชุดสิทธิ์ RBAC สำหรับ `PO_AUTH_001`–`PO_AUTH_011` ใด ๆ, wire หรือ rewire endpoint ของ integration หรือเปลี่ยนกฎการ group แบบ vendor+currency (carmen/docs § 2.3.2) ทุกการแก้ไขสะสมอยู่ใน draft แบบ pending; ไม่มีอะไร persist จนกว่าจะ Save
4. **Preview ผลกระทบ** หน้าการตั้งค่าแสดง side-panel สรุป: จำนวน PO ที่กำลังเดินซึ่งจะคงเดินภายใต้ snapshot การตั้งค่าเดิม, จำนวน PO ใหม่ที่จะใช้กฎใหม่ (ประมาณการจากอัตราการสร้างล่าสุด), stage หรือกฎที่เปลี่ยน และผู้ใช้ที่ถูกเพิ่ม / ถอนจาก `user_action.execute[]` สำหรับการเปลี่ยน threshold panel แสดงการเลื่อน band และจำนวน PO ล่าสุดที่จะ route ต่างจากเดิม สำหรับการเปลี่ยน numbering panel แสดง preview `po_no` ตัวต่อไปภายใต้ scheme ใหม่ สำหรับการเปลี่ยน integration panel รัน probe ทดสอบการเชื่อมต่อ Sysadmin สามารถแก้หรือทิ้ง draft ในขั้นตอนนี้
5. **บันทึกการตั้งค่า** ระบบเขียนการตั้งค่าใหม่พร้อม timestamp `effective_from`, บันทึกการเปลี่ยนแปลงใน configuration audit log ฝั่งระบบ (แยกจาก `tb_purchase_order_comment`) และ notify ประชากรผู้ใช้ที่ได้รับผลกระทบ (เช่น approver ที่เพิ่งถูกเพิ่ม, buyer ที่พฤติกรรม group จะเปลี่ยน) PO ที่อยู่ใน `in_progress`, `sent` หรือ `partial` อยู่แล้วจะคง **snapshot** ของการตั้งค่าเดิม — ห่วงโซ่ stage ของ workflow, band ของ threshold, รูปแบบ numbering, endpoint integration ที่ใช้ตอน submit และ context ภาษี / สกุลเงิน — ตาม snapshot semantics เดียวกับที่ปกป้อง snapshot ราคาตอนแปลง PR-to-PO PO ใหม่ที่สร้างหลัง `effective_from` ใช้การตั้งค่าใหม่
6. **ตรวจสอบการเปิดใช้** เลือก PO test ตัวแทน (หรือจำลองในสภาพแวดล้อม non-production) แล้วยืนยันว่าพฤติกรรมใหม่ทำงานตามที่คาด: numbering, การ route, gate ของ threshold, การ sync integration และผลของการ group หากพบ regression ให้ rollback โดยเปิดการตั้งค่ากลับและย้อนไปยังเวอร์ชันก่อนหน้า (ทุกเวอร์ชันที่บันทึกถูกเก็บใน configuration audit log)
7. **ปิด ticket การเปลี่ยนแปลง** พร้อม link ไปยัง configuration audit log จากจุดนี้การเปลี่ยนแปลงมีผลกับ PO ใหม่; บทบาทของ Sysadmin จบจนกว่าจะมีการเปลี่ยนการตั้งค่าครั้งต่อไป

## 3. สาขาการตัดสินใจ

- **ถ้า Auditor พบการละเมิดนโยบาย** (เช่น approval โดยผู้ใช้ที่ไม่ได้อยู่ใน `user_action.execute[]` ตาม `PO_AUTH_011`, void โดยไม่มีเหตุผลบังคับใน `tb_purchase_order_comment`, การละเมิด segregation of duties ของ `PO_AUTH_010` ที่ buyer ก็เป็นผู้ post GRN หรือ override three-way-match ที่ข้าม `PO_POST_009`): Auditor **ไม่สามารถ act กับ PO ในโมดูลได้** (read-only) ต้อง escalate ผ่านแฟ้มคดี audit — flag PO, แนบหลักฐาน (ส่วนคัดจาก activity log, timeline ของห่วงโซ่, diff ของเวอร์ชันการตั้งค่า, pairing ของ segregation of duties) แล้วส่งคดีให้ business owner ที่รับผิดชอบ (Finance, Compliance, Procurement Manager หรือ department head) เพื่อแก้ไขนอกระบบ หากการแก้ไขต้องการ action ระดับระบบ (เช่น void เพื่อยุติ PO ที่ไม่เป็นไปตามนโยบายภายใต้ `PO_AUTH_007` หรือการเริ่ม credit note ฝั่ง AP) action นั้นถูกดำเนินการโดย Procurement Manager หรือ System Administrator ภายใต้สิทธิ์ของแต่ละบทบาท ไม่ใช่โดย Auditor
- **ถ้า Sysadmin พยายามเปลี่ยนการตั้งค่าขณะที่ PO ที่กำลังเดินอยู่ขึ้นกับกฎปัจจุบัน** (เช่น เปลี่ยน high-value threshold ขณะที่ PO กำลังอยู่ที่ `in_progress` ภายใต้การ route ของ threshold เดิม หรือเปลี่ยนกฎ grouping ขณะที่ wizard แปลง PR-to-PO เดินอยู่กลางทาง): การบันทึกได้รับอนุญาต (กฎไม่ถูก lock จาก PO ที่กำลังเดิน) แต่การเปลี่ยนแปลง **ไม่** retroactive กับ PO ที่กำลังเดิน หน้าการตั้งค่าแสดงจำนวน PO ที่ `in_progress` / `sent` / `partial` ซึ่งได้รับผลกระทบใน panel preview; PO เหล่านั้นเดินต่อภายใต้ snapshot ของตน (ห่วงโซ่ stage ของ workflow, band ของ threshold, endpoint integration และ context ภาษี / สกุลเงิน ที่ถ่ายตอน submit) PR ที่อยู่ระหว่างการแปลงในขณะ save ทำการแปลงต่อภายใต้กฎที่ใช้ตอนเริ่มการแปลง; การแปลงครั้งถัดไปเริ่มภายใต้กฎใหม่
- **ถ้า Sysadmin เปิดใช้ delegation บน stage อนุมัติ** (เช่น Procurement Manager ไม่อยู่และมีผู้ได้รับมอบหมายให้ได้สิทธิ์อนุมัติชั่วคราว): ผู้ได้รับมอบหมายสืบทอด membership ใน `user_action.execute[]` ตาม scope ของ stage ในช่วงเวลาที่กำหนด notification ของ PO ใด ๆ ที่กำลังอยู่ที่ stage นั้นจะถูก re-fan ไปยังผู้ได้รับ เมื่อช่วงเวลาหมดอายุ สิทธิ์ที่สืบทอดจะหลุดอัตโนมัติ; PO ที่ยังอยู่ที่ stage นั้นเดินต่อด้วย `user_action.execute[]` ของผู้ใช้ต้นเดิม (ซึ่งไม่เคยเปลี่ยน) — ไม่ต้อง re-route PO
- **ถ้า Auditor ขอ export ที่รวมฟิลด์อ่อนไหว** (ตัวตนเต็มของ buyer / approver, snapshot pricelist ของ vendor, รายละเอียดความเบี่ยงเบนของ three-way match, payload ไฟล์แนบ): การ export เข้าสู่สถานะ **pending** และต้องได้รับการอนุมัติจากผู้อนุมัติ data-export ตามนโยบาย Auditor ไม่สามารถข้ามขั้นตอนนี้ ขณะที่ pending การ export มองไม่เห็นนอกแฟ้มคดี audit; เมื่ออนุมัติ การ export ถูก materialise และ link ดาวน์โหลดถูกบันทึกในแฟ้มคดีพร้อมตัวตนของผู้อนุมัติ
- **ถ้า Sysadmin เปลี่ยนกฎการแปลง PR-to-PO / กฎ grouping** (เช่น เพิ่มมิติ delivery location ให้กับ key ของ vendor+currency grouping ตาม carmen/docs § 2.3.2) ขณะที่ wizard แปลง PR-to-PO เดินอยู่: การแปลงที่กำลังเดินอยู่ทำต่อภายใต้ **กฎที่ใช้ตอนเริ่ม wizard** (group preview ที่ผู้ใช้ confirm แล้วจะไม่ re-shuffle ตอน save) การแปลงครั้งถัดไปที่เริ่มหลัง `effective_from` ใช้กฎใหม่ configuration audit log บันทึกการเปลี่ยนแปลงและจำนวนการแปลงที่กำลังเดิน ณ ตอน save เพื่อให้ความผิดปกติที่อาจเกิดขึ้นทันทีหลังจากนั้นโยงกลับมาที่การเปลี่ยนกฎได้ ไม่ใช่ bug ของการ route
- **ถ้าการเปลี่ยนแปลงของ Sysadmin สร้าง workflow deadlock** (เช่น การตั้งค่าถอนผู้ใช้ออกจาก `user_action.execute[]` ของ stage ที่ผู้ใช้นั้นเป็นคนเดียวที่ assign อยู่ และ PO กำลังรออยู่ที่ stage นั้นโดยไม่มี approver อื่น): panel preview จะ flag การ deadlock ในขั้นตอนที่ 4 หาก Sysadmin ยังบันทึก PO ที่ได้รับผลกระทบจะ timeout ที่ stage นั้นและต้อง intervention ด้วยมือ — โดยปกติคือ delegation ครั้งเดียว หรือ void ที่เริ่มโดย Procurement Manager / Sysadmin ภายใต้ `PO_AUTH_007` — เพื่อ unblock configuration audit log บันทึกการเปลี่ยนแปลงและ warning ของ deadlock ทำให้ audit trail ชัดเจนว่าเป็นเพราะอะไร

## 4. จุดสิ้นสุด / การส่งต่อ

แกน persona Audit / Config exit ในรูปแบบใดรูปแบบหนึ่งต่อไปนี้ โดยขึ้นกับว่าบทบาทใด act:

- **Auditor — report ถูกสร้าง** ผลของ query, drill-down ของห่วงโซ่ หรือแฟ้มคดีถูก materialise (ตรวจบนหน้าจอหรือ export เป็น CSV / PDF หลังขั้นตอนการอนุมัติการ export) **ไม่มีการเปลี่ยนสถานะ PO**: `po_status`, `workflow_current_stage`, `workflow_history`, `tb_purchase_order_comment` และทุก snapshot บนเอกสารคงสภาพเดิมตามก่อนที่ Auditor จะเปิดหน้า การส่งต่อของ Auditor เป็น **out-of-band** ไปยัง business owner ใดก็ตาม (Finance, Compliance, Procurement Manager หรือ department head) ที่รับผิดชอบการแก้ไขที่ audit เปิดเผย หากแฟ้มคดีของ Auditor แนะนำให้ void หรือ close, Procurement Manager ดำเนินการภายใต้ `PO_AUTH_007` / `PO_POST_010` (void) หรือ `PO_AUTH_008` / `PO_POST_011` (close-early); Sysadmin act เฉพาะเมื่อการแก้ไขเป็นการเปลี่ยนการตั้งค่า ไม่ใช่การเปลี่ยนสถานะรายเอกสาร
- **Auditor — แฟ้มคดีปิดโดยไม่ต้อง action** เมื่อ query / drill-down ของห่วงโซ่ไม่พบความผิดปกติ Auditor ปิดแฟ้มคดีพร้อม note "no findings" ไม่มีการเปลี่ยนสถานะ PO; แฟ้มคดีเองถูกเก็บไว้เป็นหลักฐานว่าช่วงเวลา / scope ถูกตรวจสอบ end-to-end ตลอดทั้งห่วงโซ่ PR / PO / GRN / invoice
- **Sysadmin — การตั้งค่าถูกบันทึก** เวอร์ชันใหม่ของการตั้งค่า (scheme numbering, stage และ threshold ของ workflow, map สิทธิ์ RBAC, endpoint integration, กฎการแปลง PR-to-PO / กฎ grouping หรือ document template) ถูกเขียนพร้อม timestamp `effective_from` และบันทึกใน configuration audit log **PO ที่สร้างหลัง `effective_from`** ใช้การตั้งค่าใหม่; **PO ที่อยู่ใน `in_progress`, `sent`, `partial` หรือสถานะ non-terminal ใด ๆ แล้ว** คงการตั้งค่าตาม snapshot — รวมถึงห่วงโซ่ stage, band ของ threshold, การ wire integration ตอน submit และ context ภาษี / สกุลเงิน notification ไปยังประชากรผู้ใช้ที่ได้รับผลกระทบยิงเมื่อบันทึก การส่งต่อเป็น **forward in time** — Purchaser คนถัดไปที่สร้าง PO เห็นพฤติกรรมใหม่อัตโนมัติ
- **Sysadmin — การตั้งค่าถูก rollback** หากการตรวจสอบในขั้นตอนที่ 6 ของเส้นทางหลักพบ regression Sysadmin ย้อนไปยังเวอร์ชันก่อนหน้าของการตั้งค่า การ rollback ก็เป็นการบันทึกการตั้งค่าด้วย `effective_from` ของตัวเอง; PO ที่สร้างระหว่างการเปลี่ยนแปลงเดิมและการ rollback ไม่ถูก re-evaluate ย้อนหลัง (snapshot semantics) แต่ PO ใหม่ที่สร้างหลัง rollback ใช้การตั้งค่าที่ย้อนกลับมา configuration audit log บันทึกทั้งการเปลี่ยนแปลงไปข้างหน้าและการ rollback รักษา trail ที่สะอาดไว้
- **Sysadmin — integration cutover** เมื่อการเปลี่ยนแปลงเป็นการ rewire integration (เช่น endpoint vendor-portal ใหม่, base URL ของระบบ budget ใหม่ หรือ sync window ของ pricelist ใหม่) การ save จะทริกเกอร์ลำดับ probe-and-cutover เพิ่ม — endpoint เดิมถูก drain, endpoint ใหม่รับ round-trip จำลอง และเฉพาะเมื่อสำเร็จเท่านั้นการตั้งค่าจึง go live หาก probe ล้มเหลว การบันทึกถูก hold และ endpoint เดิมยังคงมีผล; Sysadmin ถูก notify เพื่อแก้ไขก่อนลองใหม่

สถานะเอกสารทุกแบบของ exit ใน Audit / Config ถูกกำกับโดย `enum_purchase_order_doc_status = { draft, in_progress, voided, sent, partial, closed, completed }` flow ของ Auditor ไม่เคยทำให้ PO ขยับใน enum นี้; flow การตั้งค่าของ Sysadmin ก็ไม่ขยับ PO ใน enum นี้เช่นกัน (มีผลกับ PO ในอนาคตเท่านั้น) การเปลี่ยนสถานะ PO เพียงอย่างเดียวที่ทริกเกอร์โดย escalation จาก audit finding คือ **void** (Procurement Manager ภายใต้ `PO_AUTH_007` / `PO_POST_010`) และ **early close** (Procurement Manager / Inventory Manager ภายใต้ `PO_AUTH_008` / `PO_POST_011`); สิ่งเหล่านี้เป็นการดำเนินการแบบ exceptional ที่ทริกเกอร์โดย audit และทำโดย persona เชิงรายการที่เหมาะสม ไม่ใช่โดยแกน Audit / Config เอง
