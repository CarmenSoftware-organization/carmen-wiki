# ใบสั่งซื้อ — กรณีทดสอบ — ฝ่ายการเงิน (Finance)

หน้านี้รวบรวมกรณีทดสอบที่ persona ของฝ่ายการเงิน (**Finance Officer / Accounts Payable** ผู้รัน three-way match รายวันและ post AP liability บวก **Finance Manager** ผู้ใช้สิทธิ์ sign-off ทางการเงินก่อนส่ง PO สำหรับ PO มูลค่าสูงหรืออ่อนไหวด้าน FX) เป็นผู้ขับเคลื่อนโดยตรงในโมดูล `purchase-order` ฝ่ายการเงินมีจุดสัมผัสสองจุดที่แตกต่างกันบนวงจรชีวิตของ PO: **การ review ก่อนส่ง** ขณะที่ PO ยังอยู่ที่ `po_status = in_progress` (Finance Manager sign off สกุลเงิน, exchange rate, รหัสภาษี และยอดรวม ก่อนที่ `PO_POST_004` จะส่ง PO ไปยังผู้ขาย) และ **three-way match หลังการรับสินค้า** หลังการ post GRN (Finance Officer รับใบแจ้งหนี้ผู้ขาย ค้นหา PO และ GRN ที่ตรงกัน และรัน match ภายใต้ `PO_POST_008` / `PO_POST_009`) three-way match คือกิจกรรมหลักของฝ่ายการเงิน — เมื่อสำเร็จ AP เคลียร์ GRN accrual และ post AP liability ของผู้ขาย; เมื่อล้มเหลว ใบแจ้งหนี้ถูก hold ใน dispute และความไม่ตรงกันถูก flag กลับไปยัง Purchaser เพื่อแก้ไข PO เอง **ไม่** ถูก transition โดย three-way match (`PO_POST_008` ระบุชัดในเรื่องนี้) — ผลลัพธ์ match อยู่บน record ใบแจ้งหนี้ และ PO คงสถานะการรับสินค้าที่ไปถึง (`partial`, `completed` หรือ `closed`) กรณีทดสอบถูกจัดเป็น happy path ตามที่อธิบายใน [03-user-flow-finance.md](./03-user-flow-finance.md), ขอบเขต RBAC ที่บังคับโดย `PO_AUTH_009` (ขอบเขต Finance Officer — read-only บน PO + match + post AP ฝั่ง AP) และสิทธิ์ approver ก่อนส่งของ Finance Manager ภายใต้ `PO_AUTH_011` (stage-gated approval), validation rule รอบ ๆ tolerance ของ qty / price, GRN ที่หายไป, FX, period ที่ปิด และ guard ใบแจ้งหนี้ซ้ำ และชุด edge case จำนวนเล็ก ๆ เกี่ยวกับขอบ tolerance, กรณี FX no-op, multi-invoice match, concurrency และความละเอียดทศนิยมบนยอด AP เส้นทาง cross-persona ที่หมุนรอบฝ่ายการเงิน (`X-PO-05` send-back ก่อนส่ง, `X-PO-07` bounce-back dispute ของ three-way match, `X-PO-09` close-out ของการ post AP) อยู่ในหน้าภาพรวมพ่อ ไม่ได้อยู่ในหน้านี้

## 1. Happy Path

| - | ---- | ------------ | ------- | ------------ |
| FIN-HP-01 | Finance Manager review ก่อนส่ง sign off PO มูลค่าสูง | PO `PO-HV` ที่ `po_status = in_progress` ด้วย `total_amount = ฿1,250,000` (เกินเกณฑ์ high-value ของ tenant); workflow cursor อยู่ที่ stage Finance review; Finance Manager `finance.manager@blueledgers.com` อยู่ใน `user_action.execute` ตาม `PO_AUTH_011`; สกุลเงินตามสัญญา, FX rate, รหัสภาษี และ credit term ของผู้ขายทั้งหมดสอดคล้องกับ vendor master | 1. เปิด `PO-HV` จาก review queue → tab **Financial Details** 2. verify `currency_id`, `exchange_rate` เทียบกับ FX policy ของ tenant, payment terms, flag prepayment 3. verify `tax_id` / อัตราภาษีรายบรรทัดเทียบกับ tax profile ของสินค้า + การขึ้นทะเบียนภาษีของผู้ขาย; รวม `Σ line.total_price` เทียบกับ header `total_amount` ตาม `PO_CALC_008`–`PO_CALC_011` 4. กด **Sign off** | บันทึก stage approval ตาม `PO_POST_003`; ผนวก `workflow_history`; `workflow_current_stage` ขยับไป stage ถัดไป (หรือไปการอนุมัติขั้นสุดท้ายโดยตรง) เมื่ออนุมัติขั้นสุดท้าย `PO_POST_004` ทำงาน: `po_status = in_progress → sent`, `approval_date = now()`, `last_action = approved`; การส่งไปยังผู้ขายถูก trigger ผ่าน email/transmit layer entry การ sign off ของ Finance Manager ถูกบันทึกใน `tb_purchase_order_comment` (type `system`) |

| FIN-HP-04 | Price อยู่ใน tolerance — auto-pass พร้อม entry PPV | PO `PO-T` ที่ `po_status = completed` (qty 100 @ PO `unit_price = ฿50.00`); ใบแจ้งหนี้ผู้ขาย `INV-T1` สำหรับ qty 100 @ ฿50.40; tenant price tolerance = `1.5%`; สกุลเงินใบแจ้งหนี้ตรงกับ PO | 1. capture `INV-T1` 2. index เทียบกับ `PO-T` 3. รัน three-way match | ส่วนต่าง price `(50.40 − 50.00) / 50.00 = 0.8%` อยู่ใน band `1.5%` Qty match `PO_POST_008` ทำงาน: AP post ที่ราคาใบแจ้งหนี้ (`฿50.40`); variance `(50.40 − 50.00) × 100 = ฿40.00` ถูก capture เป็น entry **purchase price variance (PPV)** บนการ post AP (debit / credit บัญชี PPV ตาม GL mapping ของ tenant); GRN accrual เคลียร์ที่ valuation ที่ GRN บันทึก ใบแจ้งหนี้ approved for payment PO `unit_price` ยังเป็นราคาตามสัญญา; ไม่มีการสร้าง PO amendment |
| FIN-HP-05 | สกุลเงินไม่ตรงกันภายใน FX tolerance ของ tenant — post FX adjustment | PO `PO-F` ที่ `po_status = completed` ด้วย `currency_id = USD`, `exchange_rate = 36.50` (เทียบกับสกุลฐาน `THB`); ใบแจ้งหนี้ผู้ขาย `INV-F1` รับเป็น `EUR` ตามสัญญา dual-currency; นโยบาย tenant อนุญาต `EUR ↔ USD` สำหรับ vendor นี้; rate `EUR → USD` วันใบแจ้งหนี้ = `1.08` และ rate `USD → THB` วันใบแจ้งหนี้ = `36.65` | 1. capture `INV-F1` เป็น `EUR` 2. index เทียบกับ `PO-F` 3. ระบบยอมรับ vendor แบบ dual-currency; ใช้ FX cross-rate วันใบแจ้งหนี้ `EUR → USD → THB` 4. รัน three-way match 5. confirm การ post AP | Match รันบน qty + (price ที่ปรับ FX แล้ว) ภายใน tolerance `PO_POST_008` ทำงาน: AP post ใบแจ้งหนี้เป็น `EUR` (สกุลธุรกรรมบนใบแจ้งหนี้) พร้อม **entry FX adjustment** เทียบกับสกุลตามสัญญาของ PO (`USD`); มูลค่าสกุลฐาน `THB` คำนวณที่ FX cross-rate `EUR → THB` วันใบแจ้งหนี้ ไม่ใช่ `exchange_rate = 36.50` ที่ค้างของ PO ส่วนต่างระหว่างมูลค่า FX snapshot ของ PO กับมูลค่า FX วันใบแจ้งหนี้ถูก post เป็น entry **realised FX gain/loss** ตาม GL mapping ของ tenant ใบแจ้งหนี้ approved for payment PO ไม่เปลี่ยน |
| FIN-HP-06 | Post AP liability ด้วย GL account ที่ถูกต้องเมื่อ match สะอาด | PO `PO-G` ที่ `po_status = completed` (qty 50 @ ฿200.00, `tax_rate = 7%`, GRN post แล้ว); ใบแจ้งหนี้ผู้ขาย `INV-G1` ตรงกับ PO + GRN พอดี; GL mapping ของ tenant: GRN-accrual = `2110-GRN`, AP liability = `2100-AP-Trade`, input VAT = `1310-VAT-Input` | 1. capture `INV-G1` (`net = ฿10,000`, `tax = ฿700`, `total = ฿10,700`) 2. index เทียบกับ `PO-G` 3. รัน three-way match 4. confirm การ post AP | `PO_POST_008` ทำงานด้วย GL entry ต่อไปนี้บนการ post AP: **Dr** `2110-GRN` `฿10,000` (เคลียร์ GRN accrual), **Dr** `1310-VAT-Input` `฿700` (รับ input VAT), **Cr** `2100-AP-Trade` `฿10,700` (post vendor payable) Position inventory-receipt สุทธิเป็น 0 บน `2110-GRN`; vendor payable เพิ่ม `฿10,700`; input VAT รับคืนได้ ใบแจ้งหนี้ approved for payment ผนวก `tb_purchase_order_comment` ด้วย entry `system` ยืนยันการ post AP (ref ใบแจ้งหนี้ + ref batch GL) PO ไม่เปลี่ยน |

## 2. Permission / Authorization

| # | กรณี | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | ---- | ---------------------------------------- |
| FIN-PERM-01 | Finance Officer รัน three-way match บน PO ที่ `po_status ∈ {partial, completed, closed}` | **Allow** ตาม `PO_AUTH_009` (Finance Officer อ่านได้บน PO) และสิทธิ์ฝั่ง AP ในการ capture + match ใบแจ้งหนี้ อัลกอริทึม match ภายใต้ `PO_POST_008` ทำงาน; เมื่อสำเร็จ `PO_POST_008` post AP; เมื่อล้มเหลว `PO_POST_009` park ใบแจ้งหนี้ไว้ใน dispute record ของ PO ไม่ถูก mutate โดย Finance Officer (read-only) |
| FIN-PERM-02 | Finance Officer post AP liability เมื่อ three-way match สำเร็จ | **Allow** การ post AP เป็น action ของโมดูล AP ที่ขับโดย `PO_POST_008` (rule cross-module `PO_XMOD_007`) Finance Officer ถือสิทธิ์ post AP ฝั่ง AP; โมดูล PO ถูก consult (read-only) เพียงเพื่อข้อมูลอ้างอิงในการ match — `vendor_id`, `currency_id`, `unit_price` บรรทัด, การเชื่อม GRN GL entry ถูก write ลงบัญชี AP / inventory-clearing / VAT ตาม mapping ของ tenant |
| FIN-PERM-03 | Finance Officer พยายาม override match ที่ล้มเหลว (บังคับ post AP แม้มีความไม่ตรงกันด้าน qty / price) | **Deny — Manager-only** การ override match ที่ล้มเหลวต้องใช้ role Finance Manager (หรือ AP Manager ตามการตั้งค่า tenant) UI ของ Finance Officer ซ่อน / ปิด action **Override and Post** บน match ที่ล้มเหลว; ใบแจ้งหนี้ยังคงอยู่ใน dispute state ตาม `PO_POST_009` เรียก API ตรง ๆ ตอบกลับ `"Override of failed three-way match requires the Finance Manager role."` |
| FIN-PERM-04 | Finance Manager sign off PO มูลค่าสูงที่ stage review ก่อนส่ง | **Allow** ตาม `PO_AUTH_011` (stage-gated approval) เมื่อ user ปรากฏใน `tb_purchase_order.user_action.execute` ที่ `workflow_current_stage` ปัจจุบัน การ sign off บันทึก `PO_POST_003` (ภายใน `in_progress`) หรือบน stage สุดท้ายทำให้ `PO_POST_004` ทำงาน (`in_progress → sent`) ต่ำกว่าเกณฑ์ high-value workflow อาจข้าม stage Finance ทั้งหมด |
| FIN-PERM-05 | Finance Officer พยายาม post AP โดยไม่มี three-way match ที่สำเร็จ | **Deny** การ post AP ถูก gate บนความสำเร็จของ `PO_POST_008` — โมดูล AP reject การ post ตรง ๆ เทียบกับใบแจ้งหนี้ที่ยังไม่ match / match ล้มเหลว server ตอบกลับ `"Cannot post AP liability — three-way match has not succeeded on invoice <invoice_no>. Resolve qty / price discrepancy or escalate to the Finance Manager."` GL entry ไม่ถูก write; ใบแจ้งหนี้ยังอยู่ใน state `pending_match` หรือ `dispute` |
| FIN-PERM-06 | Finance Officer พยายามแก้ header ของ PO (vendor, สกุลเงิน, ราคา, qty) | **Deny — Purchaser only** ตาม `PO_AUTH_002` (ขอบเขต Purchaser) และ `PO_AUTH_010` (segregation of duties) Finance Officer ไม่มีสิทธิ์แก้บน record ของ PO เอง — เพียง read access ภายใต้ `PO_AUTH_009` หน้ารายละเอียด PO ในมุมมองของ Finance Officer เป็น read-only; หน้าจอโมดูล AP เป็นที่เดียวที่ Finance ทำงานได้ และหน้าจอเหล่านั้น mutate PO ไม่ได้ เรียก API แก้ตรง ๆ ตอบกลับ `"You are not authorised to edit this PO at the current stage."` |
| FIN-PERM-07 | Finance Manager ส่งกลับ PO ที่ stage review ก่อนส่ง | **Allow** ตาม `PO_POST_005` Finance Manager (หรือ approver ของ stage ใด ๆ) อาจ reject PO ที่ stage review พร้อมข้อความเหตุผล; `po_status = in_progress → draft`; reset `workflow_current_stage`; `last_action = rejected`; บันทึก comment rejection ใน `tb_purchase_order_comment` (type `system`); ownership กลับสู่ Purchaser เพื่อแก้ไขและส่งใหม่ |

## 3. Validation / Error

| # | กรณี | Trigger | Error ที่คาดหวัง |
| - | ---- | ------- | ---------------- |
| FIN-VAL-01 | Qty ไม่ตรงกันนอก tolerance — flag กลับไปยัง Purchaser | PO `PO-Q1` ที่ `po_status = completed` (qty 10 รับผ่าน GRN); ใบแจ้งหนี้ผู้ขาย `INV-Q1` สำหรับ qty 12 @ unit price ตรงกัน; tenant qty tolerance = `0%` | Reject match ตาม `PO_POST_009` server ตอบกลับ `"Three-way match failed — invoice qty (12) exceeds GRN accepted qty (10) outside tenant tolerance (0%)."` ใบแจ้งหนี้ hold ใน state `dispute`; การ post AP ถูกบล็อก; ผนวก comment `system` บน PO บันทึกความล้มเหลว (บรรทัด, มิติ = qty, ขนาด); เปิด deviation record ฝั่ง vendor / vendor-pricelist; Purchaser ถูกแจ้งผ่าน activity log เพื่อขอ credit note (over-invoicing) หรือ supplementary GRN (under-receipt) |
| FIN-VAL-02 | Price ไม่ตรงกันนอก tolerance — flag กลับไปยัง Purchaser | PO `PO-P1` ที่ `po_status = completed` (qty 100 @ PO `unit_price = ฿50.00`); ใบแจ้งหนี้ผู้ขาย `INV-P1` สำหรับ qty 100 @ `฿55.00`; tenant price tolerance = `1.5%` (variance จริง = `10%`) | Reject match ตาม `PO_POST_009` server ตอบกลับ `"Three-way match failed — invoice unit price (฿55.00) is outside the price-tolerance band (±1.5% of PO unit price ฿50.00)."` ใบแจ้งหนี้ hold ใน dispute; การ post AP ถูกบล็อก; ผนวก comment `system` บน PO + deviation บน vendor pricelist; Purchaser ขอ credit note (over-billing) หรือ PO amendment ภายใต้ `PO_VAL_016` (การเปลี่ยน price หลัง `sent` ถูกจำกัด) |
| FIN-VAL-03 | GRN หายไป — match ไม่ได้หากไม่มี | PO `PO-N1` ที่ `po_status = sent` (ยังไม่มี GRN post); ใบแจ้งหนี้ผู้ขาย `INV-N1` รับเข้ามา | Reject match server ตอบกลับ `"Three-way match cannot run — no GRN posted against PO <po_no>. Confirm receipt with the Receiver / Purchaser before re-presenting the invoice."` ใบแจ้งหนี้ park ใน `pending_match` (ไม่ใช่ dispute); การ post AP ถูกบล็อก; ถ้า vendor ส่งของแล้วแต่ GRN ยังไม่ post ให้แจ้ง Receiver / Purchaser ตาม GRN — เมื่อ post แล้ว match รันใหม่อัตโนมัติ ถ้า vendor ออกใบแจ้งหนี้ล่วงหน้าก่อนการส่ง ใบแจ้งหนี้ถูกส่งกลับให้ vendor พร้อม notice ไม่รับของ |
| FIN-VAL-04 | ใบแจ้งหนี้สำหรับ PO ที่ voided — reject | PO `PO-V1` ที่ `po_status = voided` (void โดย Procurement Manager ตาม `PO_AUTH_007`); ใบแจ้งหนี้ผู้ขาย `INV-V1` เข้ามาอ้างอิง `po_no` ที่ void แล้ว | Reject match server ตอบกลับ `"PO <po_no> is at status voided — no invoice can be matched against this PO."` ใบแจ้งหนี้ถูก flag ส่งกลับ vendor; การ post AP ถูกบล็อก; activity log บันทึกความพยายาม; เส้นทางแก้คือการแก้ vendor (vendor ต้องถอนใบแจ้งหนี้) — ไม่มี action ฝั่ง PO เนื่องจาก `voided` เป็น terminal ตาม `PO_POST_010` |
| FIN-VAL-05 | สกุลเงินไม่ตรงกันเกิน FX tolerance — reject | PO `PO-C1` ที่ `po_status = completed` ด้วย `currency_id = USD`; ใบแจ้งหนี้ผู้ขาย `INV-C1` รับเป็น `EUR` แต่นโยบาย tenant ไม่อนุญาตการออกใบแจ้งหนี้ `EUR` สำหรับ vendor นี้ (สัญญาสกุลเดียวตาม `PO_VAL_013`) | Reject match server ตอบกลับ `"Invoice currency (EUR) does not match PO currency (USD); dual-currency invoicing is not permitted for this vendor per tenant policy."` ใบแจ้งหนี้ hold ใน dispute; การ post AP ถูกบล็อก; ผนวก comment `system` บน PO; Purchaser ถูก flag ให้ติดต่อ vendor เพื่อออกใบแจ้งหนี้ใหม่เป็น `USD` แตกต่างจาก FIN-HP-05 ซึ่ง dual-currency ได้รับอนุญาต |
| FIN-VAL-06 | พยายาม match ใบแจ้งหนี้ซ้ำ — reject | PO `PO-D1` match + post AP กับใบแจ้งหนี้ `INV-D1` แล้ว (state = `approved for payment`); ใบแจ้งหนี้ที่สอง `INV-D2` ที่มีเลขที่ใบแจ้งหนี้ของ vendor เดียวกันถูก capture เทียบกับ PO เดียวกัน | Reject ที่ขั้น capture AP (ก่อน match รัน) server ตอบกลับ `"Duplicate invoice — vendor invoice number <invoice_no> from <vendor> has already been captured and matched against PO <po_no>."` ใบแจ้งหนี้ที่สองไม่ถูก write; การ post AP ถูกบล็อก; activity log บันทึกความพยายาม; แจ้ง Purchaser / vendor เพื่อ re-issue ถ้าใบแจ้งหนี้ที่สองเป็น amendment ที่ชอบด้วยกฎหมาย (ในกรณีนั้น vendor ต้องใช้เลขที่ใบแจ้งหนี้ที่ต่างกัน โดยทั่วไป suffix `-CN` หรือ `-AM`) |
| FIN-VAL-07 | พยายาม post AP เทียบกับ period บัญชีที่ปิด — reject | PO `PO-CP` ที่ `po_status = completed`; ใบแจ้งหนี้ผู้ขาย `INV-CP` match สะอาด; period GL ปัจจุบันสำหรับวันที่ใบแจ้งหนี้ (`2026-04-15`) **ปิด** (ใช้ cutoff period-close) | Reject ที่ขั้น post AP server ตอบกลับ `"Cannot post AP — accounting period <yyyy-mm> is closed. Either re-date the invoice to the open period or request period re-open from the Finance Manager."` GL entry ไม่ถูก write; ใบแจ้งหนี้ hold ที่ state `pending_post`; เส้นทางแก้คือ Finance Manager re-open period (audit-logged) หรือ re-date ใบแจ้งหนี้ด้วยความเห็นชอบของ vendor PO ไม่เปลี่ยน |

## 4. Edge Cases

| # | กรณี | เงื่อนไข | ผลที่คาดหวัง |
| - | ---- | -------- | ------------ |
| FIN-EDGE-01 | ขอบ tolerance พอดีทั้ง price และ qty | PO `PO-B1` ที่ `po_status = completed` (qty 100 @ `฿50.00`); ใบแจ้งหนี้ผู้ขาย `INV-B1` สำหรับ qty 102 @ `฿50.75`; tenant qty tolerance = `2%` (ขอบบน = 102); tenant price tolerance = `1.5%` (ขอบบน = `฿50.75`) | ยอมรับ — ทั้งสองมิติลงที่ขอบพอดี `PO_POST_008` ทำงาน: variance qty `(102 − 100) / 100 = 2%` และ variance price `(50.75 − 50.00) / 50.00 = 1.5%` ทั้งคู่เท่ากับ tolerance ไม่เกิน (ขอบรวมพอดีตามการตั้งค่า tenant) AP post ที่ค่าใบแจ้งหนี้; การรับเกิน qty reconcile เทียบกับ tolerance รับเกินของ GRN ที่ Receiver post; variance price ไหลเข้า PPV การข้ามขอบใดขอบหนึ่ง (qty = 103 หรือ price = `฿50.76`) จะ trigger `PO_POST_009` ตาม FIN-VAL-01 / FIN-VAL-02 |
| FIN-EDGE-02 | FX rate ตรงกับ snapshot ของ PO พอดี — ไม่ post FX adjustment | PO `PO-NX` ที่ `po_status = completed` ด้วย `currency_id = USD`, `exchange_rate = 36.50`; ใบแจ้งหนี้ผู้ขาย `INV-NX` รับเป็น `USD` (match สกุลเดียว); rate `USD → THB` วันใบแจ้งหนี้บังเอิญเป็น `36.50` พอดี (ตรงกับ snapshot ของ PO ที่ 5 ทศนิยม) | ยอมรับ ไม่มี entry FX `PO_POST_008` ทำงาน: match ผ่านบน qty + price; AP post ใบแจ้งหนี้เป็น `USD`; มูลค่าสกุลฐาน `THB` ใช้ rate วันใบแจ้งหนี้ `36.50` — ซึ่งเท่ากับ snapshot `exchange_rate` ของ PO — ดังนั้น entry realised FX gain/loss คือ `฿0.00` และไม่ถูก write ลง GL (no-op suppression ตามนโยบาย tenant) แตกต่างจาก FIN-HP-05 ที่ cross-rate ต่างจาก snapshot |
| FIN-EDGE-03 | Multi-invoice partial-then-full match เทียบกับ PO เดียวกัน | PO `PO-MI` ที่ `po_status = partial` (`order_qty = 10`, สอง GRN: `GRN-MI1` สำหรับ qty 4, `GRN-MI2` สำหรับ qty 6); vendor ส่งใบแจ้งหนี้สองฉบับ: `INV-MI1` สำหรับ qty 4 ก่อน แล้ว `INV-MI2` สำหรับ qty 6 ทีหลัง ทั้งคู่ที่ unit price ตรงกัน | ใบแจ้งหนี้แต่ละฉบับ match อิสระ **รอบแรก:** `INV-MI1` match `GRN-MI1` (qty 4) → `PO_POST_008` ทำงาน → AP post `INV-MI1` liability สำหรับ 4 หน่วย; matched-but-unbilled บน PO ลดจาก 10 เป็น 6 **รอบสอง:** `INV-MI2` match `GRN-MI2` (qty 6) → `PO_POST_008` ทำงานอีกครั้ง → AP post `INV-MI2` liability สำหรับ 6 หน่วย; matched-but-unbilled ลดเป็น 0 PO `po_status` ตาม transition ที่ขับโดย GRN อิสระ (`sent → partial → completed`); ใบแจ้งหนี้แต่ละฉบับ approved for payment ตามลำดับ; ไม่มี logic match แบบยอดสะสม — โมเดลเป็น match แบบ **GRN-by-GRN** ไม่ใช่ running-totals |
| FIN-EDGE-04 | post invoice-match แบบ concurrent บน PO เดียวกันจาก 2 session ของ AP | Finance Officer A และ B ทั้งคู่ capture ใบแจ้งหนี้เทียบกับ `PO-X` (`po_status = completed`, GRN เดียวที่ post, รับเต็มแล้ว); A capture `INV-XA` สำหรับ qty ทั้งหมดที่ `T`; B capture ใบแจ้งหนี้ vendor ซ้ำ `INV-XB` สำหรับ qty เดียวกันที่ `T + 500ms`; ทั้งคู่พยายาม match | match แรกชนะแบบ atomic: match ของ A รัน → `PO_POST_008` post AP สำหรับ `INV-XA` → เลขที่ใบแจ้งหนี้ vendor `INV-XA` ถูกบันทึกเทียบกับ `PO-X` แล้ว ความพยายาม match ของ B ถูก reconcile: (a) duplicate-invoice guard (FIN-VAL-06) จับ B ด้วย `"Duplicate invoice — vendor invoice <INV-XB> has already been matched..."` ถ้า `INV-XB == INV-XA`, หรือ (b) ถ้า `INV-XB` เป็นเลขที่ใบแจ้งหนี้ที่ต่างแต่ match GRN เดียวกัน optimistic-concurrency guard ระดับ GRN reject B ด้วย `"This GRN's invoice match was claimed by another user. Please refresh and re-capture."` ไม่มี double-post; ไม่มี over-credit `2100-AP-Trade`; B retry หลัง refresh |
| FIN-EDGE-05 | ความละเอียดทศนิยมบนยอด AP — ยอดรายบรรทัด reconcile กับ header ที่ 2 ทศนิยม | PO `PO-DP` มี 2 บรรทัด: `L1` (qty 7 @ `฿14.33`, `tax 7%`), `L2` (qty 3 @ `฿49.99`, `tax 7%`); ใบแจ้งหนี้ผู้ขาย `INV-DP` match พอดี | การคำนวณบรรทัดตาม `PO_CALC_001`–`PO_CALC_005` ใช้กฎ rounded-prior-step จาก PR_046–PR_055 `L1`: subtotal `Round(14.33 × 7, 2) = ฿100.31`, tax `Round(100.31 × 0.07, 2) = ฿7.02`, total `฿107.33` `L2`: subtotal `Round(49.99 × 3, 2) = ฿149.97`, tax `Round(149.97 × 0.07, 2) = ฿10.50`, total `฿160.47` Header ตาม `PO_CALC_008`–`PO_CALC_010`: `total_price = ฿250.28`, `total_tax = ฿17.52`, `total_amount = ฿267.80` AP post ลง GL: **Dr** GRN-accrual `฿250.28`, **Dr** VAT-input `฿17.52`, **Cr** AP-Trade `฿267.80` ไม่มี rounding drift ระหว่างผลรวมบรรทัดกับ header; ความละเอียดการเก็บ `Decimal(20, 5)` คงไว้ภายใน; การปัด display 2 ทศนิยมใช้กับ AP voucher เท่านั้น |
