# ใบสั่งซื้อ — กรณีทดสอบ — เจ้าหน้าที่จัดซื้อ (Purchaser)

หน้านี้รวบรวมกรณีทดสอบที่ persona ของ Purchaser (อีกชื่อหนึ่งคือ **Procurement Officer**) เป็นผู้ขับเคลื่อนโดยตรงในโมดูล `purchase-order` Purchaser เป็นเจ้าของ PO ตั้งแต่สร้างจนถึงส่งให้ vendor — ช่วงตั้งแต่ `po_status = draft` ถึง `sent` ([03-user-flow-purchaser.md](./03-user-flow-purchaser.md) หัวข้อ 2) เส้นทางสร้าง PO มีสองทางที่บรรจบกันที่ flow เดียวกัน: **manual PO** (`po_type = manual`) ที่ฝ่ายจัดซื้อขึ้นเอง และ **PR-sourced PO** (`po_type = purchase_request`) ที่เกิดจาก Convert-to-PO จากโมดูล [[purchase-request]] ต้นน้ำผ่าน bridge table `tb_purchase_order_detail_tb_purchase_request_detail` ([01-data-model.md](./01-data-model.md) หัวข้อ 2.5) กรณีทดสอบถูกจัดเป็น happy path ตามที่อธิบายใน user flow, ขอบเขต RBAC ที่บังคับโดย `PO_AUTH_001`–`PO_AUTH_003` และ `PO_AUTH_006` (ขอบเขต edit / submit / transmit ของ Purchaser) เทียบกับ `PO_AUTH_004`, `PO_AUTH_005`, `PO_AUTH_007` (การ escalate ไปยัง Procurement Manager), validation rule ใน [02-business-rules.md](./02-business-rules.md) หัวข้อ 2 (`PO_VAL_001`–`PO_VAL_016`) ที่ Purchaser สามารถ trigger ได้ตอน create / edit / submit และชุด edge case จำนวนเล็ก ๆ เกี่ยวกับความละเอียดทศนิยม, currency, concurrency และการเปลี่ยนสถานะของ vendor master เส้นทาง cross-persona ที่หมุนรอบ Purchaser (`X-PO-01`, `X-PO-02`, `X-PO-05`, `X-PO-10`) อยู่ในหน้าภาพรวมพ่อ ไม่ได้อยู่ในหน้านี้

## 2. Permission / Authorization

| # | กรณี | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | ---- | ---------------------------------------- |
| PUR-PERM-01 | Purchaser เปิด PO ของตนเอง (ผู้ใช้ปัจจุบันตรงกับ `tb_purchase_order.buyer_id` หรืออยู่ใน `user_action.execute[]` ที่ `workflow_current_stage` ปัจจุบัน) | **Allow** อ่านทุก status และ action create / edit / submit / transmit ตาม `PO_AUTH_001`–`PO_AUTH_003` และ `PO_AUTH_006` หน้ารายละเอียดโหลดด้วย header + items + panel Item Details (TC-PO-060301, TC-PO-060302) |
| PUR-PERM-02 | Purchaser ดู PO ของ Purchaser คนอื่น (อ่านตาม RBAC) | **Allow** อ่านเมื่อ RBAC ของ tenant ให้ `purchase-order:read` กับผู้ใช้ทุกคนที่ `enum_stage_role = purchase` (ค่าเริ่มต้น); **deny** action ที่ write (Edit / Submit / Send-to-Vendor / Close) เพราะ `PO_AUTH_002` กำหนดให้ผู้ใช้ต้องเป็น buyer ที่ถูกจัดให้ หรือถือ `workflow_current_stage` ปัจจุบัน เรียก API edit โดยตรงจะตอบกลับ `"You are not authorised to edit this PO at the current stage"` |
| PUR-PERM-03 | Purchaser แก้ PO ที่ `draft` ของตนเอง | **Allow** ตาม `PO_AUTH_002` (`po_status ∈ {draft, in_progress}` และผู้ใช้เป็น buyer ที่ถูกจัดให้) แก้ฟิลด์ header และบรรทัดได้ทั้งหมด; ปุ่ม **Edit / Delete / Submit** ปรากฏบน `draft` (TC-PO-060303); การแก้ qty / เพิ่มบรรทัด / ยกเลิก edit / submit ทำงานสำเร็จทั้งหมด (TC-PO-060401..TC-PO-060405) |
| PUR-PERM-04 | Purchaser แก้ PO ที่ `sent` ของตนเอง | **Allow** แบบจำกัดฟิลด์ตาม `PO_VAL_016` — มีเพียง `cancelled_qty` และ note รายบรรทัดเท่านั้นที่แก้ได้หลัง `sent` การแก้ vendor / currency / ราคาบรรทัด / qty บรรทัดถูก reject ด้วย `"PO can no longer be amended at status sent. Void or close instead."`; UI แสดงหน้ารายละเอียดในโหมด **read-only** สำหรับ header และ ราคา / qty บรรทัด (TC-PO-060304 best-effort) พร้อม panel Amendment ที่เปิดเผยเฉพาะฟิลด์ที่อนุญาต |
| PUR-PERM-05 | Purchaser พยายาม approve PO ของตนเอง (`in_progress → sent` self-approval โดย total ≤ tenant high-value threshold) | **Allow เฉพาะเมื่อนิยาม workflow มอบหมาย Purchaser ให้ stage อนุมัติ** (ตาม `PO_AUTH_004` — "ต่ำกว่า threshold Procurement Officer สามารถ self-approve ไปยัง `sent` ได้ถ้า workflow อนุญาต") **Deny** เมื่อ workflow ส่ง `in_progress` ไปที่ Procurement Manager ไม่ว่ายอดจะเป็นเท่าใด หรือเมื่อ segregation of duties (`PO_AUTH_010`) ถูกตั้งให้บังคับใช้ approver คนอื่น เรียก stage จากผู้ใช้ที่ไม่อยู่ใน list ถูก reject โดย `PO_AUTH_011` ด้วย `"You are not in the authorised approver set for this stage"` |
| PUR-PERM-06 | Purchaser void PO ที่ `draft` ของตนเองผ่าน soft-delete (path abandon) | **Deny direct hard-delete; allow escalation** `PO_AUTH_005` สงวน action soft-delete-in-draft ไว้ให้ Procurement Manager เส้นทางของ Purchaser คือ abandon-as-draft (ปล่อย PO เป็น `draft` ไว้) หรือ escalate ไปที่ Procurement Manager เพื่อ soft-delete เรียก API delete จาก role Purchaser โดยตรงถูก reject ด้วย `"Soft-delete from draft is restricted to the Procurement Manager"` UI เปิดปุ่ม `Delete` บน draft (TC-PO-060303) แต่ API บังคับใช้ role check |
| PUR-PERM-07 | Purchaser พยายาม void PO ที่ `sent` หรือ `partial` | **Deny — Manager only** ตาม `PO_AUTH_007` void จาก non-`draft` non-terminal status (`in_progress`, `sent`, `partial`) สงวนไว้ให้ Procurement Manager Action Void ถูกซ่อน / ปิดบนหน้ารายละเอียดของ Purchaser สำหรับ PO ที่ไม่ใช่ `draft`; เรียก API ตรง ๆ ตอบกลับ `"Void from status <status> requires the Procurement Manager role"` |
| PUR-PERM-08 | Purchaser พยายามปิด (close) PO ที่ `partial` (early termination) | **Deny — เฉพาะ Receiver / Manager** ตาม `PO_AUTH_008` action Close (`partial → closed`) มอบให้ Inventory Manager / Receiver ที่รัน GRN posting; Procurement Manager อาจปิดผ่าน `PO_POST_011` ก็ได้ Purchaser อาจกดปุ่ม **Close** บนหน้ารายละเอียด PO ที่อนุมัติแล้ว (TC-PO-060503, TC-PO-060504) เฉพาะเมื่อ workflow เลื่อนสิทธิ์ให้ชั่วคราว — โดยค่าเริ่มต้นจะถูก reject ด้วย `"You are not authorised to close this PO"` |

## 3. Validation / Error

| # | กรณี | Trigger | Error ที่คาดหวัง |
| - | ---- | ------- | ---------------- |
| PUR-VAL-01 | Submit PO โดยไม่มี `po_no` (DB-level fallback) | force save โดยตั้ง `po_no` ว่างผ่าน direct API call (UI สร้างให้อัตโนมัติ) | Reject — `PO_VAL_001` server ตอบกลับ `"PO reference number is required and must be unique."`; DB unique index `@@unique([po_no, deleted_at])` เป็นด่านสุดท้าย |
| PUR-VAL-02 | Submit PO ที่ `vendor_id` ชี้ไปแถว `tb_vendor` ที่ soft-deleted หรือ inactive | เลือก vendor `V_old` ที่ `deleted_at IS NOT NULL` หรือ `is_active = false`; save แล้ว submit | Reject — `PO_VAL_002` server ตอบกลับ `"Vendor is required and must be from the approved vendor list."`; vendor picker ของ UI กรอง vendor inactive และ deleted ออก แต่ selection ที่ค้างไว้ถูกจับได้ตอน submit |
| PUR-VAL-03 | Submit PO ที่ `exchange_rate = 0` หรือลบ | ตั้ง `exchange_rate = 0` บน header แล้วลอง save / submit | Reject — `PO_VAL_003` server ตอบกลับ `"Transaction currency and a positive exchange rate are required."`; input ของ UI บังคับ `> 0` แต่ direct API call ถูกจับโดย validation |
| PUR-VAL-04 | Submit PO ที่ `delivery_date < order_date` | ตั้ง `order_date = 2026-05-15`, `delivery_date = 2026-05-14`; กด Save / Submit | Reject — `PO_VAL_006` server ตอบกลับ `"Delivery date must be on or after the order date."`; date picker ของ UI เตือน inline; ความพยายาม submit ถูกบล็อก |
| PUR-VAL-05 | Submit PO ที่บรรทัดมี `order_qty = 0` | ตั้ง `order_qty = 0` ที่บรรทัดหนึ่ง; กด Submit | Reject — `PO_VAL_008` server ตอบกลับ `"Order quantity must be greater than zero and a unit of measure is required."`; บรรทัดถูกทำเครื่องหมายสีแดงในตาราง items |
| PUR-VAL-06 | Submit บรรทัด PO ที่ `price < 0` หรือ `price = 0` โดยไม่มี flag FOC | ตั้ง `price = -10` บนบรรทัด หรือ `price = 0` ที่ `is_foc = false`; กด Submit | Reject — `PO_VAL_010` server ตอบกลับ `"Unit price must be non-negative; price of 0 requires the FOC flag."`; UI แสดง error inline บนบรรทัด |
| PUR-VAL-07 | บันทึก draft PO โดยไม่มีบรรทัด แล้วลอง submit | จาก wizard **Blank** save แค่ header ไม่มีบรรทัด แล้วกด Submit | Reject — `PO_VAL_012` server ตอบกลับ `"PO must contain at least one line item."`; ปุ่ม **Save** ของ UI ปิดอยู่หรือหน้าค้างอยู่ที่ `/new` ตอนไม่มีบรรทัด (TC-PO-060204) |
| PUR-VAL-08 | พยายาม submit PO ที่บรรทัดผสม vendor หรือ currency | เพิ่มบรรทัด `L1` ด้วย `vendor_id = V1, currency_id = THB` และบรรทัด `L2` ด้วย `vendor_id = V2, currency_id = THB` (force ผ่าน direct API call — UI ป้องกันไม่ให้เลือก vendor ปนกันโดยผูกบรรทัดเข้ากับ `vendor_id` บน header) | Reject — `PO_VAL_013` server ตอบกลับ `"All lines on a PO must share the header vendor and currency. Split into separate POs by vendor+currency."`; invariant single-vendor / single-currency ถูกบังคับใช้ตอน submit |
| PUR-VAL-09 | Convert PR ที่อนุมัติแล้วเป็น PO แต่ขาด bridge row | force insert แถว `tb_purchase_order_detail` ที่ `po_type = purchase_request` บน parent PO โดยไม่มี entry ใน `tb_purchase_order_detail_tb_purchase_request_detail`; กด Submit | Reject — `PO_VAL_014` server ตอบกลับ `"PR-sourced PO lines must be linked to an originating PR line via the bridge table."`; wizard From-PR เขียน bridge row ให้เสมอ validation นี้จึงเน้นป้องกัน path direct-API / data-fix เป็นหลัก |
| PUR-VAL-10 | พยายามแก้ `vendor_id` หรือ `currency_id` บน PO ที่ `sent` | เปิด PO ที่ `po_status = sent` แล้วลองเปลี่ยน vendor หรือ currency ของ header ผ่าน direct API call (UI ปิดฟิลด์) | Reject — `PO_VAL_016` server ตอบกลับ `"PO can no longer be amended at status sent. Void or close instead."`; มีเพียง `cancelled_qty` และ note รายบรรทัดเท่านั้นที่แก้ได้หลัง `sent` |

## 4. Edge Cases

| # | กรณี | เงื่อนไข | ผลที่คาดหวัง |
| - | ---- | -------- | ------------ |
| PUR-EDGE-01 | การ์ดป้องกัน zero-quantity ตอน submit รวมกับการ์ดป้องกันตอน save-line | Purchaser ลอง save บรรทัดด้วย `order_qty = 0` (ถูกจับตอน save โดย `PO_VAL_008`); จากนั้นพยายามข้าม UI โดย submit PO ด้วยบรรทัดที่ถูกแก้เป็น `0` หลัง save ผ่าน direct API call | ทั้งการรัน `PO_VAL_008` ตอน save-line และตอน submit-time reject บรรทัดด้วย `"Order quantity must be greater than zero and a unit of measure is required."`; submit abort; บรรทัดคงค่าก่อนหน้าที่ valid ไว้; ไม่มีแถว `tb_purchase_order_detail` ถูกแก้ |
| PUR-EDGE-02 | ความละเอียดทศนิยมสูงสุดที่ `price` ของบรรทัด | บรรทัด `price = 99999999999.99999` (จำนวนเต็ม 15 + ทศนิยม 5 หลัก ขอบบนของ `Decimal(20, 5)`) ด้วย `order_qty = 1.99999`; tax_rate `7%`, ไม่มีส่วนลด | บันทึกได้ที่ storage; `sub_total_price = Round(99999999999.99999 × 1.99999, 2)` คำนวณตาม `PO_CALC_001`; การปัดเศษใช้ half-up ตาม `PO_CALC_012`; `base_qty = Round(1.99999 × conv_factor, 3)` ตาม `PO_CALC_006`; header `total_qty` รวมในหน่วย UoM ฐานที่ 3 dp ตาม `PO_CALC_011`; Decimal ทั้งหมดที่บันทึกอยู่ภายใน `Decimal(20, 5)` สำหรับเงินและ `Decimal(15, 5)` สำหรับ rate |
| PUR-EDGE-03 | `delivery_date` ย้อนหลังเท่ากับ `order_date` (ขอบล่างของ `PO_VAL_006`) | ตั้ง `order_date = 2026-05-15`, `delivery_date = 2026-05-15`; กด Save / Submit | Accepted — `PO_VAL_006` บังคับ `delivery_date >= order_date` (รวม) กรณีวันเดียวกันผ่าน; PO save และ submit ปกติ การตั้ง `delivery_date = order_date − 1d` จะ fail (ครอบคลุมโดย PUR-VAL-04) |
| PUR-EDGE-04 | การ reject consolidate ข้าม currency ใน wizard From-PR | tick PR ที่อนุมัติแล้ว 2 ฉบับใน wizard: `PR-A` ที่ `currency_id = THB` และ `PR-B` ที่ `currency_id = USD` ทั้งคู่สำหรับ `vendor_id = V1` | Wizard จัดกลุ่มอัตโนมัติตาม `(vendor_id, currency_id)` → ได้ draft-PO **2** กลุ่ม (กลุ่มละ currency) ไม่ใช่ 1 PO ที่ได้แต่ละใบเคารพ `PO_VAL_013` direct API call เพื่อบังคับให้บรรทัดทั้งสองเข้า PO เดียวกันถูก reject ด้วย `"All lines on a PO must share the header vendor and currency. Split into separate POs by vendor+currency."` (`PO_VAL_013`) |
| PUR-EDGE-05 | แก้ draft PO เดียวกันพร้อมกันจาก 2 session ของ Purchaser | Purchaser A และ Purchaser B โหลด draft PO `PO-X` ที่ `doc_version = 3`; A save การเปลี่ยน qty ที่ `T` (`doc_version → 4`); B save การเปลี่ยน price ที่ `T + 1s` โดยใช้ `doc_version = 3` ที่ stale | save แรกชนะ: การเปลี่ยน qty ของ A ถูกบันทึก `doc_version = 4` save ของ B ถูก reject โดยการ์ด optimistic-concurrency ด้วย `"This PO was modified by another user. Please refresh and re-apply your changes."`; client ของ B แจ้งให้ refresh ไม่เกิด double-write หรือ lost-update |
| PUR-EDGE-06 | Transmit ขณะ vendor email หายหรือ vendor flip inactive | PO ที่ `po_status = sent`; `tb_vendor.email IS NULL` สำหรับ vendor บน PO หรือ `tb_vendor.is_active = false` หลัง PO อนุมัติแล้ว | **Send to Vendor** (TC-PO-060502) บน vendor ที่ไม่มี email ถูก reject ด้วย `"Vendor email is not configured — set up the transmission channel before sending."`; บน vendor inactive / blacklist ถูก reject ด้วย `"Vendor is no longer active and cannot receive new POs."` PO ยังเป็น `sent` (status transition ทำงานแล้วตอน final approval ตาม `PO_POST_004`); column `email` / `approval_date` ไม่ถูก update; ผนวก audit comment Purchaser ต้อง escalate ไปที่ Procurement Manager เพื่อ void (`PO_AUTH_007`) หรือ update vendor master แล้วลองใหม่ |
| PUR-EDGE-07 | Submit PO ทันทีหลัง soft-delete บน `po_no` เดียวกัน (uniqueness re-use) | PO ฉบับก่อนหน้าที่ `po_no = PO-2026-000123` ถูก soft-delete (`deleted_at IS NOT NULL`) ตาม `PO_POST_012`; Purchaser ร่าง PO ใหม่และ numbering service ออก `po_no` เดียวกัน | Accepted — unique index `@@unique([po_no, deleted_at])` อนุญาตให้ reuse ได้เพราะ `deleted_at` ต่างกัน PO ใหม่ save และ submit ปกติ; แถวที่ soft-delete ก่อนหน้ายังอยู่ใน database เพื่อ audit Hard-delete-in-draft สงวนไว้ให้ Procurement Manager (`PO_AUTH_005`) กรณีนี้จึงเกิดจากการ cleanup ที่ขับโดย Manager |
