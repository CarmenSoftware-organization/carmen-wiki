# ใบเบิกของสโตร์ (Store Requisition) — Test Scenarios — Audit / Config

> **At a Glance**
> **Persona:** Audit / Config (Inventory Controller + Finance + Sysadmin + Auditor) &nbsp;·&nbsp; **โมดูล:** [[store-requisition]] &nbsp;·&nbsp; **Scenarios:** ~27
> **หมวดหมู่:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** ส่วนใหญ่ manual / planned; automated coverage ที่ใกล้ที่สุดคือ `tests/701-sr.spec.ts` TC-SR-060005 (Delegate Approvals) ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ scenario test ที่ persona Audit / Config — ประกอบด้วย **Inventory Controller** (variance, admin void ก่อน commit, ผู้เขียน adjustment หลัง commit ร่วม), **Finance** (block งวดปิด, ตรวจสอบ journal-entry, การ reconcile cost-centre, period close), **System Administrator** (RBAC, การตั้งค่า workflow / `tb_workflow`, threshold การผ่อนคลาย SoD, การเชื่อม auto-create recipe) และ **Auditor** (review history SR อ่านอย่างเดียว, signature trace, การยืนยันการบังคับใช้ SoD) — ขับเคลื่อนโดยตรงในโมดูล `store-requisition` ต่างจาก 4 persona เชิงปฏิบัติการ (Requester, Approver, Fulfiller, Receiver) ที่ทำงานวงจรชีวิต happy-path sub-role Audit / Config กระทำที่ **periphery**: ก่อน SR ใด ๆ มีอยู่ (config), ระหว่าง flow (variance monitoring, การบังคับใช้ period-close) และหลัง commit (signoff, audit) Scenarios จัดกลุ่มเป็น **happy path** (variance review, admin void ก่อน commit, block งวดปิด, period close, RBAC / workflow config, audit signature trace), **RBAC** (admin void เฉพาะก่อน commit, อำนาจ Finance งวดปิด, Sysadmin workflow config, Auditor อ่านอย่างเดียว), **validation** (test เชิงลบเรื่อง workflow ที่ route ผิด, การละเมิด SoD ที่ตรวจพบที่ audit, ความล้มเหลวของ auto-create recipe) และ **edge case** เรื่อง config multi-tenant, การเปิดงวดใหม่, batch ปลายงวดขนาดใหญ่ และ test sampling ของ Auditor Handoff ข้าม persona ที่ pivot จาก persona นี้ (Scenarios 7, 10, 11, 13, 14 ในภาพรวมแม่ — การ resolve ความคลาดเคลื่อน, การ resolve block งวดปิด, admin void, period close, การเปลี่ยน config workflow / RBAC) อยู่ใน [04-test-scenarios.md](./04-test-scenarios.md) ไม่ใช่ที่นี่

## 2. Permission / Authorization

| # | Scenario | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | -------- | --------------------------------------- |
| AC-PERM-01 | Inventory Controller / Sysadmin void SR ก่อน commit | **Allow** ตาม `SR_AUTH_013` ปุ่ม **Admin Void** enable บน SR `draft` และ `in_progress` สำหรับ role เหล่านี้ เหตุผลบังคับ; void เขียน state จุดสิ้นสุด `voided`; ไม่กระทบ inventory / GL |
| AC-PERM-02 | Inventory Controller พยายาม void SR `completed` | **Deny — สถานะจุดสิ้นสุด** ตาม `SR_AUTH_013` ปุ่ม Admin Void ถูกซ่อนบน SR `completed`; การเรียก API โดยตรง reject ด้วย `"Cannot void a committed SR; post-commit corrections must use inventory-adjustment."` ตามกฎ "Voiding a `completed` SR is not allowed" ของ `SR_POST_010` |
| AC-PERM-03 | Finance ปิดงวด | **Allow** การปิดงวดของ Finance ล็อก `SR_VAL_014` กับการ commit ต่อด้วยวันที่ post ในงวดปิด Action ปิดงวดอยู่ในโมดูล finance (ไม่ใช่ในโมดูล SR); โมดูล SR อ่าน state งวดปิดที่ commit |
| AC-PERM-04 | ผู้ใช้ที่ไม่ใช่ CFO พยายามเปิดงวดปิดใหม่ | **Deny — สิทธิ์เปิดงวดใหม่** การเปิดงวดใหม่โดยทั่วไปต้องการลายเซ็น CFO / Finance Manager โมดูล SR เป็นผู้บริโภคปลายน้ำของ state งวด; authorization บังคับใช้ในโมดูล finance จากฝั่ง SR ผลที่เห็นเดียวคือ `SR_VAL_014` continue block commit จนกว่างวดจะเปิดใหม่ |
| AC-PERM-05 | Sysadmin แก้ config `tb_workflow` | **Allow** Sysadmin เป็นเจ้าของ workflow configuration การเปลี่ยน apply prospective จากเวลา save; SR ที่อยู่ใน flow **ไม่** ถูก re-route อัตโนมัติ (Sysadmin ประสานกับ Inventory Controller สำหรับ re-route ถ้าจำเป็น) |
| AC-PERM-06 | ผู้ใช้ที่ไม่ใช่ Sysadmin พยายามแก้ RBAC | **Deny** RBAC matrix ต่อ location / department Sysadmin-only การเรียก API โดยตรง reject ด้วย `"You are not authorized to modify role assignments."` |
| AC-PERM-07 | Auditor พยายามเขียนคอลัมน์ใดบน `tb_store_requisition` | **Deny — Auditor อ่านอย่างเดียว** Auditor ไม่มีสิทธิ์เขียนบนคอลัมน์ตาราง SR ใด การพยายามเขียนโดยตรง (ผ่าน API หรือ UI workarounds) reject ด้วย `"Auditor role is read-only."` การเขียน comment (เช่น audit findings) โดยทั่วไป log ในระบบ audit-findings แยก ไม่ใช่บนตาราง comment ของ SR |
| AC-PERM-08 | Inventory Controller พยายาม post inventory-adjustment โดยไม่มี Finance co-auth (เหนือ threshold) | **Deny — ต้องการการอนุญาตร่วม** ตาม `SR_XMOD_009` การแก้หลัง commit ผ่าน `[[inventory-adjustment]]` สำหรับมูลค่าสาระสำคัญต้องการการอนุญาตร่วมโดย Inventory Controller AND Finance ใต้ threshold ของ tenant Inventory Controller อาจ post คนเดียว; เหนือ threshold adjustment ถูกถือเพื่ออนุมัติของ Finance Block บังคับใช้ในโมดูล inventory-adjustment ไม่ใช่โมดูล SR |

## 3. Validation / Error

| # | Scenario | Trigger | Error ที่คาดหวัง |
| - | -------- | ------- | --------------- |
| AC-VAL-01 | Admin void โดยไม่มีเหตุผล | Inventory Controller คลิก **Admin Void** แต่ submit ด้วยข้อความเหตุผลว่าง | **Reject** — เหตุผลบังคับสำหรับ `SR_POST_010` Server คืน `"Reason text is required for administrative void."` |
| AC-VAL-02 | ปิดงวดขณะมี SR `in_progress` ค้าง | Finance พยายามปิดงวด `2026-05` แต่มี SR 3 ใบยังอยู่ที่ `doc_status = in_progress` ด้วยขั้น workflow ค้าง | **Warn — ความผิดปกติการปิดงวด** Action ปิดงวดอาจดำเนินต่อ (ขึ้นกับนโยบาย tenant) แต่ Finance ถูกแจ้ง SR ค้างและถูกขอประสานกับ Inventory Controller (resolve ผ่าน admin void หรือ push commit) การปิดงวดเองเป็น action ของโมดูล finance; บทบาทโมดูล SR คือ surface จำนวน SR ค้างให้ dashboard ปิด |
| AC-VAL-03 | การเปลี่ยน config workflow สร้าง SR ที่ orphan ใน flow | Sysadmin ลบขั้น workflow ที่มี SR ใน flow ที่นั่น; SR สูญการ map `workflow_current_stage` | **Warn ตอน save** — UI แสดง warning ("This change orphans 7 in-flight SRs at stage `<removed-stage>`; proceed to migrate them to a different stage?") Sysadmin ยกเลิกการเปลี่ยน เลือกขั้นเป้าหมายที่จะ migrate หรือดำเนินต่อและยอมรับว่า SR ที่ orphan ต้องการการแทรกแซงด้วยมือ ตามความหมาย `SR_AUTH_014` (stage gating ต้องการขั้นที่ใช้ได้) |
| AC-VAL-04 | การละเมิด SoD ตรวจพบที่ audit | Auditor พบ SR ที่ `requestor_id = approved_by_id` บนบรรทัดอย่างน้อยหนึ่ง (SoD `SR_AUTH_011` ควรได้ block แล้ว) | **Audit finding — สอบสวน root cause** การละเมิดบ่งชี้ gap ระบบ (check SoD ของโมดูล SR ล้มเหลว) หรือ workflow misconfig (workflow อนุญาตผู้ใช้คนเดียวอยู่ทั้งสอง role) Auditor เผยแพร่ finding; Sysadmin / Inventory Controller สอบสวนเส้นทาง; corrective action ครอบคลุมตั้งแต่ patch ระบบ (ถ้า check SoD ถูก bypass) ถึงการเสริมนโยบาย (ถ้า workflow ถูก config permissive ตั้งใจ) |
| AC-VAL-05 | Auto-create recipe ล้มเหลวสำหรับ event banquet ที่วางแผน | โมดูล `[[recipe]]` ล้มเหลวในการ post SR `draft` (recipe-product mapping พัง, การ resolve source-location ล้มเหลว, ไม่พบผู้ใช้ requester) | **System alert — Sysadmin สอบสวน** Sysadmin check การเชื่อม: recipe-product mapping, การ resolve source-location, การ lookup ผู้ใช้ requester, การ check สิทธิ์ location Ad-hoc fix post SR draft ด้วยมือพร้อม `info.recipe_id` carry over; การเชื่อมพื้นฐานถูกแก้สำหรับ event ถัดไป |

## 4. Edge Cases
