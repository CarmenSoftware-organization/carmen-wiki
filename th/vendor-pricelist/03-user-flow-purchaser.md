# รายการราคาผู้ขาย — User Flow — Purchaser (Vendor Pricelist — User Flow — Purchaser)

> **At a Glance**
> **Persona:** Purchaser / Purchasing Staff (+ Purchasing Manager) &nbsp;·&nbsp; **โมดูล:** [[vendor-pricelist]] &nbsp;·&nbsp; **Workflow stage:** Template draft → active; Campaign draft → active → completed; Pricelist submitted → active / inactive / expired &nbsp;·&nbsp; **สิทธิ์สำคัญ:** สร้าง template, launch campaign, เชิญ vendor, review / approve / reject submission, การ curate preferred-vendor, high-value sign-off (Manager)
> **สิ่งที่ persona นี้ทำ:** เป็นเจ้าของ lifecycle ของ pricelist end-to-end ฝั่ง Carmen — template, campaign, vendor submission, การ approve และ flag preferred-vendor

## 1. Role ในโมดูลนี้

ไฟล์ persona **Purchaser** รวบ role องค์กรสองตัวที่บันทึกใน [index.md](./index.md) Section 4: **Purchaser / Purchasing Staff** (operator ที่สร้าง price-collection template, ตั้งเวลา campaign, ส่ง invitation, review และ approve pricelist ที่ submit, จัดการการแก้ price-item รายตัวและ flag preferred-vendor และ upload การ submission ที่ส่งทาง email ของ vendor ในนาม vendor ที่ไม่สามารถใช้ portal) และ **Purchasing Manager** (ชั้นการ escalation ที่ approve pricelist high-value, ตั้งค่า business rule สำหรับ logic preferred-vendor และ price-assignment, sign off บนราคา multi-currency ข้ามพรมแดน และใช้สิทธิ์ override ต่อ queue ของ Purchaser ปกติ) ทั้งสอง role ใช้หน้าจอเดียวกันและไฟล์ persona เดียวกันเพราะ action การ activate, approve และ reject เหมือนกัน — Manager แตกต่างจาก Purchaser ใน **ขอบเขต authorization** (`VPL_AUTH_005`, `VPL_AUTH_006` สำหรับ high-value approve, การตั้งค่า business-rule และ sign-off multi-currency) ไม่ใช่ใน **surface หน้าจอ** Persona นี้เป็นเจ้าของ lifecycle pricelist เชิงปฏิบัติการ end-to-end ฝั่ง Carmen: ทุกการเปลี่ยนที่ไม่ได้ขับโดย cron job (`VPL_POST_021` auto-expire), event ของ vendor portal (`VPL_POST_015`–`VPL_POST_016`) หรือ actor Finance / Audit / Sysadmin ภายนอกทำที่นี่ Purchaser ยังปฏิบัติการกับ **Receiver / Store Keeper** เป็นผู้บริโภคทางอ้อม — การ post GRN ของ Receiver อ่าน pricelist active สำหรับการคำนวณ variance ([[vendor-pricelist/02-business-rules]] § `VPL_XMOD_005`) และข้อค้นพบ variance route กลับไปยัง Purchaser สำหรับ resolution ผ่านการ amend, inactivate หรือ re-collect ผ่าน campaign ใหม่ ร่วมกันสิ่งนี้ทำให้ Purchaser เป็น **เจ้าของสัญญาคุณภาพข้อมูล** ที่โมดูล PR / PO / GRN ปลายน้ำพึ่งพา

## 2. จุดเข้าและ Primary Flow

**จุดเข้า:** Sidebar → **Vendor Management** workspace → tab **Templates** / **Campaigns** / **Pricelists** การดำเนินการวันต่อวันใช้ queue **Pricelists → Submitted (awaiting review)** เป็นรายการงานหลัก; การ launch campaign และการสร้าง template เป็น activity เป็นช่วง ๆ ขับโดยปฏิทินรอบ procurement (โดยทั่วไปรายไตรมาส) หรือคำขอเก็บ ad-hoc จาก category manager

**Primary flow (happy path — รอบ 6-phase เต็ม):**

1. **สร้าง template** Sidebar → **Templates** → **New Template** ป้อน `name` (unique ตาม `VPL_VAL_001`), description, vendor instruction (render บน portal + เนื้อหา email), default `currency_id`, `validity_period` ในวัน และ schedule reminder `[14, 7, 3, 1]` (`VPL_VAL_004`) Save ใน `draft`
2. **เพิ่มสินค้าไป template** เปิด template → tab **Products** → เลือกสินค้าผ่าน category / subcategory / item group / search สำหรับแต่ละสินค้า ตั้งค่า default order unit และ MOQ tier อย่างน้อยหนึ่ง (`VPL_VAL_006`); รูปร่าง JSON บน `tb_pricelist_template_detail.order_unit_obj` carry `{ default_order: {unit_id, unit_name}, moq: [ {unit_id, unit_name, qty, note}, ... ] }` สินค้าแต่ละชิ้นปรากฏได้ครั้งเดียวต่อ template (`VPL_VAL_005` — unique `(template_id, product_id)`)
3. **Activate template** คลิก **Activate** `VPL_POST_002` fire: `tb_pricelist_template.status = draft → active`; template ตอนนี้เลือกได้สำหรับ campaign ใหม่ `system` comment บันทึก actor และ timestamp การ activate
4. **Launch campaign** Sidebar → **Campaigns** → **New Campaign** → เลือก template active ป้อน `name`, `start_date`, `end_date` (ต้องตรง `VPL_VAL_010` — `start < end`, `end > now()`, window ขั้นต่ำ 3 วัน), `custom_message` และ `email_template_id` (tenant-id ของ email template) เลือก vendor cohort เพื่อเชิญ คลิก **Launch**
5. **อีเมล invitation dispatch** Carmen materialise แถว `tb_request_for_pricing_detail` — หนึ่งต่อ vendor ที่เลือก — ด้วย `pricelist_url_token` cryptographic ใหม่ต่อแถว Email dispatch ผ่าน channel ที่ตั้งค่า; แต่ละฉบับ carry URL portal ต่อ vendor ฝัง token Campaign status ตอนนี้ `active` (derive ตาม `VPL_POST_006`)
6. **ติดตามความคืบหน้า campaign** เปิด campaign → tab **Vendor Tracking** แต่ละแถว invitation แสดงสถานะ invitation ที่ derive (`pending` → `in-progress` → `submitted` → `approved` / `expired`) Telemetry การส่ง email (sent / delivered / opened / clicked) จับเป็น `system` comment ใน `tb_request_for_pricing_detail_comment` พร้อม payload structured ใน JSON `attachments` / `message` ([[vendor-pricelist/01-data-model]] § 2.7)
7. **รับ vendor submission** เมื่อ vendor คลิก Submit ที่ portal `tb_pricelist.submitted_at = now()` ถูกเขียนและ Purchaser ได้รับการแจ้งเตือน Pricelist ที่ submit surface ใน queue **Pricelists → Submitted (awaiting review)**
8. **Review pricelist ที่ submit** เปิด pricelist → panel **Validation** surface output ของ engine ที่เขียนไปยัง `tb_pricelist.info.validation_results` และ `info.quality_score` (`VPL_CALC_006`) เดินแต่ละแถว `tb_pricelist_detail` — product, unit, MOQ qty, price-without-tax, tax, lead-time, effective unit price ต่อ UoM ฐานที่คำนวณอัตโนมัติ (`VPL_CALC_003`) แถว multi-MOQ บนสินค้าเดียวกัน auto-sort และ validate เป็น non-increasing (`VPL_VAL_020`)
9. **ตัดสินใจ** สามเส้นทาง:
   - **Approve** pricelist (`draft + submitted_at → active`, `VPL_POST_017`) ต้องการ `VPL_AUTH_004` (ต่ำกว่าขีดจำกัด high-value) หรือ `VPL_AUTH_005` สำหรับ high-value / `VPL_AUTH_010` สำหรับ multi-currency Mark แถวที่เหมาะสมเป็น preferred ผ่าน `is_preferred = true` `system` comment จับการ approve
   - **Reject** pricelist (`draft + submitted_at → draft`, `submitted_at` reset, `VPL_POST_018`) ต้องการ `VPL_AUTH_006` จับข้อความเหตุผล; vendor ได้รับ email + คง token ต้นฉบับสำหรับ resubmission จนกว่า campaign `end_date`
   - **Request edits** โดยไม่ reject — เปิด pricelist สำหรับการแก้ฝั่ง purchaser ภายใต้ `VPL_AUTH_001` และปรับแถวเฉพาะ (เช่น toggle `is_preferred`, แก้ typo หน่วยที่ชัดเจน) แล้ว approve การแก้ถูก log ใน `tb_pricelist_detail_comment` พร้อม actor และ diff
10. **Activate ความพร้อมปลายน้ำ** เมื่อ `tb_pricelist.status = active` แล้ว pricelist กลายเป็นการอ้างอิงสดสำหรับ `(vendor, product, currency, validity window)` และโมดูล PR / PO / GRN เห็นมันที่การอ่านครั้งต่อไป การสร้างบรรทัด PR default ราคาจากแถว preferred ที่ตรงกับสินค้า + สกุลเงิน + MOQ bracket ของบรรทัดตาม [[vendor-pricelist/02-business-rules]] § `VPL_XMOD_001`
11. **จัดการเส้นทาง email / manual submission** เมื่อ vendor return Excel ทาง email แทนการ upload ผ่าน portal Purchaser เปิดแถว invitation campaign ที่ตรง → **Upload pricelist on vendor's behalf** → drop ไฟล์ Excel แอป parse เป็น `tb_pricelist` + แถว detail ด้วย `submission_method = email` และ `system` comment แนบบันทึก email ต้นฉบับและ staff ที่ upload (ตาม `VPL_AUTH_003`) Pricelist จากนั้นเข้าสู่ flow review มาตรฐานที่ Step 8
12. **รัน close-out ที่สิ้นสุด campaign** เมื่อ `end_date` ผ่าน (หรือทุก vendor ที่เชิญ submit แล้ว) campaign auto-complete (`VPL_POST_008`) Invitation ที่ค้าง auto-expire (`VPL_POST_014`) และ token ถูก revoke Purchaser review analytics ของ campaign — อัตราการตอบสนอง, เวลาการ submission เฉลี่ย, การกระจาย quality-score — เพื่อ inform รอบครั้งต่อไป

## 3. สาขาการตัดสินใจ

- **การ curate Preferred-vendor ข้าม vendor หลายรายที่มีสินค้าเดียวกัน** เมื่อ vendor สองรายขึ้นไป submit pricelist active ครอบคลุมสินค้าเดียวกัน Purchaser เปิด matrix **Preferred-Vendor** → สำหรับแต่ละ cell สินค้า / หน่วย / MOQ-bracket toggle `is_preferred = true` บนแถว `tb_pricelist_detail` หนึ่งแถวเท่านั้น Toggle ถูก log ใน `tb_pricelist_detail_comment`; การ default บรรทัด PR ตาม toggle (`VPL_XMOD_001`) เมื่อ business rule engine (carmen/docs `price-assignment-workflow-documentation.md`) ตั้งค่า engine ตั้ง flag อัตโนมัติตาม rule category (vendor preference, price optimisation, quality assurance, supply chain); Purchaser ยังสามารถ override manual การ assignment อัตโนมัติ
- **High-value หรือ multi-currency approval route ไปยัง Manager** เมื่อมูลค่า aggregate ที่ projected ของ pricelist ที่ submit เกินขีดจำกัด high-value ของ tenant (คำนวณจากการตั้งราคา MOQ × demand ประวัติ) หรือเมื่อสกุลเงินของ pricelist ต่างจากสกุลเงินฐานของ tenant และข้ามขีดจำกัด cross-border-FX Purchaser ไม่สามารถ approve (`VPL_AUTH_004` floor) Pricelist route ไปยัง queue Purchasing Manager (`VPL_AUTH_005`) และสำหรับ multi-currency queue Finance Manager แบบขนาน (`VPL_AUTH_010`) การ approve ต้องการทั้งสอง sign-off ก่อน `VPL_POST_017` fire
- **Vendor ติดอยู่ที่ `in-progress` ใกล้ deadline** อีเมล reminder fire อัตโนมัติตาม schedule template (`reminder_days`) แต่ถ้า vendor มี portal token แต่ยังไม่ submit ด้วยเวลาที่เหลือ 24 ชั่วโมง Purchaser สามารถ (a) เปิด campaign → **Reminders** → คลิก **Send one-off reminder**, (b) ขยาย campaign `end_date` (เขียน `system` comment) หรือ (c) escalate ไปยังผู้ติดต่อทางโทรศัพท์ / email นอกแบนด์ การ escalation ถูก log เป็น `user` comment ดังนั้น chain audit ถูกรักษา
- **Vendor submit ข้อมูลคุณภาพต่ำ** ถ้า validation engine รายงาน `quality_score < tenant_threshold` (default 70 ต่อ `VPL_XMOD_009`) pricelist route ไป Manager review แทนการ auto-approve Manager (a) approve ด้วยคะแนนต่ำในบันทึก (เช่น vendor ใหม่เข้าแคตตาล็อก), (b) reject ภายใต้ `VPL_AUTH_006` พร้อมคำแนะนำการแก้เฉพาะ หรือ (c) ขอการแก้ inline ก่อน activate Score drift ข้ามรอบหลายรอบจับบน profile performance ของ vendor ป้อนเข้า business-rule engine
- **Vendor ปฏิเสธการใช้ portal — flow email-only** Purchaser ส่ง Excel template ทาง email (ดาวน์โหลดจากหน้า campaign / template), รับ workbook ที่กรอกแล้วทาง reply และ upload ผ่าน Step 11 Pricelist's `submission_method = email` และ activity log บันทึก manual entry Portal token ของ vendor ยังออกและไม่ใช้; auto-expire ที่สิ้นสุด campaign โดยไม่ส่งผลต่อ pricelist ที่มาจาก email
- **ต้องการการ revoke token mid-flight** ถ้า email ผู้ติดต่อของ vendor ถูก compromise หรือถ้า Compliance flag vendor รอ review Purchaser (พร้อมการ approve Manager) escalate ไปยัง Sysadmin ตาม `VPL_AUTH_015` เพื่อ revoke token การ revoke invalidate การเข้า portal ทันที; pricelist draft in-progress ใด ๆ คงที่ `draft` และสามารถ (a) ลบถ้าไม่มีข้อมูลที่กู้คืน หรือ (b) re-route โดยการออก invitation ใหม่พร้อม token ใหม่ใน campaign แยก
- **Pricelist active ต้องการการแก้** ตาม `VPL_VAL_025` pricelist active เป็น immutable ยกเว้น status เพื่อแก้แถว Purchaser inactivate pricelist (`VPL_POST_019`), เปิด campaign ใหม่ targeting vendor + template เดียวกัน vendor re-submit พร้อมการแก้ และ pricelist ที่แก้ approve ใหม่ Pricelist เก่า (inactive) ยัง queryable สำหรับ audit Cohort PR / PO / GRN ปลายน้ำที่อ้างอิง pricelist เก่าระหว่างการ activate และ inactivate รักษาโดย snapshot semantic ฝั่งโมดูลที่บริโภค ([[purchase-order/02-business-rules]] § `PO_XMOD_005`)
- **Mass close-out ที่สิ้นสุด campaign พร้อม straggler** เมื่อ campaign `end_date` ผ่านโดยมี invitation บางตัวยัง `pending` / `in-progress` Purchaser review laggard บน tab **Vendor Tracking** และตัดสินใจต่อ vendor: (a) ขยาย campaign สำหรับ vendor เฉพาะ (admin override), (b) ยอมรับ auto-expire และติดต่อ vendor สำหรับรอบครั้งหน้า, (c) cancel campaign ทั้งหมด (`VPL_POST_009`) และ re-launch ด้วย template ที่แก้ แต่ละ action ถูก log

## 4. จุดออก / Handoff

การมีส่วนร่วมของ Purchaser ใน pricelist / campaign ที่กำหนดจบที่หนึ่งในจุดที่บันทึก; สถานะเอกสารที่แต่ละ handoff anchor กับ lifecycle ใน [03-user-flow.md](./03-user-flow.md) § 2

- **Approve → active** Purchaser approve pricelist ที่ submit; `tb_pricelist.status = active` Handoff ไปยัง **ผู้บริโภคปลายน้ำ** — การ default บรรทัด PR (`VPL_XMOD_001`), snapshot PO (`VPL_XMOD_003`), variance GRN (`VPL_XMOD_005`) ความรับผิดชอบของ Purchaser ถ่ายไปยังการติดตาม dashboard **Active Pricelists** และ resume ที่การแก้, inactivation หรือการหมดอายุ
- **Escalate high-value หรือ multi-currency** เมื่อขีดจำกัดถูกเกิน Purchaser handoff ไปยัง **Purchasing Manager** (และ **Finance Manager** สำหรับ multi-currency) Sign-off ของ Manager / Finance Manager return file ไปยัง Purchaser สำหรับการคลิก activate สุดท้าย; หรือ Manager activate โดยตรงภายใต้สิทธิ์ของตน สถานะเอกสารที่ handoff: `draft + submitted_at IS NOT NULL`
- **Reject → vendor resubmission** Purchaser reject submission; handoff ไปยัง **Vendor** พร้อม portal token ต้นฉบับสำหรับ resubmission จนกว่า campaign `end_date` สถานะเอกสารที่ handoff: `draft + submitted_at = NULL` บวก `system` rejection comment จุดออก Vendor (ดู [03-user-flow-vendor.md](./03-user-flow-vendor.md)) ป้อน Purchaser กลับที่ Step 7 ข้างต้น
- **Inactivate → ปลายน้ำ historical-only** เมื่อ Purchaser inactivate pricelist active handoff ไปยัง **ผู้บริโภคปลายน้ำ** ซึ่งตอนนี้ปฏิบัติกับ pricelist เป็น historical-only Purchaser โดยทั่วไปติดตามโดย launch campaign ใหม่เพื่อ re-collect coverage; สถานะเอกสารที่ handoff: `inactive`
- **Cancel campaign** เมื่อ Purchaser (หรือ Manager / Sysadmin) cancel campaign mid-flight handoff ไปยัง **vendor ที่เชิญ** (ผ่าน email cancellation) และไปยัง **Auditor** สำหรับ review post-hoc portal token ทั้งหมด revoke สถานะเอกสารที่ handoff: campaign `cancelled`; pricelist draft ที่เขียนไปจนถึงตอนนี้คงที่ `draft` และอาจถูก soft-delete โดย Purchaser หรือเก็บไว้สำหรับ audit
- **การ escalation revoke token** เมื่อต้องการการ revoke token Purchaser handoff ไปยัง **System Administrator** ภายใต้ `VPL_AUTH_015` สถานะเอกสารที่ handoff: invitation ย้ายไป `expired` ทันทีที่การ revoke token; pricelist draft ที่เกี่ยวข้อง (ถ้ามี) คงที่ `draft` จนกว่าจะ soft-delete หรือ re-route
- **การสืบสวน variance จากปลายน้ำ** เมื่อ **Finance** หรือ **Receiver** raise variance เทียบกับบรรทัด GRN / invoice ที่ post Purchaser re-enter flow ที่ dashboard **Active Pricelists**, review หลักฐาน variance และเลือกระหว่าง (a) ยอมรับ variance เป็นภายในนโยบาย (Finance post PPV ตาม [[purchase-order/02-business-rules]] § `PO_XMOD_007`), (b) inactivate pricelist และ re-collect หรือ (c) ติดต่อ vendor สำหรับ credit note / amendment สถานะเอกสารที่ handoff: pricelist `active` (ไม่เปลี่ยน); entry variance log สำหรับ analytics

Auto-transition (`VPL_POST_021` expiry; `VPL_POST_014` invitation auto-expire) ไม่ใช่ action ของ Purchaser — fire จาก cron background และ Purchaser สังเกตบน dashboard
