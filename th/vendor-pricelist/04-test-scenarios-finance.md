# รายการราคาผู้ขาย (Vendor Pricelist) — Test Scenarios — Finance

> **At a Glance**
> **Persona:** Finance (Finance Officer / AP + Finance Manager) &nbsp;·&nbsp; **โมดูล:** [[vendor-pricelist]] &nbsp;·&nbsp; **Scenario:** ~26
> **Categories:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** ไม่มี — dashboard variance และ coverage co-signoff รันที่ระดับ API / integration รอ roadmap item; ไม่มี spec dedicated ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ test scenario ที่ persona Finance (Finance Officer / AP + Finance Manager) ขับโดยตรงในโมดูล `vendor-pricelist` Finance มี **surface เขียนที่ไม่มีบน pricelist เอง** — `VPL_AUTH_009` เป็นอ่านอย่างเดียวข้าม pricelist, campaign และ invitation การมีส่วนร่วมของ Finance Manager ในการ activate pricelist multi-currency / high-value จับเป็น `system` co-signoff comment ใน `tb_pricelist_comment` ไม่ใช่การเปลี่ยน status โดยตรง; flip `VPL_POST_017` บันทึกเทียบกับ Purchaser / Purchasing Manager Touch point หลักของ Finance คือ **การ audit variance หลังการรับ** — เมื่อการ post GRN / invoice ใน chain [[purchase-order]] ปลายน้ำ flag variance เทียบกับ pricelist active (ตาม `VPL_XMOD_005`) Finance สืบสวน, จัดประเภท และ (a) ยอมรับ variance ภายในขีดจำกัดด้วย entry PPV ฝั่ง AP, (b) route vendor over-bill หรือ pricelist out-of-date กลับไปยัง Purchaser สำหรับ action หรือ (c) attribute variance FX-only ไปยังการ post ข้ามสกุลเงิน Scenario จัดกลุ่มเป็น happy path ที่อธิบายใน [03-user-flow-finance.md](./03-user-flow-finance.md), ขอบเขต RBAC ที่แยก Finance Officer อ่านอย่างเดียวจากสิทธิ์ co-signoff ของ Finance Manager (`VPL_AUTH_010`) และจากข้อจำกัด no-direct-write-on-pricelist (`VPL_AUTH_009`), surface validation รอบการบังคับใช้นโยบาย FX / สกุลเงิน และชุดเล็ก ๆ ของ edge case รอบ race ของ variance และกรณีขอบเขตข้ามสกุลเงิน การ handoff ข้าม persona ที่ pivot จาก Finance (`X-VPL-03` co-signoff multi-currency, `X-VPL-06` audit variance GRN) อยู่ใน parent overview ไม่ใช่ที่นี่

| VPL-FIN-EDGE-02 | Variance ตรงขอบเขต tolerance พอดี | ราคา pricelist `฿50.00`; ราคา invoice `฿50.75`; ขีดจำกัด tenant ตรง `1.5%`; variance ตรง `1.5%` | การจัดการขอบเขตตามการตั้งค่า tenant: inclusive (variance `= tolerance` คือ within) หรือ exclusive (variance `= tolerance` คือ outside) Default inclusive — variance จัดประเภท `within-tolerance` อัตโนมัติฝั่ง PO ตาม `PO_POST_008`; dashboard ของ Finance Officer แสดงแถวพร้อม indicator ขอบเขตแต่ไม่ต้องการ action Tenant สามารถ flip เป็น exclusive ใน registry validation-rule `VPL_XMOD_009` ที่ Sysadmin ตั้งค่า |
| VPL-FIN-EDGE-03 | Race co-signoff multi-currency — Finance Manager และ Purchasing Manager save ที่ instant เดียวกัน | Pricelist multi-currency `PL-V-EUR` พร้อมสำหรับการ activate; ทั้ง Finance Manager และ Purchasing Manager คลิก Sign Off / Approve พร้อมกันภายใน 500 ms | Backend serialise ทั้งสอง signoff write โดย timestamp signoff ที่สองที่ลงเป็นตัวที่ trigger `VPL_POST_017` — ที่จุดนั้นทั้งสอง signoff มีอยู่ใน `tb_pricelist_comment` (ในลำดับใดลำดับหนึ่ง), การ activate fire และ pricelist transition เป็น `active` ไม่มี double-activation; `system` comment การ activate fire ครั้งเดียวพอดีโดย actor เป็น signoff ใดก็ตามที่ลงที่สอง |
| VPL-FIN-EDGE-04 | Variance FX-only ที่ข้ามขีดจำกัดการเปิดรับสะสม | Variance FX-only หลายตัวเหนือ 30 วันเทียบกับ vendor เดียวกัน; การเปิดรับ FX สะสม `THB`-equivalent เกินขีดจำกัด tenant (เช่น `฿1,000,000` สะสม FX loss ที่เกิดขึ้นจริง) | Dashboard ของ Finance Officer surface indicator การเปิดรับสะสม; แนะนำการ escalate ไปยัง Finance Manager สำหรับ review นโยบาย hedging out-of-band ไม่มี action บน pricelist รายตัว; ข้อค้นพบสะสม log ที่ระดับ vendor / period เป็น object audit Finance Manager review และอาจแนะนำการเปลี่ยนนโยบาย currency-hedging ไปยัง Sysadmin (flow แยกนอกโมดูลนี้) |
| VPL-FIN-EDGE-05 | Pricelist ไม่มี coverage สำหรับสินค้าบน PR | การสร้างบรรทัด PR ไม่พบ pricelist active สำหรับ `(V1, P1, THB)` และถูกสร้างด้วย `pricelist_type = manual_input` ตาม `VPL_XMOD_002`; ภายหลัง GRN post ที่ราคาของบรรทัดนั้น; dashboard variance ของ Finance Officer ไม่สามารถคำนวณ variance เพราะไม่มี anchor pricelist | Dashboard variance surface แถวพิเศษ: `"Pricelist coverage gap — no anchor for variance comparison."` Finance Officer route เคสไปยัง Purchasing Manager พร้อมการแนะนำ launch campaign สำหรับ `(V1, P1)` coverage ไม่มีแถว `tb_pricelist` ที่จะ comment บน; comment อยู่บน registry vendor / สินค้าแทน ทำให้แน่ใจว่า gap surface ใน cycle การวางแผน campaign ครั้งต่อไป |
| VPL-FIN-EDGE-06 | ความแม่นยำ Decimal บน FX-converted variance amount | PO `currency_id = USD`, มูลค่าบรรทัด = `$10,000.00`; invoice `$10,005.00` (variance 0.05 %); วัน PO FX `36.50`; วัน invoice FX `36.55`; ขีดจำกัด tenant `0.1%` | Variance คำนวณฝั่งสกุลเงิน invoice ก่อน (`$5.00 ÷ $10,000.00 = 0.05%`); ภายในขีดจำกัด — การจัดประเภท within-tolerance เดลตา FX `THB`-equivalent คือ `0.13%` (`(36.55 − 36.50) ÷ 36.50`) จัดประเภทแยกเป็น `FX-only` ทั้งสองด้านติดตามแยก; ไม่มี double-counting AP post FX gain / loss ที่เกิดขึ้นจริง; PPV คือ `$5.00` × `36.55` = `฿182.75` |
