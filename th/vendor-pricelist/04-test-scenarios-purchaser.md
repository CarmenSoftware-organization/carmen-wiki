# รายการราคาผู้ขาย — Test Scenarios — Purchaser (Vendor Pricelist — Test Scenarios — Purchaser)

> **At a Glance**
> **Persona:** Purchaser (Purchasing Staff + Purchasing Manager) &nbsp;·&nbsp; **โมดูล:** [[vendor-pricelist]] &nbsp;·&nbsp; **Scenario:** ~39
> **Categories:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** ไม่มี vendor-pricelist spec dedicated; wizard From-Price-List exercise ผ่าน `tests/402-po-purchaser-journey.spec.ts` (TC-PO-060205..060208) ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ test scenario ที่ persona Purchaser (รวบ Purchaser / Purchasing Staff + Purchasing Manager ตาม [03-user-flow-purchaser.md](./03-user-flow-purchaser.md)) ขับโดยตรงในโมดูล `vendor-pricelist` Purchaser เป็นเจ้าของ lifecycle เชิงปฏิบัติการเต็มฝั่ง Carmen: template create / edit / activate, campaign launch / pause / cancel, การออก invitation, การ upload manually pricelist ที่ email-submit, การ review และ approve / reject pricelist ที่ submit, การ curate preferred-vendor และการ inactivate Role Purchasing Manager บนไฟล์ persona เดียวกันขยายด้วยสิทธิ์ high-value approval, การตั้งค่า business-rule และ multi-currency sign-off Scenario จัดกลุ่มเป็น happy path ที่อธิบายใน user flow, ขอบเขต RBAC บังคับใช้โดย `VPL_AUTH_001`–`VPL_AUTH_006` (ขอบเขต Purchaser) เทียบกับ `VPL_AUTH_005`–`VPL_AUTH_006` (การ elevate Manager), กติกา validation ใน [02-business-rules.md](./02-business-rules.md) § 2 (`VPL_VAL_001`–`VPL_VAL_025`) ที่ Purchaser สามารถ trigger ข้าม surface template / campaign / pricelist และชุดเล็ก ๆ ของ edge case รอบ concurrency, ขอบเขต MOQ, ค่าขอบเขตสกุลเงิน และ snapshot semantic ระหว่างการ inactivate การ handoff ข้าม persona ที่ pivot จาก Purchaser (`X-VPL-01`, `X-VPL-02`, `X-VPL-03`, `X-VPL-04`, `X-VPL-05`, `X-VPL-09`, `X-VPL-10`, `X-VPL-12`) อยู่ใน parent overview ไม่ใช่ที่นี่

## 2. Permission / Authorization

| # | Scenario | พฤติกรรมที่คาด (allow/deny + reason) |
| - | -------- | --------------------------------------- |
| VPL-PUR-PERM-01 | Purchaser สร้าง / แก้ template ที่ตนเป็นเจ้าของ (`draft` หรือ `active`) | **Allow** ตาม `VPL_AUTH_001` Surface template แสดง Save, Edit, Activate, Inactivate Role Manager inherit สิทธิ์ Purchaser ทั้งหมดบวก authority ที่ elevate ด้านล่าง |
| VPL-PUR-PERM-02 | Purchaser launch campaign ที่อ้างอิง template `inactive` | **Deny** ตาม `VPL_VAL_009` Server return `"Campaign must reference an active template — draft or inactive templates cannot drive campaigns."` UI ซ่อน template `inactive` จาก picker template ของ campaign; การเรียก API ตรงถูกจับโดย validation |
| VPL-PUR-PERM-03 | Purchaser approve pricelist ต่ำกว่าขีดจำกัด high-value | **Allow** ตาม `VPL_AUTH_004` `VPL_POST_017` fire; pricelist activate |
| VPL-PUR-PERM-04 | Purchaser พยายาม approve pricelist high-value (เหนือขีดจำกัด tenant) | **Deny — Manager-only** ตาม `VPL_AUTH_005` การ approve high-value ต้องการ role Purchasing Manager ปุ่ม Approve ถูก disable ใน UI ของ Purchaser; การเรียก API ตรง return `"High-value pricelist activation requires the Purchasing Manager role."` Pricelist route ไปยัง queue Manager review |
| VPL-PUR-PERM-05 | Purchaser พยายาม approve pricelist multi-currency โดยไม่มี Finance Manager co-signoff | **Deny — ต้องการ co-approval** ตาม `VPL_AUTH_010` การ activate multi-currency ต้องการ `system` co-signoff comment ของ Finance Manager บน `tb_pricelist_comment` ก่อน `VPL_POST_017` fire ได้ UI surface ความต้องการ co-signoff; การเรียก API ตรง return `"Multi-currency pricelist requires Finance Manager co-signoff before activation."` |
| VPL-PUR-PERM-06 | Purchaser พยายามแก้แถวบน pricelist `active` | **Deny** ตาม `VPL_VAL_025` pricelist active เป็น immutable ยกเว้น status Server return `"Active pricelist is immutable — open a new pricelist via a fresh campaign, or move this pricelist to inactive first."` UI แสดงเฉพาะ **Inactivate** + การเพิ่ม comment บน pricelist active; การแก้ API ตรงถูกบล็อก |
| VPL-PUR-PERM-07 | Purchaser พยายาม revoke portal token ของ vendor โดยตรง | **Deny — Sysadmin / Manager only** ตาม `VPL_AUTH_015` การ revoke token สงวนไว้สำหรับ System Administrator (หรือ Purchasing Manager ผ่านการ escalation) UI ไม่แสดง action Revoke ต่อ Purchaser; เส้นทางของ Purchaser คือการ escalate ไปยัง Sysadmin / Manager พร้อมเหตุผล การเรียก API ตรง return `"Token revocation requires the System Administrator or Purchasing Manager role."` |
| VPL-PUR-PERM-08 | Purchaser upload pricelist ที่ email ในนาม vendor | **Allow** ตาม `VPL_AUTH_003` การ upload เขียน `submission_method = email` และ `system` comment บังคับบันทึกแหล่ง email และ identity staff ที่ upload |
| VPL-PUR-PERM-09 | Purchaser พยายาม bypass การแยกหน้าที่ (user เดียวกันสร้างแถว pricelist AND approve ที่ high-value) | **Deny** ตาม `VPL_AUTH_014` user ที่แก้แถว detail ของ pricelist high-value อย่างมีนัยสำคัญต้องไม่เป็น user เดียวกันที่ approve มัน บังคับใช้ที่การเปลี่ยน `VPL_POST_017` Server return `"Segregation of duties: cannot approve a high-value pricelist you have substantially edited."` |

## 3. Validation / Error

| # | Scenario | Trigger | Error ที่คาด |
| - | -------- | ------- | -------------- |
| VPL-PUR-VAL-01 | สร้าง template ที่มีชื่อซ้ำ | หยิบ `name` ที่มีอยู่บนแถว `tb_pricelist_template` ที่ไม่ถูก soft-delete | Reject — `VPL_VAL_001` Server return `"Template name is required and must be unique."` Fallback ระดับ DB ผ่าน `pricelist_template_name_deletedat_u` |
| VPL-PUR-VAL-02 | Activate template ที่ไม่มีแถวสินค้า | เปิด template draft ที่ count `tb_pricelist_template_detail` = 0; คลิก Activate | Reject — `VPL_VAL_002` Server return `"Template must contain at least one product before it can be activated."` ปุ่ม UI Activate ถูก disable จนกว่าจะเพิ่มสินค้าอย่างน้อยหนึ่ง |
| VPL-PUR-VAL-03 | Save แถว template detail ที่ `MOQ qty = 0` หรือมีค่า MOQ ลดลง | เพิ่มสินค้าด้วย `moq: [{qty:50}, {qty:1}]` (ลดลง) หรือ `[{qty:0}, {qty:50}]` | Reject — `VPL_VAL_006` Server return `"Order unit and at least one MOQ tier with positive quantity are required; MOQ quantities must be strictly increasing."` |
| VPL-PUR-VAL-04 | Launch campaign ที่ `start_date >= end_date` | ตั้ง `start_date = 2026-06-15`, `end_date = 2026-06-10`; คลิก Launch | Reject — `VPL_VAL_010` Server return `"Campaign window must run for at least the tenant minimum (default 3 days) and end strictly after start; end_date must be in the future at launch."` |
| VPL-PUR-VAL-05 | Launch campaign ที่ไม่มี vendor เลือก | เปิด New Campaign, กรอก header, เลือก vendor zero, คลิก Launch | Reject — `VPL_VAL_011` Server return `"Campaign must invite at least one vendor and every invited vendor must have a contact email."` |
| VPL-PUR-VAL-06 | เพิ่ม vendor เดียวกันสองครั้งใน campaign | เลือก vendor `V1` แล้วลองเพิ่ม `V1` อีกครั้งผ่าน vendor picker | Reject — `VPL_VAL_012` Server return `"Vendor has already been invited to this campaign."` UI ป้องกันการเลือกซ้ำ; fallback ระดับ DB ผ่าน `request_for_pricing_detail_request_for_pricing_id_vendor_id_u` |
| VPL-PUR-VAL-07 | Approve pricelist ที่ submit ที่แถวมี `price < price_without_tax + tax_amt` | บังคับแถวที่ `price = ฿10.00` แต่ `price_without_tax = ฿10.00` และ `tax_amt = ฿0.70`; พยายาม approve | Reject — `VPL_VAL_021` Server return `"Price-without-tax must be non-negative; tax_amt and price must reconcile to the configured rounding rule (5 dp internal, 2 dp display)."` Pricelist return ไป vendor สำหรับการแก้ (หรือ Purchaser แก้ inline ภายใต้ `VPL_AUTH_001`) |
| VPL-PUR-VAL-08 | Submit pricelist ที่มีราคา MOQ-tier ลดลง (MOQ ที่สูงกว่าที่ราคาต่อหน่วยสูงกว่า) | Vendor (หรือ Purchaser upload ในนาม) submit สินค้า `P1` พร้อมสามแถว: `MOQ 1 @ ฿10.00`, `MOQ 50 @ ฿11.00`, `MOQ 100 @ ฿12.00` | Reject ที่ submit — `VPL_VAL_020` Server return `"MOQ-tier pricing must be non-increasing as MOQ quantity increases."` Save (เทียบกับ submit) ส่ง warning; submit บล็อกจนกว่าจะแก้ |
| VPL-PUR-VAL-09 | Submit pricelist ที่ `effective_to_date <= effective_from_date` | ตั้ง `effective_from_date = 2026-07-01`, `effective_to_date = 2026-06-30`; คลิก submit | Reject — `VPL_VAL_016` Server return `"Effective-from date must precede effective-to date; effective-from cannot precede submission."` |
| VPL-PUR-VAL-10 | Submit pricelist ที่ `currency_id` ไม่อยู่ในรายการที่ tenant อนุญาต | Vendor's portal session เลือก `BTC` (หรือสกุลเงินอื่นที่ไม่อนุญาต) | Reject — `VPL_VAL_015` + `VPL_AUTH_008` Server return `"Vendor and currency are required and must reference active master-data records."` (ข้อความ generic; แอป surface "Currency not permitted for tenant" ให้ vendor inline) |
| VPL-PUR-VAL-11 | Approve pricelist ที่ submit ผ่าน API ตรงขณะ `submitted_at IS NULL` | Vendor save draft แต่ไม่เคยคลิก Submit; บังคับการเรียก API ตรงเพื่อ approve | Reject — `VPL_VAL_023` + state guard implicit Server return `"Pricelist must contain at least one valid line item to submit."` (UI ไม่เคยเสนอ Approve บน draft ที่ไม่ submit; guard API คือ fallback) |
| VPL-PUR-VAL-12 | Inactivate pricelist ที่อ้างอิงโดย chain fallback `manual_input` บรรทัด PR in-flight | Pricelist active `PL-V1`; PR ปลายน้ำมีบรรทัดที่ `pricelist_detail_id` ชี้ไปที่ `PL-V1`; พยายาม inactivate ขณะ PR อยู่ระหว่างการ approve | **Allow** การ inactivate (snapshot semantic ปกป้องบรรทัด PR) Pricelist transition `active → inactive` ตาม `VPL_POST_019`; บรรทัด PR คงราคา snapshot; เฉพาะ **บรรทัด PR ใหม่** จาก moment inactivation ไปข้างหน้าตกผ่าน chain fallback Validation ไม่บล็อกกรณีนี้; activity log บันทึกการ inactivate อย่างสะอาด |

## 4. Edge Cases

| # | Scenario | เงื่อนไข | คาด |
| - | -------- | --------- | -------- |
| VPL-PUR-EDGE-01 | การแก้พร้อมกันบน draft template เดียวกันจากสอง Purchaser session | Purchaser A และ B โหลด draft template `T1` ที่ `doc_version = 3`; A save การเพิ่มสินค้าที่ `T` (`doc_version → 4`); B save การ rename ที่ `T + 1s` ใช้ `doc_version = 3` ที่ stale | Save แรกชนะ: การเพิ่มของ A persist, `doc_version = 4` Save ของ B ถูก reject โดย guard optimistic-concurrency ด้วย `"This template was modified by another user. Please refresh and re-apply your changes."`; client ของ B prompt refresh ไม่มี lost update |
| VPL-PUR-EDGE-02 | ความแม่นยำ decimal สูงสุดบน `price` แถว | แถว pricelist detail พร้อม `price_without_tax = 99999999999.99999` (15 + 5 หลัก, ขอบบนของ `Decimal(20, 5)`), `tax_rate = 0.07000`, `moq_qty = 1` | ยอมรับที่การ store `tax_amt = Round(99999999999.99999 × 0.07, 5)` ตาม `VPL_CALC_001`; `price = Round(price_without_tax + tax_amt, 5)` ตาม `VPL_CALC_002`; Decimal ที่เก็บทั้งหมด fit ภายในความแม่นยำที่ประกาศ; การแสดงผลปัดเศษเป็น 2 dp ตอน render Effective unit price ต่อ UoM ฐานคำนวณถูกต้องผ่าน `VPL_CALC_003` |
| VPL-PUR-EDGE-03 | กรณีขอบเขต MOQ-tier — ราคาเท่ากันข้ามสอง tier ที่ติดกัน | Vendor submit `MOQ 1 @ ฿10.00` และ `MOQ 50 @ ฿10.00` (ราคาเดียวกัน; vendor ยังไม่เสนอ bulk discount แต่ผูกพันกับ MOQ ที่ใหญ่กว่า) | ยอมรับ — `VPL_VAL_020` ต้องการ non-increasing (`price[i+1] <= price[i]`) ไม่ใช่ลดลงเข้มงวด สองแถวที่ติดกันทั้งคู่ผ่านที่ `price[2] == price[1]` Surface Purchaser แสดงเป็น annotation "no bulk discount yet"; การ lookup preferred-vendor ของบรรทัด PR หยิบแถวใดแถวหนึ่งขึ้นกับ bracket MOQ |
| VPL-PUR-EDGE-04 | การ re-activate pricelist ที่หมดอายุภายใน window | Pricelist `PL-V1-Q1` auto-expire ที่ `effective_to_date = 2026-04-30` (`status = expired`); ที่ 2026-04-15 (ยังภายใน window) Sysadmin / Purchaser re-activate เพราะ auto-expire fire ก่อนเวลา | การ re-activate `VPL_POST_020` อนุญาตภายใน window เท่านั้น; server check `now() < effective_to_date` ตอนสำเร็จ `status = expired → active` (transition พิเศษไม่อยู่ใน state diagram มาตรฐาน); `system` comment จับสาเหตุและ actor นอก window การ re-activate ถูก reject ด้วย `"Cannot re-activate an expired pricelist outside its validity window — open a new pricelist via a fresh campaign."` |
| VPL-PUR-EDGE-05 | Soft-delete บน draft pricelist พร้อม portal session ยังเปิด | Vendor's portal มี draft in-progress ที่ `tb_pricelist.status = draft`; Purchaser soft-delete pricelist (เช่น ในส่วนของการ cleanup campaign cancellation) | Soft-delete `VPL_POST_022` อนุญาต (status เป็น `draft`) `deleted_at = now()`, `deleted_by_id = user` การ save ครั้งต่อไปของ vendor's portal return `404 — pricelist not found`; portal redirect ไปหน้า "session terminated" อธิบายสถานการณ์ Unique index รวม `deleted_at` ดังนั้น `pricelist_no` เดียวกันสามารถใช้ซ้ำสำหรับ pricelist สดภายใต้ campaign ที่ relaunch |
| VPL-PUR-EDGE-06 | Race ตัวเลขบนการ toggle `is_preferred` ข้ามสอง cell ที่ส่งผลต่อสินค้าเดียวกัน | Purchaser A toggle preferred สำหรับ `(P1, THB, MOQ 1)` จาก `PL-V1` ไป `PL-V2`; พร้อมกัน Purchaser B toggle preferred สำหรับ `(P1, THB, MOQ 50)` จาก `PL-V1` ไป `PL-V3` | การแก้ทั้งสองดำเนินต่อโดยอิสระเพราะปฏิบัติการบนแถว `tb_pricelist_detail` ที่ต่างกัน (`moq_qty` ต่างกัน) Invariant one-preferred-per-cell รักษาต่อ cell ไม่ใช่ข้าม MOQ tier ของสินค้าเดียวกัน `PL-V1.is_preferred = false` บนทั้งสองแถวที่ได้รับผลกระทบหลังการ save สอง Activity log บันทึก actor ทั้งสอง |
| VPL-PUR-EDGE-07 | การใช้ pricelist numbering uniqueness ซ้ำหลัง soft-delete | Pricelist ก่อนหน้า `PL-2026-001` ถูก soft-delete (`deleted_at IS NOT NULL`); pricelist ใหม่ถูกสร้างและ service การกำหนดเลขออก `pricelist_no = PL-2026-001` เดียวกัน | ยอมรับ — unique index `pricelist_pricelist_no_u` เป็น `@@unique([pricelist_no, deleted_at])` Pricelist ใหม่ save และ submit ปกติ; แถวที่ soft-delete ก่อนหน้ายังคงใน database สำหรับ audit |
| VPL-PUR-EDGE-08 | การ switch registry email-template mid-campaign | Sysadmin เปลี่ยน default email-template ของ tenant ที่ `T`; campaign launch ที่ `T - 1 hour` อ้างอิง id template เก่าตอนนี้; reminder email fire ที่ `T + 24 hours` | Campaign คง **snapshot** `email_template_id` ที่ launch ตาม snapshot semantic ([[vendor-pricelist/03-user-flow-audit-config]] § 4 exit) Reminder email ดำเนินต่อใช้ template ที่ snapshot; การ switch ของ Sysadmin ใช้กับ campaign ใหม่ที่ launch หลัง `T` เท่านั้น Audit log บันทึก snapshot vs current divergence |
