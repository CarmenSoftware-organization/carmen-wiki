# รายการราคาผู้ขาย — Test Scenarios — Audit / Config (Vendor Pricelist — Test Scenarios — Audit / Config)

> **At a Glance**
> **Persona:** Audit / Config (Auditor อ่านอย่างเดียว + System Administrator การตั้งค่า) &nbsp;·&nbsp; **โมดูล:** [[vendor-pricelist]] &nbsp;·&nbsp; **Scenario:** ~34
> **Categories:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** ไม่มี — surface การ query audit-log และการตั้งค่า exercise ที่ระดับ API / integration รอ roadmap item; ไม่มี spec dedicated ใน `../carmen-inventory-frontend-e2e/`

หน้านี้จับ test scenario ที่แกน persona **Audit / Config** ขับโดยตรงในโมดูล `vendor-pricelist` สอง role ที่แตกต่างอยู่บนแกนนี้: **Auditor** (อ่านอย่างเดียวข้าม chain การตั้งราคา procurement เต็ม — Template → Campaign → Invitation → Vendor Portal Submission → Pricelist Approval → การบริโภค PR / PO / GRN ปลายน้ำ) ที่ใช้ activity log ของแต่ละ tier, สถานะ campaign / invitation ที่ derive ระดับแอป, output validation engine ที่เก็บใน `tb_pricelist.info`, telemetry portal-token ที่เขียนเป็น `system` comment บนแถว invitation และ bridge ข้ามเอกสารเพื่อตรวจสอบ compliance นโยบาย, การแยกหน้าที่ (`VPL_AUTH_014`), ความถูกต้องของ validation-engine และ traceability end-to-end; และ **System Administrator** (การตั้งค่าเท่านั้น) ที่เป็นเจ้าของ scheme การกำหนดเลข pricelist, map role-to-permission RBAC สำหรับ `VPL_AUTH_001`–`VPL_AUTH_015`, นโยบาย portal-token, การเดิน wiring email-integration, registry กติกา-validation, แหล่ง currency / FX ที่บริโภคโดย Finance, นโยบายการเก็บ audit-log และสิทธิ์ revoke token ต่อ invitation (`VPL_AUTH_015`) ทั้งสอง role ไม่ transition pricelist ข้าม `enum_pricelist_status`: Auditor ออกผ่านรายงานที่สร้าง (ไม่มีการเปลี่ยนสถานะเอกสาร), Sysadmin ออกผ่านการตั้งค่าที่ save พร้อม timestamp `effective_from` (เอกสารที่มีอยู่คงการตั้งค่าที่ snapshot; เอกสารใหม่ใช้การตั้งค่าใหม่) Touch ตรงเดียวของ Sysadmin บนเอกสารธุรกรรมคือการ revoke token ซึ่ง mutate `tb_request_for_pricing_detail.pricelist_url_token` และ append `system` comment แต่ **ไม่** เปลี่ยน `status` ของ pricelist Scenario จัดกลุ่มเป็น happy path ที่อธิบายใน [03-user-flow-audit-config.md](./03-user-flow-audit-config.md), ขอบเขต RBAC ที่แยก read-only-audit จาก configure-only-sysadmin จาก no-direct-transact, กติกา validation รอบการรักษา snapshot, การ approve export และ race ของการตั้งค่า และชุดเล็ก ๆ ของ edge case รอบขอบเขต in-flight, rollback, การป้องกัน deadlock และ registry กติกา-validation ไม่มี audit-config E2E spec dedicated ขณะนี้; coverage อยู่ที่ระดับ API / integration รอ roadmap item ใน `../carmen/docs/vendor-pricelist-management/tasks.md`

## 4. Edge Cases

| # | Scenario | เงื่อนไข | คาด |
| - | -------- | --------- | -------- |
| VPL-AUD-EDGE-01 | ขอบเขต snapshot in-flight — config save ตรงที่ submission | Pricelist `PL-SB-01` อยู่ที่ `submitted_at = NULL`; vendor คลิก Submit ที่ `T`; Sysadmin save การเปลี่ยนกติกา validation ด้วย `effective_from = T` (millisecond เดียวกัน) | Race resolve โดยการเปรียบเทียบ `effective_from` atomic ถ้า timestamp การเขียน `VPL_POST_016` ของ `PL-SB-01` น้อยกว่า `effective_from` อย่างเข้มงวด submission จับ snapshot กติกา validation เก่า; ถ้า `>= effective_from` submission รันกติกาใหม่ Entry แรกของ activity log `tb_pricelist_comment` ของ pricelist บันทึกเวอร์ชันกติกา validation ที่ผูก ดังนั้น chain audit ไม่กำกวม Log audit การตั้งค่าบันทึก count submission in-flight ที่ขอบเขต |
| VPL-AUD-EDGE-02 | Rollback ของ config การเปลี่ยนที่ไม่ดี | Sysadmin save config RBAC ใหม่ที่ `T1`; การ verify พบว่า role approver ที่ต้องการถูกลบโดยอุบัติเหตุ; Sysadmin revert ไปเวอร์ชันก่อนที่ `T2 > T1` | Rollback เองเป็น save config ด้วย `effective_from = T2` Pricelist ที่สร้างระหว่าง `T1` และ `T2` ไม่ถูก re-evaluate ย้อนหลัง (snapshot semantic); ดำเนินต่อภายใต้ snapshot broken-RBAC — โดยทั่วไป rescue ผ่าน delegation ครั้งเดียวหรือ patch RBAC ฉุกเฉิน Pricelist ที่สร้างหลัง `T2` ใช้การตั้งค่าที่ revert Log audit การตั้งค่าคงทั้งสองเวอร์ชัน |
| VPL-AUD-EDGE-03 | Gap audit-log — system event ขาดบน activation | Auditor เดิน timeline ของ `PL-GAP-01`; พบ `tb_pricelist_comment` มี entry submitted-at ที่ `T0` และ entry activation ที่ `T2` แต่ MISSING การเขียน validator-output ที่ `T1` (ซึ่งควรอยู่ระหว่างพวกมัน) | Gap เองเป็นข้อค้นพบ audit Auditor flag `PL-GAP-01` ในไฟล์เคสด้วยโน้ต "missing validator-output system entry", แนบ excerpt timeline + count comment validator ที่คาด และ route ไปยัง Sysadmin สำหรับการสืบสวน สาเหตุที่เป็นไปได้: การ corrupt replay-log, code path bypass validator หรือ comment ที่ถูก soft-delete โดยอุบัติเหตุ Auditor **ไม่** ปิดเคสเป็น "no findings"; เคสอยู่เปิดจนกว่า gap จะอธิบาย |
| VPL-AUD-EDGE-04 | Event การ revoke token จำนวนมาก | Sysadmin ค้นพบ incident การ takeover email-account ที่ส่งผลต่อ vendor 20+ รายในแคมเปญ active; ต้องการ revoke token ที่ได้รับผลกระทบทั้งหมด | Sysadmin ใช้ action bulk-revoke: filter แถว invitation campaign โดย `contact_email IN (compromised-list)`; คลิก **Revoke selected** การ revoke แต่ละครั้งเขียน entry `system` ของ `tb_request_for_pricing_detail_comment` ของตนและ count ไปยัง log audit การตั้งค่า Atomic — การ revoke 20+ ทั้งหมดสำเร็จหรือไม่มีเลย (transactional) ตาม `VPL_AUTH_015` second-factor ต้องการครั้งเดียวสำหรับ action bulk Log audit การตั้งค่าบันทึก event bulk พร้อมการอ้างอิง token 20+ ทั้งหมด |
| VPL-AUD-EDGE-05 | Query timeout บน dataset ขนาดใหญ่ | Auditor ตั้งช่วงวันที่ = `2024-01-01 .. 2026-05-15` (29 เดือน) + ไม่มี filter อื่น; tenant มี pricelist 50,000+ ใน scope | Query hit limit scan ช่วงวันที่ (VPL-AUD-VAL-04 ป้องกันเคสว่าง แต่ single-filter 29 เดือนยังเสี่ยง timeout) ระบบ chunk query — return 1,000 แถวแรก + cursor "Continue to next page", persist query plan, รันที่เหลือแบบ async Auditor สามารถ page synchronously หรือ schedule export full-extract (ซึ่งจากนั้นเข้า flow การ approve export VPL-AUD-HP-05 / VPL-AUD-VAL-02) ไม่มี error timeout; ประสบการณ์ degrade อย่างนุ่มนวล |
| VPL-AUD-EDGE-06 | Config save vs race submission in-flight | Sysadmin คลิก Save บนกติกา validation ใหม่ที่ `T`; พร้อมกัน vendor 5 รายคลิก Submit บน portal session ของพวกเขาใน window ที่ติดกัน; ทั้ง 6 action arrive ที่ backend ภายใน window 500 ms เดียวกัน | Backend serialise โดย timestamp `effective_from` บนแถว config และ timestamp การเขียน `VPL_POST_016` บนแต่ละ pricelist Submission แต่ละตัว tag atomic ด้วยเวอร์ชัน validation-rule-set ที่ active ที่ instant ของการเขียน `VPL_POST_016` ของมัน ของ 5 submission บางตัวอาจจับชุดกติกาเก่าและบางตัวจับใหม่; entry แรกของ `tb_pricelist_comment` ของแต่ละ pricelist บันทึกชัดเจนว่า WHICH version มันผูก ไม่มี submission จบใน state hybrid; ไม่มี submission ถูก reject เป็น "rule set in transition" |
| VPL-AUD-EDGE-07 | การ review ย้อนหลังของกติกา validation | Sysadmin เพิ่มกติกา LEAD_TIME_CAP ที่ `T`; หกสัปดาห์ต่อมา Auditor review submission ล่าสุดและสังเกตว่ากติกาถูก bypass โดยอุบัติเหตุบนสาม submission (comment ของ validator แสดงว่ากติกาไม่ถูก evaluate) | ข้อค้นพบ Audit: การ include rule registry ถูกต้องแต่ logic การเลือกของ validator มีบั๊กที่ exclude กติกาสำหรับการรวม vendor-category เฉพาะ Auditor flag สาม submission, file โน้ตไฟล์เคส และ route ไปยัง Engineering ผ่าน Sysadmin Pricelist ที่ได้รับผลกระทบสามตัวคงที่สถานะปัจจุบัน; ไม่มีการ invalidate ย้อนหลัง การ fix กติกา deploy และ validate; ไฟล์เคส audit คง trail หลักฐาน |
