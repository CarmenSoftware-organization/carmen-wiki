# รายการราคาผู้ขาย — User Flow — Audit / Config (Vendor Pricelist — User Flow — Audit / Config)

> **At a Glance**
> **Persona:** Auditor + System Administrator &nbsp;·&nbsp; **โมดูล:** [[vendor-pricelist]] &nbsp;·&nbsp; **Workflow stage:** off-path — สังเกต chain Template / Campaign / Invitation / Pricelist; Sysadmin ตั้งค่านโยบาย &nbsp;·&nbsp; **สิทธิ์สำคัญ:** audit/อ่านข้าม chain + PR/PO/GRN ปลายน้ำ (Auditor); การกำหนดเลข, RBAC, นโยบาย portal-token, กติกา validation, การ revoke token (Sysadmin)
> **สิ่งที่ persona นี้ทำ:** Auditor ตรวจสอบ trail การตั้งราคา procurement end-to-end และ SoD; Sysadmin เป็นเจ้าของการกำหนดเลข, RBAC, นโยบาย token, การ validate, การ integrate และการเก็บ audit

## 1. Role ในโมดูลนี้

แกน persona **Audit / Config** จัดกลุ่มสอง role ที่แตกต่างที่ทั้งคู่อยู่ **นอกเส้นทาง happy path เชิงธุรกรรม** ของโมดูล `vendor-pricelist` แต่จำเป็นต่อการกำกับและความเป็นปฏิบัติการของมัน **Auditor** เป็น persona อ่านอย่างเดียวที่ความสนใจครอบคลุม **chain การตั้งราคา procurement end-to-end** — Template → Campaign → Invitation → Vendor Portal Submission → Pricelist Approval → การบริโภค PR / PO / GRN ปลายน้ำ — และใช้ activity log ของแต่ละ tier (`tb_pricelist_template_comment`, `tb_request_for_pricing_comment`, `tb_request_for_pricing_detail_comment`, `tb_pricelist_comment`, `tb_pricelist_detail_comment`), สถานะ campaign / invitation ที่ derive ระดับแอป ([[vendor-pricelist/02-business-rules]] § 5.2, § 5.3), output validation engine ที่เก็บใน `tb_pricelist.info.validation_results` + `quality_score`, telemetry portal-token ที่เขียนเป็น `system` comment บนแถว invitation และ bridge ข้ามเอกสาร (PR `pricelist_detail_id` back-reference, PO snapshot, entry GRN variance) เพื่อตรวจสอบ compliance นโยบาย, การแยกหน้าที่ (`VPL_AUTH_014`: Vendor ≠ Purchaser, high-value Purchaser ≠ Approver), ความถูกต้องของ validation-engine, posture ความปลอดภัย token (`VPL_AUTH_007`) และ traceability จากใบเสนอราคา vendor ผ่านไปยังตำแหน่งที่วางบิล Auditor มี **surface เขียนที่ไม่มีในโมดูล**: ไม่สามารถ approve, reject, inactivate, แก้แถว detail, revoke token หรือเปลี่ยนการตั้งค่า **System Administrator** เป็น persona ตั้งค่าที่เป็นเจ้าของ **surface นโยบายและ integration** สำหรับโมดูล — scheme การกำหนดเลข pricelist ที่ขับ `tb_pricelist.pricelist_no`, map role-to-permission RBAC สำหรับ `VPL_AUTH_001`–`VPL_AUTH_015`, นโยบาย portal-token (default expiration, IP allowlist default, ขีดจำกัด concurrent-session, การติดตาม IP), การเดิน wiring email-integration (SMTP / provider transactional-email, registry email-template, การจัดการ bounce), registry กติกา-validation ที่บริโภคโดย validation engine (`VPL_XMOD_009`), แหล่งอัตรา FX ของ tenant และรายการสกุลเงินที่อนุญาตที่บริโภคโดย Finance (`VPL_CALC_005`, `VPL_XMOD_008`), นโยบายการเก็บ audit-log และ — เมื่อต้องการ — สิทธิ์ **revoke token** (`VPL_AUTH_015`) ที่ invalidate การเข้า portal ของ vendor ทันที ทั้งสอง role ไม่อยู่บนเส้นทาง happy path create-to-activate; แต่ละตัวมีจุดเข้า, surface และ semantic การออกของตน: Auditor ออกผ่านรายงานที่สร้างขึ้นโดยไม่มีการเปลี่ยนสถานะ pricelist, Sysadmin ออกผ่านการตั้งค่าที่บันทึก (หรือ token ที่ revoke) ที่มีผลสำหรับ pricelist / campaign / invitation ใหม่ในขณะที่รักษา snapshot semantic สำหรับสิ่งที่อยู่ในการบิน

## 2. จุดเข้าและ Primary Flow

ทั้งสอง role มีจุดเข้าและ flow แยก; แต่ละตัวจัดการด้านล่าง

### 2.1. Flow Auditor

**จุดเข้า:** Sidebar → workspace **Audit** → **Pricelist Activity Queries** (หรือเมื่อเริ่มจากเอกสารที่ทราบ Sidebar → workspace **Vendor Management** → เปิด Template / Campaign / Pricelist → tab **Activity Log** / tab **Related Documents**) Auditor ลงบน surface query-builder ที่ scope ไปยังตระกูลเอกสาร pricelist-tier (template / campaign / invitation / pricelist) บวก chain ข้ามเอกสารปลายน้ำ (PR → PO → GRN → invoice) ไม่ใช่บน queue Submitted-awaiting-review หรือ Active Pricelists ที่ Purchaser ใช้

**Primary flow (happy path — Auditor):**

1. จาก **Audit → Pricelist Activity Queries** เลือก template query audit (เช่น "campaign ทั้งหมดที่ launch ใน period", "pricelist ทั้งหมดที่ approve โดย buyer", "invitation ที่หมดอายุโดยไม่มี submission", "การเบี่ยงเบน pricelist เกินขีดจำกัดตาม `VPL_XMOD_005`", "การกระจาย quality-score ตาม vendor", "event การ revoke token", "การละเมิดการแยกหน้าที่") หรือสร้าง query ad-hoc กับตาราง comment, JSON `info` และ bridge ข้ามเอกสาร
2. ใช้ **filter**: ช่วงวันที่ (`created_at`, `submitted_at`, `effective_from_date`, `effective_to_date`), vendor, สกุลเงิน, business unit, สถานะ (template `status`, campaign / invitation status ที่ derive, pricelist `status`), วิธีการ submission, band quality-score, ประเภทความล้มเหลว validation Chip filter render เหนือตารางผลลัพธ์; ชุด filter ว่างถูก reject เพื่อป้องกัน scan unbounded
3. Review **ชุดผลลัพธ์**: แต่ละแถวคือเอกสารหนึ่ง (template, campaign, invitation หรือ pricelist) พร้อมฟิลด์สำคัญ, state ปัจจุบัน, actor ล่าสุด และข้อเท็จจริง audit ที่เกี่ยวข้องกับ query (เช่น actor การ approve, เหตุผลการ reject, ขนาดการเบี่ยงเบน) เรียงตามคอลัมน์ใด ๆ; คลิกเข้าแถวเพื่อ drill เข้า **chain เต็ม** สำหรับเอกสารนั้น
4. บนหน้า drill-down เดิน **chain ไปข้างหน้า** — จาก template ผ่าน campaign ที่ใช้, ผ่าน invitation ที่ออกให้แต่ละ vendor, ผ่าน pricelist ที่ submit ของ vendor, ผ่าน trail การ approve / reject และต่อไปยังบรรทัด PR ที่ default จาก pricelist active, บรรทัด PO ที่ snapshot จากมัน และการ post GRN ที่รัน variance กับมัน — และ **chain ย้อนกลับ** — จาก GRN / invoice ที่ post ย้อนกลับผ่าน pricelist active ที่ anchor การคาดหวังราคา ตรวจสอบว่า chain ต่อเนื่อง (ไม่มี gap, ไม่มี timestamp out-of-order), ว่าการแยกหน้าที่ถูกเคารพ (`VPL_AUTH_014`: vendor user ถือ token ≠ Carmen user approve; high-value Purchaser ≠ Approver) และว่าทุกการเปลี่ยนมีทั้ง actor และเหตุผลที่ต้องการ
5. ถ้าพบความผิดปกติ (เช่น การ approve บันทึกนอก matrix authorisation ตาม `VPL_AUTH_004`/`VPL_AUTH_005`, การ revoke token โดยไม่มี `system` comment เหตุผล, `quality_score < threshold` approve โดยไม่มี Manager review, การเข้า vendor portal จาก IP ที่ไม่อยู่ใน allowlist) **flag** เอกสารในไฟล์เคส audit พร้อมโน้ต การ flag **ไม่** เปลี่ยน pricelist — เขียนไปยังร้านฝั่ง audit เท่านั้น
6. **Export รายงาน** เป็น CSV / PDF สำหรับ period หรือไฟล์เคส การ export ฟิลด์ละเอียดอ่อน (vendor identity เต็ม, pricing snapshot, portal-token / IP telemetry, รายละเอียดผู้ติดต่อ) ต้องการการ approve ทุติยภูมิตามนโยบายการ export ข้อมูล — Auditor submit คำขอ export และผู้ approve การ export release รายงานที่ export และบันทึกการ approve เองเป็น object audit

### 2.2. Flow System Administrator

**จุดเข้า:** Sidebar → workspace **Configuration** → **Pricelist Numbering & Templates** (สำหรับ scheme `pricelist_no` และ `campaign name`), **Pricelist RBAC & Roles** (สำหรับ map role-to-permission `VPL_AUTH_*`), **Portal Token Policy** (สำหรับ default expiration, IP allowlist, ขีดจำกัด concurrent-session, ขีดจำกัด suspicious-activity), **Email Integration Settings** (SMTP / provider wiring, registry email-template, การจัดการ bounce), **Validation Rule Registry** (สำหรับกติกาที่ validation engine รันที่ `VPL_VAL_018`–`VPL_VAL_023` และ `VPL_XMOD_009`), **Currency & FX Sources** (รายการสกุลเงินที่อนุญาตและแหล่งอัตรา FX ที่บริโภคโดย Finance) หรือ **Audit Retention Policy** แต่ละ surface คือหน้าแยกภายใต้ workspace เดียวกัน เพิ่มเติม Sysadmin สามารถใช้ action **Revoke Token** ต่อ invitation จากหน้า Campaign / Invitation เดียว (นี่เป็นที่เดียวที่ Sysadmin touch เอกสารธุรกรรมโดยตรง ตาม `VPL_AUTH_015`)

**Primary flow (happy path — Sysadmin, การเปลี่ยนการตั้งค่า):**

1. **ระบุความต้องการการตั้งค่า** Trigger จากภายนอก — เช่น Compliance ขอ IP allowlist portal-token ที่เข้มขึ้น, Procurement ต้องการ prefix ปีของการกำหนดเลข pricelist ใหม่, validation engine ต้องการ business rule ใหม่สำหรับข้อจำกัดเฉพาะอุตสาหกรรม, registry email-template ต้องการเวอร์ชันแปลภาษาสำหรับภูมิภาคใหม่, token ของ vendor ถูก compromise และต้องการการ revoke ทันที เปิด ticket การเปลี่ยนแปลงและ link การอ้างอิงนโยบาย (memo / การ approve) ก่อนเปิด surface การตั้งค่า
2. **เปิดหน้าการตั้งค่าที่เกี่ยวข้อง** สำหรับการกำหนดเลข: **Configuration → Pricelist Numbering & Templates** สำหรับ RBAC: **Configuration → Pricelist RBAC & Roles** สำหรับนโยบาย portal-token: **Configuration → Portal Token Policy** สำหรับการเชื่อม email: **Configuration → Email Integration Settings** สำหรับการ validate: **Configuration → Validation Rule Registry** สำหรับ currency / FX: **Configuration → Currency & FX Sources** สำหรับการเก็บ audit: **Configuration → Audit Retention Policy** สำหรับการ revoke token ครั้งเดียว: เปิด campaign เฉพาะ → แถว invitation → action **Revoke Token**
3. **ปรับการตั้งค่า** ใน staged editor: เปลี่ยน scheme การกำหนดเลข, แก้ permission set RBAC, ขัน default expiration portal-token หรือ IP allowlist, register email template ใหม่, add / remove / reorder กติกา validation, เปลี่ยน URL แหล่งอัตรา FX, อัปเดต window การเก็บ audit การแก้ทั้งหมดสะสมใน draft การตั้งค่าที่รออยู่; ไม่มีอะไร persist จนกว่า Save
4. **Preview ผลกระทบ** หน้าการตั้งค่าแสดง summary side-panel: count campaign / invitation / pricelist in-flight ที่จะดำเนินต่อภายใต้ snapshot การตั้งค่าเก่า, count เอกสารใหม่ที่จะใช้กติกาใหม่, กติกา validation ที่เปลี่ยน, ผู้ใช้ที่ได้รับผลกระทบจากการเปลี่ยน RBAC และ (สำหรับการเปลี่ยน token-policy) count portal token ที่ถูกต้องในปัจจุบันที่จะได้รับผลกระทบในการเข้า portal ครั้งต่อไป สำหรับการเปลี่ยนการกำหนดเลข panel preview `pricelist_no` ครั้งต่อไปที่ออกภายใต้ scheme ใหม่ สำหรับการเปลี่ยนการเชื่อม email panel รัน probe การเชื่อมต่อ Sysadmin สามารถ revise หรือ discard draft ที่จุดนี้
5. **Save การตั้งค่า** ระบบเขียนการตั้งค่าใหม่ด้วย timestamp `effective_from`, บันทึกการเปลี่ยนใน log audit การตั้งค่าฝั่งระบบ (แยกจากตาราง comment ฝั่งโมดูล) และแจ้ง user population ที่ได้รับผลกระทบ (เช่น Purchaser ที่ pricelist ใหม่จะใช้ scheme การกำหนดเลขที่ต่างกัน; Finance ที่แหล่งอัตรา FX ถูกอัปเดต) Template / campaign / pricelist ที่อยู่ในการบินคง **snapshot** ต้นฉบับของการตั้งค่าที่มีผลที่การเปลี่ยน state ของแต่ละ — workflow stage chain, การกำหนดเลข, นโยบาย token, validation rule set, email template, URL แหล่ง FX ทั้งหมดรักษาตาม snapshot semantic ที่ปกป้อง snapshot PR / PO / GRN ปลายน้ำ เอกสารใหม่ที่สร้างหลัง `effective_from` ใช้การตั้งค่าใหม่
6. **การ revoke token (variant action เดียวของ flow การตั้งค่า)** จากหน้า campaign / invitation คลิก **Revoke Token** บนแถว `tb_request_for_pricing_detail` ที่ได้รับผลกระทบ ยืนยันด้วย flow second-factor / step-up auth Action เขียน `tb_request_for_pricing_detail.pricelist_url_token = NULL` และ append `system` comment ใน `tb_request_for_pricing_detail_comment` จับเหตุผลและ actor การเข้า portal ครั้งต่อไปของ vendor return `401`; auto-save draft รักษาฝั่ง server; เคสสามารถ re-route โดยการออก invitation ใหม่พร้อม token ใหม่ (flow Purchaser Step 4–5)
7. **ตรวจสอบการ activate** หยิบ pricelist ทดสอบใหม่ที่เป็นตัวแทน (หรือจำลองหนึ่งใน environment non-production) และยืนยันว่าพฤติกรรมใหม่ fire ตามคาด — การกำหนดเลข, RBAC, นโยบาย token, การ render email-template, การ fire กติกา validation, การ quote แหล่ง FX, window การเก็บ audit ถ้าพบ regression rollback โดยเปิดการตั้งค่าใหม่และ revert ไปยังเวอร์ชันก่อนหน้า (ทุกเวอร์ชันที่บันทึกคงไว้ใน log audit การตั้งค่า)
8. **ปิด ticket การเปลี่ยนแปลง** ด้วย link log audit การตั้งค่า จากจุดนี้การเปลี่ยนแปลงมีผลสำหรับเอกสารใหม่; การมีส่วนร่วมของ Sysadmin จบจนกว่าการเปลี่ยนการตั้งค่าครั้งต่อไป

## 3. สาขาการตัดสินใจ

- **ถ้า Auditor พบการละเมิดนโยบาย** (เช่น pricelist approve โดย user ที่ขาดสิทธิ์ `VPL_AUTH_004`/`VPL_AUTH_005`, pricelist multi-currency activate โดยไม่มี Finance Manager co-signoff ตาม `VPL_AUTH_010`, portal token ใช้จาก IP ที่ไม่รู้จักโดยไม่มีเหตุผล, `quality_score < 70` auto-approve แทนการ route ไป Manager review): Auditor **ไม่สามารถ act บนเอกสารใน-โมดูล** (อ่านอย่างเดียว) Auditor escalate ผ่านไฟล์เคส audit — flag เอกสาร, แนบหลักฐาน (excerpt activity-log, diff เวอร์ชันการตั้งค่า, trace กติกา validation, การจับคู่การแยกหน้าที่) — และ route เคสไปยังเจ้าของธุรกิจที่รับผิดชอบ (Purchasing Manager, Finance Manager, Compliance หรือ department head) สำหรับ remediation out-of-band ถ้า remediation ต้องการ action ระดับระบบ (เช่น การ inactivate ภายใต้ `VPL_POST_019`, launch campaign ใหม่, การ revoke token) action นั้นทำโดย Purchaser / Purchasing Manager / Sysadmin ภายใต้สิทธิ์ authorization ของตน ไม่ใช่โดย Auditor
- **ถ้า Sysadmin พยายามเปลี่ยนการตั้งค่าขณะ pricelist in-flight พึ่งพากติกาปัจจุบัน** (เช่น การเปลี่ยน RBAC ขณะ pricelist multi-currency อยู่ระหว่าง Finance-review; การเปลี่ยนกติกา validation ขณะ vendor portal session อยู่ระหว่าง submission): save อนุญาต (กติกาไม่ถูกล็อกโดยเอกสาร in-flight) แต่การเปลี่ยน **ไม่** re-apply ย้อนหลังไปยังเอกสาร in-flight ที่มีอยู่ หน้าการตั้งค่า surface count เอกสาร in-flight ที่ได้รับผลกระทบใน panel preview; เอกสารเหล่านั้นดำเนินต่อภายใต้ snapshot การตั้งค่า Session portal ที่ authenticate แล้วดำเนินต่อภายใต้นโยบาย token ก่อนหน้า; นโยบายใหม่ใช้เฉพาะกับคำขอเข้า portal สดหลัง `effective_from`
- **ถ้า Sysadmin revoke portal token ขณะ vendor มี draft in-progress** การ revoke token ภายใต้ `VPL_AUTH_015` เป็นทันที; การเข้า portal ครั้งต่อไปของ vendor return `401` Draft pricelist ใน `tb_pricelist` (`status = draft`) รักษาบน server สำหรับ audit และอาจถูก (a) soft-delete โดย Purchaser ถ้าข้อมูลไม่กู้คืน, (b) re-route ผ่าน invitation ใหม่ไปยัง vendor เดียวกันภายใต้ campaign แยก (Purchaser launch campaign ใหม่ vendor ได้ token ใหม่ กรอกสด) หรือ (c) ยอมรับเป็นที่ Purchaser upload draft บางส่วนเป็น pricelist email-method หลังการยืนยัน vendor out-of-band Log audit การตั้งค่าบันทึก actor และเหตุผลการ revoke
- **ถ้า Auditor ขอ export ที่รวมฟิลด์ละเอียดอ่อน** (vendor identity เต็ม, telemetry portal IP, รายละเอียดผู้ติดต่อ, pricing snapshot, รายละเอียดการเบี่ยงเบน validation, แนบสัญญา): การ export ไปยัง state **pending** และต้องการการ approve จากผู้ approve การ export ข้อมูลตามนโยบายการ export Auditor ไม่สามารถ bypass step นี้ ขณะ pending การ export มองไม่เห็นนอกไฟล์เคส audit; เมื่อ approve การ export materialise และ link ดาวน์โหลดบันทึกในไฟล์เคสด้วย identity ของผู้ approve
- **ถ้าการเปลี่ยนกติกา validation ของ Sysadmin จะทำให้ submission in-progress ล้มเหลว** Panel preview surface ความเสี่ยงนี้: "การ save การเปลี่ยนกติกานี้จะทำให้ N session vendor portal in-progress ได้รับ error validation ใหม่ที่การ save ครั้งต่อไป" Sysadmin สามารถ (a) cancel, (b) เลื่อนการเปลี่ยนไปยัง window ที่ไม่มี submission in-progress หรือ (c) save พร้อม warning ที่ acknowledge ถ้า save vendor mid-flight เห็นข้อความ validation ใหม่ที่ save ครั้งต่อไป; log audit การตั้งค่าจับ count session ที่ได้รับผลกระทบและการ acknowledge ของ Sysadmin สำหรับ attribution forensic
- **ถ้าการเปลี่ยน Sysadmin สร้าง deadlock authorisation** (เช่น การเปลี่ยน RBAC ลบ user เดียวที่ authorise approve pricelist high-value multi-currency ระหว่าง window วันหยุดสาธารณะ): preview flag deadlock ถ้า save ต่อ pricelist ที่ได้รับผลกระทบจะ time out ที่ stage approve และต้องการการแทรกแซง manual — โดยทั่วไป delegation หรือ Manager-elevated approval Log audit การตั้งค่าบันทึก warning deadlock ดังนั้น trail audit ทำให้สาเหตุชัดเจน

## 4. จุดออก / Handoff

แกน persona Audit / Config ออกด้วยวิธีต่อไปนี้ ขึ้นกับ role ใดที่ action:

- **Auditor — รายงานสร้างขึ้น** ผลลัพธ์ query, drill-down chain หรือไฟล์เคส materialise (review บนหน้าจอหรือ export เป็น CSV / PDF หลัง flow การ approve การ export) **ไม่มีการเปลี่ยนสถานะ pricelist**: ทุก status, snapshot และ entry activity-log คงเหมือนเดิมก่อนที่ Auditor จะเปิดหน้า Handoff ของ Auditor เป็น **out-of-band** ไปยังเจ้าของธุรกิจใด (Purchasing Manager, Finance Manager, Compliance หรือ department head) ที่รับผิดชอบ remediation ที่ audit surface ถ้าไฟล์เคสของ Auditor แนะนำการ inactivate Purchaser execute `VPL_POST_019` ภายใต้ authority ของตน; ถ้าแนะนำการ revoke token Sysadmin act ภายใต้ `VPL_AUTH_015`
- **Auditor — ไฟล์เคสปิดโดยไม่มี action** เมื่อ query audit / drill-down chain ไม่พบความผิดปกติ Auditor ปิดไฟล์เคสด้วยโน้ต "no findings" ไม่มีการเปลี่ยนสถานะ pricelist; ไฟล์เคสเองคงเป็นหลักฐานว่า period / scope ถูก audit end-to-end
- **Sysadmin — การตั้งค่าบันทึก** เวอร์ชันใหม่ของการตั้งค่า (การกำหนดเลข, RBAC, นโยบาย portal-token, การเชื่อม email, กติกา validation, แหล่ง currency / FX, การเก็บ audit) เขียนด้วย timestamp `effective_from` และบันทึกใน log audit การตั้งค่า **เอกสารที่สร้างหลัง `effective_from`** ใช้การตั้งค่าใหม่; **เอกสารที่อยู่ในการบินแล้ว** คงการตั้งค่า snapshot ต้นฉบับ การแจ้งเตือนไปยัง user population ที่ได้รับผลกระทบ fire ตอน save Handoff เป็น **ไปข้างหน้าในเวลา** — Purchaser คนต่อไปที่สร้าง campaign / pricelist เห็นพฤติกรรมใหม่อัตโนมัติ
- **Sysadmin — การตั้งค่า rollback** ถ้าการตรวจสอบพบ regression Sysadmin revert ไปยังเวอร์ชันก่อนหน้า Rollback เองเป็นการ save การตั้งค่าที่มี `effective_from` ของตน; เอกสารที่สร้างระหว่างการเปลี่ยนต้นฉบับและ rollback ไม่ถูก re-evaluate ย้อนหลัง (snapshot semantic) แต่เอกสารใหม่ที่สร้างหลัง rollback ใช้การตั้งค่าที่ revert log audit การตั้งค่าจับทั้งการเปลี่ยนไปข้างหน้าและ rollback
- **Sysadmin — การ revoke token** การเขียน `tb_request_for_pricing_detail.pricelist_url_token = NULL` ครั้งเดียวบวก `system` comment Handoff ไปยัง **Vendor** (สูญเสียการเข้า portal ทันที) และไปยัง **Purchaser** (การแจ้งเตือน — ดำเนินการ re-issue ถ้าต้องการ) สถานะเอกสารที่ handoff: invitation `expired` (มีผลทันที); draft pricelist in-progress ใด ๆ รักษาที่ `draft` สำหรับการตัดสินใจของ Purchaser
- **Sysadmin — integration cutover** เมื่อการเปลี่ยนเป็นการ rewire integration (เช่น provider transactional-email ใหม่, แหล่งอัตรา FX ใหม่) save trigger sequence probe-and-cutover เพิ่มเติม — endpoint เก่าถูก drain, endpoint ใหม่ได้รับ round-trip สังเคราะห์ และเฉพาะเมื่อสำเร็จการตั้งค่าจึง live ถ้า probe ล้มเหลว save ถูกค้างและ endpoint ก่อนหน้ายังมีผล; Sysadmin ถูกแจ้งให้ remediate ก่อน retry

สถานะเอกสารข้ามทุกการออก Audit / Config ถูกกำกับโดยสาม enum สถานะ (template `draft / active / inactive`, pricelist `draft / active / inactive / expired`) และสถานะ campaign / invitation ที่ derive ระดับแอป Flow Auditor ไม่เคยย้ายเอกสารข้ามสิ่งเหล่านี้; flow การตั้งค่าของ Sysadmin ไม่เคยย้ายเอกสารข้ามสิ่งเหล่านี้ (เฉพาะเอกสารในอนาคตที่ได้รับผลกระทบ); action การ revoke token ของ Sysadmin ย้าย **invitation** ไป `expired` ที่ derive ทันทีแต่ไม่เปลี่ยน `status` ของ pricelist การเปลี่ยน `status` ของ pricelist ที่ trigger โดยการ escalation จากข้อค้นพบ audit เท่านั้นคือ **การ inactivate** (Purchaser ภายใต้ `VPL_POST_019`) และที่ Purchasing Manager ตัดสินใจ launch การเก็บใหม่ วงจร `VPL_POST_015`–`VPL_POST_017` ของ pricelist ใหม่
