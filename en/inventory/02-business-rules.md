# Inventory — Business Rules

## 1. Overview

This page captures the operational business rules that govern the **inventory module** — the ledger that every other movement-generating module (GRN receipts, store-requisition issues / transfers, physical / spot counts, credit notes, manual stock-in / stock-out) posts into. Because inventory is movement-driven rather than document-driven, the rule set here looks different from the GRN / PR / PO catalogues: there is no `doc_status` lifecycle to manage, no save / approve / commit progression, and no single "posting event" that fires per row. Each transaction is itself the posting event; validation runs **before** the transaction is written, calculation populates the cost-layer and (downstream) the period snapshot, authorization is RBAC-plus-source-document, posting effects are the immediate ledger and (where applicable) GL fan-out, and the period-end process is a separate set of rules that locks the ledger against backdated changes.

Two structural points colour every rule below and are worth restating up front. **First**, there is **no `tb_stock_balance` row** — current on-hand at `(location_id, product_id, lot_no)` is derived as `last_snapshot.closing_qty + Σ (cost_layer.in_qty − cost_layer.out_qty)` since the snapshot, scoped to that key. So rules that look like "increment the balance" actually mean "write a new `tb_inventory_transaction_cost_layer` row with the right `in_qty` / `out_qty` / `cost_per_unit` so the derived sum advances correctly" (see [[inventory/01-data-model]] § 5 item 1). **Second**, `location_type` (`inventory`, `direct`, `consignment` on `tb_location.location_type`) is the **single most consequential setting** for posting behaviour — it determines whether a movement debits the Inventory asset, the Department Expense, or a memo register, and whether the movement creates a `tb_inventory_transaction_cost_layer` row at all. Several rules below are gated by `location_type` and the rule's behaviour changes accordingly.

## 2. Validation Rules

Rule IDs follow `INV_VAL_NNN`. Validation runs **at post time** — i.e. when the source module (GRN, SR, count, credit note, manual stock-in / stock-out) attempts to write the `tb_inventory_transaction` row and its children. There is no save-time / commit-time split because there is no draft state on the transaction itself; the source document carries that progression.

| Rule ID | Condition | When enforced | Error / behaviour |
| ------- | --------- | ------------- | ----------------- |
| `INV_VAL_001` | `tb_inventory_transaction.inventory_doc_type` is a valid value in `enum_inventory_doc_type` and `inventory_doc_no` resolves to a non-soft-deleted source-document row of the matching type (e.g. `inventory_doc_type = good_received_note` ⇒ `inventory_doc_no` exists in `tb_good_received_note`). | Post | Reject with `"inventory_doc_type / inventory_doc_no do not resolve to a valid source document."` No transaction row written; no cost-layer rows written; source-document commit is rolled back. |
| `INV_VAL_002` | Each `tb_inventory_transaction_detail` row has non-null `product_id` referencing an active, non-soft-deleted `tb_product`. `location_id` is non-null **except** for system-level rollups (`enum_inventory_doc_type ∈ {close, open}`) where the location is implicit in the snapshot scope. | Post | Reject the transaction with `"Product is required on every inventory transaction detail line; location is required for non-rollup movements."` Atomic rollback of the whole transaction. |
| `INV_VAL_003` | When `location_id` is set, the referenced `tb_location.is_active = true` and `tb_location.deleted_at IS NULL`. | Post | Reject with `"Cannot post to an inactive or soft-deleted location."` Source-document edit returns the error per line. |
| `INV_VAL_004` | Movement direction matches detail sign: inbound transactions (`enum_transaction_type ∈ {good_received_note, transfer_in, adjustment_in, eop_in, open_period, credit_note_quantity-with-positive}`) require `qty > 0`; outbound (`{transfer_out, issue, adjustment_out, eop_out, close_period, credit_note_quantity-with-negative}`) require `qty < 0` on the detail and `out_qty > 0` on the cost layer. | Post | Reject with `"Quantity sign does not match transaction type (inbound transactions require positive qty; outbound transactions require negative qty)."` |
| `INV_VAL_005` | Outbound movements (`out_qty > 0` on the cost layer) cannot consume more than the available qty at `(location_id, product_id, lot_no, lot_index)`. Available qty is the derived sum (Section 1, structural point 1). Negative balance is **forbidden by default** — even if costing math is unaffected by short-stocking, the inventory ledger rejects the post. Tenant config may relax this to "allow negative balance" for trusted cost centres (typically `location_type = direct` permits implicit negative because no balance is tracked). | Post | Reject with `"Outbound movement would drive on-hand at (location, product, lot) below zero. Available: <X>, requested: <Y>."` Source-document edit returns the error per line. The exception path is documented in `INV_AUTH_*` for the relaxed-tenant case. |
| `INV_VAL_006` | When `enum_transaction_type` involves a specific lot (`from_lot_no` on outbound, `current_lot_no` on inbound), the lot identity is well-formed: `lot_no` is non-empty and globally unique within `(product_id, location_id)` for new lots; for outbound, the referenced `(lot_no, lot_index)` exists in `tb_inventory_transaction_cost_layer` with positive remaining balance. | Post | Reject with `"Lot <lot_no> at location <location> is not available for consumption."` for outbound; `"Lot <lot_no> already exists for product <product> at location <location>; lot identity must be unique."` for inbound new-lot collision. |
| `INV_VAL_007` | All monetary fields are non-negative: `cost_per_unit ≥ 0` on `tb_inventory_transaction_detail`; `in_qty ≥ 0` and `out_qty ≥ 0` (never both non-zero on the same cost-layer row) on `tb_inventory_transaction_cost_layer`; `average_cost_per_unit ≥ 0`. | Post | Reject with `"Cost per unit and qty must be non-negative; in_qty and out_qty are mutually exclusive on a single cost-layer row."` |
| `INV_VAL_008` | The accounting period containing the transaction date is not `closed` or `locked` on `tb_period.status`. Backdated postings (transaction date in a `closed` / `locked` period) are rejected outright. Re-opening a `closed` period is a separate elevated action (`INV_AUTH_*`); `locked` periods cannot be re-opened by any role. | Post | Reject with `"Cannot post into period <YYMM>: period is <status>. Re-open the period (closed only) or post a current-period restatement (locked)."` |
| `INV_VAL_009` | When `location_type = direct`, the transaction is restricted to inbound types (`good_received_note`, `adjustment_in`, `eop_in`, `open_period`) — direct-cost locations cannot consume from a non-existent balance and cannot transfer out. Outbound consumption equivalents post into the source GRN's expense account at receipt time and never enter the cost-layer ledger. | Post | Reject with `"Cannot post outbound movement to direct-cost location <location>. Direct locations bypass inventory; consumption is implicit at receipt."` |
| `INV_VAL_010` | When `location_type = consignment`, ownership is the vendor's until consumption. Inbound (`good_received_note`) posts a memo cost-layer row but **does not** raise an AP liability or debit Inventory; outbound (`issue`) posts COGS and AP simultaneously per `INV_POST_006`. Cross-type transfers from `consignment → inventory` (taking ownership) require an explicit `tb_stock_in` re-classification document and are not implicit. | Post | Reject implicit consignment-to-inventory transfers with `"Consignment-to-inventory ownership transfer requires an explicit re-classification stock-in document."` |
| `INV_VAL_011` | For credit-note transactions (`enum_inventory_doc_type = credit_note`), the source credit note's `inventory_doc_no` resolves to a `committed` source document and the credit note's affected lot exists in the cost layer. Amount-only credit notes (`enum_transaction_type = credit_note_amount`) write `diff_amount` only with `in_qty = out_qty = 0`; quantity credit notes (`credit_note_quantity`) consume from the originating receipt's lot. | Post | Reject with `"Credit note source GRN is not in a state that permits adjustment, or the affected lot does not exist."` |
| `INV_VAL_012` | For period-rollup transactions (`enum_inventory_doc_type ∈ {close, open}`), the user / role is the System / scheduled-job context (`INV_AUTH_007`); the period is `closed` when posting `eop_out` / `close_period` and `open` when posting `eop_in` / `open_period`. A user-initiated close / open transaction is rejected. | Post | Reject with `"Period rollforward transactions can only be written by the period-end job."` |
| `INV_VAL_013` | Soft-delete on `tb_inventory_transaction`: `deleted_at` is **only** set as part of a compensating-transaction sequence (write the reversal, then mark the original `deleted_at` if tenant policy is "hide reversed transactions"). Direct soft-delete without a compensating reversal is rejected. | Soft-delete | Reject with `"Cannot soft-delete an inventory transaction without a compensating reversal."` |

## 3. Calculation Rules

All monetary values are stored as `Decimal(20, 5)` on the cost-layer and detail rows. Display rounding is half-up to 2 decimals for currency amounts and 3 decimals for quantities. Intermediate computations use the full 5dp precision; only the final stored / displayed value is rounded.

Rule IDs follow `INV_CALC_NNN`.

| Rule ID | Formula |
| ------- | ------- |
| `INV_CALC_001` (detail total_cost) | `total_cost = qty × cost_per_unit` (signed by `qty`'s sign). Stored at 5dp on `tb_inventory_transaction_detail`. |
| `INV_CALC_002` (cost-layer total_cost — inbound) | For `in_qty > 0`: `total_cost = in_qty × cost_per_unit` on `tb_inventory_transaction_cost_layer`. |
| `INV_CALC_003` (cost-layer total_cost — outbound) | For `out_qty > 0`: `total_cost = out_qty × cost_per_unit` where `cost_per_unit` is picked by the costing method (see `INV_CALC_005` / `INV_CALC_006`). |
| `INV_CALC_004` (current on-hand — derived) | `on_hand_qty(location, product, lot) = last_snapshot.closing_qty + Σ_layer (cost_layer.in_qty − cost_layer.out_qty)` for layers with `created_at > last_snapshot.snapshot_at`, scoped to the key. When no prior snapshot exists, the sum starts at 0. |
| `INV_CALC_005` (FIFO outbound cost pick) | For products with `costing_method = FIFO` (configured on `tb_product`): when posting an outbound layer, iterate `tb_inventory_transaction_cost_layer` rows at `(location_id, product_id)` ordered by `lot_seq_no` ascending, consuming `out_qty` from each layer's remaining balance until satisfied. Each consumed layer produces its own outbound cost-layer row with `cost_per_unit = layer.cost_per_unit`; the outbound transaction may therefore have **multiple** cost-layer rows for a single detail row when consumption spans lots. |
| `INV_CALC_006` (Weighted-average outbound cost pick) | For products with `costing_method = WEIGHTED_AVERAGE`: outbound `cost_per_unit = current_average_cost(location, product) at post time`. The current average is the most recent `tb_inventory_transaction_cost_layer.average_cost_per_unit` at `(location_id, product_id)` for that product (refreshed by every inbound — see `INV_CALC_007`). |
| `INV_CALC_007` (Weighted-average refresh on inbound) | When an inbound posts under WEIGHTED_AVERAGE: `new_average = (prior_on_hand × prior_average + in_qty × in_cost_per_unit) / (prior_on_hand + in_qty)` rounded to 5dp. Written to `average_cost_per_unit` on the new inbound cost-layer row. Subsequent reads at `(location_id, product_id)` pick up this value. |
| `INV_CALC_008` (Period snapshot — opening) | `opening_qty = previous_period.closing_qty`; `opening_cost_per_unit = previous_period.closing_cost_per_unit`; `opening_total_cost = opening_qty × opening_cost_per_unit`. Written by the period-open job (`enum_inventory_doc_type = open`, `enum_transaction_type = open_period`). |
| `INV_CALC_009` (Period snapshot — receipt / issue / adjustment buckets) | For the closing period, `receipt_qty = Σ in_qty for inbound cost layers`; `receipt_total_cost = Σ in_qty × cost_per_unit`; similarly `issue_qty / issue_total_cost = Σ out_qty × cost_per_unit`; `adjustment_qty / adjustment_total_cost = Σ from adjustment_in / adjustment_out / credit_note_* layers`. All sums scoped to the period × location × product × lot key. |
| `INV_CALC_010` (Period snapshot — closing) | `closing_qty = opening_qty + receipt_qty − issue_qty + adjustment_qty`; `closing_cost_per_unit = closing_total_cost / closing_qty` (set to the period-end weighted average; for FIFO, this is the residual lot's `cost_per_unit`); `closing_total_cost = opening_total_cost + receipt_total_cost − issue_total_cost + adjustment_total_cost + diff_amount`. The `diff_amount` bucket captures price-revaluation effects (credit-note-amount adjustments). |
| `INV_CALC_011` (Credit-note amount adjustment) | For `enum_transaction_type = credit_note_amount`: write a cost-layer row with `in_qty = out_qty = 0`, `diff_amount = vendor_credit_amount` (signed; negative for vendor concession reducing cost), `transaction_type = credit_note_amount`. The cost on the originating lot is recalculated: `new_lot_cost_per_unit = (original_lot_total_cost + diff_amount) / original_lot_qty`. Downstream consumption from the same lot picks up the revalued cost; already-consumed portions are not retroactively adjusted (the price variance flows through `diff_amount`). |
| `INV_CALC_012` (Rounding) | All rounding is half-up. Stored values use the column precision; intermediate computations carry 5dp; on-the-fly aggregations re-read rounded values from each step. |

### 3.1 Worked example (Weighted-average product, ฿ THB)

A product with `costing_method = WEIGHTED_AVERAGE`, no prior on-hand. Three movements in chronological order at the same inventory-type location:

- **GRN receipt #1** (`enum_transaction_type = good_received_note`): `in_qty = 100`, `cost_per_unit = ฿10.00`.
  - Cost layer: `in_qty = 100`, `cost_per_unit = ฿10.00`, `total_cost = ฿1,000.00`.
  - `new_average = (0 × 0 + 100 × 10.00) / (0 + 100) = ฿10.00000` per `INV_CALC_007`.
- **GRN receipt #2** (`good_received_note`): `in_qty = 50`, `cost_per_unit = ฿14.00`.
  - Cost layer: `in_qty = 50`, `cost_per_unit = ฿14.00`, `total_cost = ฿700.00`.
  - `new_average = (100 × 10.00 + 50 × 14.00) / (100 + 50) = (1,000 + 700) / 150 = ฿11.33333` per `INV_CALC_007`.
- **Store-requisition issue** (`issue`): `out_qty = 80`.
  - Outbound cost-layer row: `out_qty = 80`, `cost_per_unit = ฿11.33333` (picked from `current_average_cost` per `INV_CALC_006`), `total_cost = ฿906.67` (`80 × 11.33333` rounded to 2dp).
  - `new_average` remains `฿11.33333` (issues don't change the average under weighted-average; only inbound does).
- Current on-hand per `INV_CALC_004`: `0 (no prior snapshot) + (100 + 50 − 80) = 70 units`; valuation `70 × 11.33333 = ฿793.33`.

For a **FIFO** variant of the same product (`INV_CALC_005`): the issue would consume from receipt #1 first (`lot_seq_no` ascending) — `out_qty = 80` at `cost_per_unit = ฿10.00` (one cost-layer row), `total_cost = ฿800.00`. Receipt #1's remaining balance becomes `100 − 80 = 20` units at `฿10.00`; receipt #2's balance is untouched at `50` units at `฿14.00`. Closing inventory valuation: `20 × 10.00 + 50 × 14.00 = ฿900.00`.

### 3.2 Worked example (location_type = direct)

A product receipt to a `direct` location. `INV_VAL_009` permits inbound. The GRN commit writes a `tb_inventory_transaction` row (for audit) but **does not** create a `tb_inventory_transaction_cost_layer` row — the receipt expense is recognised immediately:

- **GRN receipt** (`good_received_note`, location_type = direct): `qty = 30`, `cost_per_unit = ฿8.00`.
  - Inventory transaction header + detail written (audit trail).
  - **No cost-layer row written** (no asset ledger entry).
  - GL: `Dr Department Expense ฿240.00 / Cr AP ฿240.00` per [[good-receive-note/02-business-rules]] `GRN_POST_006` direct-variant.
  - On-hand at `(direct_location, product, *)` remains 0 (direct locations carry no balance).

## 4. Authorization Rules

Rule IDs follow `INV_AUTH_NNN`. Authorization is enforced by RBAC at the source-document layer plus a small set of inventory-module-specific rules. The inventory transaction itself has **no `user_action.execute` set** because there is no workflow stage — RBAC on the source module (GRN, SR, count, credit note, manual stock-in / stock-out) is what permits the post. The inventory module owns the rules below that gate **direct** inventory writes (manual stock-in / stock-out, period-end rollforward, configuration of location and product-location parameters).

| Rule ID | Subject | Right | Constraint |
| ------- | ------- | ----- | ---------- |
| `INV_AUTH_001` | Store Keeper | Post inbound stock-in via `tb_stock_in` (manual adjustment, found-stock, count overage) | The Store Keeper can post `enum_inventory_doc_type = stock_in` with `enum_transaction_type = adjustment_in` to locations they have a `tb_user_location` mapping for. Above the tenant-configured threshold, an Inventory Controller approval is required first (the source `tb_stock_in.doc_status` follows `draft → in_progress → completed` workflow). |
| `INV_AUTH_002` | Store Keeper | Post outbound stock-out via `tb_stock_out` (write-off, breakage, count shortage) | Same scope as `INV_AUTH_001` but for outbound. Reason code is required (`adjustment_type_id` references `tb_adjustment_type`). Above-threshold write-offs always require Inventory Controller approval per the source-module workflow. |
| `INV_AUTH_003` | Inventory Controller | Approve and post adjustments above the Store Keeper threshold | Inventory Controller is the second signature on stock-in / stock-out documents above the threshold. They also have direct-post authority for variance adjustments arising from a `tb_count_stock` or `tb_spot_check` run (the count-difference becomes a system-staged stock-in / stock-out that the Controller approves). |
| `INV_AUTH_004` | Inventory Controller | Edit `tb_product_location` policy (min / max / par / reorder) | Read-only for Store Keeper. The Controller owns replenishment-policy maintenance; changes apply prospectively. |
| `INV_AUTH_005` | Finance | Approve cost-impact adjustments and post manual journal entries reflecting inventory-to-GL variance | Finance is the third signature on adjustments whose cost impact exceeds the Controller threshold, and is the role that posts the periodic inventory-to-GL reconciliation entry. They cannot edit inventory transactions directly — corrections flow through stock-in / stock-out / credit-note documents that they approve. |
| `INV_AUTH_006` | Finance Manager | Lock period (`status = closed → locked`); re-open period (`closed → open`) | Period locking is reserved to Finance Manager. Re-opening a `closed` period (to back-post an exceptional adjustment) requires Finance Manager authorisation and is audit-logged; re-opening a `locked` period is **not permitted** for any role. |
| `INV_AUTH_007` | System / scheduled job | Post `enum_inventory_doc_type ∈ {close, open}` rollforward transactions | Period-end rollforward is a system-context post. User-initiated `close` / `open` transactions are rejected per `INV_VAL_012`. The job runs under a system service account; activity log records `created_by_id = <system_user_id>`. |
| `INV_AUTH_008` | System Administrator | Configure `tb_location` (including `location_type` and `physical_count_type`), `tb_adjustment_type`, the costing method per product (`tb_product.costing_method`), and integration endpoints | Configuration-only role; cannot post transactions. Changes apply prospectively to new movements; `committed` (already-posted) movements retain the configuration in effect at post time per the snapshot rule. Changing `location_type` on a location with non-zero on-hand is **blocked** — must be drained (or re-classified via stock-in / stock-out) first. |
| `INV_AUTH_009` | Auditor | Read-only access to all inventory transactions, cost-layer rows, period snapshots, and configuration history | Auditor (folded under Audit / Config) holds read scope across the full inventory dataset including soft-deleted (compensated) transactions. Export of sensitive fields (unit cost, vendor terms via the joined `tb_good_received_note.vendor_id`) requires secondary approval per the parent-module audit pattern. |
| `INV_AUTH_010` | Segregation of duties — Receiver / SR Approver / Adjuster ≠ Self-receipt | The user posting an outbound `tb_stock_out` (write-off) MUST NOT be the same user who created the originating receipt (`tb_good_received_note.created_by_id`) for the lot being written off **when the write-off exceeds the tenant SoD threshold**. Routine count-shortage write-offs of small amounts are exempt; large write-offs require an independent adjuster. | Enforced at post. Mirrors the SoD pattern in [[good-receive-note]] `GRN_AUTH_010` and [[purchase-order]] `PO_AUTH_010`. |

## 5. Posting Rules

The inventory module does not have a document `doc_status` lifecycle — every posted transaction is immutable on write. The state diagram is therefore degenerate: each transaction transitions `(none) → posted` and may, via a compensating reversal, be marked `deleted_at` (logically reversed but never physically removed). The interesting rules are the **fan-out effects** that fire on the write.

Rule IDs follow `INV_POST_NNN`.

| Rule ID | Transition / Event | Effects |
| ------- | ------------------ | ------- |
| `INV_POST_001` | Post inbound (`good_received_note`, `transfer_in`, `adjustment_in`) to `inventory`-type location | Insert `tb_inventory_transaction` (header) and `tb_inventory_transaction_detail` (per product, `qty > 0`). For products with a costing method that needs layer tracking, insert one `tb_inventory_transaction_cost_layer` row with `in_qty > 0`, `cost_per_unit`, `lot_no`, `lot_index`, `lot_seq_no`, and `at_period` set; recompute `average_cost_per_unit` per `INV_CALC_007` and write it onto the new layer row. GL: **Dr Inventory** at `total_cost` / **Cr** the source-document credit account (AP for GRN, Inventory at other location for transfers, Adjustment Suspense for stock-in). |
| `INV_POST_002` | Post outbound (`issue`, `transfer_out`, `adjustment_out`) from `inventory`-type location | Insert `tb_inventory_transaction` + `tb_inventory_transaction_detail` (`qty < 0`). Cost-layer rows depend on costing method: FIFO writes one or more outbound layer rows consuming from `lot_seq_no` ascending until `out_qty` is satisfied (each row carries its lot's `cost_per_unit`); weighted-average writes a single outbound layer row at the current `average_cost_per_unit`. `INV_VAL_005` (no-negative-balance) is checked before the write. GL: **Dr** the destination-account (COGS / Department Expense for issues, Inventory at destination for transfers, Adjustment Suspense for stock-out) / **Cr Inventory** at the picked cost. |
| `INV_POST_003` | Post receipt to `direct`-type location | Insert header + detail for audit. **No cost-layer row.** No balance impact. GL: **Dr Department Expense** / **Cr** the source-document credit account (AP for GRN, etc.). This is the `INV_VAL_009` direct-cost path. |
| `INV_POST_004` | Post receipt to `consignment`-type location | Insert header + detail for audit. Insert a cost-layer row flagged as consignment (typically via `dimension` or `info` JSON; not via a separate enum value). **No AP liability raised, no Inventory asset debited at receipt.** Memo-only — the goods are tracked for visibility but the ownership remains with the vendor. |
| `INV_POST_005` | Post issue from `consignment`-type location | Two-leg GL atomically: **Dr COGS / Cr AP** at the consignment cost. The cost-layer row records the consumption (`out_qty > 0`); the AP entry is raised at consumption time, not at receipt. This is the consignment payment-on-consumption pattern documented in `../carmen/docs/Inventory/location-type-and-financial-treatment.md`. |
| `INV_POST_006` | Transfer between locations (`store_requisition` transfer with source ≠ destination) | Two cost-layer rows under a single `tb_inventory_transaction` (or two coupled transactions, depending on the SR module's posting model): `transfer_out` at source (`out_qty > 0`, FIFO or WA cost pick), `transfer_in` at destination (`in_qty > 0` with `cost_per_unit = transfer_out.cost_per_unit`, `parent_lot_no = source_lot`). `INV_VAL_005` runs against the source location's balance. GL: typically **Dr Inventory@Dest / Cr Inventory@Source** — net zero on the Inventory control account if both locations are inventory-type. If destination is direct, the transfer hits expense instead (rare; usually a re-classification stock-in / stock-out pair is used). |
| `INV_POST_007` | Credit-note amount adjustment (`credit_note_amount`) | Insert cost-layer row with `in_qty = out_qty = 0`, `diff_amount = vendor_credit_amount` (signed), `transaction_type = credit_note_amount`. Update the originating lot's `cost_per_unit` per `INV_CALC_011`. Downstream consumption from the same lot picks up the revalued cost; already-consumed portions are not retroactively adjusted. GL: **Dr AP / Cr Inventory** at the credit amount (vendor concession reduces both AP and inventory carrying value). |
| `INV_POST_008` | Credit-note quantity adjustment (`credit_note_quantity`) | Insert cost-layer row with `out_qty = credit_qty` consuming from the originating receipt's lot at the lot's `cost_per_unit`. GL: **Dr AP / Cr Inventory** at `out_qty × cost_per_unit`. The receipt-side counterpart on the source GRN's quantity is not edited (`committed` is terminal); the credit-note is the symmetric reversal. |
| `INV_POST_009` | Period close (`enum_inventory_doc_type = close`) | System job sweeps all open transactions in the period and writes `tb_period_snapshot` rows per `(location_id, product_id, lot_no, lot_index)` — `closing_qty / closing_cost_per_unit / closing_total_cost` and the bucket aggregates per `INV_CALC_009`. Writes one cost-layer row per affected lot with `transaction_type = close_period`. Sets `tb_period.status = closed`. Backdated postings into this period are now rejected per `INV_VAL_008`. |
| `INV_POST_010` | Period open (`enum_inventory_doc_type = open`) | System job (typically chained with `INV_POST_009`) opens the next period — sets `tb_period.status = open` on the new period and writes `tb_inventory_transaction_cost_layer` rows with `transaction_type = open_period`, `in_qty = previous.closing_qty`, `cost_per_unit = previous.closing_cost_per_unit`, `lot_seq_no` preserved from the closed lot. The previous period's `lot_seq_no` ordering carries forward, preserving FIFO sequence across period boundaries. |
| `INV_POST_011` | Period lock (`closed → locked` on `tb_period.status`) | Authorisation per `INV_AUTH_006`. No further re-open permitted. No transaction-side fan-out. |
| `INV_POST_012` | Reversal (compensating transaction) | Post a new `tb_inventory_transaction` with `qty` negated against the original detail's `qty`; FIFO consumes from / reverses the original lot to keep cost-layer integrity. The original transaction is **not** edited. If tenant policy is "hide reversed transactions", set `deleted_at` on the original row **after** the compensating transaction posts (per `INV_VAL_013`). Use case: correcting a posted-by-mistake stock-out, reversing a receipt for an item later determined to belong to a different PO, etc. |

State diagram (degenerate, single-state):

```
[*] → posted → (compensating reversal optional) → posted (with deleted_at set on original)
```

The transaction row is immutable after write; reversal is a new row, not a state change.

## 6. Cross-Module Rules

Rule IDs follow `INV_XMOD_NNN`.

| Rule ID | Related module | Rule |
| ------- | -------------- | ---- |
| `INV_XMOD_001` | [[good-receive-note]] | A GRN commit (`saved → committed`) writes one `tb_inventory_transaction` per detail_item with `inventory_doc_type = good_received_note` and `inventory_doc_no = tb_good_received_note.id`. Each detail_item's `inventory_transaction_id` column is stamped with the inserted transaction id (no Prisma `@relation`; application-resolved). The GRN's `received_qty` and `accepted_qty` resolve into the cost-layer `in_qty` (`accepted_qty` for inventory-type locations; full `received_qty` for direct only as expense, not as cost-layer). Lot data (`current_lot_no`, `expiry_date`) lives on `tb_inventory_transaction_detail` / `tb_inventory_transaction_cost_layer`, **not** on the GRN row (see [[good-receive-note/01-data-model]] § 5 item 3). |
| `INV_XMOD_002` | [[store-requisition]] | An approved SR issue writes outbound transactions with `inventory_doc_type = store_requisition` and `enum_transaction_type = issue` (for issue) or `transfer_out` + `transfer_in` paired (for transfers between locations). The SR detail's `inventory_transaction_id` carries the transaction id. `INV_VAL_005` (no-negative-balance) is enforced at the source location at SR post time. |
| `INV_XMOD_003` | [[physical-count]] | A completed physical count produces a `count_difference` per `(location_id, product_id, lot_no)` line — if positive (system understated), the variance posts as `tb_stock_in` with `enum_transaction_type = adjustment_in`; if negative, as `tb_stock_out` with `adjustment_out`. Each variance write requires Inventory Controller approval per `INV_AUTH_003`. The physical-count document remains the source-of-truth audit anchor for the count event; the inventory transactions are the ledger consequence. |
| `INV_XMOD_004` | [[spot-check]] | Same posting path as physical-count but scoped to a subset of products / locations. Spot-check variances post under the same `stock_in` / `stock_out` umbrella; the difference is procedural (partial count, mid-period) not structural. |
| `INV_XMOD_005` | [[inventory-adjustment]] | Manual adjustments unrelated to a count run (data fix-ups, found stock, vendor-replaced damaged stock) post via `tb_stock_in` / `tb_stock_out` with the appropriate `adjustment_type_id` reason. Threshold-based approval per `INV_AUTH_001` / `INV_AUTH_002` / `INV_AUTH_003`. The originating reason code (e.g. `BREAKAGE`, `EXPIRY_WRITE_OFF`, `FOUND_STOCK`, `MIGRATION_FIX`) is required and is what distinguishes adjustment-types in reporting. |
| `INV_XMOD_006` | [[costing]] | Costing reads `tb_inventory_transaction_cost_layer` for FIFO layer ordering and for the moving-average `average_cost_per_unit`. The costing module does not write to inventory — it consumes the cost data. COGS for an outbound is `out_qty × cost_per_unit` as picked at `INV_CALC_005` / `INV_CALC_006` and aggregated per period for the Income Statement. |
| `INV_XMOD_007` | [[good-receive-note]] credit note path | Vendor credit notes (`tb_credit_note`) post `credit_note_amount` (price adjustment, no qty change) or `credit_note_quantity` (qty return) transactions per `INV_POST_007` / `INV_POST_008`. The credit-note document is the audit anchor; the inventory transaction is the ledger effect. The Finance module reconciles the credit-note's AP impact against the receipt's AP entry. |
| `INV_XMOD_008` | Finance / GL | Inventory-to-GL reconciliation runs as a period-close step. The sum of cost-layer `total_cost` and `diff_amount` across all `inventory`-type movements in the period must equal the GL Inventory control account's net change. Variance flags a reconciliation exception that Finance investigates before period lock (`INV_AUTH_006`). |
| `INV_XMOD_009` | [[product]] | The product's `costing_method` (FIFO vs WEIGHTED_AVERAGE) is read by the inventory module at every outbound post. Changing the costing method on a product with non-zero on-hand is **blocked** (the cost-layer history would be inconsistent under a new method); the change requires draining the stock first or running a one-time migration entry under elevated authorisation. |
| `INV_XMOD_010` | All movement-generating modules | Every module that mutates inventory (GRN, SR, count, spot-check, credit note, manual stock-in / stock-out) writes via the same `tb_inventory_transaction` API. There is no module-specific bypass; the polymorphic `inventory_doc_type` is the only differentiator at the ledger level. This is the single chokepoint that makes audit trail trustable. |
