# Inventory — Test Scenarios — Store Keeper

> **At a Glance**
> **Persona:** Store Keeper &nbsp;·&nbsp; **Module:** [inventory](/en/inventory/inventory) &nbsp;·&nbsp; **Scenarios:** ~29
> **Categories:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** maps to `501-grn.spec.ts`, `720-stock-issue.spec.ts` in `../carmen-inventory-frontend-e2e/`

This page captures the test scenarios that the Store Keeper persona directly drives in the `inventory` module. The Store Keeper initiates non-GRN / non-SR adjustments (manual stock-in for found stock / count overage / vendor-replacement, manual stock-out for breakage / expiry / count shortage), executes physical and spot counts at the location level, and operates within the auto-approve threshold for below-cost-impact documents. Their inventory-module ownership begins when a discrepancy is identified at the floor and ends when the source document (`tb_stock_in` / `tb_stock_out` / `tb_count_stock`) either auto-posts (below threshold) or hands off to the Inventory Controller for approval. Scenarios are grouped into **happy paths** (stock-in for found stock, stock-out for breakage, multi-line count-derived rollup, FOC / no-cost stock-in for vendor-supplied free replacement, perishable / lot-tracked entry), **RBAC** (Store Keeper scope vs Inventory Controller scope, segregation of duties on receive-then-write-off), **validation** (negative tests against `INV_VAL_001`–`INV_VAL_013` that the Store Keeper can trigger at submit time), and **edge cases** around tolerance boundaries, decimal precision on lot qty, concurrent posts, location-type gates, and count-rollup behaviour. Cross-persona handoffs that pivot off the Store Keeper (Scenarios 1, 2, 4, 7, 9, 10 in the parent overview) live in [04-test-scenarios.md](./04-test-scenarios.md), not here.

## 2. Permission / Authorization

| # | Scenario | Expected behaviour (allow/deny + reason) |
| - | -------- | --------------------------------------- |
| SK-PERM-01 | Store Keeper raises stock-in / stock-out below threshold | **Allow.** Per `INV_AUTH_001` / `INV_AUTH_002`, the Store Keeper holds create authority on `tb_stock_in` / `tb_stock_out` documents within their `tb_user_location` scope. Below the auto-approve threshold, submit auto-advances `draft → completed` and posts to inventory directly. |
| SK-PERM-02 | Store Keeper attempts to approve another Store Keeper's pending stock-out | **Deny — Inventory Controller required.** Per `INV_AUTH_003`, only the Inventory Controller (or above) can fire the `in_progress → completed` transition on a document above the auto-approve threshold; below threshold, the system auto-fires. A Store Keeper attempting to approve another Store Keeper's `in_progress` document via API returns `"Approval of above-threshold inventory adjustments requires the Inventory Controller role."` Document stays `in_progress`; no inventory transaction. |
| SK-PERM-03 | Store Keeper attempts to edit a completed (`posted`) inventory transaction | **Deny — terminal state.** Per `INV_POST_012`, every `tb_inventory_transaction` is immutable on write; the reversal is a new compensating transaction, not an edit. Direct `PATCH /inventory-transaction/{id}` from any role returns `"Inventory transactions are immutable; reverse via a compensating stock-in / stock-out."` The Store Keeper raises a compensating stock-in / stock-out instead. |
| SK-PERM-04 | Store Keeper attempts to post to a location outside their `tb_user_location` scope | **Deny — scope.** The location picker filters to the user's scope; a direct API call with an out-of-scope `location_id` returns `"You do not have permission to post to location <LOC-X>."` `INV_VAL_003` runs at submit, but the scope check runs at the API gateway before that — both layers block. |
| SK-PERM-05 | Store Keeper attempts to post to a `direct`-cost or `inactive` location | **Deny per `INV_VAL_009` / `INV_VAL_003`.** Direct-cost locations carry no balance; stock-in / stock-out is meaningless there and rejected. Inactive (soft-deleted or `is_active = false`) locations are filtered from the picker; direct API submission returns the corresponding validation error. |
| SK-PERM-06 | Segregation of duties — write-off of stock the Store Keeper originally received | **Deny when above SoD threshold per `INV_AUTH_010`.** When a Store Keeper attempts to write off a lot that they themselves received (the originating GRN's `created_by_id` matches the stock-out's user) and the cost impact exceeds the tenant SoD threshold, the submit is rejected with `"You created the receipt for this lot; an independent adjuster must initiate the write-off (SoD INV_AUTH_010)."` Routine small write-offs are exempt. The Controller (or a different Store Keeper) initiates instead. |
| SK-PERM-07 | Store Keeper attempts to post into a `closed` or `locked` period | **Deny per `INV_VAL_008`.** Server returns `"Cannot post into period <YYMM>: period is <status>. Re-open the period (closed only) or post a current-period restatement (locked)."` Document stays `draft`; the Store Keeper changes the transaction date to the current open period or escalates to Finance Manager to re-open the closed period. Locked periods cannot be re-opened. |

## 3. Validation / Error

| # | Scenario | Trigger | Expected error |
| - | -------- | ------- | -------------- |
| SK-VAL-01 | Missing product or location at submit (`INV_VAL_002`) | Store Keeper adds a stock-in line with `product_id = null` (cleared by mis-click) or `location_id = null` (scope picker not selected); clicks Submit. | **Reject at submit** — `INV_VAL_002`. Server returns `"Product is required on every inventory transaction detail line; location is required for non-rollup movements."` Atomic — the whole document is rejected, not just the line. Store Keeper picks the missing field and re-submits. |
| SK-VAL-02 | Inactive location (`INV_VAL_003`) | A `tb_location` row was deactivated by Sysadmin between when the Store Keeper opened the screen and when they submitted (rare race); the stale `location_id` is no longer valid. | **Reject at submit** — `INV_VAL_003`. Server returns `"Cannot post to an inactive or soft-deleted location."` Store Keeper refreshes the picker (which now excludes the inactive location) and re-picks. |
| SK-VAL-03 | Quantity sign mismatch (`INV_VAL_004`) | Direct API call with `inventory_doc_type = stock_in` but a negative `qty` on the detail (or vice versa for `stock_out`); typically a client-side bug. | **Reject at submit** — `INV_VAL_004`. Server returns `"Quantity sign does not match transaction type (inbound transactions require positive qty; outbound transactions require negative qty)."` The form layer normally guards this; backend re-checks. |
| SK-VAL-04 | Negative balance attempt (`INV_VAL_005`) | Outbound stock-out for `P-1` at `LOC-A`; available qty at `LOT-1` is 5; Store Keeper enters `qty = 7`. | **Reject at submit** — `INV_VAL_005`. Server returns `"Outbound movement would drive on-hand at (LOC-A, P-1, LOT-1) below zero. Available: 5, requested: 7."` Document stays at `draft`; Store Keeper reduces qty to 5 (or to a multi-lot split — 5 from `LOT-1`, remainder from `LOT-2`), or escalates to Controller to investigate if the discrepancy suggests a missed inbound. Maps to parent Scenario 10. |
| SK-VAL-05 | Existing-lot collision (`INV_VAL_006`) | Store Keeper enters a `lot_no` claiming "new lot" but the lot already exists for the same `(product_id, location_id)`; or enters an outbound from a `lot_no` that doesn't exist at the location. | **Reject at submit** — `INV_VAL_006`. For inbound new-lot collision: `"Lot <lot_no> already exists for product <product> at location <location>; lot identity must be unique."` For outbound missing-lot: `"Lot <lot_no> at location <location> is not available for consumption."` |
| SK-VAL-06 | Negative cost (`INV_VAL_007`) | Direct API call with `cost_per_unit = -฿5.00` on a stock-in (typically a client-side validation bypass attempt). | **Reject at submit** — `INV_VAL_007`. Server returns `"Cost per unit and qty must be non-negative; in_qty and out_qty are mutually exclusive on a single cost-layer row."` Document stays `draft`. |
| SK-VAL-07 | Backdated post into a closed period (`INV_VAL_008`) | Store Keeper sets the stock-in's transaction date to a date inside a now-closed period (e.g. forgot to post yesterday; period closed overnight). | **Reject at submit** — `INV_VAL_008`. Server returns `"Cannot post into period <YYMM>: period is closed. Re-open the period or post a current-period restatement (locked)."` Store Keeper resets the date to today or escalates to Finance Manager. Maps to parent Scenario 11. |
| SK-VAL-08 | Stock-in to a direct-cost location (`INV_VAL_009`) | The location picker filters to inventory-type, but a direct API call with `location_id` referencing a direct-cost location attempts to post a stock-in. | **Reject at submit** — `INV_VAL_009`. Server returns `"Cannot post outbound movement to direct-cost location <location>. Direct locations bypass inventory; consumption is implicit at receipt."` (Symmetrically rejected for stock-out; direct locations carry no balance to adjust.) Store Keeper picks a valid inventory-type location. |
| SK-VAL-09 | Consignment-to-inventory implicit transfer rejected (`INV_VAL_010`) | A consignment location's stock is taken into ownership without an explicit re-classification document; a direct API call attempts to write a transfer from consignment to inventory. | **Reject per `INV_VAL_010`** — server returns `"Consignment-to-inventory ownership transfer requires an explicit re-classification stock-in document."` Store Keeper raises an explicit stock-in document for the inventory side and (under elevated authority) a stock-out from the consignment side. |

## 4. Edge Cases

| # | Scenario | Condition | Expected |
| - | -------- | --------- | -------- |
| SK-EDGE-01 | Exact tolerance boundary on threshold | Auto-approve threshold = `฿500.00`; Store Keeper submits a stock-in with `qty × cost_per_unit = ฿500.00` exactly. | **Boundary inclusive — auto-approve.** The system treats `≤` as the auto-approve condition. Document advances `draft → completed` without Controller handoff. `฿500.01` would exceed the threshold and route to Controller. |
| SK-EDGE-02 | Decimal precision on lot qty | Store Keeper enters `qty = 5.12345` (5dp precision per `Decimal(20, 5)`); cost-per-unit `฿11.33333` (5dp average from weighted-average product). | **Accepted at storage.** Stored at full 5dp on `qty` and `cost_per_unit`; `total_cost = qty × cost_per_unit` rounded display to 2dp (`฿58.04` from `5.12345 × 11.33333 = 58.04148...`); intermediate computation carries 5dp. Mirrors `INV_CALC_012`. |
| SK-EDGE-03 | Concurrent inbound posts to same lot from two sessions | Store Keeper A submits stock-in for `(LOC-A, P-1, LOT-1)` qty 5 at `T`; Store Keeper B submits the same at `T+200ms`; both below threshold. | **Both post — append-only.** No row-level lock conflict; both inventory transactions land successfully; cost-layer rows ordered by `lot_seq_no` (or by `created_at` for same `lot_no`); on-hand at `(LOC-A, P-1, LOT-1)` advances by 10 (5+5); weighted-average recomputed sequentially per `INV_CALC_007`. Maps to parent Scenario 9. |
| SK-EDGE-04 | Multi-line stock-in spanning above-threshold aggregate but below per-line | Stock-in with 6 lines: each line `฿100.00` (below threshold per line); aggregate `฿600.00` (above threshold). | **Routes for Controller approval based on aggregate, not per-line.** Per Section 3 decision branch of [03-user-flow-store-keeper.md](./03-user-flow-store-keeper.md), the threshold check sums across lines. The whole document routes to Controller; the Controller may approve as-is or send back for split (the Store Keeper raises two smaller documents to stay under threshold per document — typically a process change, not a system rule). |
| SK-EDGE-05 | Count-derived rollup with mixed outcomes (overage AND shortage on same product across lots) | A count run identifies `LOT-1` short by 1, `LOT-2` over by 2 for the same product `P-1` at `LOC-A`. | **Two rollup documents, not netted.** The count commit creates one `tb_stock_in` (overage on `LOT-2`, qty 2) AND one `tb_stock_out` (shortage on `LOT-1`, qty 1) — not a single netted `tb_stock_in` for qty 1. This preserves the per-lot audit trail. Both auto-advance to `completed` under Controller authority per parent Scenario 4. |
| SK-EDGE-06 | Period rolls over while a stock-in is `draft` | Store Keeper starts a stock-in at `2026-04-30 23:50`; submits at `2026-05-01 00:05`; the period rolled `2026-04 → 2026-05` overnight (`2026-04` auto-closed by scheduled period-end job). | **Transaction date drives the period.** If the Store Keeper's transaction date on the document is `2026-04-30`, submit is rejected per `INV_VAL_008` (the period is now `closed`). If the date is `2026-05-01`, submit succeeds against the open `2026-05` period. Default transaction date is the submit time, so the document typically posts into `2026-05` correctly. |
