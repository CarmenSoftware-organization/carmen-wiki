# Inventory — Test Scenarios — Inventory Controller

> **At a Glance**
> **Persona:** Inventory Controller &nbsp;·&nbsp; **Module:** [inventory](/en/inventory/inventory) &nbsp;·&nbsp; **Scenarios:** ~26
> **Categories:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** maps to `720-stock-issue.spec.ts`, `900-period-end.spec.ts` in `../carmen-inventory-frontend-e2e/`

This page captures the test scenarios that the Inventory Controller persona directly drives in the `inventory` module. The Controller owns balance accuracy: they approve `tb_stock_in` / `tb_stock_out` documents above the Store Keeper auto-approve threshold per `INV_AUTH_003`, commit count-variance rollups arising from physical / spot counts per `INV_XMOD_003` / `INV_XMOD_004`, maintain per-product / per-location stock policy on `tb_product_location` per `INV_AUTH_004`, and sign off on variance before period close. Their inventory-module ownership begins when a Store Keeper escalates a document or a count completes, and ends when (a) the document posts on Controller approval (below Finance threshold), (b) the document escalates further to Finance for very-large cost impact, (c) the count is committed and the rollup posts, or (d) the period-end variance sign-off is recorded. Scenarios are grouped into **happy paths** (approval, count-variance commit, stock-policy edit, period-end sign-off), **RBAC** (Controller scope vs Store Keeper / Finance / Sysadmin, self-approval prevention), **validation** (negative tests on approval gates), and **edge cases** (concurrent approvals, count rollup with mixed outcomes, threshold-boundary routing).

## 1. Happy Path

| # | Scenario | Pre-condition | Steps | Expected |
| - | -------- | ------------- | ----- | -------- |
| IC-HP-01 | Approve above-threshold stock-out — fires the post | Store Keeper has submitted a `tb_stock_out` document for breakage write-off at `LOC-A`, qty 50 of `P-1` at FIFO-picked cost `฿10.00` (aggregate `฿500.00` — above auto-approve `฿500.00 - epsilon` so routes to Controller, but below Finance ceiling `฿5,000.00`). Document at `doc_status = in_progress`; Controller `controller@blueledgers.com` logged in. | 1. Open the adjustment approval queue. 2. Click into the document; review the lines, the reason code (`BREAKAGE`), and the cost-pick preview. 3. Confirm Store Keeper attached photo evidence. 4. Click **Approve**. | `tb_stock_out.doc_status = in_progress → completed`; one outbound `tb_inventory_transaction` written (`inventory_doc_type = stock_out`); detail row with `qty = -50`, `cost_per_unit = ฿10.00`; outbound cost-layer row(s) per FIFO (one row if all 50 fit in one lot, multiple if spanning lots per `INV_CALC_005`); on-hand reduced. Activity log records `{ actor: controller, action: 'approve_post' }`. Maps to parent Scenario 2. |
| IC-HP-02 | Reject a stock-in document — returns to Store Keeper | Store Keeper has submitted a stock-in for a new lot (per SK-HP-03) but the cost is anomalous — `฿50.00` per unit when the vendor pricelist last-price is `฿15.00`. Document at `in_progress` in the Controller's queue. | 1. Open the document. 2. Review the cost vs the pricelist. 3. Click **Reject** with comment "Cost-per-unit `฿50.00` diverges from pricelist last-price `฿15.00`; please verify with vendor pricing reference and re-submit." | Document `doc_status = in_progress → draft`; Controller's comment appended to the activity log; Store Keeper's inbox shows the rejection; no inventory transaction written; on-hand unchanged. The Store Keeper resubmits with corrected cost or voids the document. |
| IC-HP-03 | Approve a new-lot stock-in (`SK-HP-03` follow-on) | Continued from SK-HP-03: Store Keeper has submitted a new-lot stock-in for `LOT-NEW` at `LOC-A`, `P-1`, qty 10, cost `฿8.00`, total `฿80.00` (below cost threshold but new-lot rule routes for Controller). | 1. Controller opens the document. 2. Reviews the new-lot creation (no prior `LOT-NEW` in the cost-layer ledger for `(LOC-A, P-1)`). 3. Confirms cost reference (`[vendor-pricelist](/en/inventory/vendor-pricelist)` last-price match within tenant tolerance). 4. **Approve**. | Document `completed`; inbound inventory transaction; inbound cost-layer row with new `lot_no = LOT-NEW`, `lot_index = 1`, `lot_seq_no` = max existing + 1; `current_lot_no = LOT-NEW` on the detail row; on-hand at `(LOC-A, P-1, LOT-NEW)` advances to 10. |
| IC-HP-04 | Commit count-variance rollup — aggregate below Finance threshold | A physical count at `LOC-A` has completed (`tb_count_stock.status = completed`) with variance lines: overage on `P-1` (+2 units, `฿20.00` cost impact at `LOT-1`), shortage on `P-2` (-5 units, `฿100.00` at WA cost), shortage on `P-3` (-1 unit, `฿15.00` at `LOT-3`). Aggregate cost impact `฿135.00` (well below Controller threshold; well below Finance threshold). Controller has reviewed the variance dashboard. | 1. Open the count-variance dashboard. 2. Open the count for `LOC-A`. 3. Per-line decision: `P-1` overage `post`; `P-2` shortage `post`; `P-3` shortage `post`. 4. Click **Commit count decisions**. | One `tb_stock_in` document auto-created with `P-1` overage line (`doc_status = completed` immediately under Controller authority); one `tb_stock_out` with `P-2` + `P-3` shortage lines (`completed`). Each writes a `tb_inventory_transaction` with the appropriate cost-layer rows. `tb_count_stock.status = completed_posted`. Activity log on each rollup carries `info.source_count_id = <count_id>`. Maps to parent Scenario 4. |
| IC-HP-05 | Route count-variance line back for re-count | Count at `LOC-B` shows shortage of `P-HV` (high-value product, expected qty 10, counted 6 — gap of 4 at `฿250.00` per unit = `฿1,000.00` cost impact); Controller is suspicious (no theft/breakage event known; product is high-value). | 1. Open the count-variance dashboard. 2. Open the count for `LOC-B`. 3. Per-line decision on `P-HV`: `re-count`. 4. Commit count decisions (only the other variance lines post; the `re-count` line is excluded). | The flagged line stays staged in the count document with a `re-count` status; routes back to Store Keeper for a re-count window; `tb_count_stock.status` stays `completed` (not `completed_posted` until all lines resolve). Other variance lines (if any) post via the normal rollup; the re-count line stays open until the Store Keeper completes the re-count. |
| IC-HP-06 | Edit stock policy for a product / location | Controller has reviewed the consumption pattern for `P-1` at `LOC-A` and decides to raise `min_qty` from 10 to 15 (consumption velocity has increased post-summer-promotion). | 1. Open the stock policy dashboard. 2. Filter to `LOC-A`. 3. Edit `P-1`'s `min_qty` to 15; save. | `tb_product_location.min_qty = 15` persisted with `doc_version` incremented; activity log records the change. Replenishment alert triggers fire on the new threshold. Existing on-hand unaffected; future on-hand reads compare against the new `min_qty`. Maps to `INV_AUTH_004`. |
| IC-HP-07 | Period-end variance sign-off | All in-flight stock-in / stock-out / count-stock documents for closing period `2026-04` are at terminal state (`completed` or `voided`); no `investigate`-flagged count line remains open; Controller has reviewed the period's variance summary. | 1. Open the period-end pre-flight dashboard for `2026-04`. 2. Review the variance summary (period totals: total overage cost, total shortage cost, total breakage / expiry / write-off, per-location). 3. Click **Sign off variance review for 2026-04**. | Sign-off recorded with `actor = controller`, `signed_at = now()`; the Finance period-close dashboard shows the Controller sign-off as a green checkmark; Finance can now run the period close. Activity log: `{ event: 'variance_signoff', period: '2026-04', actor: controller }`. Maps to parent Scenario 5 pre-condition. |

## 2. Permission / Authorization

| # | Scenario | Expected behaviour (allow/deny + reason) |
| - | -------- | --------------------------------------- |
| IC-PERM-01 | Controller approves above-Store-Keeper-threshold adjustments | **Allow per `INV_AUTH_003`.** The **Approve** button is enabled on documents at `in_progress` within the Controller's threshold; clicking fires the post. Below Controller threshold but above Store Keeper auto-approve — the Controller is the approver. Above Controller threshold — routes to Finance per Scenario IC-PERM-04. |
| IC-PERM-02 | Controller cannot approve their own document | **Deny — SoD `INV_AUTH_010` analogue.** When a Controller is also a Store Keeper (rare in production but common in small teams), the Approve button is hidden on documents whose `created_by_id = current_user`; a direct API call returns `"Cannot approve your own inventory adjustment; SoD requires an independent approver."` The system routes to a peer Controller or to Finance for the approval. |
| IC-PERM-03 | Controller edits stock policy on `tb_product_location` | **Allow per `INV_AUTH_004`.** The policy panel is editable for the Controller role; Store Keeper sees it read-only; Finance and Sysadmin see it editable too (for cross-coordination). Changes apply prospectively per the snapshot rule. |
| IC-PERM-04 | Controller attempts to approve an above-Finance-threshold adjustment | **Allow Controller's first signature; Finance-pending sub-state.** The Controller's approval click moves the document to a Finance-pending state (not directly to `completed`). Per `INV_AUTH_005`, Finance is the role that fires the post on documents above the Finance threshold. Maps to parent Scenario 3. |
| IC-PERM-05 | Controller attempts to lock the period | **Deny — Finance Manager only.** Per `INV_AUTH_006`, only the Finance Manager can advance `tb_period.status = closed → locked`. The Controller's role is variance sign-off, not period state advancement. The lock button is hidden for the Controller role. |
| IC-PERM-06 | Controller attempts to edit a `posted` inventory transaction | **Deny — terminal state per `INV_POST_012`.** Same as SK-PERM-03 — every `tb_inventory_transaction` is immutable. Reversal is via a compensating stock-in / stock-out, which the Controller can initiate as the approving authority on the compensation. |
| IC-PERM-07 | Controller attempts to configure adjustment-type / location-type / costing-method | **Deny — Sysadmin only per `INV_AUTH_008`.** The configuration console is not visible to the Controller role. The Controller may request configuration changes (e.g. a new adjustment-type reason code) via service ticket; the Sysadmin owns the change. |

## 3. Validation / Error

| # | Scenario | Trigger | Expected error |
| - | -------- | ------- | -------------- |
| IC-VAL-01 | Controller approves a stock-out that would drive on-hand negative | A `tb_stock_out` document submitted while on-hand was `10`; between submit and approve, another stock-out posted reducing on-hand to `3`; the document's `qty = 7` would now drive on-hand negative. | **Reject at approve** — `INV_VAL_005`. Server returns `"Outbound movement would drive on-hand below zero at (location, product, lot). Available: 3, requested: 7."` Document stays `in_progress`; Controller routes the document back to Store Keeper with the new state, or splits / reduces the qty on the Store Keeper's behalf (with an edit-and-approve action) per the system's auth model. |
| IC-VAL-02 | Controller approves an adjustment in a now-`closed` period | A document submitted while period was `open`; between submit and approve, the period auto-closed (rare race with scheduled period-end job); the transaction date is now inside the `closed` period. | **Reject at approve** — `INV_VAL_008`. Server returns `"Cannot post into period <YYMM>: period is closed."` The Controller updates the transaction date to the current open period (typically with the Store Keeper's concurrence) and re-approves, or routes to Finance Manager to re-open the closed period for the exceptional case. |
| IC-VAL-03 | Controller approves a count rollup with mismatched aggregate | A count rollup whose aggregate cost impact has changed between count-commit time and Controller-confirm time (because an interim physical inventory change altered the system qty). | **Reject at commit** — defensive guard: server returns `"Count rollup aggregate has changed since review. Refresh the variance dashboard and re-commit."` The Controller refreshes the count, re-reviews the variance lines, and re-commits with the current aggregate. |
| IC-VAL-04 | Controller attempts to commit a count with unresolved `re-count` lines | The count dashboard shows the count with one or more lines flagged `re-count`; the Controller clicks Commit anyway (mis-click). | **Reject at commit** — server returns `"Count cannot be committed while <N> lines are flagged for re-count. Resolve the re-count lines (Store Keeper re-counts) before committing."` The Controller waits for the re-count to complete or removes the `re-count` flag on the line (escalating it to `post` or `investigate`) and re-commits. |
| IC-VAL-05 | Controller attempts policy edit with invalid bounds | Controller sets `min_qty = 20`, `max_qty = 15` on `tb_product_location` (max < min — incoherent). | **Reject at save** — shape validation: server returns `"max_qty must be ≥ min_qty on stock-policy row."` The Controller corrects one of the values and re-saves. (Similar shape rules: `re_order_qty ≥ 0`, `par_qty ≥ 0`.) |
| IC-VAL-06 | Controller signs off period with open in-flight documents | Controller clicks **Sign off variance review for 2026-04** while a `tb_stock_out` document for the period is still at `doc_status = in_progress` (not yet approved). | **Reject at sign-off** — server returns `"Cannot sign off period <2026-04> while <N> in-flight inventory documents remain at draft or in_progress."` The Controller resolves the in-flight documents (approve or reject each) before re-signing-off. The Finance period-close run respects the same gate. |

## 4. Edge Cases

| # | Scenario | Condition | Expected |
| - | -------- | --------- | -------- |
| IC-EDGE-01 | Threshold boundary on Controller approval routing | Document aggregate cost impact = `฿5,000.00` (exact Finance threshold). | **Boundary inclusive — at threshold, Controller approval is terminal.** The system treats `≤` as Controller-final. `฿5,000.01` would route to Finance. Same boundary-inclusive rule as SK-EDGE-01. |
| IC-EDGE-02 | Count-rollup with overage AND shortage on same product across lots | Count identifies `LOT-1` short by 3, `LOT-2` over by 2 for `P-1` at `LOC-A`. | **Two rollup documents per parent Scenario 4 / SK-EDGE-05.** Single `tb_stock_in` (overage on `LOT-2`, qty 2) AND single `tb_stock_out` (shortage on `LOT-1`, qty 3); not netted. The Controller approves both in the same commit. Per-lot audit preserved. |
| IC-EDGE-03 | Concurrent approvals on different documents in the queue | Controller A approves document `D-1` at `T`; Controller B (peer) approves document `D-2` (different `(location, product, lot)`) at `T+200ms`. | **Both succeed — no shared row contention.** Each approval is row-level on its document; both transactions post; no conflict. The peer-controller model is supported via RBAC — multiple Controllers may have the same approval authority on overlapping or disjoint scope. |
| IC-EDGE-04 | Concurrent approval on the same document by two Controllers | Both Controllers click Approve on `D-1` at the same time. | **First click wins; second is rejected** via optimistic concurrency. The losing click returns `"This document was approved by another user at <timestamp>. Please refresh."` Document is `completed`; one inventory transaction; one set of cost-layer rows; no double-post. |
| IC-EDGE-05 | Stock-policy edit during in-flight replenishment | A purchase request was just raised for `P-1` at `LOC-A` based on the old `min_qty = 10`; Controller raises `min_qty` to 15 while the PR is still `draft`. | **PR unaffected by policy snapshot rule.** The PR retains its create-time snapshot of the policy (`min_qty = 10` at create); the new policy applies prospectively to subsequent PR raises. The Controller's edit on `tb_product_location` succeeds; the in-flight PR's evaluation against `min_qty` doesn't re-run. |
| IC-EDGE-06 | Period-end sign-off with `investigate`-flagged count line | A count line was flagged `investigate` by the Controller earlier in the period; resolution is still pending (Finance reviewing); Controller attempts sign-off. | **Reject at sign-off** per IC-VAL-06 variant: server returns `"Cannot sign off period <2026-04> while <1> count variance line is flagged investigate (route via Finance)."` Resolution happens before sign-off: either Finance clears the investigate flag (with or without a post) or the Controller un-flags after resolution. |
