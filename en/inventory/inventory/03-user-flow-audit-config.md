# Inventory — User Flow — Audit & Config

> **At a Glance**
> **Persona:** Audit / Config (Auditor + System Administrator) &nbsp;·&nbsp; **Module:** [[inventory]] &nbsp;·&nbsp; **Workflow stages:** Off-path observers — Sysadmin configures `tb_location` types, `tb_product.costing_method`, `tb_adjustment_type`, `tb_period`, thresholds, RBAC, integration endpoints; Auditor reads transactions / cost-layer / snapshots, runs lot-recall traces and period-snapshot reconciliation queries &nbsp;·&nbsp; **Key permissions:** Sysadmin owns configuration (`INV_AUTH_008`); Auditor read-only across full dataset including soft-deleted rows
> **What this persona does:** Configures the inventory module's structural primitives (Sysadmin); audits movement / cost-layer / period-snapshot chains and lot-recall traces (Auditor).

## 1. Role in This Module

The **Audit / Config** persona folds two non-transactional roles into a single flow because neither participates in the inventory movement-and-period lifecycle. The **System Administrator** sub-persona configures the structural primitives that every other inventory action depends on: `tb_location` definition (in particular `location_type` per `enum_location_type` — `inventory` / `direct` / `consignment` — and `physical_count_type`), the costing method per product on `tb_product.costing_method` (FIFO / WEIGHTED_AVERAGE — which the cost-layer post engine reads at every outbound), the `tb_adjustment_type` reason-code list that Store Keeper picks from and Inventory Controller / Finance approve against, the `tb_period` definitions for accounting periods, the threshold values that drive Store Keeper auto-approve / Controller-route / Finance-route routing, RBAC role definitions and the `tb_user_location` scope mapping per `INV_AUTH_008`, and the integration endpoints with the upstream source modules (GRN, SR, count, credit note, manual stock-in / stock-out). The **Auditor** sub-persona is **strictly read-only** across the full inventory dataset — all `tb_inventory_transaction` rows including soft-deleted (compensated) reversals, the full `tb_inventory_transaction_cost_layer` ledger, every `tb_period_snapshot`, the configuration history, and the cross-module activity logs (GRN commits, SR posts, count completions) — and runs **two specialised query patterns**: (1) **lot-recall traces** that walk the cost-layer ledger forward from a receiving GRN to all downstream consumption movements (per the recall pattern in [[good-receive-note/03-user-flow-audit-config]]), and (2) **period-snapshot reconciliation queries** that verify the snapshot rows match the cost-layer activity for the closed period (an audit defence against silent ledger drift). Neither sub-persona ever appears in the transactional flow: the Sysadmin's edits change rules that apply to **future** movements (in-flight source documents at `draft` / `in_progress` retain a snapshot of the rules in force when they were created; `posted` inventory transactions are immutable and retain their configuration snapshot), and the Auditor's queries return data without writing any state. Critically, the segregation-of-duties enforcement `INV_AUTH_010` (write-off ≠ original receiver above threshold) and the period-lock prohibition `INV_AUTH_006` (locked periods are immutable for all roles) live in the RBAC / posting-rule layer that this persona owns the configuration of.

## 2. Entry Point and Primary Flow

**Entry points:** Sub-persona-specific, no overlap with the transactional inventory list views.

- **Auditor — inventory audit-log workspace** — read-only screen that surfaces `tb_inventory_transaction` rows with their `inventory_doc_type` source linkage, `tb_inventory_transaction_cost_layer` activity, `tb_period_snapshot` rollups, and the configuration-history feed. Filter facets: date range, location, product, lot number, source-document type (GRN / SR / count / credit note / stock-in / stock-out), period.
- **Sysadmin — inventory configuration console** — administrative screen scoped by tenant, with panels for **Locations** (definition, `location_type`, `physical_count_type`, delivery point linkage), **Products** (costing method per product — typically deferred to the [[product]] module's config console with a read-only pass-through here), **Adjustment Types** (`tb_adjustment_type` reason codes for stock-in / stock-out), **Periods** (definition of `tb_period` rows, period-cadence schedule), **Thresholds** (Store-Keeper auto-approve, Inventory-Controller approval cap, Finance approval ceiling), **RBAC** (`tb_user_location` mapping, role-to-action scope), **Integrations** (the source-module endpoints).

### 2.1 Auditor flow (read-only, lot-trace + period-snapshot verify) — 6 steps

1. **Open the inventory audit workspace** scoped to the period under review. The screen aggregates `tb_inventory_transaction` rows joined to their source documents (via the polymorphic `inventory_doc_no`), the cost-layer ledger, and the period-snapshot rollups into a single navigable view.
2. **Filter the scope.** Typical investigation scopes: a specific lot (for recall), a specific location (for shrinkage analysis), a specific product (for vendor-performance / cost-variance review), a specific period (for period-close audit). Filter results are render-only; no edit affordances.
3. **Drill into a movement.** Open a specific `tb_inventory_transaction` row to see the header (source-document linkage), the detail rows (per-product / per-lot ledger lines), and the cost-layer rows (in_qty / out_qty / cost_per_unit / lot_no with FIFO ordering by `lot_seq_no`). Walk the source linkage back to the originating document (e.g. GRN, SR, count) for the full audit chain.
4. **Optional: lot-recall trace** (full trace for a specific lot). Enter `lot_no` and run the trace. **Backward** trace: identify every inventory transaction that introduced the lot (via `current_lot_no` on the inbound detail row, or via `lot_no` on the cost-layer's `transaction_type = good_received_note` or `adjustment_in` rows). **Forward** trace: identify every downstream movement that consumed from the lot (via `from_lot_no` on outbound detail rows and `transaction_type = issue` / `transfer_out` / `adjustment_out` / `credit_note_quantity` cost-layer rows). Produces the full chain-of-custody report.
5. **Optional: period-snapshot reconciliation query.** For a closed period, verify that `Σ tb_inventory_transaction_cost_layer.in_qty − Σ out_qty` (in the period, scoped to a location / product / lot key) equals `tb_period_snapshot.closing_qty − opening_qty` for the same key. A gap indicates ledger drift — a defensive audit query.
6. **Export the report.** Audit trail or lot-trace exports to CSV / PDF for the recall file, the external auditor binder, the incident-response report, or the period-close audit working papers. Sensitive-field exports (unit costs, vendor terms via joined tables) require secondary approval per the audit-pattern in [[good-receive-note/03-user-flow-audit-config]] (Auditor / Controller co-approval).

### 2.2 Sysadmin flow (configuration change) — 6 steps

1. **Identify the configuration need.** Triggers include opening a new location (need `tb_location` row with `location_type`, delivery point), retiring an old location (need confirmation that on-hand is zero — see Decision Branches), switching a product's costing method (need migration plan — see Decision Branches), adding / retiring an adjustment reason code, defining a new accounting period (typically automated, but exception periods like a short period for a fiscal-year change land here), adjusting an approval threshold, granting `tb_user_location` scope to a new user, or rewiring an integration endpoint (rare).
2. **Open the inventory configuration console** and navigate to the relevant panel.
3. **Adjust the configuration.** Edit the relevant fields per the panel's shape validation rules (location code unique within tenant, costing method one of {FIFO, WEIGHTED_AVERAGE}, adjustment type code unique, period start/end non-overlapping, threshold non-negative, RBAC role / action / scope triple well-formed, integration endpoint URL well-formed).
4. **Preview and impact analysis.** Before saving, the console runs an impact preview: how many in-flight source documents (`tb_stock_in.doc_status = draft` / `in_progress`, etc.) reference the affected configuration, how many `tb_inventory_transaction` rows in the current open period are at risk of the change affecting reporting, what user sessions hold the affected RBAC role / location scope, what downstream modules consume the affected configuration (e.g. changing a location's `location_type` from `inventory` to `direct` would affect [[good-receive-note]] receipt posting). The preview flags blockers (active references that would deadlock) and warnings (changes that need user-side adoption).
5. **Save the change.** The config is persisted with an effective-from timestamp; the previous version is retained in the configuration history for audit. The change is reflected immediately for **new** movements created after the save; in-flight source documents at `draft` / `in_progress` retain a snapshot of the configuration in force when they were created (location definition, costing method, threshold, RBAC at create time) unless the field is explicitly recomputed by re-saving. **Posted inventory transactions** always retain the historical configuration via their source-document snapshot — the cost-layer's `cost_per_unit` was computed at post time with then-current rules and is immutable.
6. **Effective for new movements.** New movements pick up the new configuration. The configuration-history activity log surfaces the change for the Auditor to review under the Sysadmin's audit-trail.

## 3. Decision Branches

- **Auditor lot-recall investigation** (read-only, escalates): the Auditor delivers the chain-of-custody trace and hands off to the **Quality / Recall lead** (write-off booking, customer notification) — they do not edit any document or transaction, do not initiate the write-off; the recall execution lives on [[inventory-adjustment]] (write-off via `tb_stock_out`) and on [[good-receive-note]] credit-note path (vendor concession). The Auditor's deliverable is the trace report.
- **Sysadmin location-type change blocked by non-zero on-hand**: an attempt to change `tb_location.location_type` from `inventory` to `direct` (or any inventory ↔ non-inventory transition) on a location with non-zero on-hand is **rejected** at the impact preview per `INV_AUTH_008`'s drain requirement. Resolution: drain the on-hand (transfer to a different inventory location or write off via stock-out), then re-attempt the location-type change. The system surfaces the on-hand summary in the rejection message.
- **Sysadmin costing-method change blocked by non-zero on-hand**: switching a product's `costing_method` (FIFO ↔ WEIGHTED_AVERAGE) on a product with non-zero on-hand is **blocked** per `INV_XMOD_009` because the cost-layer history is structurally tied to the method (FIFO requires `lot_seq_no` ordering; WA reads `average_cost_per_unit`). Resolution: drain stock (rare) or run a one-time elevated migration script that re-bases the cost layers under the new method — this is a Finance-coordinated event, not a routine Sysadmin action.
- **Sysadmin adjustment-type retirement blocked by historical references**: an attempt to hard-delete an `tb_adjustment_type` row referenced by historical `tb_stock_in` / `tb_stock_out` documents is **rejected**. Soft-deprecate (mark inactive) is offered as the resolution path — preserves audit-trail integrity while preventing the code from being picked on new documents. Same pattern as [[good-receive-note]]'s reason-code retirement rule.
- **Sysadmin period definition change**: standard periods are auto-generated by the scheduled job; exception periods (short period for fiscal-year change, mid-period boundary adjustment for a corporate event) land here as one-off Sysadmin edits. The change cannot overlap an existing period; the boundary must align to whole-day boundaries; periods cannot be redefined once any `tb_inventory_transaction` exists with a date in the period. Tight gates.
- **Sysadmin RBAC change affecting active session**: same pattern as the [[good-receive-note]] Audit / Config flow — the change takes effect on the next action attempt, not retroactively. Force-session-revocation is offered as an option. SoD rule changes (`INV_AUTH_010`) re-evaluate at the next adjustment post, not at the moment of the rule change.
- **Sysadmin threshold change retroactive scope**: lowering the Store-Keeper auto-approve threshold means in-flight documents below the old threshold that are still `draft` now route through Controller approval on next submit; documents already submitted (`in_progress` or later) stay on their original routing path. Raising the threshold means new draft documents route via auto-approve more often; in-flight `in_progress` documents complete on their current path.
- **Auditor sensitive-field export requires secondary approval**: exports including unit costs, vendor terms (joined via `inventory_doc_no → tb_good_received_note.vendor_id`), or PII (user identities beyond role) require a secondary approval per the audit pattern in [[good-receive-note/03-user-flow-audit-config]]. Plain activity-log exports (movement counts, lot lists, anonymised) do not require secondary approval.
- **Sysadmin period-lock RBAC enforcement** (forbidden — locked is immutable): a Sysadmin or Finance Manager attempt to re-open a `locked` period is rejected outright per `INV_AUTH_006` — locked is terminal. The Sysadmin's role in this branch is to ensure the RBAC layer correctly gates the lock → re-open path (no role has permission to advance `locked → open`); the only correction path for a locked period is a current-period restatement (a compensating adjustment in the currently-open period).
- **Sysadmin integration endpoint cutover**: cutting a source-module integration endpoint (e.g. GRN module API endpoint, SR module API endpoint, period-end job runner) runs a dual-write window where both old and new endpoints receive the transaction-side fan-out, followed by a reconciliation and a hard cutover. During the dual-write window the Auditor monitors the reconciliation report for fan-out mismatches; mismatches are resolved before the hard cutover.

## 4. Exit Point / Handoffs

The Audit / Config persona's involvement on a given inventory thread ends at one of three boundaries:

- **Auditor — trace / report generated.** The lot-trace or audit-log export is handed off to the requester (external auditor, Quality / Recall lead, Controller, Data Protection Officer). The Auditor does not edit any inventory transaction, does not initiate any downstream document (stock-in, stock-out, credit note); the deliverable is the read-only artefact. Follow-up handoffs from the report (recall write-off, vendor blacklisting, period-lock re-review) live on the owning persona's flow — Store Keeper / Inventory Controller / Finance / Finance Manager.
- **Sysadmin — configuration saved, effective for new movements.** The new configuration is persisted with an effective-from timestamp; the configuration history records the change for audit. New movements pick up the new rules; in-flight `draft` / `in_progress` source documents retain the snapshot of the rules in force at create time unless explicitly recomputed; `posted` transactions are immutable. The Sysadmin notifies the affected user community (Store Keepers, Inventory Controllers, Finance) of the rule change via the standard release-note channel.
- **Sysadmin — change rejected by impact preview, escalated.** Where the impact preview blocks a change (non-zero on-hand at a location whose type would change, in-flight references to a retiring code, RBAC deadlock, integration-endpoint dependency), the Sysadmin escalates to the controller or the cross-module change board to schedule a migration window, request a soft-deprecation path, or roll back the requested change. The inventory module itself does not transition; the configuration remains at the pre-change state until the migration plan is approved.
