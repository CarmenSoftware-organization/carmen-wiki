# Inventory — User Flow — Store Keeper

> **At a Glance**
> **Persona:** Store Keeper &nbsp;·&nbsp; **Module:** [inventory](/en/inventory/inventory) &nbsp;·&nbsp; **Workflow stages:** Initiate manual stock-in / stock-out drafts &nbsp;·&nbsp; auto-post below threshold (transaction posts directly into `tb_inventory_transaction`) &nbsp;·&nbsp; route above-threshold documents to Controller &nbsp;·&nbsp; execute physical counts on scoped locations &nbsp;·&nbsp; **Key permissions:** initiate / post below threshold; cannot edit cost-layer ledger; SoD (`INV_AUTH_010`) limits write-off of own-received lots above threshold
> **What this persona does:** Records day-to-day movements at the floor / location level and hands above-threshold documents off for Controller approval.

## 1. Role in This Module

The **Store Keeper** persona is the operator at the floor / location level — the user whose hands actually touch the stock. Within the inventory module their authority is **bounded**: they may **initiate** any routine movement (manual stock-in for found stock / count overage / vendor-replaced damaged stock, manual stock-out for breakage / expiry write-off / count shortage, physical-count execution on locations they have a `tb_user_location` mapping for) and they may **post** movements below the tenant-configured Inventory Controller threshold, but every above-threshold adjustment routes upward for approval. The Store Keeper does **not** edit the inventory cost-layer ledger directly, does **not** configure location type / costing method / adjustment-type reason codes, does **not** approve other Store Keepers' adjustments, and does **not** participate in period close. Their inventory-module ownership ends when (a) the source document (`tb_stock_in` / `tb_stock_out` / `tb_count_stock`) leaves their hands en route to Inventory Controller review, (b) the source document is auto-approved below threshold and posts directly into `tb_inventory_transaction`, or (c) a routine GRN receipt at the dock posts through to inventory (the GRN side of which is owned in detail by the [good-receive-note](/en/inventory/good-receive-note) Receiver persona; here the Store Keeper is the same physical operator but the inventory-side concern is just "did the movement land correctly"). Segregation of duties (`INV_AUTH_010`) restricts large write-offs of stock from a lot the same Store Keeper originally received; for those, an independent adjuster (Inventory Controller or a different Store Keeper) is required.

## 2. Entry Point and Primary Flow

**Entry points:** Three paths into the inventory side of the Store Keeper's daily work, plus the implicit fourth (GRN receipt) which lives in the GRN persona file.

- **Inventory module → Stock-In** — manual inbound for non-GRN reasons (found stock, count overage, recall replacement, vendor-supplied free-replacement that bypasses a GRN). Creates a `tb_stock_in` document at `doc_status = draft`.
- **Inventory module → Stock-Out** — manual outbound for non-issue reasons (breakage / damage write-off, expired-stock disposal, count shortage, recall write-off). Creates a `tb_stock_out` document at `doc_status = draft`.
- **Physical-count module → Run count** — sweeps the assigned location for a scheduled count window; the count outcome stages variance lines that, on approval, post as `tb_stock_in` (overage) or `tb_stock_out` (shortage) per `INV_XMOD_003`.
- *(Implicit)* **GRN receipt at the dock** — the Store Keeper may also be the Receiver on the GRN side; the inventory effect (cost-layer write at `(location, product, new_lot)`) fires on GRN commit per `INV_XMOD_001`. The Store Keeper's hands-on inventory work in this path is around lot-number entry and accepted-qty resolution; the full flow lives in [good-receive-note](/en/inventory/good-receive-note)'s Receiver persona file. This page references it but does not duplicate it.

**Primary flow (stock-in for found stock / count overage, 8 steps — illustrative of the manual-adjustment pattern):**

1. **Identify the discrepancy.** During a routine bin-stock check, a count run, or a vendor-replacement handover, the Store Keeper sees more physical stock on the shelf than the system shows (or finds stock that was previously thought lost). They confirm the discrepancy by recounting and by checking the lot label.
2. **Open the Stock-In screen.** Inventory module → Stock-In → New. The screen creates `tb_stock_in` at `doc_status = draft` with the Store Keeper as `created_by_id` and the assigned location pre-filled from the user's primary `tb_user_location` mapping (overridable to another location the user has scope for).
3. **Pick the adjustment reason.** Required: `adjustment_type_id` references `tb_adjustment_type` — typical reason codes for inbound are `FOUND_STOCK`, `COUNT_OVERAGE`, `RECALL_REPLACEMENT`, `VENDOR_FREE_REPLACEMENT`, `DATA_FIX`. The reason code drives downstream classification in the GL and in vendor-performance reporting.
4. **Add lines.** Per product: `product_id`, `qty` (positive), the lot's `lot_no` (existing lot if the stock matches a known lot label; new lot if found stock has no prior record), `cost_per_unit` (must match the originating receipt's lot cost if `lot_no` is existing; user-entered with Controller-approval gate if `lot_no` is new and the cost is non-zero). For perishable products, expiry date is required on a new lot.
5. **Verify the cost.** The screen renders the most recent `tb_inventory_transaction_cost_layer.cost_per_unit` for the lot (if existing) — the entry must match within tenant tolerance; mismatched costs flag for Controller review per `INV_AUTH_003`. For new-lot found stock, the cost must be defensible — typically read from a `[vendor-pricelist](/en/inventory/vendor-pricelist)` last-price or set to zero with an explanatory note.
6. **Submit for approval.** Click **Submit** — the source document transitions `draft → in_progress` per `enum_doc_status` on `tb_stock_in`. If the total `qty × cost_per_unit` is **below** the Store Keeper's auto-approve threshold, the document auto-advances and posts to inventory immediately (skip to step 8). If **above** threshold, the document routes to the Inventory Controller's queue per `INV_XMOD_005`.
7. **Inventory Controller reviews and approves** (or rejects with comment). On Controller approval, the document posts to inventory (`tb_stock_in.doc_status = completed`); on rejection, the document returns to `draft` with the Controller's comment in the activity log — the Store Keeper edits and re-submits, or voids the document if the discrepancy was a counting error on the Store Keeper's side.
8. **Post fires.** The system writes `tb_inventory_transaction` (`inventory_doc_type = stock_in`, `inventory_doc_no = tb_stock_in.id`), `tb_inventory_transaction_detail` (`qty > 0`), and `tb_inventory_transaction_cost_layer` (`in_qty > 0`, `transaction_type = adjustment_in`, `cost_per_unit` from the line, `lot_no` and `lot_index` per the inbound rules, `at_period` and `period_id` populated from the current `open` period). The Store Keeper's inventory-side ownership ends here; the on-hand at `(location, product, lot)` is now the new derived value per `INV_CALC_004`.

The **stock-out** flow follows the same shape but with negated direction (line `qty` negative, cost-layer `out_qty > 0`, `transaction_type = adjustment_out`), and routinely the cost-per-unit is **not** Store-Keeper-entered — it is FIFO- or weighted-average-picked per the product's costing method by the post engine (`INV_CALC_005` / `INV_CALC_006`). Above-threshold stock-out (breakage write-off, expiry write-off) routes to the Inventory Controller; very-large stock-out (large recall write-off) routes Controller → Finance for cost-impact approval.

The **physical-count** flow is owned by the [physical-count](/en/inventory/physical-count) module's persona files; the inventory-side post from a count is mediated through stock-in (overage) / stock-out (shortage) and follows steps 6–8 of the primary flow above.

## 3. Decision Branches

- **Existing-lot vs new-lot inbound.** When the discrepancy is on a lot the system already knows (`(location, product, lot_no)` exists in `tb_inventory_transaction_cost_layer`), enter the existing `lot_no`; the cost-per-unit is read from the lot's most recent layer and is not editable by the Store Keeper. When the discrepancy is on stock with no prior system record (true found stock), enter a new `lot_no` and the cost is editable — but the entry routes for Controller approval regardless of threshold because new-lot creation by a Store Keeper is a sensitive event.
- **Below vs above auto-approve threshold.** Below — document auto-advances to `completed` on Submit, inventory transaction posts immediately, Store Keeper's inventory-side work is done. Above — document routes to Inventory Controller's queue per `INV_XMOD_005`; the Store Keeper waits (the document is read-only to them while in `in_progress`).
- **Stock-out cost-per-unit not user-entered.** For stock-out (write-off / shortage / breakage), the Store Keeper enters `qty` only — the `cost_per_unit` is picked at post time by the costing engine. The Store Keeper sees the picked cost preview (FIFO from the oldest lot, or current weighted-average) before submitting, so they can flag if the picked cost looks anomalous (e.g. an unusually expensive lot being consumed for a routine write-off).
- **Negative-balance attempt rejected**: when a stock-out `qty` would drive `(location, product, lot)` below zero per `INV_VAL_005`, the document **cannot be submitted**. The Store Keeper either reduces the `qty`, picks a different lot, or escalates to the Controller to investigate (the discrepancy may indicate a missed inbound that wasn't recorded). The system's "no negative balance" rule is enforced at submit, not at post.
- **Location-type gate**: stock-in and stock-out documents cannot be raised against `location_type = direct` locations per `INV_VAL_009` — direct-cost locations carry no balance, so there is no "stock" to adjust. The screen filters the location picker to inventory-type (and consignment-type, where applicable). The Store Keeper sees a tooltip if they try to switch to a direct location.
- **Multi-line consolidation**: a single stock-in or stock-out document can carry multiple product lines (typical for a count-variance run — several products with shortage or overage). The threshold check sums across lines; a multi-line document where the total `Σ qty × cost_per_unit` exceeds the threshold routes for Controller approval even if each individual line is below threshold.

## 4. Exit Point / Handoffs

The Store Keeper's involvement on a given inventory movement ends at one of four boundaries:

- **Auto-approve post complete.** Below-threshold stock-in / stock-out posts immediately on submit. The Store Keeper's inventory-side work is done; no handoff. The activity log records `{ doc_status: 'completed', action: 'auto_approve_post' }`. If the post affects a count run, the count document advances accordingly per `INV_XMOD_003`.
- **Above-threshold handoff to Inventory Controller.** Document at `tb_stock_in.doc_status = in_progress` (or `tb_stock_out.doc_status = in_progress`) routes to **Inventory Controller** ([03-user-flow-inventory-controller.md](./03-user-flow-inventory-controller.md)) for approval. The Store Keeper re-engages only if the document is rejected (returned to `draft`) or if the Controller asks for additional supporting evidence (photos, count-sheet attachment, vendor RMA reference) on the activity log.
- **Count-variance handoff via the count document**. When the inventory-side post is the consequence of a physical / spot count completing, the handoff anchor is the count document's `tb_count_stock.status = completed` and the staged variance lines. The Store Keeper's count execution feeds Inventory Controller variance review per `INV_XMOD_003` and `INV_XMOD_004`; the Controller decides on the post (approve / adjust / re-count). The full count flow lives in [physical-count](/en/inventory/physical-count) / [spot-check](/en/inventory/spot-check).
- **Document voided pre-post.** If the discrepancy was a Store Keeper counting error caught before submit (or by the Controller on rejection), the document voids — `tb_stock_in.doc_status = voided` (or `tb_stock_out.doc_status = voided`) with reason text. No inventory transaction written, no balance change. The voided document remains in the database for audit per the standard soft-delete pattern.
