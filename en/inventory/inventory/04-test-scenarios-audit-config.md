# Inventory — Test Scenarios — Audit & Config

> **At a Glance**
> **Persona:** Audit / Config (Auditor + System Administrator) &nbsp;·&nbsp; **Module:** [inventory](/en/inventory/inventory) &nbsp;·&nbsp; **Scenarios:** ~32
> **Categories:** Happy Path &nbsp;·&nbsp; Permission &nbsp;·&nbsp; Validation &nbsp;·&nbsp; Edge Case
> **E2E coverage:** maps to `080-location.spec.ts`, `031-adjustment-type.spec.ts`, `900-period-end.spec.ts` in `../carmen-inventory-frontend-e2e/`

This page captures the test scenarios that the Audit / Config persona drives in the `inventory` module. The persona splits into two non-transactional sub-personas: the **Auditor** (strictly read-only across the full inventory dataset — `tb_inventory_transaction`, `tb_inventory_transaction_cost_layer`, `tb_period_snapshot`, configuration history; runs lot-recall traces, audit-log queries, and period-snapshot reconciliation queries) and the **System Administrator** (configures `tb_location` including `location_type` per `enum_location_type` and `physical_count_type`, the costing method per product on `tb_product.costing_method` — owned by [product](/en/inventory/product) module config, surfaced here, `tb_adjustment_type` reason codes, `tb_period` definitions, the threshold tier values that route Store Keeper / Controller / Finance approval, RBAC role definitions and `tb_user_location` scope mapping per `INV_AUTH_008`, and integration endpoints to the source modules). Neither sub-persona participates in the inventory movement lifecycle — the Auditor's queries return data without writing any state; the Sysadmin's edits apply to future movements (in-flight source documents retain a snapshot of the rules in force when they were created; `posted` inventory transactions are immutable per `INV_POST_012`). The shared rule IDs this persona's tests exercise are `INV_AUTH_006` (period lock / re-open authority — Sysadmin configures the RBAC), `INV_AUTH_008` (Sysadmin's config scope and the drain requirement on location-type / costing-method change), `INV_AUTH_009` (Auditor read-only access), `INV_AUTH_010` (SoD configuration), `INV_POST_011` / `INV_POST_012` (lock and reversal-via-compensation patterns the Sysadmin's RBAC layer gates), and `INV_XMOD_009` (costing-method drain requirement). Scenarios are grouped into **happy paths** (Auditor audit-log query, lot-recall trace, period-snapshot reconciliation query; Sysadmin location-type config, costing-method change with drain, adjustment-type maintenance, threshold edit, RBAC scope grant, integration cutover), **permission** (Auditor read-only allow, Sysadmin config-only deny on transact), **validation** (drain requirement on type change, period definition non-overlap, snapshot preservation on in-flight documents, RBAC deadlock prevention), and **edge cases** (in-flight snapshot boundary, rollback of bad config, lot-trace gap, query timeout, mass-RBAC, period-history hard-delete protection).

## 2. Permission / Authorization

| # | Scenario | Expected behaviour (allow/deny + reason) |
| - | -------- | --------------------------------------- |
| AUD-PERM-01 | Auditor reads inventory transactions, cost-layer ledger, period snapshots, configuration history | **Allow read-only per `INV_AUTH_009`.** Full read scope across all inventory entities including soft-deleted (compensated) rows. Render-only screens; no Save / Edit / Post / Approve affordances; backing API endpoints reject Auditor write attempts. |
| AUD-PERM-02 | Auditor attempts to edit any inventory entity | **Deny — role scope.** Direct API write (`POST /inventory-transaction`, `PATCH /inventory-transaction/{id}`, `POST /stock-in`, etc.) from Auditor token returns `403 Forbidden` with `"Auditor role is read-only on the inventory module."` The rejection itself logged on the audit-of-audit trail. |
| AUD-PERM-03 | Sysadmin configures location / costing method / adjustment types / periods / thresholds / RBAC / integrations | **Allow — config scope per `INV_AUTH_008`.** The configuration console renders the panels for the Sysadmin role; transactional users (Store Keeper, Controller, Finance) do not see the console. |
| AUD-PERM-04 | Sysadmin attempts to post an inventory transaction | **Deny — separation by design.** The Sysadmin role is configuration-only; not in any `tb_user_location` mapping by default; not in any approval workflow. Direct API `POST /stock-in` from Sysadmin token returns `"Sysadmin role is not authorised to transact on the inventory module — configuration only."` This is the deliberate separation that lets Sysadmin maintain `INV_AUTH_010` without bypassing it. |
| AUD-PERM-05 | Auditor exports a sensitive-field report — secondary approval required | **Allow with secondary approval.** Per AUD-HP-04, sensitive-field exports (unit cost, vendor terms, PII via joined tables) require Controller or DPO co-approval. Plain audit-log exports (movement counts, lot lists, anonymised user roles) bypass this. |
| AUD-PERM-06 | Sysadmin elevated period-lock or re-open attempt | **Deny — Finance Manager only per `INV_AUTH_006`.** The Sysadmin's role is to configure the RBAC that gates lock / re-open; the action itself is Finance Manager's. Sysadmin attempt to advance `tb_period.status = closed → locked` (or `locked → open`) returns `"Period state advancement requires the Finance Manager role; locked → open is forbidden for all roles."` |
| AUD-PERM-07 | Non-audit / non-sysadmin user accesses the audit / config workspace | **Deny.** Store Keeper / Controller / Finance navigating to `/admin/inventory-config` or `/audit/inventory-log` receives `403 Forbidden`. Each role's audit-trail visibility is scoped to their own actions (Store Keeper sees their own documents' activity log within the operational screens, etc.). |

## 3. Validation / Error

| # | Scenario | Trigger | Expected error |
| - | -------- | ------- | -------------- |
| AUD-VAL-01 | Sysadmin attempts location-type change on non-zero on-hand location | Existing `inventory`-type location `LOC-X` with on-hand qty `100` across various products / lots; Sysadmin attempts to switch `location_type` to `direct`. | **Reject at impact preview per `INV_AUTH_008` drain requirement.** Server returns `"Cannot change location_type on LOC-X: 100 units of on-hand stock remain. Drain via transfer or write-off before changing type."` Configuration not saved; Sysadmin's resolution is to transfer the stock or write it off, then re-attempt. |
| AUD-VAL-02 | Sysadmin attempts costing-method change on non-zero on-hand product | Product `P-Y` currently FIFO with non-zero on-hand at multiple locations; Sysadmin attempts to switch to WEIGHTED_AVERAGE. | **Reject at impact preview per `INV_XMOD_009`.** Server returns `"Cannot change costing_method on P-Y: 250 units of on-hand stock remain at FIFO. Drain stock (rare) or run elevated migration script under Finance authorisation."` AUD-HP-06 happens after drain; this is the negative path. |
| AUD-VAL-03 | Sysadmin attempts adjustment-type retirement with historical references | `RECEIVING_SHORT_DELIVERY` reason code referenced by 15 historical `tb_stock_out` documents over the past 12 months; Sysadmin attempts to hard-delete the code. | **Reject at delete.** Server returns `"Cannot hard-delete adjustment_type RECEIVING_SHORT_DELIVERY: 15 historical documents reference it. Use soft-deprecate to mark inactive while preserving audit-trail."` Soft-deprecate option is offered; on soft-deprecate, the code is hidden from the new-document dropdown but remains readable on existing documents. |
| AUD-VAL-04 | Sysadmin attempts overlapping period definition | Existing `tb_period` rows: `2026-04 (2026-04-01..2026-04-30)`, `2026-05 (2026-05-01..2026-05-31)`; Sysadmin attempts to create an exception period `2026-04W (2026-04-15..2026-04-29)` overlapping `2026-04`. | **Reject at save.** Server returns `"Period 2026-04W (2026-04-15..2026-04-29) overlaps existing period 2026-04 (2026-04-01..2026-04-30). Period date ranges must be non-overlapping."` Sysadmin re-plans the period boundaries (typically the exception period replaces the original, not overlaps). |
| AUD-VAL-05 | Sysadmin attempts to redefine an existing period with movements | `tb_period.id = 2026-04` exists; `tb_inventory_transaction` rows exist with transaction dates in `2026-04`; Sysadmin attempts to change `start_at` to a different date. | **Reject at save.** Server returns `"Cannot redefine period 2026-04: <N> inventory transactions reference this period. Period boundaries are immutable once movements exist."` Sysadmin's alternative is to create a fresh adjacent period for the new boundary, or to coordinate a one-off migration under Finance Manager authorisation. |
| AUD-VAL-06 | Sysadmin RBAC change deadlock prevention (no commit_grn role) | Sysadmin attempts to revoke `commit_grn` action from the only role that holds it; impact preview detects zero remaining users with commit authority. | **Reject at impact preview.** Server returns `"RBAC change would leave 0 users with commit authority on the GRN module, which would deadlock GRN → inventory flow."` Sysadmin re-plans the change (grant the action to a different role first, or grant to specific users before revoking from the role). |
| AUD-VAL-07 | Sysadmin invalid configuration shape (e.g. negative threshold) | Sysadmin enters Store Keeper auto-approve threshold = `-฿500.00` (negative). | **Reject at save.** Shape validation: `"Threshold values must be non-negative."` Sysadmin corrects and re-saves. Similar shape rules for tax rate, currency code, period date ranges, RBAC role / action / scope triple. |
| AUD-VAL-08 | Sysadmin attempts integration endpoint dual-write disable while reconciliation fails | Dual-write mode for `erp-v1` / `erp-v2` is on; reconciliation has been failing for 2 hours (some `erp-v2` writes failing intermittently); Sysadmin attempts to disable `erp-v1` and switch primary to `erp-v2`. | **Reject hard cutover with warning.** Server returns `"Cannot hard-cut to erp-v2: reconciliation has 12 unresolved mismatches in the last 2 hours. Resolve mismatches before cutover."` Sysadmin investigates the integration issue, fixes, re-runs reconciliation, then cuts over. Per AUD-HP-10 decision branch. |
| AUD-VAL-09 | Snapshot rule — Sysadmin changes adjustment-type reason that an in-flight draft references | Sysadmin attempts to retire reason code `OLD_REASON` that is referenced by 1 `tb_stock_in.doc_status = draft` document; impact preview surfaces the reference. | **Block hard-retire; offer paths per Section 3 of [03-user-flow-audit-config.md](./03-user-flow-audit-config.md).** Server returns `"Cannot retire OLD_REASON: 1 in-flight draft document references it. Resolution: (a) wait for draft to commit or void; (b) bulk re-save drafts against new config; (c) soft-deprecate."` Sysadmin picks resolution; in-flight draft retains its create-time snapshot per the snapshot rule. |

## 4. Edge Cases

| # | Scenario | Condition | Expected |
| - | -------- | --------- | -------- |
| AUD-EDGE-01 | In-flight snapshot boundary at threshold change | Store Keeper has a `tb_stock_in` document open at `draft` at `T`; Sysadmin saves a higher Store Keeper auto-approve threshold at `T + 100ms`; Store Keeper clicks Submit at `T + 1s`. | **Snapshot applies at submit time, not at draft create time.** The document's routing at submit reads the **current** threshold (the new higher one); if the document is now below the new threshold, it auto-approves rather than routing to Controller. Mirrors the snapshot rule but reversed direction — threshold is evaluated lazily. Stock-in / stock-out documents read threshold at the routing decision point. |
| AUD-EDGE-02 | Rollback of bad config change after no in-flight effect | Sysadmin saves a bad threshold value at `T` that incorrectly routes high-value documents below threshold (e.g. typo turned `฿500` into `฿50,000`); caught by monitoring at `T + 5min`; rolls back. | **Restore prior config from configuration history.** Per `INV_AUTH_008` config-history retention, the prior version is preserved; Sysadmin clicks **Roll back to previous version**; prior threshold becomes active immediately; new submits use the restored value. No movements in `T..T+5min` were affected because no Store Keeper submitted in the window; had any submitted, those documents would have followed the bad routing path and require Finance review to determine if reposting is needed. |
| AUD-EDGE-03 | Lot-recall trace with gap (orphan inventory transaction) | A `tb_inventory_transaction_cost_layer.lot_no = LOT-GAP-9` row exists but the inventory transaction's source document (e.g. the GRN with `inventory_doc_no = X`) is soft-deleted by a historical migration error; lot exists but provenance is broken. | **Surface the gap in the trace report.** Forward trace from `LOT-GAP-9` returns the downstream consumption successfully; backward trace flags `"Source document <X> for inventory transaction <Y> is soft-deleted or missing; provenance gap. Recommend forensics."` Chain-of-custody export includes the gap flag for the external auditor; no auto-repair (preserves forensic integrity). |
| AUD-EDGE-04 | Auditor query timeout on huge dataset | Auditor enters a filter spanning 12 months across all locations; row count exceeds tenant compute budget. | **Graceful timeout with partial result.** Server cancels at configured timeout (e.g. 30s); returns partial result up to T+30s with `truncated = true` flag; recommends offline export job for full dataset. Same pattern as the GRN audit module. |
| AUD-EDGE-05 | Mass-RBAC activation — N users granted across N locations | Onboarding of 10 new locations with 10 Store Keepers each (100 users × 10 locations = 1000 `tb_user_location` mappings); Sysadmin bulk-imports. | **Allow with atomic transaction; configuration-history entry per mapping.** All-or-nothing — if any single mapping fails (e.g. SoD conflict with existing role), the entire batch rolls back; on success, 1000 configuration-history entries written; users see scope on next login. |
| AUD-EDGE-06 | Period history hard-delete protection | A `tb_period_snapshot` row exists for `2026-03` (a closed period); Sysadmin or Finance Manager attempts to soft-delete or hard-delete the period definition. | **Reject delete on period with snapshots.** Server returns `"Period 2026-03 has <N> snapshot rows and <M> inventory transactions; periods with movements cannot be deleted."` The period definition is immutable post-movements; this preserves the audit chain. Same protection extends to `tb_period_snapshot` row deletion (forbidden) and `tb_inventory_transaction_cost_layer` row deletion (only via the compensating-transaction soft-delete pattern of `INV_VAL_013`). |
