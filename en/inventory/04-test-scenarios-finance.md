# Inventory — Test Scenarios — Finance

This page captures the test scenarios that the Finance persona (Finance Officer / Inventory Accountant for day-to-day reconciliation, plus Finance Manager / Controller for period-lock authority) directly drives in the `inventory` module. Finance is the valuation authority — the role that approves above-Controller-threshold adjustments per `INV_AUTH_005`, runs the inventory-to-GL reconciliation per `INV_XMOD_008`, orchestrates the period-end close per `INV_POST_009` / `INV_POST_010`, and (at the Finance Manager level) advances `tb_period.status = closed → locked` per `INV_AUTH_006` / `INV_POST_011`. Finance does **not** post inventory transactions directly — corrections flow through stock-in / stock-out / credit-note documents that Finance approves; period-end fan-out fires under system context per `INV_AUTH_007`. Their inventory-module ownership begins when (a) a Controller escalates a cost-impact adjustment, (b) a periodic reconciliation pass is due, or (c) the Controller signs off variance for the closing period, and ends when (a) the document posts, (b) the reconciliation is clean, (c) the period transitions `open → closed`, or (d) the Finance Manager advances `closed → locked`. Scenarios are grouped into **happy paths** (cost-impact approval, reconciliation pass, period close + open-next, period lock, period re-open within audit window), **RBAC** (Finance scope vs Controller / Sysadmin, Finance Manager exclusivity on lock), **validation** (reconciliation variance, close-with-open-items, lock-after-window), and **edge cases** (boundary variance, FX revaluation at period end, multi-period reconciliation, locked-period correction path).

## 2. Permission / Authorization

| # | Scenario | Expected behaviour (allow/deny + reason) |
| - | -------- | --------------------------------------- |
| FIN-PERM-01 | Finance Officer approves above-Controller-threshold adjustments | **Allow per `INV_AUTH_005`.** Finance is the second signature on documents above the Controller cap. The Approve button is enabled in the cost-impact approval queue. |
| FIN-PERM-02 | Finance Officer runs inventory-to-GL reconciliation | **Allow.** Reconciliation runs under Finance role per `INV_XMOD_008`; the dashboard is read+post (read the sub-ledger sums, post compensating journals to align). |
| FIN-PERM-03 | Finance Officer attempts to lock the period | **Deny — Finance Manager only per `INV_AUTH_006`.** The **Lock** button is hidden for the Finance Officer role; a direct API call returns `"Period lock requires the Finance Manager role."` The Officer runs the close (`closed → open → closed`); the Manager advances to `locked`. |
| FIN-PERM-04 | Finance Manager re-opens a closed period | **Allow with audit log per `INV_AUTH_006`.** The Re-open action requires a justification string; the action is audit-logged. Multiple re-opens of the same period within the audit window are permitted but increasingly scrutinised; locked periods cannot be re-opened by any role. |
| FIN-PERM-05 | Anyone attempts to re-open a locked period | **Deny per `INV_AUTH_006`.** Locked is terminal. Server returns `"Period <YYMM> is locked; locked periods cannot be re-opened by any role."` The only correction path is current-period restatement. |
| FIN-PERM-06 | Finance Officer attempts to edit a `posted` inventory transaction | **Deny — terminal state per `INV_POST_012`.** Same rule as Controller / Store Keeper: every transaction is immutable. Finance can post a compensating credit-note or approve a corrective stock-in / stock-out. |
| FIN-PERM-07 | Finance Officer attempts to configure period definition / costing method | **Deny — Sysadmin only per `INV_AUTH_008`.** Finance consumes the configuration; they do not edit it. Period definitions for normal cadences are auto-generated by the scheduled job; exceptional periods require Sysadmin action. |

## 3. Validation / Error

| # | Scenario | Trigger | Expected error |
| - | -------- | ------- | -------------- |
| FIN-VAL-01 | Approve a stock-out that would drive on-hand negative at approve time | Same race as IC-VAL-01 but at Finance threshold: between Controller approval and Finance approval, on-hand at `(location, product, lot)` has dropped further due to other postings. | **Reject at Finance approve** — `INV_VAL_005` re-checked per `INV_POST_002`. Server returns the same negative-balance error; Finance returns the document to Controller for re-evaluation (split / reduce qty / re-route). |
| FIN-VAL-02 | Reconciliation pass marked clean with above-tolerance variance | Finance Officer attempts to click **Mark reconciliation clean** on a `LOC-A` variance of `฿250.00` (above the `฿1.00` tolerance) without resolving. | **Reject at mark-clean** — server returns `"Variance ฿250.00 exceeds tolerance ฿1.00; resolve via compensating journal or corrective adjustment before marking clean."` Finance must drill, resolve, re-run reconciliation, then mark. |
| FIN-VAL-03 | Period close attempted with `in_progress` source documents | Finance clicks **Close period 2026-04** while a `tb_stock_in` document for the period is still `in_progress`. | **Reject at close** — server returns `"Cannot close period 2026-04: <N> source documents are still at draft or in_progress. Resolve all in-flight documents before closing."` Finance routes back to Controller / Store Keeper for resolution. |
| FIN-VAL-04 | Period close attempted without Controller variance sign-off | Per FIN-HP-05 — the same gate from the close-orchestration side. | **Reject at close.** See FIN-HP-05. The pre-close checklist surfaces the missing sign-off; the **Close period** button is disabled until Controller signs off. |
| FIN-VAL-05 | Period close attempted with unresolved reconciliation variance | Reconciliation for the closing period has not been run, or the latest run has unresolved variance above tolerance. | **Reject at close** — server returns `"Cannot close period 2026-04: inventory-to-GL reconciliation has unresolved variance above tolerance (variance: ฿X, tolerance: ฿Y)."` Finance must resolve before close. |
| FIN-VAL-06 | Period re-open by Finance Officer (not Manager) | Finance Officer attempts to re-open `closed` period via API. | **Reject** per FIN-PERM-03. Server returns `"Period re-open requires the Finance Manager role."` |
| FIN-VAL-07 | Period lock attempted before audit window passes | Finance Manager attempts to lock `2026-04` (closed) at current date `2026-04-15` (audit window not yet passed; tenant policy = 60 days). | **Reject at lock with override path** — server returns `"Period 2026-04 audit window has not yet passed (lock-eligible date: 2026-06-01). Confirm early lock with audit-grade justification to proceed."` If Finance Manager confirms with justification, the lock fires; otherwise stays `closed`. Tenant policy may forbid early lock outright. |
| FIN-VAL-08 | Lock attempted with open audit-correction requests | Finance Manager attempts to lock `2026-04` (closed, audit window passed) but the audit-correction-request tracker shows one open item. | **Reject at lock** — server returns `"Cannot lock period 2026-04: 1 open audit-correction request remains."` Finance Manager resolves the request (typically a credit-note post or a current-period restatement) before re-attempting lock. |

## 4. Edge Cases

| # | Scenario | Condition | Expected |
| - | -------- | --------- | -------- |
| FIN-EDGE-01 | Reconciliation variance exactly at tolerance | Variance `฿1.00` exactly equals tolerance `฿1.00`. | **Boundary inclusive — clean.** Treats `≤ tolerance` as clean. `฿1.01` would fail. Same inclusive pattern as SK-EDGE-01 / IC-EDGE-01. |
| FIN-EDGE-02 | Concurrent close attempts by two Finance Officers | Officer A clicks **Close period 2026-04** at `T`; Officer B clicks at `T+200ms`. | **First click wins; second is rejected.** A's close fires; period transitions to `closed`. B's click returns `"Period 2026-04 was closed at <timestamp> by <user A>. Refresh."` No double-close; no double-snapshot-write. |
| FIN-EDGE-03 | FX revaluation at period end (multi-currency inventory) | Foreign-currency-sourced inventory (e.g. imported `P-USD` from USD vendor) sits at `2026-04` close; the FX rate at `2026-04-30` differs from the rate at receipt. | **Snapshot uses receipt-time cost; FX revaluation is a separate GL journal.** `tb_period_snapshot.closing_cost_per_unit` uses the cost-layer's `cost_per_unit` which is in base currency at receipt-time FX. FX revaluation to current rate (if tenant policy requires) is a manual journal Finance posts to the FX Gain/Loss account; the inventory snapshot itself is **not** rewritten with the new FX rate. The cost-layer ledger preserves the original cost. |
| FIN-EDGE-04 | Multi-period reconciliation (prior period still open) | `2026-03` and `2026-04` are both `open` (Finance hasn't run the `2026-03` close yet because of a Q1 audit hold); Finance attempts to reconcile `2026-04`. | **Allow reconciliation on the current open period.** Sub-ledger sums per period scoped on `at_period` (`YYMM`); reconciliation runs against the GL entries timestamped within the period. The Finance close must run periods in order: `2026-03` close must happen before `2026-04` close (else the `2026-04` opening doesn't have a valid prior closing to read from). |
| FIN-EDGE-05 | Locked-period correction via current-period restatement | Period `2026-03` is `locked`; auditor identifies a missed write-off that should have posted in `2026-03`; Finance posts the corrective stock-out in current open period `2026-05` with a note linking back to `2026-03`. | **Posts cleanly in `2026-05`; audit trail via `note` / `info` JSON.** The corrective stock-out's transaction date is `2026-05` (the current open period); the `note` field carries the cross-period reference. Restatement disclosure in the financial reports for `2026-05` is a Finance reporting concern, not an inventory-module concern; the module just posts the movement. |
| FIN-EDGE-06 | Decimal precision on AP-reconciliation amount | Cost-layer sum involves many small-decimal amounts; rounding accumulates to `฿2.34` variance over thousands of rows. | **Per-row 5dp; aggregate rounded to 2dp; absorbed into reconciliation tolerance.** Each cost-layer row stored at 5dp; aggregation re-reads rounded prior step per `INV_CALC_012`; accumulated rounding drift is within typical tolerance (e.g. `฿1.00` per cost-centre per period for active accounts). Persistent above-tolerance drift indicates a systemic issue and routes to Sysadmin / Auditor for investigation. |
