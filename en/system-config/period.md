# Period

## 1. Purpose

Periods define the accounting calendar Carmen operates on — one row per fiscal month, identified by a `YYMM` code plus explicit `fiscal_year` / `fiscal_month` integers and a `[start_at, end_at)` date range. Every period has a status — `open`, `closed`, or `locked` — that gates which dated documents can post into it. Periods are the unit at which inventory cost is closed and snapshotted: when finance closes January, no further January GRNs / issues / adjustments may post, and the closing balance becomes the opening balance of February.

The companion table `tb_period_snapshot` stores the per-location / per-product / per-lot inventory snapshot at a point in time within (or at the close of) a period. The snapshot is the audit-grade frozen view of opening qty, in-period movements (receipts, issues, adjustments), closing qty, and per-unit and total cost — generated by the costing engine and used for trial balances, GL postings, and roll-forward to the next period.

## 3. Usage / Cross-References

- [[inventory]] — current-stock writes must pass the "period open" guard before posting; closing snapshots roll forward into the next period's opening balance.
- [[costing]] — the costing engine reads movement layers (`tb_inventory_transaction_cost_layer` keyed by `period_id`) and writes `tb_period_snapshot` at close.
- [[good-receive-note]] — `posting_date` must fall within an `open` period; back-dated postings into a `closed` period are rejected.
- [[inventory-adjustment]] — adjustments check period status before commit; closed-period adjustments require re-opening or a journal in the next open period.
- [[physical-count]] — count documents are frozen against a period via `tb_physical_count_period`; variance posting cannot land in a `closed` or `locked` period.
- [[spot-check]] — spot-check variance postings inherit the same period guard as IA.

## 4. Configuration UI

Managed by **Sysadmin** (typically the Finance Manager role) under System Configuration → Period. The screen lists periods chronologically with status badges; admin actions are *Close* (locks new postings but allows last-minute corrections by Finance), *Lock* (no postings, no corrections), and *Reopen* (only available with explicit audit reason — emits a `tb_period_comment`). The snapshot view is read-only and shows the per-location balance grid produced at close.

## 5. Business Rules

- **Uniqueness.** Both `period` and `(fiscal_year, fiscal_month)` are unique among non-deleted rows. The two should agree (`period == fiscal_year-2000 * 100 + fiscal_month` for 2000-2099). Mismatches are an input error.
- **Date range integrity.** `start_at` must be strictly before `end_at`. Adjacent periods must not overlap and should be contiguous (the close of January is the open of February).
- **Status progression.** Allowed transitions: `open → closed`, `closed → open` (reopen), `closed → locked`. `locked` is terminal under normal operation — unlock requires DB-level override and is audit-logged via [[reporting-audit/activity]].
- **Posting guard.** Every document with a posting_date attribute must resolve to a period whose status is `open`. `closed` and `locked` reject. The runtime resolves the period by date-range membership.
- **Snapshot generation.** A snapshot row is keyed by `(period_id, snapshot_at)` per `(location, product, optional lot)`. Re-running close for the same period either overwrites or appends depending on Finance policy; recommended pattern is *append at a new `snapshot_at`* so the audit trail is preserved.
- **Roll-forward.** Opening balances of period N+1 must equal closing balances of period N for every (location, product, lot) tuple. Discrepancies are surfaced in `diff_amount`.
- **Deletion guards.** Periods with any related snapshot, cost layer, or document posting cannot be deleted. Soft-delete only after archive; live deletion breaks the GL chain.
