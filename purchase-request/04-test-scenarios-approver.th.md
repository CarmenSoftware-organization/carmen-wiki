# ใบขอซื้อ — กรณีทดสอบ — ผู้อนุมัติ (Approver)

หน้านี้รวบรวมกรณีทดสอบที่ persona ของ Approver เป็นผู้ขับเคลื่อนโดยตรงในโมดูล `purchase-request` Approver ครอบคลุมผู้ตัดสินใจสามตำแหน่งในห่วงโซ่อนุมัติ — **Department Head** (Stage 1), **Budget Controller** (Stage 2) และ **Finance Officer / Manager** (Stage 3) — ซึ่งใช้หน้าจอ review-and-decide เดียวกัน กรณีทดสอบถูกจัดเป็นสี่ action ของ happy path ตามที่อธิบายไว้ใน [03-user-flow-approver.th.md](./03-user-flow-approver.th.md) หัวข้อ 2 (approve, send-back, reject, split-reject), ขอบเขต RBAC ที่บังคับโดย `PR_AUTH_002`–`PR_AUTH_006` ใน [02-business-rules.th.md](./02-business-rules.th.md) หัวข้อ 4, validation rule ที่ trigger เมื่อทำ approval action (โดยเฉพาะ `PR_VAL_013` กับ `approved_qty` และ `PR_VAL_016` กับ optimistic concurrency) และชุด edge case จำนวนเล็ก ๆ ที่มาจาก state machine แบบหลาย stage เส้นทาง cross-persona ที่หมุนรอบ Approver (`X-PR-02`, `X-PR-03`, `X-PR-05`, `X-PR-06`, `X-PR-07`) อยู่ในหน้าภาพรวมพ่อ ไม่ได้อยู่ในหน้านี้

## 2. Permission / Authorization

| # | กรณี | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | ---- | --------------------------------------- |
| APP-PERM-01 | Approver เปิด PR ในคิวของตัวเอง (ผู้ใช้ปัจจุบันอยู่ใน `user_action.execute[]` ของ `workflow_current_stage`) | **Allow** อ่านและเห็นปุ่ม action (Approve / Send Back / Reject) `PR_AUTH_002` ผ่านสำหรับ stage ปัจจุบัน |
| APP-PERM-02 | Approver เปิด PR ที่อยู่คนละ stage (เช่น HOD เปิด PR ที่ตอนนี้อยู่ที่ stage Budget Controller) | **Deny action, read-only** ปุ่ม action (Approve / Send Back / Reject / Split toolbar) ถูก disable พร้อม inline ข้อความ; `PR_AUTH_002` recompute `user_action.execute[]` ทุกครั้งที่เปลี่ยน stage และ HOD ไม่อยู่ในรายชื่อนั้นแล้ว; การมองเห็นใน list อาจยังให้แถวปรากฏใต้ **All Documents** ขึ้นกับสิทธิ์ |
| APP-PERM-03 | Approver กด **Approve** ที่ stage ของตัวเอง | **Allow** `PR_AUTH_002` ผ่านสำหรับ stage ปัจจุบัน; `PR_AUTH_003` ให้ approve / send-back / reject เป็นสิทธิ์รายบรรทัด; `PR_POST_004` (หรือ `PR_POST_005` ถ้า stage สุดท้าย) commit การเปลี่ยน |
| APP-PERM-04 | Approver กด **Approve** บน PR ที่เลย stage ของตนไปแล้ว (race หลังจาก Approver อื่นทำพร้อมกัน) | **Deny** ฝั่ง server `PR_AUTH_002` reject เพราะ `user_action.execute[]` ถูก recompute สำหรับ stage ใหม่ และผู้เรียกไม่อยู่ในรายชื่อแล้ว; UI แสดง "PR has moved to a later stage — please refresh" และ disable แถบ action เมื่อ refresh |
| APP-PERM-05 | Department Head พยายามอนุมัติ PR ของแผนกอื่นที่ตนไม่รับผิดชอบ | **Deny** routing default-chain ของ Stage 1 จำกัด `user_action.execute[]` ให้อยู่ในขอบเขตของแผนกผู้ร้องขอ (หรือ HOD ที่ตั้งไว้ใน `tb_workflow` สำหรับแผนกนั้น) HOD ของแผนกอื่นไม่อยู่ในรายชื่อ; `PR_AUTH_002` บล็อก action |
| APP-PERM-06 | Approver พยายามอนุมัติ PR ที่ตนเป็นผู้ submit เอง (segregation of duties) | **Deny** บทบาท stage `create` และ stage `approve` ปลายน้ำเป็นชุดไม่ทับซ้อนตาม workflow definition แม้จะตั้ง misconfig ให้ผู้ใช้คนเดียวกันอยู่ในสอง stage `user_action.execute[]` ก็ถูก recompute ทุก stage และเจ้าของ stage `create` ถูก filter ออกก่อนที่ action จะรัน (ดู [02-business-rules.th.md](./02-business-rules.th.md) หัวข้อ 4 default chain) |
| APP-PERM-07 | ผู้ใช้ delegate (ตั้งตาม `PR_AUTH_006`) ทำ action บน PR ที่ delegate ในช่วง delegation window | **Allow** การ delegate ถูก materialise เข้าไปใน `user_action.execute[]` สำหรับช่วง delegation; delegate ได้สิทธิ์ approve / send-back / reject / split-reject เฉพาะในช่วงนั้น `last_action_by_id` บันทึก delegate; audit comment บันทึกแหล่ง delegation |
| APP-PERM-08 | Approver เปิด URL ของ PR ใด ๆ ด้วย fixture `noAuth` / session หมดอายุ | **Deny — redirect ไปหน้า login** เมื่อไม่มี auth context ก็ไม่สามารถ evaluate `user_action.execute[]` กับผู้ใช้ปัจจุบันได้; `PR_AUTH_002` ผ่านไม่ได้; ครอบคลุมโดย fixture `noAuthTest` ใน `301-pr.spec.ts` |

## 3. Validation / Error

| # | กรณี | Trigger | ข้อผิดพลาดที่คาดหวัง |
| - | ---- | ------- | -------------------- |
| APP-VAL-01 | Approver act บน PR ที่ตนไม่มีสิทธิ์ | Approver ที่ **ไม่** อยู่ใน `user_action.execute[]` ของ `workflow_current_stage` เรียก endpoint Approve ตรง ๆ (ข้าม UI ที่ disable ปุ่มไว้) | `PR_AUTH_002` — reject ด้วย `"You are not authorised to act on this PR at the current stage"` รายชื่อถูก recompute ทุก stage transition; client ที่ stale จะวิ่งเข้า path นี้ |
| APP-VAL-02 | Approver กด **Send Back** โดยไม่กรอกเหตุผล | เปิด dialog send-back, ปล่อยช่อง reason ว่าง, กด Confirm | Reject — เหตุผลเป็น mandatory ตาม contract ของ dialog; ปุ่ม **Confirm** ก็ถูก disable ใน UI เมื่อ reason ว่าง แต่ถ้าเรียก API ตรง server จะ reject ด้วย `"A reason is required to send back this PR"` comment `type = system` ที่ `PR_POST_003` เขียนต้องมีข้อความเหตุผล ถ้าไม่มี การเขียน fail |
| APP-VAL-03 | Approver กด **Reject** โดยไม่กรอกเหตุผล | เปิด dialog reject, ปล่อย reason ว่าง, กด Confirm | Reject — `PR_AUTH_004` + `PR_POST_006` ต้องการ system comment ที่แก้ไม่ได้ พร้อมเหตุผลการปฏิเสธ (`PR_POST_008`); server reject การเรียกที่ไม่มีเหตุผลด้วย `"A reason is required to reject this PR"` |
| APP-VAL-04 | Approver ตั้ง `approved_qty = 0` แล้วกด Approve | บนบรรทัด พิมพ์ `0` ลงในช่อง approved-qty | `PR_VAL_013` — reject ด้วย `"Approved quantity must be positive and may not exceed requested quantity"` กฎคือ `> 0` strict ไม่ใช่ `≥ 0` (ดู APP-EDGE-01 — UI ก็ถือว่า `approved_qty = 0` เป็นการ reject รายบรรทัดเมื่อทำผ่าน bulk split toolbar; validation path ใต้ toolbar ยัง reject การ save `0` แบบดิบ ๆ) |
| APP-VAL-05 | Approver ตั้ง `approved_qty > requested_qty` | บนบรรทัดที่ `requested_qty = 100` พิมพ์ `120` ในช่อง approved-qty | `PR_VAL_013` — reject ด้วย `"Approved quantity must be positive and may not exceed requested quantity"` การเปรียบเทียบทำหลังการแปลง UoM ด้วย `approved_unit_conversion_factor` |
| APP-VAL-06 | Approver แก้ `approved_qty` แต่ไม่ใส่ `approved_unit_id` / `approved_unit_conversion_factor` | บังคับ save โดยให้ `approved_qty` มีค่าและ `approved_unit_id` เป็น null (ปกติเข้าได้ผ่าน API call ตรง ๆ เท่านั้น) | `PR_VAL_013` — reject ด้วย `"Approved quantity must be positive and may not exceed requested quantity"` (กฎยังกำหนดให้ `approved_unit_id` และ `approved_unit_conversion_factor` ต้องมาคู่กับ `approved_qty`) |
| APP-VAL-07 | Approver พยายามอนุมัติ PR ที่ `workflow_id` ที่ต้องใช้ไม่ active แล้ว (scope ถูกเปลี่ยน หรือถูก soft-delete กลางคัน) | แถวใน `tb_workflow` ถูก deactivate หรือ scope ถูก flip จาก `purchase-request` หลังจาก PR นี้ submit ไปแล้ว; Approver กด Approve | Reject — `PR_VAL_004` ถูก re-evaluate ตอนทำ action; `"A valid PR workflow must be selected"` PR ติดอยู่จนกว่า admin จะ re-activate workflow ที่ valid หรือ void PR (`PR_AUTH_007`) |
| APP-VAL-08 | Approver แก้ `approved_qty` บน `doc_version` ที่ stale (มีการแก้คู่ขนานอีกเส้น) | Approver สองคน (หรือ Approver กับ delegate) load PR เดียวกัน; A save แก้ `approved_qty`; B (ยังถือ `doc_version` เดิม) กด Approve | `PR_VAL_016` — reject ด้วย `"Document was modified by another user; reload and retry"` การเขียนที่สำเร็จจะ bump `doc_version` ขึ้น 1 |
| APP-VAL-09 | Split-reject ที่ไม่มีเหตุผลรายบรรทัดบนบรรทัดที่ปฏิเสธ | Mark `L2` เป็น **reject** แต่ปล่อยเหตุผลรายบรรทัดว่าง แล้วกด Approve ที่ header | Reject — การปฏิเสธรายบรรทัดต้องมีเหตุผลตาม `PR_AUTH_003` และ contract ของการเขียน comment รายบรรทัด (`PR_POST_008` ต้องการ payload comment) UI ก็ disable ปุ่ม Approve จนกว่าทุกบรรทัดที่ reject จะมีเหตุผลที่ไม่ว่าง |
| APP-VAL-10 | Approver พยายาม act หลังจาก PR ถูก void โดย Finance / sys-admin | `pr_status` เพิ่งถูก flip เป็น `voided` (`PR_AUTH_007`); Approver โหลดหน้าก่อนการ void แล้วกด Approve | Reject — state-machine guard: `voided` เป็น terminal และ freeze เอกสาร; endpoint Approve / Send Back / Reject reject ด้วย `"This PR is voided and cannot be acted on"` |

## 4. Edge Cases
