# ใบขอซื้อ — กรณีทดสอบ — ผู้จัดการฝ่ายจัดซื้อ (Procurement Manager)

หน้านี้รวบรวมกรณีทดสอบที่ persona ของผู้จัดการฝ่ายจัดซื้อ (Procurement Manager) เป็นผู้ขับเคลื่อนโดยตรงในโมดูล `purchase-request` Procurement Manager ดูแล **สองพื้นผิว** ตามที่อธิบายใน [03-user-flow-procurement-manager.th.md](./03-user-flow-procurement-manager.th.md): พื้นผิว **transactional** — stage อนุมัติสุดท้ายแบบ escalate ที่ทำงานเมื่อ `base_total_amount` เกิน threshold มูลค่าสูงที่ตั้งไว้ตาม `PR_AUTH_005` และใช้หน้าจอ review-and-decide เดียวกับห่วงโซ่ผู้อนุมัติ (Approver chain) พื้นฐานแต่มีขอบเขตกว้างกว่า (รวมถึงสิทธิ์ override การ send-back ของผู้อนุมัติก่อนหน้า) — และพื้นผิว **configurational** — ชุดกฎ Allocate Vendor (scoring weight ของ vendor ranking, priority override รายผู้ขาย และ bulk action ใน Stuck PR Oversight) ที่ขับเคลื่อนการเลือก vendor ของ Purchaser ปลายน้ำ กรณีทดสอบจัดเป็น happy path ตาม [03-user-flow-procurement-manager.th.md](./03-user-flow-procurement-manager.th.md) หัวข้อ 2 (escalated high-value approve / reject / send-back, ปรับ scoring weight, ปรับ priority ของ Allocate Vendor, bulk-action กับ stuck PR, override กฎ delegation), ขอบเขต RBAC จาก `PR_AUTH_002`–`PR_AUTH_008` ใน [02-business-rules.th.md](./02-business-rules.th.md) หัวข้อ 4, validation ที่ทำงานบนแต่ละพื้นผิว (`PR_AUTH_005` semantics ของ threshold, `PR_VAL_013` กับ `approved_qty`, การคง snapshot ตาม `PR_CALC_006`) และชุด edge case เฉพาะกรณีที่ persona ถือสิทธิ์ทั้ง transactional และ configurational เส้นทาง cross-persona ที่หมุนรอบ Procurement Manager (`X-PR-05` threshold escalation, `X-PR-10` returned-PR round trip เมื่อ PR ที่ escalate ถูก send-back กลับเป็น draft) อยู่ในหน้าภาพรวมพ่อ ไม่ได้อยู่ในหน้านี้

## 2. Permission / Authorization

| # | กรณี | พฤติกรรมที่คาดหวัง (allow/deny + เหตุผล) |
| - | ---- | --------------------------------------- |
| PM-PERM-01 | Procurement Manager เปิด PR ที่ escalate (ผู้ใช้ปัจจุบันอยู่ใน `user_action.execute[]` ของ stage escalate) | **Allow** อ่านได้และเห็นแถบ action เต็ม (Approve / Send Back / Reject / Split-Reject) `PR_AUTH_002` ผ่านสำหรับ stage escalate; stage escalate เองเป็น stage role `approve` โดยการ route ของ `PR_AUTH_005` จึงได้สิทธิ์ระดับบรรทัดและระดับ header เหมือน Approver chain พื้นฐานตาม `PR_AUTH_003` และ `PR_AUTH_004` |
| PM-PERM-02 | Procurement Manager เปิด PR ที่ **ไม่ได้** escalate ถึงตน (PR ปัจจุบันอยู่ที่ stage Budget Controller ซึ่งเป็น stage ก่อนหน้าในห่วงโซ่) | **Deny action, read-only** `PR_AUTH_002` re-compute `user_action.execute[]` ของ stage ปัจจุบันทุกการเปลี่ยน stage; Procurement Manager ไม่อยู่ใน executor list ของ Stage 2 list visibility อาจยังทำให้แถวโผล่ใน **All Documents** ตาม grant แต่ปุ่ม Approve / Send Back / Reject ถูก disable พร้อมคำอธิบายอินไลน์; การเรียก API ตรง ๆ ถูก reject ที่ server ด้วย `"You are not authorised to act on this PR at the current stage"` |

| PM-PERM-05 | Procurement Manager รัน bulk action กับ PR หลายฉบับจากมุมมอง Stuck PR Oversight | **Allow per-PR** bulk action ยังคงผ่าน `PR_AUTH_002` รายฉบับ; PR ที่มีสิทธิ์เดินต่อและเขียน audit comment รายฉบับ (`PR_POST_008`); PR ที่ Procurement Manager ไม่มีสิทธิ์ (เช่น PR ของ business unit อื่นที่อยู่นอก `user_action.execute[]` ของตน) ถูกข้ามเงียบ ๆ ในผลของ bulk โดยมีบรรทัดอธิบายใน audit log dialog ผลลัพธ์แสดงจำนวน PR ที่ได้รับผล vs ที่ถูก exclude |

| # | กรณี | Trigger | ข้อผิดพลาดที่คาดหวัง |

| PM-VAL-01 | Threshold ของการอนุมัติไม่ตรง — PR ไม่ได้เกิน threshold ของ Procurement Manager จริง | PR ที่ถูก route ผิดด้วย `base_total_amount = 50,000 ฿` หล่นเข้าคิว Escalated PRs แต่ threshold escalation ที่ตั้งไว้คือ `> 100,000 ฿` ตาม `PR_AUTH_005`; Procurement Manager กด Approve | Reject — `PR_AUTH_005` re-validate threshold ตอน action server reject ด้วย `"This PR does not meet the escalated-stage threshold and should not be at this stage"`; action ถูก block และ workflow engine ถูก flag ให้ซ่อม stage routing PR คงอยู่ใน `in_progress` ที่ stage ก่อนหน้าที่ควรจะหยุด |
| PM-VAL-02 | บันทึก rule ไม่มี diff (no-op save) | เปิด Vendor Allocation Rules → กด **Save** โดยไม่เปลี่ยน weight หรือ override ใด ๆ | Reject — configuration writer ต้องการ diff ที่ไม่ว่างเทียบกับ rule set ที่ active server reject ด้วย `"No changes detected — nothing to save"` UI ก็ disable ปุ่ม **Save** จนกว่าจะมี staged change อย่างน้อยหนึ่งรายการ ไม่มีการเขียนแถว effective-from ใหม่และไม่ส่ง notification |
| PM-VAL-03 | บันทึก rule เปลี่ยนขณะที่ PR ระหว่างทางยังขึ้นกับกฎเดิม — การคง snapshot ตาม `PR_CALC_006` | PR `in_progress` มี `vendor_id`, `vendor_name`, `pricelist_detail_id`, `pricelist_price` ที่ snapshot ไว้ตอน PR submit ภายใต้ rule set **เก่า**; Procurement Manager บันทึก rule set ใหม่ 5 นาทีต่อมา; ผู้อนุมัติ approve PR หลังจากนั้น | **อนุญาตให้ save แต่ snapshot ถูกคงไว้** การ save ฝั่ง configurational ไม่ถูก block ด้วย PR ที่อยู่ระหว่างทาง (ตาม [03-user-flow-procurement-manager.th.md](./03-user-flow-procurement-manager.th.md) หัวข้อ 3); แต่ PR ที่อยู่ระหว่างทางคง snapshot allocation **เดิม** ไปจนถึงการแปลงเป็น PO กฎใหม่มีผลเฉพาะกับ: (a) PR ที่สร้างหลัง timestamp effective-from, (b) PR ที่ถูกส่งกลับเป็น `draft` ด้วย send-back และ (c) PR ที่ถูกรันผ่าน bulk re-allocation ครั้งเดียว การพยายาม re-rank vendor บน PR `in_progress` ย้อนหลังผ่าน API ตรง ๆ ถูก reject ด้วย `"Cannot re-allocate vendor on an in-flight PR — use Stuck PR Oversight bulk re-allocation"` |
| PM-VAL-04 | Weight ของ vendor priority ไม่ summed เป็น 100 % | เปิด Scoring Weights และใส่ `{ vendor_rank: 50, lowest_price: 30, last_receiving: 30 }` (sum = 110) | Reject — scoring weight ต้องรวมเป็น 100 พอดีระหว่าง 3 เกณฑ์ server reject ด้วย `"Scoring weights must sum to 100"` UI ก็ขึ้นตัวบ่งชี้ `Σ = 110 ≠ 100` แบบ live และ disable ปุ่ม **Save** จนกว่าผลรวมจะถูก panel จำลองไม่ re-rank จนกว่า validation จะผ่าน |
| PM-VAL-05 | scoring weight ไม่ถูกต้อง — ค่าเป็นลบ | ใส่ `{ vendor_rank: -10, lowest_price: 60, last_receiving: 50 }` | Reject — scoring weight แต่ละค่าต้อง `0 ≤ weight ≤ 100` server reject ด้วย `"Scoring weight must be a non-negative percentage (0–100)"`; UI flag input ที่ติดลบเป็น inline error และ disable **Save** path เดียวกันก็ reject ค่าที่เกิน 100 ด้วย operator เดียวกัน |
| PM-VAL-06 | Approve / Reject / Send Back โดยไม่กรอกเหตุผลในที่ที่บังคับ | เปิด dialog **Reject** หรือ **Send Back** ปล่อย reason textarea ว่าง กด Confirm | Reject — เหตุผลเป็น mandatory ตาม contract ของ dialog Reject (`PR_AUTH_004` + `PR_POST_006`) และ Send Back (`PR_POST_003`); ทั้งคู่ต้องการ payload ของ system comment ที่ immutable (`PR_POST_008`) UI disable ปุ่ม Confirm เมื่อ reason ว่าง; การเรียก API ตรง ๆ ถูก reject ด้วย `"A reason is required to reject this PR"` (Reject) หรือ `"A reason is required to send back this PR"` (Send Back) Approve ใส่ความเห็นเป็น optional |
| PM-VAL-07 | แก้ `approved_qty` ที่ stage escalate แต่ผิด `PR_VAL_013` | บนบรรทัดที่มี `requested_qty = 100`, Procurement Manager พิมพ์ `approved_qty = 120` แล้วกด Approve | `PR_VAL_013` — reject ด้วย `"Approved quantity must be positive and may not exceed requested quantity"` stage escalate inherit กฎ validation ระดับบรรทัดทั้งหมดของ Approver พื้นฐาน; กฎเป็น strict `> 0` และ `≤ requested_qty` ประเมินหลังแปลง UoM ด้วย `approved_unit_conversion_factor` |
| PM-VAL-08 | พยายาม override การ send-back ของ Approver บน PR ที่จบไปแล้ว | Approver ก่อนหน้าได้กด Send Back แล้ว PR กลับเป็น `draft` แล้ว; ผู้ร้องขอยังไม่ resubmit; Procurement Manager พยายาม override การ send-back จาก UI ที่ stale ด้วยการกด Approve | Reject — state-machine guard: PR ไม่ได้อยู่ใน `in_progress` ที่ stage escalate แล้ว `PR_AUTH_002` re-evaluate กับ `workflow_current_stage` ปัจจุบัน (stage `create` ของผู้ร้องขอ) แล้วพบว่า Procurement Manager ไม่อยู่ใน `user_action.execute[]` ของ stage นั้น server reject ด้วย `"This PR has been sent back to the Requestor and cannot be approved without resubmission"` UI ของ Procurement Manager prompt ให้ refresh |

## 4. Edge Cases

| # | กรณี | เงื่อนไข | ผลที่คาดหวัง |
| - | ---- | -------- | ------------ |
| PM-EDGE-01 | ขอบเขต snapshot ระหว่าง PR กำลังเดินอยู่กับ rule ใหม่ | PR-A submit ที่ `T-0` ภายใต้ rule set `R1`; vendor snapshot ถูก take ตอน submit Procurement Manager บันทึก rule set `R2` ที่ `T+5m` PR-B ถูกสร้างที่ `T+6m` และ submit ที่ `T+7m` | PR-A คง snapshot `R1` ผ่านการ approve และเข้า PO conversion — column `vendor_id`, `vendor_name`, `pricelist_*` บน `tb_purchase_request_detail` ไม่เปลี่ยน vendor snapshot ของ PR-B ตอน submit ถูก take ภายใต้ `R2` — weight ใหม่, per-vendor override ใหม่ถูก apply ขอบเขตคือ `timestamp` effective-from ของ `R2` ไม่ใช่เวลาที่ PR ถูก approve สะท้อนแพทเทิร์น snapshot ของ exchange rate ใน `PR_CALC_006` |
| PM-EDGE-02 | ขอบเขต threshold เป๊ะ — `base_total_amount` เท่ากับขอบล่างของ Procurement Manager | escalation threshold ตั้งไว้ที่ `base_total_amount > 100,000.00000 ฿`; ผู้อนุมัติปรับ `approved_qty` จน `base_total_amount = 100,000.00000 ฿` (เป๊ะ) | การเปรียบเทียบ threshold เป็น strict `>` ตาม `PR_AUTH_005` ดังนั้นค่าขอบเป๊ะ **ไม่** trigger การ escalate; PR **ไม่** route ไปยัง Procurement Manager และจบที่ stage Finance ปกติแทน ค่า `100,000.00001 ฿` จะ escalate (จำนวน threshold เฉพาะคอนฟิกได้ต่อองค์กร; operator คือ invariant ไม่ใช่ตัวเลข — แพทเทิร์นเดียวกับ APP-EDGE-05) |
| PM-EDGE-03 | Bulk-action สำเร็จบางส่วน — บาง PR สำเร็จ บาง PR ล้มเหลว | Procurement Manager เลือก 10 PR จาก Stuck PR Oversight สำหรับ Escalation Ping 7 ฉบับผ่าน `PR_AUTH_002` ของ Procurement Manager และอยู่ที่ stage ปัจจุบันที่ถูกต้อง; 2 ฉบับ scope กับ business unit ที่ Procurement Manager ไม่มีสิทธิ์; 1 ฉบับเพิ่งถูก Finance void (`PR_AUTH_007`) ภายในนาทีที่ผ่านมา | First-pass aggregation: `PR_AUTH_002` และ state-machine guard ทำงานต่อ PR 7 ฉบับที่มีสิทธิ์และอยู่ในสถานะที่ถูกต้องได้ ping และ comment `type = system` บันทึกแหล่ง bulk (`PR_POST_008`) 2 ฉบับที่ไม่มีสิทธิ์ถูกข้ามเงียบ ๆ พร้อมบรรทัดอธิบายใน log 1 ฉบับที่ถูก void ถูก reject ที่ state-machine guard ด้วย `"This PR is voided and cannot be acted on"` และโผล่ใน failure breakdown dialog ผลลัพธ์แสดง `7 succeeded / 3 failed (2 unauthorised, 1 voided)` ไม่มี transactional rollback บางส่วน — PR ที่สำเร็จคง ping ไว้; PR ที่ล้มเหลวไม่ถูกแตะ |
| PM-EDGE-04 | Delegation chain มาที่ Procurement Manager หมดอายุระหว่างตัดสิน | Procurement Manager A ลาและ delegate อำนาจ stage escalate ให้ Procurement Manager B ตาม `PR_AUTH_006` ในหน้าต่าง `[T-2d, T]`; B เปิด PR ที่ escalate ที่ `T-1m` ตรวจสอบ และกด Approve ที่ `T+30s` (หลัง delegation หมดอายุ) | Reject ที่ server `PR_AUTH_002` re-evaluate `user_action.execute[]` กับ clock ปัจจุบัน; delegate B ที่หมดอายุไม่อยู่ใน list อีกแล้ว server ส่ง `"Your delegation has expired — please refresh"` UI โชว์ข้อความและ prompt ให้ refresh Procurement Manager A เดิม (หรือ delegate ใหม่) ต้อง act แพทเทิร์นเดียวกับ APP-EDGE-04 ใน Approver scenarios |
| PM-EDGE-05 | Configurational save vs one-off re-allocation ที่ทำพร้อมกัน | Procurement Manager เปิด **Stuck PR Oversight** และเริ่ม one-off re-allocation สำหรับ 3 PR ที่ `T`; ก่อนที่ bulk job จะ commit, Procurement Manager B บันทึก rule set ใหม่ที่ `T+1s` | one-off re-allocation อ่าน rule set ณ ช่วงเวลาที่ per-PR re-allocation แต่ละฉบับรัน per-PR re-allocation ที่รันก่อน `T+1s` apply `R_old`; ที่รันหลัง `T+1s` apply `R_new` ไม่มีการรับประกันแบบ transactional ว่าทั้ง 3 ฉบับใช้ rule set เดียวกันใน window ของการ write พร้อมกัน audit log ของ bulk action บันทึก timestamp ของ rule set ที่ใช้ต่อ PR ทำให้สังเกตการแยกได้ ในเชิง operational เกิดยาก — `Stuck PR Oversight` และ rule editor ระบุไว้ว่าใช้โดย Procurement Manager ทีละคน |
| PM-EDGE-06 | บันทึก rule set ลงพอดีกับช่วงที่ PR ใหม่กำลัง submit | ที่ `T`, ผู้ร้องขอกด Submit บน PR-X (ซึ่ง trigger routine ของการ snapshot vendor); ในวินาทีเดียวกัน `T`, Procurement Manager บันทึก rule set `R2` (effective-from `T`) | Tie-break ตาม clock ระบบที่เวลา write ถ้า routine snapshot ของ PR-X resolve rule set ที่ `T-1ms` จะอ่าน `R1`; ถ้าที่ `T+1ms` จะอ่าน `R2` ทั้งสองผลถูกต้อง — contract คือ "ใช้ rule set ที่ effective ณ ช่วงเวลาของ snapshot" ไม่ใช่ "ให้ PR-X ใช้ rule set ที่ preferred" audit trail บนทั้งแถว configuration และแถว PR detail บันทึก timestamp ทำให้สังเกตการเลือกได้ |
